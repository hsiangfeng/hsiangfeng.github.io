Which Claude Code Files Should You Commit? CLAUDE.md, Settings, Rules — A Complete Breakdown

Intro

If you've already been using Claude Code for development, you've probably hit this problem: your project suddenly has a bunch of files you've never seen before — CLAUDE.md, .claude/settings.json, .claude/rules/, etc. — and now you're hesitating about whether to commit any of this.

For me personally, the first time I used Claude Code on a team project, I really hesitated about whether to push CLAUDE.md. After all, my teammates weren't necessarily using Claude Code — would pushing it be weird? Then I found that the project also had settings.json and settings.local.json — the two looked really similar, so what's the difference?

Once I dug into Claude Code's settings architecture, I realized it's actually been designed to clearly separate "team-shared" from "personal preferences". As long as you understand the layering, you don't need to hesitate at all about what to commit.

So this post is about Claude Code's settings file layers and load order. Once you understand that, you'll know where each file fits.

Claude Code's settings layers

Before talking about individual files, let's understand Claude Code's overall settings architecture — that's the key to deciding what to commit.

Claude Code's settings aren't a single layer — they're divided into four:

LayerWho configures itScope
Managed (org level)IT / DevOps adminsEveryone in the org
User (user level)YouAll your projects
Project (project level)The teamEveryone on this project
Local (machine level)YouThis project, just you

In short, Managed is rules forced by company admins, User is your personal cross-project preferences, Project is shared team standards for the project, Local is your personal settings for a specific project.

This layering runs through CLAUDE.md, Settings, and Rules — all three settings systems. So just remember one principle: anything inside the project directory and meant for the team gets committed; anything in your Home directory or with local in the name doesn't get committed.

Let me walk through each file with this framework.

CLAUDE.md load order

CLAUDE.md is Claude Code's most central rules file — but it's not just for the project root. You can put it in several places, and Claude reads them all and merges them at startup.

A real example makes this easier to understand. Say you run Claude Code from my-project/packages/api/. It loads the following CLAUDE.md files in order:

  1. Managed level (if your company has set it up)
  2. ~/.claude/CLAUDE.md (your user level)
  3. my-project/CLAUDE.md (project root)
  4. my-project/packages/CLAUDE.md (intermediate directory, if any)
  5. my-project/packages/api/CLAUDE.md (the directory you ran the command from)

A diagram makes it clearer:

Claude Code CLAUDE.md load order diagram

Left to right is load order. The further right, the higher the priority — and the brighter the green. So if your user-level CLAUDE.md says "respond in English" but the project's CLAUDE.md says "respond in Traditional Chinese", Claude follows the project.

Also, CLAUDE.md files in subdirectories (like src/components/CLAUDE.md) don't all get loaded at startup — Claude dynamically loads them when it accesses files in that directory, so the entire project's CLAUDE.md doesn't get crammed into Context all at once.

So which ones should you commit? Easy — any CLAUDE.md inside the project gets committed, whether it's the root one or in a subdirectory, because that's the team-wide standard. The user-level ~/.claude/CLAUDE.md is in your Home directory anyway — not in the Repo — so no need to worry about it.

If you don't know how to write CLAUDE.md, check out my earlier post CLAUDE.md and Rules for AI writing tips (article in Mandarin).

So where do I put my personal preferences?

This is a question lots of people have: the project already has a CLAUDE.md — where do my own preferences go? Should I just modify the project's CLAUDE.md?

Don't do that. Personal preferences should go in ~/.claude/CLAUDE.md. That's your user-level config and applies to all your projects, and it's not in any Repo, so it won't affect your teammates.

For example, the project's CLAUDE.md might look like this:

markdown
# My Project

- Use pnpm as the package manager
- commit messages use Conventional Commits format

Your personal ~/.claude/CLAUDE.md can contain your own preferences:

markdown
# Personal preferences

- Always respond in Traditional Chinese
- Don't use emojis in responses
- When explaining code, keep it simple

Claude reads both — team standards and personal preferences both apply. If they conflict, the project level has higher priority, so your personal preferences won't override team standards.

How Settings get merged

The Settings concept is similar to CLAUDE.md — also layered — but with an extra Local layer.

Looking at the actual files:

  • ~/.claude/settings.json → your personal global settings (User layer)
  • .claude/settings.json → team-shared project settings (Project layer)
  • .claude/settings.local.json → your personal settings for this project (Local layer)

Priority from highest to lowest: Managed > Local > Project > User.

Claude Code Settings layer diagram

Here's an important detail: Settings array values aren't overridden — they're merged.

A concrete example makes this clearer. Say the project's .claude/settings.json allows two commands:

json
{
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test)"
    ]
  }
}

And your personal ~/.claude/settings.json allows one more:

json
{
  "permissions": {
    "allow": [
      "Bash(npm run dev)"
    ]
  }
}

Claude Code doesn't override your personal settings with the project's — it merges both, so you end up with all three allowed commands: npm run lint, npm run test, and npm run dev.

This design lets the team set shared baseline permissions, with each person adding their own on top — without interfering with each other.

So which one should you commit? .claude/settings.json is team-shared — commit it. .claude/settings.local.json is personal — don't commit it. And if you used claude init to initialize, it's already added to .gitignore for you.

The Rules directory

Rules is Claude Code's modular rules system. You can split rules into multiple .md files under .claude/rules/ — easier to manage than stuffing everything into one CLAUDE.md.

Like CLAUDE.md, Rules are layered:

  • ~/.claude/rules/*.md → your personal rules common across all projects
  • .claude/rules/*.md → team-shared project rules

User-level Rules load first, project-level Rules load after with higher priority.

A neat feature of Rules is you can use paths to scope where they apply:

markdown
---
paths:
  - "src/api/**/*.ts"
---
# API development standards

- All API calls must be wrapped in try-catch

Rules without paths get fully loaded at startup. Rules with paths only load when Claude is processing files matching those paths — saves Tokens and keeps unrelated rules from interfering.

The project's .claude/rules/*.md should be committed. Personal ~/.claude/rules/ is in your Home directory and you don't need to worry about it.

The Skills directory

Skills is Claude Code's skill-pack system. You can package the knowledge and procedures for a specific task into a standalone Skill, and Claude automatically loads it when relevant.

Like Rules, Skills are also two-layered:

  • ~/.claude/skills/ → your personal Skills common across all projects
  • .claude/skills/ → team-shared project Skills

Each Skill is a folder that contains at least a SKILL.md:

text
.claude/
└── skills/
    └── code-review/
        └── SKILL.md

For teams, things like the Code Review process, pre-deploy checklists, or framework-specific dev standards — you make these into Skills under the project's .claude/skills/. The whole team can use them, so they should be committed.

For personal Skills (like your own habitual debug flow), put them under ~/.claude/skills/ — not in the Repo.

If you want to dig into Skills details and how to write them, check out Agent Skills: More Token-Efficient Than MCP — Demoed With Claude Code.

Auto Memory

Auto Memory files live under ~/.claude/projects/<project>/memory/ — that's your Home directory, not in the Repo. They don't show up in git status at all, so there's nothing to do.

If you want to learn more about Auto Memory, check out Claude Code Auto Memory complete guide.

What if my teammate doesn't use Claude Code?

This is something a lot of people worry about: I pushed CLAUDE.md and .claude/ up, but my teammate isn't using Claude Code at all — does it affect them?

Not at all. To someone not using Claude Code, CLAUDE.md is just a plain Markdown file. .claude/settings.json is just a JSON file sitting there quietly — no side effects on any dev tool.

Conversely, committing this stuff has an upside: if your teammate ever decides to try Claude Code, they can clone and instantly use the existing rules — no setup from scratch.

Recommended .gitignore settings

Actually you only need to add one line:

gitignore
# Claude Code - personal settings
.claude/settings.local.json

Everything else personal (~/.claude/CLAUDE.md, ~/.claude/settings.json, ~/.claude/rules/, Auto Memory) is in your Home directory, not in the Repo — no need to add it to .gitignore.

If you initialized with claude init, this line is usually already added.

A reminder about sensitive info

Last reminder — whether it's CLAUDE.md or Rules, do not put API Keys, Tokens, passwords, or anything like that inside.

These config files don't usually have this problem because they hold rules and preferences, not values. But if you write something like "use this API Key to call XXX" inside CLAUDE.md, that's risky.

If you really need to reference sensitive info in your rules, use environment variables. Write "use environment variable XXX_API_KEY" — don't put the actual value in.

Wrap up

Back to the original question — Claude Code's settings architecture has actually already drawn a clear line between team and personal. Anything under the project's .claude/ (settings.json, rules/, skills/) plus the project's CLAUDE.md is team-shared — commit it. Anything with local in the name (settings.local.json) is personal — don't commit it. Everything under ~/.claude/ is in your Home directory — has nothing to do with Git.

For personal preferences, don't try to cram them into the project's settings. Claude Code's user level (~/.claude/CLAUDE.md, ~/.claude/settings.json, ~/.claude/rules/) is designed for personal preferences — write them there. They apply to all your projects and don't affect your teammates.
