CLAUDE.md and Rules for AI writing tips

Introduction

If you're already using AI dev tools like Claude Code or GitHub Copilot, you've probably heard of "Rules for AI" — basically a document that tells the AI how to do things in your project.

But a lot of people write one and find that the AI still doesn't really listen. Either it ignores the rules, or the code it produces still misses the mark. Most of the time, the real problem is that the rules file itself isn't written well.

So this post is about how to write that file properly, so the AI actually understands what your project needs. I'll mostly use Claude Code's CLAUDE.md as the example, but the ideas apply to any AI dev tool you're using.

Note

If you don't know what CLAUDE.md is yet, check out my earlier post 想用 Claude Code 開發？這篇帶你從入門到進階 (article in Mandarin) for a full walkthrough.

Rules files across AI tools

Before we dig in, a quick look at what each tool calls its rules file and where it lives. The names differ but the core idea is the same.

AI toolRules fileLocation
Claude CodeCLAUDE.mdProject root
Cursor.cursor/rules/*.mdc.cursor/rules/
GitHub Copilot.github/copilot-instructions.md.github/

Format and placement vary, but the goal is the same: tell the AI about your project's conventions and preferences.

So the writing tips below apply to whichever tool you use — only the syntax differs slightly.

Common ways people get this wrong

Before getting into the right way, let's look at a few common bad patterns that leave the AI with no real direction.

Too vague

Markdown
# Project rules

Please write good code.
Use best practices.
Be mindful of performance.

This is basically writing nothing, right? "Good code", "best practices", "be mindful of performance" — all way too abstract. There's nothing concrete here for the AI to act on.

Too long-winded

Markdown
# Project rules

This project was founded in 2023. After lengthy team discussions, we decided to
build it with React, having considered Vue.js, Angular, and Svelte. We chose
React because most of the team was already familiar with it and the ecosystem
was more mature. In 2024 we then introduced TypeScript because...
(another 500 lines of history)

The AI doesn't need your project's life story. It just needs to know what to do right now. All that backstory burns tokens and dilutes the parts that actually matter.

Contradictory rules

Markdown
- Every function must have a JSDoc comment
- Code should be self-explanatory; no unnecessary comments

These two contradict each other. The AI has no way to pick one consistently, so the result is random — and your code style ends up inconsistent.

Tip 1: Use concrete, actionable instructions

Good rules should be specific and clear enough that the AI knows exactly what to do after reading them.

Bad:

Markdown
Please use appropriate naming.

Good:

Markdown
## Naming conventions

- Variables and functions: camelCase (e.g. `getUserName`, `isLoading`)
- Components: PascalCase (e.g. `UserProfile`, `LoginForm`)
- Constants: UPPER_SNAKE_CASE (e.g. `API_BASE_URL`, `MAX_RETRY_COUNT`)
- CSS classes: kebab-case (e.g. `user-profile`, `login-form`)
- Boolean variables: prefix with `is`, `has`, `should` (e.g. `isActive`, `hasPermission`)

See the difference? The good version doesn't just say what to use — it also gives examples the AI can match against.

Tip 2: Lean on positive phrasing and examples

Telling the AI what to do tends to work better than telling it what not to do. AI sometimes glosses over the "don't" and ends up doing the very thing you were forbidding.

Bad:

Markdown
- Don't use the any type
- Don't use var
- Don't call APIs directly inside functions

So how do you do it well? Use a right-vs-wrong contrast so the AI clearly sees what you want.

Good (TypeScript example):

All variables and functions must have explicit types. any is forbidden.

Correct:

TypeScript
const getUserName = (userId: string): Promise<string> => { ... }

Wrong:

TypeScript
const getUserName = (userId: any): any => { ... }

Good (API call example):

API calls go in the services/ folder. Don't use fetch or axios directly in business logic.

Correct:

TypeScript
····// services/userService.ts
export const getUser = async (userId: string): Promise<User> => {
  const response = await apiClient.get(`/users/${userId}`);
  return response.data;
};

// In business logic
import { getUser } from '@/services/userService';

const user = await getUser('123');

Wrong:

TypeScript
// Calling the API directly inside business logic
const response = await fetch('/api/users/123');
const user = await response.json();

With that kind of contrast the AI gets a much clearer picture of the style you want — and giving sample code helps it produce code that matches your style.

Tip 3: Structure the rules file

A good rules file should have a clear structure so the AI can quickly find the right section. Here's the structure I personally recommend:

Markdown
# Project name

Short description (1-2 sentences)

## Tech stack

- Framework: Vue 3 + TypeScript 5.7
- Build tool: Vite
- State management: Pinia
- Styling: SCSS
- Testing: Vitest
- Package manager: pnpm

## Project structure

src/
├── components/    # Shared components
├── views/         # Page components
├── composables/   # Composable functions
├── services/      # API calls
├── stores/        # Pinia state
├── types/         # TypeScript type definitions
└── utils/         # Utility functions

## Development conventions

(naming, style, testing, etc.)

## Common commands

- `pnpm dev`: Start dev server
- `pnpm build`: Build the project
- `pnpm test`: Run tests
- `pnpm lint`: Run linter

A few things worth highlighting:

  1. Pin versions in your tech stack: Tell the AI which version you're on, so it doesn't reach for deprecated APIs
  2. List the project structure: So the AI knows where files belong
  3. List common commands: So the AI knows which command to run for which task

Tip 4: Separate concerns

Once your rules grow, don't cram everything into one file. Splitting it into multiple files is much easier to maintain. Personally I try to keep each file under 100-200 lines — beyond that, the AI starts skipping over things.

For Claude Code there are two ways to split:

Option 1: Markdown link references

Claude Code's CLAUDE.md officially supports @path/to/file syntax to import other files, and imported files get loaded into context at startup. Personally I use plain ./ Markdown links more often, because then on GitHub readers can click through to the linked doc — handy for teammates who aren't using Claude Code but still want to read the conventions.

Text
CLAUDE.md                          # Main file: project overview + references
docs/
├── PROJECT-STRUCTURE.md           # Project architecture
├── CODE-STYLE.md                  # Code style rules
├── GIT-COMMIT.md                  # Commit conventions
└── API-GUIDELINES.md              # API design rules

The main CLAUDE.md stays lean and just references the rest:

Markdown
# My Project

An e-commerce platform built with Vue 3 + TypeScript.

## Important documents

- [Project structure](./docs/PROJECT-STRUCTURE.md)
- [Code style rules](./docs/CODE-STYLE.md)
- [Commit conventions](./docs/GIT-COMMIT.md)
- [API design rules](./docs/API-GUIDELINES.md)

Option 2: The .claude/rules/ directory

Another way is to put rules under .claude/rules/, with each .md file covering one topic:

Text
············.claude/
├── CLAUDE.md
└── rules/
    ├── code-style.md    # Code style
    ├── testing.md       # Testing conventions
    └── git-commit.md    # Commit conventions

Rule files without a paths setting get loaded entirely at startup, while ones with paths only load when Claude is working on matching files (more on this in the next tip).

Both approaches share two upsides:

  1. Save tokens: Avoid one giant monolithic file
  2. Easier to maintain: Each doc owns one topic, so edits don't ripple into unrelated rules

Tip 5: Scope rules by path

Some rules don't apply to the whole project — only to certain files or folders. That's where path-scoped rules come in.

For Claude Code's .claude/rules/*.md, you can use the paths field in the YAML frontmatter to scope which files the rule applies to:

Markdown
··---
paths:
  - "src/api/**/*.ts"
---
# API development rules

- All API calls must be wrapped in try-catch
- Use a unified error response format
- Return types must be explicitly annotated

paths supports glob patterns. Common ones:

PatternMatches
**/*.tsAll TypeScript files
src/**/*Everything under src/
src/**/*.{ts,vue}.ts and .vue files under src/
tests/**/*.test.tsTest files

You can also list multiple paths:

Markdown
····---
paths:
  - "src/api/**/*.ts"
  - "src/services/**/*.ts"
---

This way the AI only loads these rules when actually working on matching files. Unrelated rules don't bleed into other parts of the project, and you save tokens at the same time.

Tip 6: Spell out the why behind decisions

For a few key rules, beyond telling the AI what to do, briefly say why. That way, when it hits a similar but not-explicitly-covered situation, it can still make a call that matches your intent.

Markdown
## State management

Use Pinia for state management — not Vuex.
Why: Pinia is the officially recommended state manager for Vue 3, has a simpler
API, and supports TypeScript natively.

## Testing strategy

Integration tests are primary; unit tests are secondary.
Why: Component interactions are complex, so integration tests give better
confidence in real behaviour. Every page should have at least one integration test.

That said — you don't need a "why" for every rule. Only add one when it's a rule the AI might second-guess or interpret differently. Otherwise the doc just bloats.

Tip 7: Add step-by-step guidance for common tasks

This one's lesser-known: you can put SOPs for common tasks directly in the rules file. So when someone says "create a new page", the AI just follows the flow — no re-explaining each time.

Markdown
## Common task playbooks

### When the user says "create a new page"

1. Create the corresponding `.vue` file under `src/views/`
2. Add the route in `src/router/`
3. If there's shared logic, create a composable under `src/composables/`
4. Create the matching test file

### When the user says "add an API integration"

1. Create or update the service file under `src/services/`
2. Define the request and response TypeScript types under `src/types/`
3. Use `apiClient` (don't call fetch or axios directly)
4. Create the matching test file

This is especially good for team development. Encode the patterns once, and the AI can follow the same flow every time without you having to explain it verbally.

Tip 8: Set the response language and style

Last tip — if you want the AI to respond in a specific language or style, write that down too.

Markdown
## Response conventions

- Always respond in Traditional Chinese
- Use Conventional Commits for commit messages (e.g. `feat: add login feature`)
- Commit message descriptions in Traditional Chinese
- Variable names and code comments in English

Looks simple, right? But if you don't write it down, the AI will randomly switch between English and Simplified Chinese, and your project's language ends up inconsistent.

A complete example

Pulling all the tips together, here's a full CLAUDE.md example you can use as a reference:

Markdown
# E-Commerce Platform

E-commerce frontend built with Vue 3 + TypeScript.

## Tech stack

- Framework: Vue 3.5 (Composition API + `<script setup>`)
- Language: TypeScript 5.7 (strict mode)
- Build tool: Vite 6
- Routing: Vue Router 4
- State: Pinia
- Styling: SCSS + BEM naming
- Testing: Vitest + Vue Test Utils
- Package manager: pnpm
- Node.js: v22

## Important documents

- [Project structure](./docs/PROJECT-STRUCTURE.md)
- [Code style rules](./docs/CODE-STYLE.md)
- [Commit conventions](./docs/GIT-COMMIT.md)
- [API design rules](./docs/API-GUIDELINES.md)
- [Testing conventions](./docs/TESTING.md)

## Common commands

- `pnpm dev`: Start dev server (port 5173)
- `pnpm build`: Build the project
- `pnpm test`: Run tests
- `pnpm test:coverage`: Run tests with coverage report
- `pnpm lint`: Run ESLint

## Development conventions

### Naming

- Variables and functions: camelCase (`getUserName`)
- Components and types: PascalCase (`UserProfile`, `UserType`)
- Constants: UPPER_SNAKE_CASE (`API_BASE_URL`)
- Component filenames: PascalCase (`UserProfile.vue`)
- Non-component filenames: camelCase (`useUser.ts`, `userService.ts`)
- Booleans: `is`/`has`/`should` prefix (`isActive`)

### TypeScript

- `any` is forbidden — use `unknown` with type guards if needed
- Use `interface` for object types, `type` for unions or utility types
- All exported functions must annotate their return type

### Vue components

- Use `<script setup lang="ts">` consistently
- Props use `defineProps<T>()` with TypeScript types
- Emits use `defineEmits<T>()` with TypeScript types
- Extract shared logic into composables (`composables/useXxx.ts`)
- Components over 200 lines should be split into sub-components

### Error handling

- Wrap API calls in try-catch
- Use a unified `AppError` class for errors
- Catch global errors with `app.config.errorHandler`

## Response conventions

- Always respond in Traditional Chinese
- Commit messages use Conventional Commits, descriptions in Traditional Chinese
- Variable names and code comments in English

## Common tasks

### Create a new page

1. Create the corresponding `.vue` file under `src/views/`
2. Add the route in `src/router/`
3. If shared logic exists, create a composable under `src/composables/`
4. Create the matching test file

### Add a new API integration

1. Create the service file under `src/services/`
2. Define request and response TypeScript types under `src/types/`
3. Use `apiClient` — don't call fetch or axios directly
4. Create the matching test file

Maintaining your rules file

Once it's written, the other important part is keeping it up to date.

As the project evolves, the tech stack might shift and conventions might change. If the rules file doesn't keep up, the AI will keep working off the old rules — and that creates more problems than it solves.

A few things I'd suggest:

  1. Review regularly: When you do a major version bump or change a key technical decision, update the rules file
  2. Pair with code review: If the AI's output isn't matching your expectations, first check whether the rules file is unclear
  3. Treat it as team-owned: For team projects, the rules file should live in version control so everyone can read and propose changes
  4. Use /insights: If you're on Claude Code, run /insights periodically to analyse usage patterns and find places to improve (see 優化 Claude Code 沒方向？超簡單指令給你方向！ (article in Mandarin))

Why I don't recommend /init

One last thing — Claude Code has an /init command that auto-generates a CLAUDE.md, but personally I don't really recommend it.

The reason is simple: the output is generic. It just scans your project structure and produces something that looks complete but is mostly hollow. The rules are general descriptions that don't reflect your team's habits or preferences, so you'll still spend time heavily rewriting it — sometimes scrapping and rewriting from scratch.

Better to write it yourself from the start. Begin with the few rules that matter most, and grow it as the project grows. That way the rules file actually fits your project, and the AI can genuinely follow what you mean.

Wrap-up

Whichever AI dev tool you use, a well-written rules file is the key to actually integrating AI into your workflow.

So to recap the main points:

  1. Concrete and actionable: Give the AI specific rules, not abstract descriptions
  2. Positive phrasing + examples: Tell the AI what to do, paired with code samples
  3. Structured: Clear headings and categories so the AI can find what it needs
  4. Separate concerns: Split into multiple files so no single one gets bloated
  5. Path scoping: Set rules that only apply to specific paths
  6. Explain the why: Add reasoning for the key decisions
  7. Task playbooks: Include step-by-step flows for common tasks
  8. Language and style: Set response language and commit format explicitly

Hope this helps you write a better rules file — and gets your AI a little closer to understanding your project, right?
