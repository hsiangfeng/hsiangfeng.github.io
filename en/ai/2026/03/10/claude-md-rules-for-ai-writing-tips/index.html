CLAUDE.md and Rules for AI: Writing Tips That Make AI Understand Your Project

#AI #Claude Code #GitHub Copilot #AI 開發 #CLAUDE.md #Rules for AI

Introduction

If you’re already using AI dev tools like Claude Code or GitHub Copilot, you’ve probably heard of “Rules for AI” — basically a document that tells the AI how to do things in your project.

But a lot of people write one and find that the AI still doesn’t really listen. Either it ignores the rules, or the code it produces still misses the mark. Most of the time, the real problem is that the rules file itself isn’t written well.

So this post is about how to write that file properly, so the AI actually understands what your project needs. I’ll mostly use Claude Code’s CLAUDE.md as the example, but the ideas apply to any AI dev tool you’re using.

Note
If you don’t know what CLAUDE.md is yet, check out my earlier post 想用 Claude Code 開發？這篇帶你從入門到進階 (article in Mandarin) for a full walkthrough.

Rules files across AI tools

Before we dig in, a quick look at what each tool calls its rules file and where it lives. The names differ but the core idea is the same.

AI tool Rules file Location
Claude Code CLAUDE.md Project root
Cursor .cursor/rules/*.mdc .cursor/rules/
GitHub Copilot .github/copilot-instructions.md .github/

Format and placement vary, but the goal is the same: tell the AI about your project’s conventions and preferences.

So the writing tips below apply to whichever tool you use — only the syntax differs slightly.

Common ways people get this wrong

Before getting into the right way, let’s look at a few common bad patterns that leave the AI with no real direction.

Too vague

This is basically writing nothing, right? “Good code”, “best practices”, “be mindful of performance” — all way too abstract. There’s nothing concrete here for the AI to act on.

Too long-winded

The AI doesn’t need your project’s life story. It just needs to know what to do right now. All that backstory burns tokens and dilutes the parts that actually matter.

Contradictory rules

These two contradict each other. The AI has no way to pick one consistently, so the result is random — and your code style ends up inconsistent.

Tip 4: Separate concerns

Once your rules grow, don’t cram everything into one file. Splitting it into multiple files is much easier to maintain. Personally I try to keep each file under 100-200 lines — beyond that, the AI starts skipping over things.

For Claude Code there are two ways to split:

Option 1: Markdown link references

Claude Code’s CLAUDE.md officially supports @path/to/file syntax to import other files, and imported files get loaded into context at startup. Personally I use plain ./ Markdown links more often, because then on GitHub readers can click through to the linked doc — handy for teammates who aren’t using Claude Code but still want to read the conventions.

Option 2: The .claude/rules/ directory

Another way is to put rules under .claude/rules/, with each .md file covering one topic:

Rule files without a paths setting get loaded entirely at startup, while ones with paths only load when Claude is working on matching files (more on this in the next tip).

Both approaches share two upsides:

  1. Save tokens: Avoid one giant monolithic file
  2. Easier to maintain: Each doc owns one topic, so edits don’t ripple into unrelated rules

Tip 5: Scope rules by path

Some rules don’t apply to the whole project — only to certain files or folders. That’s where path-scoped rules come in.

For Claude Code’s .claude/rules/*.md, you can use the paths field in the YAML frontmatter to scope which files the rule applies to:

paths supports glob patterns. Common ones:

Pattern Matches
**/*.ts All TypeScript files
src/**/* Everything under src/
src/**/*.{ts,vue} .ts and .vue files under src/
tests/**/*.test.ts Test files

Tip 6: Spell out the why behind decisions

For a few key rules, beyond telling the AI what to do, briefly say why. That way, when it hits a similar but not-explicitly-covered situation, it can still make a call that matches your intent.

That said — you don’t need a “why” for every rule. Only add one when it’s a rule the AI might second-guess or interpret differently. Otherwise the doc just bloats.

Tip 7: Add step-by-step guidance for common tasks

This one’s lesser-known: you can put SOPs for common tasks directly in the rules file. So when someone says “create a new page”, the AI just follows the flow — no re-explaining each time.

This is especially good for team development. Encode the patterns once, and the AI can follow the same flow every time without you having to explain it verbally.

Tip 8: Set the response language and style

Last tip — if you want the AI to respond in a specific language or style, write that down too.

Looks simple, right? But if you don’t write it down, the AI will randomly switch between English and Simplified Chinese, and your project’s language ends up inconsistent.

Maintaining your rules file

Once it’s written, the other important part is keeping it up to date.

As the project evolves, the tech stack might shift and conventions might change. If the rules file doesn’t keep up, the AI will keep working off the old rules — and that creates more problems than it solves.

A few things I’d suggest:

  1. Review regularly: When you do a major version bump or change a key technical decision, update the rules file
  2. Pair with code review: If the AI’s output isn’t matching your expectations, first check whether the rules file is unclear
  3. Treat it as team-owned: For team projects, the rules file should live in version control so everyone can read and propose changes
  4. Use /insights: If you’re on Claude Code, run /insights periodically to analyse usage patterns and find places to improve (see 優化 Claude Code 沒方向？超簡單指令給你方向！ (article in Mandarin))

Why I don’t recommend /init

One last thing — Claude Code has an /init command that auto-generates a CLAUDE.md, but personally I don’t really recommend it.

The reason is simple: the output is generic. It just scans your project structure and produces something that looks complete but is mostly hollow. The rules are general descriptions that don’t reflect your team’s habits or preferences, so you’ll still spend time heavily rewriting it — sometimes scrapping and rewriting from scratch.

Better to write it yourself from the start. Begin with the few rules that matter most, and grow it as the project grows. That way the rules file actually fits your project, and the AI can genuinely follow what you mean.

Wrap-up

Whichever AI dev tool you use, a well-written rules file is the key to actually integrating AI into your workflow.

So to recap the main points:

  1. Concrete and actionable: Give the AI specific rules, not abstract descriptions
  2. Positive phrasing + examples: Tell the AI what to do, paired with code samples
  3. Structured: Clear headings and categories so the AI can find what it needs
  4. Separate concerns: Split into multiple files so no single one gets bloated
  5. Path scoping: Set rules that only apply to specific paths
  6. Explain the why: Add reasoning for the key decisions
  7. Task playbooks: Include step-by-step flows for common tasks
  8. Language and style: Set response language and commit format explicitly

Hope this helps you write a better rules file — and gets your AI a little closer to understanding your project, right?
