Want to Develop With Claude Code? This Guide Takes You From Beginner to Advanced

Intro

As a developer, you've probably heard of AI Agent dev tools like Gemini, Codex, Claude Code, and so on. Most developers these days subscribe to a few AI Agent tools to help speed up dev workflows. This post picks one AI Agent tool — Claude Code — and walks through how to use it for development. There may be acceleration techniques and usage concepts here you don't know yet, and this post includes practical examples so you can get up to speed faster.

What is Claude Code?

As always, let's first understand what Claude Code is. Claude Code is an AI Agent tool developed by Anthropic, primarily based on Large Language Models (LLM) to help developers with code generation, debugging, and code optimization. The design goal of Claude Code is to let developers more efficiently complete various programming tasks — whether writing new code or maintaining existing code, it provides solid support.

Anthropic has many products, for example:

  • Claude.ai: basically another ChatGPT — primarily for conversational AI interaction.
  • Claude Code: AI Agent tool designed for developers — helps with code-related tasks. The star of this post. Beyond the terminal, there are also Desktop App, Web App (claude.ai/code), VS Code, and JetBrains extensions.
  • Claude Cowork: in short, you can use Cowork to manage your computer — let it organize your desktop, write documents, handle daily tasks.
  • Claude Design: a visual design tool released in April 2026 — can build visual mockups, prototypes, presentations, and pass designs to Claude Code for implementation.

There are also some well-known features that aren't products per se but more like features or protocols:

  • Claude in Chrome: a browser extension that lets users use Claude features directly in the browser.
  • Claude in Slack: lets users interact with Claude directly in Slack workspaces to help with various work tasks.
  • Claude in Excel: yes, this lets Excel users use Claude for data analysis, report generation, and so on.
  • MCP: MCP is a standard proposed by Anthropic — basically a standardized protocol that lets legacy systems (or systems that can't interact with AI) interact with other AI systems through MCP.
  • Skills: you could say this was designed to solve MCP causing Context bloat — lets developers build dedicated skills for specific tasks, with the AI calling these skills based on task needs. This helps reduce Context usage, since Tokens are expensive.

After getting acquainted with Anthropic, let's understand how to use Claude Code for development — Claude Code is the star of this post.

Installing Claude Code

There are quite a few install methods. The currently recommended one is the Native Installer — no need to install Node.js separately, and supports auto-update:

macOS/Linux/WSL (recommended):

bash
curl -fsSL https://claude.ai/install.sh | bash

Windows PowerShell (recommended):

powershell
irm https://claude.ai/install.ps1 | iex

Windows CMD:

cmd
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd

Homebrew:

Homebrew offers two casks, and the difference is which update channel they track:

bash
# Stable channel: skips releases with major regressions, but usually about a week behind the latest
brew install --cask claude-code

# Latest channel: gets new versions the moment they ship
brew install --cask claude-code@latest

WinGet:

powershell
winget install Anthropic.ClaudeCode

npm (requires Node.js 18+):

bash
npm install -g @anthropic-ai/claude-code

Note Whichever cask you pick, Homebrew won't auto-update — you'll need to run brew upgrade claude-code or brew upgrade claude-code@latest yourself (depending on which one you installed) to update.

So which one should you pick? claude-code (stable) skips releases with major regressions, at the cost of getting new features about a week late. claude-code@latest stays on the bleeding edge. The risk with the latter is you might hit a fresh bug that just shipped and hasn't been reported yet — basically you're the guinea pig for Anthropic. Personally I switched to @latest because I want to play with new stuff like the Workflows feature below the moment it lands, but you've got to accept the occasional landmine. If you'd rather stay safe, just use the claude-code stable cask.

Pick the install method that fits your needs. Once installed, you can start using Claude Code for development.

Then just cd into your project and type claude to launch the Claude Code interactive interface:

bash
cd your-project
claude

If it's your first time, you'll be asked to log in. If not, you can also type /login to log in:

bash
/login

When logging in, you have two account options:

  • Claude.ai account: subscription plans — Claude Pro, Claude Max, etc.
  • Claude Console: basically API-based — relatively pricier since it's billed by usage, similar to OpenAI's API.

What's the difference between the two?

  • Subscription: gives you a usage cap. Current plans:
    • Pro: $20/month, suits regular developers.
    • Max 5x: $100/month, 5x the usage of Pro — for heavy users.
    • Max 20x: $200/month, 20x the usage of Pro — for developers needing extended heavy use.
    • When you exceed the cap, you can't use it temporarily — wait for the quota to refresh or upgrade your plan.
  • API usage billing: billed by actual Tokens used. Since there's no cap, you can use Claude Code without limit. Downside: the price is relatively higher. Usually for enterprises or developers who need heavy usage.

Note Claude Code requires Pro or higher — the free Claude.ai account can't use Claude Code.

In short, this is asking you to confirm the project's source — to avoid security risks.

Some basic commands you must know — there's a high chance you'll use them:

  • /help: shows help info and available commands.
  • /init: ask Claude Code to initialize this project. Note: "initialize" here doesn't mean making changes to the project — it means letting Claude Code understand the project's structure, language, framework, etc., and produce a "project description document (CLAUDE.md)" so you can pick up the project faster. We call this a "memory file".
  • /login: log into a Claude account.
  • /logout: log out. If you have multiple accounts, you can use this command to switch — log out, then log into another.
  • /usage: check current account usage — remaining Tokens, usage count, etc.
  • /clear: clear the current conversation history and start a new one.
  • /compact: clear conversation history, but compress it first so the conversation content is more concise — avoids Context bloat.
  • /resume: bring up previous conversation history so you can continue.
  • /memory: edit or view the current memory file (CLAUDE.md). Has project memory and user memory (covered later).
  • /mcp: manage MCP config files.
  • /skills: see available Skills.
  • /model: view or switch the current model. Most cases use the default — only switch for special needs.
  • @: mention a file so Claude Code can act on it, e.g., @app.js.

These are the commands and tricks I personally use most often in development.

Memory file (CLAUDE.md)

Next let's get familiar with the "memory file (CLAUDE.md)" produced after running /init. This file is really important — it determines how Claude Code performs during development, because it tells Claude Code about the project's structure, language, framework, and even which docs to reference during dev.

The CLAUDE.md file has multiple types — each location has a different purpose:

  • Personal memory (~/.claude/CLAUDE.md): your personal preferences — always use Traditional Chinese, prefer some programming language or framework, particular dev style, etc. All projects on your computer share this personal memory.
  • Project memory (<project-root>/CLAUDE.md or <project-root>/.claude/CLAUDE.md): rules document for this project — the company's internal coding standards, dev process, team conventions, etc.
  • Project rules (<project-root>/rules/*.md): things like specific language guides, framework usage standards, particular dev rules — making Claude Code's dev work better fit project needs.
  • Private project memory (<project-root>/CLAUDE.local.md): special dev preferences for this file that you don't want to share with other devs — usually personal preferences, so it's added to .gitignore to avoid getting pushed to remote.

Why do we need these files? The purpose is to let Claude Code, after initialization (typing claude in the project), understand the project's structure, language, framework, and which docs to reference during dev.

How do we know CLAUDE.md is being correctly read? When you type claude, if you see a prompt asking you to type /init, that means Claude Code didn't read a CLAUDE.md file. So you need to type /init to have Claude Code initialize for this project.

CLAUDE.md file present

Note From my testing, I'd recommend placing it at <project-root>/CLAUDE.md rather than <project-root>/.claude/CLAUDE.md — sometimes Claude Code can't correctly read CLAUDE.md from the .claude folder. Unless you use /memory to edit the location.

CLAUDE.md file example

CLAUDE.md, in plain English, is a project README.md. When a new person joins, you wouldn't make them re-learn the project from scratch every time — you need a doc to help newcomers quickly understand the project's structure, language, framework, etc., so they can pick it up faster.

A standard project memory CLAUDE.md should contain at minimum:

  • Project name
  • Brief project description
  • Important project documents
    • Project structure (PROJECT-STRUCTURE.md)
    • Project commands (PROJECT-COMMANDS.md)
    • Dev process (DEVELOP-FLOW.md)
    • Dev standards (CODE-STYLE.md)
    • Commit standards (GIT-COMMIT.md)
    • Dev style (GUIDELINE.md)
    • Dev tasks (TASK.md)
    • Test process (TEST-FLOW.md)
    • Deploy process (DEPLOY-FLOW.md)
    • FAQ (FAQ.md)

Note Has AI made engineers' lives easier? I don't think so — instead, we have to spend more time maintaining all these docs and constantly reviewing AI-produced code quality.

Here's a simple example:

md
# Project name

This is a...

## Important project documents

- [Project structure](/docs/PROJECT-STRUCTURE.md)
- [Project commands](/docs/PROJECT-COMMANDS.md)
- [Dev process](/docs/DEVELOP-FLOW.md)
- [Dev standards](/docs/CODE-STYLE.md)
- [Commit standards](/docs/GIT-COMMIT.md)
- [Dev style](/docs/GUIDELINE.md)
- [Dev tasks](/docs/TASK.md)
- [Test process](/docs/TEST-FLOW.md)
- [Deploy process](/docs/DEPLOY-FLOW.md)
- [FAQ](/docs/FAQ.md)

You'll notice this is different from what /init generates. To avoid CLAUDE.md becoming overly long, I'd suggest splitting content into multiple files and referencing them from CLAUDE.md — making Claude Code more efficient at finding docs.

For each referenced doc, you can write so-called YAML frontmatter to help Claude Code understand the doc's purpose:

md
---
description: This is the project structure document for this project. When the user asks for a project structure explanation or wants to do project structure-related tasks, please reference this document's contents.
---
# Project structure

...

For files under <project-root>/rules/*.md, you can add a path field to specify which file paths this doc applies to — avoiding Claude Code referencing this doc in the wrong place:

md
---
path: src/api/**/*.js
---

The path field supports formats like **/*.js, src/**/*, src/api/**, etc.

Adding MCP Server

MCP stands for "Model Context Protocol" — a standardized protocol that lets systems unable to directly interact with AI to interact with other AI systems via MCP.

Since I've covered this in detail in earlier articles, I'd recommend reading these directly:

  • Heard of LLM, RAG, Prompt? A quick guide to common Generative AI terms
  • GitHub Copilot + MCP Server in VSCode: Build Your AI Dev Workflow
  • Build Your Own MCP Server for AI Agents — A Complete Implementation Guide With GitHub Copilot Chat

I'd particularly recommend reading the Heard of LLM, RAG, Prompt? post — it has good intro coverage of various AI terms.

The MCP Server I'd strongly recommend using during development is Context 7. The advantage of this open source MCP Server is that it lets your AI fetch real-time data. Why is that needed? AI is trained on data — when its training data is outdated, the results aren't accurate. So you need an MCP Server that lets AI fetch real-time data, improving the accuracy of AI's output.

Context specifically collects the latest official dev docs and real, usable code samples — basically a standard config for AI Agents.

How do you add it? The easiest way right now is the official one-click install command:

bash
npx ctx7 setup --claude

This handles OAuth verification, generates an API Key, and installs the corresponding MCP Server config automatically.

If you prefer manual setup, you can first apply for a free API Key at Context 7 (no credit card needed), then choose one of two ways to add the MCP Server:

  • Local mode: add MCP Server through Claude Code commands:
    bash
    claude mcp add --scope user context7 -- npx -y @upstash/context7-mcp --api-key YOUR_API_KEY
  • Remote mode: configure the MCP Server through the Context 7 official site, then Claude Code will auto-fetch from it:
    bash
    claude mcp add --scope user --header "CONTEXT7_API_KEY: YOUR_API_KEY" --transport http context7 https://mcp.context7.com/mcp

Replace YOUR_API_KEY with the key you obtained.

Note You can use Context7 without an API Key, but you'll have a lower rate limit (free accounts get about 1,000 requests/month). I'd recommend applying for a free API Key for higher usage.

What's the difference between remote and local? In short, in local mode the backend code runs on your computer and fetches from the remote DB. In remote mode, even the backend code runs remotely — the upside is you don't need to install anything locally.

Which is recommended? Both are fine — pick what fits your usage habits.

Custom Commands

Beyond the above, you can also customize Claude Commands to better fit your needs. I'd recommend two I personally find quite useful:

  • /test: for running tests. Have Claude Code write test cases or fix failing tests — significantly boosts your testing efficiency.
  • /review: for code review. Have Claude Code check code quality, style, or find potential issues — ensures your code quality.

How to create these? Easy — just create a .claude/commands folder in the project root and put two Markdown files inside, named test.md and review.md.

bash
mkdir -p .claude/commands
touch .claude/commands/test.md
touch .claude/commands/review.md

The content can be customized, but here are reference examples:

test.md:

md
---
description: Run automated testing flow — run existing tests, fix failures, or add tests for critical logic
allowed-tools: Bash(npm run *, npm test), Filesystem(*)
metadata:
  author: Ray
  version: "2026.1.30"
---
# Code testing command

Please run the following testing and quality assurance flow. Your goal is to ensure code correctness and robustness.

## Steps
1. **Command analysis**: check the `package.json` script definitions below to confirm the correct test command (e.g., `test` or `test:coverage`).
2. **Run tests**: use the Bash tool to execute the test command.
3. **Fix errors (if tests fail)**:
   - Analyze the stack trace.
   - **Don't simply modify the test to make it pass** unless you're sure the test case itself is wrong.
   - Fix the original code logic to pass the test.
4. **Add tests (if missing)**:
   - If the project lacks tests or coverage is insufficient, write unit tests for core business logic.
   - Ensure tests are **atomic** — don't depend on external environment (use Mock/Stub).

## Reference info (Context)
To help you pick the right command, here are the project's script settings:

`package.json` Scripts:
!`grep "scripts" -A 15 package.json`

---
description: Review code changes in the current working directory (including Staged and Unstaged)

Then just type /test or /review to launch these Commands for testing or code review.

test

In these Markdown files, you'll see a few special YAML properties:

  • description: lets users know what this Command is for — shown in Claude Code's command list.
  • allowed-tools: limits which tools this Command can use — avoids the Command using unnecessary tools and creating security issues.
  • metadata: records additional info like author, version, etc. — helps you better manage these Commands. Claude Code automatically ignores this and won't use it.

There's also a magical block — !. This block lets you execute Bash commands directly in the Command and insert results into the Command document — making the Command more dynamic and able to operate based on project state.

Claude Hooks

Yes — Claude Code also supports Hooks, letting you trigger custom behavior on specific events. This makes Claude Code more flexible and able to operate based on your needs.

For example, before each code commit, you can have Claude Code check code quality, raising your code quality.

The Hook events Claude Code supports (as of April 2026, 28 events total):

Session lifecycle:

  • SessionStart: triggered when a Claude Code session starts.
  • SessionEnd: triggered when a Claude Code session ends.
  • Stop: triggered when a Claude Code turn ends.
  • StopFailure: triggered when a turn ends due to API error.

User interaction:

  • UserPromptSubmit: triggered when you submit a Prompt, before Claude Code processes it.
  • UserPromptExpansion: triggered when a Slash Command expands into a Prompt.
  • Notification: triggered when Claude Code emits a notification.

Tool-related:

  • PreToolUse: triggered before a tool is used.
  • PostToolUse: triggered after a tool call succeeds.
  • PostToolUseFailure: triggered after a tool call fails.
  • PermissionRequest: triggered when Claude Code requests permission.
  • PermissionDenied: triggered when Auto Mode's classifier denies an operation.

Subagent and tasks:

  • SubagentStart: triggered when a subagent starts.
  • SubagentStop: triggered when a subagent ends.
  • TeammateIdle: triggered when an Agent Team teammate is about to go idle.
  • TaskCreated: triggered when a task is created.
  • TaskCompleted: triggered when a task is marked complete.

Context and configuration:

  • PreCompact: triggered before compressing conversation history.
  • PostCompact: triggered after compressing conversation history.
  • InstructionsLoaded: triggered when CLAUDE.md or rules files are loaded.
  • ConfigChange: triggered when config files change during the session.

Files and environment:

  • CwdChanged: triggered when the working directory changes.
  • FileChanged: triggered when monitored files change on disk.
  • WorktreeCreate: triggered when a Worktree is created.
  • WorktreeRemove: triggered when a Worktree is removed.

MCP-related:

  • Elicitation: triggered when an MCP Server requests user input during a tool call.
  • ElicitationResult: triggered after the user responds to MCP Elicitation.

Why introduce this? Few people use it, but Hooks are really useful in certain situations:

  • After completing a specific task, run tslint or prettier to format the code — perfect for the PostToolUse Hook.
  • After Claude Code finishes a task, notify relevant people via Slack, Discord, etc. — use the Notification Hook.
  • When the user submits a Prompt, do some preprocessing like filtering sensitive words — use the UserPromptSubmit Hook.
  • When Claude Code wants to run rm -rf, do a double confirmation to avoid accidental deletion — use the PermissionRequest Hook.
  • When Claude Code wants to read sensitive info like .env, AWS credentials, etc., encrypt first — use the PreToolUse Hook.
  • Before creating or editing files, ask Claude Code to do code review for code quality — use the PostToolUse Hook.

What's the format? This involves placement again:

  • Personal Hooks (~/.claude/settings.json): personal Hook config — affects Hook behavior across all projects on your computer.
  • Project Hooks (<project-root>/.claude/settings.json): Hook config for this project — affects Hook behavior for this project, can be uploaded to GitHub for shared use with other devs.
  • Private project Hooks (<project-root>/.claude/setting.local.json): private Hook config for this project — usually personal preferences, so add to .gitignore to avoid pushing to remote.

Content format:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "your-script.sh"
          }
        ]
      }
    ]
  }
}

matcher filters which tools to trigger on. For example, Edit|Write triggers when Claude Code edits or writes files. command is the command to execute.

Hooks receive JSON data via stdin — you can parse it with jq. Example:

json
{
  "session_id": "abc123",
  "transcript_path": "/path/to/transcript.json",
  "cwd": "/path/to/project",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/path/to/file.js",
    "old_string": "const x = 1",
    "new_string": "const x = 2"
  }
}

Common Hook examples for development

Now let me introduce some Hook examples commonly used in development, so you can pick up Claude Code's Hook feature faster.

Auto-format (Prettier)

After Claude Code edits a file, auto-run Prettier to format the code:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

Auto ESLint fix

After editing files, auto-run ESLint fix:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx eslint --fix"
          }
        ]
      }
    ]
  }
}

Auto Git Commit before edit (create backup point)

Before Claude Code modifies a file, create a backup commit so you can roll back later:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "git add -A && git commit -m 'backup: before Claude edit' --allow-empty || true"
          }
        ]
      }
    ]
  }
}

Auto commit after task completion

When Claude Code stops, auto-commit all changes:

json
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "git add -A && git commit -m 'feat: Claude Code auto-commit' || true"
          }
        ]
      }
    ]
  }
}

Block dangerous operations

Intercept dangerous commands like rm -rf to prevent accidental deletion:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "~/.claude/hooks/block-dangerous.sh"
          }
        ]
      }
    ]
  }
}

~/.claude/hooks/block-dangerous.sh:

bash
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# Intercept dangerous commands
if [[ "$COMMAND" == *"rm -rf"* ]] || [[ "$COMMAND" == *"rm -r /"* ]]; then
  echo "Dangerous command blocked: $COMMAND" >&2
  exit 2  # exit 2 = block operation
fi

exit 0

Writing Hooks with JavaScript

If you're not familiar with Bash, you can also write Hooks in JavaScript:

~/.claude/hooks/format-lint.js:

javascript
#!/usr/bin/env node
const { execSync } = require("child_process");

async function main() {
  // Read stdin
  const chunks = [];
  for await (const chunk of process.stdin) {
    chunks.push(chunk);
  }
  const input = JSON.parse(Buffer.concat(chunks).toString());
  const filePath = input.tool_input?.file_path;

  // Only handle JS/TS files
  if (!filePath?.match(/\.(js|jsx|ts|tsx)$/)) {
    process.exit(0);
  }

  try {
    // Prettier format
    execSync(`npx prettier --write "${filePath}"`, { stdio: "inherit" });

    // ESLint fix
    execSync(`npx eslint --fix "${filePath}"`, { stdio: "inherit" });
  } catch (error) {
    // Ignore errors, continue execution
  }
}

main();

Remember to make the script executable:

bash
chmod +x ~/.claude/hooks/format-lint.js

Do Hooks consume a lot of Tokens?

This depends on the Hook type, since Hooks have three execution methods:

Hook typeToken consumptionDescription
type: "command"NonePure local script execution — no AI involvement needed
type: "http"NoneSends HTTP POST to specified URL — good for Webhook notifications
type: "prompt"YesUses LLM for decision-making
type: "agent"YesUses AI agent for multi-tool verification

If you just want simple automation flows like auto-format or auto-commit, command type is sufficient — no Token consumption.

But if you need AI to judge "does this code conform to standards?" or "is this operation safe?", you need prompt or agent type — that consumes Tokens.

json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Check whether all requested tasks were completed in the conversation. If there's unfinished work, respond {\"ok\": false, \"reason\": \"Still need to complete: ...\"}."
          }
        ]
      }
    ]
  }
}

So bottom line: most automation scenarios use command type and don't consume Tokens. Only when you need AI to judge do you need prompt or agent types.

See? The Hook feature is actually really powerful — like a team having fixed processes during dev. That's why I'm specifically introducing this feature, because it lets you automate these processes and make Claude Code better fit your needs.

Permission modes

I've covered a lot, but haven't introduced one thing yet — Mode.

In Claude Code, Mode defines Claude Code's behavior when executing tasks. Different Modes affect how Claude Code handles commands, uses tools, and interacts with users.

Claude Code currently supports 6 Permission Modes:

  • Default Mode: the default mode. Claude Code asks for confirmation before file edits or shell commands — every operation is under user control.
    • Best for: first-time use, important projects, when you need careful review of every change.
  • Plan Mode: a read-only mode. Claude Code can only explore code and make plans — can't modify files.
    • Best for: analyzing unfamiliar codebases, planning large refactors, understanding scope of impact before action.
  • Auto-accept Edits Mode: auto-accepts file edits but still confirms shell commands.
    • Best for: rapid iteration, lots of file modifications, trust Claude Code's edits but still want to control system commands.
  • Auto Mode (added March 2026): all operations auto-execute, but a background AI safety classifier (Sonnet 4.6) reviews each operation, auto-blocking dangerous behavior. Currently in Research Preview.
    • Best for: fully automated dev flow with safety guardrails.
    • Limitation: requires Max, Team, Enterprise, or API plan (Pro doesn't support).
  • Don't Ask Mode: only allows pre-approved tools to execute — all unauthorized operations are auto-denied.
    • Best for: CI/CD environments, automated scripts — strict tool usage scope control.
  • Bypass Permissions: skips all permission checks (except protected paths) — all operations execute directly.
    • Best for: only in isolated containers or VMs — strongly not recommended on local dev environments.

You can use Shift+Tab to switch between Default → Auto-accept Edits → Plan — the three commonly used modes. If your account plan supports Auto Mode, it appears in the cycle too. Other modes need to be started via the CLI parameter --permission-mode.

Context management techniques

When using Claude Code, Context management is really important — Context bloat consumes Tokens fast and even affects Claude Code's response quality.

What is Context?

Context is all the info Claude Code references when handling tasks, including:

  • Conversation history: your Prompts, Claude Code's responses
  • Memory files: CLAUDE.md, rules/*.md, etc.
  • Read file contents: code or docs Claude Code read during the conversation
  • Execution results: command output, error messages
  • MCP Server data: if MCP is enabled, related response data also counts
  • Skills content: triggered Skills file contents

These accumulate in memory so Claude Code can understand the conversation's context. Because of this, Context grows continuously as the conversation progresses.

Signs of Context overload

When you see these, Context might be overloaded:

  • Claude Code starts "forgetting" previously discussed content
  • Response quality drops noticeably — repetitive or contradictory content
  • Slower execution
  • Token limit warnings appear

How to manage Context

  1. Use /compact: compresses current conversation history, keeps important info but reduces Token usage. Use periodically after long conversations.
  2. Use /clear: if starting a brand new task, just clear conversation history and start over.
  3. Use /resume smartly: if you need to review previous conversation, use this to load history.
  4. Split large tasks: instead of having Claude Code handle an entire large refactor at once, split into multiple small tasks. After each completes, use /compact or /clear.
  5. Describe needs precisely: avoid vague descriptions — precise Prompts reduce back-and-forth, which reduces Context accumulation.

Monitoring Token usage

You can type /usage anytime to check current Token status, helping you decide whether to do Context management.

Note Note this mode is only available for subscriptions.

Workflows — multi-agent orchestration (ultracode)

This is probably the most notable update of late May 2026 — "Workflows" (dynamic workflows), shipped alongside Claude Opus 4.8.

Before this, if you wanted Claude Code to fire off a bunch of subagents at once, it had to decide the next step turn by turn on its own, and the intermediate results from all those agents got stuffed into the main conversation's Context. Once the task got big, things blew up easily.

Workflows works differently: it writes the orchestration logic — how many agents to spawn, who goes first, who verifies whom — into a script and runs it in the background. A few key points:

  • Runs in the background: the workflow runs in an isolated environment, so it doesn't block your current conversation — you can keep doing other things.
  • Massively parallel: it can run multiple subagents at once (roughly 16 at a time), and a single run can spin up dozens or even hundreds of agents over its lifetime.
  • Doesn't eat your main Context: intermediate results live in script variables instead of getting dumped back into the main conversation like regular subagents do.
  • Cross-verification: a common pattern is having several agents independently do the same thing, then verify each other's results — which helps cut down on AI hallucination.

So how do you kick one off? Because this thing can spin up dozens of agents and burn a lot of Token, it won't run on its own by default — you have to explicitly ask for it:

  • Add the ultracode keyword to your Prompt, for example:
    txt
    ultracode: scan every API endpoint under src/routes/ and check for missing permission validation
  • Or just say it in plain language — "use a workflow to..." or "fan out multiple agents to..." — and Claude Code will decide whether to orchestrate one for you.
  • You can also use the built-in /deep-research <your question>, which is itself a workflow that fans out a bunch of web searches and then cross-checks the sources.

Note Note that ultracode and the ultrathink from earlier aren't the same thing. ultrathink gives Claude more thinking budget for this one turn; ultracode asks it to handle the whole task with a multi-agent workflow. Totally different purposes — don't mix them up.

Once it's running, you can use the /workflows command to watch the show — it lists running and completed workflows, showing each phase, how many agents got spawned, how much Token was spent, and how long it ran. You can also drill into a single agent's details, or just pause or stop the whole thing.

So when should you reach for a workflow? The official guidance is roughly these "too big to fit in one Context" scenarios:

  • Auditing an entire codebase: e.g. scanning the whole project for a certain class of bug or security hole.
  • Large migrations: like a framework or package upgrade across hundreds of files.
  • Research or planning that needs multiple angles: investigating the same question from several directions, letting them poke holes in each other, then converging on a conclusion.

On the flip side, if it's just a few small tasks, a regular subagent is fine — no need to go through the whole workflow song and dance, since it really does burn Token.

Note This feature requires a paid plan, and Pro users may need to turn on "Dynamic workflows" manually in /config first.

Common questions

Some issues I've personally hit while using Claude Code, with solutions for reference.

Can't switch Node.js versions with nvm after installing Claude Code

This was a particular situation when I installed Claude Code. I tried to recreate it but couldn't. To make sure other readers don't hit the same problem, I'm writing it up.

I installed via Homebrew, then suddenly couldn't switch Node.js versions with nvm. Investigation showed a global Node.js had been installed and overrode everything, breaking nvm.

The fix is simple. First, remove the Homebrew-installed Claude Code:

bash
brew uninstall claude-code

Then remove the bundled Node.js:

bash
brew uninstall --ignore-dependencies node

Note--ignore-dependencies ignores dependencies, ensuring only Node.js gets removed without affecting other packages.

By this point, your nvm should work normally.

If you hit this issue from installing the curl version of Claude Code, you can try:

bash
rm ~/.local/bin/claude

"installMethod is native, but claude command not found at /Users/users/.local/bin/claude"

This is because Claude Code calls and uses the executable at /Users/users/.local/bin/claude, but sometimes due to install method or system environment, that path's executable isn't found.

First, check where Claude Code is currently installed:

bash
which -a claude

For me, since I installed via Homebrew, it shows:

bash
/opt/homebrew/bin/claude

That path is correct, but the "installMethod is native, but claude command not found at /Users/users/.local/bin/claude" error still shows. This means Claude Code's initialized path got stuck. Just run:

bash
rm -rf ~/.claude

Remove the .claude folder, then restart Claude Code and re-run initialization and login — the error should be gone.
