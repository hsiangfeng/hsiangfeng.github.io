Agent Skills as a replacement for MCP? Demoed with Claude Code

Intro

If you've used Claude Code, Gemini, or Codex, you've almost certainly used MCP (Model Context Protocol). The problem is these tools have a hard Context limit, and MCP burns through it fast — once you start using MCP, blowing past the Context limit becomes really easy, and the model stops working properly. So this post walks through Anthropic's Agent Skills — also from Anthropic — and demos how to use them with Claude Code.

What are Agent Skills?

Before going deep on Agent Skills, we need to look at how MCP actually works — otherwise the "why use Agent Skills" question doesn't really land.

For a long time, AI models had two big problems: outdated information, and no visibility into private data. MCP showed up as a standardized protocol that lets models talk directly to external tools (databases, local files, search engines) and pull fresh data on demand.

The catch is that MCP is fundamentally about "data shipping". When you call an MCP tool, it usually dumps the raw data it pulled straight into the conversation log. That creates two main pain points:

  • A heavy Token tax: large logs and JSON blobs eat through model Tokens fast.
  • Noise: the data is so big that your original Prompt gets buried, the model loses focus, and answer quality drops.

That's why MCP solves the stale-data problem but doesn't really improve a model's "comprehension" or "reasoning" — there's just too much noise.

To address this, Anthropic released Agent Skills.

The design philosophy of Agent Skills is "knowledge shipping", not "data shipping".

Note

Agent Skills aren't really a replacement for MCP — they're built on top of it, and the two work together. Plenty of systems still rely on MCP to feed data to the AI.

So what's the difference between knowledge shipping and data shipping? Simple — give the AI only the key points, not everything. The upside is fewer wasted Tokens and less noise, so the model can stay focused on what matters.

Demoing Agent Skills with Claude Code

Both Gemini and Claude Code support Agent Skills, but I'll focus on Claude Code here.

First thing to know: Skills aren't a new technology or a new format — they're just plain Markdown files. The catch is there are some authoring conventions and placement rules.

Where to put them

Claude Code supports two locations:

  • Project-level: <project root>/.claude/skills/
  • Global: ~/.claude/skills/ (macOS/Linux)

Heads up: when a Skill with the same name exists in both locations, the priority order is Global > Project. So a global Skill overrides a project-level Skill of the same name.

Note

The full priority order also includes Managed and Plugin: Managed > Personal > Project > Plugin. But regular developers rarely touch Managed or Plugin, so I'll skip them here.

Monorepo nested directories

If your project is a Monorepo, Claude Code will search nested directories for Skills.

For example, when you're editing packages/frontend/src/App.vue, Claude Code searches the root .claude/skills/ and automatically searches packages/frontend/.claude/skills/ too.

Text
····················································································monorepo-project/
├── .claude/
│   └── skills/
│       └── shared-skill/        # shared across the whole project
│           └── SKILL.md
└── packages/
    ├── frontend/
    │   ├── .claude/
    │   │   └── skills/
    │   │       └── react-skill/ # frontend only
    │   │           └── SKILL.md
    │   └── src/
    │       └── App.vue
    └── backend/
        ├── .claude/
        │   └── skills/
        │       └── api-skill/   # backend only
        │           └── SKILL.md
        └── src/
            └── index.ts

This is super practical for Monorepos — each sub-project gets its own Skills without stepping on each other.

Either way, Skills live under .claude/skills/, with a few rules:

  • Each Skill is its own folder (e.g. .claude/skills/custom-name-skill/)
  • Each Skill folder must contain at least one SKILL.md file (uppercase, fixed filename)

So the standard structure looks like:

Text
························project-root/
└── .claude/
    └── skills/
        └── custom-name-skill/
            └── SKILL.md

You can drop other files alongside it inside the Skill folder:

Text
················································.claude/
└── skills/
    └──custom-name-skill/
        ├── SKILL.md (required - fixed filename)
        ├── reference.md (detailed API docs - loaded when needed)
        ├── examples.md (usage examples - loaded when needed)
        └── scripts/
            └── helper.py (helper script - run at execution time)

If your structure is more complex you can even split it into multiple subfolders:

Text
····························································································.claude/
└── skills/
    └──custom-name-skill/
        ├── SKILL.md (required - fixed filename)
        ├── reference/
        │   ├── api.md (detailed API docs - loaded when needed)
        │   └── data-model.md (data model docs - loaded when needed)
        ├── examples/
        │   ├── example-1.md (usage example 1 - loaded when needed)
        │   └── example-2.md (usage example 2 - loaded when needed)
        └── scripts/
            ├── helper.py (helper script - run at execution time)
            └── data_fetcher.py (data fetching script - run at execution time)

Whichever structure you pick, the key file is still SKILL.md — Claude Code reads it to understand what the Skill does and when to use it.

Authoring conventions

Now for how to write SKILL.md. The basic info goes into a YAML Front Matter block:

markdown
---
name:
description: 
---

But there are more fields than just those two. Here's the full list:

  • name (required): lowercase letters, numbers, and hyphens only — no spaces or uppercase
  • description (required): the most important field, because Claude Code uses it to decide when to use this Skill
  • allowed-tools (optional): restrict which tools the Skill can use
  • model (optional): specify which model runs this Skill (defaults to whatever model the conversation is using)
  • context (optional): set the Skill's runtime context
  • agent (optional): specify which Agent type the Skill runs under
  • hooks (optional): define lifecycle events for the Skill
  • user-invocable (optional): control whether users can call the Skill directly
  • disable-model-invocation (optional): only allow manual triggering, block AI from auto-calling

I'll walk through everything except name and description below.

Optional field (allowed-tools)

Limits the Skill to a specific set of tools that Claude can use without asking the user for permission.

markdown
······---
name: example-skills
description:  This is an example skill for demonstration.
allowed-tools: 
  - Read
  - Grep
  - Glob
---

Optional field (model)

You can specify which model the Skill runs on, picked from Anthropic's model list.

markdown
---
name: example-skills
description:  This is an example skill for demonstration.
model: claude-haiku-4-5-20251001
---

Optional field (context)

Run this Skill in an isolated sub-agent environment with its own conversation history. After it finishes, only the result returns to the main conversation — the intermediate steps don't pollute the main context.

markdown
---
name: example-skills
description:  This is an example skill for demonstration.
context: fork
---

Optional field (agent)

This field must be used together with context: fork. It specifies which Agent environment the Skill runs in.

Claude Code ships with three built-in Agent types:

Agent typeWhat it's for
ExploreSearching files, understanding architecture, etc.
PlanDesigning solutions, analysing requirements, etc.
general-purposeGeneral-purpose Agent for mixed tasks (default)
markdown
---
name: code-analysis
description: Analyze code quality and generate detailed reports.
context: fork
agent: Explore
---

You can also point to a custom Agent by name. Custom Agents live under .claude/agents/:

Text
············.claude/
├── agents/
│   └── security-reviewer.md  # custom Agent
└── skills/
    └── security-check/
        └── SKILL.md

Then reference it from the Skill:

markdown
---
name: security-check
description: Perform security analysis on the codebase.
context: fork
agent: security-reviewer
---
Note

When you use context: fork, the Skill runs in an isolated sub-agent environment with its own conversation history. After it finishes, only the result is returned to the main conversation — the intermediate process doesn't pollute the main context. This is especially useful for tasks that need a lot of searching or analysis.

Optional field (hooks)

Yes, you can define lifecycle hooks for a Skill — things to do before loading or after running.

markdown
········································---
name: secure-operations
description: Perform operations with additional security checks
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/security-check.sh $TOOL_INPUT"
          once: true
---

The full list of hooks lives in the Hooks documentation.

Optional field (user-invocable)

Controls whether the user can invoke the Skill directly. What does "invoke directly" mean? It means calling it via the / slash command.

markdown
---
name: example-skills
description:  This is an example skill for demonstration.
user-invocable: true
---

Optional field (disable-model-invocation)

This field blocks the AI from auto-calling the Skill. Set it to true and the Skill can only be triggered manually by the user typing /skill-name — Claude won't decide to call it on its own. Useful when you want a Skill to fire only on explicit user request, not when Claude thinks it's a good idea.

markdown
---
name: dangerous-operation
description: Perform a dangerous operation that requires explicit user consent.
user-invocable: true
disable-model-invocation: true
---

To make the difference between user-invocable and disable-model-invocation clearer, here's a comparison table:

Setting/ menuAuto-called by ClaudeUse case
DefaultVisibleAllowedGeneric use
user-invocable: falseHiddenAllowedClaude can use it, but you don't want users calling it manually
disable-model-invocation: trueVisibleBlockedUsers can call it, but you don't want Claude deciding for them

String substitution

Inside SKILL.md, you can use string substitution to dynamically insert info:

VariableDescription
$ARGUMENTSAll arguments passed when the Skill was invoked
${CLAUDE_SESSION_ID}Unique identifier of the current session

This shines for logging or tracking scenarios:

markdown
---
name: session-logger
description: Log activity for the current session.
---

Please log the following content to `logs/${CLAUDE_SESSION_ID}.log`:

$ARGUMENTS

When the user runs /session-logger finished the login feature today, $ARGUMENTS gets replaced with "finished the login feature today" and ${CLAUDE_SESSION_ID} gets replaced with the current Session ID.

A sample Skill

Time for a small example. I'll walk through a Code Review Skill specialised in JavaScript code review. To make it easier to follow, I'm writing the description in English here (in real projects English is generally recommended too).

Creating a Skill is straightforward. Open your project and run:

Bash
mkdir -p .claude/skills/code-review # create the folder
touch .claude/skills/code-review/SKILL.md # create SKILL.md

Then drop the following into .claude/skills/code-review/SKILL.md:

markdown
---
name: code-review
description: Use when the user asks for a JavaScript or TypeScript code review. Triggers on phrases like "review this", "check this code", or "what's wrong with this".
---

# JavaScript Code Review

When reviewing code, follow these steps:

## Review focus areas

1. **Naming**: are variables and functions clearly named, following camelCase?
2. **Error handling**: is there proper try-catch or error boundary handling?
3. **Performance**: any unnecessary loops, redundant rendering, or memory leak risks?
4. **Security**: any potential XSS, injection, or other risks?
5. **Readability**: is the code easy to follow, or does it need refactoring?

## Output format

Reply using the following format:

### Must fix
- [Problem] -> [Suggested fix]

### Suggested improvements
- [Problem] -> [Suggested fix]

### Strengths
- [Things done well]

Folder layout afterwards:

Text
························project-root/
└── .claude/
    └── skills/
        └── code-review/
            └── SKILL.md

Now drop this intentionally bad JavaScript snippet into your project (say, /bad-code.js) and let Claude Code review it:

JavaScript
··············································function f(d) {
  var x = [];
  for (var i = 0; i < d.length; i++) {
    var item = d[i];
    if (item.n != null && item.n != undefined && item.n != '') {
      var el = document.getElementById('list');
      el.innerHTML = el.innerHTML + '<li onclick="handle(\'' + item.n + '\')">' + item.n + '</li>';
      x.push(item.n);
    }
  }
  return x;
}

function handle(n) {
  eval('alert("Hello, ' + n + '")');
}

async function getData() {
  const response = await fetch('/api/data');
  const data = await response.json();
  return data;
}

Some obvious problems with this code:

TypeIssue
Namingf, d, x, n reveal nothing about intent
SecurityinnerHTML string concatenation -> XSS risk
Securityeval() is a famously dangerous function
Errorsfetch has no try-catch and no response.ok check
PerformancegetElementById and DOM updates inside the loop
SyntaxUses var instead of const/let
Comparison!= null && != undefined && != '' can be simplified

Now open Claude Code (I'm using the VSCode Plugin) and type this into the prompt:

Text
Please review @bad-code.js for me

Claude Code asks Do you want to proceed with Skill? — hit "Yes".

Do you want to proceed with Skill?

You'll see Claude Code use the code-review Skill we just defined and return a result like this:

Best practices when authoring Skills

A few practices that make Skills more effective.

SKILL.md file size

Anthropic recommends keeping SKILL.md under 500 lines for good loading performance. If your Skill needs a lot of reference material, link out to external files:

Text
.claude/skills/my-skill/
├── SKILL.md          # keep concise, under 500 lines
├── reference.md      # detailed API docs
└── examples.md       # usage examples

Reference depth

Use single-level references (SKILL.md -> reference.md). Avoid deep nesting (SKILL.md -> reference.md -> other.md -> more.md) — it cuts loading time and keeps things simple.

Use scripts when it makes sense

If your Skill needs complex logic, put it in a script. The script's output gets returned to Claude, but the script source itself doesn't load into the Context — saving Tokens:

Text
····.claude/skills/data-processor/
├── SKILL.md
└── scripts/
    └── process.py    # complex data processing logic

Troubleshooting

Skill isn't getting triggered?

The most common reason is that the description is too vague. A good description should answer two questions:

  1. What does this Skill do? — list specific capabilities
  2. When should it be used? — include trigger phrases the user might say

Bad:

YAML
description: Helps with documents

Good:

YAML
description: Extract text and tables from PDF files, fill forms, merge PDFs. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.

Wrapping up

A common reaction at this point is "do I really need a Code Review Skill? Can't I just ask the AI to review my code directly?"

Fair question — but think of Skills as packaging expertise for the AI. Just like every company has its own technical style and conventions, Skills let you bundle that expertise so the AI matches your needs when it executes a task.

That's also why I included an Output format section instead of letting the AI freestyle — it ensures the response matches what I actually want.

So treating Skills like a team or a skill pack is the right mental model — it keeps the model focused on the knowledge you care about, instead of getting buried in raw data.
