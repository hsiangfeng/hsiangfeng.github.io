Use Dev Containers to Build a Safe, Isolated Environment for AI-Driven Development

#AI #Dev Containers #Automated Development #Isolated Environment

Intro

A lot of people are excited about automated development — some would love to let AI Agents handle every requirement end-to-end. But this comes with all kinds of considerations: security, environment consistency, dependency management, and so on. Not to mention there have been multiple incidents where Auto AI Agents, without an isolated environment, ran commands directly on the host and wiped systems or destroyed data. So this post is about how to use Dev Containers to build an isolated development environment where an AI Agent can do its work in a controlled space.

A quick intro to Dev Containers

Dev Containers is a container-based approach to isolating your development environment — the underlying tech is just a Docker Container. On top of that, Dev Containers gives you some extras: you can connect Visual Studio Code directly into the container to develop, and you can define environment settings, dependencies, and more through configuration files.

So why am I making a big deal about Dev Containers? Well, as AI Agents get more capable, we want to hand off as much work to them as possible. But AI Agent behavior is unpredictable — even with Prompt Engineering nudging it in the right direction, things can still go sideways. The Agent might run a dangerous command, install an unsafe package, and so on — any of which can damage your system.

The most famous recent example is probably the Google Antigravity incident, where Antigravity wiped a developer’s D drive completely. There are tons of these AI Agent stories — if you don’t give it an isolated environment, it can absolutely damage your system.

And that’s just one Agent. As you start running multiple AI Agents in parallel for different tasks, conflicts will pop up — incompatible dependency versions, resource contention, and so on. So using Dev Containers to build an isolated dev environment becomes really important.

What “isolation” actually means in Dev Containers

I want to be clear about what “isolation” means here:

  • File system isolation: the container can only access the project folder you mount in — it can’t access other files on your system (like your D drive, your desktop, etc.)
  • Execution environment isolation: packages and global settings installed in the container don’t pollute your system
  • Dependency isolation: different projects can use different versions of Node.js, Python, etc. — no interference

But heads up: the network is open by default.

What does that mean? Some people think Dev Containers are sealed off — completely cut off from external networks. That’s not the case.

Dev Containers default to Docker’s bridge network mode, so containers can access the external network normally. Things like git push to GitHub or npm install for downloading packages all work as expected.

Setting up a Dev Containers environment

So how do we actually use Dev Containers to build an isolated dev environment? I’ll go with a simple environment and the easiest setup — using Visual Studio Code to build a Dev Containers environment.

First, head to the Visual Studio Code Extensions marketplace and install the Dev Containers extension.

Next, run these commands to create a new folder and move into it:

You’ll have an empty folder. Open it in Visual Studio Code, hit F1, type Dev Containers: Add Dev Container Configuration Files, and pick that option.

A window pops up asking you where to add the configuration files. You’ll see two options:

  • Add configuration to workspace
  • Add configuration to user data folder

Normally you want to pick Add configuration to workspace — that way the config files live in the project folder and your team can share them.

Next you’ll see Select a container configuration template or enter a custom template ID. Plain English: pick a template that fits your dev environment. We’re building a Node.js dev environment, so I’ll pick Node.js.

For the version, pick the Node.js version you need. I’ll go with 20 (default) here.

You’ll then see Select additional features to install. Check whatever you need — I’m just going to skip past this.

After all this, your project directory will have a .devcontainer folder containing a devcontainer.json config file. Some templates also include a Dockerfile. These files define your Dev Containers environment.

Starting up the Dev Container

Next we start up the Dev Container. Before that, make sure you’ve got Docker Desktop installed. If you don’t, hit F1 and type Dev Containers: Install Docker — VS Code will guide you through it.

Once Docker is installed, hit F1 and type Dev Containers: Reopen in Container. This command reopens Visual Studio Code and connects it to the Dev Container (this might take a moment).

That means you’ve successfully entered your isolated Dev Containers environment.

What about when you’re done developing?

Once you’re done with your work in the Dev Container, you can just close Visual Studio Code. The Dev Container stops automatically and won’t affect your system in any way.

You might be worried — “will the code I wrote in the container disappear?” The answer is no.

Dev Containers default to Volume Mount, which mounts your project folder into the container. So any changes you make to project files inside the container sync back to the project folder on your host in real time.

To save your progress, you just use Git like normal:

These Git commands work the same whether you run them inside the container or after closing the container on your host.

You might also wonder — “how do I authenticate with GitHub inside the container? Do I need to set it up again?” Nope.

The Dev Containers extension handles authentication for you automatically:

  • SSH Key forwarding: SSH agent gets forwarded automatically, so the SSH keys you’ve already set up on your host work inside the container
  • Git Credential forwarding: HTTPS authentication is handled too
  • Git config forwarding: global settings like user.name and user.email get carried into the container

So as long as you’ve set up GitHub auth on your host, once you’re in the Dev Container you can just git push — no extra setup needed.

When you want to continue development next time, just hit F1 → Dev Containers: Reopen in Container again and you’re back in the same dev environment.

How do other team members join in?

Once you’ve pushed your project to GitHub, how do other team members get in on the action? Two ways:

Option 1: Clone first, then open the container

  1. git clone the project locally
  2. Open the project folder in Visual Studio Code
  3. VS Code detects the .devcontainer folder and pops up a prompt in the bottom right asking if you want to reopen in container
  4. Click “Reopen in Container” and it builds and enters the container automatically

Option 2: Clone directly into the container Volume

The other way is to use F1 → Dev Containers: Clone Repository in Container Volume. This clones the project directly into a Docker volume rather than the file system.

The upside is better file I/O performance — particularly handy for macOS and Windows users since Docker’s file system mount performance on these systems is relatively poor. The downside is the project files don’t show up on your file system — they’re only accessible through VS Code.

Wrap up

As AI Agents get more powerful, automated development isn’t a far-off dream anymore. But that also means we need to take dev environment security more seriously. With Dev Containers, we can let an AI Agent work inside a controlled, isolated environment — getting all the convenience of automation without worrying about it damaging the host system.

If you’re thinking about letting an AI Agent help you write code, I strongly recommend setting up Dev Containers first so you can hand off work without worry.
