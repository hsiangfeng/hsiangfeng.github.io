The AI model runs remotely, but a Coding Agent's tools may run on your local machine

Intro

We had an internal training session at work today, and a coworker asked a couple of questions that really stuck with me:

Is Claude actually running inside my computer? Otherwise how can it find my project, modify files, and even run commands on its own?

Honestly, I think that's a totally fair question. When you watch AI Agent tools like Claude Code, Codex CLI, or Gemini CLI read files, write and edit code, and run tests all by themselves, it really does look like there's an AI living inside your machine.

But the real answer actually splits into two parts. What you see is only partly running on your local machine — the other part runs on a remote server (for example, Claude's models usually run in a remote data center).

So this post is here to clear that up, and I'm aiming it at the total-beginner level, which means I'm not going to start from complicated APIs or JSON.

Once you separate "the model" from "the Coding Agent," the whole thing gets a lot easier to understand.

The model is the brain, the Coding Agent is the hands

When you use Claude Code, there are really three things to keep separate:

  1. Claude's model lives in a remote data center: the model's main job is to understand your request, read what it's given, and decide what to do next.
  2. Claude Code is an app installed on your computer: it's the program that talks to the remote model, manages permissions, and runs tools.
  3. Reading files, editing files, running tests — those really do happen on your computer: the thing actually doing the work is the Claude Code installed on your machine, not the remote model.

If that's still a bit fuzzy, here's an analogy that might help:

  • The remote model is the brain, the local Coding Agent is the hands.
  • The brain figures out the next step; the hands are what actually touch your files and run commands.

How Claude's model and the local Coding Agent divide the work

So when you install Claude Code, you're not downloading a full Claude model onto your computer — you're installing an app that can talk to the model and use your local tools.

Coding Agents like Codex CLI and Gemini CLI share the same core idea, though each product differs in which tools it can use, where those tools run, and when it needs your permission.

So when you say "fix this bug for me," what actually happens?

Let's set up a scenario. Say you type this into the Claude Code chat window:

Text
Please fix the bug in XXX.js, then run the tests.

Behind the scenes, the flow looks roughly like this:

  1. Claude Code (local) receives your request (the text, a.k.a. the Prompt)
  2. Claude Code sends your request plus whatever info is currently needed to the remote side (Claude's model)
  3. The remote model receives it and decides it should read XXX.js first
  4. The model returns a tool call: "please read XXX.js"
  5. Claude Code (local) checks permissions, then actually reads the file (if it doesn't have permission, it'll pop up and ask you to authorize)
  6. Claude Code hands the file contents back to the remote model
  7. The model reads through the contents, then decides how to change the code and which tests to run

And this whole thing doesn't run just once.

The model reads a file, searches for other code, edits the contents, runs the tests — and if the tests fail, it decides the next step based on the error message.

This loop of "model decides the next step → tool executes → result goes back to the model → model decides the next step again" is called the Agentic Loop.

The Agentic Loop — local tools and the remote model going back and forth while Claude Code fixes a bug

What is a tool call?

Tool calls are... a little tricky. In short, when you install Claude Code, it ships with a bunch of small programs bundled inside — things like:

  • A file reader (File Reader)
  • A file writer (File Writer)
  • A file deleter (File Deleter)

And so on. So the remote model never reaches directly into your computer — all it does is tell Claude Code:

Text
For my next step, I want to use the "file reader" tool, targeting XXX.js.

Only after Claude Code receives that request does it decide, based on your current permission settings:

  • Is this tool allowed to run?
  • Which folders can it read?
  • Does this operation need to ask the user first?
  • Once it succeeds, what result should go back to the model?

So "the model wants to use a tool" and "the tool actually ran successfully" are two different things.

The model proposes the next step; actually running it is the job of an external program or execution environment.

On top of that, tools don't always run on your computer. Some run on the vendor's servers or in a cloud sandbox — things like Codex cloud tasks, Code Interpreter, and Web Search.

But... why can the remote model see my code?

This is the fun part, and it ties back to the earlier point about "reading files."

The remote model doesn't get to see your entire hard drive, and it doesn't go wandering around your folders on its own.

The reason it can see XXX.js is usually that the local Coding Agent used the file reader tool to grab the contents, then handed the model whatever it needs to finish the task.

Besides file contents, every time the model works on a task it might also receive:

  • The request you just made
  • Earlier conversation history
  • Project rules and operation limits
  • Which tools the model is allowed to use
  • Command output, error messages, and test results

All of this together is what we call the Context.

The model needs to see enough context to know what it did earlier, and what it should do next and how to handle it.

When the conversation gets too long, the Coding Agent might keep only the relevant parts, or summarize and compress the older stuff.

Think of it like this: when your desk can't hold every document at once, you put the unimportant ones away and keep only what the current task needs on top. The technical term for that is Auto Compact.

Warning

Even though commands run locally, that doesn't mean none of your data ever leaves your computer. To help the remote model understand your project, the Coding Agent does send file contents, error messages, and tool results. So exactly which data gets sent depends on the product's design, your settings, and the terms of service.

Why do more Agents make your computer lag?

Okay, after all that setup, we can finally talk about the title — because you kind of needed the earlier flow explained before this makes sense.

When a Coding Agent is waiting for the remote model to reply, your computer is just sitting there keeping the app, the display, and the network connection alive. That part doesn't really put any load on your machine.

What actually slows your computer down is multiple Agents doing local work at the same time — things like:

  • Searching or reading a large number of project files simultaneously
  • Running tests, lint, type checks, or full builds at the same time
  • Spinning up multiple dev servers and browsers
  • Installing packages, unzipping files, or building caches
  • Reading and writing lots of files at once, or working across different Git worktrees

Multiple Agents running local work at the same time, stacking up CPU, RAM, and disk load

Say you have three Agents each running a full test suite at the same time — that's basically no different from you opening three terminals and running the tests yourself.

All that work hits the same local CPU, RAM, and disk, so of course your computer can slow down.

So a more accurate way to put it is:

The more Agents you open, the more local work can run at the same time. The main reason your computer lags is the stacked-up workload — not a remote model reaching through the air to eat your local GPU.

Of course, opening three Agents doesn't automatically make your computer three times slower (imagine if it did — how scary would that be, and how beefy would your machine need to be).

Three Agents all waiting for a remote reply, versus three Agents all building a large project at once, put completely different loads on your computer.

So why does everyone say AI is so GPU- and memory-hungry?

The answer here is really about the model itself — the model needs a ton of compute and memory. But when you're just using a cloud model, what's being consumed is the remote data center's hardware, not your laptop, desktop, or server.

Let me put this a different way:

  • The model weights are like one giant set of reference books: for the model to use what it learned, that data has to be accessible to the system.
  • Memory is like a desk: the bigger the desktop, the more model data, conversation history, and temporary results you can spread out at once.
  • The GPU is like a big crew that can all compute at the same time: the model needs to do a huge amount of math, and a GPU is great for chewing through that repetitive, massive workload together.

Model weights as reference books, memory as a desk, the GPU as a large crew all computing at once

So when the model runs, memory mainly holds:

  • The model's own weights
  • The KV Cache needed for the current conversation and input
  • Intermediate results during inference, plus the execution framework
Note

When we say "memory" here, we generally mean CPU memory (RAM) and GPU memory (VRAM), not HDD or SSD storage.

For example, running a 7B model needs roughly 14 GB just for the weights (a rough estimate), and actually running it adds KV Cache and other overhead on top. Quantization can shrink the model, but it can also come with trade-offs in quality, speed, or compatibility.

Large cloud models might even spread their weights across multiple GPUs or multiple machines, so you can't picture it as "the whole model definitely crammed into a single graphics card."

One simple table to see it clearly

Next, here's a table to make the difference between "where the model computes" and "where the tools run" clear:

SituationWhere the model computesWhere the tools runWhat your computer mainly does
Regular cloud chat, no toolsRemote data centerNo extra toolsRender the display, send requests, receive replies
Local Coding Agents like Claude Code, Codex CLIModel usually remoteFile read/write, terminal, etc. often localRun the Agent and tools, using CPU, RAM, disk
Cloud Agents like Codex cloud taskRemote data centerCloud container or sandboxSubmit the task, check results, apply changes
Local models like Ollama, LM StudioYour computerUsually also on your computerEven model inference is on you — can heavily use GPU / unified memory

The most important takeaway from this table is: where the model computes and where the tools run are two separate questions.

The same AI product can offer both local tools and remote tools at once — things like data analysis, Code Interpreter, Web Search, or server-side Code Execution.

So when you see "the AI ran a program for me," you can't immediately assume the program definitely ran on your own computer.

In the end it comes back to which feature you're using, and what that tool's execution environment is.

Local tools really do touch your computer — permissions matter

The model usually lives remotely, and when data is sent over, the model vendor does some basic sensitive-info filtering. But the actions a local Coding Agent runs can genuinely affect your computer.

Because it really can:

  • Modify or create files
  • Delete content
  • Install packages
  • Run tests or start servers
  • Connect to external systems when permission allows

Coding Agents like Claude Code and Codex limit what they can do through permissions, approval rules, or sandboxes. But you need to be clear that not every action is guaranteed to ask you first — it depends entirely on the mode you chose (Auto Mode), the boundaries of your workspace, and the type of operation.

If this is your first time using a Coding Agent, I'd strongly suggest starting with more conservative permissions. When the Coding Agent pops up a permission prompt, check what it's about to read, write, or delete, whether it needs network access, and whether the action is destructive.

If you just mindlessly hit Yes, there's a real chance your privacy and your computer could get leaked or damaged.

Conclusion

Let me wrap up the key points:

  1. Cloud models like Claude and GPT usually do their computation and inference in a remote data center
  2. Claude Code and Codex CLI are Coding Agent apps that run on your local machine
  3. The model only proposes content and next steps; the actual work is done by local, cloud, or third-party tools
  4. Whether opening more Agents causes lag depends on how much local work they're running at the same time
  5. Only when you use a local model does inference directly and heavily use your own GPU / unified memory — though local tools can also use the GPU depending on the work

So, back to the original question: "Is Claude running inside my computer?"

I'm confident you already have a clear answer by now. As long as you look at "where the model computes" and "where the tools run" separately, you won't mix up "AI needs a lot of GPU" with "a Coding Agent slows down my machine" as if they were the same thing.
