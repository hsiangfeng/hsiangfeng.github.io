Want to Run Your Own LLM? Understanding Model Parameters and Quantization

#AI #LLM #語言模型 #Ollama #LM Studio #Hugging Face #量化

Intro

If you’ve ever tried downloading an AI model to run on your own machine, you’ve probably hit Hugging Face and seen 7B, Q4_K_M, GGUF — a wall of codes with no clue which one your computer can handle. This post unpacks what those codes actually mean and how to pick a model that fits your hardware.

Note
If you don’t know how to run a model locally yet, check out Setting Up Your Own AI Language Model Lab With LM Studio (article in Mandarin) first — it covers everything from installation to usage.

What does the B after the model name mean?

When you’re browsing models, you’ll often see a number followed by B after the model name — 4B, 7B, 70B. That B stands for Billion, and it’s how many parameters (Parameters) the model has.

So what’s a parameter? Simple: parameters are the numerical values the model learned from massive amounts of data during training. If you want a more everyday analogy, think of parameters as the books a person has read — someone who has read tens of thousands of books can answer way more questions than someone who has read a few thousand, right? It’s the same idea for parameters: 7B means 7 billion parameters, 70B means 70 billion. More parameters means the model has learned more.

I touched briefly on B/M/T notation in Setting Up Your Own AI Language Model Lab With LM Studio (article in Mandarin) — this post goes deeper into how parameter count translates to actual usage.

Bigger parameter count is always better, right?

You might think: “Why not just grab the model with the biggest number?”

It’s not that simple. More parameters means more hardware demand — your machine has to be beefy enough to actually run it. And more parameters doesn’t automatically mean smarter. A well-trained 14B model can outperform a mediocre 70B one. So parameter count is just one signal, not the whole story.

What kind of parameter range fits what kind of task? Here’s a rough breakdown:

Parameters Tier Suited for
1B - 3B Lightweight Simple chat, summarization, translation — phones and low-end devices
4B - 9B Entry Coding assist, Q&A, general chat — the starting point for most people
13B - 14B Mid Noticeably better reasoning, can handle more complex problems
27B - 32B Advanced Multi-step reasoning, domain-specific Q&A
70B+ High-end Approaches cloud-service quality, hardware bar is high

Looking at the table — for most people, 4B - 9B is already plenty. No need to jump straight to 70B.

Will my computer actually run it?

Once you know the parameter count, the next very practical question is: how much memory do I actually need?

Easy rule: with no compression, every billion parameters takes about 2 GB of memory. So a 7B model needs around 14 GB, and a 70B model needs about 140 GB.

Seeing that 140 GB number, you might already be sweating — “My machine only has 16 GB of RAM, so I can only run the smallest models?”

Hold on — that’s exactly what quantization solves. We’ll get to it shortly. First, let’s look at what else is hiding in those model names.

Decoding every part of a model name

Say you spot this model on Hugging Face:

It looks dense, but each segment means something specific:

Part Meaning
bartowski Uploader (a well-known community quantizer)
Meta Developer (Meta AI)
Llama Model family name

So what that whole string is telling you: a Meta Llama 3.1 model, repackaged by bartowski, 8B parameters, instruction-tuned, stored in GGUF format.

What do tags like Instruct, Base, Chat mean?

Model names often carry tags that signal what kind of training the model went through. The most common ones:

  • Base: the raw model — like a student who has read a lot of books but hasn’t learned how to converse yet. Not great for chatting directly
  • Instruct / IT: instruction-tuned — follows your instructions and replies accordingly. This is what most people want
  • Chat: optimised for multi-turn conversation
  • Coder: specialised for code generation
  • Distill: distilled from a larger model — keeps most of the capability but is smaller

Personally — if you just want to chat locally or have AI help with everyday tasks, see Instruct, pick it.

Note
The LM Studio post covers the basic concept of Distill vs. Instruct — feel free to flip back if you’re curious.

What’s MoE? Don’t panic when you see A22B

A lot of recent models include the MoE tag, like:

You see 235B and immediately think: “235 billion parameters? My machine has zero chance, right?”

Hold on — there’s an A22B after it. MoE stands for Mixture of Experts. What does that mean? Simply put, the model contains many “experts”, but it doesn’t activate all of them for every question — it only picks a few relevant ones to handle the task.

So 235B is the total parameter count across all experts (235 billion), and A22B is the Active parameter count — only 22 billion actually run for any given query. In other words, this model has the knowledge breadth of 235B parameters, but the runtime cost of a 22B-class model.

That’s why MoE has been getting popular — it makes giant models that used to be exclusively in the cloud actually runnable on consumer hardware.

Note
One catch: MoE models still need disk space for all parameters — the file size doesn’t shrink because only some experts are active. The savings are at runtime in memory and compute.

What is quantization? It’s how your machine runs big models

Earlier I mentioned that an uncompressed 7B model needs 14 GB of memory — still too much for most people. That’s where quantization comes in.

So what’s quantization? Think of it like photo resolution. A 50-megapixel photo is the sharpest, but the file is huge. If you compress it down to 12 megapixels, you basically can’t tell the difference on your phone screen, but the file is way smaller. Quantization works similarly — it lowers the precision of each parameter, shrinking the file and the memory footprint.

What do Q4, Q8 mean?

When you download a model, you’ll see codes like Q4_K_M and Q8_0. The number is the quantization precision — smaller means more compression and smaller files, but with some quality loss.

Using a 7B model as the example:

Quantization 7B model size Quality impact
FP16 (no compression) About 14 GB None
Q8_0 (8-bit) About 7 GB Almost imperceptible
Q5_K_M (5-bit) About 4.8 GB Very minor
Q4_K_M (4-bit) About 3.8 GB Minor — most people’s pick
Q3_K_M (3-bit) About 3 GB Starts to feel noticeable
Q2_K (2-bit) About 2.5 GB Clearly degraded — not recommended

Look at that table — the same 7B model goes from 14 GB to 3.8 GB at Q4_K_M, so an 8 GB machine can handle it. And Q4 quality loss is genuinely small — for everyday chat you won’t really feel it.

So what about K_S, K_M, K_L?

Quantization codes often have K_S, K_M, or K_L suffixes, which describe the compression strategy:

  • K_L (Large): keeps the most detail, biggest file
  • K_M (Medium): balanced — the most commonly recommended option
  • K_S (Small): most compressed, smallest file, fastest

The K refers to K-Quants, which is a smarter form of compression. Instead of squashing everything to the same precision, it figures out which parameters matter more and keeps higher precision for those, only compressing the less important ones harder. So overall quality is much better than naive compression.

Personally — for most situations, Q4_K_M is a great starting point.

How do I pick between GGUF, GPTQ, and AWQ?

Beyond quantization level, you’ll also see different file formats. Here are the three most common:

GGUF is the dominant format for running models locally — Ollama, LM Studio, and llama.cpp all use it. The big upside is that it runs on both CPU and GPU and isn’t picky about hardware. For most people, just pick GGUF.

GPTQ and AWQ are NVIDIA-GPU-only formats. They preserve quality well, but you have to have an NVIDIA card. No NVIDIA GPU? Skip them.

So if you’re not sure what to pick, just remember one combo: GGUF + Q4_K_M. It works for most situations.

Picking a model based on your hardware

With those concepts down, the practical question becomes: based on the hardware I have, what should I download?

8 GB RAM (no discrete GPU)

This tier is limited but still usable. Aim for 3B - 4B models with Q4_K_M, like Phi-4 Mini (3.8B) or Llama 3.2-3B — files are around 2 - 3 GB. Simple chat and summarization are fine, though inference will be slow.

16 GB RAM or 8 GB VRAM

This is the sweet spot for most people right now. You can run 7B - 8B models with Q4_K_M — Llama 3.1-8B or Qwen 2.5-7B — file sizes around 4 - 6 GB. Coding assist, Q&A, and translation all work well.

32 GB RAM or 16 - 24 GB VRAM

If you have an RTX 4090 (24 GB VRAM) or 32 GB+ RAM, your options open up a lot. 30B-class models can use Q5_K_M or even Q6_K, and 70B can run at Q4_K_M. Quality is approaching cloud-service territory.

Apple Silicon Mac users

If you’re on a Mac, here’s some good news. Apple Silicon uses Unified Memory — system RAM can be used directly as GPU memory, which is a huge advantage for running LLMs.

For example, a 36 GB MacBook Pro can run 30B-class models (Q4 - Q8). A similarly priced Windows laptop, where system RAM and VRAM are separate, would have to fall back to CPU-only inference and be much slower.

Mac config Memory Models it can run
M-series base 8 - 16 GB 3B - 7B (Q4)
M-series Pro 18 - 36 GB 13B - 30B (Q4 - Q8)
M-series Max 64 - 96 GB 70B (Q4) or 30B (near-lossless)
M-series Ultra 128 GB+ 100B+ MoE models

A practical rule of thumb

A general guideline:

Rather than picking a big model and squeezing it into very low quantization, pick a slightly smaller model at higher quantization.

What does that mean? If your machine’s memory is limited, an 8B model at Q5_K_M will usually outperform a 13B model crushed down to Q3_K_M. Over-compression strips too much capability away.

Where to download models

Now that you know how to pick, where do you download them?

Hugging Face

Think of it as the GitHub of AI models — it’s the largest model hub right now. Beyond original models, the community uploads pre-quantized versions. When looking for GGUF models, recognised quantizers like bartowski and hugging-quants are reliable bets.

Ollama

If you don’t want to mess with details, Ollama is the easiest option. Open your terminal and run a single command to download and start a model:

It auto-downloads the model, picks a quantization level appropriate for your hardware, and drops you straight into a chat — no file picking required.

LM Studio

If you prefer a GUI, LM Studio lets you search, download, and load models entirely inside an app — no command line involved.

This space moves fast — new models drop almost every month, so these picks will shift over time. If you want to track the latest, watch Hugging Face’s Open LLM Leaderboard for benchmark rankings across models.

Wrapping up

Quick recap:

  • The B in model names means billions of parameters — bigger means more knowledge, but heavier hardware demands
  • Pick the Instruct variant when downloading; Base is for researchers
  • For MoE models with an A-prefixed number, the active parameter count is much smaller than the total — way easier on resources
  • Quantization compresses models; Q4_K_M is the best starting point for most people
  • For file format, GGUF is the safe pick — runs on both CPU and GPU
  • Pick a model that fits cleanly in your memory rather than cramming a giant model down to extreme quantization
  • Mac users have a natural edge for local inference thanks to Unified Memory

Hope this gives you a clearer picture of LLM parameters and quantization — next time you’re picking a model on Hugging Face, those codes won’t feel like alphabet soup anymore.
