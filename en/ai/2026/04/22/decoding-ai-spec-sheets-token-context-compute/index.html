Decoding AI Spec Sheets: Token, Context Window, Compute — What Are These Numbers Actually About?

#AI #LLM #Context Window #Token #Tokenizer #BPE #Temperature #FLOPS #GPU #Compute

Intro

If you’re like me and you’ve been checking out a lot of AI product intro pages lately, you’ve probably seen this kind of stuff: “Supports 1M Token Context Window”, “100 billion parameters”, “Input only $3.00 per million Tokens”. Looks impressive, but what do these numbers actually mean?

I think a lot of people, when they first see these spec sheets, default to “bigger numbers = better, right?” But each number behind the scenes means something different — bigger isn’t always better. Once you understand these terms, you can actually judge which AI fits your needs, instead of just being led around by marketing numbers.

So this post breaks down all the jargon AI vendors love to use, so the next time you see one of these spec sheets, you won’t be lost.

Token: the smallest currency unit in the AI world

What is a Token?

If you’ve used ChatGPT or Claude, you’ve definitely heard the word “Token”, because everything from API pricing (an API is basically an interface that lets developers call AI services from code) to Context Window limits revolves around Tokens.

In short, a Token is the smallest unit a language model (the core tech behind ChatGPT, Claude, etc.) uses when processing text. You can think of it as the “basic currency” the AI uses when reading and producing text. Every time the AI reads your input or produces a response, the calculation is done in units of Tokens.

But a Token isn’t the same as “one character” or “one word” — how it gets split depends on the Tokenizer each model uses. For example, in English "darkness" might get split into two Tokens, "dark" and "ness", while "I" is one complete Token. So how does the Tokenizer decide how to split text?

Wait, doesn’t a computer only understand 0 and 1? Why do we need Tokens?

Before getting into Tokenizers, you might have a more basic question: “Doesn’t a computer only understand 0 and 1? Why doesn’t AI just process text in binary directly — why this extra Token concept?”

True, a computer’s most fundamental operation is 0 and 1, and all text in a computer ends up as a string of 0s and 1s. The English letter A is stored as 01000001 (8 bits of 0 and 1). The Chinese character 你 is more complex and needs 24 bits to represent. So if text can already be represented in 0s and 1s, why does a language model need an additional Token layer?

The reason is efficiency and semantics.

If you make a language model process text one character at a time, the length the model has to handle becomes really long. Take "Hello" — character by character it’s 5 units, which seems fine right? But take "The quick brown fox jumps over the lazy dog" — character by character it’s 43 units. If we use Tokens via the splitting tech we’ll cover later, that same sentence might only be about 9-10 Tokens — several times shorter.

The difference becomes more pronounced with long text, because a language model’s compute requirements scale dramatically with length (roughly the square of the length). So 4x the length means 16x the compute — that’s just impractical.

Beyond efficiency, Tokens also capture more meaningful semantic units. The character d on its own doesn’t mean much, but the Token "dark" carries clear meaning — the model can directly learn that “dark” represents the concept of “darkness”, instead of having to slowly piece this concept together from d, a, r, k. So Tokens are essentially a balance point between “the binary world of computers” and “the semantic world of humans” — not so granular that efficiency tanks, not so coarse that the vocabulary list explodes.

How does a Tokenizer split text?

Most current language models use a splitting algorithm called BPE (full name: Byte Pair Encoding — no need to memorize that). GPT and Claude both use BPE or variations of it. The core logic isn’t hard to understand: it starts by splitting text into the smallest units (usually individual characters), then repeatedly merges the most commonly co-occurring “adjacent character pairs” into new Tokens, repeating until the vocabulary list (the “list of all Tokens this Tokenizer recognizes”) reaches a preset size.

Words like "darkness" might get split into "dark" and "ness" — two Tokens — because both parts are very common combinations in English. Interestingly, "brightness" also gets split into "bright" and "ness" — you’ll notice the suffix "ness" shares the same Token ID, which helps the model understand that “darkness” and “brightness” have some semantic relationship.

The final trained Tokenizer has a fixed vocabulary list, usually between 32,000 and 200,000 Tokens. For example, OpenAI’s vocabulary (called cl100k_base — 100k means about 100,000) contains roughly 100,000 Tokens, and all text input gets split against this list.

Why does Chinese consume more Tokens than English?

Here’s a real-world issue. Most language model Tokenizers are trained on text data that’s mostly English, so English Token efficiency is really high — about every 4 English letters maps to 1 Token, which works out to roughly 1-1.5 Tokens per English word.

Chinese is different. Because the Tokenizer mainly learned how to split text from English data, Chinese characters take up a relatively small share of the vocabulary. Most Chinese characters need 1.5 to 2 Tokens to represent, and rarer characters might need 3 Tokens. The reason is that the Tokenizer’s vocabulary size is limited (say 100,000 slots), but the common Chinese, Japanese, and Korean characters alone add up to tens of thousands — there’s no way every one gets its own slot, so many Chinese characters have to be represented as combinations of multiple Tokens.

A concrete comparison:

Content Text Approximate Token count
English “Hello, how are you?” About 6 Tokens
Chinese “你好，你好嗎？” About 11 Tokens

Both express roughly the same meaning, but the Chinese version uses nearly twice as many Tokens. This means that under the same Context Window limit, Chinese users can fit less info in than English users. And if you’re calling the model via API, Chinese costs a bit more than English too.

Note
This problem has been gradually improving in recent years. OpenAI’s later o200k_base Tokenizer and various models’ multilingual optimizations have narrowed the gap in Token efficiency between Chinese and English — but the difference still exists.

Input Token vs Output Token

Another important concept is that Tokens are split into “Input Tokens” and “Output Tokens”.

Input Tokens are everything you send to the AI — your question, the documents you upload, the System Prompt (system instructions developers preset to tell the AI how to respond), conversation history, and so on. Add it all up and that’s your Input Token consumption.

Output Tokens are what the AI sends back to you — every Token produced consumes one Output Token.

Why split them out? Because generating text takes more compute than reading text, so Output Tokens usually cost several times more than Input Tokens. For mainstream models today, Output is typically priced 4-6x higher than Input. That’s why you’ll see API pricing tables list Input and Output separately.

In short, the longer your question and the more the AI replies, the more Tokens consumed and the higher the cost.

Context Window: the model’s “working memory” limit

What is Context Window?

If you’ve ever chatted with ChatGPT for a while and noticed it starts forgetting things you mentioned earlier, you’ve already personally experienced the Context Window limit.

Context Window is just the “maximum number of Tokens” a language model can handle in a single conversation. You can think of it as the AI’s work desk — bigger desk means more documents spread out for reference at once, but the desk has a finite size, and content beyond the limit gets pushed off or compressed.

Important to emphasize: Context Window includes everything — your System Prompt, conversation history, documents you upload, and the AI’s responses are all packed into the same Context Window. So when an AI vendor says “Supports 1M Token Context Window”, that doesn’t mean you can send a 1M-Token question and then get a 1M-Token answer. It means the entire conversation’s “input plus output total” can’t exceed that limit.

So how big is 1M Tokens really? Quick note on the units commonly seen in AI spec sheets: K stands for Thousand and M stands for Million. So 1M Tokens is 1 million Tokens, and 128K is 128,000 Tokens. 1 million Tokens converts to roughly 750,000 English words, or about 500,000-650,000 Chinese characters (since Chinese consumes more Tokens, as mentioned earlier) — basically several books worth of content.

How big are Context Windows these days?

If you check the intro pages from various AI vendors, you’ll see Context Windows ranging from hundreds of thousands to tens of millions of Tokens. This number gets updated every few months, so I won’t list specific model specs here — you can always check the latest numbers on each vendor’s official website.

But there’s one concept worth understanding: Context Window and “Maximum Output Tokens” are two different limits. Context Window refers to the total cap for the entire conversation (input + output), while Maximum Output Tokens refers to how many Tokens the model can produce in a single reply.

For example, say a model has a 1M Token Context Window but only 64K max output. That means it can “read” a lot (the entire Context Window can hold stuff), but it can only “write” up to 64K Tokens at a time — it can’t reply with an entire book in one shot. Some models even claim Context Windows of tens of millions, but their max output might only be a few thousand Tokens. So just looking at the Context Window number isn’t enough — you also need to pay attention to the max output limit.

Marketing numbers vs actual performance — different things

Here’s a really important concept: the advertised Context Window size and what the model can actually effectively handle are two different things.

According to various independent tests, most models start showing noticeable accuracy drops as they approach their Context Window limit. The model might miss important info you mentioned earlier, or its understanding of context can drift. Generally, the actually usable range is only about 60-70% of the advertised number.

This is similar to humans, actually. It’s like reading a 500-page book — by the time you’re at the end, you don’t remember the details from the early chapters that clearly. Language models have a similar problem. Research shows models remember the beginning and end of Context most strongly — content in the middle is what gets ignored most easily. This is called the “Lost in the Middle” phenomenon — basically, content stuffed in the middle is most easily ignored by the model.

So next time you see an AI vendor advertising “Supports 10M Tokens”, don’t immediately assume you can dump 10M Tokens in and expect a perfect response. In actual use, you still need to consider the effective handling range.

Note
A practical tip: if you need the AI to process a large amount of documents, instead of stuffing them all in at once, processing them in batches and summarizing before merging usually works better.

Compute: the engine behind AI

What is compute?

You’ve probably seen “compute”, “GPU”, and “NVIDIA” mentioned together in the news a lot. So what is compute?

In short, compute is “how many calculations a computer can do in a given time”, usually measured in FLOPS — Floating Point Operations Per Second, or “how many floating-point operations per second”. Floating-point operations means calculations involving numbers with decimal points, like 3.14 × 2.71. Almost all calculations inside an AI model are this type, so FLOPS has become the most common unit for measuring compute. The bigger this number, the more calculations per second this computer (or GPU — the processor chip on a graphics card, more on this later) can do.

Common compute units:

Unit Full name Order of magnitude
GFLOPS Giga FLOPS 1 billion ops per second
TFLOPS Tera FLOPS 1 trillion ops per second
PFLOPS Peta FLOPS 1,000 trillion ops per second
EFLOPS Exa FLOPS 1 million trillion ops per second

These numbers sound astronomical, but training a large language model needs astronomical compute on top of astronomical compute, which is why thousands of GPUs running in parallel are needed.

Training vs inference

In the AI world, compute usage splits into two main phases: Training and Inference.

Training is the process of getting the AI model to “learn knowledge” — like having a student read all the textbooks, papers, and web content, then “compressing” that knowledge into the model’s parameters. This process eats massive amounts of compute. Training a GPT-4-class model is estimated to need tens of thousands of GPUs running continuously for several months.

Inference is the model “applying its knowledge” after learning — every time you ask ChatGPT a question or have Claude write code, the calculations the model performs are inference. Compared to training, single inference calls need much less compute, but with millions of users worldwide using these AI services daily, the cumulative inference compute demand is also staggering.

A simple analogy: training is like spending years to develop a doctor, while inference is like that doctor seeing patients every day. Developing them requires huge time and resource investment, but once developed, each patient visit is relatively quick.

Why are GPUs better suited for AI than CPUs?

You might wonder why AI uses GPUs instead of regular CPUs. Quick intro to both:

  • CPU (Central Processing Unit): the “brain” inside your computer, handling all sorts of compute tasks — opening a browser, running programs, all on it
  • GPU (Graphics Processing Unit): originally a chip on graphics cards specifically for processing game graphics and 3D imagery

So why use a chip originally meant for gaming graphics to do AI? Because AI calculations are essentially massive amounts of numbers that need to be added and multiplied at the same time — and GPUs are naturally great at “doing tons of simple calculations simultaneously”. CPUs are better at “handling one really complex thing at a time”.

An analogy: CPUs are like a really smart math professor who can only do one problem at a time but does it fast and accurately. GPUs are like a thousand average calculators — each one only does simple addition, but with 1,000 working in parallel, the total speed far exceeds the professor. AI calculations happen to be that “simple but huge volume” kind of work, so GPUs became the most central hardware for AI.

The AI training market is currently almost monopolized by NVIDIA, with over 90% market share — one of the main reasons NVIDIA’s stock and market cap have grown so dramatically the past few years.

Parameters: the model’s “brain capacity”

What are parameters?

When you see news saying “this model has 70 billion parameters” or “GPT-5 supposedly has trillions of parameters”, what are these “parameters” actually?

In short, parameters are the “values” the language model learned during training. Every parameter is a number representing the model’s understanding of some language pattern. These numbers combined form the model’s ability to judge “what the next character should be”. Using a brain analogy, parameters are kind of like “synaptic connections” in the brain — more synapses, more complex info the brain can handle.

Note
If you want to dive deeper into the technical details of parameters and quantization, including how to pick a model based on your hardware, check out Want to run your own LLM? Understand parameters and quantization so you don’t pick the wrong model — it has more thorough explanations.

Are more parameters always better?

A lot of people intuitively think more parameters = stronger. There was a trend in this direction early on — the industry called it “Scaling Law”: the idea that as long as you make models bigger, feed more data, and throw enough compute at it, model capabilities keep improving. But by 2025-2026, people started noticing that simply increasing parameter count brings less and less improvement.

The more popular approach now is an architecture called MoE (Mixture of Experts). You can think of it like a big company with many departments specializing in different things, but each project only sends the most relevant departments to handle it — not the whole company. So with this architecture, the model’s “total parameter count” might reach trillions, but each query only activates a small subset of parameters. The actually-used parameters are called “Active Parameters”. For example, an MoE model claiming 2 trillion total parameters might only use 200 billion active parameters per query, keeping strong capabilities without burning through compute.

So when looking at parameter counts now, it’s more important to distinguish “total parameters” from “active parameters” — looking only at total parameters doesn’t accurately reflect a model’s actual capabilities anymore.

Why do some vendors not disclose parameter counts?

You might have noticed that OpenAI stopped publicly disclosing model parameter counts starting with GPT-4. Anthropic’s Claude series has never officially disclosed them either, and the same goes for Google’s Gemini.

The reason is mostly competitive considerations — disclosing parameter counts tells competitors your model architecture details, while also avoiding the market doing crude parameter-count-only comparisons. Most current industry estimates of parameter counts are unofficial numbers reverse-engineered from model behavior, inference speed, and pricing — so when you see these estimates, take them as reference only, no need to take them too seriously.

The “seasoning” for model inference: Temperature, Top-P, Top-K

Beyond the “hard specs” above, if you’ve used the API or some advanced AI tools, you might have run into Temperature, Top-P, Top-K. These aren’t model specs themselves — they’re “knobs” you can adjust at the time the model is generating a reply (the “inference” stage I mentioned), and they determine the style and randomness of the model’s reply.

Temperature

Temperature is the most common and most intuitive parameter. It controls the “creativity level” of the model’s reply, usually ranging from 0 to 2. I mentioned earlier that when a language model predicts the next character, it calculates the probability (likelihood) of each candidate. Temperature adjusts whether the model “takes risks” choosing characters with lower probabilities.

  • Lower Temperature (closer to 0): the model leans toward picking the highest-probability character, replies are more deterministic and consistent — good for code or factual questions
  • Higher Temperature (closer to 1 or above): the model is more willing to pick lower-probability characters, replies are more varied and creative — good for storytelling or brainstorming

In short, low Temperature = “conservative and stable”, high Temperature = “bold and risky”.

Top-P (nucleus sampling)

Top-P’s logic is “only pick from the most likely candidates until their combined probability reaches P%”. For example, Top-P set to 0.9 means the model first sorts all candidates from highest to lowest probability, then only considers the ones whose combined probability reaches 90% — the rest, which are very unlikely, are excluded.

The advantage of Top-P is that it adjusts dynamically. If the model is very confident about the next character, it might only consider 1-2 candidate Tokens, but if it’s uncertain, it considers more candidates — making sampling more flexible.

Top-K

Top-K is more straightforward: “only consider the top K Tokens by probability”. For example, Top-K set to 50 means no matter how the combined probabilities of those 50 Tokens stack up, the model fixes its choice to those 50.

Compared to Top-P’s dynamic adjustment, Top-K’s candidate count is fixed, which can be less flexible in some cases. In 2026 practice, most API vendors mainly expose Temperature and Top-P for users to adjust — Top-K is less commonly seen in commercial APIs.

Note
The general advice is to only adjust either Temperature or Top-P at a time — adjusting both can sometimes produce conflicting effects. If you’re not sure how to set them, the defaults are usually plenty good for most situations.

Token pricing: how do AI services charge?

After understanding the terms above, let’s wrap up with a practical question: how is AI API pricing actually structured?

Most mainstream AI APIs price per “million Tokens” (usually abbreviated as MTok), and Input Token vs Output Token pricing is calculated separately — Output is usually several times more expensive than Input.

Specific prices vary by vendor and model, and they change really fast, so I won’t list specific numbers here — you can check each vendor’s official pricing page for the latest. But a few concepts don’t change:

  1. Stronger models cost more: each vendor usually has a “flagship” tier and a “lightweight” tier. Flagship is the most capable but also the most expensive; lightweight is much cheaper but with limited capability — the price gap can be tens of times.
  2. Output is more expensive than Input: because generating text takes more compute than reading text, Output Token pricing is typically 4-6x Input.
  3. Prices keep dropping: AI API prices have been dropping rapidly — equivalent capabilities cost a fraction of what they did two years ago. Good news for both developers and regular users.

So a good usage strategy is to give simple tasks to cheap lightweight models, and reserve flagship models for genuinely complex tasks — striking a nice balance between cost and performance.

Wrap up

After reading this post, you shouldn’t be lost the next time you see an AI product spec sheet. Here’s a quick recap:

  • Token: the smallest unit AI uses for text, not the same as character count, Chinese consumes more Tokens than English
  • Context Window: the maximum total Tokens the model can process in one go — actual effective capacity is usually less than advertised, and pay attention to max output as a separate limit
  • Compute (FLOPS): a measure of computing power — training needs massive compute, inference is smaller per call but cumulatively staggering
  • Parameters: the count of learned numerical values in the model — distinguish total parameters from active parameters now
  • Temperature / Top-P: knobs for adjusting reply style — neither higher nor lower is universally better

Understanding the core concepts behind these terms matters more than memorizing specific numbers. Numbers update every few months, but concepts don’t change. Hopefully this post helps!
