What Are Claude Code's Built-in Tools Doing? Understand How It Picks Tools — Then Write Better Prompts

#AI #Claude Code #AI Agent #Claude Code Tools #AI Development

Intro

This post is a quick chat about Claude Code’s built-in Tools. If you’ve used Claude Code, you’ve probably noticed something — sometimes when you ask it to read a file, it just opens it with Read. Other times when you ask “what components does this project have,” it goes off and runs Glob and Grep. And occasionally, it takes a detour through Bash’s cat or grep, which is confusing because the built-in tools are right there. Claude actually has its own logic for picking tools — this post walks through what built-in Tools Claude Code has, how it decides which one to use, and how to reverse-engineer its tool usage to fix your prompts.

Note
If you’re not familiar with Claude Code basics yet, start with Want to develop with Claude Code? This post takes you from beginner to advanced (article in Mandarin) — it has a thorough beginner walkthrough. Every tool name in this post matches the official Tools reference.

Why bother understanding the built-in Tools

You might be thinking — I’m not Claude, why do I need to know what tools it uses?

Short answer — once you can read which tool Claude picked, you can reverse-engineer where your Prompt was unclear.

Here’s a concrete example. Say you tell Claude “take a look at the login logic” and it fires off seven or eight Globs without finding anything, then finally lands the answer with Grep. If you can’t read what it’s doing, you’ll just feel like “wow, this is slow.” But if you know that Glob matches by filename pattern and Grep searches by content, you’d realize the issue — your description “login logic” is an abstract concept, the filename probably doesn’t even contain login. Next time, you’d just hand it the path src/auth/ directly, and things speed up instantly.

The other reason is that Claude Code’s Hook configs, Skill allowed-tools, and permissions deny rules are all built around tool names. If you don’t know which tools exist and what they do, you can’t even write these advanced configs correctly.

What built-in Tools does Claude Code have?

A quick overview first — I’ll expand on the common ones afterward.

Category Tool name Main use
Read Read Read file contents
Read Glob Find files by filename pattern
Read Grep Search by content (powered by ripgrep)
Edit Edit Precise string replacement on a file
Edit Write Overwrite the entire file
Edit NotebookEdit Edit cells in a Jupyter Notebook
Execute Bash Run Shell commands
Execute PowerShell Run PowerShell commands (Windows-focused)
Execute Monitor Run a command in the background and stream every output line back to Claude
Delegate Agent Dispatch a Subagent to handle a task
Code LSP Type-check, jump to definition, find references via Language Server
Tasks TaskCreate / TaskUpdate / TaskList Manage task lists
Interaction AskUserQuestion Ask the user a multiple-choice question
Plan EnterPlanMode / ExitPlanMode Enter or exit Plan Mode
Schedule CronCreate / CronDelete / CronList Schedule tasks within the session
Web WebFetch Fetch a web page
Web WebSearch Search the web
Other Skill Run a Skill
Other ToolSearch Load deferred tool definitions

It looks like a lot, but day-to-day you’ll see Read, Glob, Grep, Edit, Write, Bash, and Agent the most. The rest only show up occasionally. Next, I’ll dig into those key ones, and at the end I’ll add a few that are still worth knowing about.

The reading trio: Read, Glob, Grep

A lot of people starting with Claude Code wonder why reading files needs three different tools. Reading is reading, right? Actually, these three do completely different things, and mixing them up means Claude picks wrong.

Read

Read is the most direct file-reading tool. Hand it an absolute path and it returns the file contents — and crucially, with line numbers attached:

Those line numbers aren’t decoration. The Edit tool relies on them later, because Edit needs precise line content to do its replacement.

A few more things about Read worth knowing:

  • It reads up to 2000 lines at a time — bigger files need offset and limit to chunk it
  • It can read images, PDFs, and Jupyter Notebooks. Images are presented to Claude visually
  • The path must be absolute — relative paths aren’t allowed

So why doesn’t Claude just use Bash’s cat? Because cat doesn’t give line numbers, which would break a follow-up Edit. And cat dumps the whole file in one go with no chunking — big files would blow out your Context. The official system prompt explicitly says “if you can use Read, don’t use cat.”

Glob

Glob finds files by filename pattern, with wildcard support like this:

Results come back sorted by modification time — most recently modified first. That ordering is genuinely useful, because when you’re debugging something, the most recently changed files are usually the ones you want to look at.

Short version — Glob is for when you already have a sense of what the filename looks like. If you’re searching for files that contain some snippet of content, that’s Grep’s job.

Grep

Grep wraps ripgrep under the hood, so it’s much faster than Bash’s grep, and it supports full regex.

It has three output modes:

  • files_with_matches (default): only returns matching file paths
  • content: returns the actual matching lines
  • count: returns how many matches per file

Personally, I use content mode the most — paired with -n (show line numbers) and -C (before/after context). That way Claude sees the search hit plus a few lines of surrounding context in one shot, so it doesn’t have to pop open Read again to confirm.

Note
Grep only searches single lines by default. If you need to match a pattern that spans multiple lines (like “from a function definition to the first return”), set multiline: true. People forget this a lot, so when Claude is searching for cross-line structures and comes up empty, give it a nudge.

When to use which

To wrap up these three, here’s when each one fits:

  • You already know the path: use Read directly
  • You know the filename pattern but not where it lives: use Glob
  • You only know that some string appears in the content: use Grep

If your prompt is upfront about it — like “Grep the whole project for the constant LEGACY_AUTH“ — Claude won’t waste a Glob round and will Grep straight away.

Edit and Write are not the same thing

After reading comes editing. Claude Code has two file-editing tools — Edit and Write. People often think they’re roughly the same, but they’re really not.

Edit is precise replacement

Edit takes an old_string and a new_string — Claude Code finds an exact match for old_string in the file and swaps it for new_string. The keyword is exact match — whitespace, indentation, line endings all have to line up.

Because of this, Edit has an important constraint: you have to Read the file before you can Edit it. Editing without reading first gets blocked, to stop Claude from editing files based on stale memory.

If the same string appears multiple times in the file, Edit errors out by default and asks you to provide more context to make the string unique. If you actually want to replace all of them, pass replace_all: true.

Write overwrites the whole file

Write rewrites the whole file. If the path exists, the old contents get blown away.

Sounds convenient, right? But it’s actually a fairly dangerous tool, because it wipes out anything you edited manually that Claude didn’t see. Imagine you have two windows open — Claude Code on one side, VSCode on the other — and Claude does a Write. Whatever you just edited in VSCode is gone.

The official system prompt strongly recommends “if you can use Edit, don’t use Write.” Write is mostly for the “create a brand-new file” case.

If you notice Claude keeps using Write instead of Edit, it’s usually because its remembered file content doesn’t match what’s on disk anymore (you edited something in between), so Edit can’t find a match no matter what it tries and falls back to Write. Telling it to Read again before editing usually fixes it.

Bash: when does Claude take the detour?

The Bash tool is exactly what it sounds like — running Shell commands. But Claude Code’s system prompt specifically nudges it: if there’s a built-in tool that does the job, don’t use Bash. The reason — cat, grep, find and friends all have built-in equivalents. The built-ins have better UX, easier review and approval flows, and friendlier output formats for Claude’s downstream processing.

So when does Claude still use Bash? Mostly these cases:

  • You ask it to run tests or builds: things like pnpm test, pnpm build
  • Git operations: git status, git diff, git log (Claude Code lists read-only git commands as auto-approved)
  • Starting a dev server: e.g. pnpm dev
  • Package manager commands: pnpm install
  • Things the built-in tools don’t cover: complex pipes (like finding the five largest files)

Bash also has a run_in_background parameter that runs a command in the background. Useful for long-running tasks like a dev server or a long build — Claude doesn’t have to sit and wait, it can move on to other things, and gets notified when the background command finishes.

Note
One detail worth knowing: the Read and Edit deny rules only block the built-in tools, not Bash subprocesses. So if you put Read(./.env) in deny, Claude reading .env via Read gets blocked — but it can still get through with cat .env. To actually block this at the OS level, you need sandboxing.

Agent: handing tasks off to a Subagent

This is the tool most people don’t get — and the one with the biggest payoff.

How a Subagent differs from the main conversation

Short version — the Agent tool spins up a fresh Subagent to handle a task, and that Subagent has a few traits:

  • Its own independent Context Window — completely separate from the main conversation
  • Returns only a summary back to the main conversation when done
  • It can’t see the main conversation’s history, so the instructions you give it have to be fully self-contained
  • Each Subagent inherits the parent’s permissions plus extra tool restrictions
  • You can specify the Subagent type — more on that next

To put it in everyday terms — a Subagent is like asking an intern to go do something. You hand them the task description plus the context they need, they come back with the conclusion, and the books they flipped through and the notes they took along the way are invisible to you.

The three built-in Subagents

Claude Code ships with three built-in Subagents, each designed for slightly different uses:

Subagent Model Tool permissions Good fit
Explore Haiku (fast) Read-only (no Write/Edit) Wide-scope file searches, understanding a codebase
Plan Same as main conversation Read-only (no Write/Edit) Plan Mode helper for researching the project
general-purpose Same as main conversation All tools Complex tasks that need to read and edit

What this table makes obvious is that Explore is the one most often auto-triggered — it’s cheap and fast, and read-only so it can’t mess anything up. When dispatching Explore, Claude also picks a thoroughness level (quick, medium, very thorough). The more specific your Prompt, the lower the thoroughness it can get away with — saving more Tokens.

Beyond the built-ins, you can define your own Subagents under .claude/agents/. That’s more advanced, so I’m leaving it out of this post.

What kinds of tasks fit the Agent tool

Tasks that need to read lots of files to draw a conclusion are the sweet spot:

  • “Scan the whole project and tell me if any old version of some API is still in use”
  • “Trace the implementation of this feature, from the router all the way down to the service”
  • “Where in this codebase is there test coverage and where isn’t there”

If you let the main conversation do this directly, every file it reads gets stuffed into the main Context, and you barely have any Token budget left for the actual work. With Agent, the Subagent does the legwork and the main conversation just gets a summary back — Context stays intact.

I touched on this technique in Is your Claude Code Context getting eaten alive? Full guide to AGENTS.md and Context management (article in Mandarin). One more thing worth adding — you can ask Claude to dispatch multiple Agents in parallel. Like “find all the API endpoints AND all the React components at the same time” — both Subagents run concurrently, much faster than doing them one at a time.

What kinds of tasks don’t fit the Agent tool

Agent isn’t a silver bullet though. Some cases don’t fit:

  • Tasks that need back-and-forth discussion: Subagents can’t see the main conversation history, so every dispatch you have to re-explain the background. For things that need lots of iteration, Subagent actually slows you down
  • Single-file edits with a clear goal: Just Read and Edit yourself — handing off to a Subagent is the long way around
  • Tasks that genuinely depend on the main conversation’s context: If you can’t be bothered re-explaining, just let the main conversation handle it

Note
If you don’t want Claude to auto-dispatch a particular Subagent, add a rule like Agent(Explore) to permissions.deny. The syntax only blocks that specific Subagent — others still work.

Other tools worth knowing

Above were the day-to-day heavy hitters. Here are a few that show up less often but are still worth knowing.

WebFetch and WebSearch

The names look similar but they do different things. WebFetch takes a URL and pulls the page down as Markdown — good for “I have an official doc URL, please follow it” cases. WebSearch actually searches keywords — good for “how do I use the latest version of some API” cases where you need to find information.

If you tell Claude “look up Vue 3.5’s latest features,” it’ll usually WebSearch first to find the official docs URL, then WebFetch to grab the full content. That’s a common combo move.

AskUserQuestion

When Claude hits a choice it’s not sure about, it sometimes pops a question at you — that’s the AskUserQuestion tool. It’s multiple-choice. For instance, if you ask it to scaffold a project but didn’t specify React or Vue, it might pause and offer you a few options instead of guessing.

If you’d rather it not stop to ask, write your defaults into CLAUDE.md and it’ll decide for itself instead of popping up.

LSP

LSP is a newer tool (via Language Server Protocol), good for code intelligence — jumping to function definitions, finding where a symbol is referenced, listing implementations of an interface, things like that. The most noticeable effect is that after every Edit, LSP automatically reports type errors and warnings for that file, so Claude doesn’t have to run a build separately to know whether it broke something.

Heads up though — this tool only kicks in if you’ve installed the Code Intelligence Plugin for the relevant language. Without it, Claude can’t see anything from LSP.

Monitor

Monitor lets Claude run a command in the background and streams every output line back as it happens, good for:

  • Watching a log file and pinging when an error shows up
  • Monitoring CI status changes
  • Watching a folder for new files

The difference vs Bash’s run_in_background — Monitor “keeps feeding data to Claude,” while Bash background “notifies once when it’s done.”

TaskCreate / TaskUpdate / TaskList

These are how Claude manages task lists internally. When a task is complex, it builds a todo list and ticks off items one by one. When you see Claude pulling up a checklist in the Claude Code UI, that’s these tools at work.

A small detail here — interactive sessions (the everyday claude command) use TaskCreate / TaskUpdate / TaskList, while Headless mode and the Agent SDK use an older tool called TodoWrite. Same concept, different implementation.

How does Claude decide which tool to use?

Now that we’ve covered the tools, how does Claude actually decide?

Short version — there are two sources.

The first is the System Prompt. Claude Code loads a long system prompt at startup that explicitly says which tools to prefer and what each tool’s constraints are — rules like “if you can use a built-in tool, don’t use Bash” or “Read before Edit.” So part of Claude’s tool-picking logic is taught up front.

The second is Claude’s own judgment. The model uses your request, the current conversation context, and the file info in Context to guess which tool fits best. This part is where Prompt clarity matters — the vaguer your Prompt, the more likely Claude takes a detour.

A concrete example to make it click:

  • You say “open src/auth/login.ts“ → Claude goes straight to Read because the path is explicit
  • You say “find the login logic” → Claude probably Globs **/login* or **/auth/** first, falls back to Grep if nothing matches
  • You say “scan the whole project for usage of deprecated APIs” → A smarter response is dispatching the Explore Subagent

From this angle, the more specific your Prompt, the more accurately Claude uses tools. That’s exactly why I wanted you to learn how to read tool usage patterns earlier.

Reverse-engineering your Prompt from how Claude uses tools

This is the heart of this post. Everything above was setup for this.

Claude keeps Read-ing different files without finding the target

Usually means your description was too abstract and it’s guessing at file locations. For example, you say “fix the home page,” but it has no idea which file the home page is, so it keeps Read-ing things that look home-page-shaped.

Fix — give the path or a precise description in your Prompt, like “fix the search logic in src/pages/Home.vue.”

Claude keeps Glob-ing and finds nothing

Means your keywords don’t line up with the actual filenames. Like you say “find login-related files” but the project actually uses Auth* naming, so Globbing for *login* finds nothing.

Fix — tell it the naming convention, or switch to Grep on the import strings — that’s usually more reliable.

Claude keeps failing Edits and falls back to Write

If you see Claude trying Edit a few times, getting “old_string not found” errors, and finally switching to Write, two common causes:

  1. You edited the file in your IDE in the meantime — Claude’s remembered content doesn’t match disk
  2. Inconsistent indentation or trailing whitespace (especially tabs vs spaces)

Fix — tell it to Read the file again before editing, or be explicit: “please Read the latest content first before editing.”

Claude doesn’t dispatch a Subagent and grinds through files itself

If a task should’ve gone to Agent (like a project-wide scan) but Claude is Read-ing files one by one, it’s because it didn’t recognize this as a big task.

Fix — write it directly into your Prompt: “please use the Explore subagent to scan the whole project for me.” Be explicit about wanting a Subagent dispatched.

Claude keeps detouring through Bash

If it should be Read but it’s running cat, or it should be Grep but it’s running grep, it’s usually one of these:

  • permissions.deny is blocking the built-in tools and forcing Claude to fall back to Bash (remember, the deny rule only blocks built-ins, not Bash)
  • There are unnecessary nudges in CLAUDE.md
  • The thing it’s doing genuinely needs a pipe combo, like “find the five largest log files”

The first two can be fixed via /permissions and reviewing CLAUDE.md. The third is fine — let it be.

Wrap-up

Quick recap of the main points:

  • Claude Code’s built-in Tools mostly fall into reading (Read, Glob, Grep), editing (Edit, Write), executing (Bash), and delegating (Agent), plus newer ones like LSP, Monitor, and TaskCreate
  • Read brings line numbers, Edit needs Read first, and use Write sparingly to avoid overwriting your manual edits
  • Glob finds filename patterns, Grep finds content strings — getting these straight stops Claude from picking wrong
  • Agent is the most powerful and most overlooked tool. Claude Code has three built-in Subagents — Explore, Plan, and general-purpose. For tasks that need to read lots of files, Explore is the most cost-effective
  • permissions.deny rules for Read/Edit only block the built-ins, not Bash subprocesses. To block at the OS level, go with sandboxing
  • Once you can read which tool Claude picked, you can reverse-engineer where your Prompt was unclear

Hope this gave you a clearer picture of Claude Code’s built-in Tools — next time it picks the wrong tool, you’ll know how to tweak the Prompt.
