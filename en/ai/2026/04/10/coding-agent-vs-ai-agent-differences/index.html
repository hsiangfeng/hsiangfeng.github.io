AI Agent Isn't One Thing Anymore — Understanding Coding Agents and Other AI Agent Types

#AI #Claude Code #GitHub Copilot #Cursor #AI Agent #Coding Agent #Code Agent #Devin

Intro

A while back I wrote Have you heard of LLM, RAG, and Prompt? A walkthrough of common generative AI terms (article in Mandarin), where I introduced AI Agent as a catch-all term — a system that can understand a task, break it into steps, and use tools to finish the job.

But here in 2026, you keep seeing more specific terms popping up — Coding Agent, Research Agent, Browser Agent. AI Agent isn’t a fuzzy umbrella term anymore. So this post breaks down what these subcategories actually mean and how they differ from each other.

A quick recap of AI Agent

Before getting into subcategories, let me quickly recap what an AI Agent is.

The current industry consensus, in plain terms, is something like this:

A system where an LLM uses tools in a loop to achieve a goal.

A few things in that sentence matter:

  • The LLM makes the decisions: You don’t hardcode the flow — the LLM itself decides what to do next
  • It uses tools: It can read files, call APIs, browse the web, run commands, and so on
  • It runs in a loop: Take an action, observe the result, decide the next step, take another action — repeat until the task is done
  • It’s goal-oriented: There’s a clear job to finish, not aimless chatting

There’s an important distinction here. In their Building Effective Agents post, Anthropic separates Workflow from Agent. A Workflow is an LLM and tools running through a flow you wrote ahead of time — every step is something you defined. An Agent is the LLM dynamically choosing which tool to use and how, with the LLM in control of the whole process.

To put it in everyday terms — a Workflow is like hiring a part-timer to follow an SOP, where every step is spelled out. An Agent is more like hiring an experienced employee: you tell them the goal, and they figure out how to get there.

The main categories of AI Agent

Through 2025 and into 2026, specialized AI Agent tools have exploded, and the industry has started carving out subtypes. There’s no single agreed-upon taxonomy, but for the sake of communication, here are the common categories grouped by what these tools are aimed at:

Category Description Representative tools
Coding Agent Agent specialized for software development Claude Code, Cursor, GitHub Copilot, Devin
Research Agent Gathers data, cross-references, produces reports Google Deep Research, Perplexity
Browser Agent Automates web pages, fills forms, completes online tasks OpenAI Operator, Google Project Mariner

These categories aren’t strictly separate. Plenty of tools span multiple categories. Take Claude Code — it’s primarily a Coding Agent, but it can absolutely search the web if you ask it to. The point of categorizing isn’t to draw hard boundaries, it’s to tell you where each tool’s home turf is.

There’s also a trend worth mentioning: Multi-Agent (multi-agent collaboration). The idea is to compose different types of Agents together — a Research Agent gathers data, a Coding Agent implements, an analysis Agent verifies the result. It’s a bit like microservices — each Agent focuses on one thing, and they coordinate through some orchestration layer.

What is a Coding Agent?

A Coding Agent is an AI Agent specialized for software development. Its toolset and workflow are designed around how software actually gets built.

MIT’s Missing Semester course added an Agentic Coding lecture in 2026 that defines a Coding Agent like this:

A conversational AI model equipped with tools to read and write files, search the web, and execute terminal commands.

On the surface that sounds a lot like a regular AI Agent, right? The difference is that a Coding Agent’s entire design is built around the software development loop. Here’s a side-by-side:

Aspect Regular AI Agent Coding Agent
Domain General tasks Software development
Toolset General tools (search, APIs, etc.) Dev tools (filesystem, Git, terminal, test framework, Linter)
Feedback loop Depends on the task Write code → run tests → read errors → fix → run tests again
Output Text, decisions, actions Code changes, commits, Pull Requests
Context understanding Depends on the task Needs to grasp the whole Codebase structure, dependencies, and type system
Verification Depends on the domain Tests, compilation, and lint can verify automatically

That last row is actually a huge advantage Coding Agents have over other types. Software development comes with objective verification baked in — whether the code runs, whether tests pass, whether there are syntax errors — these are all binary outcomes that don’t need a human to judge. That lets a Coding Agent keep trying and self-correcting without anyone stepping in. Other domains usually don’t have such a clean automated verification path.

Coding Agent vs Code Assistant

There’s another thing that’s easy to mix up: a Coding Agent is not the same as a Code Assistant.

The AI coding tools we used to use were mostly at the Code Assistant level. The original GitHub Copilot is the classic example. A Code Assistant does this — while you’re typing code, it suggests the next line or the next function. You’re still the main operator, it’s just helping along the way.

A Coding Agent, on the other hand, takes a goal from you and plans which files to change, makes the changes, runs the tests, and fixes errors on its own. You basically just review the final result.

To put it in everyday terms — a Code Assistant is like having a co-pilot in the passenger seat helping you read the road, but your hands are still on the wheel. A Coding Agent is more like autopilot — you give it a destination, it drives, and you just keep an eye on it to make sure things are fine.

These days many tools do both. GitHub Copilot started as a Code Assistant, but after adding Agent Mode and Coding Agent it became both. Cursor is similar — Tab autocomplete is Assistant-level, but Agent Mode is Coding Agent-level.

What Coding Agents are out there?

The main Coding Agents on the market right now look something like this:

Claude Code

Anthropic’s CLI dev tool — it runs straight in your terminal. It can understand the whole Codebase structure, read files, edit files, run commands, run tests on its own, and supports advanced features like Hooks, Skills, and MCP Servers. Personally, Claude Code is the Coding Agent I use the most.

Note
If you want a deeper dive into Claude Code, check out Want to develop with Claude Code? This post takes you from beginner to advanced (article in Mandarin).

GitHub Copilot Coding Agent

GitHub’s cloud-based Coding Agent that runs inside the GitHub Actions environment. You assign it to an Issue, and it spins up a branch in the background, writes code, runs tests, and opens a PR when it’s done. This is different from Copilot’s Agent Mode — Agent Mode runs inside VSCode, the Coding Agent runs in the cloud.

Cursor Agent Mode

Cursor’s Agent Mode can search the entire Codebase, edit across files, and run terminal commands. In 2026, Cursor also launched Background Agents — they clone your Repo to the cloud and work on isolated branches, so you can run multiple of them in parallel.

Devin

Cognition’s fully autonomous Coding Agent — the most extreme end of the automation spectrum. You give it a task and it runs the whole thing in a cloud environment from start to finish — coding, debugging, deploying. It’s a good fit for well-defined tasks that don’t need a lot of back-and-forth.

OpenAI Codex

OpenAI’s cloud-based Coding Agent that can run dev tasks asynchronously in the background. You give it an instruction and it finishes in a sandbox environment, then hands you the result for review.

Others

There’s also a handful of open-source or more niche tools, like Aider (terminal tool), Cline (VSCode extension), and OpenHands (self-hostable platform).

Note
If you want a full comparison of these tools, check out my earlier post How to pick AI Coding tools in 2026? Full comparison of GitHub Copilot, Cursor, Claude Code, Kiro, and Antigravity (article in Mandarin).

The core workflow of a Coding Agent

No matter which Coding Agent you pick, the core workflow is pretty much the same:

  1. Understand the task: You tell it what you want, it reads the relevant code to figure out the current state
  2. Plan an approach: Decide which files to change, how, and in what order
  3. Make the changes: Start editing — could be one file, could be a dozen
  4. Verify the result: Run tests, lint, compile, see what breaks
  5. Self-correct: If a test fails, read the error and try again
  6. Deliver: Commit, open a PR, or just leave the changes in your working directory

Coding Agents work because software development has clear verification (tests, compilation), which lets the Agent keep iterating in this loop until it gets the right result.

Wrap-up

Quick recap of the main points:

Term Description
AI Agent The umbrella term — a system where an LLM uses tools in a loop to achieve a goal
Coding Agent A subtype of AI Agent specialized for software development — can read/write files, operate Git, run tests
Code Assistant Helps you write code (autocomplete, suggestions). You’re still the main operator
Research Agent Gathers data, cross-references, produces research reports
Browser Agent Automates web pages and completes online tasks
Multi-Agent Multiple Agents collaborating on more complex tasks

The short version — AI Agent is still the umbrella term, but as tools multiply and applications widen, the industry has started carving out more specific names for different domains. For us as engineers, the one we’ll bump into most is the Coding Agent — those AI dev tools that can read code, edit code, run tests, and open PRs on their own.

Hope this gave you a clearer picture of how these terms differ.
