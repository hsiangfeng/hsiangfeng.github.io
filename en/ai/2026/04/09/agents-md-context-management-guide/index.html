Is Your Claude Code Context Getting Eaten Alive? AGENTS.md and the Complete Context Management Guide

Intro

This post is about Claude Code's Context management problem. A lot of people install a bunch of Skills, Plugins, and MCP Servers, then write a giant CLAUDE.md — and before they've even started working, half their Context is already gone. We'll also cover AGENTS.md (a cross-tool universal standard) and how to make sure your Context goes where it actually matters.

Note If you're not familiar with Claude Code's basic operations yet, check out Want to develop with Claude Code? This guide takes you from beginner to advanced (article in Mandarin) first. If you want to learn more about writing CLAUDE.md, see CLAUDE.md and Rules for AI writing tips (article in Mandarin).

A quick intro to the Context Window

Before we get into management, let me recap what the Context Window is.

Whether you're using Claude Code, Cursor, Copilot, or any other AI dev tool, there's a Context Window concept behind all of them. You can think of it as the AI's short-term memory. Everything the AI needs for this conversation has to fit inside this space — system instructions, rule files, conversation history, file contents it's read, command execution results, and so on. The fuller the Context, the worse the AI performs. This concept applies to any AI dev tool.

This post uses Claude Code as the example, but the concepts apply regardless of which tool you use. The Context Window sizes for Claude Code right now are:

  • Standard: 200,000 tokens
  • Extended (Opus 4.6 / Sonnet 4.6): 1,000,000 tokens

But the space you actually have to work with is way less than you'd think, because Claude Code has already stuffed a bunch of stuff in there before you even start typing. The System Prompt alone takes several thousand Tokens, plus tool definitions, CLAUDE.md, Auto Memory, Skill descriptions, MCP tool names, environment info — for a project with a lot loaded up, you're already 10K+ Tokens in before you've started working.

You can type /context in Claude Code to see your current Token usage. It uses a colored grid to show how much each item takes up, plus optimization suggestions.

And the Context Window doesn't have to be totally full to cause problems. From my own experience, around 60% in the AI's attention starts to drop — it might ignore stuff in your rule files, or forget things mentioned earlier in the conversation. When Context is about to fill up, Claude Code triggers Auto Compaction, which compresses prior conversation history into a summary to free up space. But Auto Compaction is actually pretty unreliable and easily loses important info, so don't rely on auto-compaction. I'll cover the right approach later.

The culprits eating your Context

Now that we understand Context, let's talk about what's most likely silently devouring your Context.

CLAUDE.md is too long

A lot of people stuff entire project docs, API docs, and detailed architecture explanations into CLAUDE.md, ending up with a several-hundred-line file. The problem is, CLAUDE.md is loaded every conversation — whether you need it for that conversation or not, it sits there occupying your Context.

For me personally, I'd suggest keeping CLAUDE.md under 200 lines and only including stuff Claude can't infer from the code itself — build/test commands, team Code Style conventions (if they differ from defaults), important architectural decisions, common gotchas, and so on. If some rules only apply to specific directories, you can use .claude/rules/ with paths to scope them, so they only load when Claude accesses the relevant files.

I covered this in more detail in CLAUDE.md and Rules for AI writing tips (article in Mandarin).

Too many MCP Servers

Each MCP Server has a bunch of tool definitions. If Tool Search isn't enabled, all the tools' full definitions get loaded into Context at startup — the more Servers you install, the more Tokens get eaten up.

Good news: Claude Code has Tool Search (also called Deferred Tool Loading) enabled by default. This mechanism only loads MCP tool names initially — full definitions only get loaded when Claude actually needs to use a particular tool. This significantly reduces MCP Token consumption at startup. You can use /mcp to manage MCP Server connection states — disconnect Servers you don't need right now to save a lot of Context.

Too many Skills and Plugins installed

Skills load similarly to MCP — only names and descriptions are loaded at startup, with full content only loading when you actually invoke a Skill. But Skill descriptions have a total cap (1% of Context Window by default), so installing too many can cause descriptions to get truncated, which means Claude might mismatch your instruction to a Skill.

Here's a really useful tip: if you have Skills that are manually triggered (like deploy, commit, etc.), add disable-model-invocation: true in the SKILL.md, and Claude won't see those Skill descriptions at all. They only load when you manually invoke /skill-name, taking zero Context. I covered this fully in Agent Skills: More Token-Efficient Than MCP — Demoed With Claude Code.

Letting conversations go too long without cleaning

This is probably the gotcha most people hit. People do a lot of different things in a single conversation — fix feature A, then feature B, ask a few unrelated questions in between — and the conversation history gets longer and longer, Context gets fuller and fuller. Worse, if you keep correcting Claude's mistakes, those failed attempts stay in Context too, which actually messes with Claude's later reasoning.

"Just use /compact to compress, right?" you might think.

But /compact is actually pretty unreliable. When it compresses, it easily drops important details — which files you changed, current task progress, test results, etc. After compression, Claude might forget what it did before.

So the approach I'd actually recommend is manually save + restart conversation. The flow looks like this:

  1. When Context hits about 60% (use /context to check), or when you've just finished a feature
  2. Have Claude enter Plan Mode (press Shift+Tab twice)
  3. Ask it to save current progress, modified files, todo list, etc. to a Markdown file
  4. Use /clear to wipe the entire conversation
  5. Start a new conversation, ask Claude to read that Markdown file and continue

For example, you could tell Claude something like:

text
Save my current work progress as markdown to .claude/progress.md,
including:
- What task I'm currently working on
- Which files I've modified
- What's still left to do
- Test results

After saving, run /clear, then start a new conversation with:

text
Read .claude/progress.md and continue the previous work.

This lets you seamlessly pick up where you left off, and you can precisely control what info to keep and what to drop. Once you clear the conversation, those failed attempts and irrelevant chatter get cleared too — Claude's attention is more focused.

A few other small tips: between different tasks, just /clear and start over — no need to save. For small questions, use /btw — the answer shows up in a floating window without going into conversation history, so it doesn't take Context. And if you've failed to correct Claude two times in a row, stop trying — just /clear, restart, and bake what you learned into the new Prompt. It'll be faster.

What is AGENTS.md?

Now let's talk about something related to CLAUDE.md: AGENTS.md.

You might wonder: I already have CLAUDE.md, why do I need AGENTS.md?

Here's the background. There are more and more AI dev tools out there now, and each tool has its own rule file format. Claude Code uses CLAUDE.md, Cursor uses .cursor/rules/*.mdc, GitHub Copilot uses .github/copilot-instructions.md. If your team has people using Claude Code, others using Cursor, others using Copilot, you need to maintain several different rule files in different formats — and the content is all about the same. That's a real waste of time.

AGENTS.md was born to solve this. In short, it's a cross-tool universal standard — a Markdown file in your project root that any tool supporting this standard can read.

AspectCLAUDE.mdAGENTS.md
ScopeOnly Claude Code reads itCross-tool universal standard
Supported toolsClaude CodeCodex CLI, Cursor, Copilot CLI, Gemini CLI, etc.
Special features@imports, hierarchical loading, path rules, /init generationBasic Markdown instructions
Claude Code supportNative readingWon't auto-read, must import via CLAUDE.md

One important thing here: Claude Code doesn't auto-read AGENTS.md — it only knows CLAUDE.md. So if your project has AGENTS.md, you need to use @ syntax in CLAUDE.md to import it:

markdown
@AGENTS.md

## Claude Code-specific settings
Use Plan mode for changes under `src/billing/`.

So when do you use which? For me personally, if your team is all on Claude Code, you don't really need AGENTS.md — just use CLAUDE.md. But if your project uses multiple AI tools, AGENTS.md is a better choice. Put shared rules in AGENTS.md and have each tool import it plus add its own specific settings — at least you don't have to update rules in multiple places every time.

Practical Context management tips

We've talked about a lot of problems above. Here are some practical tips you can actually use.

Use Subagents for exploration

If you need Claude to do exploratory work — like "look at the architecture of this codebase" — use a Subagent. What does that mean? A Subagent has its own independent Context Window. It can read a bunch of files, do the analysis, and return only a concise summary to your main conversation. That way your main Context doesn't get blown out with file contents.

Use path-scoped rules

Put rules that only apply to specific directories under .claude/rules/, using paths to scope when they load:

markdown
---
paths:
  - src/api/**
---

API routes always use RESTful style. Standard response format is { data, error, message }.

This rule only loads when Claude accesses files under src/api/ — otherwise it doesn't take Context.

Use Plan Mode to save Tokens

Press Shift+Tab twice to switch to Plan Mode. In this mode, Claude thinks through the plan first before doing anything, which usually saves a fair amount of Tokens — it's not trying things and failing as it goes.

Add a save-progress rule to CLAUDE.md

I mentioned the manual save + restart approach above. You can add a section to CLAUDE.md so Claude knows exactly what to save when you ask:

markdown
## Progress save rules

When asked to save progress, organize the following info into Markdown at .claude/progress.md:
- Description of the current task
- List of modified files
- Outstanding todo items
- Test commands and test results
- Any gotchas to watch out for

This way, every time you need to clean Context, just tell Claude "save" and it knows what to do.

Use permissions.deny to exclude unneeded files

If you have files you don't want Claude Code reading — like .env, secrets/, build/, sensitive or unnecessary stuff — you can exclude them with permissions.deny in .claude/settings.json:

json
{
  "permissions": {
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(./secrets/**)",
      "Read(./build)"
    ]
  }
}

Excluded files disappear from search results, and Claude can't read them — so unrelated file contents don't take up Context.

Use Status Line for real-time monitoring

If you want to know how much Context you're using at any given moment, set up Status Line at the bottom of the terminal to show real-time usage. I covered this in detail in Claude Code Status Line setup guide.

Wrap up

To summarize the key points of this post:

  • AGENTS.md is a cross-tool universal standard — useful when teams mix multiple AI tools. But Claude Code doesn't auto-read it, so you need to import it via @AGENTS.md in CLAUDE.md.
  • Keep CLAUDE.md under 200 lines, only include stuff Claude can't infer from the code.
  • Regularly use /context to check Token usage.
  • At ~60% Context or after finishing a feature, manually save + /clear and restart. Don't rely on Auto Compaction.
  • Disconnect MCP Servers you're not using. Use /mcp to manage connections.
  • For manually triggered Skills, add disable-model-invocation: true so they don't take Context.
  • Use Subagents for exploratory work — don't blow out the main Context.
  • Use path-scoped rules, Plan Mode, and permissions.deny to precisely control Context usage.

Hopefully this gives you a better grasp of Context management. After all, Tokens aren't cheap — save where you can.
