From 3 to 100: Passing isitagentready's AI Agent Readiness Test

#AI #AI Agent #robots.txt #MCP #Cloudflare #Cloudflare Worker #Agent Skills #WebMCP

Intro

If you run a tech blog like I do, you’re probably already familiar with SEO. I’ve written before about AEO (Answer Engine Optimization) (article in Mandarin) — basically how to get AI search engines to cite your content. But recently I came across something that goes a step further: Cloudflare’s isitagentready.com, a tool that scans your site and tells you, “Is your website ready for AI Agents to drop by?”

Out of curiosity I threw my own blog at it — and almost fell out of my chair when the results came back. Out of 12 items I only passed 3, with a sad little score of 3 out of 100. The Level it gave me? The lowest possible.

So this post walks through the whole process of getting from 3 all the way to 100. If you want to try the same thing on your own site, you can follow along.

What is Agent Readiness, and why should you care?

Before we start fixing anything, let’s talk about what this thing actually is.

Have you ever stopped to think about how ChatGPT, Claude, Perplexity, and other AI tools actually “see” your website when they access it? They don’t have eyeballs like humans — they need standardized signals to figure out what’s on your site and what it can do.

Imagine you run a small shop. Human customers walk in, look at the sign, read the menu, done. But if your customer is an “AI customer,” it needs you to hang a machine-readable QR code or some kind of standardized label out front so it knows what you sell and how to order.

Agent Readiness is exactly that — does your website have those “signs for AI” properly hung up?

What is isitagentready.com actually testing?

The tool grades your site across four dimensions, totaling 12 items. Let me break down what each one is doing in plain English.

Category 1: Discoverability (can AI even find you?) — 3 items

This category checks whether AI can “discover” your site, like whether your shop has a sign out front:

  • robots.txt: a file that tells crawlers “here’s what you’re allowed to look at, and here’s what’s off-limits”
  • Sitemap: a map of your site that tells AI all the pages you have
  • Link headers: signposts placed inside your HTTP response that point AI toward more resources (API catalogs, RSS, etc.)

Category 2: Content (can AI understand what you have?) — 1 item

  • Markdown Negotiation: when an AI says, “hey, can you give me this in Markdown?”, can your site actually return Markdown instead of a pile of HTML tags?

Why does AI prefer Markdown? Because Markdown is clean — no CSS classes, no nested div soup. AI processes it faster and burns way fewer tokens.

Category 3: Bot Access Control (rules for crawlers) — 2 items

  • AI bot rules: does your robots.txt have specific rules for AI crawlers (like GPTBot, ClaudeBot)?
  • Content Signals: have you clearly told AI, “you can use my content for search, but not for training”?

Category 4: Protocol Discovery — 6 items

This category is more “advanced.” It’s mostly about hosting standardized JSON files on your site so AI Agents know which interaction methods you support:

  • API Catalog: publish your API directory
  • OAuth/OIDC: describes your authentication methods
  • OAuth Protected Resource: describes protected resources
  • MCP Server Card: basic info about your MCP server
  • Agent Skills index: which “skills” your site exposes to AI
  • WebMCP: expose tools to AI directly through JavaScript on the page

Honestly, these 6 don’t mean much for a pure blog — they’re more relevant for sites that actually offer API services. But hey, if we’re going for the perfect score, let’s go all the way.

My starting point: 3 out of 100

The very first scan looked like this:

  • ✅ robots.txt (already had one)
  • ✅ Sitemap (Hexo generates it automatically)
  • ✅ AI bot rules (set these up back when I wrote the AEO post)
  • ❌ The other 9 items: all failing

Alright, time to fix them one by one.

Fix #1: Content Signals

This is the easiest one. Just add a single line to your robots.txt. The line tells AI: “you’re welcome to use my content for search, just don’t use it to train your models.”

What the three values mean:

  • ai-train=no: don’t use my content to train AI
  • search=yes: feel free to index for AI search
  • ai-input=no: don’t pipe my content directly into AI as input material

If your robots.txt is hand-maintained, just slap it on the end. In my case, I use Hexo, and the robots.txt is auto-generated by a script — so I had to go modify the script that generates it.

Fix #2: Link Response Headers

This one checks whether your homepage’s HTTP response includes a Link header.

Quick refresher on HTTP headers: every time a browser (or AI Agent) requests data from your site, your site returns the HTML — plus some “extra info” attached. That extra info is the headers. The Link header is one of them, used to say, “hey, here are some related resources you might want to check out.”

Important note: this needs to be an HTTP header, not the <link> tag inside your HTML. Two different things.

If your site goes through Cloudflare (the orange cloud is on for your DNS), you can use Cloudflare’s Transform Rule to add it:

  1. Go to the Cloudflare Dashboard, pick your domain
  2. Go to Rules > click Create Rule in the top right > select Response Header Transform Rule
  3. Name the rule Agent Discovery Link Headers
  4. Set the condition: field “URI Path”, operator “equals”, value /
  5. Scroll down to the “Then…” section and choose “Add static”
  6. Header name: Link, value: whatever resources you want to point to (API catalog, Sitemap, RSS, etc.)

Once that’s saved, every time someone (or some AI) hits your homepage, Cloudflare will tack the Link header onto the response automatically.

Fix #3: Markdown Negotiation

This one wants your site to support “content negotiation” — meaning when an AI Agent requests a page with Accept: text/markdown in the header, your site should return Markdown instead of regular HTML.

The approach here is to use a Cloudflare Worker. Think of a Worker as a “middleman” — every request that goes through Cloudflare passes through the Worker first. The Worker can check whether this request is from an AI Agent (does it have Accept: text/markdown?), and if so, convert the HTML to Markdown before returning it.

I’ll cover this Worker together with the .well-known solution below, since they share the same Worker.

Fix #4: WebMCP

WebMCP is a newer standard that lets your webpage tell an AI Agent — via JavaScript — “hey, here are the things you can do on my site.”

Implementation is simple: add a snippet of JavaScript to your page that calls navigator.modelContext.provideContext() to register some “tools.”

Fix #5: The .well-known series

Now for the final five items. They all require placing JSON files under your site’s /.well-known/ path. /.well-known/ is a web standard for a public information directory, letting AI Agents discover what services and capabilities your site offers via fixed paths.

The files you need to host:

  • /.well-known/api-catalog - API catalog
  • /.well-known/openid-configuration - authentication info
  • /.well-known/oauth-protected-resource - protected resource info
  • /.well-known/mcp/server-card.json - MCP server info
  • /.well-known/agent-skills/index.json - Agent Skills index

You might be thinking, “I’m just a static blog — I don’t have an API or OAuth, why do I need any of this?” Fair. These items were originally designed for sites with backend services, and realistically no AI Agent is going to actually call your OAuth endpoint on a static blog. But the scanner only checks “do these files exist, and is the format correct?” — so if you want a perfect score, just throw them up there.

A heads up if you’re hosting on GitHub Pages: GitHub Pages can’t properly serve files under .well-known, even with .nojekyll in place. So we hand this off to a Cloudflare Worker too.

Worker deployment steps

  1. Go to the Cloudflare Dashboard, on the left sidebar select Workers & Pages
  2. Click Create > Create Worker
  3. Pick any name, e.g. agent-readiness
  4. Click “Deploy” to push the default code first
  5. After deploying, click “Edit Code” and paste your Worker script in
  6. Click “Deploy” in the top right

After deploying, you also need to set up routes so Cloudflare knows which requests should go through this Worker:

  1. Go back to your domain’s management page
  2. On the left, choose Workers Routes
  3. Add a new route, set the route to yourdomain.com/*, and pick the Worker you just created
  4. Save

You can also turn off the Worker’s workers.dev public URL — you only need it bound through the route, no need for a separate public entry point.

Note
You might be wondering if the free Cloudflare Worker plan’s 100,000 daily requests is enough. For a typical tech blog with a few hundred to a few thousand daily views, you’ll use less than 1% of the free quota. Don’t sweat it.

Final result: a perfect 100

After fixing everything, I rescanned, and:

  • Discoverability: 100 (3/3)
  • Content: 100 (1/1)
  • Bot Access Control: 100 (2/2)
  • API, Auth, MCP & Skill Discovery: 100 (6/6)
  • Total: 100, Level 5 Agent-Native 🎉

Quick recap of how each item was solved:

Item How it was fixed
robots.txt Already there
Sitemap Already there
AI bot rules Already there
Content Signals Added a Content-Signal line to robots.txt
Link headers Cloudflare Response Header Transform Rule
Markdown Negotiation Cloudflare Worker
WebMCP A snippet of JavaScript in the HTML
.well-known (5 files) All handled by Cloudflare Worker

A few extra notes

  1. GitHub Pages doesn’t support the .well-known directory: even with .nojekyll, it doesn’t work. You’ll need a Cloudflare Worker to serve those files
  2. Markdown Negotiation needs a Worker: a static site on its own can’t do content negotiation, you need the Worker to do it

Wrapping up

The whole process really wasn’t that complex — most items were just “add a file” or “add a setting” and call it a day. The bulk of the time was actually spent stepping on landmines and debugging.

That said, honestly, some of these items (looking at you, OAuth) don’t have any real meaning for a pure blog — they’re just there for the points. But the Discoverability and Content items? Those are genuinely worth doing. The AI Agent ecosystem is only going to get more mature, and getting this foundation in place early means your content has a much better shot at being correctly accessed and cited by AI down the line — so yeah, worth it.
