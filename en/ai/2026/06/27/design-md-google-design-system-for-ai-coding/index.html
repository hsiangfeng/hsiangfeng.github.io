design.md: If Your AI-Generated UI Keeps Drifting, You're Missing Google's design.md

#Claude Code #AI #Cursor #design.md #Google Stitch #Stitch #Design System #Tailwind

Note
design.md is a format spec released by Google Labs that splits a design system into two layers — “YAML design Tokens” and “Markdown design rationale” — so that AI Coding tools like Claude Code and Cursor see more than just hex codes and font sizes; they actually understand the design intent behind them. The built-in CLI handles lint (including WCAG contrast checks), diff, and export to Tailwind or W3C DTCG. It’s currently in alpha.

Intro

Have you ever done some Vibe Coding and noticed the AI’s layout is always a little off, and no matter how many times you say “the spacing is wrong here, the colour is inconsistent there”, it still produces a slightly broken screen? If yes, it’s because you haven’t given the AI the rules of your design system properly.

To solve exactly this, this post walks through design.md, an AI design system spec released by Google Labs. The goal is to let AI Coding tools like Claude Code, Cursor, and GitHub Copilot actually read your design system and produce consistent UI, so you don’t have to keep repeating “wrong colour, wrong font size” over and over.

Note
If you’re not familiar yet with rules files for AI like CLAUDE.md / AGENTS.md, take a look at CLAUDE.md and Rules for AI: Writing Tips That Make AI Understand Your Project and Agent Skills: More Token-Efficient Than MCP first — it’ll make a lot more sense why design.md exists.

Where design.md came from: incubated from Stitch

As always, let’s do a quick history detour first.

design.md actually comes out of Stitch, Google’s AI design tool (first shown at Google I/O 2025). Stitch is a UI design tool powered by Google Gemini — you give it a text Prompt or upload a sketch, and it quickly generates a UI screen (frontend code included).

So when it launched, it was honestly pretty wild.

Then somewhere around 2026, Google released the design.md format itself under Apache 2.0 on GitHub (google-labs-code/design.md).

What does that mean? It basically means they’re trying to “standardise” Stitch’s design system rules.

After all, the most common pain point for Vibe Coding developers is inconsistent UI and inconsistent style. But if you open-source this design spec, the business value gets pretty huge — whether you use Claude Code, Cursor, GitHub Copilot, or any other AI Coding tool, you can use this spec to produce consistent UI.

Wait, you’re wondering what business value open-sourcing brings?

A few examples:

  • Becoming the industry standard means owning the definition. Think of it like OpenAPI or Kubernetes back in the day — when they were first open-sourced they looked harmless, but once everyone started using them, Google’s voice became the loudest about where the spec should go, what fields to add, what Token types to support. Because they’re the one in charge.
  • Other AI tools voluntarily become allies. Once Claude Code, Cursor, and Copilot start marketing “we support design.md” as a selling point, those competitor tools are basically promoting Google’s format for free. Google doesn’t even need to spend marketing budget — super cheap.
  • The training data flywheel. Once it’s open-sourced, people will put their own project’s design.md in public GitHub repos. Google can then use those files to train “AI that generates UI” — structured design Tokens plus human-written design rationale is honestly the perfect combo for AI training data. Future Gemini gets to use this public data to become stronger and get a better excuse to charge more per Token.
  • Stitch’s free marketing funnel. Once developers get used to design.md, when they need a “design tool that speaks design.md”, the first thing they think of is Stitch — because Stitch is the source.

So when big companies open-source things and it looks like they’re losing money, they’re often trading time for ecosystem. Chromium, Kubernetes, TensorFlow all played this game — on the surface it’s a community contribution, but in reality it’s using “free” to lock in the standard early.

What problem is design.md actually solving?

So… what problem does design.md actually solve? Basically just this:

Making sure the UI that AI generates uses the same colours, font sizes, and corner radii every time.

This is really, really important. I’m sure everyone Vibe Coding has run into this — you ask the AI to make a Button, and then…

  • The first time it gives you #3B82F6
  • The second time #2563EB
  • The third time straight up bg-blue-500

The screen looks fine, all blue. But they’re not actually the same blue, so your project ends up with a whole watercolour palette of “blues”. Font sizes have the same issue — sometimes 1.25rem, sometimes 20px, sometimes text-xl.

This kind of inconsistency makes the UI feel disjointed, which makes the user experience inconsistent, which hurts the brand.

So why does this happen? The main reason is you haven’t given the AI a “design system rules” document.

Someone might say: “Can’t I just put it in CLAUDE.md?”

You can. But you have to know that CLAUDE.md (and AGENTS.md) are “natural-language rules” for the AI — there’s no structured colour palette or Token system. After reading, the AI still has to guess “which hex is Primary” or “which corner radius should this button use”, and it ends up guessing in the end anyway.

So design.md is basically turning that “design system rules” document into a file that machines can parse precisely + humans can understand the reasoning behind.

So what does design.md actually look like?

We’ve kept saying “machines can parse precisely + humans can understand the reasoning behind”, so what does this file actually look like? Pretty simple — design.md is just a Markdown file.

The structure splits into two parts:

  • YAML Front Matter: the precise design Tokens (colours, typography, spacing, corner radius, components).
  • Markdown body: the design rationale — why this colour, when to use it, when not to use it.

By splitting the file this way, the AI doesn’t just know “Primary is #1A1C1E“ — it also knows “Primary is the deep ink used for headlines and core text”. So when it produces UI, it’s a lot less likely to use the wrong colour for a button or the wrong size for a piece of text.

A few key points:

  • colors, typography, rounded, spacing, components are the five main Token categories.
  • Tokens can reference each other — for example, button-primary‘s backgroundColor is written as "{colors.tertiary}", so when you change the tertiary colour later, everything using it updates automatically.
  • Colours support any valid CSS colour string: hex, rgb, hsl, oklch — all good.

If you’ve used Tailwind’s tailwind.config.js or the W3C Design Tokens Format Module (DTCG), this’ll feel really familiar.

Markdown body block

After the closing --- of the YAML, you get the design reasoning written for humans (and AI) to read.

Here’s the officially recommended section order:

  • Overview
  • Colors
  • Typography
  • Layout
  • Elevation & Depth
  • Shapes
  • Components
  • Do’s and Don’ts

Only Colors (and the primary colour inside it) is required. The other sections are optional — write them only if your design system has that piece.

Wait — there’s a CLI!?

Yep, design.md also ships with some really handy tools. Mostly they do four things:

  • lint
  • diff
  • export
  • spec

So let’s go through what each tool does.

lint: validate the DESIGN.md structure

I’m pretty sure lint will feel familiar to any developer — it’s the same concept as ESLint, Stylelint, Prettier.

In short, lint checks the following:

  • Whether the YAML schema is correct
  • Whether Token references are broken (e.g. you wrote {colors.tertiery}, missing an a?!)
  • Whether there’s a primary colour
  • Whether WCAG AA contrast passes (I think this one’s the killer feature)
  • Whether there are orphan Tokens that are defined but never used
  • Whether the section order matches the recommendation

For the WCAG AA rule, it’ll straight up tell you “on-secondary against secondary only hits X.X:1 contrast, doesn’t pass AA”. So accessibility checks are baked right into the design system file.

diff: compare two versions

This one is honestly pretty fun — just like git diff, you can compare two versions of DESIGN.md. Usage is simple:

It’ll tell you which Tokens changed, which were deleted, which were added.

The main use case: when you’re doing a version bump on your design system, swapping primary from #1A1C1E to #0A0A0A, and you want to know “how many components in the project are going to change colour because of this” — diff lists every affected Token so you can quickly see what needs updating.

export: convert to other formats

The export tool is genuinely impressive in my opinion, especially if you want to treat design.md as the “upstream” source and convert it into Tailwind / CSS / DTCG formats that the frontend actually uses.

spec: hand the spec to the AI

Last one is spec, also dead simple to use:

This command outputs design.md’s own format spec.

After that, you can hand the spec to your AI Coding tool of choice (Claude Code, Cursor, Copilot) and tell it “please follow this spec when reading my DESIGN.md”. The AI then knows how to parse the YAML schema, that Token references use {path.to.token} syntax, what’s required, what’s optional.

Add --rules and it’ll output the linting rules as well.

So how do you get a DESIGN.md in the first place?

In case you’re wondering where DESIGN.md actually comes from, there are basically two ways:

  1. Write one by hand, following the YAML + Markdown structure above.
  2. Pick a style you like from VoltAgent/awesome-design-md, drop its DESIGN.md straight into your project root, then tell the AI:

Please generate UI following the spec in DESIGN.md

That’s it. Pretty simple, right?

How do you use it with AI Coding tools?

Now that we’ve covered design.md’s structure and CLI tools, let’s talk about how to use it with AI Coding tools.

Since design.md’s whole design goal is “turn the UI design system into something that can be continuously understood by AI”, the usage is pretty simple:

  1. Put DESIGN.md in your project root.
  2. In CLAUDE.md / AGENTS.md / .cursor/rules/*.mdc, add a line: “For UI-related tasks, please read DESIGN.md first and strictly follow the Tokens and design rationale inside.”

Yep, that’s it. Super easy.

One small reminder at the end, though, and it’s kind of an old chestnut at this point:

Don’t stuff design.md with irrelevant junk.

The whole design philosophy is to give the AI a “distilled” version of your design knowledge — same direction as Agent Skills that I shared before. Give the AI the key points, not everything.

Note
Right now, design.md doesn’t ship with official integration examples for Claude Code / Cursor / Copilot — only the CLI and TypeScript API. So you need to guide the AI to read design.md via your CLAUDE.md / AGENTS.md / rules file.

Wrap-up

Let me wrap this up.

So what exactly is design.md?

  • A design system document for AI Coding tools, in a “YAML Token + Markdown design rationale” format.
  • The pain point it solves is AI-generated UI having inconsistent colours, font sizes, and spacing.
  • The built-in CLI can lint (with WCAG contrast checks), diff, export to Tailwind / DTCG, and output a spec for the AI.

Hope this post helps you fix the “AI generates UI with inconsistent colours, font sizes, and spacing” pain so your AI Coding workflow goes a lot more smoothly.
