Is Claude Code Auto Memory Hogging Your Memory? A Cleanup & Diagnostics Guide

#Claude Code #AI #Auto Memory #Claude Code Memory #Context Window #cleanupPeriodDays #AI Development

Note
Auto Memory writes your feedback into ~/.claude/projects/<project>/memory/. It doesn’t use RAM, but it does eat Context Window and disk space. The good news is the memory files are usually just KB in size — the real reason ~/.claude/projects/ balloons to hundreds of MB or even GB is the conversation jsonl files sitting in the same folder. Claude Code uses cleanupPeriodDays (30 days by default) to auto-clean old session files, but if you just ran a massive session recently, waiting 30 days is a bit much. You can use find to fish out the big files instead.

Intro

A friend pinged me out of the blue the other day: “Hey Ray, my disk just lost 300–500 MB out of nowhere, and every time I launch Claude my Context drops way faster than usual. Any ideas?”

I had a pretty good guess. They’d been hammering Claude Code hard recently, so my first instinct was to check ~/.claude/projects/. Sure enough, that folder alone had eaten 300+ MB — and that’s tied to Claude Code’s Auto Memory mechanism. So this post is about that.

I also wrote a previous post — What Is Claude Code Auto Memory? A Complete Guide to the Auto Memory Feature — covering the basics and how Auto Memory works. This one picks up from there to talk about what Auto Memory actually “occupies,” what’s really eating your disk and Context Window, and how to deal with it.

What does Auto Memory actually occupy?

Quick answer first — Auto Memory occupies two things, and neither of them is RAM:

Resource How much Why
Context Window First 200 lines or 25KB of MEMORY.md Auto-loaded at the start of every conversation
Disk space Usually just KB-level Memory content is written as Markdown files in your Home directory

Note
Heads up — when I say “memory” in this post I don’t mean RAM. I mean Context Window and disk space.

So how do you find this MEMORY.md? It’s just the index file inside your project’s memory/ folder, at ~/.claude/projects/<project>/memory/MEMORY.md. It lists all your memory files (Markdown files), one entry per line. Claude uses that index to find and read the corresponding memory content.

Context Window: how does MEMORY.md eat tokens?

Auto Memory content lives as Markdown files on disk. Every time you start a new conversation, Claude Code auto-loads the content of MEMORY.md into the Context Window so Claude can reference those memories during the conversation.

But MEMORY.md doesn’t get loaded in its entirety — there’s a limit:

Auto Memory only loads the first 200 lines or first 25KB (whichever comes first) of MEMORY.md into the Context Window.

In other words, if your MEMORY.md is over 200 lines or 25KB, the part past that threshold doesn’t get loaded into Context at all. This is also why some people swear Claude was supposed to remember something — they clearly told it, Claude even confirmed it remembered — but it has no recollection later. It just never saw that memory entry.

Here’s what an actual MEMORY.md looks like:

Each line is just a one-sentence index pointing at the actual memory file. With this kind of setup, even if your memory grows to dozens of entries, only these index lines get loaded into Context Window at startup. The actual detail behind each entry only gets read when Claude needs it in the conversation.

To make it concrete, the feedback_uiux_skill_for_ui_work.md mentioned above expands into something like this:

You’ll notice the name, description, type, originSessionId fields — the frontmatter format looks a lot like a Skill’s SKILL.md. You can think of Auto Memory as another flavor of Skill: Claude uses this metadata to figure out what kind of memory this is, which session it came from, and what it contains. That’s why the description field is critical.

If you ever find Claude can’t seem to recall a specific memory you thought it had, it’s probably because that file’s description isn’t clear enough — Claude can’t tell what the entry is about, so it doesn’t read it.

How to slim down MEMORY.md

If your MEMORY.md is approaching 200 lines, here are a few ways to trim it:

  1. Split details into Topic files: Keep MEMORY.md as just the index — one line per entry with a link, and put the actual content in separate .md files. Claude reads them on demand.
  2. Delete outdated memories: Just tell Claude “forget the previous rule about X” in the conversation, or manually delete the corresponding file under ~/.claude/projects/<project>/memory/.
  3. Edit manually: Memory files are regular Markdown — open them in your editor and clean up.

Note
Most of the time you’d ask the AI to organize this for you rather than hand-editing — the whole point of Auto Memory is letting the AI manage these memories. Doing it manually kind of defeats the purpose. That said, understanding how the structure and loading mechanism work is still useful when something goes wrong and you need to debug it.

Disk space: the real culprit isn’t Auto Memory

Now the disk side — this is where most people misunderstand Auto Memory.

Let’s first see how much Auto Memory itself actually takes up. Using one of my own projects as an example, the memory folder looks like this:

The entire memory/ folder adds up to 52KB. At this scale, you’ll basically never need to clean it.

Now what about the whole projects/ folder?

Surprised? 339 MB! So who’s actually taking up all that space?

The answer is conversation jsonl — Claude Code saves every single conversation as a .jsonl file, where each line is a JSON object containing your prompt, Claude’s response, tool calls, tool results — everything written down. For long conversations or sessions with heavy file reads/writes, a single file can easily hit MB-level.

See for yourself:

The biggest single conversation file is already up to 8 MB.

So next time you notice ~/.claude/projects/ growing, don’t blame Auto Memory — run du and you’ll see the actual culprit is these .jsonl files.

So how do I auto-clean these? Meet cleanupPeriodDays

OK, good news — there’s a way to handle this. Claude Code has a built-in auto-cleanup mechanism that deletes session files older than 30 days when it starts up. The setting is called cleanupPeriodDays.

Note
cleanupPeriodDays only cleans conversation logs — it doesn’t touch Auto Memory’s memory files. Whatever memories you’ve written stay put. No need to worry.

Cleaning out large files manually

Once you delete a session’s jsonl, that session can’t be brought back via claude --resume. If you still want to keep recent conversations, ls -lh first and check the modification times — only delete the ones you’re sure you won’t go back to.

While you’re at it, check which project folders are heaviest

Each folder under ~/.claude/projects/ corresponds to a directory you’ve opened with Claude Code. If a project is long since abandoned, there’s no point keeping its folder around. To see which folders are heaviest:

For any project you’re sure you won’t resume anymore, just delete the whole folder.

Anything else in that folder?

Besides conversation jsonl and memory/, each project folder may also contain a few UUID-named subfolders (no .jsonl extension — just folders). Those are scratch spaces used by subagents during task execution. You generally don’t need to touch them — cleanupPeriodDays will clean those up too.

If you’ve been running subagents with Worktree, the old worktrees also get auto-cleaned at startup, controlled by the same cleanupPeriodDays.

Hope this gave you a clearer picture of what Auto Memory actually “occupies.”

Next time your disk is running tight, du first to see what’s actually eating your space — don’t go blaming Auto Memory.
