Claude Code Routines: How to Set Up /schedule (and Why It Pushes to claude/ Instead of main)

Note Claude Code has three ways to schedule work: Routines (cloud-based, runs even with your computer off), Desktop scheduled tasks (local, app needs to be open), and /loop (session-level, terminal needs to stay open). The /schedule CLI is the entry point for Routines — minimum 1-hour interval, and by default it only pushes to claude/-prefixed branches. To push to main, flip "Allow unrestricted branch pushes" on the web side.

Intro

I've been using Claude Code's /schedule for a while now. It builds a scheduled task that fires at whatever time I set, and after the run I usually have it push the result to GitHub. The thing is, I kept hitting this one situation — open GitHub the next day and the new branch is claude/main-xxxx instead of the main I asked for.

So this post is me writing up the scheduling feature in Claude Code, while also pinning down the claude/main-xxxx thing that's been bugging me.

Oh, and if you've already read Claude Cowork Dispatch and Remote Control (article in Mandarin), think of this post as a parallel branch — Dispatch and Remote Control let you control Claude right now from your phone, while scheduling lets it work for you automatically while you're not around.

What are Routines?

In short, a Routine is a prompt plus the repo it should work in (and a few other settings), parked on Anthropic's cloud. It fires on whatever schedule or trigger you set up, and runs your task on its own.

Two main things separate it from a regular Claude Code session:

  1. It runs on Anthropic's cloud. Not on your computer — so it still fires when you're asleep, when you've gone out, or when your laptop's off.
  2. It runs autonomously. No permission prompts (no Yes/No popups), it just follows your prompt all the way through. Which means the prompt has to be really clear — otherwise it might do something you didn't expect.

Note If you want to squeeze the most out of your monthly subscription, Routines are actually pretty useful for recurring tasks — checking for new PRs to review daily, summarizing the week's commit log on Fridays, opening a monthly PR to update the stats chart in your README, that kind of thing.

Creating a schedule with /schedule

There are two ways to create a Routine:

  1. Web: claude.ai/code/routines
  2. CLI: /schedule

Either works — they both edit the same record, so creating or editing on one side syncs to the other instantly.

Basic usage:

bash
/schedule

Just type /schedule with nothing after it, and Claude walks you through it conversationally — asking what the prompt should be, which repo, how often it should run, etc.

If you already know exactly what you want, you can write it inline:

bash
/schedule daily PR review at 9am

Or for a one-off reminder:

bash
/schedule in 2 weeks, open a cleanup PR that removes the feature flag

Claude parses the time you wrote, converts it to a concrete timestamp, confirms with you, and saves it.

Note The /schedule CLI only creates Schedule-triggered Routines. For API or GitHub event triggers, you need to edit the routine on claude.ai/code/routines.

The CLI also has commands for managing existing routines:

CommandPurpose
/schedule listList all your routines
/schedule updateEdit a routine (including setting cron)
/schedule runTrigger right now, don't wait for the next slot

Note If you're really lazy, you can also just tell the AI to use /schedule to set the schedule for you...

Why does it push to claude/xxx when I asked for main?

OK now the part that confused me — the "I asked for main but it pushed to claude/xxx" thing.

When a Routine runs a prompt that pushes code, Claude can only push to claude/-prefixed branches by default. So even if your prompt says "push directly to main when done", Claude opens something like claude/main or claude/update-main-xxxx and pushes there. Your actual main stays untouched.

This isn't a bug — it's a feature (really). The reasoning is to "stop routines from accidentally messing up important branches". Scheduled tasks run autonomously, nobody's watching, and there are no permission prompts. If they could write straight to main, one badly written prompt could blow up a production project.

So how do you handle it? Two options:

  1. Live with the default. Let it push to claude/xxx, then open a PR for review. This is the recommended workflow — since nobody's watching when the routine runs, having changes land on a separate branch first and merging when you have time is just a lot safer.
  2. Turn on "Allow unrestricted branch pushes". If you really want it to push to main directly, you can flip this in the Permissions section when editing the routine. No more branch prefix restriction — but you'd better make sure the prompt is rock-solid and won't do anything you don't want.

If you really do want it to push straight to a specific branch, here's how:

  1. Go to claude.ai/code/routines and click into the routine
  2. Hit the pencil icon to enter Edit routine
  3. Find the Permissions section at the bottom
  4. Check "Allow unrestricted branch pushes" for that repo
  5. Save

Where to find Allow unrestricted branch pushes

Note If your main is protected with a branch protection rule, GitHub will still block the push even with "Allow unrestricted branch pushes" turned on. At that point you either add an exception to the protection rule, or just go through the PR flow — which is probably more practical anyway.

Where it runs and what it can access

Besides the wrong-branch thing, there are a couple of Routine details that are easy to miss the first time you set one up.

Every run is a fresh clone of the default branch

Every time a Routine fires, it clones a fresh copy of the default branch from your chosen repo — it doesn't carry over a working tree from the last run. So you can't write a schedule that "continues yesterday's half-finished work". Every run starts from scratch.

If your task needs to reference past state, write that state into a file in the repo, or hook up to a third-party service.

One-off schedules don't count against the daily cap

Routines have a daily run cap (the max number of runs per day), but one-off schedules don't count against that cap — they just draw from your subscription's general usage. So if you want to use scheduling as a "remind me in two weeks" tool, you can stack a bunch of these without burning through the daily cap.

Managing routines

Two ways to manage existing routines.

On the CLI side there's /schedule list, /schedule update, and /schedule run for listing, editing, and force-triggering.

On the web side, go to claude.ai/code/routines, click into a routine's detail page, and you get more:

  • Run now — fire it immediately without waiting for the next slot
  • Repeats toggle — pause or resume the schedule; pausing doesn't delete the routine itself
  • Edit — name, prompt, repo, environment, Connectors, trigger type
  • Delete — removes the routine; past run sessions stay in your session list

Speaking of which — every time a routine runs, a new session shows up in your session list. You can click in, see what Claude did, review changes, open a PR, or just keep the conversation going from where the routine left off.

Wrapping up

Claude Code's scheduling feature isn't all that complicated once you get it — the main things are: know the differences between the three scheduling types, learn how /schedule works, and remember the claude/-prefixed branch default. After that it's just tuning trigger types and environment settings to match your needs.

My own habit at this point — daily "run this small thing every day" goes into Routines as a daily schedule, batch tasks that need local file access become Desktop scheduled tasks, and anything I want to poll repeatedly inside a current session uses /loop. They each have their own slot.

As for the "always pushes to claude/xxx" thing — once I knew it was intentional design, I made my peace with it. Letting it push to claude/ and reviewing through a PR is honestly more reassuring anyway. If you've been hitting the same confusion, hopefully this saved you a trip into the docs.
