What Are Claude Code Hooks? Auto-Run Lint, Block rm -rf, Inject Context on SessionStart — A Complete Guide

Note Claude Code Hooks are "automatic trigger points" in Claude Code's lifecycle — they fire your Shell command at the moment you specify. The common ones are PreToolUse (before a tool runs), PostToolUse (after a tool runs), UserPromptSubmit (every time you send a prompt), and SessionStart (when a session opens). You just write the config into the hooks block of settings.json and you're set.

Intro

After using Claude Code for a while, you've probably hit a situation like this: every time it edits a file, you have to remind it to run Prettier and ESLint again (sure, you could just run npm run format yourself, but humans are lazy). Or you wanted to go fast so you flipped on --dangerously-skip-permissions, but now you're nervous it might fire off something like rm -rf. Or your Context got Compacted and Claude completely forgot some important project conventions you'd already explained, so you have to explain them again — ugh.

So this post is about a really important Claude Code technique: Hooks. They let you "hook into" Claude Code's lifecycle and trigger automated actions, making sure it does the right thing at the right moment.

I've also written Which Claude Code Files Should You Commit? CLAUDE.md, Settings, Rules, Skills — A Complete Breakdown which covers the layering concept of .claude/settings.json. This post is a follow-up that pulls the hooks feature buried in settings out into its own deep dive.

What's the difference between Hooks, Skills, CLAUDE.md, and Rules?

Before getting into how to set up hooks, let's quickly compare it against the other "teach Claude what to do" mechanisms, so it's clearer what each one is actually for.

MechanismWho runs itWhen it triggersWhat it's for
CLAUDE.mdClaudeLoaded on every conversationProject conventions, naming, workflows
RulesClaudeLoaded on startup OR when relevant paths are touchedModular rules that can fire by path
SkillClaudeClaude decides on its own when to useBundling an entire workflow into a reusable capability
HookYour ShellClaude Code lifecycle eventsForced automated actions

If the table didn't quite click, think of it this way:

  1. CLAUDE.md and Rules are "rules"
  2. Skill is a "capability" — these three all rely on Claude reading them and choosing to follow
  3. Hook is different — hooks are run by Claude Code itself, Claude has no choice, the moment the trigger fires the command will run

Here's a concrete example. Say in CLAUDE.md you write "run Prettier after every file edit". Most of the time Claude listens, but occasionally it forgets (Context might've blown up?!). If you use a PostToolUse hook to set "run Prettier automatically after every Edit or Write", then regardless of whether Claude feels like it, your Shell command is guaranteed to run.

So hooks fit the scenario of...

Things I absolutely cannot tolerate Claude missing

Like lint, format, security guardrails. The fuzzier stuff that needs Claude's judgment is still better off in CLAUDE.md.

What does the hook config file look like?

All hook config goes into the hooks block of settings.json. The structure looks like this:

json
{
  "hooks": {
    "EventName": [
      {
        "matcher": "filter condition",
        "hooks": [
          {
            "type": "command",
            "command": "the Shell command you want to run"
          }
        ]
      }
    ]
  }
}

The outer key is the "event name" (like PreToolUse or PostToolUse), wrapping an array where each entry is a config group containing matcher (filters when this group activates) and hooks (the actual commands to run).

If you've read Claude Code Status Line Setup Guide (article in Mandarin), this structure should feel familiar — like Status Line, it's "Claude Code hands data to your Shell, and you decide how to respond".

Where does the config file go?

Like other Claude Code settings, hooks also have layers:

LocationScopeCommit it?
~/.claude/settings.jsonAll your projectsWon't end up in the project repo
.claude/settings.jsonThe whole teamYes, commit it
.claude/settings.local.jsonJust you personallyDon't commit (already in .gitignore)

In short: hooks the whole team should have (like lint) go in .claude/settings.json, personal preferences (like desktop notifications) go in ~/.claude/settings.json, and sensitive-path local rules go in .claude/settings.local.json.

What events are available?

Now we get to the main event of this post: "what events can we actually hook into?"

Honestly, Claude Code offers more than 20 events. Listing them all would scare people off, so I'll group them by "how often I actually use them in real work".

Here's a diagram of a typical Claude Code session lifecycle, with the events that fire on each arrow — you can match it up with where each hook fits:

Diagram showing when Claude Code Hooks events fire

The most commonly used ones

EventWhen it firesClassic use case
PreToolUseBefore a tool runsBlock dangerous commands, rewrite parameters
PostToolUseAfter a tool succeedsAuto-format, run lint
UserPromptSubmitWhen you press Enter to send a promptInject extra context, filter sensitive info
SessionStartWhen a session opens or resumesLoad project context, output git status
StopWhen Claude finishes responding and is about to stopVerify the task was really completed
NotificationWhen Claude needs your attentionDesktop notifications, voice prompts

These six are the most frequently used — most real-world examples revolve around them.

Other ones that are rarer but worth knowing

EventWhen it firesClassic use case
PermissionRequestWhen a permission dialog is about to appearAuto-approve specific actions
PermissionDeniedWhen auto mode blocks a tool callLog blocked commands
PostToolUseFailureWhen a tool execution failsNotify on failure or auto-retry
SubagentStopWhen a subagent finishes a taskVerify subagent results
PreCompact / PostCompactBefore / after context compactionBack up the conversation, reinject rules after compaction
SessionEndWhen the session closesClean up temp files, write logs
ConfigChangeWhen a settings file is modified during the sessionAudit config changes
CwdChanged / FileChangedWhen the working directory or a watched file changesReload environment variables (pair with direnv)

At this point, you should see that hooks pretty much cover every node Claude Code passes through. Almost anything you can imagine in the shape of "I want X to run automatically when Y happens" is doable — the concept is the same as the "lifecycle events" you'd find in a framework.

How do you write a matcher?

Time to talk about how to actually write the hook condition.

You saw the hook example earlier looks like this:

json
{
  "hooks": {
    "EventName": [
      {
        "matcher": "filter condition",
        "hooks": [
          {
            "type": "command",
            "command": "the Shell command you want to run"
          }
        ]
      }
    ]
  }
}

matcher decides "under what conditions this hook group activates", and what it compares against depends on the event. For the commonly used PreToolUse / PostToolUse, it compares against the "tool name", so the most intuitive way to write it is to just drop the tool name into the matcher field.

Here's a concrete example — this hook is "before Claude runs Bash, show a reminder in the terminal":

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -n '{ systemMessage: \"Bash is about to run\" }'"
          }
        ]
      }
    ]
  }
}

After this, you'll see the Bash is about to run reminder pop up:

Screenshot showing the 'Bash is about to run' reminder

The value of matcher is the tool name Bash, meaning "this hook only fires when Claude uses the Bash tool".

Note Why use jq here instead of echo? Because Claude Code talks to hooks in "JSON format". If you just write echo "Bash is about to run", what Claude Code receives is unstructured plain text — it doesn't know what you want it to do, so it just treats it as an execution log entry and Claude never sees it. But jq -n '{ systemMessage: "Bash is about to run" }' produces a JSON output, which is basically telling Claude Code "this field is called systemMessage, please show its contents to Claude". When Claude Code sees that recognized format, it actually feeds the message in.

If you want to watch multiple tools at once, separate them with |. For example, this hook is "run Prettier automatically every time Claude uses Edit or Write to modify a file":

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

The command looks a bit complicated, so let me break it down:

  1. When Claude finishes editing a file with Edit / Write, Claude Code sends a JSON payload to this hook that looks roughly like { "tool_input": {"file_path": "/path/to/file.ts" }, ... }, telling you which file it just changed.
  2. Then jq -r '.tool_input.file_path' parses that JSON and extracts the value of tool_input.file_path (the file path). The -r means "give me raw text, no quotes".
  3. | (pipe) — think of it like a relay baton: "take the output of the previous command and feed it as input to the next one".
  4. xargs takes that incoming text and "appends it as an argument to the next command", so what actually runs is npx prettier --write /path/to/file.ts.
  5. npx prettier --write calls Prettier to format that file.

Noteprettier is a code formatter. The --write flag means "actually modify the file in place". If you'd rather just see the diff without writing, swap it for --check. If your project uses ESLint instead of Prettier, replace npx prettier --write with npx eslint --fix.

Basically, in later examples, whenever you see jq -r '.xxx', it's doing this same thing:

jq -r parses the JSON that Claude Code sent in, pulls out the field you want, and hands it off to the next command. The jq -n from earlier is the reverse — it "generates" a JSON payload from scratch to feed back to Claude Code.

Finally, if you want to "trigger no matter what tool runs", just leave matcher as an empty string:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_name' >> ~/.claude/all-tools.log"
          }
        ]
      }
    ]
  }
}

This hook appends the tool name to a log file before any tool is used by Claude — basically logging every tool call across the whole session.

This covers 95% of cases. If you want to look up what each tool is called, see my earlier post What do Claude Code's built-in tools actually do? Understanding how it picks tools — it lists every tool by name.

Advanced: use regex to match MCP tools

The remaining 5% is when you want to match tools provided by an MCP server. MCP tools are named like mcp__<server>__<tool> — for example, the search tool from GitHub's MCP server is called mcp__github__search_repositories.

If you want to watch every tool from a given MCP server at once, use the regex .* to match anything:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "mcp__github__.*",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'GitHub MCP tool called' >&2"
          }
        ]
      }
    ]
  }
}

.* is regex for "any character", so mcp__github__.* means "starts with mcp__github__, followed by anything". Claude Code's rule for when to use regex is simple: if the matcher is made of anything other than letters, digits, _, and |, it runs as a JavaScript regex; otherwise it's compared as a literal string. So earlier writings like Bash or Edit|Write won't accidentally be treated as regex.

Not all events match on tool name

Everything above was tool-related events, but some events don't match on tool name — they match on some other "category of the event itself".

The most common example is SessionStart, which matches on "how the session was started". This hook is "when a session rebuilds after Context is Compacted, remind Claude of the project rules":

json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Reminder: use pnpm for package management, not npm'"
          }
        ]
      }
    ]
  }
}

When you see echo 'Reminder: use pnpm...', you might be thinking...

"Wait, didn't you just say echo doesn't work with PreToolUse? How come it works here?"

That's because SessionStart and UserPromptSubmit are two special events — whatever the hook outputs to stdout gets automatically picked up by Claude Code as "extra context to inject for Claude", so echo works and Claude sees it. Other events (like the PreToolUse we saw earlier) have to go through jq -n '{systemMessage: ...}' JSON format to talk to Claude Code.

matcher: "compact" restricts this to "only fire when the session rebuilds after Compact" — opening Claude Code for the first time or resuming an old session won't trigger it. SessionStart's valid values are startup (first open), resume (resuming an old session), clear (cleared context), and compact (after compaction).

Notification follows similar logic — it matches on "notification type". The common ones are permission_prompt (Claude is waiting for your approval) and idle_prompt (Claude has nothing to do and is waiting for the next message); rarer ones include auth_success, elicitation_dialog, elicitation_complete, and elicitation_response. For what each event's matcher corresponds to, check the official hooks reference — I won't list them all here.

Advanced: use if to filter on tool arguments too

matcher alone isn't precise enough because it only compares tool names. If you want to say "only trigger the hook when Claude runs git push, ignore other Bash commands", you need the more granular if field:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "if": "Bash(git push *)",
            "command": "echo 'Blocked: please go through a PR, do not push to main directly' >&2; exit 2"
          }
        ]
      }
    ]
  }
}

What this example does: when Claude wants to run any Bash command starting with git push, the hook outputs a blocking message and uses exit 2 to block the action. But if Claude wants to run git status or npm test, the hook won't even fire.

The difference between the two filter fields:

  • matcher: "Bash" is the first-level coarse filter: "only look at the Bash tool".
  • if: "Bash(git push *)" is the second-level fine filter — it also matches the actual command Bash is about to run. * is the "anything after this" wildcard, using the same syntax as the permissions settings.

The biggest upside is "the hook subprocess only spins up when the condition is actually matched". Compared to firing the hook every time and then using jq to parse arguments to decide what to do, the performance is way better.

Note The if field was added later, so older versions of Claude Code might not recognize it and just ignore it. If you wrote if but the hook behaves as if it isn't there, run claude --version first to check your version and update.

How does a hook talk to Claude Code?

Now that matcher makes sense, let's look at how a hook actually "exchanges messages" with Claude Code. Here's the smallest possible example:

bash
#!/bin/bash

# 1. Capture the JSON Claude Code sent into the INPUT variable
INPUT=$(cat)

# 2. Output a message back to Claude Code — this doesn't show up in your terminal,
#    Claude Code picks it up as the hook's response
echo "Hello Claude"

# 3. Exit the script. Returning 0 means "all good, continue".
#    If you write `exit 2`, that means "something's wrong, block this action".
exit 0

To give you a PreToolUse example: when Claude wants to run npm test, your hook receives a JSON payload like this on stdin:

json
{
  "session_id": "abc123",
  "cwd": "/Users/ray/myproject",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm test"
  }
}

Note A lot of people writing their first hook trip up here: they assume exit 1 will block. But only exit 2 blocks — any other non-zero exit code doesn't.

Using JSON output for finer control

Are exit 0 and exit 2 your only options? Not quite — you can do more granular things (like auto-approving, or directly rewriting the command Claude is about to run). The trick is exit 0 plus outputting a JSON payload to stdout:

json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Please use pnpm, not npm"
  }
}

permissionDecision has four values for PreToolUse:

  • allow: skip the permission prompt and allow directly
  • deny: block the action and feed the reason back to Claude
  • ask: fall back to the normal permission flow and prompt you
  • defer: the hook has no opinion, go back to the normal permission flow (same effect as not writing this field)

Note Don't mix exit 2 and JSON output. When Claude Code sees exit 2, it reads stderr as the reason and ignores any JSON on stdout. Pick one approach: either exit 2 with stderr, or exit 0 with JSON.

Five hook types

All the examples so far used type: "command", which is the most common form. But Claude Code has four others:

TypeHow it runsWhat it's for
commandRun a Shell commandMost situations — flexible and simple
httpPOST event data to a URLHooking into a remote service for shared auditing or org-wide validation
mcp_toolCall a tool on a connected MCP serverReusing existing MCP tools as hook logic
promptHand data to a Claude model to decideSituations needing semantic judgment instead of hard-coded rules
agentDispatch a subagent with tools to verifySituations where you need to actually read files or run commands to decide

90% of people writing hooks stop at command, but the other three (especially prompt) are pretty worth knowing about — they upgrade what hooks can do from "dumb rules" to "gatekeepers with judgment".

Here's a prompt hook example. Say you want another model to check whether Claude really finished the requested task after it stops talking:

json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Check whether Claude actually completed every requested task. If not, reply {\"ok\": false, \"reason\": \"what's missing\"}."
          }
        ]
      }
    ]
  }
}

When this hook fires, Claude Code automatically calls a model to do a single-turn yes/no decision. If it replies that the task wasn't finished, the main Claude keeps going — basically a free quality-control checkpoint.

Real-world examples

Enough concept, time for actual examples. The ones below are the ones I find most useful day-to-day, highest bang-for-buck.

Auto-format modified files

After Claude uses Edit or Write to modify a file, run Prettier automatically to fix the formatting:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

This uses jq to parse the file_path out of the JSON stdin and pipe it to Prettier — all in one line. If your system doesn't have jq, on Mac it's brew install jq, on Ubuntu it's apt install jq.

If your project uses ESLint instead of Prettier, swap npx prettier --write for npx eslint --fix.

This hook is tied to the project's lint/format rules, so put it in .claude/settings.json and commit it for the whole team.

Block dangerous commands like rm -rf

It's easier to maintain hooks as standalone Shell scripts. Start by creating .claude/hooks/block-rm.sh:

bash
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# Only block actual rm -rf / or rm -r /. Other paths (like /tmp/foo) are allowed through
if echo "$COMMAND" | grep -qE 'rm[[:space:]]+-rf?[[:space:]]+/([[:space:]]|$)'; then
  echo "Blocked: rm -rf from root is not allowed" >&2
  exit 2
fi

exit 0

I'm using the POSIX character class [[:space:]] instead of \s here because \s is a GNU grep extension — macOS's default BSD grep doesn't necessarily support it, which would silently break the match. [[:space:]] is the POSIX standard syntax and works in every grep on Mac and Linux.

The (/([[:space:]]|$)) part forces / to be followed by whitespace or end-of-line, so:

  • rm -rf /: blocked
  • rm -r /: blocked
  • rm -rf /tmp/foo: allowed, because what follows / isn't whitespace or end-of-line

This pattern is intentionally conservative — it only blocks literal rm -rf / and rm -r /. If you want to go harder (like blocking rm -rf /*, rm -rf ~, rm -rf $HOME and other variants), follow the same model and chain more grep -qE checks, each one with its own exit 2.

Don't forget to make it executable:

bash
chmod +x .claude/hooks/block-rm.sh

Then register it in .claude/settings.json:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/block-rm.sh"
          }
        ]
      }
    ]
  }
}

$CLAUDE_PROJECT_DIR is a built-in Claude Code environment variable — it expands to the project root, safer than hardcoding an absolute path.

Note This hook fires even under --dangerously-skip-permissions. PreToolUse runs before the permission mode check, so hooks are one of the few mechanisms that can actually block actions in "bypass permissions" mode.

This kind of safety guardrail goes in .claude/settings.json committed to the repo so the whole team is protected. If you only want personal insurance, putting it in ~/.claude/settings.json is fine too.

Reinject project context after Compact

Once Context gets Compacted, Claude often forgets important conventions. You can use SessionStart with the compact matcher to inject a reminder every time after compaction:

json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Reminder: use pnpm for package management, not npm; always run pnpm test before commit.'"
          }
        ]
      }
    ]
  }
}

What SessionStart and UserPromptSubmit hooks output to stdout gets added as extra context for Claude — that's what makes these two events different from the others.

If you want to inject dynamic content (like the current git branch or uncommitted files), swap the echo for your own script:

bash
#!/bin/bash
BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null)
DIRTY=$(git status --porcelain | wc -l)
echo "Current branch: $BRANCH, uncommitted files: $DIRTY"

This hook is best placed in .claude/settings.json, because "use pnpm" and "run pnpm test before commit" are internal project conventions — committing it means everyone who clones the repo picks it up automatically, no one has to set it up themselves.

Mac desktop notification

Get a notification when Claude is waiting for your permission or your next prompt, so you can confidently go do other things:

json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code is waiting\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

On Linux, swap osascript for notify-send; on Windows PowerShell, use MessageBox.Show — same idea.

Note On Mac the first run might not show anything, because osascript sends notifications through the built-in Script Editor app. You need to go to System Settings → Notifications, find Script Editor, and enable its notification permission. If Script Editor isn't even showing up in the list, run osascript -e 'display notification "test"' once in Terminal first — that will make it appear in the notification settings list.

Desktop notifications are purely personal preference — put this in ~/.claude/settings.json, no need to commit.

Send every Claude action to Discord

If you don't want to babysit the terminal, or you want to share what Claude is up to with the team via a Discord channel, you can use a PreToolUse hook to call a Discord webhook.

First, generate a webhook URL in your Discord channel settings, then export it in your shell profile (don't hardcode it in the script — committing it to a repo basically leaks it):

bash
export DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/your-webhook-id/your-token"

Next, create ~/.claude/hooks/discord-notify.js:

javascript
#!/usr/bin/env node
const WEBHOOK_URL = process.env.DISCORD_WEBHOOK_URL;

async function main() {
  // Read the JSON Claude Code sent in via stdin
  const chunks = [];
  for await (const chunk of process.stdin) chunks.push(chunk);
  const input = JSON.parse(Buffer.concat(chunks).toString());

  const toolName = input.tool_name || "unknown tool";
  const toolInput = input.tool_input || {};

  // Build a Discord embed so the message looks nicer
  const embed = {
    color: 0x7289da,
    author: { name: `🔧 ${toolName}` },
    description: toolInput.file_path
      ? `\`${toolInput.file_path}\``
      : toolInput.command
        ? `\`\`\`bash\n${toolInput.command}\n\`\`\``
        : "",
    timestamp: new Date().toISOString(),
  };

  await fetch(WEBHOOK_URL, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ embeds: [embed] }),
  });
}

main().catch(console.error);

Finally, register it in ~/.claude/settings.json (using personal settings so it works across all projects):

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "node ~/.claude/hooks/discord-notify.js"
          }
        ]
      }
    ]
  }
}

matcher: "" means "notify on any tool", so no matter what Claude is about to do — Read, Edit, Bash — you'll get a Discord message. If you want different emoji, colors, or fields per tool type, extend along these lines.

This whole hook is for personal notifications and the webhook URL is yours alone — put it in ~/.claude/settings.json at the personal layer, no need to commit.

Note The webhook URL should always be read from an environment variable, never hardcoded in the script. If you really want to share one webhook with the team, the hook itself can live in .claude/settings.json, but the webhook URL should still only live in each person's shell profile or .claude/settings.local.json (which doesn't get committed).

Log every Bash command

If you want to audit which Bash commands Claude has actually run, add a one-liner to PostToolUse:

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' >> ~/.claude/bash.log"
          }
        ]
      }
    ]
  }
}

Every Bash command Claude runs gets appended to ~/.claude/bash.log, easy to review later. This is personal auditing — put it in ~/.claude/settings.json.

How do multiple hooks combine?

When you set multiple hooks on the same event, they all run in parallel — they don't block each other, and Claude Code combines all the results at the end. For PreToolUse, the permission decision follows the "strictest wins" rule: deny beats ask, ask beats allow.

A concrete example — you can set both "log every Bash command" and "block rm -rf" at the same time:

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r .tool_input.command >> ~/.claude/bash.log"
          },
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/block-rm.sh"
          }
        ]
      }
    ]
  }
}

When Claude tries to run rm -rf /tmp/build, both hooks fire at the same time — the log gets written, the rm gets blocked. The log doesn't miss the entry just because rm was blocked, because the two hooks run independently.

How do you confirm a hook actually ran?

You've set up a hook but aren't sure if it's even firing? Claude Code gives you two ways to check.

The first is the /hooks command — type /hooks inside Claude Code to bring up a list of currently configured hooks, grouped by event, each one drillable to see its matcher and command. /hooks is read-only — to actually edit a hook, you still need to edit settings.json.

Screenshot of the /hooks command interface

The second is the debug log. When you launch, add --debug-file /tmp/claude.log:

bash
claude --debug-file /tmp/claude.log

Then open another window and run tail -f /tmp/claude.log — every hook trigger will show its full stdout, stderr, and exit code in the log. If you only want hook-related output without other debug noise drowning it out, use claude --debug hooks with a filter.
