Vibe Coding's 2026 Upgrade: A Six-Stage Agentic Engineering SOP with Templates and a Real Case

#Claude Code #AI #Vibe Coding #Agentic Engineering #Sub-agent #AGENTS.md #Spec-Driven Development #Plan Mode #AI Development Workflow

Intro

If you’ve used Claude Code, Cursor, Codex, or any of these AI coding tools, you’ve probably hit the same wall: it feels magical when you start, but once you try to build a real project at scale, the gremlins show up — context overflows and AI forgets everything, you fix A and break B, AI overwrites good code while editing, multiple files get touched at once and you can’t roll any of it back cleanly.

None of these are tool problems. They’re workflow problems — you haven’t built a discipline for collaborating with AI.

Worth mentioning here: Andrej Karpathy coined the term “Vibe Coding” in February 2025 (original tweet). Then in April 2026, at Sequoia Ascent 2026, he upgraded the concept himself — the talk is titled “Software 3.0, Agentic Engineering, and Jagged Intelligence”, and the key line is:

“Vibe coding raises the floor. Agentic engineering is about extrapolating the ceiling.”

Translated into plain English: vibe coding raises the floor, so people who’ve never written code can still build things. Agentic engineering raises the ceiling — it’s how professional engineers should actually work.

So what’s the difference between an agentic engineer and a vibe coder? Based on the talk’s core message, the work isn’t about describing a need, letting AI write it, and shipping. Instead, AI plays the role of implementer and you play supervisor — designing specs, supervising plans, reviewing diffs, writing tests, building evaluation loops, managing permissions, isolating worktrees, maintaining quality. An agentic engineer doesn’t blindly accept AI-generated code — they wrap engineering discipline around AI speed.

I’ve previously written AI Vibe Coding: A Look at Today’s Mainstream BDD and SDD and Want to Start Vibe Coding? Fundamentals Beginners Should Pick Up First — the first one’s more methodology, the second’s more beginner basics. But both still sit inside the vibe coding frame. This post pushes that one level up, using agentic engineering discipline to lay out the full hands-on workflow. Every section ships with templates you can lift directly, and at the end we’ll walk a personal e-commerce MVP (Nuxt 3 + Supabase + Stripe) through the whole thing.

Note
Why insist on discipline? Per CodeRabbit’s December 2025 “State of AI vs Human Code Generation” report (analyzing 470 GitHub open-source PRs — 320 AI co-authored + 150 human-only), AI-written code has 1.7x more major bugs than human-written code and 1.57x more security issues overall. Within that, XSS vulnerabilities are 2.74x and insecure password handling 1.88x. AI generates fast but quality isn’t guaranteed — what makes it safe is the workflow you design to catch its mistakes.

Vibe Coding vs Agentic Engineering: Which one for 2026?

Before getting into the workflow, here’s a table laying out the differences:

Dimension Vibe Coding Agentic Engineering
When introduced Feb 2025, Karpathy Apr 2026, Karpathy’s upgrade
Core spirit Describe by gut feel, AI writes, minimal review Design spec, supervise plan, check diffs, write tests, build evals
When to use Demos, PoCs, throwaway projects Production services, long-term maintainable projects
Workflow discipline Loose, improvised Structured, has an SOP
Human role A user describing ideas An engineer supervising AI employees
Failure mode Works, but can blow up anytime Slower but more controlled

You can see from the table — these aren’t replacements for each other, they’re for different situations. Building a weekend toy? Vibe coding is fine. Building something other people will use, something you need to maintain? Time to switch to agentic engineering.

The SOP that follows is written entirely from the agentic engineering discipline.

1. Planning phase: Spec → Plan → Tasks, three-layer documentation

The first principle of agentic engineering is the plan should be documented. Mainstream frameworks line up here — GitHub’s 2025 Spec Kit (Specify → Plan → Tasks → Implement, each stage producing spec.md / plan.md / tasks.md files), Anthropic’s Claude Code best practices (Explore → Plan → Implement → Commit, with plan mode letting you press Ctrl+G to open the plan in your editor directly). The core idea is identical — plans and specs should be readable, editable files, not things that only live in a chat.

Specifically, three layers:

  1. Spec: what this project is, who it’s for, where the boundaries are
  2. Plan: which features to build, by when
  3. Tasks: how the current task breaks down, where you are right now

Each layer in its own file. Each solves a different context problem.

1.1 Spec: write a CLAUDE.md (with AGENTS.md) as your project constitution

Before you start, one important thing: different AI coding tools read different config files.

  • Claude Code only reads CLAUDE.md, not AGENTS.md (Anthropic’s official docs explicitly say “Claude Code reads CLAUDE.md, not AGENTS.md“)
  • Other tools (Cursor, Codex, Windsurf, Aider, etc.) read AGENTS.md

What is AGENTS.md?

AGENTS.md is an open standard OpenAI launched in August 2025, jointly developed with Amp, Google Jules, Cursor, Factory, and other AI tool vendors. In December 2025 OpenAI donated it to the Linux Foundation’s Agentic AI Foundation (AAIF), joining MCP (donated by Anthropic) and Goose (donated by Block) as the three founding projects of AAIF. As of May 2026, over 60,000 open-source projects use it, with support in 20+ AI coding tools — Amp, Codex, Cursor, Devin, Factory, Gemini CLI, GitHub Copilot, Jules, VS Code, and more.

How to set it up?

The most practical approach is both files, with no content duplication:

  • AGENTS.md holds the project conventions themselves (shared across tools)
  • CLAUDE.md starts with @AGENTS.md to import AGENTS content, then adds Claude-Code-specific rules + @ imports for ROADMAP / tasks

Or even simpler: ln -s AGENTS.md CLAUDE.md creates a symlink so one file serves both names (on Windows you need Administrator or Developer Mode — if neither is set, @AGENTS.md import is safer).

If you only use Claude Code and don’t care about cross-tool, just write CLAUDE.md. No need for AGENTS.md.

@import mechanism (this is the Claude Code key)

Notice the @<path> syntax in the ## Read before starting section of the template above — this is Anthropic’s CLAUDE.md extension (confirmed in official docs). When the AI starts up and hits @<path>, it pulls that file’s content into context too. It recurses up to 5 hops deep, and supports both relative and absolute paths (e.g. @~/.claude/personal-rules.md).

This mechanism matters because Claude Code only auto-loads CLAUDE.md itself (plus CLAUDE.md from ancestor paths) on startup — anything under docs/ like ROADMAP.md or tasks.md doesn’t get auto-read. If you write ROADMAP without @-referencing it from CLAUDE.md, the AI starting fresh in a new session can’t see your plan at all. You’d be writing for nothing.

Note
@<path> is Anthropic’s CLAUDE.md extension. The AGENTS.md spec itself doesn’t define an import syntax, so other tools’ AGENTS.md implementations may not support @import. So in cross-tool scenarios, it’s safer to inline important rules directly into AGENTS.md — don’t assume other tools will resolve @import. Claude Code still works as usual: pull the full AGENTS.md content via @AGENTS.md, then hook in plans via @docs/ROADMAP.md.

AI tools have built-in todo lists (Claude Code’s todo system, for example), but those live in context — once you run /compact, close the session and reopen, or context overflows, the list disappears with it. 2. Division of labor: your three types of AI employees

Karpathy emphasized the agentic in agentic engineering for a reason — in 2026, you’re no longer the engineer writing code by hand. You’re the supervisor managing a team of AI employees. Claude Code’s 2025-2026 evolution gave us three employee configurations, each solving a different problem.

2.1 Skill: standard workflow injected into the main agent

A Skill is a markdown instruction pack stored at .claude/skills/<name>/SKILL.md. When Claude detects a task matching the Skill’s description, it auto-loads it into the main context. Skills share context and share the model with the main agent — good for repeatable standard moves.

2.2 Sub-agent: dedicated employee with isolated context and isolated permissions

The biggest difference between a Sub-agent and a Skill: a sub-agent has its own context window, its own system prompt, its own tool permissions, and can even run a different model. After the main Claude agent delegates a task to a sub-agent, the sub-agent works in its own context and returns only a summary — the main agent’s context stays clean.

2.3 Agent Teams: advanced mode with parallel sessions

Agent Teams is a 2026 advanced mode — multiple Claude instances running in isolated worktrees, coordinating via git. Each agent claims its own task, merges its own work, resolves its own conflicts. Practical experience puts the sweet spot at 3-5 parallel agents — above 10, costs go up but the marginal benefit drops.

This mode is too complex for beginners. Master Skill + Sub-agent first, then consider Agent Teams when one person + one main Claude can’t keep up with the project.

3. Development discipline: three rules to stick to every day

Once you’re in implementation, three rules to enforce every day. Each one is a net catching a specific kind of AI mistake.

3.1 Add tests in the same turn as the code change

This rule comes up the most when reminding AI: every time you change code, add the tests in the same turn — don’t punt to “I’ll batch them up in a separate task later”.

Why insist? Because AI development has this very easy trap: fixing A and breaking B. You tell it to adjust some composable’s logic, the code that comes back looks fine on the surface, but it’s actually side-effected somewhere else and your eye can’t catch it. Tests are the net that catches that side effect.

So why insist on the same turn? Because tests added later tend to become tests that exist just to make code pass — not tests that actually verify behavior. When you write them in the same turn, the AI’s context for the implementation and the tests is connected — the requirement, the logic, and the cases all stay alive. If you punt to a later task, the context has drifted by then, and the cases AI writes will only catch the surface — not the real edge cases.

3.2 UI changes always go to a dedicated Skill or Sub-agent

Any time the change involves UI/UX (layout, component styling, Tailwind class adjustments), call the dedicated UI employee — don’t let the main agent freestyle it.

Why split it?

Specialization, basically. The main agent is strong at architecture decisions and writing logic. But once UI/UX comes in, it tends to commit the usual sins: slapping !important on everything to override existing styles, color choices that show zero design system awareness, spacing like mt-5 mb-3 with no logic.

UI-dedicated employees usually have design system knowledge, color conventions, and accessibility principles baked in. They know filled inputs don’t need borders, focus rings go through tokens, touch targets are at least 44px — all the basics.

3.3 One logical change = one commit, don’t stack them up

The third rule is commit discipline. Without explicit rules, AI tends to dump ten file changes into a single commit. Later when you need to revert a portion of it, everything’s wrapped together and there’s no clean way to separate them.

So spell it out: one logical change = one commit. For example:

  • Add tests for component A → one commit
  • Fix router bug in component B → another commit
  • Refactor shared logic C → another commit

That way reverting any one piece later doesn’t disturb the others. Combined with the rollback strategy from section 5, the cost of recovery stays low.

4. Wrap-up validation: three safety nets catching AI mistakes

Writing the code isn’t the same as finishing. Wrap-up has three safety nets, each catching a different kind of mistake.

4.1 First net: typecheck + test + build, three validations

Every time a task wraps up and gets delivered, have AI run three validations:

  1. pnpm typecheck (or tsc --noEmit) — confirms types pass
  2. pnpm test — confirms existing tests still pass
  3. pnpm build — confirms a clean production bundle

These three catch different layers:

  • typecheck catches type errors — e.g. AI passing a User object to a function that takes string
  • test catches behavior errors — e.g. AI changed a function and a case that should return true now returns false
  • build catches errors that only show after bundling — e.g. an import path that works in dev mode but blows up in production build

Only running one of these isn’t enough. All three together is a reasonably complete safety net.

Pre-delivery checklist template

4.2 Second net: Writer / Reviewer dual-Claude mode

This is Anthropic’s officially recommended advanced workflow in best practices. Simply put: open two Claude sessions, one writes and one reviews. Why? Claude reviewing code it just wrote suffers from confirmation bias — it can’t catch its own problems easily. A fresh-context Claude catches more edge cases.

How to do it:

Session A (Writer) Session B (Reviewer)
Implement the rate limiter middleware for A component
Review @src/middleware/rateLimiter.ts. Find edge cases, race conditions, inconsistencies with the existing middleware pattern
Apply the review feedback: [Session B output]

A more advanced approach: turn the Reviewer into a sub-agent (like the security-reviewer from section 2.2). Then every time you finish writing, you can fire off a review with one call — no need to manually open a new session.

4.3 Third net: cross-check AI’s cited versions, packages, APIs against the official docs

The third safety net is anti-stale-memory: every time AI cites a package version, an API usage, a cloud console step — anything that updates frequently — open the official docs and verify before letting that memory ship.

Why so paranoid? Because AI’s training data has a cutoff date. Package versions, API specs, cloud console UIs change fast — what AI answers from memory is often several months out of date. It might write API usage for some package that’s been renamed in the new major version. It might tell you to click X in some cloud console, but the console UI’s been redesigned.

5. Mistake recovery: rollback decision table

In practice, AI is going to mess up code at some point. The most wasteful response when this happens is to keep piling on fixes in the wrong direction — the hole just gets deeper. So you need an explicit rollback judgment rule.

In 2026 Claude Code added a feature called /rewind (or Esc + Esc) — rolls you back to any checkpoint and restores the code state Claude changed. That’s the first layer of recovery.

Note
Important caveat — Anthropic’s docs explicitly say checkpoints only track files Claude changed, not file changes from Bash commands (e.g. node_modules touched by npm install, things deleted by rm, build artifacts). For those, you need git. So git commit discipline can’t be skipped.

The rule in one sentence: was the broken part an execution detail, or the planning direction?

Rollback decision table

Situation Action
AI’s just-finished change doesn’t pass review /rewind (or Esc + Esc) to last checkpoint
Single file’s implementation detail wrong, direction OK git restore <file> and redo
Multiple related files broken, direction OK git restore multiple files, redo
AI applied wrong architecture / misread the requirement Scrap the plan and rewrite from the planning phase
Tests added don’t cover the implementation intent Scrap tests, re-examine the plan, re-add tests
AI changed too much, can’t tell what’s good and what’s bad First git status to confirm + git stash as insurance, then git restore . to roll everything back (this is why small commits matter)
typecheck / test failing at large scale High chance the direction’s wrong, stop and check the plan

You can see from the table — rollback isn’t failure, it’s part of the workflow. Treat it as a tool for catching AI mistakes and your whole development rhythm gets much steadier.

6. Memory system: two-layer architecture + positive pattern templates

The final piece is memory system design. Once this is in place, all the discipline above becomes things AI does on its own — you don’t have to remind it every time.

Memory has two layers:

6.1 Global layer: ~/.claude/CLAUDE.md or ~/.claude/rules/ for cross-project rules

For rules that should apply across projects — rules you explicitly want Claude to follow every time (package manager conventions, test discipline, no over-engineering, no scope creep) — Anthropic’s docs define two locations:

  • ~/.claude/CLAUDE.md — single file, auto-loaded every session. Good for 5-10 core principles.
  • ~/.claude/rules/<topic>.md — split into per-topic files (one rule per file), also auto-loaded every session. The advantage: you can use YAML frontmatter paths: for path-scoped rules (e.g. a rule that only loads when working in src/api/**)

New projects inherit them automatically — no need to re-teach.

Note
Worth clarifying a common confusion — Claude Code also has a feature called Auto memory, which Claude accumulates on its own (not what you write by hand), stored at ~/.claude/projects/<project>/memory/, scoped per project, not global. The “global rules” in this section refers to specs you write by hand for AI to follow — different thing from Auto memory. For the full mechanism, see What is Claude Code Auto Memory? 6.3 When writing rules, positive patterns beat prohibitions

Last key thing: when writing rules, mix prohibitions with positive patterns. Prohibitions alone aren’t enough.

Type Example Effect
Prohibition Don’t lead with bullet outlines, don’t use emojis, don’t over-engineer AI avoids it, but doesn’t know what “right” looks like
Positive pattern A good plan doc includes goals, task breakdown, what’s out of scope, and acceptance criteria AI knows what to actively produce

You can see from the table — prohibitions only block traps. They can’t actively guide good output. So they need positive patterns alongside.

Where this SOP doesn’t work: limits and failure modes

Time to be honest about limits — this SOP isn’t a silver bullet. A few rounds in, you’ll hit these enemies:

  1. Overkill for small projects: weekend demo, one-shot script, PoC to test the waters — maintaining the full CLAUDE.md + AGENTS.md + ROADMAP + tasks.md + sub-agent configuration is heavy. Vibe coding’s fine here — the “when to use” column in the earlier table already says so.
  2. Token costs aren’t cheap: Writer/Reviewer means running Claude twice per review. Sub-agents are fresh-context every call. With Opus heavy usage, monthly bills hit three-digit USD easily. A leaner config: heavy tasks (plan, implement) on Opus, lighter tasks (Reviewer, test-writer, ui-engineer sub-agents) on Sonnet or Haiku.
  3. Context decay is this SOP’s biggest enemy: past 100K+ tokens of conversation, AI starts dropping rules, forgetting CLAUDE.md discipline, and forgetting to sync tasks.md. Build the habit of running /compact regularly and opening new sessions when needed — don’t let one conversation run too long.
  4. Sub-agent routing isn’t reliably stable: Claude’s auto-detection of “should I invoke ui-engineer for this task” isn’t 100% accurate. Over time the main agent picks up Tailwind work itself. The reliable approach is to name the sub-agent explicitly in the prompt — don’t rely entirely on auto-trigger.
  5. More memory isn’t always better: past 20-30 entries, AI won’t follow every single one. Important rules and minor rules get diluted when mixed. Prune regularly — drop outdated or low-priority entries instead of always adding new ones.

These limits aren’t reasons not to use the SOP. They’re context for setting realistic expectations. The SOP raises stability — it doesn’t guarantee stability. What’s actually stable is the underlying Git, tests, typecheck, build, and human review. AI just makes you faster.

Wrapping up: make this SOP your own

Recap — what this post gives you is a complete agentic engineering workflow from spec to ship:

  1. Planning phase: CLAUDE.md (with AGENTS.md) as project constitution + ROADMAP.md (feature plan) + tasks.md (progress sync), tied together with @<path> import
  2. Division of labor: Skill injected into main context, Sub-agent with isolated context, Agent Teams running in parallel
  3. Development discipline: tests in the same turn + UI delegated to specialists + one logical change per commit
  4. Wrap-up validation: typecheck + test + build three validations + Writer/Reviewer dual-Claude + AI citation cross-check
  5. Mistake recovery: /rewind → git restore → scrap plan and rewrite, three layers of recovery
  6. Memory system: global + project two-layer architecture + positive pattern templates

There’s a common spirit running through all six sections: every single rule is a discipline against AI amnesia, direction drift, and side effects. Karpathy’s core point about agentic engineering is exactly this — you’re no longer the engineer writing code by hand, you’re the engineer supervising a team of AI employees. Your work is designing specs, supervising plans, reviewing diffs, writing tests, building evals, managing permissions.

More importantly, once you start running this workflow, you’ll gradually hit traps unique to your project. When you do, don’t just solve them in the moment — write them into memory or CLAUDE.md / AGENTS.md to make them next time’s rule. Same trap never gets hit twice. Every trap caught becomes another step of solidified workflow. Eventually you end up with an agentic engineering SOP that’s genuinely yours.

Hope this helps you build your own agentic engineering workflow. If you run a few rounds and your workflow diverges from mine, that’s totally normal — every team has different habits. Pick the parts that work for you, try them out, and slowly tune your own version.
