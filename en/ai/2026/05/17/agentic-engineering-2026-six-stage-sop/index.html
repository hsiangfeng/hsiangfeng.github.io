Vibe Coding's 2026 Upgrade: A Six-Stage Agentic Engineering SOP with Templates and a Real Case

Intro

If you've used Claude Code, Cursor, Codex, or any of these AI coding tools, you've probably hit the same wall: it feels magical when you start, but once you try to build a real project at scale, the gremlins show up — context overflows and AI forgets everything, you fix A and break B, AI overwrites good code while editing, multiple files get touched at once and you can't roll any of it back cleanly.

None of these are tool problems. They're workflow problems — you haven't built a discipline for collaborating with AI.

Worth mentioning here: Andrej Karpathy coined the term "Vibe Coding" in February 2025 (original tweet). Then in April 2026, at Sequoia Ascent 2026, he upgraded the concept himself — the talk is titled "Software 3.0, Agentic Engineering, and Jagged Intelligence", and the key line is:

"Vibe coding raises the floor. Agentic engineering is about extrapolating the ceiling."

Translated into plain English: vibe coding raises the floor, so people who've never written code can still build things. Agentic engineering raises the ceiling — it's how professional engineers should actually work.

So what's the difference between an agentic engineer and a vibe coder? Based on the talk's core message, the work isn't about describing a need, letting AI write it, and shipping. Instead, AI plays the role of implementer and you play supervisor — designing specs, supervising plans, reviewing diffs, writing tests, building evaluation loops, managing permissions, isolating worktrees, maintaining quality. An agentic engineer doesn't blindly accept AI-generated code — they wrap engineering discipline around AI speed.

I've previously written AI Vibe Coding: A Look at Today's Mainstream BDD and SDD and Want to Start Vibe Coding? Fundamentals Beginners Should Pick Up First — the first one's more methodology, the second's more beginner basics. But both still sit inside the vibe coding frame. This post pushes that one level up, using agentic engineering discipline to lay out the full hands-on workflow. Every section ships with templates you can lift directly, and at the end we'll walk a personal e-commerce MVP (Nuxt 3 + Supabase + Stripe) through the whole thing.

Note

Why insist on discipline? Per CodeRabbit's December 2025 "State of AI vs Human Code Generation" report (analyzing 470 GitHub open-source PRs — 320 AI co-authored + 150 human-only), AI-written code has 1.7x more major bugs than human-written code and 1.57x more security issues overall. Within that, XSS vulnerabilities are 2.74x and insecure password handling 1.88x. AI generates fast but quality isn't guaranteed — what makes it safe is the workflow you design to catch its mistakes.

Vibe Coding vs Agentic Engineering: Which one for 2026?

Before getting into the workflow, here's a table laying out the differences:

DimensionVibe CodingAgentic Engineering
When introducedFeb 2025, KarpathyApr 2026, Karpathy's upgrade
Core spiritDescribe by gut feel, AI writes, minimal reviewDesign spec, supervise plan, check diffs, write tests, build evals
When to useDemos, PoCs, throwaway projectsProduction services, long-term maintainable projects
Workflow disciplineLoose, improvisedStructured, has an SOP
Human roleA user describing ideasAn engineer supervising AI employees
Failure modeWorks, but can blow up anytimeSlower but more controlled

You can see from the table — these aren't replacements for each other, they're for different situations. Building a weekend toy? Vibe coding is fine. Building something other people will use, something you need to maintain? Time to switch to agentic engineering.

The SOP that follows is written entirely from the agentic engineering discipline.

The full workflow at a glance: one feature, requirement to ship

Before getting into details, here's a flowchart that compresses the whole SOP into a feature lifecycle. Three roles — you (the user), Leader (main Claude), and the employees (Sub-agents + Skills). Seven steps total:

Text
····················································································[User]                       [Leader / Main Claude]           [Employees / Sub-agents + Skills]
   │                              │                                  │
   │ ① Submit a requirement       │                                  │
   ├─────────────────────────────▶│                                  │
   │                              │ ② Plan mode breaks down work     │
   │                              │   (calls sub-agents if needed)   │
   │                              ├─────────────────────────────────▶│
   │ ⚠ When requirements unclear, │   Produces docs/tasks.md         │
   │   Leader proactively         │◀─────────────────────────────────│
   │   AskUserQuestion            │                                  │
   │◀─────────⇄───────────────────│                                  │
   │                              │ ③ Build (UI → ui-engineer)       │
   │                              ├─────────────────────────────────▶│
   │                              │   Tests added in the same turn   │
   │                              │◀─────────────────────────────────│
   │                              │ ④ Code Review                    │
   │                              │   (separate session or sub-agent)│
   │                              ├─────────────────────────────────▶│
   │                              │   Fresh context catches bias     │
   │                              │◀─────────────────────────────────│
   │                              │ ⑤ Acceptance + three validations │
   │                              │   typecheck + test + build       │
   │                              ├─────────────────────────────────▶│
   │                              │   Against spec (CLAUDE.md/AGENTS)│
   │                              │◀─────────────────────────────────│
   │ ⑥ Report + update ROADMAP/tasks                                 │
   │◀─────────────────────────────│                                  │
   │ ⑦ You review diff + commit / push                               │
   ├──────────⇄───────────────────│                                  │

Key things to notice:

  • You only step in for steps ①, ⑥, and ⑦ (submit the requirement, read the report, review diffs + commit). Everything between ② and ⑤ is the Leader orchestrating.
  • Ideally Leader should act more like an orchestrator — break down, delegate, collect reports, push implementation to employees. In practice, the main agent often still picks up code itself, so unless you explicitly design the delegation flow (e.g. prompt directly with "please invoke X sub-agent"), don't assume it auto-routes.
  • The ⚠ step is Anthropic's officially recommended interview pattern — when a requirement is fuzzy, Leader uses AskUserQuestion to ask back, so downstream doesn't go the wrong way.
  • ④ Code Review needs a fresh context — done by another Claude session or sub-agent to avoid self-confirmation bias.
  • ⑤ Acceptance tests are based on the spec (CLAUDE.md / AGENTS.md), not whatever Leader made up.

Each step maps to a later section: ① ② in 1. Planning phase, ③ in 2. Division of labor + 3. Development discipline, ④ ⑤ in 4. Wrap-up validation, ⑦ in the commit cadence under 3. Development discipline.

If Leader goes off the rails mid-way, jump to the rollback table in 5. Mistake recovery. After you catch a trap, write it into 6. Memory system to make it a long-term rule.

1. Planning phase: Spec → Plan → Tasks, three-layer documentation

The first principle of agentic engineering is the plan should be documented. Mainstream frameworks line up here — GitHub's 2025 Spec Kit (Specify → Plan → Tasks → Implement, each stage producing spec.md / plan.md / tasks.md files), Anthropic's Claude Code best practices (Explore → Plan → Implement → Commit, with plan mode letting you press Ctrl+G to open the plan in your editor directly). The core idea is identical — plans and specs should be readable, editable files, not things that only live in a chat.

Specifically, three layers:

  1. Spec: what this project is, who it's for, where the boundaries are
  2. Plan: which features to build, by when
  3. Tasks: how the current task breaks down, where you are right now

Each layer in its own file. Each solves a different context problem.

1.1 Spec: write a CLAUDE.md (with AGENTS.md) as your project constitution

Before you start, one important thing: different AI coding tools read different config files.

  • Claude Code only reads CLAUDE.md, not AGENTS.md (Anthropic's official docs explicitly say "Claude Code reads CLAUDE.md, not AGENTS.md")
  • Other tools (Cursor, Codex, Windsurf, Aider, etc.) read AGENTS.md

What is AGENTS.md?

AGENTS.md is an open standard OpenAI launched in August 2025, jointly developed with Amp, Google Jules, Cursor, Factory, and other AI tool vendors. In December 2025 OpenAI donated it to the Linux Foundation's Agentic AI Foundation (AAIF), joining MCP (donated by Anthropic) and Goose (donated by Block) as the three founding projects of AAIF. As of May 2026, over 60,000 open-source projects use it, with support in 20+ AI coding tools — Amp, Codex, Cursor, Devin, Factory, Gemini CLI, GitHub Copilot, Jules, VS Code, and more.

How to set it up?

The most practical approach is both files, with no content duplication:

  • AGENTS.md holds the project conventions themselves (shared across tools)
  • CLAUDE.md starts with @AGENTS.md to import AGENTS content, then adds Claude-Code-specific rules + @ imports for ROADMAP / tasks

Or even simpler: ln -s AGENTS.md CLAUDE.md creates a symlink so one file serves both names (on Windows you need Administrator or Developer Mode — if neither is set, @AGENTS.md import is safer).

If you only use Claude Code and don't care about cross-tool, just write CLAUDE.md. No need for AGENTS.md.

CLAUDE.md template

Copy this into your project's root CLAUDE.md:

markdown
# Project Name - Claude Code Instructions

@AGENTS.md

## Read before starting
@docs/ROADMAP.md
@docs/tasks.md

## Project Overview
(One or two sentences: what this project is, who it's for)

## Build & Test
- Install: pnpm install
- Dev: pnpm dev
- Test: pnpm test (must be all green before commit)
- Build: pnpm build
- Typecheck: pnpm typecheck

## Code Style
- Use ES modules (import/export), not CommonJS
- Naming: composables use useXxx, services use xxxService, pure functions use camelCase
- Indent: 2 spaces
- Quotes: single quotes

## Testing Rules
- Every time you change code, add the tests in the same turn — don't punt to the next task
- Test file naming: xxx.test.ts, in the same folder as the implementation
- Pull out pure functions first to test (easier than mocking)

## Git Workflow
- Commit messages use Conventional Commits (feat / fix / test / refactor / docs / chore)
- One logical change, one commit — don't stack them
- Main branch: main; feature branches: feature/xxx

## Security
- Don't put API keys, tokens, or passwords in code
- .env files don't get committed
- Sensitive files: .env*, *.key, credentials.json

## Explicitly out of scope
- Don't introduce [some kind of feature], reason: [reason]
- Don't use [some technology], reason: [reason]
Note

Line 3 (@AGENTS.md) in the template above is for the dual-file setup. If you only have CLAUDE.md without AGENTS.md, delete that line (or Claude Code will complain it can't find AGENTS.md). Conversely, if you do go with the dual-file setup: AGENTS.md content looks the same as the CLAUDE.md template above, except no @AGENTS.md line (since it's the import target itself). And the ## Read before starting section is better kept only in CLAUDE.md, because @import is Anthropic's CLAUDE.md extension — other tools' AGENTS.md implementations may not support it.

@import mechanism (this is the Claude Code key)

Notice the @<path> syntax in the ## Read before starting section of the template above — this is Anthropic's CLAUDE.md extension (confirmed in official docs). When the AI starts up and hits @<path>, it pulls that file's content into context too. It recurses up to 5 hops deep, and supports both relative and absolute paths (e.g. @~/.claude/personal-rules.md).

This mechanism matters because Claude Code only auto-loads CLAUDE.md itself (plus CLAUDE.md from ancestor paths) on startup — anything under docs/ like ROADMAP.md or tasks.md doesn't get auto-read. If you write ROADMAP without @-referencing it from CLAUDE.md, the AI starting fresh in a new session can't see your plan at all. You'd be writing for nothing.

Note

@<path> is Anthropic's CLAUDE.md extension. The AGENTS.md spec itself doesn't define an import syntax, so other tools' AGENTS.md implementations may not support @import. So in cross-tool scenarios, it's safer to inline important rules directly into AGENTS.md — don't assume other tools will resolve @import. Claude Code still works as usual: pull the full AGENTS.md content via @AGENTS.md, then hook in plans via @docs/ROADMAP.md.

1.2 Plan: write a ROADMAP.md listing features and stages

CLAUDE.md (with AGENTS.md) is the project constitution — doesn't change much. ROADMAP.md is the plan for this sprint or this month's features — gets updated as development progresses.

ROADMAP.md template

Put it in docs/ROADMAP.md:

markdown
# Project Name ROADMAP

## Vision
(One or two sentences: what problem this project solves, who it's for)

## Week 1: MVP
- [ ] Feature A: [brief behavior + acceptance criteria]
- [ ] Feature B: [same]

## Week 2: Reinforcement
- [ ] Feature C: [same]
- [ ] Feature D: [same]

## To evaluate
- Feature X: maybe, maybe not — depends on Week 1 completion
- Feature Y: same

## Explicitly out of scope
- Feature Z: because [reason]

The ## Explicitly out of scope section matters. AI loves making up features and crossing boundaries — writing out what not to do helps keep it in lane.

1.3 Tasks: write a tasks.md to sync current progress

The third layer is current-task progress tracking. AI tools have built-in todo lists (Claude Code's todo system, for example), but those live in context — once you run /compact, close the session and reopen, or context overflows, the list disappears with it. The more durable approach is to also have the AI write progress to an actual file.

tasks.md template

Put it in docs/tasks.md:

markdown
········# Current task: [ Brief description ]

## Todo
- [x] 1. Understand the requirement, plan saved to docs/ROADMAP.md
- [x] 2. Break down subtasks
- [ ] 3. Implement A
  - [ ] 3-1. Write A.composable
  - [ ] 3-2. Write A.composable.test in the same turn
- [ ] 4. Implement B
  - [ ] 4-1. Write B.service
  - [ ] 4-2. Write B.service.test in the same turn
- [ ] 5. Run typecheck + test + build, three validations
- [ ] 6. commit + push

## In progress
3-1: Writing A.composable, stuck on [description]

## Completed log
- 2026-XX-XX: plan confirmed, tasks broken down

Before starting, just tell the AI: list the todos, sync them to docs/tasks.md, update the file every time you finish an item. That way even if you reopen a fresh session tomorrow, AI picks up the project by reading the file.

2. Division of labor: your three types of AI employees

Karpathy emphasized the agentic in agentic engineering for a reason — in 2026, you're no longer the engineer writing code by hand. You're the supervisor managing a team of AI employees. Claude Code's 2025-2026 evolution gave us three employee configurations, each solving a different problem.

2.1 Skill: standard workflow injected into the main agent

A Skill is a markdown instruction pack stored at .claude/skills/<name>/SKILL.md. When Claude detects a task matching the Skill's description, it auto-loads it into the main context. Skills share context and share the model with the main agent — good for repeatable standard moves.

Skill template (API conventions)

.claude/skills/api-conventions/SKILL.md:

markdown
---
name: api-conventions
description: REST API design conventions for this project
---

# API Conventions

- URL paths use kebab-case: /api/user-profiles
- JSON properties use camelCase: userId, createdAt
- List endpoints always support pagination: ?page=1&pageSize=20
- API version in URL path: /v1/, /v2/
- Error response format: { code, message, details? }
Note

If you want the full Skill mechanism, see my previous post: Agent Skills: More Token-Efficient Than MCP — Demoed With Claude Code.

2.2 Sub-agent: dedicated employee with isolated context and isolated permissions

The biggest difference between a Sub-agent and a Skill: a sub-agent has its own context window, its own system prompt, its own tool permissions, and can even run a different model. After the main Claude agent delegates a task to a sub-agent, the sub-agent works in its own context and returns only a summary — the main agent's context stays clean.

Files go in .claude/agents/<name>.md:

Sub-agent template (security-reviewer)

.claude/agents/security-reviewer.md:

markdown
---
name: security-reviewer
description: Run a security review on freshly written code
tools: Read, Grep, Glob, Bash
model: opus
---

You're a senior security engineer. When reviewing code, focus on:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets / credentials in code
- Sensitive data handling

Provide specific line numbers and fix suggestions.

Sub-agent template (test-writer)

.claude/agents/test-writer.md:

markdown
---
name: test-writer
description: Add unit tests for freshly written composables / services
tools: Read, Edit, Bash
model: sonnet
---

You're an AI engineer specialized in writing unit tests.

When you receive a task:
1. Read the target file, understand the function / composable's behavior
2. List three categories: happy path + edge cases + error cases
3. Use Vitest to write .test.ts files
4. Run pnpm test to confirm all green
5. If any case fails, report the failure to the main agent and let it decide

Sub-agent template (ui-engineer)

.claude/agents/ui-engineer.md:

markdown
---
name: ui-engineer
description: Handle Tailwind classes, component styling, design system changes
tools: Read, Edit, Glob
model: sonnet
---

You're a frontend engineer specialized in UI/UX. When changing visuals:

- Filled inputs don't need a border
- Focus rings go through design system tokens, don't hardcode colors
- Touch targets at least 44px (buttons, icon buttons, clickable areas)
- Spacing uses design system scale (mt-2 / mt-4 / mt-6 / mt-8), no random mt-5 mb-3 with no logic
- Colors use design system color tokens, don't hardcode hex
- No !important to override existing styles — find the root cause and fix at the right layer
- RWD: write mobile first, then md:, lg: for larger breakpoints

When you receive a task:
1. Read the target component, confirm existing patterns and token usage
2. Modify Tailwind classes (don't write custom CSS unless really necessary)
3. After changes, check against the reference design / Figma (if any) for visual consistency

When to pick Skill vs Sub-agent?

SituationPick SkillPick Sub-agent
Repeatable standard moves (API conventions, commit rules)✓
Need isolated context to avoid polluting main chat (deep search, code review)✓
Want to run a different model for cost savings (main Sonnet + sub Haiku)✓
Want to restrict tool permissions (reviewer reads but doesn't write)✓
Want AI to auto-detect and use it, no manual trigger✓
Multi-step workflows with decision points✓

2.3 Agent Teams: advanced mode with parallel sessions

Agent Teams is a 2026 advanced mode — multiple Claude instances running in isolated worktrees, coordinating via git. Each agent claims its own task, merges its own work, resolves its own conflicts. Practical experience puts the sweet spot at 3-5 parallel agents — above 10, costs go up but the marginal benefit drops.

This mode is too complex for beginners. Master Skill + Sub-agent first, then consider Agent Teams when one person + one main Claude can't keep up with the project.

3. Development discipline: three rules to stick to every day

Once you're in implementation, three rules to enforce every day. Each one is a net catching a specific kind of AI mistake.

3.1 Add tests in the same turn as the code change

This rule comes up the most when reminding AI: every time you change code, add the tests in the same turn — don't punt to "I'll batch them up in a separate task later".

Why insist? Because AI development has this very easy trap: fixing A and breaking B. You tell it to adjust some composable's logic, the code that comes back looks fine on the surface, but it's actually side-effected somewhere else and your eye can't catch it. Tests are the net that catches that side effect.

So why insist on the same turn? Because tests added later tend to become tests that exist just to make code pass — not tests that actually verify behavior. When you write them in the same turn, the AI's context for the implementation and the tests is connected — the requirement, the logic, and the cases all stay alive. If you punt to a later task, the context has drifted by then, and the cases AI writes will only catch the surface — not the real edge cases.

Test-along prompt template

Before starting / before each new feature, tell the AI:

Text
For this task, every time you change a file, add the matching tests in the same turn.
Add tests as a sibling subtask of the implementation (not nested under the implementation),
so you don't forget after finishing the implementation.
Task isn't done until pnpm test is all green.

3.2 UI changes always go to a dedicated Skill or Sub-agent

Any time the change involves UI/UX (layout, component styling, Tailwind class adjustments), call the dedicated UI employee — don't let the main agent freestyle it.

Why split it?

Specialization, basically. The main agent is strong at architecture decisions and writing logic. But once UI/UX comes in, it tends to commit the usual sins: slapping !important on everything to override existing styles, color choices that show zero design system awareness, spacing like mt-5 mb-3 with no logic.

UI-dedicated employees usually have design system knowledge, color conventions, and accessibility principles baked in. They know filled inputs don't need borders, focus rings go through tokens, touch targets are at least 44px — all the basics.

UI delegation prompt template

Text
This change touches [component name / screen area], goal is [describe the desired result].
Please call the UI specialist (the ui-engineer sub-agent from section 2.2,
or if you've installed a plugin like ui-ux-pro-max, call that).
The main agent only decides which component to touch and the direction —
the actual Tailwind class changes are handed off to the specialist.

3.3 One logical change = one commit, don't stack them up

The third rule is commit discipline. Without explicit rules, AI tends to dump ten file changes into a single commit. Later when you need to revert a portion of it, everything's wrapped together and there's no clean way to separate them.

So spell it out: one logical change = one commit. For example:

  • Add tests for component A → one commit
  • Fix router bug in component B → another commit
  • Refactor shared logic C → another commit

That way reverting any one piece later doesn't disturb the others. Combined with the rollback strategy from section 5, the cost of recovery stays low.

Commit message template (Conventional Commits)

Text
feat: add Feature A
fix: fix router bug in Component B
test: add tests for Composable C
refactor: extract shared logic D
docs: update ROADMAP.md
chore: bump package versions

AI commit cadence prompt template

Tell AI before starting:

Text
Commit cadence rules: one logical change = one commit.
- Don't stack implementation + tests + bug fixes into one commit
- Split tests from implementation (test: and feat: as separate commits)
- Refactor in a standalone commit, don't mix it with other changes
Commit messages follow Conventional Commits
(feat / fix / test / refactor / docs / chore).

4. Wrap-up validation: three safety nets catching AI mistakes

Writing the code isn't the same as finishing. Wrap-up has three safety nets, each catching a different kind of mistake.

4.1 First net: typecheck + test + build, three validations

Every time a task wraps up and gets delivered, have AI run three validations:

  1. pnpm typecheck (or tsc --noEmit) — confirms types pass
  2. pnpm test — confirms existing tests still pass
  3. pnpm build — confirms a clean production bundle

These three catch different layers:

  • typecheck catches type errors — e.g. AI passing a User object to a function that takes string
  • test catches behavior errors — e.g. AI changed a function and a case that should return true now returns false
  • build catches errors that only show after bundling — e.g. an import path that works in dev mode but blows up in production build

Only running one of these isn't enough. All three together is a reasonably complete safety net.

Pre-delivery checklist template

Text
- [ ] pnpm typecheck (or tsc --noEmit) passes
- [ ] pnpm test all green
- [ ] pnpm build succeeds
- [ ] Manually run the actual feature, poke around in the browser
- [ ] git status shows no unexpected file changes
- [ ] Commit message matches Conventional Commits format

4.2 Second net: Writer / Reviewer dual-Claude mode

This is Anthropic's officially recommended advanced workflow in best practices. Simply put: open two Claude sessions, one writes and one reviews. Why? Claude reviewing code it just wrote suffers from confirmation bias — it can't catch its own problems easily. A fresh-context Claude catches more edge cases.

How to do it:

Session A (Writer)Session B (Reviewer)
Implement the rate limiter middleware for A component
Review @src/middleware/rateLimiter.ts. Find edge cases, race conditions, inconsistencies with the existing middleware pattern
Apply the review feedback: [Session B output]

A more advanced approach: turn the Reviewer into a sub-agent (like the security-reviewer from section 2.2). Then every time you finish writing, you can fire off a review with one call — no need to manually open a new session.

4.3 Third net: cross-check AI's cited versions, packages, APIs against the official docs

The third safety net is anti-stale-memory: every time AI cites a package version, an API usage, a cloud console step — anything that updates frequently — open the official docs and verify before letting that memory ship.

Why so paranoid? Because AI's training data has a cutoff date. Package versions, API specs, cloud console UIs change fast — what AI answers from memory is often several months out of date. It might write API usage for some package that's been renamed in the new major version. It might tell you to click X in some cloud console, but the console UI's been redesigned.

AI citation cross-check list

Text
When AI's code references the following, open the official docs and verify:

[Must verify]
- Package API usage (especially major version changes)
- Cloud console UI / configuration steps (AWS, GCP, Cloudflare, Firebase console, etc.)
- Environment variable / config file field names
- Framework / SDK syntax for the version you use (Vue 3 vs Vue 2, Nuxt 3 vs 2, etc.)
- Third-party API specs

[Don't need to verify]
- Pure syntax (JS built-in array methods, string handling, etc.)
- Pure logic (algorithms, pure functions)
- Modules and types inside your own project

5. Mistake recovery: rollback decision table

In practice, AI is going to mess up code at some point. The most wasteful response when this happens is to keep piling on fixes in the wrong direction — the hole just gets deeper. So you need an explicit rollback judgment rule.

In 2026 Claude Code added a feature called /rewind (or Esc + Esc) — rolls you back to any checkpoint and restores the code state Claude changed. That's the first layer of recovery.

Note

Important caveat — Anthropic's docs explicitly say checkpoints only track files Claude changed, not file changes from Bash commands (e.g. node_modules touched by npm install, things deleted by rm, build artifacts). For those, you need git. So git commit discipline can't be skipped.

The rule in one sentence: was the broken part an execution detail, or the planning direction?

Rollback decision table

SituationAction
AI's just-finished change doesn't pass review/rewind (or Esc + Esc) to last checkpoint
Single file's implementation detail wrong, direction OKgit restore <file> and redo
Multiple related files broken, direction OKgit restore multiple files, redo
AI applied wrong architecture / misread the requirementScrap the plan and rewrite from the planning phase
Tests added don't cover the implementation intentScrap tests, re-examine the plan, re-add tests
AI changed too much, can't tell what's good and what's badFirst git status to confirm + git stash as insurance, then git restore . to roll everything back (this is why small commits matter)
typecheck / test failing at large scaleHigh chance the direction's wrong, stop and check the plan

You can see from the table — rollback isn't failure, it's part of the workflow. Treat it as a tool for catching AI mistakes and your whole development rhythm gets much steadier.

6. Memory system: two-layer architecture + positive pattern templates

The final piece is memory system design. Once this is in place, all the discipline above becomes things AI does on its own — you don't have to remind it every time.

Memory has two layers:

6.1 Global layer: ~/.claude/CLAUDE.md or ~/.claude/rules/ for cross-project rules

For rules that should apply across projects — rules you explicitly want Claude to follow every time (package manager conventions, test discipline, no over-engineering, no scope creep) — Anthropic's docs define two locations:

  • ~/.claude/CLAUDE.md — single file, auto-loaded every session. Good for 5-10 core principles.
  • ~/.claude/rules/<topic>.md — split into per-topic files (one rule per file), also auto-loaded every session. The advantage: you can use YAML frontmatter paths: for path-scoped rules (e.g. a rule that only loads when working in src/api/**)

New projects inherit them automatically — no need to re-teach.

Note

Worth clarifying a common confusion — Claude Code also has a feature called Auto memory, which Claude accumulates on its own (not what you write by hand), stored at ~/.claude/projects/<project>/memory/, scoped per project, not global. The "global rules" in this section refers to specs you write by hand for AI to follow — different thing from Auto memory. For the full mechanism, see What is Claude Code Auto Memory? Complete Guide to Auto Memory.

Global rule template (pnpm convention)

~/.claude/rules/pnpm.md:

markdown
---
name: Package manager uses pnpm
description: For JS/TS projects, use pnpm not npm
type: feedback
---

For JS/TS projects, use pnpm by default, not npm.

**Why:** unified lock file convention, disk space savings, monorepo-friendly.

**How to apply:** when you see package.json, default to pnpm install / pnpm run xxx.
If the project has package-lock.json but no pnpm-lock.yaml,
ask the user first whether to migrate to pnpm.

Global rule template (test-along discipline)

~/.claude/rules/testing.md:

markdown
---
name: Add tests in the same turn as the code change
description: Every code change must be paired with tests in the same turn
type: feedback
---

Every time you add or modify code, you must add matching tests in the same work session.

**Why:** Tests added later tend to become "tests that pass for the sake of passing" — they miss edge cases.

**How to apply:**
- When writing a new composable / service / pure function, add *.test.ts in the same task
- When changing an existing file's behavior, extend or add tests
- Pull out pure functions first to test (way easier than mocking)
- Task isn't done until pnpm test is all green
- Add tests as a sibling subtask of the implementation

Global rule template (stay in scope)

~/.claude/rules/minimal-scope.md:

markdown
---
name: Stay strictly in the delivery scope
description: When user says "just X", don't drag in other files
type: feedback
---

When the user clearly says only change X, only change X — don't drag in other files.

**Why:** Scope creep introduces unauthorized changes, costs the user more review time,
and can introduce regressions.

**How to apply:**
- At the start of every task, clearly list the scope
- If you find something out of scope, note it and ask — don't act on it yet

6.2 Project layer: AGENTS.md / CLAUDE.md for project-specific rules

Project layer holds design decisions, product boundaries, and traps specific to this project. The judgment rule: would this rule still apply in a different project? If yes, put it in global. If only in this project, put it in the project layer.

If you want the full Claude Code memory mechanism, see my previous post: What is Claude Code Auto Memory? Complete Guide to Auto Memory.

6.3 When writing rules, positive patterns beat prohibitions

Last key thing: when writing rules, mix prohibitions with positive patterns. Prohibitions alone aren't enough.

TypeExampleEffect
ProhibitionDon't lead with bullet outlines, don't use emojis, don't over-engineerAI avoids it, but doesn't know what "right" looks like
Positive patternA good plan doc includes goals, task breakdown, what's out of scope, and acceptance criteriaAI knows what to actively produce

You can see from the table — prohibitions only block traps. They can't actively guide good output. So they need positive patterns alongside.

Positive pattern template

~/.claude/rules/plan-template.md:

markdown
---
name: Template for a good plan doc
description: The positive structure AI should follow when writing a plan doc
type: feedback
---

When writing a plan doc, include these four sections:

1. **Goal**: one or two sentences on what problem to solve and for whom
2. **Task breakdown**: each subtask listed independently with acceptance criteria
3. **Out of scope**: explicitly excluded (important — AI loves crossing boundaries)
4. **Development discipline notes**: package manager, tests, commit cadence, etc.

Example: see docs/ROADMAP.md.

**Why:** Prohibitions only block traps — positive patterns guide the right output.

**How to apply:** When receiving tasks like writing plans, planning features,
or organizing ROADMAPs, read this structure before writing.

Use both: prohibitions to block traps, positive patterns to guide.

Real-world case study: a personal e-commerce MVP from spec to ship

After six sections of theory, let's walk through one realistic case: building a small personal e-commerce MVP — selling handmade goods, and it has to actually take money.

Stack: Nuxt 3 + Vue 3 + Tailwind + Pinia + Supabase (DB + Auth) + Stripe (payment) + Vercel (hosting). This stack is fairly mainstream for 2026 small e-commerce MVPs — no backend to host, payments handled by Stripe, one-click deploy.

Step 1 — Write CLAUDE.md (with AGENTS.md) as the project constitution

First thing in a new project is writing CLAUDE.md (and AGENTS.md too if your team uses multiple tools — CLAUDE.md's first line is @AGENTS.md to import it). E-commerce has extra emphasis on money / security / RLS compared to a regular tool, so the constitution must spell these out:

markdown
# RaySelf Shop - Claude Code Instructions

@AGENTS.md

## Read before starting
@docs/ROADMAP.md
@docs/tasks.md

## Project Overview
Personal small e-commerce MVP selling handmade goods. Goal: live and taking money within 1 month.

## Tech Stack
- Frontend: Nuxt 3, Vue 3, Tailwind, Pinia
- Backend: Supabase (PostgreSQL + Auth + Storage)
- Payment: Stripe Checkout (Redirect mode)
- Hosting: Vercel
- Package Manager: pnpm

## Build & Test
- Install: pnpm install
- Dev: pnpm dev
- Test: pnpm test
- Typecheck: pnpm typecheck
- Build: pnpm build

## Code Style
- Vue 3 Composition API + script setup
- Composables named useXxx (useCart, useProducts, useCheckout)
- API calls extracted to composables/api/*.ts
- Pure functions (price calc, shipping calc) extracted to utils/
- Server routes in server/api/

## Testing Rules
- Pure functions (price calc, shipping, tax, discount) — 100% coverage
- API composables — at least happy path + 1 error case
- Tests in the same turn — no deferring

## Git Workflow
- Commit messages: Conventional Commits
- One logical change, one commit

## Security (e-commerce core, non-negotiable)
- **Prices always read from server-side Supabase. Never trust prices from the frontend.**
- Stripe secret key only in server routes — frontend can't touch it
- Supabase RLS always on, every table has permission policies
- Webhook signatures must be verified (block forged orders)
- .env never committed; .env.example only holds blank templates

## Explicitly out of scope
- No user signup/login (out of scope for Week 1-3, use Stripe Checkout's email for receipts)
- No product upload admin (use Supabase Studio manually for now)
- No product variants (size / color)
- No refund flow
- No multi-language

Step 2 — Write ROADMAP.md splitting three weeks

docs/ROADMAP.md:

markdown
# RaySelf Shop ROADMAP

## Vision
Let individual creators launch a small money-taking e-commerce at minimum cost.

## Week 1: Product display (public pages)
- [ ] Supabase schema: products / orders / order_items three tables + RLS policies
- [ ] Product list page `/`: fetch products, render cards
- [ ] Product detail page `/products/[slug]`: full text + add to cart
- [ ] Cart store (Pinia + localStorage sync)

## Week 2: Checkout + order writing
- [ ] Server route `/api/checkout` creates a Stripe Checkout Session
- [ ] Checkout integration: cart → Stripe Redirect
- [ ] Server route `/api/stripe-webhook` receives webhook, writes orders to Supabase
- [ ] Order success page `/success`, cancel page `/cancel`

## Week 3: Simple admin
- [ ] Admin order list page (lock with env var or basic auth)
- [ ] Manual order status updates (pending / shipped / completed)

## Explicitly out of scope
- User signup/login: later
- Product upload admin: Week 1-3 uses Supabase Studio manually
- Refund flow: later

Step 3 — Enter plan mode and discuss with AI before starting

Open Claude Code, enter plan mode:

Text
Read CLAUDE.md (it already @imports AGENTS.md and docs/ROADMAP.md) to understand the project context.
I'm starting Week 1 product display. List the file structure, main composables / pure functions,
Supabase schema, and RLS policy draft.

AI will return a plan roughly like:

  • Supabase schema:
    • products: id, slug, name, description, price_cents, image_url, stock
    • orders: id, stripe_session_id, customer_email, total_cents, status, created_at
    • order_items: id, order_id, product_id, quantity, price_cents_at_purchase
  • Code structure:
    • composables/api/useProducts.ts — products fetch
    • composables/useCart.ts — Pinia store + localStorage sync
    • utils/cart.ts — pure functions (subtotal, shipping, total)
    • pages/index.vue, pages/products/[slug].vue
    • Matching test files
  • RLS policy: products public read, orders only server-side service_role can write

Once the plan checks out, have AI write it to docs/tasks.md, then exit plan mode and start implementing.

Step 4 — Implement + add tests in the same turn

Before starting, prompt:

Text
Every time you change a file, add the matching tests in the same turn.
Add tests as a sibling subtask of the implementation.
Update docs/tasks.md every time you finish an item.

AI will:

  1. Write utils/cart.ts (pure functions: calculate subtotal, total) → same turn write utils/cart.test.ts → run pnpm test
  2. Write composables/api/useProducts.ts → same turn write matching tests (mock Supabase client)
  3. Write composables/useCart.ts → same turn write tests
  4. Write pages/index.vue + pages/products/[slug].vue UI → call the ui-engineer sub-agent (template from section 2.2) for Tailwind classes

Step 5 — Three validations + Writer/Reviewer to catch security issues

After implementation, run:

Bash
pnpm typecheck  # types pass
pnpm test       # all green
pnpm build      # bundle succeeds

Then — this is the most critical step in the e-commerce case — open another Claude session as Reviewer, specifically reviewing the security layer:

Text
You're the security-reviewer sub-agent.
Please review this Week 2 checkout flow:
- @server/api/checkout.post.ts
- @composables/useCheckout.ts
- @server/api/stripe-webhook.post.ts

Focus on:
1. Whether prices are read from server-side Supabase (do not trust frontend price)
2. Whether Stripe webhook signature is verified
3. Whether Supabase RLS blocks unauthorized orders write
4. Whether customer email and other sensitive data are handled correctly

Step 6 — Split commits

Follow Conventional Commits, one logical change per commit:

Text
feat: Supabase schema + RLS policy (products / orders / order_items)
feat: product list API composable + pure function cart utils
test: tests for cart utils and useProducts
feat: product list page + product detail page UI
feat: cart Pinia store + localStorage sync
test: tests for useCart
docs: update ROADMAP, Week 1 product display done

Down the road, if any piece of Week 1 needs reverting, every commit is an independent logical change — you can cleanly extract just that one.

Step 7 — Hitting a trap: frontend price tampering security issue

After wiring up Stripe Checkout in Week 2, run Writer/Reviewer. Reviewer Claude catches a critical issue:

server/api/checkout.post.ts directly takes price_cents from the frontend cart to build the Stripe Session — classic client-side trust vulnerability. A user can open DevTools, change localStorage, turn a 10000 product into a 1 product, and pay.

Run it through the rollback decision table:

SituationDecision
Not a single-field wrong value✗ git restore single file doesn't fit
Trust boundary in the architecture is wrong✓ Scrap plan, rewrite Stripe portion

So the right move is scrap the plan and rewrite the Stripe checkout flow. New plan:

  1. /api/checkout only accepts product_id and quantity, not price
  2. Server-side queries each product's price_cents from Supabase
  3. Use server-side computed amount to build the Stripe Checkout Session

After fixing, write this trap into AGENTS.md as a long-term rule:

markdown
## Security
- Prices always read from server-side Supabase. Never trust frontend prices.
- Any endpoint involving money calculations must use database source-of-truth.
- Before adding a new server route involving money, check this rule.

Also promote it to the global layer in ~/.claude/rules/, since this rule applies to any e-commerce, subscription, or money-handling project:

markdown
# ~/.claude/rules/money-server-side.md

---
name: Money always server-side
description: Any money calculation must be based on server-side data, never trust the client
type: feedback
---

For any money-related endpoint (checkout, top-up, subscription, refund, etc.):
client sends only identifier IDs (product_id / plan_id) — server-side queries the price and computes the total itself.

**Why:** Client-side data can be tampered with via DevTools / localStorage.
Any money value coming from the client is untrustworthy.

**How to apply:**
- The endpoint's accept signature must not include price / amount / total fields
- Server-side must query source-of-truth amounts from the DB
- Stripe / payment API money parameters only take server-side computed results

This dramatically lowers the chance of stepping on the same trap in Week 3 admin / future refund flow / future subscription mechanism. That's the full demo of the meta-pattern, trap-to-SOP: caught by review → scrap plan and rewrite → write into AGENTS.md + global ~/.claude/rules/ → dramatically lower the re-trip probability (not 100% immune — AI occasionally still misses rules — but the frequency drops sharply).

Where this SOP doesn't work: limits and failure modes

Time to be honest about limits — this SOP isn't a silver bullet. A few rounds in, you'll hit these enemies:

  1. Overkill for small projects: weekend demo, one-shot script, PoC to test the waters — maintaining the full CLAUDE.md + AGENTS.md + ROADMAP + tasks.md + sub-agent configuration is heavy. Vibe coding's fine here — the "when to use" column in the earlier table already says so.
  2. Token costs aren't cheap: Writer/Reviewer means running Claude twice per review. Sub-agents are fresh-context every call. With Opus heavy usage, monthly bills hit three-digit USD easily. A leaner config: heavy tasks (plan, implement) on Opus, lighter tasks (Reviewer, test-writer, ui-engineer sub-agents) on Sonnet or Haiku.
  3. Context decay is this SOP's biggest enemy: past 100K+ tokens of conversation, AI starts dropping rules, forgetting CLAUDE.md discipline, and forgetting to sync tasks.md. Build the habit of running /compact regularly and opening new sessions when needed — don't let one conversation run too long.
  4. Sub-agent routing isn't reliably stable: Claude's auto-detection of "should I invoke ui-engineer for this task" isn't 100% accurate. Over time the main agent picks up Tailwind work itself. The reliable approach is to name the sub-agent explicitly in the prompt — don't rely entirely on auto-trigger.
  5. More memory isn't always better: past 20-30 entries, AI won't follow every single one. Important rules and minor rules get diluted when mixed. Prune regularly — drop outdated or low-priority entries instead of always adding new ones.

These limits aren't reasons not to use the SOP. They're context for setting realistic expectations. The SOP raises stability — it doesn't guarantee stability. What's actually stable is the underlying Git, tests, typecheck, build, and human review. AI just makes you faster.

Wrapping up: make this SOP your own

Recap — what this post gives you is a complete agentic engineering workflow from spec to ship:

  1. Planning phase: CLAUDE.md (with AGENTS.md) as project constitution + ROADMAP.md (feature plan) + tasks.md (progress sync), tied together with @<path> import
  2. Division of labor: Skill injected into main context, Sub-agent with isolated context, Agent Teams running in parallel
  3. Development discipline: tests in the same turn + UI delegated to specialists + one logical change per commit
  4. Wrap-up validation: typecheck + test + build three validations + Writer/Reviewer dual-Claude + AI citation cross-check
  5. Mistake recovery: /rewind → git restore → scrap plan and rewrite, three layers of recovery
  6. Memory system: global + project two-layer architecture + positive pattern templates

There's a common spirit running through all six sections: every single rule is a discipline against AI amnesia, direction drift, and side effects. Karpathy's core point about agentic engineering is exactly this — you're no longer the engineer writing code by hand, you're the engineer supervising a team of AI employees. Your work is designing specs, supervising plans, reviewing diffs, writing tests, building evals, managing permissions.

More importantly, once you start running this workflow, you'll gradually hit traps unique to your project. When you do, don't just solve them in the moment — write them into memory or CLAUDE.md / AGENTS.md to make them next time's rule. Same trap never gets hit twice. Every trap caught becomes another step of solidified workflow. Eventually you end up with an agentic engineering SOP that's genuinely yours.

Hope this helps you build your own agentic engineering workflow. If you run a few rounds and your workflow diverges from mine, that's totally normal — every team has different habits. Pick the parts that work for you, try them out, and slowly tune your own version.
