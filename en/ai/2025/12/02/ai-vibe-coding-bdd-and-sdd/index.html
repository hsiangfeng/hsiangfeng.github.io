A Quick Take on AI Vibe Coding's Mainstream Methodologies — BDD and SDD

#AI #Vibe Coding #BDD #SDD

Intro

If you’ve used AI to develop any product or feature, you’ve probably heard terms like “AI code generation”. I’m calling this style — “describe a feature directly in natural language, and have AI generate the code” — “Vibe Coding”.

Vibe Coding

The term “Vibe Coding” is a bit awkward to translate, but I think “intuitive development” works better than “develop by feel” — because what Vibe Coding really means is “develop directly by intuition”. You don’t need to write code yourself. You just describe what you want in natural language and the AI generates the code for you.

Beyond that, Vibe Coding has some core principles:

  • Natural language description: describe features in natural language instead of writing code by hand.
  • Don’t worry about implementation details: users don’t need to care whether it’s Python or JavaScript, or whether semicolons are needed — the AI handles all of it.
  • Focus on business logic: you can focus on describing the business logic without worrying about technical details. This lets devs quickly validate ideas, putting energy into “what problem to solve” instead of “how to write the code”.
  • Real-time feedback and iteration: when the code or feature doesn’t match expectations, ask the AI to modify it instead of manually fixing it.
  • Removing the barrier: lots of people are intimidated by writing code, but Vibe Coding lets them participate in dev too — natural language is enough. Truly anyone can create.

For me personally, I’ve built quite a few small tools with Vibe Coding, even applied it to work.

The hidden problems with Vibe Coding

Relying entirely on “intuition” and AI conversation to produce code starts hitting clear bottlenecks as projects grow:

  • Not precise (Ambiguity): natural language descriptions are inherently ambiguous, and a single sentence can have multiple interpretations.
    • Example: “please write me a program that adds two numbers”.
    • For the AI: do you want “floating-point addition” or “integer addition”? The word “numbers” itself isn’t specific enough.
  • Inconsistent style (Inconsistency): developers usually have a strict Code Style for future maintainability, but Vibe Coding-generated code often varies. If you intervene manually, you lose the point of Vibe Coding.
  • Functional defects (Bugs): AI-generated code often has subtle logic errors. For non-technical people, these bugs can be hard to spot, and corner cases might not be fully covered.
  • Security concerns (Security): there can be security issues like SQL Injection, XSS, etc. — and the user has to handle these.

Note
Corner cases are situations where the system might behave abnormally under specific conditions — usually rare or extreme situations, but they can crash the system or cause errors.

These are all clear downsides of Vibe Coding. So now let me introduce BDD and SDD — two methodologies that can help us solve some of Vibe Coding’s problems.

BDD and SDD: frameworks for making AI listen better

BDD stands for Behavior-Driven Development, and SDD stands for Specification-Driven Development. Both are methodologies for better describing requirements and behavior, leading to more precise and consistent generated code.

These aren’t anything new. BDD has a strong relationship with TDD (Test-Driven Development). TDD’s core principle is “write tests first, code after”, while BDD’s is “describe behavior first, code after”.

“Isn’t TDD suitable for Vibe Coding?” you might ask.

That’s a good question. It’s not that TDD isn’t suitable for Vibe Coding — it’s that TDD is itself fairly technical. For non-technical people, TDD has a learning curve. A regular person has a hard time understanding what a “test” even is, let alone “code”.

BDD, on the other hand, lines up nicely with Vibe Coding’s philosophy. Because BDD describes behavior in natural language, it matches Vibe Coding’s “natural language description” trait. It also forces users to describe requirements in a structured way.

So… what about SDD? In short, SDD is a detailed requirements specification document — it specifies in detail what features the system should have, what behaviors, and what constraints. The AI references this document while generating code, producing code that better matches requirements.

So I’ll split this into two parts and explain how each applies to Vibe Coding.

BDD applied to Vibe Coding: telling stories

BDD typically revolves around three core steps. In AI development, we mainly focus on the first:

  1. Behavior description (Given-When-Then): users describe system behavior in natural language, usually using the “Given-When-Then” format
  2. Test implementation: based on the behavior description, implement tests to verify the system matches that behavior
  3. Code generation: finally, generate code based on the test cases

So what is Given-When-Then?

  • Given (preconditions): describes the initial state or background of the system. e.g., “Given the user is logged in”
  • When (action triggered): describes the action the user or system performs. e.g., “When the user clicks the ‘Add Product’ button”
  • Then (expected result): describes the result the system should reach. e.g., “Then the system should display the ‘Product added’ message”

With BDD covered, let’s look at how SDD helps us solve Vibe Coding’s problems.

SDD applied to Vibe Coding: defining the skeleton

If BDD is telling a story, SDD is drawing a blueprint — it tells the AI what the data should look like, which technologies to use, and what the dev standards are.

For regular developers, SDD should be the most familiar. After all, SDD stands for Software Design Document, and a requirements specification document usually contains:

  • System Architecture: describes the overall system architecture, including frontend, backend, database, etc.
  • Data Structures: defines data structures used in the system — DB table structures, API data formats, etc.
  • Technology Stack: specifies the technologies the system uses — programming languages, frameworks, tools, etc.
  • Development Standards: defines code style, naming conventions, testing standards, etc.
  • Non-Functional Requirements: includes performance, security, scalability, and so on.

So… BDD or SDD?

You’re probably wondering at this point:

“So which should I pick — BDD or SDD?”

Actually, the two can be used together. BDD focuses on “behavior description”, while SDD focuses on “technical detail” — they complement each other. But for non-technical people, I’d recommend starting with BDD, since BDD is closer to natural language description and easier for non-technical folks to pick up.

For people with dev experience, you’ll lean toward SDD — since SDD helps define the system’s technical details more precisely, leading to code that better matches needs.
