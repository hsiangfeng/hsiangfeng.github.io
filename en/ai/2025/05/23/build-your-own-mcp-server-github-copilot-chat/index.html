MCP Server

Intro

MCP Server is incredibly popular right now, and major service providers are releasing their own MCP Servers. So to better understand how MCP Server actually works, what better way than to build one yourself! This post takes you from zero to building your own MCP Server, paired with GitHub Copilot Chat to make using MCP Server more convenient.

Preparing the environment

Before starting, prep these:

  • VSCode
  • Node.js
  • GitHub Copilot

To build an MCP Server, we need a topic to work with. Here we'll use the todolist API provided by HexSchool as our template. This API has lots of features — add, delete, update — which will be the practice templates for our MCP Server implementation.

First, open your terminal and create a todolist-hexschool-mcp folder:

Bash
mkdir todolist-hexschool-mcp

Move into it:

Bash
cd todolist-hexschool-mcp

Initialize the project:

Bash
npm init -y

After initialization, the project will have a package.json file with basic info — project name, version, description, etc.

The MCP official site recommends using @types/node and typescript for development, but I want to keep this project as simple and easy-to-follow as possible, so we'll just install:

Bash
npm install @modelcontextprotocol/sdk zod axios

Quick rundown of these packages:

  • @modelcontextprotocol/sdk: the core MCP Server package — we'll use this to build our MCP Server.
  • zod: a data validation package — we'll use this to validate that data is correct.

Open package.json and add this property to make the project use ES Module syntax:

JSON
··{
  "type": "module",
}

In short, this makes the project modular so we can use ES Module syntax to write code.

Finally, create an index.js file — the main program and entry point for our MCP Server.

Bash
touch index.js

So far so good. If everything's set up, let's start writing code.

Setting up the MCP Server

The setup is simple. First, import these:

JavaScript
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import axios from "axios";
import { z } from "zod"; // We'll use this later

After importing, instantiate the MCP Server:

JavaScript
····const server = new McpServer({
  name: "hexschool-todolist",
  version: "1.0.0",
});

The McpServer constructor takes an object with two required properties: name and version. The name is our MCP Server's name, and version is its version.

Then add a server startup call at the end:

JavaScript
··········async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("HexSchool TodoList MCP Server starting...");
}

main().catch((error) => {
  console.error("Fatal error in main():", error);
  process.exit(1);
});

That's the basic MCP Server skeleton. Next, let's write some functionality so the MCP Server actually does stuff.

Writing a Tool

We're going to use a method on server called tool. This is similar to OpenAI's Function Calling/Tools Calling — it lets us define tools that the AI can call to complete tasks.

What do we do first?

For the TodoList API, the first action is "register an account", followed by "log in to the account". So we can imagine a user's natural language input:

Text
I want to register a todolist account — email is example@gmail.com, password is example123, nickname is example

When the AI receives this input, it knows the user wants to register an account, so it calls our defined tool, which then calls the API to do the work.

From the Swagger doc, the API endpoint is POST https://todolist-api.hexschool.io/users/sign_up, and the data format is:

JSON
······{
  "email": "example@gmail.com",
  "password": "example",
  "nickname": "example"
}

In pure JavaScript, we'd write:

JavaScript
······import axios from "axios";

const API_URL = "https://todolist-api.hexschool.io/users/sign_up";

const res = await axios.post(API_URL, {
  email: "example@gmail.com",
  password: "example123",
  nickname: "example"
});

Simple and straightforward. But because we're calling this API through server.tool, we have to define the API URL and data format inside the tool so the AI knows how to call it.

The server.tool syntax looks roughly like:

JavaScript
················································································server.tool(
  "tool_name", // English name required
  "tool description",
  {
    // Parameter validation
  },
  async (input) => {
    // Actual execution logic

    // Content returned to the AI
    return {
      content: [
        {
          type: "text",
          text: "returned content",
        },
      ],
    };
  },
)

With this example, you can get a feel for server.tool. Let me just paste the code:

JavaScript
········································································································import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import axios from "axios";
import { z } from "zod";

const API_URL = "https://todolist-api.hexschool.io/users/sign_up";

const server = new McpServer({
  name: "hexschool-todolist",
  version: "1.0.0",
});

server.tool(
  "hexschool_todolist_sign_up",
  "Register an account",
  {
    email: z.string().email(),
    password: z.string(),
    nickname: z.string(),
  },
  async (input) => {
    const { email, password, nickname } = input;
    const res = await axios.post(API_URL, {
      email,
      password,
      nickname,
    });

    return {
      content: [{
        type: "text",
        text: 'Registration successful',
      }]
    }
  }
);

async function main() {
  const transport = new StdioServerTransport();
  await server.connect(transport);
  console.error("HexSchool TodoList MCP Server starting...");
}

main().catch((error) => {
  console.error("Fatal error in main():", error);
  process.exit(1);
});

In terms of how server.tool works, the AI + MCP Server flow looks like:

  1. User tells the AI they want to register a Todolist account
  2. AI analyzes that it needs to use the "hexschool_todolist_sign_up" tool
  3. The "hexschool_todolist_sign_up" tool gets called
  4. The tool receives the parameters from the AI (email, password, nickname) and validates the format with zod
  5. After execution, results are returned to the AI
  6. The AI returns the result to the user

That's how server.tool operates.

Testing the MCP Server

So how do we test this MCP Server? node index.js?

No. Because we're pairing it with GitHub Copilot Chat, and to avoid polluting the VSCode global setting.json, run:

Bash
mkdir .vscode

Then create an mcp.json file:

Bash
touch .vscode/mcp.json

These two steps create a project-specific .vscode folder with an mcp.json file inside, which is used to configure the MCP Server.

Open .vscode/mcp.json and add this:

JSON
············································{
  "servers": {
    "todolist-hexschool-mcp": {
      "type": "stdio",
      "command": "node",
      "args": [
        "/Users/rayxu/todolist-hexschool-mcp/src/index.js"
      ]
    }
  }
}
Note

Replace "/Users/rayxu/todolist-hexschool-mcp/src/index.js" with your own absolute file path.

Quick rundown of this file:

  • servers: the property for our MCP Server config
  • todolist-hexschool-mcp: our MCP Server's name — you can name it anything
  • type: our MCP Server's type — here we use stdio (standard input/output)
  • command: the command for our MCP Server — node here
  • args: the file we want to run — index.js (our MCP Server's main program)

Click the "Start" button on VSCode's screen.

Start

The MCP Server will start, and you'll see this in VSCode's terminal:

Bash

How do we verify it works? Open GitHub Copilot Chat and switch to AI Agent mode.

Agent Mode

Now you can start asking the AI things like:

Text
I want to register a todolist account — email is example-mcp@gmail.com, password is example123, nickname is example

When called, dialogs will pop up asking you to confirm the tool to use. Since this is our own MCP Server, no need to worry — just hit "Continue".

Continue

You'll see the AI's response.

Registration successful

At this point, we've successfully built a basic MCP Server that can interact with GitHub Copilot Chat. Starting from a register API, we built actual executable tool logic.

Now you can try adding more features — login, add tasks, delete tasks — turning this Server into a real assistant for your AI development.
