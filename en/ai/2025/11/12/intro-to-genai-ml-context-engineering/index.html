Intro to Generative AI and Machine Learning

Intro

Today's AI applications are basically all tied to Prompt Engineering and Context Engineering. So what's the difference between the two, and how do they relate? This post digs into how these two concepts shape how AI Agents work.

What is Context Engineering?

The previous post "Intro to Generative AI and Machine Learning (1) — It's All Just Text Prediction" kept hammering one thing:

"A language model is basically a 'text prediction' system."

Given the Prompt you input, the language model predicts the next character based on the input, then keeps repeating until it finishes the full response.

Let's revisit the formula:

$f(x)=ax+b$

  • x: input prompt
  • f: language model
  • a, b: parameters inside the language model
  • f(x): model output

When you input a Prompt x, the language model f calculates the output f(x) based on its internal parameters a and b. But when the output f(x) isn't what you wanted, you can only adjust x (the prompt) or f (the language model itself).

If you're adjusting the model, that's called "Training". If you're changing the model's a and b parameters, that's called "Learning".

Usually we don't go in and tweak the model's parameters directly, so we focus on "tuning the prompt (Prompt Engineering)" — designing a better prompt x to get the language model f to output what we want as f(x). Most of the time we can't actually modify the language model itself — ChatGPT for example, you can only use OpenAI's API or interface to interact with it, you can't change its parameters directly. It's a closed online language model, so even if there's a problem, you can't fix it.

Prompt Engineering vs Context Engineering

So we've said most of the time we modify the Prompt to get the result we want — this is called "Prompt Engineering". We design a better prompt x to get the language model f to output what we want as f(x).

So what's "Context Engineering"?

The course's professor mentioned that, fundamentally, there's not much difference — it's basically just a fancier name.

But is that really the case? There are still real differences between the two. The concept is similar, but the focus is different. Prompt Engineering is about "how to design a good prompt", or even "magical-sounding prompts" that coax the language model into producing what we want.

For example:

txt
You must respond in JSON format and include the following fields:

- name: user name...

This is a classic Prompt Engineering example, because it directly tells the model what response format we want. As time has passed, AIs have gotten smarter — basically any prompt a human can understand, the AI can understand too. So we can design more complex prompts to get the result we want.

In the early days, models like GPT-3 were really rough — if you didn't put effort into the Prompt, the response usually wasn't great. So Prompt Engineering became really important, and people discovered all kinds of "magical spells" (prompts). Like, if you told the model "every response must be at least 3,000 words", the answer would actually become noticeably more precise.

magical prompts

Other "magical spells" emerged later — take three deep breaths, role-play as a particular character, even emotionally pressure the model — all of which could improve response quality.

There's an experiment at minimaxir.com that offered ChatGPT four options for "what you get if you answer correctly": Taylor Swift tickets, world peace, a tip, getting love. ChatGPT cared about world peace the most.

As time goes on, these magical prompts are slowly becoming less magical — thanks to models getting more capable.

As the magic of prompt spells fades, a new thing has emerged — "Context Engineering". The focus of Context Engineering is the automated management of "context information". We're no longer just designing a single prompt — we're managing the entire conversation's context information so the model can understand our needs better.

User Prompt: give the premise

What should Context Engineering include? Basically everything Prompt Engineering uses, like:

  1. Task description: write a TodoList Web Application.
  2. Detailed instructions (options): use Vue + Vite and Bootstrap, persist data to LocalStorage so it's available next time.
  3. Extra conditions: input fields need form validation, completed todos should be deletable.
  4. Output style: code should be readable, with comments explaining what each function does.

The clearer you can be, the better — language models can't read minds. They respond based on the info you give them.

For example, if you ask ChatGPT in Mandarin:

txt
What's a "carrier"?

ChatGPT has no idea what you're asking about, so the response will be vague:

ChatGPT

This is a classic case of not giving the model a clear "premise". Mandarin is full of nuance — a single word can have many meanings depending on context. The Mandarin word "載具" can mean different things in different situations. If you clarify:

txt
At a 7-Eleven checkout, the clerk asked me whether I wanted to use a "載具". What does "載具" mean here?

ChatGPT's response suddenly becomes really precise:

ChatGPT

(For non-Mandarin readers: in Taiwan, "載具" is the term used for a digital receipt carrier registered with the tax system. The model needs context to disambiguate from "vehicle".)

Giving the model a clear "premise" is genuinely useful — this is a key idea in Context Engineering.

User Prompt: give a reference

Beyond clear premises, providing a concrete example helps a lot. For example:

txt
Please rewrite the text below in Martian script:

Hello, I'm a blogger called Ray, and I love writing technical posts!

You'll find ChatGPT's response probably isn't what you expected:

ChatGPT

This is because "Martian script" doesn't have a single agreed definition, so the model can't figure out what you mean. If you provide an example:

txt
Please rewrite the text below in Martian script. Examples of Martian script:

- the => teh
- is => 4
- love => Love
- I => 1

Now rewrite this in Martian script:
Hello, I'm a blogger called Ray, and I love writing technical posts!

You'll find the model can now produce something matching your expectation more easily:

ChatGPT

Providing examples this way is called "In-Context Learning". Heads up: the "Learning" here doesn't change the model's parameters — it just means the model uses the context info we provide to learn within that conversation.

The course also brought up a fun case — "Karamonji language", apparently spoken by only a few thousand people, with basically no internet presence (I searched too — really no info). So the model has no way to learn anything about it. What do you do? Provide reference material like textbooks and papers, and you'll find the model can produce responses in Karamonji too.

That's a textbook "In-Context Learning" case.

So... how does the model actually learn? Again, "learning" here doesn't mean changing the model's parameters — it's letting the model use the context info to "learn" within that session.

What gives the language model the ability to translate "Karamonji" is the "example sentences" in those textbooks and papers — what really helps the model is example sentences, not grammar rules. Because the model is fundamentally a "text prediction" system that learns from massive amounts of text, give it enough example sentences and it can figure out how to translate Karamonji.

What exactly is a System Prompt?

So far we've talked about User Prompt (the user's prompt). What about System Prompt?

A System Prompt is a set of rules the developers of the language model set at the system level to guide the model's behavior. Claude has its System Prompts publicly documented — let me pull a snippet from Claude Sonnet 4.5:

txt
<behavior_instructions> <general_claude_info> The assistant is Claude, created by Anthropic.
The current date is {{currentDateTime}}.

Here is some information about Claude and Anthropic's products in case the person asks:
This iteration of Claude is Claude Sonnet 4.5 from the Claude 4 model family. The Claude 4 family currently consists of Claude Opus 4.1, 4 and Claude Sonnet 4.5 and 4. Claude Sonnet 4.5 is the smartest model and is efficient for everyday use.
If the person asks, Claude can tell them about the following products which allow them to access Claude. Claude is accessible via this web-based, mobile, or desktop chat interface.
...

It's pretty long, so I'm only showing part of it.

The first line immediately tells the model who it is, then today's date — this is why even though the model is "locked in a closed system", it still knows the date: because the System Prompt told it. A good model's System Prompt is very long, with lots of rules guiding behavior.

System Prompts usually aren't shown to users. For Claude's System Prompt, the key categories include:

  • Identity and product info: "The assistant is Claude, created by Anthropic."
  • Usage instructions and limits: "If asked about the Anthropic API, point to https://docs.anthropic.com"
  • Interaction style and user feedback: "If user is unhappy, suggest pressing the thumbs down button"
  • Safety and forbidden content: "Claude does not provide info to make chemical or nuclear weapons"
  • Response style and format: "Claude never starts its response with 'good question'"
  • Knowledge and factuality: "Claude's knowledge cutoff is Jan 2025"
  • Self-positioning and philosophy: "Claude does not claim to be human or conscious"
  • Error handling and interaction detail: "If corrected, Claude first thinks carefully before acknowledging"

So we can see — a good System Prompt is really long, with lots of rules that directly shape the model's response quality and style.

Dialogue History (short-term memory)

When you use ChatGPT, the current chat window is essentially "short-term memory".

What does that mean? Try this — in chat window A, say:

ChatGPT

Then open a new chat window B and say:

ChatGPT

You'll find ChatGPT suddenly doesn't remember what you said in A. This is because each chat window is "short-term memory" — it only remembers the current window's conversation. Open a new one and the prior conversation is gone.

In the image above, ChatGPT mentioned:

txt
If you want me to remember "Little Wang is the legendary Pharaoh" long-term, I actually can't do that right now — you'd need to go to Settings > Personalization > Memory and turn on the memory feature.

Within this conversation, I can definitely treat "Little Wang = legendary Pharaoh" as context, but it won't automatically persist to future conversations.

ChatGPT now offers a way to remember things "across windows" — it's called "Memory". You can enable it in settings, and ChatGPT will remember things across chat windows.

Long-Term Memory

So how does ChatGPT's long-term memory actually work?

ChatGPT

Obviously by going to personalization settings and turning on "Memory"!

Once you enable the Memory feature, every new chat starts with ChatGPT first reading your stored "memories" and using them as part of the context. The shape is roughly:

txt
[System Prompt]

[Long-Term Memory]

[User Prompt]

ChatGPT first loads the developer-set System Prompt, then your Long-Term Memory, then finally your User Prompt. That way ChatGPT can respond while taking your stored memories into account.

Other source references

Let's also talk about how models handle outdated information. You've probably hit a situation where AI gives you stale info — like always replying with Vue 2 patterns when we're clearly in Vue 3 territory now. What do you do?

The most common approach: search engines. Let the language model search the web for fresh info and use that as part of the context, so the model can give you up-to-date info.

Web Fetch

So before asking the model to do something, you can have it search first, then respond or generate content. You can also feed your own database to the model.

This actually has a formal name:

"Retrieval-Augmented Generation (RAG)"

Even with RAG strengthening the model's knowledge, the model's output isn't guaranteed to be correct — at the end of the day, it's still "text prediction". So we still need to review and verify the info the model provides.

Tool Use

While we're here — using a search engine is itself a "Tool Use". For example, Gemini can read and even modify your Gmail and Google Calendar.

How does the model actually use tools? Simple — in the Context, you tell the model what tools it can use. When the model needs to use one, it follows the Context's instructions, uses the tool, and returns the result to the user.

txt
If you hit a problem you can't solve, you can use tools.
Wrap tool use commands in <tool> and </tool> tags. After using a tool, put the result between <output> and </output>.

Then you tell the model how to use each tool:

txt
Available tools:

- Search engine: search for latest info. Usage: `WebSearch(value)`. Example: `WebSearch("latest AI tech in 2025")`
- Calculator: do math. Usage: `Calculator(value)`. Example: `Calculator("12345 * 6789")`

So when you give the model a User Prompt like:

txt
Please search for "latest AI tech in 2025" and tell me the key points.

The model will use the search engine tool based on the Context instructions:

txt
<tool>
WebSearch("latest AI tech in 2025")
</tool>

One thing to flag here — the model isn't actually using the tool, because that string is just text. You need a small program to convert that text into real tool invocation — usually called a "Mediator".

So when the model produces:

txt
<tool>
WebSearch("latest AI tech in 2025")
</tool>

The output gets handed to the mediator, which might look something like:

js
async function performWebSearch(query) {
  // actual search code goes here
  const response = await fetch(`https://api.example.com/search?q=${encodeURIComponent(query)}`);
  const data = await response.json();
  return data.results;
}

async function mediator(modelOutput) {
  const toolRegex = /<tool>([\s\S]*?)<\/tool>/;
  const match = modelOutput.match(toolRegex);
  if (match) {
    const toolCommand = match[1].trim();
    if (toolCommand.startsWith("WebSearch")) {
      const query = toolCommand.match(/WebSearch\("([\s\S]*?)"\)/)[1];
      const searchResults = await performWebSearch(query);
      return `<output>${searchResults}</output>`;
    }
    // extensible to other tools
  }
  return null;
}

Tool Use: Computer Use

Among all tools, the most powerful and general-purpose one is your computer — this is called "Computer Use". It lets the model use the computer for various operations like reading files, writing files, running code, etc.

ChatGPT has a feature called "AI Agent" that lets the model use the computer to do things — reading files, writing files, running code, even booking tickets, sending emails, organizing data. All of these are accomplished by letting the model use the computer.

ChatGPT

How does the model actually pull this off? The concept is the same as "Tool Use" — you tell the model what tools are available in the Context, and when it needs to use them, it follows the Context's instructions and returns the result.

txt
Available tools:

- Mouse: click, drag, scroll, move. Usage: `Mouse(action, x, y)`. Example: `Mouse("click", 100, 200)`
- Keyboard: type text, press keys. Usage: `Keyboard(action, key)`. Example: `Keyboard("type", "Hello World")`
- File system: read/write files. Usage: `FileSystem(action, path, content)`. Example: `FileSystem("write", "/path/to/file.txt", "Hello World")`
- Network request: send HTTP requests. Usage: `NetworkRequest(method, url, data)`. Example: `NetworkRequest("GET", "https://api.example.com/data")`

So when you give a User Prompt like:

txt
Please open the browser, search for "latest AI tech in 2025", then save the results to search_results.txt on the desktop.

The model uses the tools based on the Context:

txt
<tool>
Mouse("click", 100, 200)  // click browser search bar
Keyboard("type", "latest AI tech in 2025")
Keyboard("type", "Enter")  // press Enter
FileSystem("write", "/Users/username/Desktop/search_results.txt", "search result content")
</tool>

Same thing — a mediator translates the text into actual computer operations.

Reasoning

Now let's talk about "Reasoning". A lot of models claim to have "deep thinking" capability — DeepSeek series, Gemini series, certain ChatGPT models. How do they do it?

Usually a language model responds directly to your User Prompt based on the context. "Deep thinking" means "it doesn't answer right away — it does an internal mental rehearsal first", trying various approaches and self-verifying:

txt
<reasoning>
1. Approach 1: try solution A... no, doesn't work
2. Approach 2: try solution B... seems fine
3. Approach 3: try solution C... feels off
</reasoning>

Then it produces the final answer based on Approach 2.

Most models don't expose the full reasoning trace — usually only a summary. The reasoning is also part of the Context, just contributed by the model itself rather than the user.

So what is Context, anyway?

We've gone through a lot, but it's all "Context":

  • System Prompt
  • User Prompt
  • Dialogue History
  • Memory
  • Relevant info from other sources
  • Tool Use
  • Reasoning

That's a lot. So Context is really long, and the core focus of Context Engineering is:

"Avoid Context Too Long."

When Context blows up, the language model easily forgets earlier setups, info, etc., and can't respond properly.

Why the AI Agent era needs Context Engineering

Three common ways we use AI:

  • Standard usage: one question, one answer — let the AI generate content.
  • Agentic Workflow: let the AI complete a workflow you defined — a fixed flow.
  • AI Agent: rather than a fixed SOP, the AI Agent itself decides how to complete the workflow, even which tools to use. Like "book me a train ticket from Taipei to Kaohsiung" — and the model figures out what tools to use, whether to check the schedule first, etc. The core is: no fixed flow, the AI decides.

An AI Agent is like a loop — it keeps deciding the next step based on context, then re-decides based on new context, until the task is done.

Each iteration of an AI Agent has two phases:

  • Observation: the result of the previous Action — like a tool's output, an API response, an error message, or a new instruction from the user.
  • Action: what the Agent decides to do next based on the current Observation and overall context — like calling a tool, fetching info, updating the plan, or directly returning a result.

The task goal is called the "Goal".

Note When using AI Agents, be careful with the action flow — make sure it doesn't do harmful things like deleting files it shouldn't, sending spam, etc.

From the LLM's perspective, what is an AI Agent really solving?

Remember — a language model is still just playing "text prediction". AI Agents are just built on top of the LLM's prediction capability, iterating to complete a complex task. They're not some completely new technology.

The biggest challenge for AI Agents is "Context Too Long". An AI Agent keeps adding new Observations to the Context, so the Context keeps growing, eventually exceeding the model's max Context length. At that point the model can't respond properly.

You'll see models claim huge Context Window sizes, but the actually usable Context length is usually shorter than the official number — because handling the Context has overhead: System Prompt, Tool Use, Reasoning, etc. — all of which take up Context.

Many models will Auto Summarize earlier Context before hitting the limit — compressing longer Context into shorter summaries. This saves Context length but often loses information, because you don't know exactly which info got kept and which got dropped.

One more thing — being able to input millions of tokens doesn't mean the model can actually understand all of it. Same idea as asking you to read a thick book — you won't necessarily understand all the content, and you won't know what the key points are. Many experiments have shown "the more data you give, the worse the model performs" — because it can't comprehend everything.

Initially the model can handle a certain amount, but as the data grows, the model loses understanding and gets confused. Many studies have shown language models mostly remember the "beginning" and "end" of the context, not the middle. That's why when designing Context, important info should go at the beginning and end so the model is more likely to remember it.

Context Too Long

It turns out the model's memory pattern is really similar to humans — we mostly remember the beginning and end too, and forget the middle.

The basics of Context Engineering

From all the examples above, the core of Context Engineering boils down to:

"Manage your Context well."

Put the important Context in, leave the unimportant Context out, and make sure Context doesn't go too long — then the model can respond correctly.

Let's go through three common Context Engineering techniques.

Select

Pick the Context info the model actually needs — don't dump every related (or unrelated) piece of info on it, because that overloads the model and hurts response quality.

For example: you use RAG to grab a pile of related info from the web, but you should only hand the model the parts it actually needs to read, not the whole document. Like I said earlier — even humans mostly remember just the beginning and end, and models are the same.

My recent personal example — I just rewatched all 700 episodes of Naruto over a month. If you ask me what happened in the middle, I can't tell you. I can only tell you the beginning and ending, plus a few highlights — Naruto is an orphan with the Nine-Tailed Demon Fox sealed inside him, his teacher is Jiraiya, he grew up with Sasuke, Sasuke turned bad, Naruto fought him, in the end Naruto won him over and there was peace.

So... is filtering out the unnecessary info something humans should do? No — let the language model do it, not by hand.

The core point is: "Only include the Context info the model actually needs." This avoids Context blow-up. Same for tools — don't dump 100 tools at it, pick the few it actually needs.

One more thing — when the model makes a mistake, don't immediately tell it to fix the mistake. Sometimes those error traces actually make the model perform worse — like when someone says "don't think about elephants" and now you can't stop thinking about elephants.

So how do you get the AI to stop making the same mistakes? Use better techniques — give it correct examples, give clearer instructions, etc. — and then put those correct examples in the Context so the model can learn the correct behavior pattern. That's how you avoid the same mistake.

Compress

Compression mainly comes up when the Context is too long. So... do we compress it ourselves? No — let the model do it.

For example, you can ask the model to summarize and compress the previous content whenever its Context Window hits 90%, or every 100 interactions. Then subsequent responses use the compressed version. That avoids Context Too Long.

Each compression includes the previous compression's content:

Compress

So the older the content, the less detail the model can remember.

A common real-world trick: store the data in Markdown files and have the model read those files, then compress the contents, then put the compressed versions in Context.

If there's important content you don't want lost to compression, you can put it in long-term storage and use RAG to pull it back, so the important stuff doesn't get compressed away.

Multi-Agent

The Multi-Agent concept is:

You set up multiple AI agents with different specialties, then have them cooperate on a complex task — like CEO Agent, CTO Agent, Project Manager Agent, Developer Agent, Designer Agent, etc. Each agent owns its slice of work, and they collaborate.

Multi-Agent is actually a really effective way to manage Context, because each agent only focuses on its own slice and doesn't get distracted by unrelated content.

For example, without Multi-Agent, building a website would mean cramming requirements, design, development, testing, etc., all into one Context — basically having one person handle design, frontend, backend, and testing all at once. That's Single-Agent.

With Multi-Agent, you have a Lead role (like a Project Manager Agent) that coordinates the team. When a request comes in, the Lead analyzes it and decides which agent handles which piece. Each agent applies its expertise to its piece, returns the result to the Lead, and the Lead sends the result back to the user.

It looks like this:

txt
I want a website with members, product display, shopping cart, and order management.

The first to receive this is the Lead (Project Manager Agent):

txt
User wants a website with members, product display, shopping cart, and order management.

But I need to understand the specifics first, so I'll hand this to the Product Manager Agent for requirements gathering with the user.

So the overall flow looks like:

txt
User → Lead Agent → Product Manager Agent → Lead Agent → Designer Agent → Lead Agent → Developer Agent → Lead Agent → User

This is a way more effective way to manage Context.

When the task is simple, Single-Agent works fine. When it's complex, Multi-Agent shines.
