Intro

We're almost at the end of the iThome Ironman series, and I just realized I skipped something that's actually kind of important — the whole pass by value vs pass by reference thing. So this post is a quick chat about how values and references behave in Python.

Pass by Value

Let's start with plain old pass by value. In JavaScript, primitive types just hand a copy of the value to another variable, so changing one doesn't touch the original:

js
var a = 'Ray';
var b = a;
b = 'QQ';

console.log(a, b); // Ray, QQ

Now back to Python — does it work the same way? Let's just look at an example:

py
a = 'Ray'
b = a
b = 'QQ'

print(a, b) # Ray, QQ

Same result. Python and JavaScript behave the same here.

But what if we write it a different way — does the result hold up? Here's another snippet:

py
def fn(x, y):
  cache = 100
  x = cache
  y = x

x = 10
y = 20

fn(x, y)

print(x, y) # 10, 20

In this one, I passed the global variables x and y into the fn function, declared a cache = 100 inside it, then reassigned x and y. But you'll notice the changes inside fn don't affect the global variables at all. What does that tell us? It tells us we only passed the values of x and y for fn to use, so whatever happens inside the function doesn't touch the originals.

Now you might cleverly remember that back in "Learning Python From a JavaScript Angle (6) - Variable Scope" I introduced global, which lets you tinker with outside variables. But if you try using global here, you'll just hit an error:

py
def fn(x, y):
  cache = 100
  global x, y # SyntaxError: name 'x, y' is parameter and global
  x = cache
  y = x


x = 10
y = 20

fn(x, y)

print(x, y) # 10, 20

The reason you get SyntaxError: name 'x, y' is parameter and global is that if a function's parameter shares a name with the variable you declare global, things break — Python can't tell whether you mean the parameter or the variable.

You can't play it like that

So here we can take a bold guess...

"Maybe Python's numeric, string, and boolean types behave like JavaScript's primitive types?"

To make that guess more solid, let's actually test it. From earlier chapters we learned that container types are like JavaScript's arrays and objects, so here we can write out string, boolean, and numeric types and see what happens:

py
def fn(w, x, y, z):
  w = False
  x = 10
  y = 1.1
  z = 'Ray'
  print(w, x, y, z) # False, 10, 1.1, Ray


w = True # boolean
x = 9.9 # float
y = 'Ray' # string
z = 10 # integer


fn(w, x, y, z)

print(w, x, y, z) # True, 9.9, Ray, 10

From the result, Python's string, boolean, and numeric types really do behave like JavaScript's primitive types. So what about dictionaries and lists? Hold on, we'll get there — just remember we're talking about Python here, not JavaScript:

This is Python, not JavaScript

Pass by Reference

Now let's talk about the dictionaries and lists I left out. We already know Python's dicts and lists map to JavaScript's objects and arrays, but do they actually behave the same way? Let's go through this part quickly too.

First, here's a really common JavaScript example of passing an object by reference:

js
var b = {
  name: 'Ray',
};

var e = b;

e.name = 'QQ';
console.log(b.name); // QQ

This is the classic pass-by-reference concept. If you want to dig into it, I wrote a more detailed piece on it: "JavaScript Core Concepts (27) - Objects - How Object References Actually Work (article in Mandarin)", so I won't rehash it here.

Got tricked

After all that buildup, what about Python? Let's rewrite the JavaScript example directly into Python and see:

py
b = {
  'name': 'Ray'
}

e = b

e['name'] = 'QQ'
print(b['name']) # QQ

Okay... judging by the output, Python's dictionary behaves a lot like JavaScript's object when it comes to passing by reference, right?

Now let's look at another snippet:

py
def fn (x):
  x['name'] = 'QQ'

b = {
  'name': 'Ray'
}

fn(b)

print(b['name']) # QQ

This one works fine, and it sure looks heavily tied to pass by reference. But what if the code changes to the following — what's the result then?

py
def fn (x):
  x = {
    'name': 'Ray'
  }


b = {
  'name': 'Ray'
}

fn(b)

print(b['name']) # Ray

Huh? Weird, why is the result different now?

My head hurts

Okay, there's something special going on with passing objects (dictionaries) by reference. We know Python's dict is basically like JavaScript's object, so when you assign a dict to a variable, you're really assigning a memory location. I know that's a bit hard to grasp, so let me put it another way. Let's understand container types vs numeric and string types through a real-life analogy.

I'll use a car as the example. You can picture numeric and string types as stuff inside the car — tires, the steering wheel, the radio, and so on. A dictionary or list, meanwhile, is like a car that's just a shell (not even an engine). So with that analogy, it might look like this:

py
# Think of the dict as a car shell holding the car's parts
car = {
  'steeringWheel': 1, # steering wheel
  'wheels': 4, # tires
  'radio': 1, # radio
}

Now when I assign the car dict to another variable, the concept is kind of like lending the car out to someone else. Say I borrowed a car:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car

So if I change anything on the car — maybe pull off a tire — it affects the original car, because the car is borrowed:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car
ray['wheels'] = 3

print(car['wheels']) # 3

That lending-out behavior is exactly JavaScript's pass by reference — I'm just letting you use it.

Now there's a situation where I don't affect the original car — namely, I finally have enough money to buy my own car:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car 

ray = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

Since I bought my own car, I'm no longer borrowing anyone's, so no matter how I tweak my own car from here on, it won't affect the original car variable:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car 

ray = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray['radio'] = 0

print(car['radio']) # 1

The analogy is just to make this easier to read. So whenever you see a variable get reassigned to a dict or object, it means it now points to a brand-new memory location (an empty car shell). If that's still tricky, just remember: when you see a variable reassigned to a fresh {} or [], it's been pointed to a new address, so the pass-by-reference issue won't happen.

Of course, you can also use the id trick from earlier chapters to check whether two variables share the same memory location:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car

print(id(ray), id(car)) # 4463682048, 4463682048

And if it's reassigned, the memory locations differ:

py
car = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

ray = car 

ray = {
  'steeringWheel': 1,
  'wheels': 4,
  'radio': 1,
}

print(id(ray), id(car)) # 4418249408 4418249216

Some of the tricks from earlier chapters can really help you understand this, which is why those earlier concepts matter so much.

call by xxx

Finally, you've probably noticed I keep deliberately dodging certain terms — like call by xxx or pass by xxx. The main reason is I'm not super familiar with Python myself, so I don't dare say too much, mostly out of fear of getting it wrong.

But if you really pin me down, I'd go with what Wikipedia says, because by its description, Python is closest to Call by sharing.

Call by sharing was named by Barbara Liskov, and is used by languages like Python, Java (object types), JavaScript, Scheme, and OCaml.

And this happens to line up nicely with JavaScript, so honestly I think you can get away with treating Python like JavaScript here.

One last fun thing to look at — this code:

py
a = 'Ray'
b = 'Ray'

print('a', a, id(a)) # a Ray 4351631984
print('b', b, id(b)) # b Ray 4351631984

Notice I never did a = b, yet the two have the same memory location. That means when a string is created, both point to the same memory location, which is why a == b is True. I think this image helps explain it:

Memory space

But once you reassign a different value, it points straight to a new memory location:

Repointing to a new memory space

Dictionaries are similar, except a dict only repoints when it sees a fresh {} or []. I'll be a little lazy here and reuse the image from my earlier post "JavaScript Core Concepts (27) - Objects - How Object References Actually Work (article in Mandarin)":
