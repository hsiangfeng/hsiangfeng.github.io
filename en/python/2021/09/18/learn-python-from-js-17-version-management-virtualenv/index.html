So from last chapter we know that no matter what package you install, it lands in the Global environment:

The thing is, once you’ve got a lot of packages, this gets messy and can mess up your own dev setup. Sometimes when we’re working on a project we only want a specific version of a package, but using pip3 install just keeps overwriting and overwriting your Global environment. That’s exactly why the official docs have a chapter called “Virtual Environments and Packages“ — it’s there to solve this problem.

Creating a Virtual Environment

Back in Python 3.3, a module called venv got bundled in, and from what I dug up it became the recommended tool for managing virtual environments around Python 3.5. So we can use this module to spin up a virtual environment directly.

So what’s venv actually for? In short, it can create a small Python virtual environment inside your project based on whatever Python version your machine is currently running. That way when you install packages, they don’t get dumped into the Global environment. It’s basically the thing we do in npm with npm install --save. One heads-up though: once you’ve created the virtual environment, installing a package goes straight into the virtual environment — you don’t need to add any extra flags!

And personally I’d recommend using venv to keep your dev environments separate. After all, you don’t want a giant pile of Global packages on your machine, right?

Version Management

Oh! So far we’ve only covered the virtual environment side of Python. But in real-world development, can one Python version really do everything? Of course not. In my earlier Hexo series I explained how Node.js 8 and Node.js 13 have huge performance differences, and Python is exactly the same — programs get updated, new syntax gets added, bugs get fixed, which also means packages get updated along with the versions.

But sometimes a package we’re using, or a particular dev environment, simply has to run under a specific version. In that case you need to switch versions. It’s the same on the JavaScript side: for Node.js development we usually use the nvm package to switch Node.js versions.

If you haven’t used nvm yet, I’d recommend it. Here are some notes I’ve written:

  • Installing nvm and switching NodeJS versions on Windows 10 (article in Mandarin)
  • Installing nvm on Mac OS X (article in Mandarin)

Cut back to Python — so… what about Python? Is there a super-handy nvm-like thing for switching versions?

The answer is no, there’s no nvm. But there is a thing called pyenv.

Hold on, I said there’s no nvm — I didn’t say there’s no other way. So next up, let me show you how to install pyenv.

Common pyenv Commands

Let me list out some of the commonly used pyenv commands too:

  • Install a specific Python version
    • pyenv install [version]
    • The verbose version of the install command is - pyenv install -v [version]
  • List which Python versions are available to install
    • pyenv install -list
  • Remove an installed Python
    • pyenv uninstall [version]
  • List the Python versions you’ve already installed
    • pyenv versions
  • Switch Python versions
    • pyenv global [version]
      • switch the Global Python version
    • pyenv local [version]
      • switch the current project’s Python version
    • pyenv shell [version]
      • switch the current terminal’s Python version

The usage isn’t too hard, so I won’t walk through each one here.
