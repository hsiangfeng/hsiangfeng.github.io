Intro

Last time we talked through the basic pip commands and how to use them. But there were a couple of problems lurking in that chapter that I never actually explained how to solve. So this post is where we fix them.

Virtual Environments

In the previous chapter we covered the pip commands and how to install things. But there's one thing you might have noticed, and it has to do with where packages get installed.

Huh? What? You forgot? No problem, let me run through it again...

What did you forget for

(Kidding about that, by the way.)

So from last chapter we know that no matter what package you install, it lands in the Global environment:

cmd
pip3 install pytube

The thing is, once you've got a lot of packages, this gets messy and can mess up your own dev setup. Sometimes when we're working on a project we only want a specific version of a package, but using pip3 install just keeps overwriting and overwriting your Global environment. That's exactly why the official docs have a chapter called "Virtual Environments and Packages" — it's there to solve this problem.

Creating a Virtual Environment

Back in Python 3.3, a module called venv got bundled in, and from what I dug up it became the recommended tool for managing virtual environments around Python 3.5. So we can use this module to spin up a virtual environment directly.

So what's venv actually for? In short, it can create a small Python virtual environment inside your project based on whatever Python version your machine is currently running. That way when you install packages, they don't get dumped into the Global environment. It's basically the thing we do in npm with npm install --save. One heads-up though: once you've created the virtual environment, installing a package goes straight into the virtual environment — you don't need to add any extra flags!

Here's the flow for creating a virtual environment:

Create the virtual environment -> install or restore the packages you need -> start working

So first let's create a new project:

cmd
mkdir getYTVideo # create a folder called getYTVideo
cd getYTVideo # move into the getYTVideo folder

Once you're inside the folder, run the official command:

cmd
python3 -m venv tutorial-env

Now your project should have a new folder called tutorial-env. If you don't like that name you can change it — say, to PyVideo — but I'd recommend tacking on -env at the end so it's clear this is a virtual environment:

cmd
python3 -m venv PyVideo-env

PyVideo-env

But honestly, I'd suggest just keeping the default folder name, tutorial-env. If you rename it you'll have to do some extra tweaking, otherwise running pip3 list might still pick up your local setup. So sticking with tutorial-env is the safer bet, because when it's correct it looks like this:

tutorial-env

This virtual environment gets generated based on your current Python version, so it usually goes hand in hand with version control. That's why later on I'll also cover how to use pyenv to switch versions.

Installing Packages

Once the virtual environment is set up, you can install packages into the current virtual environment. We'll do some practice with this later, so for now you can just try installing a package or two:

cmd
pip3 install pytube

(It's a Python package for downloading YouTube videos.)

After it installs successfully, try running pip3 list to check it went through. You should see the pytube package sitting in your current virtual environment:

pytube

If this feels confusing, think of venv like package.json — when you install a package it goes into the local project. Without venv, packages default to installing globally across your whole machine.

And personally I'd recommend using venv to keep your dev environments separate. After all, you don't want a giant pile of Global packages on your machine, right?

VSCode

One extra thing worth mentioning here. If you're developing in VSCode, you might run into a situation where no matter what you do, VSCode keeps reading the global Python. So you'll need to tweak the VSCode setup a bit.

For VSCode specifically, the official docs suggest running this:

cmd
python3 -m venv .venv # Mac
python -m venv .venv # Windows

Usually after running that you'll see a .venv folder show up. You might hit a few issues along the way, but I've already shared how to fix those in an earlier post, so I won't repeat it here:

Setting up Python3 venv parameters in VSCode (article in Mandarin)

So consider this a little bonus for anyone developing in VSCode.

Version Management

Oh! So far we've only covered the virtual environment side of Python. But in real-world development, can one Python version really do everything? Of course not. In my earlier Hexo series I explained how Node.js 8 and Node.js 13 have huge performance differences, and Python is exactly the same — programs get updated, new syntax gets added, bugs get fixed, which also means packages get updated along with the versions.

But sometimes a package we're using, or a particular dev environment, simply has to run under a specific version. In that case you need to switch versions. It's the same on the JavaScript side: for Node.js development we usually use the nvm package to switch Node.js versions.

If you haven't used nvm yet, I'd recommend it. Here are some notes I've written:

  • Installing nvm and switching NodeJS versions on Windows 10 (article in Mandarin)
  • Installing nvm on Mac OS X (article in Mandarin)

Cut back to Python — so... what about Python? Is there a super-handy nvm-like thing for switching versions?

The answer is no, there's no nvm. But there is a thing called pyenv.

Didn't you say there wasn't one?

Hold on, I said there's no nvm — I didn't say there's no other way. So next up, let me show you how to install pyenv.

Installing on Mac

For Mac, I generally recommend installing via Homebrew. If you jumped straight to this chapter, I'd suggest going back to read the earlier ones.

The install command on Mac is:

cmd
brew update
brew install pyenv

If you hit Updating Homebrew... during the install... just give it a moment...

Updating Homebrew...

Once that finishes, you can go straight into setting up pyenv.

Oh, and one more thing — I'd recommend installing these dependencies too:

cmd
brew install openssl readline sqlite3 xz zlib

Setting Up pyenv

Normally you'll need to set up some environment variables for your terminal. On Mac we often use ZSH or Bash, so you'll need to tweak accordingly:

Bash:

cmd
echo 'eval "$(pyenv init -)"' >> ~/.bashrc

ZSH:

cmd
echo 'eval "$(pyenv init -)"' >> ~/.zshrc

That's basically all the setup you need.

Common pyenv Commands

Let me list out some of the commonly used pyenv commands too:

  • Install a specific Python version
    • pyenv install [version]
    • The verbose version of the install command is - pyenv install -v [version]
  • List which Python versions are available to install
    • pyenv install -list
  • Remove an installed Python
    • pyenv uninstall [version]
  • List the Python versions you've already installed
    • pyenv versions
  • Switch Python versions
    • pyenv global [version]
      • switch the Global Python version
    • pyenv local [version]
      • switch the current project's Python version
    • pyenv shell [version]
      • switch the current terminal's Python version

The usage isn't too hard, so I won't walk through each one here.

And with that, this chapter wraps up~
