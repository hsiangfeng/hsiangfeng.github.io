A casual chat about TypeScript — what it is and why use it

Intro

TypeScript has basically become a must-have skill for a lot of developers, but plenty of projects are still sitting on JavaScript and going back and forth about whether to migrate. So this post is a casual share of my own take on TypeScript, and why you'd want to use it.

About TypeScript

TypeScript is an open-source language developed and maintained by Microsoft...

"Wait — so TypeScript is a brand new programming language?"

Not really. TypeScript has a lot of features and its own syntax, but at the core, when you write TypeScript, you still follow JavaScript's rules. TypeScript is essentially a superset of JavaScript — you can use any JavaScript syntax inside TypeScript, but the reverse doesn't work because JavaScript doesn't know TypeScript's syntax.

So I mentioned TypeScript is a superset of JavaScript, but what does "superset" actually mean? Some folks may not be familiar with the term, so let's quickly cover that first.

As a frontend dev, most of the time you're writing JavaScript. JavaScript has different versions — ES5, ES6, ES7, and so on (ECMAScript). Every time we want to use new syntax, we have to think about browser compatibility, because not every browser supports the latest syntax. So we have to use Babel to transpile down to syntax the browsers support, which becomes a hassle (every ECMAScript generation has to wait for browser support before you can really use it).

Each ECMAScript generation is a set, meaning each version includes the syntax from the previous one. Like the diagram below:

ECMAScript sets

So every ECMAScript is a set — you can't get to ES7 and suddenly only be able to use ES7 syntax without ES5. Each ECMAScript version contains the previous one.

With that intuition, let's get back to the famous line:

"TypeScript is a superset of JavaScript."

What's a superset? Like I mentioned, when you write TypeScript, you still follow JavaScript's rules, so TypeScript is essentially still JavaScript. So why is TypeScript called a superset? Because each ECMAScript generation is a set containing the previous one, TypeScript necessarily includes ES5, ES6, ES7, and so on. But you might be thinking:

"If TypeScript just includes ES5, ES6, ES7..., that's just a set — why is it called a superset?"

Simple — TypeScript has features that JavaScript doesn't, like:

  • A type system
  • Interfaces
  • Decorators
  • Generics
  • etc.

Because TypeScript has features JavaScript doesn't, it's called a superset. So the diagram looks like:

TypeScript superset

If you think of TypeScript as an "extended" version of JavaScript, that's also fair — TypeScript really does build on top of JavaScript and add a bunch of things JavaScript doesn't have. So you can call TypeScript an extended JavaScript (though most people call it a superset).

TypeScript features

Next I'll briefly touch on TypeScript's most representative feature — the type system. This post isn't meant to teach you how to write TypeScript, so I'm not going to walk through every feature. The goal is to help you understand why TypeScript is called a superset, and why it's become a must-have skill for many developers.

In regular dev, you might declare a variable and assign a value like:

JavaScript
const myName = 'Ray';

In TypeScript, you can add a type annotation when you declare:

TypeScript
const myName: string = 'Ray';

That way when you read the code, you immediately know what type the variable is without checking its value. One small detail though: if you don't write a type annotation, TypeScript can also use type inference to figure out the variable's type automatically. So you don't have to write annotations, but if you do, TypeScript will respect them:

TypeScript
const myName = 'Ray'; // TypeScript infers myName as string
TypeScript
const myName: string = 'Ray'; // TypeScript uses your annotation

The difference isn't huge — the first one relies on the TypeScript compiler's inference, the second is your explicit annotation. You might wonder what the actual difference is between writing them yourself vs. letting TypeScript infer. Not much, but in larger projects, you'll find auto-inference can be limiting. For example:

TypeScript
let myName = 'Ray'; // TypeScript infers myName as string

myName = 123; // TypeScript errors here, because it inferred myName as string, so assigning a number fails

But if your requirement is that myName should be either string or number, you have to write the annotation yourself:

TypeScript
let myName: string | number = 'Ray'; // now string or number is fine

myName = 123; // no error

Letting TypeScript infer types is convenient and makes the code look cleaner, but for more complex requirements I'd recommend writing annotations yourself — it makes the code clearer and avoids unnecessary errors. Inference is usually best for simple cases — types you could figure out at a glance, or cases where the type is super obvious.

We just touched on TypeScript's flagship feature, the type system. You might be wondering:

"Why does frontend need a type system anyway?"

This actually comes down to JavaScript being a dynamically typed language. You don't declare variable types or function return types — and because of this, JavaScript's types are hard to track. Usually you have to run the code to know what type something is. Let me show you a simple addition example:

JavaScript
··function add(a, b) {
  return a + b;
}

An experienced dev might glance at this and say "the return type is number — as long as you pass in numbers, it always returns number". But a beginner might get tripped up by JavaScript's dynamic typing. Why?

Say we usually call it like this:

JavaScript
add(1, 2);

Then one day your brain stalls and your fingers don't cooperate, and you end up calling it like:

JavaScript
add(1, '2');

Suddenly the return value isn't number anymore — it's string. Now you have to dig into the function to figure out why. So to avoid that, you might write defensive code like:

JavaScript
··········function add(a, b) {
  if (typeof a !== 'number' || typeof b !== 'number') {
    throw new Error('a or b is not a number');
  }

  return a + b;
}

There's nothing wrong with this, but the code becomes really verbose, and you still have to run it before you can find out whether there's a problem.

With TypeScript, you can write:

TypeScript
··function add(a: number, b: number): number {
  return a + b;
}

Now passing in the wrong type makes TypeScript error out:

TypeScript
add(1, '2'); // Argument of type 'string' is not assignable to parameter of type 'number'

With TypeScript's type system, you find out about problems at dev time, not at runtime.

Beyond that, accurate type inference helps you avoid typos and pick the right methods. For example:

TypeScript
const arr: number[] = [1, 2, 3];

When you start typing arr., TypeScript will give you autocomplete suggestions:

autocomplete

So TypeScript both solves JavaScript's dynamic-typing pain and improves the dev experience.

When to use TypeScript, and things to watch out for

We've covered TypeScript's features and why it's called a superset. Now let's talk about when you should use it.

Projects with 2+ people

Personally, I think TypeScript fits best on projects with two or more people. It's not that solo projects can't use TypeScript — it's just that the more devs you have, the more you benefit from a type system, because the chance of unnecessary errors grows with team size.

Say you're working with one other person, and dev A wrote this shared utility:

JavaScript
····················································································const myMoney = (storage) => {
  let money = storage;
  return (price) => {
    return {
      nowMoney: () => money,
      count: (price) => {
        if (money < price) return `insufficient balance, current: ${money} $`;
        if (money > 0) {
          money -= price;
          return money;
        }
        return `deduction failed, current balance: ${money} $`;
      }
    };
  };
};

Dev B then has to read A's code to figure out the return type and the parameter types. With TypeScript, you don't have that problem — TypeScript auto-suggests the return type and parameter types. You just need solid annotations:

TypeScript
························································································type MoneyManager = {
  nowMoney: () => number;
  count: (price: number) => number | string;
};

const myMoney = (storage: number): (price: number) => MoneyManager => {
  let money = storage;
  return (price: number): MoneyManager => {
    return {
      nowMoney: (): number => money,
      count: (price: number): number | string => {
        if (money < price) return `insufficient balance, current: ${money} $`;
        if (money > 0) {
          money -= price;
          return money;
        }
        return `deduction failed, current balance: ${money} $`;
      }
    };
  };
};

Now dev B can see the return type and parameter types at a glance without reading A's implementation. Fewer unnecessary errors, clearer code.

When you want better maintainability

From the earlier examples, you can see TypeScript's type system makes code clearer and avoids unnecessary errors. So if you want better maintainability, TypeScript is a solid pick.

That said, I'd recommend only adopting TypeScript when the project has tests. TypeScript doesn't check logic errors for you — so if you blindly migrate, you might end up changing the original logic while fixing TypeScript errors. So I personally recommend only introducing TypeScript when the project has tests.

For example, original code:

JavaScript
············function calculateTotalPrice(cartItems) {
  return cartItems.reduce((total, item) => {
    return total + item.price * item.quantity;
  }, 0);
}

// test data
const testCartItems = [
  { id: 1, name: 'Book', price: 10, quantity: 3 },
  { id: 2, name: 'Pen', price: 5, quantity: 2 },
];

console.log(calculateTotalPrice(testCartItems)); // should be 40

When you migrate to TypeScript, you'll see TypeScript errors — it'll infer cartItems as any[], so you have to write annotations:

TypeScript
····················type CartItem = {
  id: number;
  name: string;
  price: number;
  quantity: number;
};

function calculateTotalPrice(cartItems: CartItem[]): number {
  return cartItems.reduce((total, item) => {
    return total + item.price * item.quantity;
  }, 0);
}

// test data
const testCartItems: CartItem[] = [
  { id: 1, name: 'Book', price: 10, quantity: 3 },
  { id: 2, name: 'Pen', price: 5, quantity: 2 },
];

console.log(calculateTotalPrice(testCartItems)); // should be 40

During the migration, your code's maintainability did get better through TypeScript, but if you accidentally changed the logic — say, swapping item.price * item.quantity for item.price + item.quantity — and there are no tests, you'd have a really hard time noticing. TypeScript doesn't check logic errors, which is why I'd only introduce TypeScript when the project has tests.

You also can't guarantee you won't accidentally fat-finger something during the migration, so my recommendation is: adopt TypeScript when the project has tests in place.

When your team has a TypeScript baseline

This one I think matters a lot. Some team leads see all the TypeScript hype and want to migrate the whole project, but they forget one thing — does the team actually have a TypeScript baseline? If nobody on the team knows TypeScript, you'll need time to learn it.

If the project schedule isn't pressured, sure, you can introduce TypeScript. But if the schedule is tight, I'd avoid introducing TypeScript, because you'll need time to learn it.

"So we just have to give up on TypeScript?"

Not exactly. Before trying TypeScript, you can try writing JSDoc instead. JSDoc is a comment syntax that lets you add some notes to JavaScript code — it makes the code clearer and gives you some type-checking. For example:

JavaScript
·······/**
 * sum two numbers
 * @param {number} a
 * @param {number} b
 * @returns {number}
 */
function add(a, b) {
  return a + b;
}

Getting used to writing JSDoc will help you when you eventually migrate to TypeScript — conceptually, TypeScript's type annotations look pretty close to JSDoc comments, so the learning curve is gentler. So when the schedule is tight, I'd suggest trying JSDoc first, then migrating to TypeScript later when there's more breathing room.

TypeScript is not a silver bullet

One thing I need to flag — using TypeScript doesn't mean you avoid all of JavaScript's hidden gotchas. Remember:

"TypeScript is essentially a superset of JavaScript, which means the compiled output is still JavaScript."

Don't assume TypeScript saves you from JavaScript's quirks. For example — say you write a form feature with type annotations for every form object. The compiled output is still JavaScript. A simple example:

TypeScript
············type Form = {
  name: string;
  email: string;
  phone: string;
};

const form: Form = {
  name: '',
  email: '',
  phone: '',
};

Compiles to:

JavaScript
············"use strict";
const form = {
    name: '',
    email: '',
    phone: '',
};

See the Pen typescript-example by @hsiangfeng on CodePen.

So the JavaScript logic in your code can still have hidden gotchas. The defensive coding you need to do isn't going away. Don't assume TypeScript solves everything — at the end of the day, it's still JavaScript.

Shared TypeScript conventions

I have to say, TypeScript's type system has cut down on a ton of dev pitfalls and improved the dev experience (DX). I've seen devs writing TypeScript but using any for everything — but does that mean we can never use any?

No, it means you use any based on the need. If you can know the type of a variable, don't use any. If you genuinely can't, then any is fine.

For example, you have a function but don't know its return type. Then any is fair:

TypeScript
··function getSomething(): any {
  // ...
}

The rule of thumb is: avoid any whenever possible, because using any means giving up TypeScript's type checking.

If you're going to write like that, just rename it to AnyScript

Closing thoughts

That's pretty much my take on TypeScript and why I'd use it. Hopefully this gives someone new to TypeScript a starting mental model — and the why behind using it, instead of just adopting it because everyone else is.
