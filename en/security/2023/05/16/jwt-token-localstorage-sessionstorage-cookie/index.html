Token in localStorage? sessionStorage? Or Cookie?

#localStorage #XSS #JWT #Cookie #Security

Intro

This post is about some things I’ve picked up around JWT Tokens — specifically, why you shouldn’t store JWT Tokens in localStorage.

JSON Web Token (JWT)

First, a quick description of what a JSON Web Token (JWT) is. It’s a format based on the JSON standard. Where does it usually come up?

Basically anywhere you need authorization. For example:

  • Login
  • Password reset
  • Updating user profile info

And so on.

So when a user logs in, they get a JWT Token back:

That token becomes the identifier they’ll use for talking to the server going forward. The server uses it to verify their identity.

What a JWT Token is made of

Let’s quickly look at what a JWT Token is built from. Here’s the example token from the official JWT site:

A JWT Token has three parts:

  • HEADER: ALGORITHM & TOKEN TYPE
    • Describes the algorithm used to encrypt this token, and the token’s type
  • PAYLOAD: DATA
    • Stores some data, like the user’s ID, name, etc.
  • VERIFY SIGNATURE
    • Used to verify this token is legitimate

localStorage? sessionStorage? cookie?

Once you log in, you get a JWT Token used to talk to the backend. Before going further, let me quickly recap browser storage.

A few common ones:

  • localStorage
  • sessionStorage
  • cookie

localStorage and sessionStorage are both part of the HTML5 Web Storage API. The difference: localStorage data is stored permanently (or semi-permanently — until the user deletes it) in the browser, while sessionStorage only exists within the current browser tab.

So where should the JWT Token go?

Where should you store your JWT Token? Based on what we covered, JWT Tokens are critical identifiers — you need them to fetch any data from the backend, so they absolutely cannot be leaked. If a malicious actor got hold of one, you’d be looking at a data breach.

But here’s the thing — the JWT Token may look encrypted, but it’s actually reversible. Meaning: anyone who gets hold of a JWT Token can parse it and read what’s inside.

  • sub: Subject — the user’s ID
  • name: the user’s name
  • iat: Issued At — when this JWT Token was issued

The PAYLOAD contents are decided by the backend developer, so you can put anything in there — like the user’s email or name for easy frontend display.

Why? If an attacker can run JavaScript on your site, that means they can read your localStorage data through JavaScript — which means a potential data leak.

So no matter where you put it — localStorage, sessionStorage, or Cookie — there’s a risk. Below are four takeaways to mitigate this as much as possible.

Takeaway 1: Be careful with third-party resources

Basically, if your site can run third-party JavaScript, your data is potentially at risk. This is one of the main reasons to avoid CDNs when possible — you can’t really verify whether a CDN’s resources are safe. Any third-party script — safe or not — opens up the possibility of a data leak.

Not to mention the headaches of a CDN suddenly going down or pushing a breaking update.

Takeaway 2: Stick to good practices

Normally, you’d use HTTP Only cookies to solve this — JavaScript can’t read HTTP Only cookies. But modern dev usually splits frontend and backend, and HTTP Only has to be set by the backend, so the frontend can’t set it directly.

(If you’re not split across frontend/backend, your backend can set HTTP Only cookies just fine.)

If you can’t use HTTP Only, here are some principles I’d follow:

  • Don’t put sensitive info in JWT — passwords, credit card numbers, etc.
  • Use HTTPS
  • Use short-lived tokens
  • Set up CORS allowlists so only specific domains can hit the API
  • Use CSRF Tokens to prevent CSRF attacks
  • Be careful with patterns prone to XSS, like innerHTML and eval

Sticking to these will avoid most issues.

Takeaway 3: Use Cookie

Why not store the token in localStorage? Because localStorage is semi-permanent — data doesn’t expire on its own. So if your token doesn’t expire either, time becomes a risk multiplier. And that’s before we even talk about whether you’ve also stashed other sensitive data in there.

localStorage is super convenient by design, but compared to cookies, I’d lean toward cookies. Cookies have options for setting expiration, so you can let them auto-expire after a while — no need to worry about a stale token sticking around forever.

You can also deliberately set the cookie expiration shorter than the backend token expiration. For example, if the backend token lasts 7 days, you can set the cookie to expire in 3 days. You can also restrict the cookie to specific domains, and add a Refresh Token mechanism.

One caveat: localStorage has a 5MB limit, while a cookie is limited to 4KB. So if your token is huge, you might not be able to fit it in a cookie and have to fall back to localStorage — in which case you really need to lock down the security side.

Takeaway 4: User behavior

If you’ve followed all the above and kept up with security patches, the rest is mostly on the user. If their data leaks because of how they used the site — that’s a user-side issue.

For example, logging in on a public computer and not logging out, or not closing the browser. These aren’t really developer-solvable problems — you can build auto-logout, but no amount of safeguards is foolproof.

Closing thoughts

localStorage is super convenient, but given the potential security issues, I’d recommend storing your token in a cookie with an expiration. That way you can sidestep a chunk of the risk. So in summary:

  • localStorage: semi-permanent storage. Good for data that doesn’t expire. 5MB cap, but watch for security and XSS issues
  • sessionStorage: short-term storage. Good for unimportant data that should disappear on tab close. Same caveats
  • Cookie: short-term storage. Good for expiring data like tokens, but capped at 4KB and watch for CSRF

Every storage option has its place — just stay aware of security, follow some basic principles, and you’ll avoid most of the common pitfalls.
