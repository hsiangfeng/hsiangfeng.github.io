Intro

The IIFE is one of those tricks you see absolutely everywhere — in libraries, in frameworks, in legacy code. So it's worth knowing what it actually is.

The IIFE

So what's an IIFE? The full name is "immediately-invoked function expression" — IIFE for short. In the last chapter we saw that a named function declaration only runs when you actually call it.

js
function fn() {
  console.log('Ray');
}

fn();

An IIFE doesn't wait for you to call it. The moment the code runs, it runs itself.

js
(function() {
  console.log('Ray');
})();

The key to an IIFE is that pair of parens () at the very end. You'll usually see it written one of two ways:

js
(function() {
  console.log('Ray');
})();

(function() {
  console.log('Ray');
}());

There's no right answer between these two — both run the IIFE just fine. Notice the IIFE has no name, so it's a kind of anonymous function. But it's different from the anonymous function expressions we saw earlier — a regular anonymous expression has to be called before its body runs.

js
var fn = function() {
  console.log('Ray');
}

fn();

Now someone might wonder — what counts as this?

js
(function fn() {
  console.log('Ray');
}());

That's also an IIFE. The function name isn't actually needed here — there's no point to it — but JavaScript won't warn you that you can drop it.

So the first feature of an IIFE is pretty obvious: it runs immediately (well, duh). But there's another one. Once an IIFE finishes, you can't call it again — not from outside, not at all — and any variables declared inside it are only accessible within the IIFE.

js
(function fn() {
  console.log('Ray');
  var a = 'Hello';
}());

fn(); // fn is not defined
console.log(a); // a is not defined

Let's look at MDN's description of the IIFE:

An IIFE (Immediately Invoked Function Expression) is a JavaScript function that runs as soon as it is defined.

There's another important point worth noting here:

This pattern keeps the variables inside from polluting the global scope.

What does that mean? The whole point of an IIFE comes down to understanding scope. Before block scope existed — before ES6, basically — pollution was really easy to cause, because var is scoped to the nearest function, not the nearest block. So code like this leaks straight into window:

js
for(var i = 0; i < 10; i++) {
  console.log(i);
}
console.log(i); // 10

We want to avoid polluting window while we're developing. This is exactly where an IIFE helps — it boxes in the scope so the variable i only lives inside the IIFE.

js
(function() {
  for(var i = 0; i < 10; i++) {
    console.log(i); // 0~9
  }
  console.log(i); // 10
})();
console.log(i); // i is not defined

Now you might ask — if the outside can't get at the IIFE's internal variables and functions, can we at least pass arguments in? Yep, we can. Here's how:

js
(function(name) {
  console.log(name); // Ray
})('Ray');

And since an IIFE is itself an expression, you can return a value from it and catch it with a variable:

js
var name = (function(name) {
  console.log(name); // Ray
  return name;
})('Ray');

console.log(name); // Ray

Finally, there's one classic IIFE landmine — the ;. If you don't write your semicolons correctly, you'll hit this error: Uncaught TypeError: (intermediate value)(...) is not a function.

js
(function(name) {
  console.log(name); // Ray
})('Ray')
(function(name) {
  console.log(name); // Ray
})('Ray')

The fix is just to add a ; in front of the opening paren:

js
(function(name) {
  console.log(name); // Ray
})('Ray')
;(function(name) {
  console.log(name); // Ray
})('Ray')

The root cause here ties back to a much earlier chapter — "JavaScript Core Concepts (12): Operators, Types & Grammar — ASI Automatic Semicolon Insertion (article in Mandarin)".

So to wrap up — why use an IIFE at all? The main goal is to avoid polluting the global execution context and causing collisions. In the example below, even though the variable lives inside the function, the function itself is declared directly in the global execution context:

js
function getName() {
  console.log('Ray');
}
getName();

So if there's another variable or function with a similar name floating around, you get a collision.

And sure, ES6 gave us let and const now — but the memory used by a variable declaration can't be freed, so as you declare more and more, performance naturally suffers. The IIFE still has its place.
