JavaScript Core Concepts (11): Statements vs Expressions — The One Everybody Mixes Up

#JavaScript #JavaScript Core #Operators, Types and Grammar

Intro

Statements vs expressions is one of those concepts almost everyone gets fuzzy on — and honestly, that includes me too.

Statements and Expressions

You can get a feel for which is which straight from the MDN docs. There you’ll find what JavaScript statements look like, for example.

Statements include things like:

  • var, let, const
  • if...else
  • for
  • …and so on

Expressions, on the other hand, are basically too numerous to list. The thing is, an expression usually pairs up with an operator and returns a result.

Statements

Basically, a statement doesn’t return a result. Right about now you’re probably thinking — wait, you just said expressions usually pair with an operator and return a result, so why isn’t a variable declaration like var a = 1; an expression? It even returns undefined, and it has an = operator. Doesn’t that make it an expression?

A lot of people get tripped up on why a “variable declaration” counts as a statement. The simplest way to think about a statement: it just sits there quietly waiting for you to call it, and its biggest characteristic is that it doesn’t return any result — but a statement always gets executed once. The reason is that it needs to establish its lexical scope first.

So when you type var a = 1; into the Chrome console and it returns undefined, here’s what’s happening: at declaration time, JavaScript runs through it once because it needs to set aside a chunk of memory for that variable and map that memory to the variable name. But it doesn’t return a result, which is why a variable declaration hands you back undefined. That undefined is really just JavaScript’s way of saying “I’ve got the memory ready for this variable.”

Expressions

An expression is anything that returns a value. The simplest way to see it: type 1+1 and it returns 2 underneath. That’s an expression.

MDN lists a few characteristics of expressions:

Arithmetic: evaluates to a number, for example 3.14159. (Generally uses arithmetic operators.)
String: evaluates to a string, for example “Fred” or “234”. (Generally uses string operators.)
Logical: evaluates to true or false. (Often involves logical operators.)
Primary expressions: basic keywords and general expressions in JavaScript.
Left-hand-side expressions: left values are the destination of an assignment.

So an expression usually comes with some operators, but the simplest idea to hold onto is “it returns a result, or a value.” So when you type qq = 'Ray' and it returns 'Ray', that’s an expression.

The key idea behind expressions is the operator. The equals sign is an expression — it assigns a value to qq and returns the result. Which lets us circle back to why var a = '11'; returns undefined: as I said earlier, a variable declaration returns undefined because it’s preparing a slot in memory for the variable. You’ll notice that anything that’s a statement just returns undefined.

Expressions are the opposite — an expression always returns a result, a value.

As a side note, the setTimeout and setInterval we reach for all the time in dev are expressions too.

There’s one more telltale difference between statements and expressions: a statement can’t be stored in a variable, because it doesn’t return a value.

Function Declarations

Now let’s talk about another flavor of statement-vs-expression. When you declare a function, there are actually two broad ways to do it. The first is what’s called a function declaration.

Basically, when you declare a function this way it doesn’t return a result — same as before, it just returns undefined. This kind of function declaration is also called a named function declaration.

Function Expressions

A function expression usually declares a variable, paired with an equals operator and a function. That’s a function expression.

This way of declaring a function is also called an anonymous function expression. Even though typing it into Chrome still hands you back undefined, what’s actually happening is that because this function has no name, it gets returned and stored in the fu variable. The big distinguishing feature of a function expression is that it isn’t affected by hoisting — so if you try to call it before the function expression, you’ll get an error.

Finally, one extra special case. JavaScript has a thing called a block, which is just {}. In MDN, a block is also classified as a statement, so you can write:

But if you rewrite it as an object literal, you use “:“ to separate the property from its value.

The big difference between the two is that an object literal is an expression — and an expression can be stored in a variable.

A block, though, can’t be — because it’s a statement.

So the bottom line: if it returns a result or a value, it’s an expression. If it doesn’t, it’s a statement.

The Hybrid

This name is something I kind of made up, mostly to explain var a = 1; and var a = function() {}.

I bet after reading the above, a lot of people are going “hang on, var a = 1; should be an expression, not a statement, right? It’s doing assignment!”

Right — and saying it’s an expression is also correct. The reason is that most of JavaScript is actually a hybrid (expression + statement). For example, we know an if conditional is a statement, but a conditional mainly accepts expressions, so you can write it like this:

If you really want to split hairs, the right way to describe it is a hybrid (expression + statement), since it has the traits of both the statement var a; and the assignment a = 1;. But I’d suggest understanding them separately — it’s less confusing that way.
