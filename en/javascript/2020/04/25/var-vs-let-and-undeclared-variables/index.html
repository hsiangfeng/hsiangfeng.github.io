Intro

Plenty of people online have already covered the difference between var and let, and what happens when you forget to declare a variable at all. So here's my take — the biggest differences between these two, plus the gotcha with skipping the declaration keyword.

What happens when you skip the declaration keyword

Honestly the heading should be "the difference between using var and not using it", but in plain English we tend to say "declaring a variable or not". Before I get into it though, you need to know two concepts:

  • The global environment
  • The global object

I already covered both of these in [JS Weird World] No.3 Global Environment and Global Object (article in Mandarin), so I won't repeat them here. Let me jump straight to the point.

In normal situations, the biggest difference between using var and not using it comes down to one thing: "can it be deleted with delete?"

delete is the syntax for removing a property from an object — there's a write-up on MDN. So we can use the snippet below to see the difference between declaring with var and not:

JavaScript
var a = 'a';
delete a; // false

b = 'b';
delete b; // true

That tells us something important: a variable created without var gets added as an object property. So I'd strongly recommend you always declare variables with var — a variable that can be deleted is an easy way to introduce system bugs.

I'm leaving let and const out of this part for now.

Scope

Next let's look at the scope of var versus let and const. Generally the advice is to swap all your old var declarations over to let — but what's the actual difference between let and var?

Start with var. Its scope is function scope, which means the variable's scope is decided by the function it lives in. For example:

JavaScript
··function fuA() {
  var a = 'a';
}
fuA();
console.log(a); // a is not defined

We can't reach the variable declared inside fuA, even though we've already called and run it.

What about let? Its scope is block scope. The easiest way to spot it: anywhere there's a pair of { } curly braces, that's its scope.

JavaScript
··if(true) {
  let a = 'a';
}

console.log(a);

If you try to compare the two using functions, you won't see the difference, because the scopes look the same.

JavaScript
····function fuA() {
  var a = 'a';
}
fuA();
console.log(a); // a is not defined

function fuB() {
  let b = 'b';
}
fuB();
console.log(b); // b is not defined

But once you use something like an if block, the difference shows up:

JavaScript
····if(true) {
  var a = 'a';
}

console.log(a); // a

if(true) {
  let b = 'b';
}

console.log(b); // b is not defined

So the upside of declaring with let is that we can be sure we won't pollute the global object — because any variable that isn't declared inside a function ends up polluting the global object. The if example above pollutes it:

Polluting the global object with var

You might ask: "What about const?" Same as let, basically.

I'm not going to get into Hoisting here — maybe another time.

Why you can't find a let variable on window

This is something almost nobody mentions. Back in the day, after declaring a variable like this, you could always find it under window:

JavaScript
var aName = 'Ray';
console.log(window);

window

But once we use ES6 syntax — let or const — that variable is nowhere to be found:

JavaScript
let aName = 'Ray';
console.log(window);

Can't find the variable declared with let

And yet, if we reference aName directly, we still get the declared variable back just fine:

Not on window but still retrievable

So what's the deal? In our usual mental model, declared variables get added to window. But a variable declared with let can't be found under window — even though you can still call it and get the right value. And if you try delete aName, you'll also get a false, meaning it can't be deleted.

Is it magic? Nope.

The reason lies in the global space. The global space is actually made up of two environments:

  • The global object — Object Env
  • The declarative environment — Declare Env

So the global space is really a two-environment thing, and normally we never see the Declare Env.

var is declared based on the Object Env and added to the Declare Env, while let and const are only declared in the Declare Env. That's exactly why we can't see variables declared with let or const on window, but can still get their values just fine.
