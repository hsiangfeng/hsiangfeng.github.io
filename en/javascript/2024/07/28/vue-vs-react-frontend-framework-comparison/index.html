Vue vs React: What's Actually Different, and How to Pick One (Plus Common Interview Questions)

#JavaScript #Vue #React #Interview Questions

Intro

A while back I was at a job interview and the interviewer asked: “What’s the difference between Vue and React, and why is React more popular?” It’s a pretty common question — and an interesting one — so I figured I’d write up my own take.

Note
I’m mainly a Vue developer, but I do work on React projects occasionally. I’ll try to stay objective in this post — if I’m wrong about something, corrections are welcome.

What’s actually different between Vue and React?

Before getting into the differences, there are a few things to clear up first:

  1. Why each one was created, and their official definitions
  2. Their core concepts
  3. Their ecosystems

Those three are the ones I think matter most for telling React and Vue apart. I’ll use examples throughout to make the differences more concrete.

But first, a quick conclusion — whether it’s React or Vue, both of them are…

“Implementations of JavaScript.”

Note
“Implementation” here means both Vue and React are built using JavaScript — so they don’t actually change JavaScript’s language features (variable declaration, pass-by-reference, etc.). It’s all still JavaScript underneath.

Why they were created

Both React’s and Vue’s official sites describe what they are. Quick screenshots:

Their own definitions:

  • React: “The library for web and native user interfaces”
  • Vue: “The Progressive JavaScript Framework”

Notice the difference: React calls itself a Library, Vue calls itself a Framework. That matters. We often say React, Vue, and Angular are the “big three frameworks”, but technically React is not a framework — it’s a library. There’s an impostor among us.

As for why each was created, the best way to understand is to watch the Honeypot documentaries — they explain the history really well.

React documentary:

Vue documentary:

If you can’t watch them, the short version:

“Both were trying to solve the problem of browser UI updates, and to make web dev faster and easier to maintain.”

For the full story I’d really recommend watching the documentaries — the interviews are great and they really capture each one’s origin.

So why does React get lumped in with the “big three frameworks”?

Here’s the obvious question: if React is a library, why do we group it with Vue and Angular? A few reasons:

  1. In practice, React is always paired with other packages — React DOM, React Router, Redux, etc. The combination acts like a framework.
  2. React showed up one year before Vue (React in 2013, Vue in 2014), and React’s community ecosystem is massive — enough that you can build a full app on top of it.
  3. React did simplify a lot of dev work. With React DOM generating HTML, React Router handling routes, etc., you can focus on building rather than wiring infrastructure.

Add all that up, and React-the-library acts enough like a framework in practice that we just call it one.

Note
I’ll keep calling React a Framework for the rest of this post — no need to be a pedant about it now that I’ve explained the reasoning.

Framework vs Library

Okay, so why is React called a framework when it’s technically a library? Let’s dig into Framework vs Library.

A Library is like a toolbox in the real world — there are nails, saws, screwdrivers, all independent tools you pick and combine as you like. That’s the concept.

jQuery is a great example. jQuery itself is a library — it gave you methods like $.ajax(), $('DOM').click() to develop quickly. Before React showed up, a lot of folks built everything on jQuery. Just look at how many plugins depended on it — Owl Carousel, Fancybox, pre-v5 Bootstrap, and so on.

So a Library just gives you tools — what you do with them is up to you. Can you build a SPA (Single Page Application) with jQuery? Of course — you just need to write the supporting code yourself. Same with React: pair it with React DOM for rendering, React Router for routing, etc.

A Framework, by contrast, is a more complete solution. It hands you a full dev environment ready to use, but you’re constrained by its conventions and the author’s worldview. That’s why people say Vue has lots of “extra things to learn” — on top of JavaScript itself, you also need to learn Vue syntax like v-for, v-if, v-model.

The same goes for Angular — you have to learn *ngFor, *ngIf, *ngModel, etc. Same idea as Vue, just different syntax. I’m not super familiar with Angular so I won’t go deep there — let’s stay focused on React vs Vue.

Back to the practical side. When you build a React web app, you need to include two things: React and React DOM. They’re separate tools with different jobs:

React itself provides basic syntax and APIs like React.createElement() and React.Component. They define components and describe their behaviour and state. But if you only include React without React DOM, you can’t actually render anything to HTML — because React only handles UI construction and state management, not rendering.

To render React components to HTML, you need React DOM. It provides methods like ReactDOM.render() that convert React components into browser-renderable HTML. So React DOM is the bridge — that’s why we include both when building a React app.

React DOM uses the Virtual DOM concept. Every React dev runs into it. The short version: Virtual DOM converts React components into JavaScript objects describing the structure and props, then diffs against the real DOM. But that’s a separate deep dive — let’s skip it here.

And Vue? Vue devs only need to include one CDN:

That’s because Vue bundles the rendering layer inside itself. Which fits the Framework concept — everything’s wired up for you, just start building.

Note
I’m skipping Router and similar packages — those are SPA concerns, and both Vue and React leave the choice to you. In practice both work with a Router.

So in short: Framework wins on speed-of-getting-started, loses on flexibility (you’re locked into its conventions). Library is the opposite — it gives you tools and freedom but you have to assemble more yourself.

Origins and philosophy

Now a quick run through each one’s origin and design philosophy. Won’t go too deep — these are huge topics. The documentaries cover them in detail.

React

React was originally built by Jordan Walke at Facebook. He started it to solve UI update and maintenance problems.

Facebook is huge, and as users grew the UI got harder and harder to maintain — hard to extend, hard to keep clean. A big site can’t be built by one person, so a lot of folks need to collaborate. Did Facebook have an internal framework? Yes — two engineers built one called Bolt.js, based on the MVC (Model-View-Controller) pattern.

MVC was fine for that era, but as I mentioned above, frameworks have rigid conventions. That rigidity ended up causing problems for Facebook’s devs, so Jordan started thinking about how to fix it.

Note
Facebook later became Meta Platforms — Facebook was the original company name.

With maintainability and scalability in mind, Jordan built FBolt (Functional Bolt) based on functional programming concepts. FBolt was the precursor to React.

I won’t go deep into functional programming here — it’s a huge topic. The relevant point: React is built around functional programming principles. If you’re curious, I’ve written about it before: A casual look at Functional Programming in JavaScript (article in Mandarin) and A casual look at Functional Programming in JavaScript: Currying (article in Mandarin).

Back to the main thread: React was born to solve UI update and maintenance problems.

Note
React’s launch wasn’t smooth. It broke with mainstream architectural patterns, so devs at the time weren’t super receptive. But over time, as React evolved and the community grew, it became mainstream.

React really took off when Facebook acquired Instagram. Instagram only had Android and iOS apps then, no web version. After the acquisition, Facebook decided to build the Instagram web app in React — that was the moment React really got used in production.

There are a lot of people behind React’s rise — it broke conventions, and without the right people supporting it, React wouldn’t be where it is today.

Vue

Now Vue. Vue was created by Evan You. Before Vue existed, Evan rebuilt a web version of the Clear app (a TodoList app), which got him recruited into a Google experimental team — he didn’t even apply, they called him.

His first framework at Google was Backbone.js, also based on MVC. He felt Backbone didn’t help him work efficiently enough on his assigned task, so he started thinking about how to improve his workflow.

He also took inspiration from Angular, but Angular’s a framework too — and as I said earlier, frameworks come with constraints. Angular didn’t solve Evan’s problems either. So he decided to build a new tool to address his own dev frustrations. Notice: originally Vue wasn’t a Framework, it was a Library. It only became a Framework over time.

Note
Vue was originally called Seed.js, but that name was taken. Evan used “View” as the base, ran it through Google Translate, found the French translation “Vue”, and went with that.

So Vue started as Evan trying to solve his own dev efficiency problem — keep things simple, ship fast. He never imagined it’d become one of the top frontend frameworks.

So Vue’s origin story is: built to solve dev pain and improve efficiency.

Like React, Vue wasn’t immediately well-received. With one person maintaining an open source project, that’s risky — if Evan stops, the project dies. Then Taylor Otwell, creator of Laravel (a PHP MVC framework), started looking for a frontend library/framework to pair with Laravel.

Did Taylor try React first? Yes, he did. But for someone not deeply specialized in JavaScript, React has a steep learning curve — it asks for solid JavaScript knowledge and uses functional programming as its design pattern. For a non-JS-focused dev, that’s a big jump.

So Taylor found Vue. Vue had a much lower curve for non-JS-focused devs — basic JS plus some Vue syntax, and you’re off. That’s why Taylor chose Vue, which got Vue community attention and support, and from there it became mainstream.

Why is React more popular?

This part is a bit controversial, and I’m a Vue dev who also writes React, so I’ll try to stay objective. If I’m wrong, corrections welcome.

Ecosystem

For ecosystem size, React really is bigger than Vue. React is a Library, so pair it with the right packages and you can build all kinds of apps — React + React DOM for web apps, React Native for cross-platform mobile (Android + iOS).

Vue is a framework, so it’s somewhat tied to web by default. Vue Native exists, but it really just converts Vue syntax into React Native — so it’s not an independent project, more of a translator. (There are other workarounds but I won’t go deep here.)

Note
Vue Native is no longer maintained. NativeScript is an alternative, or you could go with Flutter (Google’s cross-platform language supporting Android, iOS, Windows, macOS).

Comparing them side by side, you can see why JS-focused devs find React friendlier — it’s really just JavaScript. With Vue you also have to learn Vue’s syntactic sugar. The most obvious example: Vue uses v-for for lists, while React uses native JavaScript’s Array.prototype.map().

That’s why some devs jokingly say Vue is “a lot of magic” — without digging into Vue internals, you might not understand how v-for actually works. React doesn’t have that problem because React is just JavaScript.

Note
React mostly uses JSX to write markup. JSX is React’s syntactic sugar that lets you write HTML-like code inside JavaScript — keeps things intuitive and easy to read.

Easier debugging

Because React is a Library, debugging might be easier than Vue — React is basically JavaScript, with less API abstraction. Vue, on the other hand, gives you a lot of syntactic sugar — when the sugar misbehaves, you sometimes have to dig into how it works to figure things out.

Say v-for isn’t rendering what you expect, or v-if isn’t matching the condition you think it is — you’d have to read Vue’s docs or source to understand. With React, you’re just using JavaScript’s Array.prototype.map() and conditionals, which are language basics — so debugging tends to be more direct.

Perception

I had a couple of US-based friends ask their dev friends what they think of Vue. Most of them knew Vue exists, but hesitated to actually use it — mainly because the creator is Chinese. That association makes them more conservative, even though Vue is open source community-maintained. So when picking a stack, they tend to default to React.

Note
This is just my own observation — not representative of everyone. Plenty of folks outside Taiwan enjoy Vue’s dev experience.

Backing

As I mentioned, React is backed by Facebook, Angular by Google. Vue isn’t backed by any single company — it’s entirely community-maintained. For some companies, that’s a worry — what if Vue stops being maintained, what about its future direction? React is open source too, but with Facebook behind it.

That said, the community-only model is also Vue’s strength. Not being tied to a company means Vue’s direction is decided by the open source community, so Vue can pick up good ideas from anywhere and bring them in. That’s what’s special about Vue — it’s all the community and the contributors, no corporate gatekeepers.

Conclusion

Through all of the above, both React and Vue were born for a reason. React was solving UI update and maintenance problems; Vue was solving dev efficiency.

From my teaching experience, total beginners tend to find Vue more rewarding — you only need to learn a handful of Vue syntax sugars plus basic JavaScript and you can build a real web app. React’s different — you have to think about a lot of details yourself, like memory management and rendering behaviour. Vue handles most of that for you.

Note
“Total beginner” here means someone with zero programming background.

I don’t think it’s up to devs to decide which is “better”. Every dev gets to pick their favourite language, framework, library — that’s subjective. The real arbiter is industry: which one actually solves business problems, ships fast, improves productivity. That’s what matters.

Vue is one of the big three frameworks because it really does solve industry problems. So is React. Both earned their spot.

I’ve seen plenty of projects start with Vue to ship fast, then slowly migrate to React. And plenty going the other way too. So my take is: there’s no right or wrong technology — only fit.

For a startup needing to ship an MVP fast to grab market share, my own bias leans towards Vue. Frameworks have the speed advantage — everything’s wired up, just follow the conventions. That’s also true for larger projects — picking Vue doesn’t mean you’re destined to be reassigned to backend or lose career progression.

Whether you write Vue or React, you’ll find they share a lot — lifecycles, state management, all the things every frontend dev needs to know. So instead of fixating on Vue vs React, invest your time in solid JavaScript fundamentals. Because…

“It’s all just JavaScript.”

That’s the real question to think about. For frontend devs, these are just tools. What matters is your logic, your JavaScript depth, and your problem-solving. Maybe one day Vue or React will fade, but JavaScript — one of the foundation stones of the web — isn’t going anywhere any time soon.

Note
Languages can fade too — that’s why I said “any time soon”.
