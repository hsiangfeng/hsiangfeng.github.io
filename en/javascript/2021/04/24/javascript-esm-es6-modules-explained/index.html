Intro

First off, what is ESM? ESM stands for "ES6 Modules" — also called "JavaScript Modules." So in this post I want to walk through what ESM actually is.

ES6 Modules / JavaScript Modules

Way back, JavaScript was mostly used for DOM stuff on web pages — effects, interactions, that kind of thing. But over the last few years JavaScript has grown to do a lot of complex work: SPAs, backend development, you name it. And as that complexity kept piling up, developers had to start thinking about how to modularize all this code.

So why modularize? Let me give you a scenario.

Say you've got a web page where two different pages both need to render data on screen. So you've got two files, app1.js and app2.js, and the contents look like this:

(Heads up: this is code I wrote in my early days, so forgive how ugly and rough it is.)

js
/* app1.js */
function updataList(e) {
  var select = e.target.value;
  listTitle.textContent = select;
  var str = '';
  for (var i = 0; i < listLen; i++) {
    if (select == data[i].Zone) {
      str += `<div class="col-md-6 py-2 px-1">
        <div class="card">
          <div class="card bg-dark text-white text-left">
            <img class="card-img-top bg-cover" height="155px" src="` + data[i].Picture1 + `">
            <div class="card-img-overlay d-flex justify-content-between align-items-end p-0 px-3" style="background-color: rgba(0, 0, 0, .2)">
             <h5 class="card-img-title-lg">` + data[i].Name + `</h5><h5 class="card-img-title-sm">` + data[i].Zone + `</h5>
           </div>
          </div>
          <div class="card-body text-left">
              <p class="card-p-text"><i class="far fa-clock fa-clock-time"></i>&nbsp;` + data[i].Opentime + `</p>
              <p class="card-p-text"><i class="fas fa-map-marker-alt fa-map-gps"></i>&nbsp;` + data[i].Add + `</p>
            <div class="d-flex justify-content-between align-items-end">
              <p class="card-p-text"><i class="fas fa-mobile-alt fa-mobile"></i>&nbsp;` + data[i].Tel + `</p>
              <p class="card-p-text"><i class="fas fa-tags text-warning"></i>&nbsp;` + data[i].Ticketinfo + `</p>
            </div>
          </div>
        </div>
      </div>`
    };
  };
  list.innerHTML = str;
};
/* app2.js */
function updataList(e) {

  listTitle.textContent = select;
  var str = '';
  for (var i = 0; i < listLen; i++) {

        <div class="card">

           </div>

          <div class="card-body text-left">
              <p class="card-p-text"><i class="far fa-clock fa-clock-time"></i>&nbsp;` + data[i].Opentime + `</p>

    };
  };
  list.innerHTML = str;
};

Notice how we had to write the exact same code twice across two files? And if there were more pages, we'd have to repeat it even more times.

So naturally, we'd want to pull out the repeated, reusable chunks of code and treat them as a module instead.

Node.js

Over in Node.js, the approach is what we call CommonJS — originally named ServerJS. Why am I singling out CommonJS? Because, like I mentioned, it was a standard that came out of developers trying to figure out how to modularize complex code. It mainly uses module.exports and require:

js
/* exports.js */

module.exports = 'Hello World'

/* main.js */

const hello = require('exports');

console.log(hello); // Hello World

One thing worth noting: require loads synchronously, so it has to finish running before any of the code after it will run.

Back to the main topic. ESM mostly uses import and export. If you've used webpack before — heads up, this ESM intro isn't about the webpack side of things, it's about writing ESM directly in the browser. ESM basically absorbed the best parts of AMD and CommonJS, which is why it's considered such an important piece of JavaScript syntax.

Now, the key thing if you want to use it: you have to add type="module" to your <script> tag. Otherwise you can't use ESM at all.

On top of that, let me walk through a few key points about ESM and how to use it.

ESM is strict mode by default

Once you're in ESM, your code automatically becomes 'use strict' (strict mode). Whether or not you actually wrote 'use strict', it's always strict mode.

So what's strict mode? JavaScript is a pretty loose language, so if you don't have a solid grip on it, you're going to step on a lot of landmines while developing. Strict mode exists to keep us from stepping on those weird landmines. A really common example is typos:

js
var person;
persom = {};
console.log(persom);

Non-strict mode

Without strict mode, this throws no error and even runs just fine. Once you turn on strict mode, it's a completely different story:

js
"use strict";
var person ; 
persom = {};
console.log(persom);

Strict mode

So now you've got a basic feel for strict mode. Next, try pasting the code below into CodePen — you'll see the same error message pop up. I won't bother re-posting the screenshot here.

html
<script type="module">
  var person;
  persom = {};
  console.log(persom);
</script>

Independent scope

Every module has its own independent scope, and you can't grab values across scopes. What does that mean? Each script type="module" gets its own little space, and you can't reach into another space to get its variables or methods.

For example, normally if we write it like this, we can grab values across scripts:

html
<script>
  var person = 'Hello Ray';
</script>

<script>
  console.log(person); // Hello Ray
</script>

But once you add type="module" to the script tag where you declare the variable, it's a whole different story:

html
<script type="module">
  var person = 'Hello Ray';
</script>

<script>
  console.log(person); // ReferenceError: person is not defined 
</script>

So when you're using ESM, make sure you're at least aware of these two things — otherwise you'll keep stepping on landmines.

Named export and import

Next let me cover what a named export is. The concept is actually pretty similar to named functions vs. anonymous functions — when you export a module, it has a name, same idea. As mentioned earlier, you mainly use export, and named exports aren't really restricted: you can export objects, variables, functions, whatever:

js
/* app.js */
export const myName = 'Hello Ray';

export sayHi() {
  console.log('Hello Ray');
}

export const obj = {
  myName: 'Ray',
}

export const arr = [1, 2, 3];

The biggest thing about named exports is that you can have multiple exports in a single file. On the import side you use import, and since named exports already have names, you usually import them like import { ... } from './app.js':

html
<script type="module">
  import { arr, obj, sayHi, myName } from './app.js';
  console.log(myName); // 'Hello Ray'
  sayHi(); // Hello Ray
  console.log(obj.myName); // Ray
  console.log(arr[0]); // 0
</script>

If you don't feel like importing them one by one, you can also import everything at once — though honestly this isn't really recommended in practice, because it makes debugging hard. Note that the * below means "import everything," and as assigns it to a new variable called app:

html
<script type="module">
  import * as app from './app.js';
  console.log(app.myName); // 'Hello Ray'
  app.sayHi(); // Hello Ray
  console.log(app.obj.myName); // Ray
  console.log(app.arr[0]); // 0
</script>

And of course you can use as to rename things:

html
<script type="module">
  import { obj as obj2 } from './app.js';
  console.log(obj2.myName); // Ray
</script>

Default export and import

Basically you can think of a default export like an anonymous function, so when you import it you have to give it a variable to receive it. The most common way to write a default export is as a single object:

js
/* app.js */
export default {
  sayHello() {
    console.log('Hello Ray');
  },
  myName: 'Ray',
  obj: {
    myName: 'Ray',
  },
  arr: [1, 2, 3],
};

Like I just said, a default export is like an anonymous function, so when you import it you have to give it a name to receive it:

html
<script type="module">
  import app from './app.js';
  console.log(app.myName); // 'Ray'
  app.sayHello(); // Hello Ray
  console.log(app.obj.myName); // Ray
  console.log(app.arr[0]); // 0
</script>

Side Effect modules

At this point you're probably wondering — how did the old libraries pull off exports and imports? jQuery, for instance?

This is where Side Effects come in. Back before ESM existed, you basically used the IIFE (Immediately Invoked Function Expression) trick to do it:

js
/* jQuery.js */
(function(global){
  global.$ = {
    sayHello() {
      console.log('Hello Ray');
    },
    myName: 'Ray',
    obj: {
      myName: 'Ray',
    },
    arr: [1, 2, 3],
  }
})(window);

/* app.js */
console.log($.myName); // 'Ray'
$.sayHello(); // Hello Ray
console.log($.obj.myName); // Ray
console.log($.arr[0]); // 0

You might be wondering at the end here — why even learn this? Basically, if you understand how these fundamentals work, then down the road when you run into this kind of code, or when you want to build a more complex system yourself, you're definitely going to lean on these techniques.
