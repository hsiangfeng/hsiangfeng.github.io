Intro

This post is about how to add a JavaScript style manager to your own project. If you want your project's code to have a baseline level of quality, then ESLint is basically a must-have.

ESLint

ESLint

First, let's get a feel for what ESLint actually is. ESLint is a static analysis tool. Its main job is to help you catch problems while you're still writing JavaScript, rather than after the project is already deployed. A big reason this matters comes down to the nature of the language itself — JavaScript is really loose about how you write it, so it's easy to introduce bugs. That's why it's generally a good idea to add ESLint when you're building a project: it unifies the coding style as much as possible and surfaces problems early.

Installing ESLint

ESLint needs Node.js and npm. If you're not sure whether your machine has them, try running these in your terminal:

cmd
node -v
npm -v

If a version number shows up, you've got them installed. If not, here are a couple of my other posts you can use to get set up:

  • Installing nvm on Mac OS X (article in Mandarin)
  • Installing nvm on Windows 10 to switch Node.js versions (article in Mandarin)

Once Node.js and npm are confirmed, run this in your terminal:

cmd
npm install eslint --global

(If you're not comfortable with npm commands, check out my post "The difference between npm install commands (article in Mandarin)".)

After it finishes installing, run eslint -v to check for a version number. If nothing shows up, the install didn't go through — just go back a step and run npm install eslint --global again.

Using ESLint

Next, create a new project folder with a single file in it:

  • a.js

Here's the content for a.js:

js
var a = 'Ray'

fn()

function fn() {
  console.log("Hello Ray");
}

Once we install ESLint in a moment, this sample code is going to get those squiggly underlines.

Then run the following commands in order:

cmd
npm init -y

This should create a package.json for you. That file mainly records the packages we'll install shortly.

package.json

Now run the ESLint init command inside the project:

cmd
eslint --init

When you run that, ESLint asks you a bunch of questions. Just answer based on what you need — here's what I picked:

  • How would you like to use ESLint?
    • To check syntax only
    • To check syntax and find problems
    • To check syntax, find problems, and enforce code style
  • What type of modules does your project use?
    • JavaScript modules (import/export)
    • CommonJS (require/exports)
    • None of these
  • Which framework does your project use?
    • React
    • Vue.js
    • None of these
  • Does your project use TypeScript?
    • No
    • Yes
  • Where does your code run?
    • Browser
    • Node
  • How would you like to define a style for your project?
    • Use a popular style guide
    • Answer questions about your style
    • Inspect your JavaScript file(s)
  • Which style guide do you want to follow?
    • Airbnb: https://github.com/airbnb/javascript
    • Standard: https://github.com/standard/standard
    • Google: https://github.com/google/eslint-config-google

Note If you're not very comfortable with JavaScript yet, I'd suggest picking Standard. Going straight to Airbnb can be really frustrating, so I'd learn Standard first, get a decent handle on it, and then move to Airbnb. I'll cover the differences between these styles further down.

  • What format do you want your config file to be in?
    • JavaScript
    • YAML
    • JSON
  • Would you like to install them now with npm?
    • No
    • Yes

Once you've answered everything, ESLint will recommend installing a VSCode extension:

VS Code ESLint extension

This extension lets you see ESLint's results right inside the editor. Without it, you'd have to run commands to find out what's wrong, which is honestly too much hassle. So I'd definitely install it. Once it's in, you'll see a whole bunch of red squiggly underlines.

ESLint

A look at the ESLint styles

Airbnb

Airbnb

The Airbnb style comes from Airbnb — yep, the same company you use to book places to stay when you travel. The style guide is open source on GitHub.

In there you'll find all of their style recommendations and explanations.

In short, Airbnb's style pushes you toward ES6 syntax as much as possible — things like arrow functions, let and const, and not calling a function before it's declared. Taking the earlier example and applying it, the code becomes this:

js
const a = 'Ray';

function fn() {
  console.log('Hello Ray');
}
fn();

console.log(a);

Airbnb

Standard

Standard

Standard is a lot more relaxed than Airbnb. It doesn't force you to use ES6, and its site lays out the main rules. Here's the gist:

Two spaces for indentation Single quotes for strings — except to avoid escaping No unused variables — this one catches tons of bugs! No semicolons — it's totally fine, really! Never start a line with (, [, or `Add a space after keywords if (condition) { ... }Add a space after the function name function name (arg) { ... } Always use === instead of == — but obj == null is okay for checking null || undefined. Always handle the err parameter in Node.js Always prefix browser globals with window — except for document and navigator Avoid poorly named globals like open, length, event, and name

For me, the biggest thing about Standard is not having to write trailing semicolons. Applying it to the earlier example, you get this:

js
const a = 'Ray'

fn()

function fn () {
  console.log('Hello Ray')
}

console.log(a)

Standard

Google

Google

Last one is Google's style. Honestly I'm not very familiar with this one, but going by the Google style, the example would turn into this:

js
const a = 'Ray';

/**
 * This is a function
 */
function fn() {
  console.log('Hello Ray');
}

fn();

console.log(a);

The main thing I noticed after playing with it a bit is that it wants you to add a comment to every function (kind of unusual). Other than that, the syntax is pretty close to Airbnb.

Google

So which style is best? Each one has its own fans. For me personally, I'm more used to Airbnb, but I also love that Standard skips the semicolons. There's no absolute right answer here — it really comes down to your team and your own habits.

One last thing: if you're looking for how to add ESLint to a Vue CLI project, I'd point you to How to add ESLint to Vue CLI (article in Mandarin). Vue has its own ESLint tweaks, so if you run eslint --init yourself you might hit some issues that are a pain to deal with.

And if you're working with Laravel-Vue, check out Fixing ESLint not working statically in VSCode on Laravel-Vue (article in Mandarin).
