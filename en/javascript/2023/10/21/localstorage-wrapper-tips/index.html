A few tricks for wrapping localStorage

Intro

When you're building web apps, you end up using window.localStorage to store data all the time. The thing is, the window.localStorage API itself isn't great to work with — so we can wrap it to make it nicer.

window.localStorage

window.localStorage is a Web API the browser gives you. If you do web dev at all, you've definitely used it — it lets you store data in the browser, and that data sticks around even after you close the browser. There's also window.sessionStorage, another browser Web API, but with that one the data disappears when the browser closes. So between these two Web APIs, you've got your options for storing data.

This post focuses on window.localStorage, since its whole deal is that the data survives closing the browser. That makes it good for stashing important stuff — login info, user preferences, that kind of thing.

Notewindow.sessionStorage works and is used exactly the same way as window.localStorage — the only difference is its data vanishes when the browser closes, so it's better for less important stuff like temporary data. One interesting catch though: window.localStorage shares data across different pages/tabs, but window.sessionStorage doesn't. Makes sense — its data dies when the browser closes, so it can only share data within the same page.

window.localStorage gives you these methods:

  • window.localStorage.setItem(key, value): add a piece of data
  • window.localStorage.getItem(key): get a piece of data
  • window.localStorage.removeItem(key): delete a piece of data
  • window.localStorage.clear(): delete everything

After you run window.localStorage.setItem('myName', 'Ray'), you can open your browser's dev tools, go to the Application tab, and find your new data under Local Storage — like this:

Firefox

Note Every browser shows this in roughly the same way. I'm using Firefox, so yours might look slightly different, but it's close enough — go ahead and try it yourself.

The downsides of window.localStorage

So the window.localStorage API really is that simple, but it's got a few rough edges. What do I mean?

window.localStorage can actually store objects — for example...

js
window.localStorage.setItem('myObj', { name: 'Ray', age: 18 })

But when you try to read it back with window.localStorage.getItem('myObj'), you'll get [object Object].

Is that a bug? Do you need some special method to parse it? Nope — the reason is that window.localStorage can only store strings. So when you want to store an object, you have to turn it into a string first:

js
window.localStorage.setItem('myObj', JSON.stringify({ name: 'Ray', age: 18 }))

That stores the object correctly.

And that's not the only gotcha. Like I just said, since window.localStorage only stores strings, when you read data out you have to turn the string back into an object:

js
const myObj = JSON.parse(window.localStorage.getItem('myObj'))

Which leads to yet another problem. If window.localStorage.getItem('myObj') returns null, you're fine — JSON.parse(null) returns null and doesn't throw. But if your value is undefined, then JSON.parse(undefined) blows up:

js
JSON.parse(undefined) // Uncaught SyntaxError: JSON.parse: unexpected character at line 1 column 1 of the JSON data

Or maybe you forgot to stringify the value when saving:

js
window.localStorage.setItem('myObj', { name: 'Ray', age: 18 })

JSON.parse(window.localStorage.getItem('myObj')) // Uncaught SyntaxError: JSON.parse: unexpected character at line 1 column 2 of the JSON data

The quickest fix is to wrap it in try...catch, or return an empty object when you hit null. But that makes the code kind of ugly, so instead let's solve it by wrapping localStorage in our own helper.

Wrapping window.localStorage

There are a bunch of ways to wrap it. Some people use a Class, some use a Function — I'll show both. Personally I lean toward the Function approach, so I'll start there and cover the Class version afterward.

Usually we put wrappers like this in a utils folder, so let's create localStorage.js inside utils and start wrapping.

The first thing to wrap is reading data, so let's add a get method that wraps window.localStorage.getItem(key):

js
export const getStorageItem = (key) => {
  return window.localStorage.getItem(key)
}

Simple, right? But it's not enough. If the developer forgets to pass key, you end up with undefined. So let's check whether key exists, and if not, log a console.error to warn them:

js
export const getStorageItem = (key) => {
  if (!key) {
    console.error('Missing the key to read from localStorage');
    return
  }

  return window.localStorage.getItem(key)
}

Now whenever someone forgets to pass key, they get a warning — that stops devs from forgetting it.

Missing key

Then finally, when we read the value out, we should check too — if it's null or undefined, return an empty object:

js
export const getStorageItem = (key) => {
  if (!key) {
    console.error('Missing the key to read from localStorage');
    return
  }

  const data = window.localStorage.getItem(key)

  if (!data) {
    return {}
  }

  return JSON.parse(data)
}

Note You can return an empty object or an empty array — either works — but I prefer an empty object, because then you can just use Object.keys to check whether it's empty, like Object.keys({}).length === 0. With an empty array you'd have to use Array.isArray instead, like Array.isArray([]) && [].length === 0.

Then to use it, you just import getStorageItem and pass in key:

js
import { getStorageItem } from './utils/localStorage'

const myObj = getStorageItem('myObj')

See it? After wrapping, we don't have to call JSON.parse ourselves anymore, plus we added some checks so devs won't forget to pass key — which avoids a bunch of silly mistakes.

Next up is the most-used one, set, which adds data. Same idea for wrapping, so I'll just paste it:

js
export const setStorageItem = (key, value) => {
  if (!key) {
    console.error('Missing the key to set in localStorage');
    return
  }

  if (!value) {
    console.error('Missing the value to set in localStorage');
    return
  }

  window.localStorage.setItem(key, JSON.stringify(value))
}

Pretty simple, right? Wrapping really isn't hard, but it cuts down on a lot of silly mistakes — forgetting key or value, forgetting to stringify — all dodged by wrapping. And it keeps your code cleaner too.

There's one more advantage to wrapping. Once we've wrapped and imported it, we can check whether window.localStorage even exists, and if it doesn't, warn the developer that the current environment doesn't support window.localStorage:

js
export const getStorageItem = (key) => {
  if (!window.localStorage) {
    console.error('The current environment does not support localStorage');
    return
  }

  if (!key) {
    console.error('Missing the key to read from localStorage');
    return
  }

  const data = window.localStorage.getItem(key)

  if (!data) {
    return {}
  }

  return JSON.parse(data)
}

export const setStorageItem = (key, value) => {
  if (!window.localStorage) {
    console.error('The current environment does not support localStorage');
    return
  }

  if (!key) {
    console.error('Missing the key to set in localStorage');
    return
  }

  if (!value) {
    console.error('Missing the value to set in localStorage');
    return
  }

  window.localStorage.setItem(key, JSON.stringify(value))
}

Now here's the thing — it feels like we're adding more and more functions. So how do we manage them? This is where a Class comes in handy:

js
class LocalStorage {
  constructor() {
    if (!window.localStorage) {
      console.error('The current environment does not support localStorage');
      return
    }
  }

  get(key) {
    if (!key) {
      console.error('Missing the key to read from localStorage');
      return
    }

    const data = window.localStorage.getItem(key)

    if (!data) {
      return {}
    }

    return JSON.parse(data)
  }

  set(key, value) {
    if (!key) {
      console.error('Missing the key to set in localStorage');
      return
    }

    if (!value) {
      console.error('Missing the value to set in localStorage');
      return
    }

    window.localStorage.setItem(key, JSON.stringify(value))
  }
}

export default new LocalStorage()

To use it, you import the LocalStorage Class and call LocalStorage.get('myObj') to read data, or LocalStorage.set('myObj', { name: 'Ray', age: 18 }) to add data — like this:

js
import LocalStorage from './utils/localStorage'

LocalStorage.set('myObj', { name: 'Ray', age: 18 });

const myObj = LocalStorage.get('myObj')

When we do import LocalStorage from './utils/localStorage', that already runs new LocalStorage(). So at init time, we've already checked whether window.localStorage exists, and if it doesn't, the warning fires. That way devs won't forget to check for window.localStorage when using it.

So at this point you should have a decent feel for wrapping window.localStorage.

Below is a full example you can use as a reference:

js
class LocalStorage {
  constructor() {
    if(typeof window === 'undefined') {
      console.error('The current environment does not support window');
      return
    }
    if (!window.localStorage) {
      console.error('The current environment does not support localStorage');
      return
    }
  }

  get(key) {
    if (!key) {
      console.error('Missing the key to read from localStorage');
      return
    }

    const data = window.localStorage.getItem(key)

    if (!data) {
      return {}
    }

    return JSON.parse(data)
  }

  set(key, value) {
    if (!key) {
      console.error('Missing the key to set in localStorage');
      return
    }

    if (!value) {
      console.error('Missing the value to set in localStorage');
      return
    }

    window.localStorage.setItem(key, JSON.stringify(value))

    const data = this.get(key)
    if(!data) {
      console.error('Failed to set localStorage');
      return false
    } else {
      return true
    }
  }

  remove(key) {
    if (!key) {
      console.error('Missing the key to remove from localStorage');
      return false
    }

    window.localStorage.removeItem(key)

    const data = this.get(key)

    if(data) {
      console.error('Failed to remove from localStorage');
      return false
    } else {
      return true
    }
  }

  clear() {
    window.localStorage.clear()
  }
}

export default new LocalStorage()
