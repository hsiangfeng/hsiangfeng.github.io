JS From Zero (Day 22): Execution Context, Variable Environment & the Scope Chain

#JavaScript #RE：從零開始的學習 JS 生活 #第 11 屆 iT邦幫忙鐵人賽

Intro

Day 22 of being reincarnated into JS land. This time we’re covering four things that all kind of connect to each other: functions, the execution context, the variable environment, and the scope chain.

Execution Context

Let’s start with JavaScript’s execution context. Back on Day 19, I mentioned that JavaScript has this thing called the creation phase. When JavaScript runs, the first thing it creates is two things:

  • window
  • this

We already talked about this being a special keyword — I’ll cover it properly in a later chapter — so here I want to focus on window. Why single it out?

Think of it like a game. Every game has someone with the highest level of permissions, the GM.

That GM sits on top of an environment with a ton of permissions and features. You can think of window as the goddess who yanked us out of the normal world and into this other world (basically the GM). Once you understand her, you’ll know how to ask the goddess for the features you want — manipulating the DOM, using the BOM, that sort of thing.

What about functions? You can think of a function as a sub-manager, because it has its own execution context too. But you only need to remember one thing: the top-level boss with the highest permissions is window.

Variable Environment

Next up is the variable environment. Wherever you declare a variable, that’s where the variable shows up. This idea also ties into JavaScript’s scope chain, so I’ll introduce the scope chain concept a bit here too.

When we create the JavaScript environment, the first layer is window, and this window is what we call the global execution context, also known as the global object ↓

When we create a variable in the global execution context, that variable becomes what we call a global variable. That’s why you can find variable a under window ↓

A global variable basically means this variable can be used anywhere — as long as it’s attached to window, it’s accessible everywhere. Say I define a function A, and inside A I write console.log(a). Where does the a inside function A go looking? It searches outward, and the outer layer of function A is window, so it finds the var a = 1; we just defined ↓

This searching-upward behavior is what we call the scope chain. So why do we need to understand this? We’ve covered the global variable and the global execution context, but if there’s a global, there’s also a local. So what does local mean? A local variable is a variable that can only be used within that area. Let’s move var a = 1; inside a function, then create another function B, and inside B we again write console.log(a). Can you find var a = 1; now? Let’s just try it.

Why? Because var a = 1; is a local variable — it only lives inside function A. So in picture form it looks like this ↓

But if you create a function C inside function B, with console.log(a); as its content, it can find variable a, because it’s attached to that function’s environment ↓

So that’s the execution context. Every function builds its own execution context when it runs. You can think of it as a tree (Window) that grows a lot of branches (functions), and those branches grow other branches (other functions). Each branch has its own look — some branches have human faces (?!), some look like cats (?!) (mysterious voice: what a terrifying tree). But any variable created under a branch can’t be used by anyone else, and that’s a local variable.

So that’s a quick rundown of the scope chain, execution context, function environment, and variable environment. Hopefully I didn’t get anything too wrong in the explanation.
