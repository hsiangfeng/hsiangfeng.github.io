Intro

async and await are everyday tools in JavaScript, and they're closely tied to Promise. So before we get into them, let me first talk about what a Promise actually is.

Async vs Sync Events

First off, what even are async and sync events? Back in the "weird parts of JavaScript" stuff, we learned that JavaScript is a single-threaded language (meaning it can only do one thing at a time).

Sync Events

Think of JavaScript like cooking. The synchronous way is like saying I have to chop the veggies first → then slice the mushrooms → then pour in the broth → and finally serve the dish, strictly in that order.

Sync events

Async Events

So what about async? When you're cooking, aside from serving the dish at the very end, do we really have to chop veggies → slice mushrooms → pour broth in that exact order? Not necessarily. We could slice the mushrooms first, or pour the broth first, or even chop the veggies last. Basically, we can do whatever else can be done first, and those other tasks won't get blocked.

Now, I just said JavaScript is single-threaded, so how does it handle async events at all? In JavaScript, it finishes the stuff that needs to be done up front, then takes the things that need async handling and drops them into something called the event queue. Once the up-front work is done, it goes back and checks the event queue to see what's ready to run.

Promise

Since JavaScript has a ton of async events, there's this thing called Callback Hell. When we want one thing to finish before moving on to the next, we end up with this nightmare of nested callbacks.

Callback Hell

So how do you write a Promise?

Promise is a feature added in ES6, so you call it as a function. The common way to write it looks like this ↓

js
let newPromise = new Promise((resolve, reject) => {
  // resolve → means the operation succeeded, and passes the value back
  // reject  → means it failed, and returns an error message
})

At the start we can't know which function will finish first, but we still want them to run in order. To do that, you can write it like this ↓

js
let runPromiss = (name, time, status) => {
  return new Promise((resolve, reject) => {
    if(status) {
      console.log(`${name} is off!`)
      return setTimeout(() => {
        resolve(`${name} ran for ${time / 1000}s`)
      },time);
    } else {
     reject(new Error(`${name} hit an error`));
    };
  });
};
// Ming fails, so it returns an error message
runPromiss('Ming', 3000, false).then((res) => {
  console.log(res);
  return runPromiss('Wang', 1000, true)
}).then((res) => {
  console.log(res);
}).catch((error) => {
  console.log(error);
})

From the case above, you can see that once Ming fails, it returns the error message right away.

Ming failed

But if Ming finishes the run successfully, then next we'll see Wang start running too.

Wang takes off!

One thing to watch out for

So this is how we handle async events. There's one thing you absolutely need to know though: if a Promise doesn't return resolve, then the next .then won't run. Like this ↓

js
let runPromiss = (name, time, status) => {
  return new Promise((resolve, reject) => {
    if(status) {
      console.log(`${name} is off!`)
      return setTimeout(() => {
        console.log(`${name} ran for ${time / 1000} s`)
      },time);
    } else {
     reject(new Error(`${name} hit an error`));
    };
  });
};
// Ming fails, so it returns an error message
runPromiss('Ming', 3000, true).then((res) => {
  console.log(res);
  return runPromiss('Wang', 1000, true)
}).then((res) => {
  console.log(res);
}).catch((error) => {
  console.log(error);
})

No resolve

Callback Hell

Let me also throw in some background on what Callback Hell actually does to you.

Like I mentioned earlier, when we want JavaScript to run our code and handle events in the order we want, but we don't use ES6 Promise, you'll often end up with a pile of if...if... checks to figure out whether the data came back okay or finished processing. I've written this kind of Callback Hell myself, and it's miserable to debug, because there are so many brackets you lose track of which one closes which block.

A chunk of Callback Hell from my RagnarokShopV3 project

After switching to Promise, it got a lot easier to debug (honestly this part could still be optimized, I've just been a bit busy lately).
