Web Crypto API

Intro

When you're building things, you constantly run into moments where you need to encrypt, decrypt, or hash some data. In the past you'd probably reach for Node.js's crypto module, or a third-party package like crypto-js. But here's the thing — modern browsers already have a built-in Web Crypto API, so you can do all of that right in the browser without installing a single package. So this post is going to walk through how to actually use it.

What's Crypto?

A lot of the time when we build apps, we don't store data as plain text. We encrypt it first, then store it. The upside is that if the database ever gets breached, the attacker can't decrypt anything without the key. Sure, they could try brute-forcing the key, but that takes an absurdly long time. And that delay is exactly the point — it buys a company enough time to respond to the incident and notify users (change passwords, lock accounts, that kind of thing).

So Crypto is basically the toolbox that does all of this — things like:

  • Encryption
  • Decryption
  • Hashing
  • HMAC
  • Signing
  • Verifying

…and more. In Node.js, Crypto is a built-in module, so you don't need to install anything to use it. In the browser, though, you'd normally have to pull in a third-party package. Except — you don't actually have to, because the browser has a built-in API for this too. That's the Web Crypto API.

The Web Crypto API

The Web Crypto API is a feature browsers added later on, letting you do encryption and decryption directly in the browser. The nice part is you don't need to install anything to use it, and the API itself is pretty straightforward. Let me show you how it works.

The Web Crypto API mostly works through the window.crypto.subtle object, which gives you these common methods:

  • crypto.subtle.digest(): hashing
  • crypto.subtle.encrypt() / crypto.subtle.decrypt(): encrypt / decrypt
  • crypto.subtle.sign() / crypto.subtle.verify(): sign / verify
  • crypto.subtle.generateKey(): generate a key
  • crypto.subtle.importKey() / crypto.subtle.exportKey(): import / export a key
  • crypto.subtle.deriveKey() / crypto.subtle.deriveBits(): derive a key / derive bits

Let's go through these one by one.

Generating a Random UUID

Before we get into the more complicated encryption stuff, let's look at the simplest and most useful feature first — generating a UUID (Universally Unique Identifier). Turns out crypto.randomUUID() is one of the methods the Web Crypto API gives you, and it couldn't be easier to use:

js
const id = crypto.randomUUID();
console.log(id); // something like 'cb729eee-87de-4aa8-b455-857bfff869e2'

Pretty simple, right? You don't need to install the uuid package — the browser already has this built in, and it generates a UUID v4, which is so unlikely to collide it's basically impossible.

Note If you want to dig deeper into UUIDs, check out my earlier post "Generate a UUID right in the browser, no npm package needed! (article in Mandarin)".

Generating Random Values

Besides UUIDs, the Web Crypto API also gives you crypto.getRandomValues(), which generates cryptographically secure random values. This is safer than Math.random(), because the numbers Math.random() produces are predictable, whereas the ones from crypto.getRandomValues() aren't:

js
// Generate 16 random bytes
const randomBytes = crypto.getRandomValues(new Uint8Array(16));
console.log(randomBytes);
// something like Uint8Array(16) [23, 187, 49, 200, 11, ...]

This is super handy for things like initialization vectors (IVs) and salts when generating encryption keys — you'll see it show up again in the encryption examples below.

Hashing

Hashing is a one-way kind of encryption. Meaning you can hash data, but you can't reverse it back to the original. This is usually used for storing passwords, because we don't actually need to know the original password — we just need to check whether a given password is correct.

Common hashing algorithms include:

  • SHA-1 (no longer recommended)
  • SHA-256
  • SHA-384
  • SHA-512

So let's see how to hash with the Web Crypto API:

js
async function hash(message) {
  // Convert the string into a Uint8Array
  const encoder = new TextEncoder();
  const data = encoder.encode(message);

  // Hash using SHA-256
  const hashBuffer = await crypto.subtle.digest('SHA-256', data);

  // Convert the ArrayBuffer into a hex string
  const hashArray = Array.from(new Uint8Array(hashBuffer));
  const hashHex = hashArray.map(b => b.toString(16).padStart(2, '0')).join('');

  return hashHex;
}

hash('Hello World').then(console.log);
// output: a591a6d40bf420404a011733cfb7b190d62c65bf0bcda32b57b277d9ad9f146e

You might be wondering — why do we convert the string into a Uint8Array first? Because the digest method takes a BufferSource, which means an ArrayBuffer or a TypedArray, so we need to turn the string into a Uint8Array first.

And digest returns an ArrayBuffer too, so we convert that into a more readable hex string.

Note If you've used Node.js's crypto module before, you might notice its API is a bit simpler, e.g. crypto.createHash('sha256').update('Hello World').digest('hex'). The Web Crypto API way is slightly more involved, but it's natively supported by the browser and needs no extra packages.

Encryption and Decryption (AES-GCM)

Next up is encryption and decryption. Here we'll use the AES-GCM algorithm. AES-GCM is a symmetric algorithm, which means encryption and decryption use the same key.

Generating a Key

First we need to generate a key:

js
async function generateKey() {
  return await crypto.subtle.generateKey(
    {
      name: 'AES-GCM',
      length: 256,
    },
    true, // whether the key is exportable
    ['encrypt', 'decrypt'] // key usages
  );
}

Encrypting

Then we can use that key to encrypt:

js
async function encrypt(key, message) {
  const encoder = new TextEncoder();
  const data = encoder.encode(message);

  // Generate a random initialization vector (IV)
  const iv = crypto.getRandomValues(new Uint8Array(12));

  const encryptedBuffer = await crypto.subtle.encrypt(
    {
      name: 'AES-GCM',
      iv: iv,
    },
    key,
    data
  );

  return { iv, encryptedBuffer };
}

There's an important concept here: the iv (initialization vector). Its job is to make the same piece of text produce a different result every time you encrypt it, which boosts security. The iv doesn't need to be kept secret — it's usually stored alongside the encrypted data.

Decrypting

Finally we use the same key to decrypt:

js
async function decrypt(key, iv, encryptedBuffer) {
  const decryptedBuffer = await crypto.subtle.decrypt(
    {
      name: 'AES-GCM',
      iv: iv,
    },
    key,
    encryptedBuffer
  );

  const decoder = new TextDecoder();
  return decoder.decode(decryptedBuffer);
}

Full AES-GCM Example

So let's put together the full encrypt-and-decrypt example:

js
async function demo() {
  // Generate a key
  const key = await generateKey();

  // Encrypt
  const message = 'Hello Web Crypto API!';
  const { iv, encryptedBuffer } = await encrypt(key, message);

  console.log('Encrypted data:', new Uint8Array(encryptedBuffer));

  // Decrypt
  const decryptedMessage = await decrypt(key, iv, encryptedBuffer);
  console.log('Decrypted data:', decryptedMessage);
  // output: Hello Web Crypto API!
}

demo();

Looks pretty simple, right? With the Web Crypto API, we can encrypt and decrypt right in the browser without installing anything.

Signing and Verifying (HMAC)

HMAC (Hash-based Message Authentication Code) is a mechanism for verifying the integrity and authenticity of data. It combines a hash function with a key, letting you confirm whether data was tampered with in transit.

Generating an HMAC Key

js
async function generateHmacKey() {
  return await crypto.subtle.generateKey(
    {
      name: 'HMAC',
      hash: 'SHA-256',
    },
    true,
    ['sign', 'verify']
  );
}

Signing

js
async function sign(key, message) {
  const encoder = new TextEncoder();
  const data = encoder.encode(message);

  const signature = await crypto.subtle.sign('HMAC', key, data);

  return signature;
}

Verifying

js
async function verify(key, signature, message) {
  const encoder = new TextEncoder();
  const data = encoder.encode(message);

  const isValid = await crypto.subtle.verify('HMAC', key, signature, data);

  return isValid;
}

Full HMAC Example

js
async function hmacDemo() {
  const key = await generateHmacKey();

  const message = 'This is some data that needs verifying';
  const signature = await sign(key, message);

  // Verify the original data
  const isValid = await verify(key, signature, message);
  console.log('Verification result:', isValid); // true

  // Verify tampered data
  const isTampered = await verify(key, signature, 'This is tampered data');
  console.log('Tampered verification result:', isTampered); // false
}

hmacDemo();

With HMAC, we can confirm whether data was tampered with in transit — a really common security mechanism in API communication.

Asymmetric Encryption (RSA-OAEP)

Besides symmetric encryption, the Web Crypto API also supports asymmetric encryption — what we usually call public/private key encryption. Here we'll use RSA-OAEP to demonstrate.

The idea behind asymmetric encryption is that you have a pair of keys: a "public key" and a "private key". The public key encrypts, the private key decrypts. The public key can be shared openly, while the private key has to be kept secret.

js
async function rsaDemo() {
  // Generate an RSA key pair
  const keyPair = await crypto.subtle.generateKey(
    {
      name: 'RSA-OAEP',
      modulusLength: 2048,
      publicExponent: new Uint8Array([1, 0, 1]),
      hash: 'SHA-256',
    },
    true,
    ['encrypt', 'decrypt']
  );

  const encoder = new TextEncoder();
  const message = 'Hello RSA!';
  const data = encoder.encode(message);

  // Encrypt with the public key
  const encrypted = await crypto.subtle.encrypt(
    { name: 'RSA-OAEP' },
    keyPair.publicKey,
    data
  );

  console.log('Encrypted:', new Uint8Array(encrypted));

  // Decrypt with the private key
  const decrypted = await crypto.subtle.decrypt(
    { name: 'RSA-OAEP' },
    keyPair.privateKey,
    encrypted
  );

  const decoder = new TextDecoder();
  console.log('Decrypted:', decoder.decode(decrypted));
  // output: Hello RSA!
}

rsaDemo();

Handy Helper Functions

When you're working with the Web Crypto API, you'll notice you constantly need to convert between ArrayBuffer and strings. So it's worth wrapping up a few common helpers to make life easier:

js
// ArrayBuffer to hex string
function bufferToHex(buffer) {
  return Array.from(new Uint8Array(buffer))
    .map(b => b.toString(16).padStart(2, '0'))
    .join('');
}

// Hex string to ArrayBuffer
function hexToBuffer(hex) {
  const bytes = hex.match(/.{1,2}/g).map(byte => parseInt(byte, 16));
  return new Uint8Array(bytes).buffer;
}

// ArrayBuffer to Base64
function bufferToBase64(buffer) {
  const bytes = new Uint8Array(buffer);
  let binary = '';
  bytes.forEach(b => binary += String.fromCharCode(b));
  return btoa(binary);
}

// Base64 to ArrayBuffer
function base64ToBuffer(base64) {
  const binary = atob(base64);
  const bytes = new Uint8Array(binary.length);
  for (let i = 0; i < binary.length; i++) {
    bytes[i] = binary.charCodeAt(i);
  }
  return bytes.buffer;
}

These helpers come in really handy in real-world dev, because when we transmit or store encrypted data, we usually convert the ArrayBuffer into Base64 or a hex string first.

Web Crypto API vs. Node.js crypto

You might be curious how the Web Crypto API differs from Node.js's crypto module. Here's a quick comparison:

ItemWeb Crypto APINode.js crypto
EnvironmentBrowserNode.js
API stylePromise (async)Sync / Stream
InstallationBuilt in, no installBuilt in, no install
Supported algorithmsAES, RSA, ECDSA, HMAC, etc.Broader algorithm support
ConstraintsRequires HTTPS (Secure Context)None

One important thing to watch out for: the Web Crypto API only works in a Secure Context. That means you have to be on HTTPS or localhost to use it. On plain HTTP, it won't work.

Note Worth knowing: every method in the Web Crypto API is async (returns a Promise). That's because crypto operations tend to be expensive, and going async avoids blocking the main thread, which keeps the user experience smooth.

Browser Support

So how's browser support for the Web Crypto API? Honestly, all the major browsers support it now — Chrome, Firefox, Safari, Edge, you name it. You can basically use it without worrying.

And it's not just browsers. Node.js (v15+), Deno, and Bun all support the Web Crypto API too, so your encryption code can run across environments. That's a huge advantage.

Wrapping Up

The Web Crypto API is a really capable built-in browser API. It lets us encrypt and decrypt directly in the browser without installing anything. Its API is a touch more involved than Node.js's crypto module (mostly because of all the ArrayBuffer conversions), but the upsides are:

  • No packages to install, fewer project dependencies
  • Native browser support, better performance
  • Cross-environment support (browser, Node.js, Deno, Bun)
  • Async by design, won't block the main thread

If your project needs to do encryption or decryption on the frontend, the Web Crypto API is definitely your first pick.
