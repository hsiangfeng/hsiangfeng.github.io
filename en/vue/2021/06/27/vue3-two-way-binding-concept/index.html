Vue 3 Two-Way Binding: Concept and Implementation

#Vue #v-model #Two-way binding

Intro

This post is a quick look at Vue 3 two-way binding, including a small implementation.

Vue

In the previous post “Vue 2 Two-Way Binding: Concept and Implementation“ I covered how Vue 2’s two-way binding is built on Object.defineProperty. With Vue 3, Vue made a major change — switching from Object.defineProperty to ES6 Proxy. The most commonly cited reason is a big performance improvement.

The trade-off with Proxy isn’t just performance — Object.defineProperty had two main downsides:

  • It can’t deeply observe data. Vue 2 had to walk through objects recursively to set up watchers, which was expensive.
  • Array length changes weren’t detected.

The Vue 2 docs cover this under “Change Detection Caveats“:

Due to limitations in JavaScript, Vue cannot detect array and object changes. However there are ways to circumvent these limitations and preserve reactivity.

What does that mean in practice? You’ve probably heard the advice “define your data upfront, or things will break”. If data doesn’t include a property and you add it later, that property won’t be reactive:

Since b isn’t reactive, binding it in the template throws ReferenceError: b is not defined. To see what’s happening more clearly, try this:

We know Object.defineProperty would set up Setter and Getter — but the late-added b doesn’t have them:

For this, Vue provides Vue.$set.

For arrays, the docs call out two cases Vue can’t observe:

When you directly set an item with the index, e.g. vm.items[indexOfItem] = newValue
When you modify the length of the array, e.g. vm.items.length = newLength

Usually we use arrays for looping. Setting up Getter/Setter on every array element is expensive — there’s a discussion on this (in Chinese) that goes into the perf reasoning if you’re curious.
