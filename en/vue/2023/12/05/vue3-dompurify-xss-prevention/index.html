XSS

Intro

In Vue, v-html lets you render an HTML string as actual DOM. The catch is that this opens up XSS attacks — you can't be sure user input is safe, so you need to filter input before rendering it.

DOMPurify

Vue has three ways to render data:

  • mustache interpolations: {{ myName }}
  • v-text
  • v-html

Both {{ myName }} and v-text render the data as plain text. v-html renders it as HTML — similar to innerHTML, but not actually innerHTML since v-html participates in reactive updates whereas innerHTML doesn't.

See the Pen v-html by @hsiangfeng on CodePen.

We all know that rendering HTML directly via innerHTML is risky — it's open to XSS attacks (article in Mandarin). Mustache interpolations and v-text avoid this because Vue strips HTML tags for you, but sometimes you actually do need v-html to render HTML.

When would you need v-html? Mostly when users get to input HTML themselves — Editors, blog comment boxes, that kind of thing.

That's where DOMPurify comes in. DOMPurify strips out dangerous tags, so you can filter user-supplied HTML before rendering and avoid executing malicious scripts.

Here's a setup where a user submits a malicious script and we render it with v-html:

See the Pen v-html ajax by @hsiangfeng on CodePen.

The example uses Vue 3 Composition API. Click Get Data — after five seconds it renders the data. You'll see an Image tag that's actually a malicious script — the image fails to load, triggers the onerror handler, and alert fires. That's the XSS.

How do we prevent it? With DOMPurify. You can install via npm or use a CDN. For the demo, I'll use the CDN:

html
<script src='https://cdnjs.cloudflare.com/ajax/libs/dompurify/3.0.6/purify.min.js' integrity='sha512-H+rglffZ6f5gF7UJgvH4Naa+fGCgjrHKMgoFOGmcPTRwR6oILo5R+gtzNrpDp7iMV3udbymBVjkeZGNz1Em4rQ==' crossorigin='anonymous'></script>

CodePen demo you can fork:

See the Pen v-html ajax XSS by @hsiangfeng on CodePen.

Using DOMPurify is simple — call sanitize with the HTML string, get back a sanitised string, then v-html that:

See the Pen v-html DOMPurify by @hsiangfeng on CodePen.

Click Get Data and you'll still see a broken image — but the onerror doesn't fire, so no XSS. That's what DOMPurify does.

Other DOMPurify usage

You can check what DOMPurify filtered out with DOMPurify.removed — it returns an array of removed tags:

js
const clean = DOMPurify.sanitize('<img src="x" onerror="alert(1)">');

console.log(DOMPurify.removed); // [{...}]

Filtered tags

The array contains an object with two properties:

  • attribute: the filtered attribute
  • from: the HTML tag it came from

Quick note: per the official docs, DOMPurify.removed is for inspection only — don't rely on it in your program logic.

In real-world use you usually don't want to filter everything — some tags should be allowed. Customise with the second argument to DOMPurify.sanitize:

  • dirty: the HTML string to filter
  • config: filter rules

This part is pretty involved — just refer to the official docs.

When should you use DOMPurify?

The common case is user-supplied input — Editors, comment boxes, etc. Modern frameworks have basic XSS defence built in, and the backend usually sanitises data before storage too.

So if both the backend and the framework already filter, do you still need DOMPurify? Depends on the use case. In Vue, if you render with {{ myName }} or v-text, you don't need DOMPurify — Vue handles it. But if you use v-html, you do — DOMPurify filters out dangerous tags so malicious scripts can't execute.

In short: if your feature renders user-supplied HTML directly, sanitise it with DOMPurify on the frontend to keep malicious scripts from executing.
