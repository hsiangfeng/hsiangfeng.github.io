Use a Custom <spec lang="md"> Block in Vue SFC for Smoother AI Development

#Vue #SFC #Spec-Driven Development #AI Development

Intro

Lots of devs pair AI tools (Claude Code, Gemini, Copilot, etc.) with their workflow, and Vue Single File Components (SFC) tend to hit some friction along the way. So here’s a different approach — adding a custom <spec lang="md"> block so you can use AI tools in SFC with way less hassle.

AI development modes

There are basically three modes of AI-assisted development right now:

  • SDD (Specification Driven Development): write the spec first, then let AI generate code from it.
  • BDD (Behavior Driven Development): write the behavior description first, then let AI generate code from it.
  • TDD (Test Driven Development): write the tests first, then let AI generate code from them.

SDD is the most popular of the three. As long as you describe the spec clearly, AI can produce code that matches your requirements. I won’t go deep into how to use or write SDD here though.

But here’s the thing — what does “describe the spec clearly” actually mean?

Honestly I find this interesting. Describing your own problem clearly is already hard enough, let alone the fact that AI only generates code based on what you actually say and provide. On top of that, there’s no real standard for measuring whether your description is clear enough or what it should include.

Usually before we ask AI to write code, we’ll have it do a preliminary project analysis and generate a project spec doc. The next time we open the project and ask AI to write code, we just feed this spec doc back to AI and let it work from there.

So you end up with a pile of docs like:

  • *.spec.md: project spec doc
  • *.test.md: test case doc
  • *.guide.md: development guide
  • *.task.md: task list
  • README.md: project readme
    …

And so on — sometimes you’ll even split out code samples into separate docs to explain.

As you can imagine, dev docs pile up fast, and you have to keep updating them so they stay in sync with the code/requirements.

You might think “these docs are just for the AI anyway, no big deal”, but they actually slow down development too — you have to spend time maintaining them. You can’t just ignore them and ask AI to write code blindly.

Not to mention AI Context Windows are limited.

That’s it — now you can use a <spec lang="md"> block in your Vue SFC to write spec and requirements.

What are the upsides? Here’s what I think:

  • Centralized management: the spec doc lives directly in the component file, easy to read and maintain. Each component carries its own spec, which makes team collaboration and communication smoother.
  • Consistent context: when AI generates code, it can reference the spec right inside the component, which cuts down on misunderstanding.
  • Fewer docs: no need to maintain a pile of separate spec docs, less mess overall.

One caveat — this approach only works in .vue files. For plain js/ts files you can’t use this, so I’d recommend JSDoc or TypeScript type definitions to describe things instead.

Why is Vue SFC specifically a good fit for this?

You might be wondering — can other frameworks pull off something similar?

Honestly, not easily. Vue SFC’s design philosophy is to keep related things together — <template>, <script>, <style> all live in the same file, so adding one more <spec> is a pretty natural extension.

Plus Vue has this built-in trait where one .vue file is one component, which gives you a 1:1 mapping between spec and implementation. You won’t get confused about which spec belongs to which component, because they’re literally in the same file.

React on the other hand doesn’t have the SFC concept — one file can hold multiple components, and there’s no custom block mechanism. To get something similar you’d mostly have to rely on JSDoc or drop a .spec.md file next to it, but then you lose the colocation advantage. Svelte is also a single-file component but its compiler doesn’t support custom blocks, so this approach doesn’t translate over either.

In short, Vue SFC’s custom blocks are basically the only way among mainstream frontend frameworks to make spec docs a first-class citizen of the component.

Things to watch out for

This approach is convenient, but there are a few things worth keeping in mind.

The most common one is Spec Degradation. Basically, the code gets updated but the <spec> doesn’t, and over time the spec and the implementation drift apart. My recommendation is to check that <spec> got updated during Code Review, or add a rule to your AI config file (like copilot-instructions.md or CLAUDE.md) saying “update the <spec> block whenever you modify the component” — that way AI will maintain the spec for you automatically.

Another thing — <spec> can only describe a single component’s spec, it can’t express interactions between multiple components. If you need to describe a whole feature or page-level architecture, you’ll still need a separate spec doc for that.

One more easy-to-miss thing is Context Window consumption. The <spec> block content doesn’t get compiled into the final output (since the Vite plugin turns it into an empty module), but AI still counts it when it reads the file. So keep the <spec> content concise and focused on the key requirements and behavior — no need to stuff a full PRD in there.

Wrap-up

Overall, using a <spec lang="md"> custom block in Vue SFC to manage spec docs is a really low-barrier and practical approach. No need to install some complex framework or tool — a simple Vite plugin gets it done, and having the spec sit right next to the code in the same file is also the ideal context source for AI tools.

Of course it’s not a silver bullet. Large projects might still need other tools for more complete spec management. But as a starting point for spec-driven development, this approach is genuinely worth trying.
