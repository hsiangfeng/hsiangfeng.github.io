Day 7 - Web Framework: Meet Express.js

#Node.js #Node.js 30 Days Series

Intro

In the previous post we tried building an HTTP API with pure Node.js, and I’m sure you learned a lot. But in real-world dev, we don’t directly use Node.js to build HTTP APIs — that’d be too much hassle. Instead, we use frameworks to help. This post introduces a really popular framework — Express.js.

What is Express.js?

What is Express.js (also known as Express)? If you’ve got frontend dev experience, think of it like Vue or Angular. Why? Because their positioning is really similar — they’re all web frameworks. The difference is Vue and Angular are for frontend web pages, while Express.js is specifically for Node.js server development.

Note
By official definition, React is a Library, so I didn’t mention it. But honestly, the line between Library and Framework is getting blurrier these days.

Express is widely used. For example, Paypal and Uber use Express.js for their server development — so we know Express is a really mature framework, and we can confidently use it for server dev.

There are alternatives — Koa.js, Fastify, NestJS, etc. — but this post focuses on Express. As for the others, we’ll get to them when fate allows :D.

Middleware

What’s Middleware? Think of it as a middleman. Real-life analogy: when you want to buy or rent a house, you go through an agent who finds suitable houses for you — that process is the middle layer. In code terms, it’s a function that runs before your code, then your code runs after — that process is Middleware.

A real estate agent helps you find suitable houses, just like Middleware does extra operations on requests before passing control to the next handler or route.

Express is essentially built on the concept of Middleware, so it’s a really important concept.

Back to app.use(express.json()); — I mentioned this Middleware helps parse req.body so we can get data normally. Let’s look at the source code:

So app.use(express.json()); is actually app.use(bodyParser.json()); — Express just wrapped it. Let’s look at the json source in body-parser:

Yeah it’s complex, but the real core is return function jsonParser (req, res, next) { ... }. This function runs before req, parsing based on settings — limit, inflate, reviver, strict, type, verify, etc. Once parsed, the parsed JSON data is attached to the request object’s body property for the route handler to use.

Of course Middleware isn’t just for parsing. In practice it’s commonly used for authentication — checking if Tokens have expired, checking permissions, etc. — all handled via Middleware.

This post is about wrapping up, so let’s summarize the Middleware concept:

  • Middleware is a function
  • Middleware runs before route handlers
  • Middleware must take req, res, next as parameters
  • Middleware must call next() to continue execution
  • Middleware can be used for authentication, data parsing, error handling, etc.
