Day 4 - Node.js Built-in Modules

#Node.js #Node.js 30 Days Series

Intro

In the previous post we covered a lot and finally wrote our first simple Node.js project (even if the example was from Node.js docs). Now let’s dig into Node.js’s built-in modules.

Node.js built-in modules

We previously wrote a small example (or rather, just used the official docs example):

In that example, we didn’t really get into it. Since this post is about Node.js built-in modules, let’s also explain that example along the way.

So what’s a built-in module? In short, it’s a module that comes built into Node.js — you don’t need to install anything extra to use it. These modules are provided by Node.js itself, so we can use them directly. That’s what “built-in module” means.

In the example above, we used the http built-in module. Through this module, we can quickly build a simple server. Since this module is from Node.js itself, we can use it directly.

Built-in modules are usually written like require('node:<module-name>').

With a basic concept in mind, let’s go through the example line by line:

  • Line 1: const http = require('node:http');
    • Use require to import the http module and assign it to the http constant, so we can use http to access the module.
  • Line 2: const hostname = '127.0.0.1';
    • Define a hostname constant assigned to '127.0.0.1' — we’ll use it later.
  • Line 3: const port = 3000;
    • Define a port constant assigned to 3000 — also used later.
  • Line 5: const server = http.createServer((req, res) => { ... }
    • Use the createServer method of the http module, passing in a function. This creates a simple server, but it hasn’t started yet.
  • Line 6: res.statusCode = 200;
    • Set the statusCode property on the res object to 200, so the browser knows the request was successful.
  • Line 7: res.setHeader('Content-Type', 'text/plain');
    • Use setHeader on res to set Content-Type to text/plain, so the browser knows the upcoming response will be plain text.
  • Line 8: res.end('Hello, World!\n');
    • Use end on res with Hello, World!\n as the response content, so the browser sees the Hello, World! string.
  • Line 10: server.listen(port, hostname, () => { ... }
    • Finally, server has a listen method that starts the server with the specified port and hostname, so the browser can connect.

With this code, we’ve built a simple server. Once the project starts, you can visit http://127.0.0.1:3000/ in your browser to see the Hello, World! string.

Quick note: res stands for Response, and req stands for Request — both are objects provided by the http module. We use them to handle the request and response. Since when a user makes a request we have to tell them the format we’ll send back, we use res.setHeader('Content-Type', 'text/plain'); to tell the browser that the upcoming response will be text/plain.

(Damn, I thought I just had to explain the previous example.)

You think Node.js’s built-in modules can only build a simple server?

Wrong! Node.js has many more built-in modules. Here are some:

  • http: for building HTTP servers.
  • https: for building HTTPS servers.
  • fs: for file operations — reading, writing, deleting files, etc.
  • path: tool for handling file paths.
  • os: tool for operating system info.
  • crypto: provides encryption and decryption functionality.
  • zlib: provides compression and decompression.
  • util: provides various utilities.
  • stream: for handling Streams — useful for reading and writing large files or data.
  • querystring: for handling URL query strings.

I won’t cover every built-in module — there are way too many. I’ll just pick a few common ones. We’ve already covered http, so I won’t repeat that.

Before continuing, let me reiterate: if you see require('node:<module-name>'), that means it’s a Node.js built-in module — you don’t need to install it separately. What does “install separately” mean? It means you don’t need to run npm install <module-name> because Node.js has already prepared it for you.

path module

path is a module specifically for handling file paths. Pretty interesting right? There’s actually a dedicated module for handling file paths — aren’t paths just ../ or ./? Why do we need a dedicated module?

The reason is paths are represented differently across operating systems. On Windows the path looks like C:\Users\user\Desktop\test.txt, while on Mac it’s /Users/user/Desktop/test.txt. So we need a dedicated module to make sure paths are handled correctly across operating systems.

Pretty simple right? We used the path module’s join method, which combines multiple paths and returns the combined path. Here we combined __dirname and text.txt to get a complete path.

Now here’s the interesting part — what’s __dirname? When you log __dirname directly, you get the current file’s path, e.g., /Users/user/Desktop/fs-example. So you might be thinking…

“Where did __dirname come from? You didn’t declare it. How can it get the current project directory?”

In Node.js there’s a global object that you can use anywhere. __dirname is a property on the global object, allowing us to get the current file’s path. So through __dirname, we get the current file path.

Note
Node.js’s global object is called global, while the browser’s is window. Both can be used anywhere. But in Node.js you don’t need to use global to access the global object — Node.js sets global as the default global, so you can directly use __dirname. Think of global as the Node.js equivalent of the browser’s window.

crypto module

crypto is mainly for encryption and decryption. The most common scenario is during user registration — we encrypt the user’s password so it doesn’t leak. When the user logs in, we encrypt the input password and compare against the database to confirm the input is correct.

crypto provides many methods — sha256, sha512, md5, etc. Here’s a simple example:

In short, we used sha256 to hash the user’s password into a string of garbled characters — preventing the password from leaking. What does this mean? When we develop systems, we always need to record user accounts and passwords. But for security, we don’t store passwords in plaintext — we hash them. That way the password won’t leak, since even system admins can’t see it.

Note
Plaintext means a password without any processing — e.g., if the password is 12345678 and it’s stored as 12345678, that’s plaintext.

But sha256 is just basic — usually you also use Salt (salting). That’s not the focus here, so I won’t go into it.

Follow-up about node:

Someone asked:

“In the example, the node: prefix — it seems to work whether or not you include it. So is there any difference?”

Great question. With our earlier example:

Even if you change it to:

This relates to Node.js’s require cache mechanism. When modules are imported, they’re cached (in require.cache) to avoid re-importing. So when we import a module, Node.js first checks if the module is cached — if so, it uses the cached one; if not, it loads the module fresh. This significantly improves performance, avoids re-importing, and allows shared state.

The node: prefix was introduced around Node.js 14 (I think?). When you use the node: prefix, Node.js bypasses the require cache mechanism. What does this mean? Even if the module exists in require.cache, it’s ignored — Node.js loads the module fresh. This makes sure you’re using the latest module instead of the cached one.

So using node: is essentially telling Node.js:

“Hey! Go directly find the module after node: — don’t use the cached one!”

That’s why both with and without the node: prefix work normally.
