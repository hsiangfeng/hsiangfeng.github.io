Let's Go

Note Oh, by the way — I'll be using VSCode throughout this series. If you don't have it installed yet, grab it from the VSCode website.

So what is Node.js?

Node.js is a JavaScript Runtime built on top of V8, which lets us write server-side programs in JavaScript. In other words, JavaScript isn't limited to the browser anymore — it can run on the server too.

v8

So… what's V8?

Definitely NOT this kind of engine you might be picturing.

V8 aircraft engine (image source: Wiki)

V8 is an open-source JavaScript engine developed by Google. It compiles JavaScript code into machine code so the computer can execute it. The name V8 actually does come from the V8 car/aircraft engine (the real engine), used to suggest "as powerful and efficient as a car with a V8 engine".

Note Computers basically only understand 0 and 1 — this is called binary. Machine code is just a string of binary that the computer can execute directly. Most modern programming languages need a compiler or interpreter to convert code into machine code so the computer can run it.

So where does V8 actually get used in practice?

If you're reading this in a browser and that browser is Chrome, you're already using V8 — Chrome uses V8 to execute JavaScript. V8 is itself written in C++ (another programming language), and it's what implements ECMAScript (the JavaScript spec) for us.

One thing worth mentioning: when we do frontend development, we manipulate the DOM (Document Object Model) to change web page content. But V8 doesn't provide DOM manipulation syntax — the DOM APIs you use in browsers come from the browser itself. V8 just implements ECMAScript so we can use JavaScript to operate on the browser's DOM.

OK, let's quickly summarize V8 so we don't spend too much time on it (we're not deep-diving into V8):

  • V8 is an open-source JavaScript engine developed by Google
  • V8 compiles JavaScript code into machine code so the computer can execute it
  • V8 is implemented in C++
  • V8 mainly implements the ECMAScript spec, allowing us to use JavaScript to operate on the browser DOM

From this summary, you can see how important V8 is to Node.js — without V8, Node.js wouldn't exist.

I mentioned earlier that Node.js can execute JavaScript on the server side. So what's the server side?

In computer science basics, server-side and client-side are very common roles/concepts. The server side is the one providing services, while the client side is the one consuming services. For example, when you type a URL into your browser, a request gets sent to the server. The server processes the request based on its content, talks to the database (which is where data is stored), and returns the appropriate data to the client. The client then displays the data to the user. That's a simple server-client interaction.

Why am I bringing this up? Because originally, JavaScript wasn't designed for server-side use — it was designed to manipulate the browser DOM. But as the web evolved, JavaScript started getting used for server-side development, and Node.js is the environment that lets JavaScript run on the server.

So at this point, you should have a decent grasp of Node.js's history and background. Next, let's install Node.js.

A few things before installing Node.js

First, head to the Node.js website. You should see something like this:

Node.js

But personally, I don't recommend installing directly through the Node.js website. Why? I'll explain later.

You'll see "18.17.0 LTS" and "20.5.0 Current" on the page. Let me first explain the difference between LTS and Current.

LTS (Long-Term Support) vs Current (or Latest) — what's the difference? In short, one is the stable version and the other is the testing version. For practical development, we always go with LTS, since LTS is the stable version while Current is the testing version. Current gets new features earlier, but because it's a testing version, it may have bugs — features that don't work, code that doesn't run properly, etc. So for development we always pick LTS.

Now let's look at what 18.17.0 and 20.5.0 mean. These are version numbers, and they follow Semantic Versioning, which uses three numbers:

  • Major
  • Minor
  • Patch

What do these three numbers mean?

  • Major: when your code has a major change — for example, going from ES5 to ES6, or ES6 to ES7 — the Major version number changes
  • Minor: when your code adds new features without affecting existing code, the Minor version number changes
  • Patch: when your code fixes bugs, the Patch version number changes

So 18.17.0 means version 18, minor version 17, patch version 0. Add the LTS label and the full plain English meaning is:

"This is version 18, minor version 17, patch version 0, and it's the stable Long-Term Support release (LTS)."

What about 20.5.0 Current? Same logic:

"This is version 20, minor version 5, patch version 0, and it's the testing/Current release."

Now that we understand version numbers, I want to mention another thing — the "even/odd version install rule".

What's the even/odd version install rule?

When installing certain packages, you might see this in their docs:

Make sure to use an even numbered version (18, 20, etc).

Most packages recommend using even-numbered Node.js versions. Why? The Node.js website explains:

Major Node.js versions enter "Current" release status for six months, which gives library authors time to add support for them. After six months, odd-numbered releases (9, 11, etc.) become unsupported, and even-numbered releases (10, 12, etc.) move to "Active LTS" status and are ready for general use. LTS release status is "long-term support", which typically guarantees that critical bugs will be fixed for a total of 30 months. Production applications should only use Active LTS or Maintenance LTS releases.

releases

That description might be a bit complex, so let me simplify it.

But before I do, let me define some roles to avoid confusion:

  • Provider: the developer who provides the tool, framework, or library
  • Developer: the developer who uses the tool, framework, or library

OK with that out of the way, let me explain. Odd-numbered versions (9, 11, etc.) are for providers — the main goal is to give them six months to make sure their stuff is compatible with new versions. After six months, odd-numbered versions become unsupported (no more updates/fixes). Even-numbered versions (10, 12, etc.) are for developers — these are the stable versions, supported for 30 months, so developers don't run into weird version-related bugs. That's why developers are recommended to use even-numbered versions.

This is getting confusing — let me use real version numbers to break it down.

We mentioned earlier that for development we always pick LTS, so 18.17.0 LTS it is. According to the even/odd version install rule, this is fine — the rule is based on the major version, and 18.17.0 LTS's major is 18, which is even. The minor and patch numbers just indicate small feature additions and bug fixes, so they don't affect our usage.

Installing Node.js

Funny right? After all that talking, we're only just getting to the install. But sorry — one more detour: we won't install Node.js directly. We'll install it through a Node.js version manager.

Why use a version manager? Because in development, different projects often need different Node.js versions — some on 12.0.0, some on 14.0.0. I've personally maintained projects that only run on 6.0.0. So we need a tool to manage our Node.js versions, instead of installing directly from the Node.js website. The direct install is super inconvenient — every version switch means reinstalling, which gets old fast. So in real-world dev, we use Node.js version managers.

What Node.js version managers are out there?

  • nvm
  • n
  • nvs
  • nodist
  • volta
  • fnm

Most people I work with use nvm, so I've previously written install guides for nvm:

  • Installing nvm on Mac OS X (article in Mandarin)
  • Installing nvm on Windows 10 to switch Node.js versions (article in Mandarin)

I won't go through the nvm install process here — I've written about it so many times already, I genuinely don't want to write about nvm again, QQ.

Spare me

Note If you already have Node.js installed, please remove it before installing nvm. nvm manages Node.js versions for you, so if you have Node.js already installed, you'll end up with two Node.js versions — one you installed and one nvm installed — which causes confusion. So remove Node.js before installing nvm.

So I'll assume you have nvm installed. Open your terminal (Terminal) or command prompt (CMD) and run this to install the same Node.js version I'm using:

bash
nvm install 18.17.0

install node

Then run this to make sure your Node.js version is 18.17.0:

bash
nvm use 18.17.0
bash
node -v

node

OK, we've finally got Node.js installed. Tomorrow we'll cover some basic Node.js concepts — modules, packages, npm, package.json, etc. So that's it for today, see you in the next one!

Note If you're curious about how to make the terminal look nicer — for Mac, see my post Mac OS Terminal Customization (article in Mandarin). For Windows, see Windows PowerShell Customization (Windows Terminal) (article in Mandarin).

Random thoughts

I previously tried using volta as my package manager and personally found it pretty good. But if your dev partners aren't using it too, it's a bit useless — its main feature is auto-switching to the right Node.js version based on a custom property in package.json.

Plus, support for some package installers isn't great yet (see: Support for pnpm).

So in the end I went back to nvm — it's older and more stable, and lots of people use it, so problems are easier to find solutions for.
