About CORS and Env

Intro

In the previous post we completed the Express + Mongoose integration. Now we'll handle environment variables and CORS issues.

Cross-origin resource sharing?

Cross-origin resource sharing? No worries if you don't get it from the name — I'm sure you know what CORS is. If you're a frontend dev, you've probably seen this error message often:

bash
Access to XMLHttpRequest at 'http://localhost:3000/api/v1/users' from origin 'http://localhost:8080' has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header is present on the requested resource.

What does this mean? In short, your current domain and Port are different from the domain you're trying to access, so the browser blocks it. At this point you need to add CORS settings on your backend to tell the browser which domains are allowed to access your API.

This is also called "cross-origin resource sharing". What does that mean? When you access resources of domain B from domain A, that's cross-origin resource sharing.

For security reasons, browsers block this. So you need CORS settings on the backend to tell the browser which domains can access your API. Then there's no problem.

(The most common scenario where this happens is AJAX requests.)

We're not going to deep-dive CORS. Just remember a few key points:

  • When domains differ, browsers block requests
  • You can add CORS settings on the backend to tell the browser which domains are allowed

Now let's add CORS to the API from the previous chapter.

CORS settings

First, recap the example code from the previous chapter:

js
const express = require('express');
const mongoose = require('mongoose');
const app = express();


let connectStatus = false;

async function connectMongoDB () {
  try {
    await mongoose.connect('mongodb+srv://url')
    console.log('Connected to MongoDB...')
    connectStatus = true;
  } catch (error) {
    console.log(error)
  }
}

connectMongoDB()

app.use(express.json());

app.use((req, res, next) => {
  if (connectStatus) {
    next();
  } else {
    res.status(503).send({
      status: false,
      message: 'Server is not ready'
    });
  }
})

const todoSchema = new mongoose.Schema({
  id: Number,
  title: String,
  completed: Boolean,
},{
  versionKey: false,
  _id: false,
});

const Todo = mongoose.model('Todo', todoSchema);

app.get('/todos', async (req, res) => {
  const todos = await Todo.find().select('-__v -_id');
  res.send({
    status: true,
    data: todos,
  });
});

app.post('/todos', async (req, res) => {
  const { title, completed } = req.body;

  const todo = new Todo({
    id: new Date().getTime(),
    title,
    completed,
  });

  await todo.save();
  res.send({
    status: true,
    message: 'Create todo successfully',
  });
});

app.listen(3000)

Why add CORS settings? It's related to current mainstream development models. Today's mainstream is frontend-backend separation — frontend is independent, backend is independent. The benefit is that frontend and backend teams can develop independently without affecting each other, focusing on their own areas.

To solve this, the backend has to add CORS settings so the frontend can access the backend API.

The setup is easy. Since we're using Express, we can use the cors package to handle CORS.

bash
npm i cors

Add CORS settings in your code:

js
const express = require('express');
// Add cors package
const cors = require('cors');

// ...other code omitted

// Add cors settings
app.use(cors());

// ...other code omitted

app.listen(3000);

Done — pretty simple right? At this point you might think...

"But when I tested with Postman earlier, I didn't run into CORS issues!"

The reason is CORS only happens in browsers. Postman is a software (you can think of it as a backend program), so it doesn't have CORS issues. That's why sometimes when communicating with the backend via Postman it's fine, but when the frontend integrates the API, CORS issues appear.

But the CORS settings above aren't quite right — they allow all domains to access your API:

json
{
  "origin": "*",
  "methods": "GET,HEAD,PUT,PATCH,POST,DELETE",
  "preflightContinue": false,
  "optionsSuccessStatus": 204
}

Note The above means all domains can access your API, with allowed methods being GET,HEAD,PUT,PATCH,POST,DELETE.

That's a security issue — we need to restrict access to specific domains.

We can use the origin parameter to restrict domains. For example, only allow http://localhost:8080:

js
const express = require('express');
// Add cors package
const cors = require('cors');

// ...other code omitted

const corsOptions = {
  origin: 'http://localhost:8080',
};

app.use(cors(corsOptions));

// ...other code omitted

app.listen(3000);

Note Some older browsers require optionsSuccessStatus: 200 to work properly, but modern browsers don't need this — so I won't go into detail.

The corsOptions.origin parameter accepts a string or function. If a string, it specifies a domain. If a function, you can define your own logic — like:

js
const express = require('express');

// Add cors package
const cors = require('cors');

// ...other code omitted

const whitelist = ['http://localhost:8080', 'http://localhost:8081'];
const corsOptions = {
  origin(origin, callback) {
    if (whitelist.indexOf(origin) !== -1) {
      callback(null, true);
    } else {
      callback(new Error('Not allowed by CORS'));
    }
  }
}

app.use(cors(corsOptions))

// ...other code omitted

app.listen(3000);

This way you define your own logic. We defined a whitelist array with two domains, then defined an origin function in corsOptions. The function takes two parameters — origin (the requesting domain) and callback (a callback function). We use the callback to decide whether to allow access.

Notewhitelist means an allowlist — we only allow domains in the whitelist to access our API.

If you don't want to use the cors package, you can write your own Middleware to handle CORS:

js
const express = require('express');

const app = express();

// ...other code omitted

const whitelist = ['http://localhost:8080', 'http://localhost:8081'];

app.use((req, res, next) => {
  const origin = req.headers.origin;
  if (whitelist.indexOf(origin) !== -1) {
    res.setHeader('Access-Control-Allow-Origin', origin);
    res.setHeader('Access-Control-Allow-Methods', 'GET,HEAD,PUT,PATCH,POST,DELETE');
    res.setHeader('Access-Control-Allow-Headers', 'Content-Type');
    next();
  } else {
    res.status(403).send({
      status: false,
      message: 'Not allowed by CORS'
    });
  }
});

// ...other code omitted

app.listen(3000);

But it's still more common to use the cors package — it's simpler.

Environment Variable

For some people, the purpose of environment variables might not be clear. But for others, environment variables are really important — they make our programs more flexible and secure.

Even though we've covered a lot already, environment variables are still really important, so we have to learn how to use them.

What are environment variables mainly used for? Common uses:

  • Hiding sensitive info
  • Switching between environments
  • Improving maintainability

Hiding sensitive info

Things like database passwords, keys, API keys — these are sensitive info we don't want others to know. So we use environment variables to hide them.

If you hardcode them in your code, then once your code is exposed, your database can be compromised. So we have to hide sensitive info.

Switching between environments

Sometimes certain code only runs in specific environments. We can switch environments via environment variables — for example, switching databases via env vars to ensure test and production environments are different.

Improving maintainability

"What?! Env vars also improve maintainability?"

You read that right. Say you have code like this:

js
// a.js
axios.get('http://localhost:3000/api/v1/profile');

// b.js
axios.get('http://localhost:3000/api/v1/users');

// c.js
axios.get('http://localhost:3000/api/v1/posts');

All these access the http://localhost:3000 domain. But if one day we need to change the domain to http://localhost:3001, we'd have to change all the code. Env vars make code more flexible and maintainable.

One less place to change

Noteaxios is an HTTP request package.

Setting up env vars

First, install dotenv — this package lets us use env vars in our code.

bash
npm i dotenv

Then create a .env file. This file is where we store env vars. We can define env vars in this file like:

bash
PORT=3000

This defines a PORT env var. Now modify our code to use it:

js
const express = require('express');

// Add dotenv package
require('dotenv').config();

const app = express();

// ...other code omitted

app.listen(process.env.PORT);

Now we can use env vars. We used process.env.PORT — the env var we defined in the .env file.

Noteprocess.env is a global variable in Node.js for storing env vars. Env var names usually use uppercase letters — e.g., PORT, DB_URL, API_KEY, etc.

I mentioned env vars improve maintainability — let's see how:

js
// a.js
axios.get('http://localhost:3000/api/v1/profile');

// b.js
axios.get('http://localhost:3000/api/v1/users');

// c.js
axios.get('http://localhost:3000/api/v1/posts');

All access the http://localhost:3000 domain. If we need to change to http://localhost:3001, we'd have to change all the code. With env vars, we just define an API_URL env var in .env and use it in the code:

bash
API_URL=http://localhost:3000
js
// a.js
axios.get(`${process.env.API_URL}/api/v1/profile`);
// b.js
axios.get(`${process.env.API_URL}/api/v1/users`);
// c.js
axios.get(`${process.env.API_URL}/api/v1/posts`);

Easy right? This improves maintainability.

Quick note — why use the dotenv package for env vars? Because dotenv lets us use env vars in code, and it auto-reads the .env file and adds the env vars in it to process.env. So we can use env vars in code.

Conclusion: Node.js itself has the built-in process.env env var object, but the dotenv package gives us a better way to manage, organize, and use env vars.
