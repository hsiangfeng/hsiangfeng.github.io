Node basics

Intro

In the previous post we finally got to know Node.js and installed it via a Node.js install/manage tool. Now let's get familiar with some Node.js basics.

Node.js basics

First, open your terminal (Terminal) or command prompt (CMD) and type:

bash
node

You should see something like this:

bash
❯ node
Welcome to Node.js v18.17.0.
Type ".help" for more information.
>

node

This mode is called "Node REPL (Read-Eval-Print-Loop)". In short, it puts you into the Node.js environment where you can type JavaScript code and hit Enter to execute it directly. Each time you input something, Node.js runs it and prints the result. That's REPL.

Note This isn't unique to Node.js — Python and Ruby have it too. We just happen to be running JavaScript code in the Node.js environment here.

Let's try it. Type the following and hit Enter:

bash
> console.log('Hello World')

You should see:

bash
> console.log('Hello World')
Hello World
undefined
>

REPL

Pretty neat right? If this is hard to wrap your head around, think of it like typing code into the browser's Console and hitting Enter — the browser runs it and prints the result. Same idea.

Note To exit REPL mode, just hit Ctrl + C twice (or Command + C on Mac), or type .exit.

The node command isn't just for entering REPL though — it has other parameters too. Besides the common -v (version) parameter for checking the Node.js version, I want to introduce another one: --inspect. This lets you debug Node.js using Chrome DevTools.

Debug Node.js with Chrome DevTools? What does that mean? Try running this:

bash
node --inspect

The terminal should show:

bash
❯ node --inspect
Debugger listening on ws://127.0.0.1:9229/508515c1-f25d-4445-90d6-8b28ee9e6935
For help, see: https://nodejs.org/en/docs/inspector
Welcome to Node.js v18.17.0.
Type ".help" for more information.
>

Debugger listening

Note The ws:... you see uses ws which stands for WebSocket. WebSocket is a network protocol that allows bidirectional communication between server and client — meaning persistent two-way communication, not just one direction at a time like traditional web requests.

Now open Chrome and type chrome://inspect in the address bar. You'll see:

chrome://inspect

You can see there's a Remote Target — that's the Node.js instance we just opened in the terminal with node --inspect. Click "inspect" to open Chrome DevTools, and you'll see:

inspect

Note If your terminal is too cluttered, hit Ctrl + L (or Command + K / Control + L on Mac) or type clear to wipe the screen. See: Clearing the Terminal screen (article in Mandarin)

So what's this mode useful for? It lets us debug Node.js using Chrome DevTools — meaning we can write code here and use Chrome DevTools to debug, instead of debugging in the terminal.

Note Debugging means finding the bugs in your code and fixing them.

Let's try a simple debug. In Chrome DevTools, find the "Sources" tab, then "Snippets", and click "New snippet". You'll see:

new snippet

Type this in the side panel and hit Ctrl + S (Command + S on Mac) to save:

js
for(let i = 0; i < 10; i++) {
  console.log(i)
}

Now click in the gutter next to line 2 — it should light up and a breakpoint will appear:

Adding a breakpoint

Press the shortcut to run the code: Ctrl + Enter (Command + Enter on Mac). You'll see:

Breakpoint

The code paused at the first iteration of the loop — when the initial value 0 is set:

init for

This way you can step through the code one piece at a time and observe how things change, making it easier to find where the bugs are. That's debugging.

The node command has tons of options we won't use, so let me cover one more practical thing — running a JavaScript file via the node command. This way we don't have to enter REPL mode and type code every time — we can just execute the JS file directly.

Create a folder, then create an index.js file inside it. Put this in index.js:

js
console.log('Hello World')

Note If you're comfortable with the command line, you can run mkdir example to create the folder, cd example to enter it, and touch index.js to create the file. (On Windows, touch doesn't exist — use echo "" > index.js instead.)

Now type node and then drag the index.js file into the terminal. The terminal should show:

bash
node /Users/xxxx/Desktop/example/index.js

node index.js

Hit Enter and you'll see Hello World appear in the terminal. That means you successfully executed the JavaScript file!

By now I'm sure you've got a basic grasp of node operations. So next we're going to look at a new thing — npm.

npm

What is NPM?

NPM

If you're a frontend dev with some experience, NPM probably isn't new to you — it's a really important thing in development. But for readers without that background, here's a quick intro.

NPM stands for Node Package Manager — it's a package manager for Node.js. With NPM, you can easily install, update, remove packages, and manage their versions. It comes bundled when you install Node.js, so you don't need to install it separately.

You might wonder why we need it. In real-world dev, you can't be like a caveman building everything from scratch — there are deadlines. So we use packages other people have written, and these packages are hosted on NPM. We use NPM to install them, making it easy to use other people's work.

If that's still abstract, let me use a real-life analogy. Say you want to get from Taipei to Kaohsiung — there are many ways: drive, fly, take the train, take the high-speed rail. These ways are like packages. NPM is like a travel agent — you can tell them you want to fly, and they'll book your flight for you. That way you can easily get to Kaohsiung.

Note NPM isn't unique to Node.js — other languages have similar package managers: Python has pip, Ruby has gem, PHP has composer, etc. These all exist to help developers manage packages — install, update, remove, and version control.
