Day 3 - Your First Node.js Project

#Node.js #Node.js 30 Days Series

Intro

After two days of intro, we’re finally about to build our first Node.js project. By now you should have a decent grasp of Node.js basics, so we can comfortably get started.

Basic terminal commands

First, I want you to memorize a few terminal commands. These are used constantly in real-world dev — even when you’re using VSCode’s terminal. I’m likely to use these in the article, so I want you to at least know:

  • cd [path]: move to the specified path
    • e.g., cd /Users/username/Desktop: move to the desktop
  • cd ..: go up one folder
  • mkdir [folder name]: create a folder
    • e.g., mkdir my-project: create a my-project folder
  • touch [filename.ext]: create a file
    • e.g., touch index.js: create the index.js file
    • Note: Windows doesn’t support this command, use echo > [filename.ext] instead
  • ls: list all files and folders in the current path
  • ll: list all files and folders in the current path (with detailed info)

These are the most basic and most common terminal commands. If you’re not familiar with them, I’d suggest practicing and getting used to them — they’re going to come up constantly in development.

Note
File extension means the file’s type. For example, index.js has the extension .js and the main name index. Operating systems often hide file extensions, so if you can’t see them, you’ll need to enable that in OS settings.

I mentioned package.json is where we manage packages — calling it the heart of NPM isn’t an exaggeration, since we install, update, and remove packages through it. So this file is really important.

Quick rundown of each property currently in package.json:

  • name: project name
  • version: project version
  • description: project description
  • main: project entry point
  • scripts: project commands
  • keywords: project keywords
  • author: project author
  • license: project license
  • dependencies: project dependencies
  • devDependencies: project dev dependencies

Note
package.json actually has more properties than these. If you want to know all the special ones, just check the official npm package.json docs — there are way too many to list.

The three properties you really need to pay attention to are dependencies, devDependencies, and scripts, since you’ll definitely use these. Let’s look at what they’re for.

What about package-lock.json?

Surprised? Just package.json alone has tons of stuff to cover. So… what about package-lock.json?

package-lock.json is only generated when you run npm install. This file records detailed package version info. The benefit is that when you run npm install in different environments (e.g., different computers), it installs based on the versions in package-lock.json — avoiding the issue of different versions being installed in different environments and breaking the project.

But wait — we said earlier that packages are installed based on versions in package.json. So why do we need package-lock.json? Isn’t that redundant?

A real example might make this clearer. We just installed nodemon with npm install nodemon. Currently package.json looks like:

So package-lock.json is a really important file.

What about node_modules?

node_modules…

…is a black hole. End of story.

There’s a reason node_modules has been jokingly called a black hole. Every time you install a package, a folder gets created in node_modules containing all the package’s files. So when you’ve installed lots of packages, node_modules gets really big — sometimes over 100MB (my personal record is around 500MB). At that point you’ll notice node_modules is just like a black hole — you have no idea what’s in it, just that it keeps getting fatter.

So you basically just need to know it’s where packages we’ll use are stored. As for what’s actually in there, you really don’t need to care.

If you do care, you might consider a different package manager like pnpm — which makes node_modules much smaller because it shares packages across projects. But I won’t go into details — covering pnpm could easily take three more days.
