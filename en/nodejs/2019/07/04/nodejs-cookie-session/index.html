Cookie vs Session: What Every Web Dev Should Actually Know

#Node.js #Full-Stack Warrior's Path Node.js

Intro

Next up I want to talk about Cookie and Session. As a web developer, these two are things you really have to know — there’s no skipping them.

Cookie

A Cookie usually stores records of the sites you’ve visited — logins, preferences, that kind of thing. Here’s what makes a Cookie tick:

  • It can only store up to 4KB
  • It can be read from and written to on both the Client (browser) and the Server
  • It’s made up of key/value pairs
  • And the big one: you can set an expiration date

But there are actually two kinds of Cookie:

  • Memory Cookie
  • Disk Cookie

Memory Cookie

A memory Cookie is usually maintained by the Client (browser) and stored in memory. The moment you close the browser, it’s gone — so it sticks around for a very short time.

Disk Cookie

A disk Cookie is a bit more interesting. It’s stored on disk, so unless the user manually clears it or the time you set expires, it won’t get deleted — and that time is usually a long way off.

So if you split things up by lifespan, you can also describe Cookies as two types: non-persistent Cookie and persistent Cookie.

Downsides of Cookie

Cookies aren’t perfect, though. Since a Cookie is only 4KB, any more complex way of storing data is off the table. Cookies also add to your traffic. And finally, if you pass them over plain HTTP, you’ve got a security problem — unless you go with HTTPS.

Session

Now that we’ve covered Cookie, let’s talk about Session. So what’s special about Session?

  • It’s temporary data stored on the Server — it can live in memory or in a database
  • A Session can also store a UUID on a Cookie so the server knows who you are

Usually a Session lives in memory; it’s pretty rare to put it in a database.

What about real-world use? In practice you usually combine the two together to verify a user’s login — that’s the most common approach you’ll see out there. So let’s keep going and look at how to write this in JS.
