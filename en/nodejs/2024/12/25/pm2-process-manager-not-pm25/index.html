PM2

Intro

PM2 stands for Process Manager 2 — a Node.js process management tool. Not the abbreviation for Particulate Matter 2.5. So don't confuse them.

What is PM2 (Process Manager 2)?

The common description goes:

PM2 is a Node.js process management tool for the JavaScript runtime.

That sentence is short but loaded. Let's break it down — you'll probably understand PM2 better:

  • JavaScript runtime: when JavaScript executes, it creates an environment — what we call the runtime. The runtime executes JavaScript code.
  • Node.js: a JavaScript runtime built on Chrome's V8 engine that lets JavaScript run on servers and use system APIs.
  • Process management tool: a tool that manages processes. A process is a running program — a process manager lets us control these programs.

So PM2 is a tool for managing Node.js processes. But why do we need it?

Simplest scenario: you have a Node.js app you want kept running. The app might crash due to an unexpected bug — and without PM2, that's when you get the on-call page. PM2 helps prevent that.

Setting up the initial environment

Let's set up a simple environment so PM2's value becomes clearer. Build a quick Node.js app:

bash
mkdir pm2-example
cd pm2-example
express --no-view
npm install

What we did:

  • Created a folder pm2-example
  • cd'd into it
  • Used express to scaffold an Express app, no view engine (we just want a simple API)
  • Installed Express dependencies

Build the habit — after creating a project, always run it once to make sure it works:

bash
npm start

Test the API with Postman:

txt
GET http://localhost:3000/users

You should get:

txt
respond with a resource

Your project is working. Now we'll tweak it so PM2 can manage it instead of you running npm start manually.

Installing PM2

Open the terminal:

bash
npm install pm2 -g

Done. Very simple. With yarn:

bash
yarn global add pm2

PM2 basics

Usage is simple. The main command to remember:

  • pm2 start <filename>: start an application

That's it. To start a project directly:

bash
pm2 start app.js

But our project was scaffolded by Express, so:

bash
pm2 start bin/www

You'll see:

Start

The terminal looks totally different. Before, with npm start:

npm start

With PM2, requests and console output don't appear in the terminal — PM2 moves the running process to the background, so we don't have to worry about the project stopping when we close the terminal.

View PM2 status and list

So how do we check PM2's status? pm2 list:

bash
pm2 list

You'll see the projects running under PM2:

PM2

It's like a dashboard for PM2 projects with basic info.

There are multiple equivalent commands:

  • pm2 status
  • pm2 ls
  • pm2 list

They all show PM2 projects. I personally use pm2 list.

What each column means:

  • id: auto-generated project ID from PM2
  • name: project name, usually the filename — defaults to whatever comes after the command (bin/www → www)
  • namespace: project namespace, defaults to default — think of it as a group
  • version: project version, taken from package.json
  • mode: project mode, usually fork (single process) or cluster (multi-process)
  • pid: process ID
  • uptime: how long it's been running
  • status: usually online. Possible values: online, stopped, errored, launching, unknown
  • cpu: CPU usage
  • mem: memory usage
  • user: user running it
  • watching: whether file watching is on (usually false, true when enabled)

View PM2 logs

To view logs:

bash
pm2 logs <project name/ID/namespace>

For our example:

bash
pm2 logs 0 # 0 is the project ID
pm2 logs www # www is the project name
pm2 logs default # default is the namespace

You'll keep seeing the project's logs:

Logs

Detailed pm2 logs options:

  • pm2 logs: all project logs
  • pm2 logs <name/ID>: specific project's logs
  • pm2 logs <namespace>: like a group view — you can categorize projects and see all logs in that group

With additional flags you can be more precise:

  • pm2 logs --out: only stdout (normal output)
  • pm2 logs --err: only error output
  • pm2 logs --json > logs.json: output as JSON so you can process it programmatically
  • pm2 logs <name/ID> | grep "keyword": filter logs by keyword — e.g. pm2 logs www | grep "GET" shows only GET logs
  • pm2 logs <name/ID> --lines 100: show only the last 100 lines

That covers the common usage.

Passing flags to pm2 start

pm2 start isn't just a simple command — you can pass flags for more options:

  • --name: set project name
  • --watch: watch for file changes, auto-restart on change
  • --cron-restart: restart on a cron schedule
  • --max-memory-restart: restart when memory usage exceeds a threshold
  • --namespace: set namespace

Common usage:

bash
pm2 start bin/www --name "hexschool-example" --watch

Sets the project name to hexschool-example, watches for file changes, and auto-restarts on change.

To set the namespace:

bash
pm2 start bin/www --namespace "hexschool" --watch

To set both name and namespace:

bash
pm2 start bin/www --name "hexschool-example" --namespace "hexschool" --watch

Restart a project

When something goes wrong, restart with pm2 restart:

bash
pm2 restart <name/ID/namespace>

To restart everything:

bash
pm2 restart all

Stop a project

To stop:

bash
pm2 stop <name/ID/namespace>

To stop everything:

bash
pm2 stop all

Delete a project

pm2 stop only stops a project — it doesn't delete it. To delete, use pm2 delete:

bash
pm2 delete <name/ID/namespace>

Delete all:

bash
pm2 delete all

PM2 advanced config (Ecosystem File)

PM2 has an advanced config file called Ecosystem File. It supports JavaScript, JSON, and YAML.

In practice, ecosystem.config.js (JavaScript) is the most common.

To generate one, PM2 auto-creates it for you:

bash
pm2 ecosystem

It generates an Ecosystem File based on your current project environment.

The format looks like:

javascript
module.exports = {
  apps : [{
    script: 'index.js',
    watch: '.'
  }, {
    script: './service-worker/',
    watch: ['./service-worker']
  }],

  deploy : {
    production : {
      user : 'SSH_USERNAME',
      host : 'SSH_HOSTMACHINE',
      ref  : 'origin/master',
      repo : 'GIT_REPOSITORY',
      path : 'DESTINATION_PATH',
      'pre-deploy-local': '',
      'post-deploy' : 'npm install && pm2 reload ecosystem.config.js --env production',
      'pre-setup': ''
    }
  }
};

PM2 is genuinely powerful — Ecosystem File lets you go way deeper.

That covers PM2 basics. Hopefully this helps you use PM2 to manage your Node.js apps

PS. PM2 isn't just for Node.js — you can use it to manage Python, Ruby, Java, and more.
