CI/CD

Intro

Knowing a bit about CI/CD is genuinely useful as a developer. Even on a small side project, CI/CD pays for itself — and even if you're "just a frontend dev", there's plenty you can do with it.

What's CI/CD?

First — what are CI and CD?

CI is Continuous Integration. CD is Continuous Deployment.

OK, you might already be thinking:

"What exactly are we 'continuously' integrating and deploying?"

So let me paint a picture.

When you deploy a project to production, you usually run a few commands — npm run test to make sure nothing's broken, npm run build to bundle, then ship the build artifacts. That's a normal dev workflow.

Looks fine on paper. But humans are forgetful — skip the tests one time, forget to rebuild another time. Skipping tests is especially bad: best case, you patch in prod immediately. Worst case, your phone rings.

That's what CI fixes — it automates the stuff you should be doing anyway. Tests, builds, etc.

CD? That's about deployment. CD auto-deploys for you, which saves time and removes one more thing you can mess up.

With the concepts out of the way — what CI/CD tool to use? A few options:

  • Travis CI
  • Circle CI
  • Jenkins
  • GitLab CI
  • GitHub Actions

This post uses GitHub Actions, since most of us are already on GitHub and GitHub Actions is first-party.

YAML basics

Before touching GitHub Actions, you need one thing: .yml files. That's the format GitHub Actions configs use — what jobs to run, what to test, build, deploy. (.yaml and .yml are the same format, different extension.)

Quick rules of YAML:

  • Indentation defines hierarchy
  • # is a comment
  • Key-value pairs are written key: value
  • Use - for multiple values (like an array), or [] inline. They're equivalent — [] is just more compact. For objects, use key: value again.
  • | preserves line breaks across multiple lines, > joins them on a single line.
  • {{ }} is variable substitution. Example: {{ secrets.GITHUB_TOKEN }} — secrets is an object, GITHUB_TOKEN is the key.

Under the hood YAML is essentially a flavor of JSON. If you know JSON, YAML should click pretty fast. Side-by-side examples below.

Comment

yml
# A comment
name: 'Ray'

key: value

yml
name: 'Ray'
json
{
  "name": "Ray"
}

Multi-line text

Preserve line breaks:

yml
name: |
  Ray
  IsRayNotArray
  Hello
json
{
  "name": "Ray\nIsRayNotArray\nHello"
}

Strip line breaks:

yml
name: >
  Ray
  IsRayNotArray
  Hello
json
{
  "name": "Ray IsRayNotArray Hello"
}

Array

Block form:

yml
name:
  - Ray
  - IsRayNotArray
  - Hello
json
{
  "name": ["Ray", "IsRayNotArray", "Hello"]
}

Flow form:

yml
name: [Ray, IsRayNotArray, Hello]
json
{
  "name": ["Ray", "IsRayNotArray", "Hello"]
}

Variable

yml
name: {{ secrets.GITHUB_TOKEN }} # automatically pulls the GitHub token
json
{
  "name": "1234567890"
}

Object

yml
name:
  firstName: Ray
  lastName: IsRayNotArray
  age: 18
json
{
  "name": {
    "firstName": "Ray",
    "lastName": "IsRayNotArray",
    "age": 18
  }
}

Array of objects:

  - firstName: Ray
    lastName: IsRayNotArray
    age: 18

    age: 18

    {
      "firstName": "Ray",
      "lastName": "IsRayNotArray",
      "age": 18
    }
  ]
}

That's the gist of YAML.

GitHub Actions fundamentals

Now that YAML's behind us, time for GitHub Actions itself.

Heads up — there's no UI editor. You define everything in .yml files, called workflows, which live in .github/workflows/. Here's the official picture:

workflow

The structure:

  • Event
  • Runner 1
    • Job 1
      • Step 1: Run action
      • Step 2: Run action
      • Step 3: Run action
      • Step 4: Run action
  • Runner 2
    • Job 2
      • Step 1: Run action
      • Step 2: Run action
      • Step 3: Run action

Every workflow has an Event, which fires a Runner, which runs Jobs, which run Steps in order.

Here's an official example:

yml
name: learn-github-actions
run-name: ${{ github.actor }} is learning GitHub Actions
on: [push]
jobs:
  check-bats-version:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '14'
      - run: npm install -g bats
      - run: bats -v

Don't panic — going line by line.

yml
name: learn-github-actions
run-name: ${{ github.actor }} is learning GitHub Actions

name is the workflow name — what shows up in the GitHub Actions tab. run-name is the description for each run, and github.actor is a variable for the GitHub username — so it reads "Ray is learning GitHub Actions" or whatever.

yml
on: [push]

on: [push] is the event trigger — this workflow fires when you push.

yml
jobs:
  check-bats-version:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '14'
      - run: npm install -g bats
      - run: bats -v

jobs is what runs when the event fires. check-bats-version is the job name. runs-on: ubuntu-latest says which environment runs it. steps is the sequence.

The - uses: lines are the bit people stumble on. What's actions/checkout@v3?

actions/checkout@v3 and actions/setup-node@v3 come from GitHub's official Actions organization.

actions

In plain English:

"Use the checkout repo under the actions organization, version v3."

checkout lets your workflow fetch your code. setup-node lets you pick a Node.js version. v3 is the version tag.

So the uses: actions/checkout@v3 / actions/setup-node@v3 syntax should make a bit more sense now.

actions/setup-node@v3 also has a with: block — that's where you pass arguments. node-version: '14' sets Node.js to v14. There are more options for setup-node, but most of the time the version is all you need. See the setup-node repo for details.

Last two lines — - run: npm install -g bats and - run: bats -v — those just run shell commands, the same way you would locally. - run: means "run this command".

Here's the visualized flow from the docs:

GitHub Action

That's GitHub Actions' official example walked through.

Building real workflows

With the fundamentals down, time to write some real workflows.

I'll use an Express skeleton I've prepared:

Express

Fork and clone it.

In the project root, create .github/workflows/. GitHub Actions configs have to live there. Inside, create main.yml. Your structure should look like this:

Structure

The .yml filename doesn't matter — it's just for your own organization. I'll use main.yml (later renamed to main-pr.yml, so you can use that too).

What we're automating:

  • ESLint scan
  • Tests
  • Build
  • Deploy

The first three run on pull requests. Deploy runs on merges to the main branch. So two workflow files:

  • .github/workflows/main-pr.yml
  • .github/workflows/main-deploy.yml

Pull Requests

Open .github/workflows/main-pr.yml. We want the workflow to fire on pull_request:

yml
name: Main Pull Requests Check
on:
  pull_request:

pull_request is written as an object because it accepts a lot of sub-properties. See the pull_request docs for the full list. The most common one is branches — which target branch the PR is going into:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches:
      - main

Means: "fire this workflow when someone opens a PR against main".

Inline array form works the same:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches: [main]

Wildcards work for Git Flow-style branch names like feature/*, release/*, hotfix/*:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches: [feature/*]

For us, just main.

Event done — onto the Job. We want three steps inside one job: ESLint, tests, build. For ESLint + tests we need both packages:

bash
npm i -D eslint jest

Initialize ESLint for the project and write a basic test — I'll skip the setup details. To let GitHub Actions invoke them, add two scripts in package.json:

json
{
  "name": "2022-express-example",
  "version": "0.0.0",
  "private": true,
  "scripts": {
    "start": "node ./bin/www",
    "lint": "eslint . --ext .js",
    "test": "jest"
  },
  "dependencies": {
    "cookie-parser": "~1.4.4",
    "debug": "~2.6.9",
    "express": "~4.16.1",
    "http-errors": "~1.6.3",
    "jade": "~1.11.0",
    "morgan": "~1.9.1"
  },
  "devDependencies": {
    "eslint": "^8.41.0",
    "eslint-config-airbnb-base": "^15.0.0",
    "eslint-plugin-import": "^2.27.5",
    "jest": "^29.5.0"
  }
}

lint runs ESLint, test runs Jest. With those in place, back to the workflow.

Lint job first:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint

If ESLint finds any error, the job fails and you'll see it on the Actions tab:

lint failure

Once you fix it, the job passes:

passing

Tests and build are basically the same pattern. Since this is an Express project there's no build step, so I'll skip it below:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run test

Notice runs-on: ubuntu-latest, uses: actions/checkout@v3, uses: actions/setup-node@v3 repeat across jobs. Why? Because each job runs in its own container — independent environment, fresh state. So each needs its own setup steps.

Also worth knowing — GitHub Actions doesn't guarantee job order by default. ESLint might run before tests, or after. If you want a specific order, use needs:

yml
name: Main Pull Requests Check
on:
  pull_request:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint
  test:
    needs:
      - lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run test

Without needs, the Actions visualization looks like this:

Parallel jobs

With needs, it chains:

Sequential jobs

Note The visualization lives under the project's Actions tab.

PR workflow done. Onto deploy.

Deploy

Open .github/workflows/main-deploy.yml. This one listens for push to main — fires after a PR merges:

yml
name: Main Deploy
on:
  push:
    branches:
      - main

That's it for the event side. Easy.

Deploy to Render.com

Where to deploy? I'll demo Render.com, since Render exposes a Deploy Hook.

You might ask — why bother with CD when Render can auto-deploy on branch push directly? In some setups you specifically need your CI/CD pipeline to finish before deploy can happen (run all the tests first, etc.), so doing it explicitly with CI/CD gives you that control. That's why we're going this route instead of letting Render handle the branch watching.

Some setup first. When you create a Render project, turn off Auto Deploy — otherwise Render will deploy before your CI even finishes:

Disable Auto Deploy

For an existing project, do this in the project's Settings:

Auto Deploy settings

Right next to that is the Deploy Hook URL — copy it. Anyone with this URL can trigger a deploy of your project, so keep it secret.

Now grab an API key. Click your avatar → "Account Setting":

Account Setting

Find "API Keys" and click "Create API Key":

Create API Key

Name it whatever — I used "Example". You'll get a key — save it, you'll need it.

Key

Back to the workflow. Even though the PR already passed lint and tests, you might want to re-run them on main before deploying — extra paranoia:

yml
name: Main Deploy
on:
  push:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint
  test:
    needs:
      - lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run test

(Basically copy-pasted from the PR workflow.)

Time to add the deploy step. First, paste your Deploy Hook URL and API key into the repo's Secrets — Settings tab on GitHub:

Secrets

Click "New repository secret" and add RENDER_DEPLOY_HOOK and RENDER_TOKEN. Click "Add secret" for each.

Back to main-deploy.yml. The deploy job needs needs: [lint, test] so deploy only runs after lint and test pass:

yml
name: Main Deploy
on:
  push:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint
  test:
    needs:
      - lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run test
  deploy:
    needs:
      - lint
      - test
    runs-on: ubuntu-latest

The deploy itself uses curl to hit the Render Deploy Hook:

yml
name: Main Deploy
on:
  push:
    branches:
      - main
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run lint
  test:
    needs:
      - lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: '16'
      - run: npm install
      - run: npm run test
  deploy:
    needs:
      - lint
      - test
    runs-on: ubuntu-latest
    steps:
      - name: Deploy
        run: |
          curl -X POST -H "Authorization: Bearer ${{ secrets.RENDER_TOKEN }}" \
          -H "Content-Type: application/json" \
          -d '{}' \
          ${{ secrets.RENDER_DEPLOY_HOOK }}

Open a PR, merge it to main, and GitHub Actions runs the chain — lint, test, deploy. Render takes the deploy hook ping and ships:
