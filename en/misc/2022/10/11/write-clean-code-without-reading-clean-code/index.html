You Don't Have to Read Clean Code to Write Decent Code

#Vue #JavaScript #React #Clean Code #Code Review #Code Style

Intro

This is a slightly unusual post — it took me close to a year to write. The reason it took that long is because it’s a collection of things I’ve seen from students during my time as a teaching assistant. I’m not pasting their code verbatim — I clean and adapt each example so nobody gets singled out. The goal is just “here’s how you could write this better”.

Why I’m writing this

This isn’t a Clean Code book recap — it’s not going to teach you naming theory or refactoring patterns top-down. It’s me sharing actual code patterns I’ve seen, with notes on why they hurt readability.

I’m also going to skip the heavily theoretical concepts. The vibe is “you could write this a bit better”, not “let me lecture you”.

Everyone has their own code style, so this isn’t a “you must do it my way” post. It’s more “here’s what I’d suggest, and why”.

And I’m a Vue dev, so some of the examples lean Vue.

Avoid pinyin (or local language transliterations) in code

I once had a student name functions and variables using Mandarin pinyin:

I wouldn’t recommend this — if you ever hand the project off to a non-Mandarin-speaking developer, they’ll have no idea what’s going on. And most devs default to English naming, so this becomes friction. Stick with English.

(If your English is weak, Google Translate works.)

Cut down on var

JavaScript has var, let, and const. In real projects you should default to ES6 let / const, mostly because var is function-scoped while let and const are block-scoped.

If you declare a var and then redeclare it inside an if block a few hundred lines later, the inner one overwrites the outer:

With let or const, the {} of the if block creates its own scope and the outer value is preserved:

So use let and const in real projects to avoid scoping surprises.

Even though ES6 has been everywhere for years now, people still write var in fresh code, so this is worth calling out.

Following the previous one — JavaScript has all three (var, let, const), but mixing var and let in the same file just confuses readers about scoping. For example:

So to avoid the problem entirely, don’t mix var and let. (Also: don’t earn yourself an awkward “let’s grab coffee” chat with a coworker.)

Don’t smush everything onto one line

I’ve seen someone literally collapse multi-line code into a single line:

Please don’t try to optimize for line count manually. That kind of compression should be done by Gulp, Webpack, or your bundler — not you. Doing it by hand just kills readability.

Don’t overuse the ternary operator

The ternary operator is great — but easy to over-use. Like this:

In practice, keep ternaries to simple conditions. For complex logic, an if statement is easier to read.

Scattered console.logs everywhere

console.log is the universal debug tool — but please remove them before shipping. Style guides usually flag console.log as a dev-only construct, and there are good reasons: you might accidentally leak sensitive info, and beyond that, a codebase full of stray logs reads like the author didn’t finish their work.

Why? The lifecycle hooks are different. Options API uses created and mounted. Composition API uses onMounted. The two writing styles are also very different. Imagine inheriting a codebase that mixes both.

Closing thoughts

That’s most of what I’ve seen as a TA. Hopefully these examples help you write a bit better, and save you from the “let’s grab a coffee” chat with a coworker.

These are guidelines, not rules. Every project has its own code style — some of the brace stuff in particular is context-dependent, and people do drop braces in real codebases.

A lot of the above also gets caught by a linter. But it’s worth building good habits before turning the linter on — that way when you do enable it, it doesn’t look like a wall of red.
