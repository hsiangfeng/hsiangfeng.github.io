Note Monorepo = "Mono" (single) + "Repo" (Repository) — multiple projects in one Git repo. It's the opposite of the "one project, one repo" Monolith pattern. In JavaScript, the most common way to build one is pnpm workspace paired with Vite.

Intro

What is a Monorepo? How does it differ from a regular project layout? I had a need for one recently, so I'm writing this up.

What is a Monorepo?

Monorepo — what's that? It's two words mashed together: Mono (meaning "single") + Repo (short for Repository). So Monorepo literally means "single repository".

Sounds a bit abstract? No worries. Let's talk about other project layouts first, then come back to Monorepo.

Monolith Repository

When you build a project, you typically go through these steps:

  • Create the project
  • Initialize version control (git init)
  • Build the features
  • Push to a remote (GitHub, GitLab, Bitbucket, etc.)

Repository

The project gets its own Git repo. That's the most basic project layout, and it's called a Monolith Repository (or just "Monolith") — a single-project Repository.

The Monolith layout looks something like this:

md
- Repository
  - .git
  - README.md
  - package.json
  - src
    - components
      - Button
      - Input
      - ...
    - pages
      - index
      - shopping
      - admin
      - ...
  - public
  - ...other files

One Git repo manages the whole project.

The Monolith setup is convenient and simple — a great choice for MVPs. But it has problems too. As the project grows, it gets harder to maintain. Say the project has these feature modules:

  • Frontend home
  • Frontend shopping page
  • Admin backend
  • ...and so on

Development and deployment slow down as the project gets bigger. Even a small change requires a full build + test + deploy cycle, and one bug anywhere can take the whole thing down:

Bye bye

To solve this, we get the next layout: Multi Repository.

Multi Repository

What is Multi Repository (Multi for short)? Take the Monolith and split it into multiple repos — each project gets its own. That way each one can be developed and deployed independently, solving Monolith's main problem.

The layout becomes:

md
- A Repository
  - .git
  - README.md
  - package.json
  - src
    - components
      - Button
      - Input
      - ...
    - pages
      - index
      - ...
  - public
  - ...other files
- B Repository
  - .git
  - README.md
  - package.json
  - src
    - components
      - Button
      - Input
      - ...
    - pages
      - shopping
      - ...
  - public
  - ...other files
- C Repository
  - .git
  - README.md
  - package.json
  - src
    - components
      - Button
      - Input
      - ...
    - pages
      - admin
      - ...
  - public
  - ...other files

This looks great — no more "one bug takes down everything". Each feature module is its own project, each one can ship independently.

But new problems show up. Look at the example above: Button and Input components are duplicated in every repo. That's duplication of effort, and when one of them breaks, you've gotta fix it in every project. Maintenance gets painful. Not just shared components either — CI/CD setup, dependency versions, all of that gets harder to keep in sync. QAQ...

Crashed out

Mono Repository

To solve that, we get the layout this post is actually about: Monorepo.

Things might be getting confusing, so let me draw it out. First, Monolith:

Monolith Repository

One project, one Repository, lots of feature modules inside, with the "everything breaks together" problem.

Then Multi:

Multi Repository

Monolith split into multiple repos, each one independent. Solves Monolith's problems, introduces duplication problems.

Finally, Monorepo:

Mono Repository

In Monorepo, all the projects sit inside the same Repository, managed by one Git. Solves Multi's duplication problem because shared resources can actually be shared — they all live in the same place.

Monorepo isn't a free lunch either. A few downsides:

  • The repo gets big — git clone takes longer
  • Shared code needs careful planning, otherwise projects affect each other
  • All projects share one repo, so you can't have per-project access control
  • Git repo size grows fast

So every layout has trade-offs. When picking one, think about what your project actually needs.

Building a Monorepo with Vite

Now let me walk through building a Vue3 Monorepo with Vite + pnpm. First, set up the environment:

bash
mkdir monorepo-example-pnpm-vue3
cd monorepo-example-pnpm-vue3
pnpm init

Next, a special setting. The pnpm Workspace docs say a Monorepo needs a pnpm-workspace.yaml file in the root. So create one in this project.

Notepnpm-workspace.yaml tells pnpm "this project is a Monorepo". Running pnpm install will install dependencies for all the sub-projects.

What goes in it? Let's look at the Monorepo layout again:

Mono Repository

One Git manages multiple projects, so a typical Monorepo looks like:

md
- Git
  - pnpm-workspace.yaml
  - package.json
  - apps
  - packages

The apps folder is where the sub-projects live. Create it in the root:

bash
mkdir apps

Then write this to pnpm-workspace.yaml:

yaml
packages:
  - apps/*

That's the basic Monorepo environment ready. Now let's create a Vue3 project. Move into apps and run:

bash
cd apps
pnpm create vue@latest

I'll create one called shopping:

Vue Create

After creation, the project should look like:

Project structure

Open apps/shopping/package.json and change the name to @apps/shopping so pnpm knows this project belongs to the Monorepo:

json
{
  "name": "@apps/shopping",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "dev": "vite",
    "build": "vite build",
    "serve": "vite preview"
  },
  "dependencies": {
    "vue": "^3.2.29"
  },
  "devDependencies": {
    "@vitejs/plugin-vue": "^1.9.2",
    "vite": "^2.7.13"
  }
}

Note@app/project-name is the Monorepo naming convention. Project names need to be unique inside a Monorepo, so the pattern is @ + apps + / + project-name. E.g. a shopping project becomes @apps/shopping.

Back at the root package.json, add a script to launch it:

json
{
  "name": "monorepo-example-pnpm-vue3",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "shopping": "pnpm --filter @apps/shopping"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}

pnpm --filter @apps/shopping tells pnpm to run commands against @apps/shopping. So running pnpm shopping dev from the root will start that project.

To launch the project, run:

bash
pnpm shopping dev

Notepnpm shopping dev means "run dev in @apps/shopping", where dev is the script defined in that project's package.json.

Shared dependencies

Now, all of our sub-projects use Vue-related dependencies:

json
{
  "name": "@apps/shopping",
  // ...other fields
  "dependencies": {
    "pinia": "^2.1.7",
    "vue": "^3.4.21",
    "vue-router": "^4.3.0"
  },
  "devDependencies": {
    "@rushstack/eslint-patch": "^1.8.0",
    "@vitejs/plugin-vue": "^5.0.4",
    "@vue/eslint-config-prettier": "^9.0.0",
    "@vue/test-utils": "^2.4.5",
    "eslint": "^8.57.0",
    "eslint-plugin-vue": "^9.23.0",
    "jsdom": "^24.0.0",
    "prettier": "^3.2.5",
    "vite": "^5.2.8",
    "vite-plugin-vue-devtools": "^7.0.25",
    "vitest": "^1.4.0"
  }
}

So we can hoist dependencies and devDependencies up to the root package.json to share them. Just update the scripts. After the change, the root package.json becomes:

json
{
  "name": "monorepo-example-pnpm-vue3",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1",
    "dev:shopping": "pnpm run -C apps/shopping dev",
    "build:shopping": "pnpm run -C apps/shopping build",
    "dev:order": "pnpm run -C apps/order dev",
    "build:order": "pnpm run -C apps/order build"
  },
  // ...other fields
}

Note Since dependencies and devDependencies moved to the root, the script flag changed from --filter to -C. This is what lets pnpm know which sub-project to run.

Now you just call the right script:

bash
pnpm dev:shopping

Project launched

To add a shared dependency, install it at the root level. For example, adding vue-loading-overlay:

bash
pnpm add vue-loading-overlay -w

Note Inside a Monorepo, installs need the -w flag so pnpm knows the package should be available to all sub-projects. For devDependencies, add -D too — e.g. pnpm add eslint -w -D.

Building a shared Vue component

Suppose we want a shared Button component. In a Monorepo, here's how to handle it. From the root:

bash
mkdir packages
cd packages

Then create a new Vue3 project:

bash
pnpm create vue@latest

Trim it down to just this:

Trimmed project

For a shared component, we don't need vite.config.js, public/, or App.vue. Open ui-lib/package.json and change it to:

json
{
  "name": "@packages/ui-lib",
  "version": "0.0.0",
  "private": true,
  "type": "module",
  "main": "src/main.js"
}

Note The main field matters — point it to your main.js so other projects can import the package.

Now use it from Shopping:

bash
cd apps/shopping
pnpm add @packages/ui-lib

And use the component inside Shopping:

html
<script setup>
import Button from '@packages/ui-lib';
</script>

<template>
   <Button label="Hi from Shopping" />
</template>

That's it — Shopping can now use the shared Button from ui-lib.

So that's a basic Monorepo setup. Full example on GitHub:

  • GitHub

Note After git clone, just run pnpm install at the root to install all sub-project dependencies.
