Monorepo? Another Way to Manage Projects — Building a Vue3 Monorepo With Vite + pnpm

#Vite #pnpm #Vue3 #Monorepo

Note
Monorepo = “Mono” (single) + “Repo” (Repository) — multiple projects in one Git repo. It’s the opposite of the “one project, one repo” Monolith pattern. In JavaScript, the most common way to build one is pnpm workspace paired with Vite.

Intro

What is a Monorepo? How does it differ from a regular project layout? I had a need for one recently, so I’m writing this up.

What is a Monorepo?

Monorepo — what’s that? It’s two words mashed together: Mono (meaning “single”) + Repo (short for Repository). So Monorepo literally means “single repository”.

Sounds a bit abstract? No worries. Let’s talk about other project layouts first, then come back to Monorepo.

Monolith Repository

When you build a project, you typically go through these steps:

  • Create the project
  • Initialize version control (git init)
  • Build the features
  • Push to a remote (GitHub, GitLab, Bitbucket, etc.)

The project gets its own Git repo. That’s the most basic project layout, and it’s called a Monolith Repository (or just “Monolith”) — a single-project Repository.

The Monolith layout looks something like this:

One Git repo manages the whole project.

The Monolith setup is convenient and simple — a great choice for MVPs. But it has problems too. As the project grows, it gets harder to maintain. Say the project has these feature modules:

  • Frontend home
  • Frontend shopping page
  • Admin backend
  • …and so on

Development and deployment slow down as the project gets bigger. Even a small change requires a full build + test + deploy cycle, and one bug anywhere can take the whole thing down:

To solve this, we get the next layout: Multi Repository.

Multi Repository

What is Multi Repository (Multi for short)? Take the Monolith and split it into multiple repos — each project gets its own. That way each one can be developed and deployed independently, solving Monolith’s main problem.

The layout becomes:

This looks great — no more “one bug takes down everything”. Each feature module is its own project, each one can ship independently.

But new problems show up. Look at the example above: Button and Input components are duplicated in every repo. That’s duplication of effort, and when one of them breaks, you’ve gotta fix it in every project. Maintenance gets painful. Not just shared components either — CI/CD setup, dependency versions, all of that gets harder to keep in sync. QAQ…

Mono Repository

To solve that, we get the layout this post is actually about: Monorepo.

Things might be getting confusing, so let me draw it out. First, Monolith:

One project, one Repository, lots of feature modules inside, with the “everything breaks together” problem.

Then Multi:

Monolith split into multiple repos, each one independent. Solves Monolith’s problems, introduces duplication problems.

Finally, Monorepo:

In Monorepo, all the projects sit inside the same Repository, managed by one Git. Solves Multi’s duplication problem because shared resources can actually be shared — they all live in the same place.

Monorepo isn’t a free lunch either. A few downsides:

  • The repo gets big — git clone takes longer
  • Shared code needs careful planning, otherwise projects affect each other
  • All projects share one repo, so you can’t have per-project access control
  • Git repo size grows fast

So every layout has trade-offs. When picking one, think about what your project actually needs.
