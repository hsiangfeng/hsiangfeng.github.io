VSCode Opens Every Link in the Integrated Browser? How to Disable Auto Takeover

#VSCode #Visual Studio Code #Integrated Browser #Simple Browser #Frontend Development #Editor Settings

Intro

VSCode 1.109 shipped a new Integrated Browser, and once workbench.browser.openLocalhostLinks is on, every localhost link in the terminal or chat view gets pulled into the editor — tiny pane, painful to use. This post covers how to disable the two key settings (workbench.browser.openLocalhostLinks and simpleBrowser.useIntegratedBrowser), and how to still call the integrated browser manually from the command palette when you actually want it.

Why did this start happening?

This behavior isn’t something VSCode has always had — it landed in 1.109 (around January 2026) along with a new feature called the Integrated Browser. When it’s on, any localhost link VSCode detects gets pulled into the editor instead of being handed off to your default browser. The idea is you don’t have to switch apps to see the page.

The intent is fine, but in practice it’s pretty annoying. The editor is already split between the file explorer, the editor itself, output panels, and so on — adding another browser pane shrinks usable space further. Trying to test RWD or open DevTools in there is rough. QQ…

What’s the relationship between the Integrated Browser and Simple Browser?

If you’ve been using VSCode for a while you’ve probably heard of Simple Browser — an old built-in browser extension. It’s pretty limited:

  • Only shows static pages
  • Can’t log into auth-required sites
  • Doesn’t keep cookies

So most people only ever used it for a quick “is localhost up?” peek.

So what’s the relationship between the new Integrated Browser and Simple Browser?

In short, the Integrated Browser is an upgraded version of Simple Browser, but they coexist — the new one doesn’t replace the old. A setting decides which one the command palette opens.

The Integrated Browser fills in what Simple Browser was missing:

  • Log into auth-required sites (cookies and sessions persist)
  • Browse any page, not just localhost
  • Built-in DevTools — inspect elements directly
  • Send the selected element to Copilot as context

Basically VSCode stuffed a small Chrome-like browser into the editor, so you can fully view a page without leaving VSCode — instead of the half-broken view Simple Browser used to give.

So why does clicking a link auto-route there?

The Integrated Browser by itself is just “one more browser you can use”. The “link gets hijacked” part comes from a separate setting — workbench.browser.openLocalhostLinks. When that’s on, anywhere in VSCode (terminal, chat, Markdown preview, output panel) — clicking a localhost link routes to the Integrated Browser instead of your system default browser.

Note
The 1.109 release notes describe it as “you can turn this on”, implying it’s off by default. But in some setups it gets flipped on without you noticing — maybe you clicked through a prompt, or a plugin enabled it for you. Next thing you know, every localhost link is opening inside VSCode.

While you’re here: Simple Browser’s own setting

After disabling workbench.browser.openLocalhostLinks, localhost links should behave normally again. But it’s worth knowing about another related setting — simpleBrowser.useIntegratedBrowser. This one belongs to the Simple Browser extension (a VSCode built-in extension), not to VSCode core. You can see it from the naming.

Setting Belongs to What it controls
workbench.browser.openLocalhostLinks VSCode core Whether clicking a localhost link auto-routes to the integrated browser
simpleBrowser.useIntegratedBrowser Simple Browser (built-in extension) Whether running the Simple Browser command uses the new integrated browser

The prefix tells you where the setting comes from — workbench.* is VSCode’s own, simpleBrowser.* is the Simple Browser extension’s. So if you’ve disabled the core setting but the command palette still routes you to the new integrated browser, this one’s still on — search and disable.

Closing thoughts

The Integrated Browser as a feature is actually nice — it’s just the default auto-takeover that gets in the way for anyone using an external browser as their main dev browser. That tiny pane really isn’t a substitute for Chrome or Edge in daily work, so disabling the auto-route is the better default.
