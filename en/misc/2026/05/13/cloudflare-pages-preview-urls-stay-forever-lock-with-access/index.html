Cloudflare Pages Preview URLs Stay Live Forever? Lock Them Down with Cloudflare Access

#Cloudflare #Cloudflare Pages #Cloudflare Access #Zero Trust #One-time PIN #Preview Deployments #Security

Intro

Every Cloudflare Pages deployment gets a hash-based preview URL like d682e28c.your-project.pages.dev. This post starts with the security problem of those URLs persisting forever, then walks through how to use Cloudflare Access (free for up to 50 users) to put a Self-hosted application policy in front of them — locking the previews behind an “Emails ending in” rule. Covers the step-by-step setup, how to write OR rules in Access policies, and three things to check when the One-time PIN email doesn’t arrive.

If you don’t have a Cloudflare domain or Pages project yet, you might want to read Google Domains shutting down? Complete walkthrough for setting up a domain and deploying with Cloudflare (article in Mandarin) first — it has the full Pages deployment flow and pairs nicely with this one.

The story

I’ve been playing with Hexo blog auto-deployment lately, wiring the project up to Cloudflare Pages so every git push triggers a new deploy. Cloudflare is nice about it — beyond the production your-project.pages.dev URL, every deployment also gets its own preview URL:

That d682e28c is the first 8 chars of the commit hash. It’s a super useful mechanism — you can share it with a colleague or a teacher to show “here’s what this version looks like”, without merging to main first.

But then something hit me: how long does that preview URL stay live? If the link ever got shared by accident, could it become a security problem? Some companies preview sensitive content on these (internal projects, unreleased features) — if the preview URLs hang around forever, that’s basically a permanent snapshot just sitting there, waiting for someone to dig it up.

I went and looked it up… yep. Forever.

How long do Cloudflare Pages preview URLs last?

Like I said: forever.

The official docs for Preview deployments spell it out pretty clearly:

When a preview deployment is published, it is given a unique, hash-based address — for example, <hash>.<project>.pages.dev. These are atomic and may always be visited in the future.

Translation: every preview deployment gets a hash-based URL, that URL is atomic (immutable), and it can always be accessed in the future.

In other words, unless you manually delete that deployment from the Cloudflare dashboard, that URL stays accessible forever, showing whatever you pushed that time.

Note
Not only does Cloudflare not auto-clean old previews — per the official docs, once a Pages project hits 100+ deployments, even deleting the whole project fails. You have to clean up old deployments via Wrangler first.

So what’s the risk?

The hash has 8 random hex characters, so brute-forcing one is impractical — these URLs aren’t going to get picked up by random scanners.

But that doesn’t mean there’s no risk. A few common scenarios:

  • You paste the preview link into a Slack or Discord group.
  • You send the link to a designer, PM, or teacher.
  • You tweet it out or screenshot it into a blog post.
  • You demo to a client and forget to delete it after the meeting.

If any of these end up in front of someone who shouldn’t see it, you can’t take the URL back. Even if you later delete the branch or rename the project, that preview deployment URL stays alive. That’s what bugs me about it.

So here’s the approach I use: put Cloudflare Access in front of the preview URL, so only authorized people can get in.

Locking previews with Cloudflare Access

Cloudflare Access is a ZTNA (Zero Trust Network Access) service under Cloudflare Zero Trust. Its job is to put an authentication gate in front of your site — anyone trying to get in has to authenticate first.

Does it cost money? Per the official Cloudflare Zero Trust pricing page, the Zero Trust Free plan supports up to 50 users for free. For a personal blog, that’s basically free-tier territory.

Step 1: Create a Self-hosted application

Go to the Cloudflare dashboard and find Zero Trust in the left sidebar. (First time here, you’ll be asked to pick a Team name — that becomes your login URL https://your-team-name.cloudflareaccess.com.)

Then follow this path:

In the destination section, fill in:

Field What to enter
Subdomain *
Domain Your Pages domain, e.g. your-project.pages.dev
Path leave blank

Setting subdomain to * is the key part — that way <any hash>.your-project.pages.dev all get covered by this application, and the production your-project.pages.dev is protected too.

Step 2: Configure an Access policy

Next, decide who’s allowed in. There’s an Access Policies block below:

Click Create new policy and fill in:

Field What to enter
Policy name e.g. Internal only
Action Allow
Rule selector Emails ending in
Value @gmail.com (your company domain)

If you also want your personal gmail to be able to log in, add another rule under the same policy:

Rule Selector Value
Rule 1 Emails ending in @gmail.com
Rule 2 Emails your-name@gmail.com

The two rules are OR’d together — either condition true is enough.

Step 3: Save the application

Last, fill in the application name and session duration (I’d suggest 24 hours), hit Create, done.

Now open an incognito window, paste your preview URL — you should see Cloudflare’s login page asking for an email. Enter a valid one, you get a PIN email, enter the PIN, and you’re in.

Set everything up but the PIN email isn’t coming?

If you’re like me, set everything up excitedly, and then the PIN email just… doesn’t arrive — stay calm. Three places to check.

First: spam folder. The Cloudflare PIN email is sent from noreply@notify.cloudflare.com, and Gmail very often files it under Spam or the Promotions tab. Subject looks something like Your one-time code is.... Check both folders.

Second: the email you entered doesn’t match the policy. This is the sneaky one. If your policy says “ends in @hexschool.com“ but you logged in with gmail.com, Cloudflare silently denies — doesn’t send the email, doesn’t tell you why. Just silence.

Note
From a security perspective this design makes sense (don’t tell attackers which emails are whitelisted), but for someone setting this up the first time it’s incredibly confusing. Double-check the email you entered actually matches the policy.

Third: One-time PIN isn’t enabled. Go to Zero Trust → Settings → Authentication → Login methods and confirm One-time PIN is active. If not, hit Add to enable it.

If your goal isn’t “block people” but “just nuke the old previews entirely”, you can use Cloudflare’s Wrangler CLI to list everything:

Combined with wrangler pages deployment delete <id> to delete one at a time, this is way faster than clicking through the dashboard. As mentioned earlier, once you’ve got 100+ deployments, even deleting the whole project fails — this command becomes essential for maintenance.

Wrapping up

Cloudflare Pages previews are double-edged: super convenient when you want to share, slightly scary when you forget the URL is permanent. Before writing this post I was naively treating preview URLs as “temporary test environments” — looking back, they’re permanent snapshots just sitting there waiting to be discovered.

Hopefully this gives you a clearer picture of how Cloudflare Pages previews work and how Access can lock them down.
