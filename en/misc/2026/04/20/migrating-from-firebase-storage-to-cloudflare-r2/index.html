Migrating from Firebase Storage to Cloudflare R2 — the Full Move

Intro

A while back I wrote Cache Firebase Storage images with Cloudflare Workers, completely free. The idea was to put a Cloudflare CDN layer in front of Firebase Storage to cut down on bandwidth costs.

After running with that for a while I noticed a problem: a cache is still just a cache. On cache miss, traffic still hits Firebase Storage. And as of February 2026, Firebase forces all projects to upgrade to the Blaze (paid) plan — which means the free-tier umbrella is gone, and going past the free quota means you get billed. Worse, there's no notification — the bill just quietly grows.

My blog runs two Firebase Storage buckets with around 4800 images total, billed separately. Every time I wanted to check the bill, I had to open two Firebase consoles. The whole thing felt fiddly enough that I decided to just rip the bandage off and move everything to Cloudflare R2.

Why Cloudflare R2?

Before deciding to move, I compared the costs:

Firebase Storage (Blaze)Cloudflare R2
Storage$0.026/GB/month$0.015/GB/month
Egress (download)$0.12 ~ $0.15/GB$0 (free)
Free tier5GB storage (specific region only)10GB storage + 10 million reads/month

The killer difference is egress. Firebase charges $0.12–$0.15 per GB downloaded; R2 has zero egress fees — no matter how many times your images get read, no charge.

Blog images are the textbook "write once, read many" workload. Every article view loads its images; if any post ever goes viral, Firebase's bandwidth bill spikes immediately. R2 has none of that risk.

For my 4800 images at around 10GB, I'm comfortably inside R2's free tier — meaning after the migration, monthly cost is $0.

Final architecture

After moving, the architecture looks like:

txt
User views article
    ↓
images.israynotarray.com (Cloudflare Worker)
    ↓
Check Referer (hotlink prevention)
    ↓
Read image from R2 bucket
    ↓
Return image (with cache headers)

Pretty similar to the previous Workers-caching-Firebase setup. The only change is what's behind the Worker — R2 instead of Firebase. Since R2 and Workers are both Cloudflare, integration is simpler and there's no external network hop, so it's faster too.

Migration steps

Walk-through of the whole process.

Step 1: Download all images from Firebase Storage

First, get all the images off Firebase Storage. 4800 images is not a "click each one" job, so I wrote a Node.js script.

You need a Firebase service account key to run it. Firebase Console → Project Settings → Service Accounts → Generate New Private Key gets you a JSON file.

bash
pnpm install @google-cloud/storage
js
import { Storage } from '@google-cloud/storage';
import { dirname, join } from 'path';
import { mkdirSync, existsSync, writeFileSync, readFileSync } from 'fs';

const BUCKET_NAME = 'your-project.appspot.com';
const DOWNLOAD_DIR = './firebase-images';
const KEY_FILE = './your-service-account-key.json';
const PROGRESS_FILE = './firebase-images/.progress.json';
const CONCURRENCY = 10;

function loadProgress() {
  if (existsSync(PROGRESS_FILE)) {
    return JSON.parse(readFileSync(PROGRESS_FILE, 'utf-8'));
  }
  return { downloaded: [] };
}

function saveProgress(progress) {
  writeFileSync(PROGRESS_FILE, JSON.stringify(progress, null, 2));
}

async function downloadWithRetry(file, destPath, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      await file.download({ destination: destPath });
      return true;
    } catch (err) {
      if (i === retries - 1) {
        console.error(`Download failed: ${file.name} - ${err.message}`);
        return false;
      }
      await new Promise((r) => setTimeout(r, 1000 * (i + 1)));
    }
  }
}

async function main() {
  const storage = new Storage({ keyFilename: KEY_FILE });
  const bucket = storage.bucket(BUCKET_NAME);

  const [files] = await bucket.getFiles();
  console.log(`Found ${files.length} files`);

  const progress = loadProgress();
  const alreadyDownloaded = new Set(progress.downloaded);
  const remaining = files.filter((f) => !alreadyDownloaded.has(f.name));

  let completed = alreadyDownloaded.size;
  const total = files.length;

  for (let i = 0; i < remaining.length; i += CONCURRENCY) {
    const batch = remaining.slice(i, i + CONCURRENCY);

    await Promise.all(
      batch.map(async (file) => {
        const destPath = join(DOWNLOAD_DIR, file.name);
        const destDir = dirname(destPath);

        if (!existsSync(destDir)) {
          mkdirSync(destDir, { recursive: true });
        }

        const ok = await downloadWithRetry(file, destPath);
        if (ok) {
          completed++;
          progress.downloaded.push(file.name);
          console.log(`[${completed}/${total}] ${file.name}`);
        }
      })
    );

    saveProgress(progress);
  }

  console.log(`Done! Successfully downloaded ${completed} files`);
}

main();

Key points in this script:

  • 10 parallel downloads: not so fast that it gets rate-limited, not so slow you're sitting around.
  • Resume support: progress is saved to .progress.json, so re-running after an interruption skips already-downloaded files.
  • Auto-retry: each file retries up to 3 times on failure.

After running, my numbers were 3920 files succeeded, 48 failed. But looking at the failure list, all the failures were paths ending in / like blogImg/Git/ — not actual files, just Firebase Storage's placeholder objects for "folders". So in practice, everything downloaded successfully, nothing to worry about.

Note Downloading from Firebase Storage incurs egress charges. For 4800 images at ~10GB, that's roughly $0.12/GB × 10GB = $1.2 USD, about 40 TWD. No need to panic.

Step 2: Upload to Cloudflare R2

Now upload to R2. R2 is S3-compatible, so I used the AWS SDK.

bash
pnpm install @aws-sdk/client-s3

In the Cloudflare dashboard, create your R2 bucket, then go to R2 → Manage R2 API Tokens → Create API Token to get the Access Key ID and Secret Access Key.

Create a .env.r2:

txt
CLOUDFLARE_ACCOUNT_ID=your_account_id
R2_ACCESS_KEY_ID=your_access_key_id
R2_SECRET_ACCESS_KEY=your_secret_access_key
R2_BUCKET_NAME=your_bucket_name

Upload script:

js
import { S3Client, PutObjectCommand } from '@aws-sdk/client-s3';
import { readFileSync, existsSync, readdirSync, statSync } from 'fs';
import { join, extname } from 'path';

const MIME_TYPES = {
  '.png': 'image/png',
  '.jpg': 'image/jpeg',
  '.jpeg': 'image/jpeg',
  '.gif': 'image/gif',
  '.webp': 'image/webp',
  '.svg': 'image/svg+xml',
};

// Read .env.r2
const env = {};
readFileSync('.env.r2', 'utf-8')
  .split('\n')
  .filter((line) => line.trim() && !line.startsWith('#'))
  .forEach((line) => {
    const [key, ...rest] = line.split('=');
    env[key.trim()] = rest.join('=').trim();
  });

const client = new S3Client({
  region: 'auto',
  endpoint: `https://${env.CLOUDFLARE_ACCOUNT_ID}.r2.cloudflarestorage.com`,
  credentials: {
    accessKeyId: env.R2_ACCESS_KEY_ID,
    secretAccessKey: env.R2_SECRET_ACCESS_KEY,
  },
});

function getAllFiles(dir) {
  const files = [];
  for (const entry of readdirSync(dir)) {
    const fullPath = join(dir, entry);
    if (statSync(fullPath).isDirectory()) {
      files.push(...getAllFiles(fullPath));
    } else {
      files.push(fullPath);
    }
  }
  return files;
}

const UPLOAD_DIR = './firebase-images';
const allFiles = getAllFiles(UPLOAD_DIR);

for (const filePath of allFiles) {
  // Important: strip the local directory prefix, keep only the actual path
  const key = filePath.replace(/^\.?\/?firebase-images\//, '');
  const ext = extname(filePath).toLowerCase();
  const contentType = MIME_TYPES[ext] || 'application/octet-stream';

  await client.send(
    new PutObjectCommand({
      Bucket: env.R2_BUCKET_NAME,
      Key: key,
      Body: readFileSync(filePath),
      ContentType: contentType,
      CacheControl: 'public, max-age=31536000, immutable',
    })
  );

  console.log(`Uploaded: ${key}`);
}

There's a gotcha here I'll cover later.

Step 3: Set up the Cloudflare Worker

After uploading, you need a Worker to proxy image reads from R2. In the Cloudflare dashboard, create a Worker, then Settings → Bindings → add an R2 Bucket binding. Variable name BUCKET, select your bucket.

Worker code:

js
export default {
  async fetch(request, env) {
    const url = new URL(request.url);

    if (url.pathname === '/' || url.pathname === '') {
      return Response.redirect('https://yourdomain.com/', 301);
    }

    if (request.method !== 'GET' && request.method !== 'HEAD') {
      return new Response('Method Not Allowed', { status: 405 });
    }

    // Hotlink prevention
    const allowedOrigins = [
      'https://yourdomain.com',
      'https://www.yourdomain.com',
      'http://localhost:4000',
    ];

    const referer = request.headers.get('Referer') || '';
    const origin = request.headers.get('Origin') || '';

    const isAllowed =
      (!referer && !origin) ||
      allowedOrigins.some((o) => referer.startsWith(o) || origin.startsWith(o));

    if (!isAllowed) {
      return Response.redirect('https://yourdomain.com', 302);
    }

    const path = decodeURIComponent(url.pathname.slice(1));

    const object = await env.BUCKET.get(path);

    if (!object) {
      return new Response('Not Found', { status: 404 });
    }

    const headers = new Headers();
    object.writeHttpMetadata(headers);
    headers.set('etag', object.httpEtag);
    headers.set('Cache-Control', 'public, max-age=31536000, immutable');
    headers.set('CDN-Cache-Control', 'max-age=31536000');
    headers.set('Access-Control-Allow-Origin', '*');

    return new Response(object.body, { headers });
  },
};

Compared to the old Worker that cached Firebase, the biggest change is the backend: fetch(firebaseUrl) is replaced by env.BUCKET.get(path), reading directly from R2 — no external network hops.

Step 4: Bulk-replace article URLs

Once the Worker's deployed and images are loading correctly, the last step is rewriting all the old URLs across your articles.

The original Firebase Storage URL looks like:

txt
https://firebasestorage.googleapis.com/v0/b/your-project.appspot.com/o/blogImg%2FGit%2F202107092329.png?alt=media&token=xxx

Should become:

txt
https://images.yourdomain.com/blogImg/Git/202107092329.png

I wrote a script:

js
import { readFileSync, writeFileSync } from 'fs';
import { execSync } from 'child_process';

const NEW_BASE = 'https://images.yourdomain.com';
const FIREBASE_PATTERN =
  /https:\/\/firebasestorage\.googleapis\.com\/v0\/b\/your-project\.appspot\.com\/o\/([^?]+)\?alt=media&token=[a-f0-9-]+/g;

const files = execSync(
  'grep -rl "firebasestorage.googleapis.com/v0/b/your-project" source/',
  { encoding: 'utf-8' }
)
  .trim()
  .split('\n')
  .filter(Boolean);

for (const file of files) {
  let content = readFileSync(file, 'utf-8');

  const newContent = content.replace(FIREBASE_PATTERN, (match, encodedPath) => {
    // Convert only %2F to /, leave other encodings untouched
    const urlPath = encodedPath.replace(/%2F/gi, '/');
    return `${NEW_BASE}/${urlPath}`;
  });

  writeFileSync(file, newContent, 'utf-8');
}

My blog had 289 files and 1436 URLs to replace, plus a second bucket with 154 files and 835 URLs — totaling 443 files and 2271 URLs, all done by the script.

Gotchas I hit

The migration went pretty smoothly overall, but I did hit some sharp edges. Logging them here in case you're heading down the same path.

Gotcha 1: URL encoding

This one hurt the most.

Firebase Storage URLs encode path separators (/) as %2F, and Chinese characters and spaces are also percent-encoded. When replacing, I initially used decodeURIComponent on the whole path — which left spaces and Chinese characters directly in the URL:

txt
https://images.israynotarray.com/blogImg/Hexo/試著學 Hexo 系列/截圖 2020-07-11 16.15.43.png

Markdown's image syntax ![](url) breaks on a literal space, so every image with Chinese or spaces in the path was busted.

The right approach is to convert only %2F to /, leave everything else encoded:

js
// Wrong
const decodedPath = decodeURIComponent(encodedPath);

// Right
const urlPath = encodedPath.replace(/%2F/gi, '/');

Gotcha 2: Upload path prefix

When uploading to R2, the key should be blogImg/Git/202107092329.png, but my prefix replacement didn't quite match (./firebase-images/ vs firebase-images/), so the key ended up as firebase-images/blogImg/Git/202107092329.png — with an extra directory level.

You can patch around it in the Worker, but it's ugly. Better to use a regex in the upload script:

js
// Don't use plain string replace, regex is safer
const key = filePath.replace(/^\.?\/?firebase-images\//, '');

Gotcha 3: Google Analytics tracking parameters

Some image URLs had a mysterious &_gl=1*5qrrwu*_ga*... tacked on. This is Google Analytics's cross-domain tracking param that gets attached when you copy a URL from the Firebase Console — and it makes images fail to load.

Cleanup is trivial:

js
content = content.replace(/&_gl=[^\s)"']*/g, '');

20 files and 55 URLs in my blog had this. Cleaned them all up and everything came back fine.

After the migration

The big win is no longer worrying about bandwidth costs. No matter how many people read which post or how many times the images get loaded, the cost is $0. I can finally sleep at night (lol).

Closing thoughts

The whole migration took about an afternoon, most of which was waiting for downloads and uploads. Once the scripts were running, it was just "wait for it to finish". Most of the gotchas were URL encoding stuff, but once you remember the rule "only convert %2F, leave the rest alone", you'll mostly avoid the pitfalls.

If you're also using Firebase Storage for images and you're tired of looking at that bandwidth bill, moving to R2 is genuinely the most cost-effective option right now.
