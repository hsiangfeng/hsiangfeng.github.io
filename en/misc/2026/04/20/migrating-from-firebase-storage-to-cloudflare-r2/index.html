Migrating from Firebase Storage to Cloudflare R2 — the Full Move

#Node.js #Cloudflare #Firebase #Cloudflare R2 #Cloudflare Workers

Intro

A while back I wrote Cache Firebase Storage images with Cloudflare Workers, completely free. The idea was to put a Cloudflare CDN layer in front of Firebase Storage to cut down on bandwidth costs.

After running with that for a while I noticed a problem: a cache is still just a cache. On cache miss, traffic still hits Firebase Storage. And as of February 2026, Firebase forces all projects to upgrade to the Blaze (paid) plan — which means the free-tier umbrella is gone, and going past the free quota means you get billed. Worse, there’s no notification — the bill just quietly grows.

My blog runs two Firebase Storage buckets with around 4800 images total, billed separately. Every time I wanted to check the bill, I had to open two Firebase consoles. The whole thing felt fiddly enough that I decided to just rip the bandage off and move everything to Cloudflare R2.

Why Cloudflare R2?

Before deciding to move, I compared the costs:

Firebase Storage (Blaze) Cloudflare R2
Storage $0.026/GB/month $0.015/GB/month
Egress (download) $0.12 ~ $0.15/GB $0 (free)
Free tier 5GB storage (specific region only) 10GB storage + 10 million reads/month

The killer difference is egress. Firebase charges $0.12–$0.15 per GB downloaded; R2 has zero egress fees — no matter how many times your images get read, no charge.

Blog images are the textbook “write once, read many” workload. Every article view loads its images; if any post ever goes viral, Firebase’s bandwidth bill spikes immediately. R2 has none of that risk.

For my 4800 images at around 10GB, I’m comfortably inside R2’s free tier — meaning after the migration, monthly cost is $0.

Pretty similar to the previous Workers-caching-Firebase setup. The only change is what’s behind the Worker — R2 instead of Firebase. Since R2 and Workers are both Cloudflare, integration is simpler and there’s no external network hop, so it’s faster too.

Migration steps

Walk-through of the whole process.

Step 1: Download all images from Firebase Storage

First, get all the images off Firebase Storage. 4800 images is not a “click each one” job, so I wrote a Node.js script.

You need a Firebase service account key to run it. Firebase Console → Project Settings → Service Accounts → Generate New Private Key gets you a JSON file.

Key points in this script:

  • 10 parallel downloads: not so fast that it gets rate-limited, not so slow you’re sitting around.
  • Resume support: progress is saved to .progress.json, so re-running after an interruption skips already-downloaded files.
  • Auto-retry: each file retries up to 3 times on failure.

After running, my numbers were 3920 files succeeded, 48 failed. But looking at the failure list, all the failures were paths ending in / like blogImg/Git/ — not actual files, just Firebase Storage’s placeholder objects for “folders”. So in practice, everything downloaded successfully, nothing to worry about.

Note
Downloading from Firebase Storage incurs egress charges. For 4800 images at ~10GB, that’s roughly $0.12/GB × 10GB = $1.2 USD, about 40 TWD. No need to panic.

Gotchas I hit

The migration went pretty smoothly overall, but I did hit some sharp edges. Logging them here in case you’re heading down the same path.

Gotcha 1: URL encoding

This one hurt the most.

Firebase Storage URLs encode path separators (/) as %2F, and Chinese characters and spaces are also percent-encoded. When replacing, I initially used decodeURIComponent on the whole path — which left spaces and Chinese characters directly in the URL:

Markdown’s image syntax ![](url) breaks on a literal space, so every image with Chinese or spaces in the path was busted.

The right approach is to convert only %2F to /, leave everything else encoded:

Gotcha 2: Upload path prefix

When uploading to R2, the key should be blogImg/Git/202107092329.png, but my prefix replacement didn’t quite match (./firebase-images/ vs firebase-images/), so the key ended up as firebase-images/blogImg/Git/202107092329.png — with an extra directory level.

Gotcha 3: Google Analytics tracking parameters

Some image URLs had a mysterious &_gl=1*5qrrwu*_ga*... tacked on. This is Google Analytics’s cross-domain tracking param that gets attached when you copy a URL from the Firebase Console — and it makes images fail to load.

After the migration

Before/after comparison:

Before After
Storage 2 Firebase Storage buckets 1 Cloudflare R2 bucket
Management 2 Firebase consoles 1 Cloudflare dashboard
Egress fee $0.12/GB, no ceiling $0
Monthly cost Depends on traffic, unpredictable $0 (within free tier)
Image domain 2 different domains Unified images.israynotarray.com
Traffic spike risk Yes — bill jumps No, bandwidth is always free

The big win is no longer worrying about bandwidth costs. No matter how many people read which post or how many times the images get loaded, the cost is $0. I can finally sleep at night (lol).

Closing thoughts

The whole migration took about an afternoon, most of which was waiting for downloads and uploads. Once the scripts were running, it was just “wait for it to finish”. Most of the gotchas were URL encoding stuff, but once you remember the rule “only convert %2F, leave the rest alone”, you’ll mostly avoid the pitfalls.

If you’re also using Firebase Storage for images and you’re tired of looking at that bandwidth bill, moving to R2 is genuinely the most cost-effective option right now.
