Cache Firebase Storage Images with Cloudflare Workers — Completely Free Bandwidth Savings

Intro

If you're using Firebase Storage to host blog images like I was, you've probably noticed bandwidth costs creeping up. Every time someone visits an article, the images load straight from Firebase Storage, and all of that bandwidth lands on your Firebase bill. The more articles and readers you have, the worse the bandwidth bill gets.

Is there a way to put a cache layer in front so images don't get fetched from Firebase Storage every time? Yes — and completely free. This post walks through how to use Cloudflare Workers with a custom subdomain to cache Firebase Storage images, letting Cloudflare's global CDN absorb most of the traffic.

How does this work?

Before getting into the implementation, let's cover the principle. Otherwise you won't know why it works.

In short, we're going to put Cloudflare in front of Firebase Storage as a "middleman" that caches the images:

txt
User → images.yourdomain.com → Cloudflare CDN (cache layer) → Firebase Storage
                                      ↑
                              If cached, return directly — never touch Firebase

The first time someone requests an image, Cloudflare fetches it from Firebase Storage and stashes a copy at the edge. Every subsequent request goes straight from the edge — Firebase doesn't get hit, which means no bandwidth charge on the Firebase side.

Is this really free?

Before we go any further, let me cover the cost — that's probably what most people care about anyway:

ItemCost
Cloudflare accountFree
Cloudflare DNS hostingFree
Cloudflare WorkersFree (100,000 requests/day, no credit card)
Cloudflare CDN cached trafficFree (no bandwidth cap)
Custom domain for WorkerSupported on the free plan

All of it free. Cloudflare is one of the few CDNs that doesn't bill on bandwidth, and people have run 5–9 TB/month on the free plan without trouble.

And once a request hits cache, it doesn't go through the Worker at all — so it doesn't consume your daily 100k Worker invocation quota either. For blog images, this is more than enough.

Note The one requirement is that you own a domain and host its DNS with Cloudflare (which is also free).

Watch out for the paid traps

The Workers free plan doesn't require a credit card, but the dashboard has features (Durable Objects, Workers AI, etc.) that might nudge you toward the $5/month paid plan. Don't click any "upgrade" button — we only need basic Worker functionality, none of those advanced features.

Why a subdomain?

You might be thinking: "why not just use yourdomain.com/images/...?"

Simple — if your main domain is already running a blog or another project, the paths can collide easily. A subdomain (like images.yourdomain.com) is fully isolated and won't interfere with anything on the main site. And the final image URL ends up looking like:

txt
https://images.yourdomain.com/blog2/ai/my-post/screenshot.png

Way cleaner than the long Firebase Storage URL.

Let's build it

Now that we've covered the principle and cost, on to the implementation.

Step 1: Host your domain DNS on Cloudflare

If your domain isn't on Cloudflare yet:

  1. Sign up for a Cloudflare account (free).
  2. Click Add a site → enter your domain.
  3. Pick the Free plan.
  4. Cloudflare gives you two nameservers.
  5. At your domain registrar, change nameservers to the ones Cloudflare gave you.
  6. Back in Cloudflare, click confirm, wait for DNS to propagate (usually minutes to hours).

Note If your domain's already on Cloudflare, skip to Step 2.

Step 2: Create a Cloudflare Worker

  1. Cloudflare dashboard → left sidebar → Workers & Pages.
  2. Click Create → Create Worker.
  3. Pick "Start from Hello World!".
  4. Give it a name, e.g. image-proxy.
  5. Click Deploy to create it, then Edit code.
  6. Delete the default code and paste this:
js
export default {
  async fetch(request) {
    // Restrict allowed origins
    const allowedOrigins = [
      'https://yourdomain.com',
      'https://www.yourdomain.com',
      'http://localhost', // for local dev
    ];

    const referer = request.headers.get('Referer') || '';
    const origin = request.headers.get('Origin') || '';

    const isAllowed =
      (!referer && !origin) || // allow direct browser address-bar access
      allowedOrigins.some((o) => referer.startsWith(o) || origin.startsWith(o));

    if (!isAllowed) {
      return new Response('Forbidden', { status: 403 });
    }

    // Build the Firebase Storage URL from the path
    const url = new URL(request.url);
    const path = encodeURIComponent(url.pathname.slice(1));
    const firebaseUrl = `https://firebasestorage.googleapis.com/v0/b/YOUR_PROJECT.appspot.com/o/${path}?alt=media`;

    const response = await fetch(firebaseUrl);

    if (!response.ok) {
      return new Response('Not Found', { status: 404 });
    }

    // Set cache headers
    const headers = new Headers(response.headers);
    headers.set('Cache-Control', 'public, max-age=31536000, immutable');
    headers.set('CDN-Cache-Control', 'max-age=31536000');
    headers.set('Access-Control-Allow-Origin', '*');

    return new Response(response.body, {
      status: response.status,
      headers,
    });
  },
};

Note Remember to replace YOUR_PROJECT.appspot.com with your actual Firebase project name, and allowedOrigins with your own domain.

  1. Click Save and Deploy.

What does this code do?

In case you're wondering what's actually happening — three things:

  1. Domain restriction: checks the request origin and only allows your own domain and local dev. Stops other people from hotlinking your image proxy.
  2. Reverse proxy: builds the Firebase Storage URL from the request path, fetches from Firebase.
  3. Cache headers: adds cache-control headers on the response, telling the Cloudflare CDN and the browser to cache the image for a year.

The localhost entry exists so local development still shows images. Since we're using startsWith for matching, any port (4000, 3000, 8080…) works fine.

Step 3: Bind a subdomain

  1. In the Worker → Settings → Domains & Routes.
  2. Click Add → Custom Domain.
  3. Enter images.yourdomain.com.
  4. Cloudflare creates the DNS record for you, click confirm.

Step 4: Configure Firebase Storage Rules

Because our Worker requests Firebase Storage without a token, the images need to be publicly readable.

Firebase Console → Storage → Rules:

txt
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /blog2/{allPaths=**} {
      allow read;
    }
  }
}

This only opens up blog2/ for public read — other paths are unaffected.

You might be worried about the security implications, but consider: these images were already public via blog articles — anyone could see them. The original token was sitting in your HTML for anyone who opens devtools to grab anyway. Switching to public read doesn't actually add any new risk.

Step 5: Verify

Open a browser and type in:

txt
https://images.yourdomain.com/blog2/your-folder/your-image.png

If you see the image, you're good.

Updating image links

Now you need to replace all the Firebase Storage image URLs in your blog with the new format.

Original looks like:

txt
https://firebasestorage.googleapis.com/v0/b/YOUR_PROJECT.appspot.com/o/blog2%2Ffolder%2Fimage.png?alt=media&token=xxx

Becomes:

txt
https://images.yourdomain.com/blog2/folder/image.png

The rules:

  1. Swap the Firebase domain for images.yourdomain.com.
  2. Drop the ?alt=media&token=xxx.
  3. Convert %2F to /.

If you have lots of articles, doing it by hand isn't realistic. Bulk replace with sed:

bash
# Step 1: replace domain and strip token
grep -rl "firebasestorage\.googleapis\.com/v0/b/YOUR_PROJECT\.appspot\.com" source/ | \
while IFS= read -r f; do
  sed -i '' 's|https://firebasestorage\.googleapis\.com/v0/b/YOUR_PROJECT\.appspot\.com/o/\([^?]*\)?alt=media&token=[a-f0-9\-]*|https://images.yourdomain.com/\1|g' "$f"
done

# Step 2: replace %2F with /
for i in $(seq 1 5); do
  grep -rl "images\.yourdomain\.com/[^)]*%2F" source/ 2>/dev/null | \
  while IFS= read -r f; do
    sed -i '' 's|\(images\.yourdomain\.com/[^)]*\)%2F|\1/|g' "$f"
  done
done

Note Step 2 needs to run multiple times because sed only replaces one %2F per pass — the loop ensures all of them get converted. Also, if you have images from multiple Firebase projects, double-check you're only replacing URLs from the target project so you don't accidentally touch other projects' image URLs.

Important: use / instead of %2F in URL paths

One detail worth highlighting.

Firebase Storage's URL format uses %2F to separate paths (because the whole file path is treated as a single URL segment), but on our own subdomain, you must use a regular /:

txt
❌ https://images.yourdomain.com/blog2%2Fai%2Fmy-post%2Fimage.png
✅ https://images.yourdomain.com/blog2/ai/my-post/image.png

Why? Because Cloudflare doesn't handle %2F correctly in this context — you'll get 404s. The Worker internally converts / back to Firebase's %2F format (via encodeURIComponent), so on your end you just use normal paths and don't need to worry about encoding.

Getting URLs for new images

How do you get URLs for new images going forward? The rule is just:

txt
https://images.yourdomain.com/ + the path in Firebase Storage

For example, if you upload an image to blog2/ai/my-post/screenshot.png in Firebase Storage, the URL is:

txt
https://images.yourdomain.com/blog2/ai/my-post/screenshot.png

No more tokens. Just concatenate the path. Way more convenient.

How the caching works

Last bit: let's cover how the caching actually flows, so you can feel more confident about the setup.

The Worker sets two key headers:

js
headers.set('Cache-Control', 'public, max-age=31536000, immutable');
headers.set('CDN-Cache-Control', 'max-age=31536000');

These set up two cache layers.

Layer 1: Cloudflare CDN edge cache

CDN-Cache-Control tells Cloudflare's edge nodes: "cache this for a year". On the first request, Cloudflare fetches from Firebase and stores a copy at the edge closest to the user. After that, anyone requesting the same image gets it straight from the edge — no Worker invocation, no Firebase hit.

Layer 2: Browser local cache

Cache-Control: public, max-age=31536000, immutable tells the browser: "cache this locally for a year, and it won't change". So when the same user visits the same post again, the browser uses its local copy — doesn't even hit Cloudflare.

Overall flow

First request (cold cache):

txt
User → Cloudflare edge (no cache) → Worker → Firebase Storage
                ↓
        Store copy at edge + return to user

Second request (CDN hit):

txt
User → Cloudflare edge (cached) → return directly

Same user revisiting (browser cache hit):

txt
User → browser local cache → render directly

Since blog images don't change after upload, setting a 1-year cache with immutable is totally reasonable.

Closing thoughts

With Cloudflare Workers + CDN caching, you can dramatically cut Firebase Storage bandwidth — for $0. Setup doesn't take much time, but the savings are long-term.

Recap of the benefits:

  • Completely free: no credit card, no paid plan needed.
  • Global acceleration: Cloudflare has 300+ edge nodes worldwide; images come from the closest one.
  • Reduced bandwidth: cache hits never touch Firebase, bandwidth goes to zero.
  • Domain restriction: only your own site can access — stops hotlinking.
  • Shorter URLs: from long Firebase URLs to clean subdomain paths.

If you're also using Firebase Storage for images, this approach is worth trying.
