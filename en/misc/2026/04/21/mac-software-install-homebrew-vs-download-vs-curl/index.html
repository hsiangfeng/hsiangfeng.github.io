How Should You Install Software on a Mac? Homebrew vs Official Download vs curl

#Mac #macOS #nvm #Homebrew #mise #Software Installation #Development Environment

Intro

If you’ve just started using a Mac for development, you’ve probably noticed something: the same tool has several different install paths. Take Git — some say Xcode Command Line Tools is enough, others recommend Homebrew. Take Node.js — some say download from the official site, some say use nvm, some say use Homebrew.

This post sorts through the common Mac install methods and pairs them with concrete examples so you know which one fits which situation.

Why everyone recommends Homebrew

Before going through the various install methods, let me cover why most tutorials default to Homebrew.

Homebrew is the most-used package manager on macOS. If you haven’t installed it yet, you can follow my older post Make your Mac terminal pretty — the only guide you need (article in Mandarin). In short, it lets you manage software with terminal commands. What makes it better than just downloading from official sites?

Manage all your software in one place

Whether you install command-line tools or GUI apps through Homebrew, you can manage all of them with commands:

If you download tools from official sites instead, updates mean going back to those sites manually, and uninstalling means hunting down leftover files. After 20–30 tools, it gets really hard to manage.

Automatic dependency handling

Some tools require other tools to be installed first — like a package that needs openssl or cmake. If you’re installing by hand, you have to track down each dependency yourself. Homebrew handles them automatically — whatever’s missing, it pulls in.

Use Brewfile to back up and restore your whole machine

This is one of Homebrew’s most useful features. You can export everything you’ve installed via Homebrew as a Brewfile:

So when you switch to a new Mac or want to sync dev environments with a colleague, share the Brewfile and they’re one command away from having the same setup. You can’t do that by downloading from individual sites.

Formulae vs Cask

Homebrew has two install modes for two kinds of software:

brew install installs command-line tools. brew install --cask installs GUI apps. Using Cask gives you the same end result as downloading a .dmg, with the bonus that you can run brew upgrade --cask to update everything in one shot and brew uninstall --cask for a clean uninstall.

Other install methods

Beyond Homebrew, there are a few other approaches, each with their place.

Xcode Command Line Tools

If you’re not doing iOS development, you don’t need full Xcode — Xcode Command Line Tools is enough. It includes Git, make, clang, and the basic dev tools:

This is usually the first command you run on a new Mac, since many tools (Homebrew included) depend on it.

Official .dmg / .pkg downloads

Going to the software’s official site and downloading the installer. Simple and direct, but updates are on you, and uninstalls don’t always clean up — some software scatters config files around the system.

If the same software is available via Homebrew, prefer Homebrew. Much easier to manage.

curl install scripts

Some tools give you a curl one-liner:

This pattern usually shows up when the tool needs to modify your shell config (like .zshrc), which isn’t a great fit for Homebrew’s standard flow. Homebrew itself installs via curl.

Note
Before running a curl script, make sure the source is the official repo. You’re literally executing code from the internet on your machine.

Version managers

If you’re maintaining multiple projects that each need different language versions (project A on Node.js 18, project B on Node.js 20), you want a version manager — not a plain Homebrew install.

Common version managers:

Tool Manages Install method
nvm Node.js curl script
pyenv Python Homebrew or curl script
rbenv Ruby Homebrew
mise Everything (900+ tools) Homebrew or curl script

Of these, mise can manage Node.js, Python, Ruby, Go, Java, etc., all at once — so you don’t have to install a separate version manager per language.

If you want to see how nvm works in practice, I wrote about it in Auto-switch Node.js version with nvm use when opening a project on Mac (article in Mandarin).

Real examples

Now let’s walk through a few common tools and the install method that fits each.

Git

Git comes with Xcode Command Line Tools, but the bundled version is usually older. If you want a newer version, just use Homebrew:

The Homebrew version auto-overrides the Xcode CLT one. Updates are just brew upgrade git.

Recommendation: Xcode CLT is fine for most users. If you care about version, use Homebrew.

Node.js

Node.js is the tool people get confused about the most:

Downloading the .pkg from the official site isn’t great because it installs at a system-level path, which makes switching versions or uninstalling painful.

Recommendation: Frontend or Node.js devs should use nvm or mise. If you only touch Node.js occasionally, Homebrew is enough.

Python

Mac ships with Python, but it’s an old version and not great for dev:

Recommendation: If you write Python regularly, use pyenv or mise. If something else just needs Python as a dependency, Homebrew is fine.

VS Code

VS Code is a GUI app — both methods get you the same result:

Or just download .dmg from the VS Code site.

Recommendation: If you’re already on Homebrew, use Cask for consistency.

Summary

Quick reference table:

What you’re installing Recommended method
Command-line tools (git, wget, jq) Homebrew
GUI apps (VS Code, Chrome) Homebrew Cask or official download
Languages needing multiple versions (Node.js, Python) Version manager (nvm, pyenv, mise)
Xcode, Mac-only apps Mac App Store
Version managers themselves (nvm, mise) curl script or Homebrew

My personal new-Mac install order looks something like this:

  1. xcode-select --install (basic dev tools)
  2. Install Homebrew (the package management core)
  3. All command-line tools via Homebrew
  4. Node.js via nvm
  5. GUI apps via Homebrew Cask or official download depending on the situation

You don’t have to memorize every method. The rule of thumb is: if Homebrew can manage it, use Homebrew; if you need multiple versions, use a version manager. That alone covers most decisions.
