Core Web Vitals for SEO: LCP, FID, and CLS

#SEO #Chrome #Google #Google Search Console #LCP #FID #CLS

Note
This post was written in May 2021. In March 2024, Google replaced FID
with INP (Interaction to Next Paint) in the Core Web Vitals set. The
conceptual framing here still applies, but if you’re reading this in
2024+ and want to act on it, treat the FID section as historical and
check Google’s docs for INP guidance.

Intro

A heads-up for the post — I’m writing this because I noticed my own site getting flagged with a bad page experience score. So I’m capturing what I learned.

What happened

Recently I poked around Google Search Console and saw a new tab — “Page Experience / Core Web Vitals”. I clicked in and immediately got a wall of red.

Looked it up — turns out around mid-2020, Google quietly added “Core Web Vitals” reporting. (Embarrassed I missed it.)

Google’s been pushing for better page experience for years — mobile friendliness, HTTPS, etc. Then in 2021 they rolled out Core Web Vitals, and the important bit is:

“Core Web Vitals factor into SEO ranking.”

My site got diagnosed with CLS issues on both mobile and desktop.

So writing this post to capture the basics.

Core Web Vitals

Google emphasizes in Search Console: bad performance = bad UX. From their docs:

Why page performance matters.
The longer a page takes to load, the more bounce rate climbs. For example:
If load time goes from 1s to 3s, bounce rate increases by 32%.
If load time goes from 1s to 6s, bounce rate increases by 106%.

(Bounce rate = the probability a user leaves your site without engaging further.)

Core Web Vitals has three key metrics:

  • LCP (Largest Contentful Paint)
  • FID (First Input Delay)
  • CLS (Cumulative Layout Shift)

LCP (Largest Contentful Paint)

The time from when the user requests a URL to when the largest visible content element in the viewport has rendered. The largest element is usually an image, video, or large block-level text. LCP matters because it gives users a clear “the page is loading” signal.
“Aggregated LCP” in the report (Agg LCP) represents the LCP time achieved by 75% of users accessing a specific URL in the group.

In plainer terms — when I land on your site, how long until the biggest thing in the viewport is rendered? Includes images and video.

Usually the slow part is images and video. So compress before uploading — I use TinyPNG:

TinyPNG is great — shows you the compression result clearly:

Google’s thresholds:

Good Needs Improvement Poor
LCP ≤ 2.5s ≤ 4s > 4s

FID (First Input Delay)

The time from a user’s first interaction with the page (clicking a link, tapping a button) to when the browser responds to that interaction. Measured against the user’s first interaction with an interactive element. This timing matters for pages that need active user engagement, because it’s the point where the page becomes interactive.
“Aggregated FID” in the report (Agg FID) represents the lowest value experienced by 75% of users for a specific URL.

Strongly related to “first impression”. Think about meeting someone new — you try to leave a good first impression, right? Same idea for sites. The user’s first interaction with your site should feel responsive. If they click a button and nothing happens for 5 seconds — that’s a brutal first impression.

Common cause — waiting on the backend, or your JS bundle is huge and the main thread is busy doing other work when the user tries to interact.

Good Needs Improvement Poor
FID ≤ 100ms ≤ 300ms > 300ms

CLS (Cumulative Layout Shift)

CLS measures the sum of all unexpected layout shifts that occur during the entire lifespan of a page. Score ranges from 0 (no shift) to any positive number — higher = more shift. The metric matters because if elements move around while users are trying to interact, it actively hurts the experience. If you can’t figure out why your score is high, try interacting with the page to see what shifts.
“Aggregated CLS” in the report (Agg CLS) represents the lowest CLS score for 75% of users accessing a specific URL (excluding extreme outliers).

This one’s special — my site has this issue. What’s CLS? Google’s image makes it crystal clear:

Basically — your page jumps around while users are trying to use it, causing accidental misclicks.

My blog flagged for CLS likely from two culprits:

  • Google AdSense — Yep, ads. They load asynchronously and shift layout. Should be ignored as a Google-affiliated service, but might not be.
  • Lazy loading — I had image lazy-loading enabled to save bandwidth. I’ve turned it off and resubmitted for Google review. If that fixes the issue, lazy loading is definitely off the table.
Good Needs Improvement Poor
CLS ≤ 0.1 ≤ 0.25 > 0.25

If your site is too new to have enough data, PageSpeed Insights is the one-off lookup. Enter the URL, get the report:

Another option is Chrome’s built-in Lighthouse. Open Chrome DevTools → Lighthouse tab. Lets you pick mobile or desktop. Make sure you’re on the page you want to test before starting:
