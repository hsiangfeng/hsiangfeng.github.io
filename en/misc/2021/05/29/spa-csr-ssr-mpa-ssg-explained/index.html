Intro

Quick post on what these terms actually mean. They show up everywhere — and if you can talk through the differences in an interview, that's a real win.

SSR (Server Side Render) and MPA (Multi-Page Application)

Let's start with SSR. SSR is Server Side Render — what's that mean?

Say you open a browser — Chrome, Firefox, Edge, the slowly-disappearing Internet Explorer (or maybe even reading this it's already extinct), or Safari (which some argue is the next IE — fight me). You type a URL or click a bookmark. The browser sends a Request over the network to the server saying "I want this page". The server responds with the page content. The browser then takes that content and renders it for you.

A friendlier framing — let's role-play this flow:

  • User — me
  • Browser — a blank piece of paper
  • Network — the postman
  • Server — a painter

The scenario (simplified for clarity):

I (the user) want to look at a picture of Godzilla. But the painter is dirt poor — every time I want a new picture I have to provide a blank piece of paper and a letter (the Request) explaining what I want. For example:

txt
Hi painter, please paint me a picture of Godzilla.

The handsome/lovely postman (the network) delivers the letter to the painter (the server). The painter reads it, prepares their brushes, paints Godzilla onto the paper I sent, and the postman delivers the finished painting back to me.

That's traditional web dev. It's called MPA (Multi-Page Application) — multi-page apps. Each page is its own HTML file. So in the analogy above, every time I want a new view I have to send the painter a new blank sheet. Page switching has a momentary blank screen while the painter does their work — which means users wait. In today's pace, if pages take more than a second or two, users bounce. For an e-commerce site that's brutal. Performance is also worse — the server does all the work, page by page, plus each page might have its own CSS/JS to download.

If you've done backend dev, you've probably seen Express + EJS, or a templating engine like pug:

pug
extends layout

block content
  .container
    h1= title

After the backend processes it, it becomes a real HTML document:

html
<!DOCTYPE html>
<html>
  <head>
    <title>Just the home page</title>
  </head>
  <body>
    <div class="container">
      <h1>Just the home page</h1>
    </div>
  </body>
</html>

Browsers only understand HTML, CSS, and JavaScript — so the backend processes everything, then hands the rendered HTML (plus CSS and JS) back to the browser. With SSR, the server handles both data and view rendering.

CSR (Client Side Render) and SPA (Single-page Application)

Now CSR. If you've used Vue, React, or Angular, by default you're doing CSR. What's CSR?

CSR is definitely not "Corporate Social Responsibility" (that's what Google might tell you first). It's Client Side Render. The Chinese term translates as "client-side rendering" — easier to think of it as "browser rendering" (technically incorrect but easier to grasp at first).

While SSR has the server rendering the page for the user, CSR flips it — the user's browser handles rendering, and the server doesn't touch the view layer.

So what does the server do then? It can be much simpler — just business logic and data processing. The browser handles the view. Servers typically just return JSON. Example:

(Source: https://github.com/hexschool/w3hexschool-API)

json
{
  "keyID": 0,
  "blogList": [ // article list
    {
      "title": "PixiJS V5 Tutorial (0)", // article title
      "url": "https://israynotarray.com/javascript/20200203/3949702627/" // article URL
    }
  ],
  "updateTime": "2020/2/18 7:49:36 PM", // form update time
  "blogUrl": "https://israynotarray.com/", // blog URL
  "name": "Ray" // display name if opted in
}

Compared to SSR, the server isn't dealing with rendering — just data.

Same role-play with CSR:

  • User — me
  • Browser — a blank piece of paper
  • Network — the postman
  • Server — a painter

Scenario (simplified):

I (the user) want to look at a picture of Godzilla. But this time the painter's modernized — they're using a fancy e-pad instead of paper. I still write a request:

txt
Hi painter, please paint me a picture of Godzilla.

The postman delivers the letter. But this time the painter doesn't paint — they write back:

txt
Tap this spot on the e-pad and you'll see it.

I follow the instructions, tap, and the picture appears.

See what happened? The painter cut out the painting step. The work happens locally on the user's e-pad — better UX, faster, no waiting on the painter.

CSR usually means SPA (Single-page Application). The whole app lives on one HTML page — like the e-pad. We don't ask for new paper every time we want a new view. CSR generally outperforms SSR because the rendering work is on the user's machine, not the server. Imagine SSR with thousands of concurrent users — every page render hammers the server. CSR pushes that work to each user's browser, so the server load is much lower, and modern personal computers are plenty powerful to handle it. The browser uses AJAX (Asynchronous JavaScript and XML) to talk to the backend for data — but that's a different post.

So once the view is the browser's job, the server can focus on data, and performance scales much better. The trade-off — first page load is sometimes slower, because the JS bundle has to download before anything renders. Once that's done, view switches are fast. And since CSR is usually SPA, you stay on the same page (typically index.html) the whole time. JavaScript-driven router simulates page changes — you'll often see # in URLs:

txt
https://israynotarray.com/w3cHexSchool-Search/#/ -> https://israynotarray.com/w3cHexSchool-Search/#/user

Same index.html under the hood, just JS swapping views.

CSR's downside — SEO. Since the view renders via JS on the browser side rather than being delivered fully-rendered by the server, search engine crawlers see "blank shell + JS" first. This is less of a problem now — Google has explicitly added support for crawling SPA-shaped pages. But traditionally CSR was SEO-hostile.

Identifying SSR vs. CSR

CSR is easier to spot. Right-click → View Source.

View Source (Firefox)

If the visible page is rich:

hexschool ironman search

But "View Source" shows something almost empty:

CSR source

Then the site is using CSR.

CSR + SPA sites usually also include these telltale tags:

html
<noscript><strong>We're sorry but w3hexschool-search-ts doesn't work properly without JavaScript enabled. Please enable it to continue.</strong></noscript>

And just a single mount root:

html
<div id="app"></div>

But if you switch to the DevTools Elements panel and look at <div id="app"></div>, you'll see a fully-rendered tree:

DevTools shows rendered content

That's CSR.

Inverse — if View Source matches what DevTools shows, it's "basically" SSR. But this is just a heuristic, not a guarantee. If you View Source on my blog right now and see the same structure as the rendered page, that's not SSR — my blog is a static site (more on that next). So the View Source method works well for spotting CSR, but is less precise for confirming SSR.

Picking CSR vs. SSR — both have pros and cons. CSR looks better on paper, and shifts the perf cost to users, leading to the joke "if you want to save the company money, pick CSR" (only half kidding). In practice it depends on the project. Some sites prioritize UX, others prioritize SEO.

You might ask — "can't you have CSR with great SEO?" Yes, doable, there are techniques. We'll skip that for now. SSR with great perf? Same — money fixes most things. (Half kidding.)

SSG (Static Site Generation)

After SSR and CSR, you've maybe also heard of SSG. What's SSG?

Two interpretations:

  • Static Site Generation — pre-build static pages
  • Server Side Generation — server-side generation

Different names, same output — a static HTML page.

Why SSG? Because CSR has an SEO problem. Even though Google's caught up on indexing SPA pages, SSR still trends better for SEO since the HTML arrives fully rendered.

Do we abandon SPA for SEO? No — SPA exists for a reason (UX). So frameworks emerged that solve both — Nuxt.js (Vue), Next.js (React).

The SSG concept turns this CSR shell:

html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta http-equiv="X-UA-Compatible" content="IE=edge">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Document</title>
</head>
<body>
  <div id="app"></div>
</body>
</html>

Into something more SSR-shaped, where the content is really written into the #app markup at build time:

html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta http-equiv="X-UA-Compatible" content="IE=edge">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>Document</title>
</head>
<body>
  <div id="app">
    <header>Ray</header>
    <ul>
      <li>Some line</li>
      <li>Some line</li>
      <li>Some line</li>
      <li>Some line</li>
    </ul>
    <footer>Footer</footer>
  </div>
</body>
</html>

How it works — at build time, the framework (Nuxt, Next, etc.) calls your AJAX endpoints once, pulls initial data, and writes it into the HTML output. So when a user "View Source"s a SSG-built page, they see a fully rendered tree, not just #app.

Which is why the "View Source = SSR vs CSR" trick is just a heuristic — you can't be 100% sure without other context.
