Can I Use? A Five-Minute Guide to Checking CSS / HTML / JS Browser Support

#CSS #JavaScript #HTML #CSS3 #Can I Use

Intro

As a frontend dev, you’ll use CSS / HTML / JS features all the time — and sometimes you’ll reach for a new one without realizing it’s not supported everywhere. Those gaps turn into bugs.

Can I Use

“Can I use… Support tables for HTML5, CSS3, etc“ is a browser support lookup. Historically people use it for CSS3 lookups, but it covers HTML and JS features too.

Open the site and use the search box on the home page:

The support legend lives at the bottom of the home page:

  • Green = Supported
  • Red = Not supported
  • Greenish yellow = Partial support
  • Gray = Support unknown

The Settings button next to the search bar lets you customize — feed in regional usage data to weight by where your users actually are, hook up Google Analytics, etc.

There are more options for which browsers to show — explore on your own.

Using it

We’ve got autoprefixer tools (PostCSS) that take care of vendor prefixes for us — but sometimes you’re writing raw CSS without a build pipeline, and you need to know what’s supported. Example — search “flex“ to see browser support for Flexbox:

Lots of cells, color-coded per the legend:

  • Green = Supported
  • Red = Not supported
  • Greenish yellow = Partial support
  • Gray = Support unknown

A “-“ badge on a cell means that version requires a vendor prefix. For example, Chrome 4-20 required -webkit-:

So box-sizing: border-box; would need to be written as -webkit-box-sizing: border-box; for those older Chrome versions.

You can also use Can I Use to check ES6 syntax support:

Can I Use uses its own data plus MDN’s data underneath.

CSS vendor prefixes

Common browser prefixes:

  • Chrome — -webkit-
  • Safari — -webkit-
  • Firefox — -moz-
  • Internet Explorer — -ms-

Internet Explorer 11 was retired in 2022, but you’ll occasionally still bump into prefix considerations, so it’s worth knowing what they are.

What’s a vendor prefix? When a new CSS feature lands in spec, browser vendors implement it under their own prefix to ship support without committing to the not-yet-finalized spec. Newer features go through a draft / experimental phase, and the prefixed form is the vendor saying “this is our implementation while the spec is still being decided”.

Adding a vendor prefix is essentially saying “yes I want the experimental version”.

Once the spec stabilizes, newer browser versions usually drop the prefix and accept the unprefixed form. If your user is on an older browser, you might still need the prefix.

You might wonder if older users see broken layouts. Mostly not — by the time a feature ships in browsers, the spec is usually pretty stable. The unprefixed standardization mostly affects fine-grained behavior, not “does it work at all”.
