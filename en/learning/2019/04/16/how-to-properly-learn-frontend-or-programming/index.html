How to Properly Learn Frontend Tech or Programming Languages

#Learning #Misc

Intro

Why am I writing this? Since switching to frontend engineering and joining HexSchool, friends, online friends, and classmates have kept asking me how to learn frontend like I did. I thought about it for a while before writing this experience-sharing post.

Anyone can be an engineer

Nobody’s unfit to be an engineer — it just comes down to whether you’re willing to spend time.

From my own experience: high school I was in Data Processing, college I studied Information Management — but I never really learned programming. School just taught me Office and how to take certification exams.

So I didn’t come from a coding background. I first encountered programming in junior year of college, when I needed to prep for my graduation project. I met a teacher who mainly taught PHP+MySQL basics, and I figured “I’ve always been pretty interested in web stuff — let me give programming a try!” That’s how my engineering journey started.

When I first started learning, I kept hitting walls and couldn’t understand why programming was so hard. I stopped coding for six months to a year, and almost gave up multiple times.

If we’re being technical, the earliest “programming” I touched was writing in-game dialogue scripts and game hacks — but those were just text files following someone else’s rules to fill in data. Not real programming.

So am I great at programming? No, I’m not. I’m still a career-switcher.

What pushed you to choose this path

Choosing this path always has a reason.

I mentioned I deliberately chose programming as my graduation project. The reason I picked web was that I thought web was cool and flashy. Wanted to try.

But starting was hard. My main language was backend (PHP+MySQL). Even though I understood CRUD, relational tables, and even server setup, I didn’t know what doing all that actually contributed to society.

So I spent 3–4 years trying various languages, hunting for what I actually liked. C, C++, C#, Python, Java, Swift — I learned all of them at least a bit, but I didn’t know what to do with any of them. Until September 2018, when my company asked me to handle frontend dev. I signed up at HexSchool to learn frontend, and after the course I realized my actual original motivation was frontend, not backend. So before learning programming, figure out what actually moved you to start.

Mindset before learning to program

Learning new tech is going to be full of setbacks.

Many people learn programming because they want to earn more money, but they often overlook the three key elements:

  1. Interest
  2. Passion
  3. Perseverance

All three matter. Missing any of them and your learning regresses or you start doubting whether you’re cut out for it. Even big paychecks won’t carry you through.

Interest and passion don’t directly equal reality, so if you rely only on them to learn new tech, you’ll max out your frustration skill fast — and might even quit programming for good. That’s when you have to lean on perseverance. If you can’t keep going, look back at why you got into this field in the first place. A lot of success comes from persistence.

How to start learning

We’re in an era of learning overload — try to find the method that fits you.

A lot of people ask me how to learn frontend, or want to copy my style. I have to say: sorry, my method may not fit you. I’d touched programming languages before, so some logic clicked easier for me.

Everyone should try to find the learning style that fits them. Early on you can mimic the way you’d mimic cooking, but later if you still don’t actively look for your own way, you’ll get phased out.

For example, when watching HexSchool’s online courses, I took notes (in Evernote) for every section. For just HTML & CSS, I had 56 notebooks, and I’d rewatch each section 2–3 times. The moment one concept wasn’t crystal clear, I’d rewatch. My daily learning time was 8–12 hours. I’m prone to forgetting, so I chose to repeat-view and write down a lot. That way the knowledge gets internalized like riding a bike.

If you’re forgetful like me, take a lot of notes. Notes clarify thinking and deepen impression.

I can’t remember the syntax

Learning a programming language is like learning a new natural language.

When learning a language, don’t memorize blindly — understand the meaning and purpose. Why do you know “Apple” means a fruit? Because you understand what it means, so when you hear “Apple”, the image of an apple shows up immediately.

Another example: CSS has a property called float. Literally, it means floating. If you don’t try to visualize or understand it, you’ll keep forgetting to clearfix (clear the float). My understanding of float: “float lifts this element up, so what’s below loses height and can’t be pushed left or right — to avoid this, you have to clear the float.”

How do I produce portfolio-level work that matches the interview standard?

Your work’s quality is set by you.

A lot of people ask this privately. The standard is set by you, but everyone’s bar is different. You’ll ask: how do I produce industry-standard work? My answer: send out resumes! After a few face-washings you’ll know whether your work clears the bar.

Also ask yourself if you’re satisfied with what you built. If yes — have you thought about how to push it further?

In HexSchool’s “JavaScript Apprentice’s Trial”, there’s a small assignment that’s a GitHub API project. Just displays results like this.

I wasn’t satisfied, so I used other tech and UI design to make it more professional and aesthetic, and kept updating with new features — added localStorage.

I even challenged myself to build the GitLab version.

Should I practice slicing layouts to the point I can write them at a glance?

No matter the layout, basic prep work has to happen.

I’ve been a frontend engineer for about half a year. I can hand-write CSS & HTML, RWD, semantic tags — but I still can’t write any layout at a glance. What I do: analyze the layout first, record class names, then build using the recorded names. Saves time during dev so I’m not stuck naming classes mid-build.

(Mysterious voice: handwriting is ugly)

How do I choose a JS framework?

Anything that solves the problem is a good framework.

Image source: link

Everyone recommends a different JS framework. Ask an AngularJS person and they’ll say Angular is great. Ask 100 people, get 100 answers. If you ask me, I’ll say Vue, because HexSchool teaches Vue. But honestly, the big three aren’t that different in concept — pick one you like and grind it to mastery. Even if you jump to another framework later, you’ll pick it up fast.

Same logic applies when learning any programming language.

What’s an engineer’s value?

If you can solve problems, you can call yourself an engineer.

Many kinds of engineers exist — MIS, hardware, app — but industry is always short on engineers. Don’t worry that becoming an engineer will tank your worth. The industry is always short on engineers who can solve problems.

One of my past jobs was special: I was super green, technically titled full-stack engineer, but I’d never really written code. I leveraged online resources to solve various problems. The boss was very satisfied and considered me skilled. So what companies often need isn’t a coding god — they need someone who solves problems. If you can solve problems well, you’re a good engineer.

Closing thoughts

Finally, if you’re stuck on whether to walk the engineer path, just ask: if this is the career you genuinely want, do you want to be old one day and regret never having tried?

If, like me, you’re interested in frontend but don’t have a good teacher, I’d recommend HexSchool. I learned a lot here, even built a large e-commerce in Vue.js.

The Vue.js final project’s design was on me to brainstorm, but the underlying code was guided step by step by the teachers.
