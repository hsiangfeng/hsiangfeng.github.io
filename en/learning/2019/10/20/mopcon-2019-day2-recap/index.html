MOPCON 2019 — One Day Recap

#Misc #Learning

Intro

Day 1 of MOPCON 2019 I couldn’t make it, so I only went to Day 2. This post is a quick recap of that one day.

My planned schedule

Here’s what I had on the schedule the night before:

  • 3F - 09:15 ~ 10:00 - Challenges and solutions of machine learning in quantitative trading - R1
  • 4F - 10:15 ~ 10:55 - From backend to architecture - R3
  • 3F - 10:30 ~ 11:00 - Agile kickoff! 3 modes of agile transformation - UnConf R3
  • 4F - 11:10 ~ 11:50 - Web Vue.js Patterns - The evolution of components - R3
  • 3F - 13:00 ~ 13:30 - From consumer to developer 2019 — KKBOX’s open API journey, year two - UnConf R3
  • 3F - 13:30 ~ 14:00 - “Pure Komplaint Engineer” architecture talk - UnConf R3
  • 4F - 13:55 ~ 14:35 - Efficient record-and-playback web automation testing techniques and trends — SideeX as an example - R3
  • 4F - 14:50 ~ 15:30 - Vue.js 3.0 and the Composition API changes - R2
  • 3F - 16:00 ~ 17:00 - Career forum — keep engineering, or move to management? - R1

Truly a technical lineup

Machine learning in quantitative trading

The first session was a pretty solid one for anyone who codes and wants to invest. Like the title says, the talk was about using machine learning to predict stock fluctuations. Why do this? Basically, if you can train ML to predict stocks well, you can create passive income, and that gives you more time. There was one slide that I thought was really nice ↓

You can see a blue line and a red line in this chart. Each represents a person. The blue one uses investing well and you can see the curve climbs steadily upward, while the red one drops to the bottom after age 60.

That said, current ML stock prediction still needs a lot of tuning, because ML is — as the name says — feeding data to a computer to learn and predict from. A few rounds of prediction won’t get you accurate results. If you want to use ML for stocks, you have to commit to letting the computer learn continuously over a long period.

Agile kickoff! 3 modes of agile transformation

I was curious about agile, and although there was also an “Agile forum — agile across generations” session, I was worried it’d clash with the other talks I planned on. So I went with this UnConf-style one for a lighter intro.

Basically, agile transformation has three modes:

  • Boss-driven (top-down)
  • Employee-driven (bottom-up)
  • Both sides collaborating (top-down + bottom-up)

After listening through, the third mode is the best — both sides communicate, and you bring in a third-party expert to help guide the rollout (nice and concise?!). I didn’t feel particularly strongly about this one, so I’ll leave it there.

Speaker slides

Web Vue.js Patterns - The evolution of components

As a Vue.js engineer, how could I not catch this one? Right at the opening, the speaker said something that stuck — “I can give you 101 ways to not use Vuex.” After hearing them out, I kind of agreed. The way we usually write Vuex is to dump code into it indiscriminately — like throwing all your clothes into a closet, then having an avalanche when you open it. Not that Vuex is bad — it’s just that there’s no need to throw non-essential code in. That just leads to maintenance headaches and a shorter lifespan for your mouse wheel.

One particularly interesting bit was Vue component-to-component communication. The way we usually think about components is like a tree ↓

The speaker pointed out that as Vue components grow, the tree gets deeper, which leads to maintenance problems. Just finding a prop becomes painful. So the recommendation is to use Provide / inject for Vue component communication (if I remember correctly). I didn’t follow this part super closely (kind of zoned out for a moment), so I looked up some Provide / inject articles afterward.

From consumer to developer 2019 — KKBOX’s open API journey, year two

This one was given by ericpi from KKBOX, walking through the pain of pushing KKBOX toward an Open Data API and the rollout difficulties. I think this is actually a pretty good API source for practice — one of the slides showed an example project: a Pomodoro timer wired up to the KKBOX API.

If you’re interested in backend or just want to know how the “Pure Komplaint Engineer” platform is built, this one gives you a solid baseline of backend ideas. For example, I’d previously only touched Repository and Service (and not very deeply, QQ) — but the speaker also covered Presenter, which mainly handles UI-related logic. Efficient record-and-playback web automation testing techniques and trends

Wandered off to chat with the boss and other dev gods. Missed this one entirely.

Vue.js 3.0 and the Composition API changes

If you follow Vue, you probably know Vue 3.0 Pre Alpha dropped on 2019/10/05. This session was a quick intro to what Vue 3.0 brings to the table. Performance alone is a pretty big jump…

What are the main highlights of Vue 3.0? Quoting from kuro-san’s slides:

  • Goal: smaller and faster
  • Stronger TypeScript support
  • More API consistency
  • Open the underlying framework for extensions
  • Change detection: Object.defineProperty → Proxy API (IE doesn’t support it, so say goodbye to old browsers)
  • Virtual DOM rewrite
  • Better compile-time optimization

The rest you can probably wait for the official release to dig into. But from my recollection of reading the official docs, Vue 3.0 recommends writing in TypeScript, so I’m guessing TypeScript becomes a must-have skill.

(Just… I’m still not used to setup() QQ)

HackMD

Career forum — keep engineering, or move to management?

The last session was the MOPCON career forum.

The three main speakers were:

  • Liao Wei-Jie — CEO of HexSchool
  • Howard — Founder of Good Ideas Studio
  • Gipi (Yu Shu-Fan) — Business mindset evangelist at Business Thinking Institute

I got a lot out of this one. A few moments really stuck — one was don’t box yourself in. Don’t lock yourself into “I can only be an engineer” — maybe switching to selling braised duck heads is also a great option.

Another moment hit close to home. Early in my career I was really focused on how others perceived me, but no matter what I did I couldn’t meet their idea of “good”, and that caused a lot of mental discomfort to deal with.

Here’s the Day 2 livestream timestamp for reference: MOPCON 2019 Day2 forum livestream.

To wrap up — honestly, more than half of my time at the event was spent in Loading… state. If you ask whether it was worth it, there’s no single answer — everyone gets something different out of it. But I did get to see a lot of dev gods in person.
