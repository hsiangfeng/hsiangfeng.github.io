You Can Understand APIs by Buying Lottery Tickets — A Real-Life Walkthrough of RESTful APIs

Intro

Let's start with a simple lottery example and step-by-step unpack the mystery of APIs, REST, and RESTful APIs.

APIs

You hear stuff like this all the time:

"Does this have an API I can use?"

"Does this have APIs available?"

"Do you know how to integrate with APIs?"

As an engineer, you've probably heard the term "APIs" a lot. So what exactly is an API?

API stands for Application Programming Interface. Most sources describe it as a "bridge" or "interface" between two applications. I think that definition is hard for a beginner to grok, so I'd say toss those definitions out for a moment.

"Toss them out? OK, then how would you explain it?"

I know you'd say that, so don't worry — I'll use a real-life example to explain APIs, so it'll be easier to understand.

APIs are actually a kind of "transaction"

Picture this in your head...

You're going to a lottery shop to buy a ticket. You walk in and say "Boss, I'll take NT$100 of Lotto!" You hand over NT$100. The boss confirms receipt, types your NT$100 order into the lottery terminal, the terminal prints a Lotto ticket with two sets of numbers, and the boss hands it to you.

Note Typically one set of Lotto numbers (6 numbers) costs NT$50, and our example buys two sets, which adds up to NT$100.

In that single transaction, we've already explained the entire API concept and behavior. Let's map API onto buying a lottery ticket:

  • You: User
  • Boss: Server
  • NT$100: Request, usually with auth data attached
  • Lottery terminal: The bank's prewritten APIs encapsulated in the terminal — the server can call corresponding functions, often with auth data, parameters, etc.
  • Lotto ticket: The Response data

Now applied to our example:

  • You walk into the lottery shop: User opens a browser, types a URL, lands on a site
  • "Boss, I'll take NT$100 of Lotto!" and hand over NT$100: User sends a request to the site, with auth data attached
  • Boss confirms receipt of NT$100: Server validates whether the request is legitimate and authorized
  • Boss enters your NT$100 order into the terminal: Server calls the API. APIs are pre-encapsulated functions or programs — there are usually multiple
  • Terminal prints a Lotto ticket: API returns data to the server
  • Boss hands you the ticket: Server returns the data to the user

The analogy isn't perfect, but you should at least have an idea now.

The key part is "Boss enters your NT$100 order into the terminal". The lottery is sold by the bank, but we can't run to the bank every time we want a ticket — we go to the shop. The terminal is what bridges the shop and the bank. That's the API concept.

So a simple summary:

"An API is the bridge (lottery terminal) between application A (boss) and application B (bank), letting A talk to B via the API — that achieves data exchange."

What about applying this to web frontend / backend dev?

  • User-visible web UI (frontend): Application A
  • Server: The execution and processing center, handling backend program logic
  • APIs: Interfaces provided by the backend program, letting the frontend talk to the backend

So when a user interacts with the UI, the UI says to the server "Hey! I'm requesting data from /api/v1/user — can you return that?" The server calls /api/v1/user, then returns data to the user. The /api/v1/user route itself calls an encapsulated function or program. That's the API concept.

API types

API types are roughly:

  • Open APIs: provided to developers so they can integrate to build their own apps
  • Internal APIs: company-internal APIs, not exposed externally
  • Third-party APIs: APIs provided by a third party so developers can integrate to build apps

Note "Third-party" means non-official APIs, typically from third-party companies or individuals.

For OpenAI: most people use the ChatGPT WebUI to chat. But if you want to integrate ChatGPT APIs to build your own app, you use the APIs OpenAI provides. Those APIs belong to "open APIs".

What's the difference between open APIs and third-party APIs? Not much, but open APIs emphasize openness. For example, open APIs might pull data from government open data platforms — but integrating directly might involve some data-logic handling. Third-party APIs solve this by handling that for you.

A quick summary:

  • Open APIs: emphasize public access and ecosystem-building. Usually provided by the original service.
  • Third-party APIs: APIs that non-official entities or other companies built by wrapping or enhancing original APIs — for easier data integration.
  • Internal APIs: Just for the company. Internal use only, not public.

Side note: if you can use an open API directly, that's ideal — third-party APIs have stability and data-security concerns since they're provided by a third party.

RESTful

The direct conclusion first:

"RESTful APIs are a style for designing APIs, not a technology or standard. It's a convention."

We know what APIs are. But what the heck is RESTful? Googling gives you a "Representational State Transfer" translation, which is incredibly hard for a beginner to follow — same for me.

Before explaining RESTful, we need to understand REST first.

What is REST?

REST stands for REpresentational State Transfer — the translation is technically "Representational State Transfer". It's a design pattern proposed for managing distributed resources, formulated by Roy Fielding in his 2000 PhD dissertation.

If your API follows REST's design style, it can be called a RESTful API.

To follow REST design style, you should respect these principles:

Uniform Interface

Achieving a Uniform Interface usually requires hitting four constraints:

  • Resource identification: every data request should have a unique URL (e.g., /api/v1/user) — analogous to how frontend uses document.getElementById('id') to grab an element.
  • Resource manipulation: the server returns data in a unified format (e.g., JSON), so integrators don't have to learn new formats. Modern dev typically uses Content-Type: application/json.
  • Self-descriptive: the server's response data should include enough info for developers to understand it without consulting docs — HTTP status codes, error messages, etc. E.g., HTTP Code: 200 for success, HTTP Code: 400 for error, with a message like {"message": "error message"}.
  • HATEOAS (hypermedia as the engine of application state): basically when you fetch data, the server returns hyperlinks too. Upside: the frontend doesn't deal with URLs — the server-returned hyperlinks fetch the data, and only the backend maintains those links. The frontend stays focused on rendering. Common links: self (current), next, prev, first, last, etc.

Hitting all four constraints satisfies the Uniform Interface principle.

Note In my own dev experience, HATEOAS is most commonly seen for managing sidebar links — e.g., after login, the admin doesn't show all frontend page links directly; the backend validates and returns the appropriate frontend page links.

Stateless

RESTful APIs must be stateless. Every request is independent and not affected by the previous request. The upside: the server doesn't have to maintain user state — just handle the request.

If that's not clear, take user login. After login, the server returns a token to the user:

json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiYWRtaW4iOnRydWUsImlhdCI6MTUxNjIzOTAyMn0.KMUFsIDTnFmyG3nMiGM6H9FNFUROf3wh7SmqJp-QV30"
}

That token gets stored in the user's browser. On subsequent requests, the user attaches the token, and the server knows who they are — no need to maintain user state (the older Session concept).

Client-Server

Client-Server is about separating client and server, so they can evolve independently without affecting each other. In early web dev, we'd mix HTML, CSS, JavaScript, and backend code together — that violated Client-Server. In plain terms: frontend / backend separation.

Layered System

Layered System means splitting the whole app into independent layers — each layer does its own thing without meddling with others.

Classic example: MVC.

  • Model: handles data logic
  • View: handles UI logic
  • Controller: handles logic flow

In MVC, View doesn't talk to Model directly — it goes through Controller. That satisfies Layered System.

Layered System is like a team where each person does their own job and only coordinates with the neighbor, keeping flow smooth.

In app dev, we'd put token validation in a separate Service / Module from user logic — also a layering concept. Applied to API design, this means APIs are split into layers. For example, user login is split into:

  • POST /api/v1/auth/: handles auth logic, doesn't handle user logic
  • POST /api/v1/user/: handles user logic, doesn't handle token logic
  • POST /api/v1/token/: handles token logic, doesn't handle user logic

Further split into more granular layers like POST /api/v1/auth/signin, POST /api/v1/auth/signup, POST /api/v1/auth/signout — satisfying Layered System.

Cacheable

Cacheable means server response data can be cached, improving performance and reducing server load.

For API integration: you can cache data that doesn't change often (set Cache-Control). On subsequent requests, data comes from cache rather than hitting the server.

This is more of a backend concern, e.g.:

js
app.get('/api/v1/user', (req, res) => {
  // Cache for 1 hour
  res.setHeader('Cache-Control', 'max-age=3600')
  res.json({ message: 'Hello World' })
})

On the next request, the user can pull from cache instead of hitting the server, improving performance.

Code on Demand

This one's optional. The principle says the server can return code for the client to execute, improving client capability. Code on Demand isn't usually required — I haven't run into this requirement myself.

RESTful + APIs = RESTful APIs

Now we know what APIs are and what REST is. Combine them: an API that follows REST design style can be called a RESTful API. This post will explain how to design RESTful APIs from a web dev angle.

Let's say there's an e-commerce system with these features:

  • User login
  • User signup
  • User logout
  • Get product list
  • Get product details
  • Add to cart
  • Remove cart item
  • Checkout

Pause and think — how would you design these API paths (communication bridges)? Real example: at company A, the backend engineer designed them like this:

  • User login: POST /login
  • User signup: POST /register
  • User logout: POST /logout
  • Get product list: GET /products
  • Get product details: GET /products/:id
  • Add to cart: POST /cart
  • Remove cart item: POST /cart/:id
  • Checkout: POST /checkout
  • Create product (admin): POST /admin/products
  • Delete product (admin): POST /admin/products/:id

Looks OK. Then I left and went to company B, where the backend engineer designed them like this:

  • User login: POST /userSignIn
  • User signup: POST /userSignUp
  • User logout: POST /userSignOut
  • Get product list: GET /productList
  • Get product details: GET /productDetail
  • Add to cart: POST /addCart
  • Remove cart item: POST /deleteCart
  • Checkout: POST /checkout
  • Create product (admin): POST /adminCreateProduct
  • Delete product (admin): POST /adminDeleteProduct

Looks OK at first glance, but every company ends up with their own naming style. Problems:

  1. Hard to use modern (standardized) tools to generate docs — Swagger, Postman, etc. are built around RESTful APIs
  2. Action-centric naming tends toward RPC (Remote Procedure Call) style, blurring resources and complicating paths
  3. Doesn't follow HTTP Method semantics (POST = create, GET = read, PUT = update, DELETE = delete). Hard for devs to infer what an API does from its method
  4. Hard to intuitively know what an API does — naming and verb/noun conventions vary
  5. Versioning and path management are awkward. Adding or modifying features will shift paths, frustrating frontend devs
  6. Every new company means re-learning their APIs or reading docs to figure out what they do

Note RPC is another design style, typically action-centric naming with required parameters to call.

So how would you redesign these paths? Here's a RESTful API style redesign:

  • User login: POST /api/v1/auth/signin
  • User signup: POST /api/v1/auth/signup
  • User logout: POST /api/v1/auth/signout
  • Get product list: GET /api/v1/products
  • Get product details: GET /api/v1/products/:id
  • Add to cart: POST /api/v1/cart
  • Remove cart item: DELETE /api/v1/cart/:id
  • Checkout: POST /api/v1/checkout
  • Create product (admin): POST /api/v1/admin/products
  • Delete product (admin): DELETE /api/v1/admin/products/:id

Then for request and response, use JSON consistently (Content-Type: application/json). You might ask:

"Why JSON?"

Because JSON is a lightweight, human-readable data exchange format that most languages support, so frontend / backend devs can parse and adjust with minimal cost.

Since we're talking web dev, pair these paths with HTTP methods:

  • POST: create (or change resource state)
  • GET: read
  • PUT: update
  • PATCH: partial update
  • DELETE: delete

And use HTTP status codes for responses, e.g.:

  • HTTP Code: 200: success
  • HTTP Code: 400: error
  • HTTP Code: 401: unauthorized
  • HTTP Code: 404: resource not found
  • HTTP Code: 500: server error

So a RESTful-style API call looks like:

User login: POST /api/v1/auth/signin

Request:

js
const data = {
  email: 'example@example.com',
  password: '123456'
}

const options = {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify(data)
}

fetch('/api/v1/auth/signin', options)
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error(error))

Response (HTTP Code: 200):

json
{
  "message": "Login successful",
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiYWRtaW4iOnRydWUsImlhdCI6MTUxNjIzOTAyMn0.KMUFsIDTnFmyG3nMiGM6H9FNFUROf3wh7SmqJp-QV30"
}

Note The token above is called a JWT token. If you're not familiar with JWT, see About JWT Authentication (article in Mandarin).

Response (HTTP Code: 400):

json
{
  "message": "Login failed",
  "error": "Wrong account or password"
}

After all this, hopefully you have a clearer sense of why we recommend RESTful design when opening APIs ~

Closing thoughts

One last thing: the HTTP methods we use here aren't equivalent to REST, and REST isn't an absolute rule. It's just a convention. As long as your API follows REST design style, it's a RESTful API.

RESTful is mainstream but not the only answer. If you need complex queries or compact data transfer, GraphQL might fit better. Some enterprise-internal systems use traditional SOAP / RPC stacks.

Also, generally don't sweat API path length. The core idea of RESTful API design is resource representation, not path length. Don't sacrifice resource clarity to shrink the path — that just makes resources ambiguous and harder to follow.
