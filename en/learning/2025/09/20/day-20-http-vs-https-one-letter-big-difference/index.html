HTTP vs HTTPS — one extra S makes that much of a difference

As technology and the internet grew, I went to cafés less and less — I had a computer at home now. To stop me from running off to cafés, my family even signed us up for HiNet broadband so I could go online at home.

Note

HiNet (full name: Chunghwa Telecom Internet Service) is Taiwan's largest ISP, offering broadband, fiber, and various network services.

The game I was most invested in was Ragnarok Online. I played at least 10 years — from elementary all the way through entering the workforce, multiple comebacks in between.

Why mention it? Because my account got hacked countless times. And almost always after typing my password on the official site and topping up points.

"Argh! Hacked again!" I complained on IM.

"What? Hacked again? What did you do beforehand?" online friends asked.

"Logged into the official site, topped up — boom, hacked. WHY!" I'd answer.

Yeah, just constant "WHY". Because I didn't get it: it was the real official site, not phishing. My computer was just reinstalled, couldn't be infected. And if it were phishing, the points cards should've been stolen too — but they weren't.

It wasn't until later, in IT class, when the teacher demonstrated Wireshark and explained that HTTP is a completely insecure protocol.

Insecure protocol

Note

The site above is "Slime's First Home". Since this site doesn't use HTTPS, the browser shows a "not secure" warning.

Suddenly it clicked — back when I typed my password on the game's official site, since they used HTTP, my credentials were intercepted mid-transit…

HTTP vs HTTPS

HTTP stands for HyperText Transfer Protocol — a protocol / standard for transmitting data over networks.

When you open the browser and enter a URL, the browser uses HTTP to request data from the server, and the server returns data to the browser — that's how you see web content.

HTTP has versions too, but we only need to know HTTP/1.1, HTTP/2, and HTTP/3.

HTTP/2 and HTTP/3 are upgrades over HTTP/1.1 for better transmission efficiency and security. We mostly need HTTP/1.1's features — RESTful API is designed on HTTP/1.1. HTTP/1.1 defines 8 request methods:

  • GET
  • POST
  • PUT
  • DELETE
  • HEAD
  • OPTIONS
  • TRACE and CONNECT

These tell the server what operation to perform — GET for fetching data, POST for submitting, etc.

GET and HEAD are defined as "safe" requests — they don't modify or affect server resources. So you can confidently use GET / HEAD to fetch data without worrying about touching the server.

Why are these two "safe"? Web crawlers work by using GET requests to fetch page data. If GET weren't safe, crawlers could modify server resources while fetching — that'd be a huge security problem.

In contrast, POST, PUT, DELETE are defined as "unsafe" — they may modify or affect server resources, so use with care.

In practice, GET isn't 100% safe — some sites use GET to modify resources.

After understanding HTTP basics, here's another important point: HTTP transmission is "plaintext transmission" — data isn't encrypted in transit, so anyone intercepting packets can read content directly.

Note

Plaintext transmission means data isn't encrypted in transit — anyone can easily read it.

To prove HTTP is plaintext, I'll demonstrate intercepting HTTP packets with Wireshark and show how easy the content reads.

Note

First, download and install Wireshark from the official site.

The examples will require writing code. If that's not convenient, don't worry — I'll screenshot each step so you can read without setting up the environment. But following along is way better for learning.

First, download this GitHub project I prepared. It includes a simple login screen with these accounts:

  • Username: admin, password: password123
  • Username: user, password: secret456
  • Username: test, password: test789

After getting the project:

  1. In the project directory, run npm install.
  2. Run npm start to start the project.
  3. Open browser and visit http://localhost:3000 to see the demo login screen.

Now open Wireshark and pick the interface to monitor (Wi-Fi or Ethernet). For this example, we monitor local traffic — pick Loopback (usually shown as lo).

Wireshark Loopback interface

You'll see Wireshark immediately filling with messages — that's it monitoring traffic in real time.

Wireshark monitoring traffic

Now go back to the project page and log in. When you see the post-login screen, switch back to Wireshark to check the packets.

Type credentials

Wireshark is probably scrolling wildly. To avoid interference, click the "Stop" button at the top to pause monitoring.

Wireshark stop monitoring

Lots of packet info — filter with: tcp.port == 3000 and http.request.method == "POST"

Wireshark filter

Note

The filter means "show only TCP packets on port 3000 with POST method", screening out irrelevant packets.

You should see a POST request packet. Click it, and the "Packet Bytes" area at the bottom shows the detailed request info — including the credentials you just typed!

Wireshark packet contents

This is exactly HTTP plaintext transmission — anyone can easily read what's sent. That's why HTTP is considered insecure.

Note

In local development, HTTP is usually the default since security isn't a concern. But in production, HTTPS is required to ensure transmission security.

What about HTTPS?

In short, HTTPS = HTTP + SSL/TLS encryption — data is encrypted in transit. Even if someone intercepts packets, they can't easily read the content.

To enable SSL/TLS, the server needs an SSL certificate, issued by a trusted Certificate Authority (CA) to ensure secure connection.

In my example project, I have a shell script to quickly generate an SSL certificate:

Bash
#!/bin/bash

mkdir -p cert
openssl req -nodes -new -x509 -keyout cert/server.key -out cert/server.crt -days 365 -subj "/C=TW/ST=Taiwan/L=Taipei/O=Demo/OU=Dev/CN=localhost"

echo "Self-signed certificate generated at cert/server.key and cert/server.crt"

This creates a cert folder in the project with server.key and server.crt — the SSL certificate we need.

Note

OpenSSL is an open-source crypto tool for generating SSL certs and private keys. Note: production environments must request SSL certs from trusted CAs. Self-generated certs are for dev / testing only.

So run this command to generate the cert, then use it to start the HTTPS server:

Bash
sh generate-cert.sh
npm start

Then visit https://localhost:3443 — note https, not http.

You'll get a Chrome warning at the login screen because we're using a self-signed cert that the browser can't verify.

HTTPS warning

The rest is the same: type credentials, log in, switch to Wireshark to check packets. The filter changes to:

Text
tcp.port == 3443

Notice http.request.method == "POST" is gone — HTTPS encrypts in transit, so we can't see the request method directly.

How do we identify? Through "packet size" — POST is usually bigger than GET. In Wireshark, check the "Length" column.

E.g., here it shows 4.059109 seconds — likely the POST request from when you logged in.

But no matter how you look, you can't see the credentials — HTTPS encryption protects the data~

Wireshark HTTPS packet contents
