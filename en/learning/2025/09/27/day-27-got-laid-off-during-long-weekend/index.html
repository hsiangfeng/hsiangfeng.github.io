【Day 27】Got My Layoff Notice While Enjoying a Long Weekend

Got my layoff notice while enjoying a long weekend

After entering the workforce, was everything smooth?
No, actually it wasn’t.

That year, during the 228 long weekend (Taiwan’s Peace Memorial Day), as I was enjoying my time off, my phone suddenly rang.

“Ray, next month our team is being laid off. When you have time, come back to the office to clean up.”
My manager’s tone was unbelievably calm, like he was discussing a routine task.

After the call ended, I couldn’t help but laugh bitterly —
Turns out “layoff notice” is basically a free career-transition reminder.

Coworkers got the notice one by one, and the group chat exploded with curses. I just calmly replied:
“Yeah, what a pain — gotta start prepping resumes and interviews now, QQ”.

After the layoff, I put all my focus on frontend. Finished the JavaScript course, repeated it 3–4 times, then bit the bullet and dove into Vue.js.

First impression:
“Whoa, new terminology everywhere!”
Componentization, one-way data flow, lifecycle…
Each one sounded like an RPG skill tree branch, and I was back in the beginner village again.

Did it feel like magic? At that moment, absolutely — write a few lines and you produce flashy effects.

Like this simple Vue component:

In pure JavaScript, this might take dozens of lines, plus handling event binding and DOM manipulation. With Vue.js, it’s easy.

I prepped for about half a year, eventually building “RagnarokShop“, an e-commerce practice site in Vue.js.

After passing the course teacher’s Code Review, I started applying for frontend engineer jobs.

The position I applied for was “Junior Frontend Engineer” — without real frontend work experience, I had to start with Junior.

A few interview questions stuck with me deeply.

The interviewer asked:
“Is JavaScript an interpreted or compiled language?” And “What’s a Framework? What’s a Library?”

Honestly I didn’t know clearly at the time. Rather than making up answers, I responded honestly:

“Um… I’m not sure, I haven’t specifically researched this.”

“You didn’t use Frameworks in PHP?” you might ask.

True, I used Frameworks for most PHP work. But back then I’d never thought about the Framework vs Library distinction — I’d just copy from official docs as long as it ran.

The difference? Heh, I just thought: “as long as it runs, who cares what it’s called!”

Throughout the interview, I got asked plenty of questions, but only these few stumped me. The interviewer even explained them at the end — left a deep impression.

Interpreted vs compiled languages

In the programming world, there are two main types: Interpreted Language and Compiled Language.

Differences:

  • Interpreted Language: code doesn’t need to be compiled. An interpreter parses and executes each line. The JavaScript we covered earlier is interpreted.
  • Compiled Language: code is compiled first, converted to machine code, then executed. Machine code is what the computer directly understands — binary. C is compiled.

Compiled:

From flows, the pros and cons:

  • Interpreted:
    • Pros: fast dev — no compile step, code runs immediately.
    • Cons: slower execution — every run has to interpret the code.
  • Compiled:
    • Pros: faster execution — code is already machine code.
    • Cons: slower dev — needs compile step before execution.

So JavaScript and PHP are interpreted. C is compiled.

To solve these issues, modern languages aren’t strictly interpreted or compiled — they combine both.

For JavaScript, the V8 engine first compiles code into bytecode, then JIT (Just-In-Time) compiles bytecode to machine code, hugely speeding up execution.

This style is called “Hybrid Language”.

Compilation techniques can be split into:

  • AOT (Ahead-Of-Time) compilation: pre-compile interpreted code into machine code.
  • JIT (Just-In-Time) compilation: combines interpreted and compiled advantages, speeding up at runtime.

So whichever language you pick, the key is understanding its features and tradeoffs. Only then can you make better decisions in dev.

Every language has its use cases and strengths. Picking the right language makes dev much smoother~

What are Framework and Library?

In software dev, Framework and Library are concepts often mentioned.

A senior at one of my early jobs once said:

“If you want to write good code, I’d recommend using fewer Frameworks and more Libraries.”

At the time I didn’t really understand. Over time, through work and learning, I gradually figured out the difference.

What are they?

  • Library: a collection of pre-written code providing specific functionality or service. Devs call these functions directly without writing from scratch.
  • Framework: a more comprehensive solution, providing a structured dev environment and dictating the overall app architecture and flow.

Think of Library as a toolbox of various tools (functions) helping you accomplish specific tasks — nails, hammers — you decide how to use them. It doesn’t interfere with your workflow.

Framework is more like a full workshop with fixed processes and rules. You follow its rules. Framework not only provides tools but tells you how to use them and sets the overall structure.

Pros and cons:

  • Library pros:
    • High flexibility — pick and choose functions (tools).
    • Doesn’t interfere with workflow — dev your own way.
  • Library cons:
    • You manage code structure and flow yourself — can add dev complexity.
  • Framework pros:
    • Provides clear structure and conventions — boosts dev efficiency.
    • Built-in features reduce reinventing the wheel.
  • Framework cons:
    • Steep learning curve — takes time to learn the framework’s conventions and usage.
    • May limit dev freedom — must follow the framework’s design philosophy.

“So should I pick Framework or Library?” beginners often ask.

Both work — depends on project needs and personal style.

For an MVP or quickly validating an idea, Framework fits — completes prototypes faster.

For a large system needing high maintainability and extensibility, Library may fit better — more flexibility to design architecture.

Note
MVP = Minimum Viable Product — a product with basic features used to validate market demand and user feedback.

Of course, using a Framework doesn’t mean you can’t write good code, and Library isn’t necessarily better. The most important thing: understand each one’s traits and tradeoffs first.

Today’s frontend space has React, Vue, Angular — all built on JavaScript. Instead of agonizing over which to pick, build solid JavaScript fundamentals first.

Because — without solid JS, switching frameworks is just swapping wine in the same bottle.

Closing thoughts

This was my first layoff experience. Tough in the moment, but also gave me a chance to reassess my career direction.

So… should I actually thank the company for laying me off?!
