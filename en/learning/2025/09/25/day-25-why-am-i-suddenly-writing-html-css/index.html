【Day 25】Why Did I Suddenly Become the HTML and CSS Person?

#Code Review of My Youth - From the BIOS Beep to Software Engineering #Intro to HTML and CSS #Frontend Web Development #Responsive Web Design #Wretch Memories

Why did I suddenly become the HTML and CSS person?

In the first 2–3 years out of school, I worked on Web apps mostly in PHP. My understanding of HTML at that time was “as long as it displays” — I didn’t care about layout structure, semantics, etc.

I’d touched web dev early on, but HTML5 wasn’t yet popular, and Table-based layouts were still mainstream. Then again, maybe that was just because I only knew Table layouts XD.

I didn’t really learn HTML until joining a new company.

“Anyone here familiar with Web, HTML, CSS?” my manager asked the team (only 5 people) while writing PHP.

“Frontend is too hard — HTML and CSS are too complex.” he immediately complained after asking.

“Oh, Ray, you go write HTML and CSS.” he pointed at me.

Honestly, I got pushed into HTML and CSS for absolutely no reason.

My original weapon was PHP, and now I was suddenly tossed onto the battlefield with <div> and <span> — <h1> and <p> were still in the beginner village. (ಥ﹏ಥ)

“OK, I’ll try.” I didn’t refuse — not because I was hyped, but because I had two newbies under me.

In reality, I didn’t know it either. I just learned while pretending I knew, then pretended to be professional while teaching them. The three of us were a beginner-village party XD.

Looking back, when searching for HTML/CSS learning resources, the first thing that came up was W3Schools — solid old-school tutorial site. But after about 30 minutes I couldn’t take it, so I went looking for other online resources — TibaMe, Udemy, etc.

Finally I bought a Udemy course called “Build a Website with HTML and CSS“.

That was my first HTML/CSS course. That’s when I really got to know the HTML/CSS world. During learning, it reminded me of the old “Wretch (無名小站)”.

Wretch was a hugely popular Taiwanese blog platform back then. I wrote articles there too. It shut down later, but it remains an important memory.

Back then, to make your blog look pretty, everyone would customize their own page layout. The tech for that customization was HTML and CSS. So when I revisited HTML and CSS, my brain immediately pulled up memories of late-night blog-customization sessions on Wretch.

Some popular platforms at the time:

  • Wretch (wretch.cc): developed by NCTU students, later spun off into “Wretch Inc.”, then acquired by Yahoo!, becoming part of Yahoo! Kimo. Shut down on 2013/12/26, along with Yahoo! Kimo Blog.
  • yam Sky Blog (blog.yam.com): a major refuge for Wretch users — many migrated here, but it also shut down and erased all data on 2018/10/01.
  • Pixnet (pixnet.net): still running, probably Taiwan’s largest blog platform (I think?). Never used it personally.
  • Blogger (blogger.com): Google’s blog platform. Not as popular as Wretch or Pixnet in Taiwan, but has a decent international user base — even some Google engineers use it for tech articles.

These were the popular platforms back then. Now most have moved to Medium, WordPress, or self-hosted blogs. But these platforms contributed hugely to spreading internet culture and tech sharing.

Note
If you want to revisit old sites, check the Wayback Machine. Enter the site URL and it’ll fetch past versions. Not all sites are archived, so heads up.

HTML basics

HTML stands for HyperText Markup Language — a language describing the content and structure of a webpage.

Basic HTML example:

File extension is .html. You might also see .htm — that’s older. Modern dev uses .html.

<div>, <span>, <h1>, <p>, etc. are HTML tags. They define and describe the page’s structure and content. From the example, basic HTML structure:

  • <!DOCTYPE html>: tells the browser this is an HTML5 document — the standard declaration.
  • <html lang="en">: defines the root element and specifies language.
  • <head>: holds the document’s metadata — charset, viewport, title.
  • <meta charset="UTF-8">: specifies character encoding as UTF-8, the most common modern choice.
  • <meta name="viewport" content="width=device-width, initial-scale=1.0">: sets viewport width and zoom ratio — crucial for responsive design.
  • <title>: defines the document title — shown on the browser tab.
  • <body>: contains the document’s main content. Everything displayed on screen goes here.

Note
Responsive Design — a design method that auto-adjusts page layout and content based on different device screen sizes and resolutions for the best UX. Usually done via CSS Media Queries, so pages display well on desktop, tablet, and phone.

You might ask:

“What about </title>, </head>, </body>?”

HTML tags often have “opening tags” and “closing tags”. The opening tag starts an element; the closing tag ends it. Some tags don’t have closing tags — called “self-closing tags” — like <img>, <br>, etc. We’ll cover this next.

If you read HTML resources, you’ll see tons of tags — for text, images, links, tables, etc. Each has its purpose and attributes. Do you need to memorize all? Honestly no — what you really need is the few common ones: <div>, <span>, <p>, <a>, <img>, etc. These cover most use cases, and their attributes are relatively simple.

Two things in HTML learning that need extra attention:

  1. Semantics: HTML tags aren’t just for displaying content — they should express content meaning. E.g., use <h1> through <h6> to express heading importance, <p> for paragraphs. Semantics aren’t just for search engines — they also help developers understand structure and intent.
  2. Element traits: every tag has its own element trait. Not just acceptable attributes — also block elements vs inline elements. Block takes the full row width; inline doesn’t. Critical for layout and styling, especially responsive design.

We mentioned self-closing tags. What does that mean?

Self-closing tags don’t need an end tag. Why? Here’s an extra concept — Void elements vs Normal elements.

In HTML, beyond inline/block, there’s also void elements. Void elements don’t need end tags because they have no content (and don’t allow content). Common void elements:

Element Use Example
<area> clickable area in image map <area shape="rect" coords="34,44,270,350" href="link.html" alt="region">
<base> base URL for all relative URLs in page <base href="https://example.com/">
<br> line break <br>
<col> table column setting (use with <colgroup>) <col span="2" style="background:yellow">
<embed> embed external resource (video, PDF) <embed src="file.pdf" type="application/pdf" width="600" height="400">
<hr> horizontal rule <hr>
<img> insert image <img src="image.jpg" alt="description">
<input> form input field <input type="text" name="username" placeholder="enter name">
<link> external resource (usually CSS) <link rel="stylesheet" href="style.css">
<meta> page metadata (SEO, encoding) <meta charset="UTF-8">
<source> media source for <audio> or <video> <source src="video.mp4" type="video/mp4">
<track> subtitles for <video> or <audio> <track src="subtitles.vtt" kind="subtitles" srclang="en" label="English">
<wbr> suggested line break (Word Break Opportunity) <wbr >

You might also see <img src="image.jpg" alt="description" />. This is because XHTML (a kind of XML) requires all tags to be closed, so <img> becomes <img src="image.jpg" alt="description" />. In HTML5, the closing slash is optional. So you can write <img src="image.jpg" alt="description"> — and in HTML5 both forms are correct.

“What’s XHTML?”

I knew you’d ask. XHTML is an extension of HTML. Most dev still uses HTML, so I won’t elaborate.

CSS basics

CSS = Cascading Style Sheets. Used to describe HTML element style and layout.

Basic CSS example:

CSS can’t run alone — it needs HTML. We usually write CSS in the HTML’s <head> using a <style> tag, or use an external CSS file.

Early on, not many people cared about CSS — browser support was inconsistent, many browsers’ CSS support was poor. As people demanded better visuals, browser CSS support improved, and CSS became essential for web dev.

CSS learners often reach this conclusion:

“Easy to learn, hard to master.”

This is correct. CSS syntax is simple and intuitive. From the example, you can quickly grasp each line:

  • background-color: background color
  • font-family: font
  • color: text color
  • font-size: font size

What about body and h1? They refer to HTML’s <body> and <h1> tags. For body, this CSS means:

“Set the HTML’s <body> tag background color to #f0f0f0, font to Arial or sans-serif.”

Pretty straightforward, right? CSS is this simple — which is exactly why you might get sloppy and not dig deeper.

What makes CSS “hard to master”? I think mainly:

  • Selectors: CSS selectors are diverse — from basic element selectors to complex combinators, pseudo-classes, etc. Time-consuming to grasp.
  • Layout models: CSS has multiple layout models — Box, Flexbox, Grid, etc. Each has its strengths and use cases. Practice needed.
  • Responsive design: as mobile devices proliferate, RWD matters more. Need deep understanding of media queries, flexible layout, etc.
  • Browser compatibility: browsers support CSS differently. Need testing and tweaking for cross-browser display.
  • Performance optimization: how to reduce repaints/reflows, pick efficient selectors — needs deep understanding.

How to start learning CSS? My suggestions:

  • Basic syntax: master CSS basics and selectors first. Real dev uses just a few syntax and patterns.
  • Layout models: before responsive design, understand layout models — Box, Flexbox, Grid. Critical. Otherwise you’ll struggle with that 1px difference when matching designs.
  • Hands-on practice: input needs output. Try replicating templates or design mockups to deepen syntax familiarity. Slow at first is fine, but you should get faster over repeated practice.
  • Responsive layout: modern users almost all have phones. Early on responsive wasn’t as critical, but now it’s essential. Learn what to adjust at what sizes, starting with media queries.

Compatibility and performance? I think you don’t need to go deep right away — wait until you have some learning momentum and satisfaction. These problems come up after real dev.

One thing worth highlighting: “compatibility”. Browsers’ CSS support varies, so display might differ. What happens if a browser fully doesn’t support a CSS property?

Simply: the property is ignored without throwing an error.

Normally when we use non-existent syntax, we get errors, or the IDE warns about non-existent properties. CSS doesn’t have this.

It’s confusing, but also a benefit:
Your page doesn’t crash because of an unsupported CSS property. But you also can’t immediately tell whether the property is supported — you have to check each browser’s compatibility yourself.

Closing thoughts

Looking back, Wretch had a lot of great-sounding songs, but as Wretch disappeared, many of those songs became unfindable, QQ.
