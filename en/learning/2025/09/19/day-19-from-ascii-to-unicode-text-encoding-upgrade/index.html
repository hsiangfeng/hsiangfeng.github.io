From ASCII to Unicode — a comprehensive text encoding upgrade

One time at an internet café, I was looking up info — I opened a webpage and the text was complete garble, looking like…

I forget what I was searching for, only that I Googled something, clicked a site, and the whole page was garbled.

“Hey, why’s this site like this?” I asked a friend at the café.

He smiled. “Easy. Open your browser settings and change the character encoding to UTF-8.”

Note
Browsers used to let you manually set encoding, but most browsers now auto-detect, so garbled text is rare now.

“Oh, it’s normal now! Why?” I asked.

“Dunno~ just change to UTF-8 if you see garble.” he said.

I just thought “solved is solved” and didn’t think more. But the next day in IT class, the teacher happened to be covering “text encoding”.

Text encoding basics

Remember ASCII from the previous post? We said ASCII is a “text encoding” — mapping binary to letters, numbers, and symbols so they display on screen.

But early ASCII only supported 7 bits — 2^7 = 128 characters. This limit meant ASCII couldn’t cover other languages like Chinese, Japanese, Korean — it was designed for English from the start.

Note
Because early ASCII was English-only, it was later renamed “US-ASCII” — “American Standard Code for Information Interchange”.

To cover ASCII’s gap, “EASCII (Extended ASCII)” emerged. The key difference: EASCII expanded to 8 bits, giving 2^8 = 256 characters — more symbol support.

EASCII is built on ASCII, so it has:

  • Backwards-compatible: EASCII’s 0–127 characters are identical to ASCII, so all ASCII characters still work.
  • Extended character set: Adds 128–255 for more symbols — special European letters, math symbols, etc.

Important caveat: EASCII is NOT an international unified standard. Different systems and apps might use different EASCII encodings — basically “every-vendor-for-themselves”. This is exactly why “garbled text” problems happened.

Reference table:

Item ASCII EASCII (Extended ASCII)
Full name American Standard Code for Information Interchange Extended ASCII
Bits 7 bits → 128 combinations 8 bits → 256 combinations
Range 0 ~ 127 0 ~ 255
Standardized? Yes, international standard Each vendor extends differently → no “globally consistent standard”
High-bit range (128~255) Undefined Vendor-specific symbols, international chars, graphic chars
Main use Basic English, numbers, punctuation, control codes European language support, graphics, business needs

Why don’t we worry about encoding issues now?

Browsers auto-detect? Hardware upgraded?

Neither — the real reason is the emergence of “UNICODE” as a standard.

Let me bust a myth — many people (including past-me) think Unicode is a kind of character encoding.

ASCII and EASCII really are encodings. But Unicode is actually NOT an encoding — it’s a “character set”.

Think of it as an “international dictionary” cataloging all the world’s text and symbols, assigning each character a unique number — the “Code Point”. For example:

Character Code Point
A U+0041
B U+0042
C U+0043
你 U+4F60
我 U+6211
中 U+4E2D
😀 U+1F600
🐱 U+1F431

This design lets Unicode support all the world’s characters and symbols, effectively avoiding garbled text. The table shows Unicode code points use U+ prefix with hexadecimal digits.

How many bits does Unicode use?

21 bits — much bigger than ASCII or EASCII:

  • ASCII: 7 bits → 2^7 = 128 code points.
  • EASCII: 8 bits → 2^8 = 256 code points.
  • Unicode: 21 bits → 2^21 = 2,097,152 code points.

Enough for all world languages and symbols.

We said Unicode is a “character set” — so what’s its encoding? Unicode has three:

  1. UTF-8: variable-length encoding, uses 1–4 bytes per character, backwards-compatible with ASCII, suited for network transmission.
  2. UTF-16: 2 or 4 bytes per character, suited for memory and file storage.
  3. UTF-32: 4 bytes per character — simple but space-heavy.

We most often use UTF-8. UTF-8 is variable-length — what does that mean?

In short, byte usage per character depends on character complexity:

  • For ASCII chars (0–127), exactly U+0000 to U+007F, uses 1 byte.
  • For chars U+0080 to U+07FF, uses 2 bytes (European characters).
  • For chars U+0800 to U+FFFF, uses 3 bytes (Chinese / Japanese / Korean / math symbols).
  • For chars U+10000 to U+10FFFF, uses 4 bytes (emoji, special symbols, rare characters).

That’s why A is U+0041 (1 byte) and 😀 is U+1F600 (4 bytes).

Last, let’s discuss UTF-8’s common positional system — “hexadecimal”. In short, hex is an extension of decimal. Decimal uses 0–9; once you go past 9, you carry. Hex adds A–F to represent 10–15.

So hex symbols:

For UTF-8, just grasp these key points:

  • UTF-8 is variable-length encoding, varies with character complexity.
  • UTF-8 is backwards-compatible with ASCII — all ASCII characters work in UTF-8.
  • Hexadecimal is UTF-8’s display format — easier to represent character codes.
  • Unicode is a character set, covering all world text and symbols, avoiding garbled text.

That’s the Unicode intro. This encoding lets us freely use various characters and symbols online without worrying about garbles~
