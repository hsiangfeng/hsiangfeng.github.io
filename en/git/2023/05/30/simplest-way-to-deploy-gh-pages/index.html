The Simplest Way to Deploy Your Own gh-pages Site

#Git #GitHub #gh-pages #GitHub Pages #Deployment

Intro

gh-pages is a free GitHub service that lets you deploy a static site for showcase purposes. For frontend devs that’s super handy and useful. But there are a bunch of ways to deploy a gh-pages site, so this post walks through the simplest one.

What is gh-pages?

Quick intro first.

gh-pages stands for GitHub Pages. It’s a free service from GitHub (the code-hosting platform) for deploying static sites. You can use it for your blog, portfolio, or quick demos.

It’s also a special GitHub branch (if you know Git). If your repo has this branch, you can access your site at https://<username>.github.io/<repository> — where <username> is your GitHub handle and <repository> is your project name.

Note
One more thing: you might’ve seen URLs like https://<username>.github.io without the repo part. That’s because the repo name matches the username. For me that would mean https://hsiangfeng.github.io/hsiangfeng.github.io, but since the repo name is the same as my username, the repo name gets dropped — leaving https://hsiangfeng.github.io.

And one more note — gh-pages is free as long as you have a GitHub account, but there are a few things to watch:

  • Static sites only. No backend — no PHP, Node.js, Python, etc. Just HTML, CSS, JavaScript. There’s no server-side compute.
  • One gh-pages branch per repo. So if you need multiple sites, you need multiple repos.
  • Each repo’s capacity is limited — 1GB. If your site goes over that, you can’t deploy.
  • Bandwidth limit is 100GB per month.
  • gh-pages only allows 10 deploys per hour. If you roll your own GitHub Actions, that cap doesn’t apply (because you’re paying anyway).
  • Free SSL certs.
  • The default domain is https://<username>.github.io/<repository>. For a custom domain, you’ll need to buy one and configure DNS. See my earlier post How to Buy a Domain and Connect Your Blog for details.

One more thing: gh-pages is free but not for commercial or illegal use — your account can get suspended. That’s in GitHub’s docs, so I won’t dwell on it.

Now let’s get to deploying gh-pages.

Deploying gh-pages

So I mentioned gh-pages is dead simple to use — all you need is a GitHub account. But how do you actually do it? Can you deploy a simple site even with minimal programming knowledge?

Yes, and it’s really straightforward. You just need basic frontend and basic Git. Let’s get into it.

A few setup steps I’ll skip:

  • Install VSCode (or any editor)
  • Create a GitHub account

So what’s next? If it were a pure static site (no build step), you’d just create a gh-pages branch in your repo, push your code there, and GitHub auto-deploys. Note
GitHub auto-detects the gh-pages branch — if it exists, it deploys. The deploy process shows up in the repo’s Actions tab under “pages build and deployment”. Once it turns green, you’re deployed.

Quick aside on how gh-pages works under the hood: in package.json we added gh-pages -d src. That tells gh-pages to push the contents of the src folder to the gh-pages branch. The -d flag is short for directory — it specifies which folder to push. Without -d, it defaults to the root, but you don’t want to push everything (including node_modules) so we specify src.

How does gh-pages know which repo to push to? It reads .git‘s remote config — that’s why we didn’t have to specify a repo URL anywhere.

Note
If you’re using CRA or Vite, read the gh-pages docs — they have specific config requirements, otherwise the site might not work. Check the official docs.

Bonus

If you’re using Vite (or similar build tools), change "deploy": "gh-pages -d dist" to "deploy": "vite build && gh-pages -d dist" so it builds before deploying.
