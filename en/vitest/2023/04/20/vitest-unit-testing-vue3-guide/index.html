Testing

Intro

I've been wanting to write this testing post for a long time but never had the time. Now that I've got some breathing room, I'm finally filling in this gap on testing.

Why write tests?

I should probably start with "why write tests?" Think about your real-world projects — have you ever hit a situation where you changed feature A, and somehow feature B broke too? Or when features get more and more complex, how do you even test them?

That's why writing tests matters. Tests help guarantee that your code is stable and reliable — meaning, after you modify a feature, you can verify the feature still works without manually checking each one by hand.

You might be thinking "what's wrong with manual testing?" Nothing, really, but as a project gets bigger, manual verification gets really painful. You can't just grab someone every time you add or change a feature and have them re-test the whole project, right?

So testing becomes really important. On the frontend side, tests are usually split into two kinds: unit testing and end-to-end (E2E) testing. This post focuses on unit testing.

Vitest basics and syntax

Before writing tests, you need to learn some basics around Vitest's syntax — otherwise you won't know where to start.

One thing to mention up front: Vitest is built on top of Jest, so whenever you look up syntax, you can also search Jest's docs — the syntax is the same between them.

Note

Jest is a JavaScript testing framework that lets you write test code. It's one of the most common testing frameworks in the industry.

describe

First up is describe. describe is used to label what this test is about, mainly so you can identify what the test is for at a glance.

JavaScript
··describe('test name', () => {
  // test contents
})

describe is a grouping concept, so you typically put many tests inside it. You can also nest describe inside another describe for clearer organization.

test

Next is test. test is the core of unit testing — this is where you actually write what you expect to be true.

JavaScript
··test('test name', () => {
  // test contents
})

test and describe look similar, but describe just labels the test, while the actual testing behavior lives inside test.

There's a shorter alias called it that does the exact same thing as test:

JavaScript
··it('test name', () => {
  // test contents
})

expect

So once you've written a test, how do you actually check whether the result is correct? That's where expect comes in. For example, if I expect 1 + 1 to equal 2, I'd write (didn't we say no 1+1=2?):

JavaScript
expect(1 + 1).toBe(2)

From a code perspective, when you run expect(1 + 1), it returns an expect object with lots of methods on it — toBe, toEqual, toBeGreaterThan, toBeLessThan, etc. These methods check whether the result matches what you expected. toBe checks for direct equality.

Like:

JavaScript
const name = 'Ray';

expect(name).toBe('Ray');

As mentioned, expect returns a bunch of methods. I won't go through all of them here — if you're curious, check the Jest docs or Vitest docs.

Lifecycle hooks

Surprise — tests have their own lifecycle hooks too. These come up a lot in practice. The common ones are:

  • beforeAll: runs before all tests
  • beforeEach: runs before each test
  • afterAll: runs after all tests
  • afterEach: runs after each test

One thing to watch out for: where you put these affects when they run. For example:

JavaScript
··import { beforeEach } from 'vitest'

beforeEach(async () => {
  console.log('beforeEach')
})

vs.

JavaScript
········import { beforeEach } from 'vitest'

describe('test', () => {
  beforeEach(async () => {
    console.log('beforeEach')
  })
})

These two run at different times. The first one applies to the entire file's test lifecycle, so it runs before every test in the file. The second one applies to that describe block's lifecycle, so it runs before each test inside that describe.

If this sounds confusing, don't worry — just remember the difference exists, and when we start writing actual examples below, it'll click.

Test files

The last concept is how to create test files. There's a convention here — typically, you create a __tests__ folder at the root of your project.

tests

All your test files go inside __tests__, and the filenames also follow a convention:

Bash
{filename}.test.{ext}
{filename}.spec.{ext}

That's pretty much everything you need before writing tests. Time to actually write some.

Writing your first tests

Before writing the first test, create a test file inside __tests__ called first.test.js with:

JavaScript
········import { test, describe, expect } from 'vitest';

describe('test', () => {
  test('1 + 1 = 2', () => {
    expect(1 + 1).toBe(2);
  });
});

Then run npm run test:unit in your terminal. You'll see:

test:unit

You'll see Test Files 2 passed (2) and Tests 2 passed (2) — that's because Vite's default template includes a sample test file in src/components, which is why there are two. If that's distracting, you can delete the files under src/components (I'll do the same to keep things clean).

test:unit

So far we've just confirmed our test runs. Now it's time to actually write real tests.

Testing App.vue

First, let's set up App.vue:

HTML
········<script setup>
</script>

<template>
  <main>
    <h1>IsRayNotArray</h1>
  </main>
</template>

<style scoped>
</style>

Then create App.test.js inside __tests__ (I won't repeat where to put these from now on). What do we test?

When writing tests, you need a specific scenario. For example, we want to test that App.vue's heading is "IsRayNotArray", so we create a test file for App.vue like this:

JavaScript
··············import { test, describe, expect } from 'vitest';
import { mount } from '@vue/test-utils';

import App from '../src/App.vue';

describe('App.vue', () => {
  const wrapper = mount(App);

  test('heading is correctly IsRayNotArray', () => {
    const text = wrapper.text();
    expect(text).toBe('IsRayNotArray');
  });
});

Now run npm run test:unit in your terminal:

test:unit

Test passed. Let's go back and look at the new pieces:

  • import { mount } from '@vue/test-utils';
  • const wrapper = mount(App);

What are these? @vue/test-utils is the official Vue testing utility — it lets you test Vue components. mount you can think of as instantiating — it actually constructs your component so you can test it.

Once the component is mounted, the returned object has tons of useful methods. Common ones:

  • wrapper.text(): get the text inside the component
  • wrapper.html(): get the HTML inside the component
  • wrapper.find(): find an element inside the component
  • wrapper.findAll(): find all matching elements
  • wrapper.vm: get the component's data (like data)

There are way too many to list all of them — if you want more, check the Vue docs.

Now say you've got some markup like this:

HTML
··································<template>
  <main>
    <h1 class="text-base">IsRayNotArray</h1>
    <ul>
      <li>Product 1</li>
      <li>Product 2</li>
      <li>Product 3</li>
    </ul>
  </main>
</template>

How do you correctly target the IsRayNotArray text? Our scenario assumes that App.vue's page must contain "IsRayNotArray", so:

JavaScript
··············import { test, describe, expect } from 'vitest';
import { mount } from '@vue/test-utils';

import App from '../src/App.vue';

describe('App.vue', () => {
  const wrapper = mount(App);

  test('heading is correctly IsRayNotArray', () => {
    const text = wrapper.find('h1.text-base').text();
    expect(text).toBe('IsRayNotArray');
  });
});

And we successfully target the "IsRayNotArray" text.

Assume these are your products. How do you test that they get rendered correctly? This is a little trickier — we need to introduce mock. Mocking means we fake an API to test whether the component renders the data correctly.

Now open App.test.js. We'll use a new piece of syntax, vi. vi is an API Vitest provides with lots of methods. The main one we need is spyOn — spyOn lets you watch whether a method gets called, or whether a method runs correctly. You can also set a return value with spyOn, which is how we fake an API for the component.

describe('App.vue', () => {
  const getProducts = vi.spyOn(axios, 'get');

Heads up: when using spyOn, the spy must come before mount, otherwise it won't work. This is important.

What should we test? Our example assumes axios.get is called once the DOM is ready, so:

JavaScript
············describe('App.vue', () => {
  const getProducts = vi.spyOn(axios, 'get');
  // ...omitted

  test('products method is called once', () => {
    expect(getProducts).toHaveBeenCalledTimes(1);
  });
});

toHaveBeenCalledTimes(1) means "has this been called exactly once?". If yes, pass; if no, fail. This lets us verify that axios.get actually got called.

toHaveBeenCalledTimes

Next? Our code has const products = ref([]);, so we should test whether products got assigned values:

JavaScript
··test('products has data with length 3', () => {
  expect(wrapper.vm.products.length).toBe(3);
});

And finally, we should check the actual content of products:

JavaScript
··test('products data is correct', () => {
  expect(wrapper.vm.products[0].title).toBe("Fjallraven - Foldsack No. 1 Backpack, Fits 15 Laptops");
})

Quick summary on remote API testing:

  • The spyOn call must come before mount, otherwise the spy won't take effect
  • When writing tests, you don't actually call the remote API — you return predefined data
  • To check whether axios.get was called, use toHaveBeenCalledTimes(1)

Testing functions

You might be thinking, "people always say unit testing means testing a function — how do I actually test one?"

Before testing a function, you need to be clear on "what is a function?" A function is a callable piece of code that can take parameters and return a value:

JavaScript
··function add(a, b) {
  return a + b;
}

This function takes two parameters and returns their sum, so:

JavaScript
··test('add returns 3', () => {
  expect(add(1, 2)).toBe(3);
});

This is the everyone-knows-this answer, and the most common example you'll see in unit testing articles online. A function needs inputs and outputs to be testable — what people often call a "pure function". If you're not familiar with pure functions, check out Functional Programming in JavaScript: a casual intro (article in Mandarin) — I won't go deeper here.

This is also why people say "your code must be testable" — meaning your code should be pure functions to be testable.

Note

Be careful with spyOn. The first parameter must be an object, and the second must be a string matching one of that object's methods, otherwise it fails. Like this:

JavaScript
··const cart = {
  getApples: () => 13,
}

const spy = vi.spyOn(cart, 'getApples')

Mocking

Mocking comes up a lot when writing tests. Say Products.vue has a button that pops up an alert when clicked:

HTML
························<script setup>
import { onMounted } from 'vue';

const props = defineProps({
  products: {
    type: Array,
    required: true
  }
});

const showAlert = (str) => {
  window.alert(str);
}
</script>

<template>
  <ul>
    <li v-for="item in products">{{ item.title }}</li>
  </ul>
  <button class="btn btn-alert" @click="showAlert('hello')">Click me</button>
</template>

<style lang="scss" scoped>
</style>

Now open Product.test.js. We're going to test showAlert:

JavaScript
··········test('button click calls alert', () => {
  const mockAlert = vi.fn();
  window.alert = mockAlert;
  const button = wrapper.find('.btn-alert');
  button.trigger('click');
  expect(mockAlert).toBeCalledWith('hello');
});

Why do we declare vi.fn() and swap out window.alert? Because we don't actually want an alert to pop up — we just want to verify that showAlert got called and that it was passed the right argument.

If you don't write vi.fn();, you basically can't verify it. So you're going to need vi.fn() either way. You might be wondering — can I use spyOn instead?

JavaScript
········test('button click calls alert', () => {
  const alertSpy = vi.spyOn(window, 'alert');
  const button = wrapper.find('.btn-alert');
  button.trigger('click');
  expect(alertSpy).toBeCalledWith('hello');
});

The test will pass, but you'll get an Error: Not implemented: window.alert error. So the correct approach is still vi.fn().

Closing thoughts

This post ended up pretty long, mostly because unit testing looks simple but actually has a lot of details to watch out for. A lot of them require checking official docs, so I can't promise this is all 100% correct — but if it helps you get started writing tests, that's already worth it.

Everyone learning unit testing hits the same wall: not knowing how to write them. People often say to write tests first, then code, but on the frontend I think that's tough — a lot of the time you need a UI first before the code shows up.

The other thing is, when you don't know what to write, the trick is: tests are written based on your code. Or simpler — imagine yourself as the user, and you're testing the user's behavior.

Take TodoList as an example. A TodoList probably has these features:

  • Add a todo
  • Delete a todo
  • Edit a todo

When writing tests, you'd write test('add todo'), test('delete todo'), test('edit todo'), then call each corresponding function and use expect to check the result. That's a complete test. So tests are usually written based on scenarios — that way you have a clear sense of what you're actually testing.

By the way, I wrote a TodoList unit test if you want to reference it.

The sample test files for this post are also here.

Last thing — you might be wondering: how do I know which parts of my code I've covered? Vitest has built-in c8 (test coverage) support. It shows your coverage percentage and generates an HTML file you can open in the browser to see it visually.
