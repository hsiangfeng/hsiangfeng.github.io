Deploying to GitHub Pages

So how do you deploy Vite to GitHub Pages? It’s actually really simple. I’ll skip the part about creating the GitHub repo here.

There are two ways to deploy, and I’ll walk through both of them.

Option 1: Deploy with a shell script

The first thing to change is vite.config.js. Add a property called base. What goes in there? Your GitHub repo name. For example, mine is example-vite-react, so I write base: '/example-vite-react/'.

One thing to watch out for: don’t drop the trailing /, otherwise you’ll hit a 404. On top of that, I like to add a condition to base so that when I’m developing locally it doesn’t tack on /example-vite-react/. That way local dev doesn’t run into weird issues.

The Vite environment

Now let’s take a quick look at why we’d choose Vite in the first place.

The thing is, when we write JavaScript, the code eventually gets more and more complex. But back in the early days, browsers didn’t support ESM (ES6 Modules) yet, so you might’ve seen a bunch of different module standards like these:

  • AMD
    • Stands for Asynchronous Module Definition, mainly relies on requireJS
  • CMD
    • Stands for Common Module Definition, basically an improvement built on top of AMD
  • UMD
    • Stands for Universal Module Definition, mainly a way to solve sharing code between front and back end
  • CommonJS
    • Mostly used in Node.js, though Node.js started supporting ESM (ES6 Modules) from 13.x onward

None of these module standards are really what this chapter is about, so just having a rough idea is fine. If you’re curious, go ask Google.

What we care about is the standard ES6 settled on later — ESM (ES6 Modules) — which is also the core foundation of Vite. Originally in frontend development we mostly had to use bundlers like Webpack, Gulp, Rollup, or Parcel. Along the way we went through all kinds of changes in frontend tooling, and there’s no denying these tools hugely improved our development experience.

But as projects get more complex, even larger, you start running into stuff like:

  • The dev server takes ages to start up — for bigger projects, anywhere from a few seconds to a few minutes
  • HMR (Hot Module Replacement) gets slow

Right now most mainstream development still centers around Webpack, so Vite’s official site uses Webpack as an example and shows this diagram:

Looking at the diagram above, when Webpack runs it has to bundle and compile all the code first, then hand the compiled code to the devServer before the developer can actually start working. But what about Vite?

Vite hands the source code straight to the browser and completely skips the bundling/compiling step. So it starts up super fast, and it only requests a specific page from the devServer when you actually need it. The devServer does a bit of light processing and sends the result straight back to the browser. So overall it’s really fast, and you don’t have to compile the whole thing every time you edit or restart. On top of that, Vite’s devServer uses Koa.js.

In dev mode, Vite also uses something called esBuild, mainly to handle the JavaScript/TypeScript side of things. Why esBuild? In short: speed. The diagram below is from esBuild’s official site:

So why is esBuild so fast? It actually has to do with the language it’s written in. esBuild is built in Go, which makes compiling really fast.

After all that, you’ll probably notice that when you run npm run build, there’s still some bundling time. The official docs actually explain why: even though most browsers support ESM, for performance reasons it’s still recommended to bundle for production. If you’re interested, check out the official explanation.

At this point you might be thinking — didn’t you just say esBuild is super fast? So why does bundling seem a bit slow? The main reason is that bundling uses Rollup, not esBuild. The official site also explains this: esBuild isn’t stable enough yet for things like CSS handling, which is why they don’t use it for bundling. But they don’t rule out using esBuild for bundling in the future.

This post is mostly just a quick chat about the Vite environment and why we’d choose Vite. So now we know that in dev mode Vite mainly relies on ESM (ES6 Modules) + Koa.js + esBuild, while for bundling/compiling it uses Rollup.

And that’s pretty much it for this chapter. See you tomorrow.
