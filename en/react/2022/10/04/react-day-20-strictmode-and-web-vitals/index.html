Intro

Picking up from the last post's CRA project, today we'll go through the remaining 20% of the generated code we haven't really looked at yet. So this one's going to be pretty light and easy.

Web Vitals

Let's start with this little chunk at the bottom of index.js:

js
// If you want to start measuring performance in your app, pass a function
// to log results (for example: reportWebVitals(console.log))
// or send to an analytics endpoint. Learn more: https://bit.ly/CRA-vitals
reportWebVitals();

From the comment we can already tell reportWebVitals has something to do with site performance. So let's open up reportWebVitals and see what's inside:

js
// reportWebVitals.js

const reportWebVitals = onPerfEntry => {
  if (onPerfEntry && onPerfEntry instanceof Function) {
    import('web-vitals').then(({ getCLS, getFID, getFCP, getLCP, getTTFB }) => {
      getCLS(onPerfEntry);
      getFID(onPerfEntry);
      getFCP(onPerfEntry);
      getLCP(onPerfEntry);
      getTTFB(onPerfEntry);
    });
  }
};

export default reportWebVitals;

If you've ever poked around SEO, a few of these names probably look familiar: CLS, FID, FCP, LCP, and TTFB. My own blog actually got flagged once for failing CLS (article in Mandarin) out of the LCP, FID, and CLS trio. These are Google's Core Web Vitals.

Basically reportWebVitals is all about those Core Web Vitals. So if you want to use it, just head back to index.js and change reportWebVitals(); to reportWebVitals(console.log);. Now you'll see the numbers and results show up in your browser console.

web vitals output in the console

So yeah, reportWebVitals ties straight into Google's Core Web Vitals.

StrictMode

Now let's back up a bit to the code above it. I trimmed the example down to focus just on the part we're about to talk about:

js
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';

const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);

In all the previous chapters we never really mentioned or used StrictMode (React.StrictMode). So what is StrictMode, exactly? In short, it's kind of like JavaScript's strict mode ('use strict';).

But what makes React's StrictMode special? Basically, any component wrapped inside <React.StrictMode> gets the following:

  • Identifying components with unsafe lifecycles
  • Warning about legacy string ref API usage
  • Warning about deprecated findDOMNode usage
  • Detecting unexpected side effects
  • Detecting legacy context API
  • Ensuring reusable state

All of these come straight from the React docs.

In short, it's there to make your code a bit safer.

There's one thing to watch out for though. If you write it like this:

js
import React from 'react';
import ReactDOM from 'react-dom/client';
import App from './App';

import Header from './Header';
import Footer from './Footer';

const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(
  <Header />
  <React.StrictMode>
    <App />
  </React.StrictMode>
  <Footer />
);

Then <Header /> and <Footer /> won't get the StrictMode treatment — only <App /> and everything underneath it (its components and child components) gets set to StrictMode.

So let's quickly walk through what actually happens once you use StrictMode. We'll write a simple example for it:

jsx
function App() {
  return (
    <div>
      <h1>React StrictMode</h1>
      <p>This is a StrictMode test example</p>
    </div>
  );
}

const root = ReactDOM.createRoot(document.querySelector('#root'));
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);

In this example we've only got one component, and it doesn't use any lifecycle stuff, so you won't see any weird error messages. But the moment you try doing something with useEffect, the results get a little different:

jsx
function App() {
  React.useEffect(() => {
    console.log('useEffect');
  }, []);

  return (
    <div>
      <h1>React StrictMode</h1>
      <p>This is a StrictMode test example</p>
    </div>
  );
}

const root = ReactDOM.createRoot(document.querySelector('#root'));
root.render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);

Open the console and you'll notice useEffect ran twice — normally it should only run once. This happens because StrictMode checks whether your component has any unsafe lifecycle behavior, so it deliberately runs things twice. The idea is to make it easier for you to spot problems.

If StrictMode sounds interesting, you can check out the React docs or this discussion.
