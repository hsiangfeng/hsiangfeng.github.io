Intro

Next up we're getting to know a new React Hook: useEffect. The name looks a little odd at first glance, because if you translate it straight into plain English it's basically "use effect." Effect of what, exactly? Let's find out.

Side Effect

Before we touch useEffect, there's one concept we need to nail down first: the Side Effect. So what's a Side Effect?

The classic analogy is the "medicine" a doctor prescribes.

Yep, exactly the kind you're thinking of — the stuff on the prescription. Picture it: after you catch a cold, the bag of meds you get usually has "side effects" printed on it somewhere, and those side effects might be things like an allergic reaction, drowsiness, or an upset stomach.

I'll admit the word "side effect" makes everything sound bad, but a side effect can actually be helpful and beneficial too. Here's a quick way to look at it:

Text
After you take the medicine your body gets better, but along with it comes an upset stomach

That little description actually contains two side effects: "your body gets better" and "an upset stomach." So side effects aren't all bad.

So why am I bringing this up? Well, the things we run into all the time — AJAX, DOM manipulation, even console.log — are all a kind of side effect. And in real development we lean on side effects constantly, so understanding them helps you avoid writing some dangerous code.

So in the world of code, what does a side effect look like? For example, reassigning a value is also a side effect:

JavaScript
var myName = 'Ray';
... // a few thousand lines later
myName = 'Array';

Bet you didn't expect that reassigning a value counts as a side effect.

But it's true — anytime you mutate the original value, that's a side effect. You can see the original value above is 'Ray', and then who-knows-how-many hundreds or thousands of lines later we reassign myName to 'Array'. That whole process can lead to unpredictable bugs that are a pain to track down.

This is just a rough overview, of course. If this area interests you, check out something I wrote earlier — "A Fun, Simple Take on Functional Programming in JavaScript" (article in Mandarin) — for more detail.

I'm only touching on what a side effect is here, since the React docs put it like this:

Data fetching, setting up a subscription, and manually changing the DOM in React components are all examples of side effects.

Alright, now that we've got a rough idea of what a side effect is, let's get to know useEffect.

useEffect

useEffect is mostly used when we want to do something after the screen renders. Basically you're telling React, "hey, after you render, remember to run this stuff for me." The things you put in here are usually side-effect-related, so as a rule of thumb, any behavior that produces a side effect is best placed inside useEffect — things like the AJAX calls and DOM changes I mentioned earlier.

And if you came from Vue, what's useEffect similar to? It's kind of like the Composition API's onMounted.

JavaScript
··························const { createApp, ref, onMounted } = Vue;

const app = createApp({
  setup() {
    const text = ref('Hello Ray.');
    
    onMounted(() => {
      console.log(text.value)
    })
  }
});

app.mount('#app');

So I said useEffect is "remember to run this stuff after rendering," but useEffect actually takes two arguments: a function and an array. The second one, the array, is optional, so depending on how you use it you'll get different results. Let's walk through the various ways to use useEffect.

After Every Render

First let's look at the first useEffect example:

jsx
············································function App (){
  const [count, setCount] = React.useState(0);

  React.useEffect(() => {
    console.log('render');
  })
  
  return (
    <div>
      <button type="button" onClick={() => setCount(count + 1)}>
        Count is: { count }
      </button>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

If you only pass useEffect a function and skip the second argument, then once you start clicking the button on screen you'll notice console.log('render') keeps firing over and over.

So this first form of useEffect means: every time the screen triggers a re-render, useEffect runs again right along with it.

Once

But in practice we often only need to run something once, not every single time the screen re-renders. So what do we do? Just pass useEffect a second argument — an empty array — and you'll get the run-once behavior:

jsx
············································function App (){
  const [count, setCount] = React.useState(0);

  React.useEffect(() => {
    console.log('once render');
  }, []);
  
  return (
    <div>
      <button type="button" onClick={() => setCount(count + 1)}>
        Count is: { count }
      </button>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

Now no matter how much you click the Count button, it won't keep re-firing useEffect like the previous example did. The concept here is pretty close to Vue's onMounted lifecycle.

On State (watching a specific value)

Besides an empty array, the second argument of useEffect can also take the thing you want to watch. Watch what, exactly? You watch a useState value — whenever that useState changes, useEffect fires.

One thing to note here, though: useEffect always runs once first, so seeing one render in the console is normal. After that, it only fires again when the value you're watching actually changes.

jsx
··················································································function App (){
  const [count, setCount] = React.useState(0);
  const [myName, setMyName] = React.useState('Ray');
  function sayHi () {
    window.alert('Hello Ray.');
  }
  
  React.useEffect(() => {
    console.log('render');
  }, [ count ])
  
  return (
    <div>
      <button type="button" onClick={() => setCount(count + 1)}>
        Count is: { count }
      </button>
      <hr />
      <button type="button" onClick={() => setMyName('QQ')}>
        Change name { myName }
      </button>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

You can see that when you click Count, useEffect fires along with it, but when you click the Change name button, useEffect doesn't fire. By watching a specific value you can avoid the "After Every Render" situation.

And of course, you can bind multiple states too.

jsx
······React.useEffect(() => {
    console.log('render');
  }, [ count, myName ])

This concept is also a lot like Vue's watch.

cleanup function

The cleanup function is a little special.

So how do you write a cleanup function? It's actually simple — as long as your useEffect returns a function, that returned function is the cleanup function:

JavaScript
··········useEffect(() => {
  console.log('useEffect');
  return () => {
    console.log('cleanup function');
  }
})

So when does it fire? On the first time you land on the screen, console.log('useEffect'); fires, and console.log('cleanup function'); does not. The cleanup function only gets called once when you trigger a re-render of the screen:

JavaScript
··························································function App (){
  const [count, setCount] = React.useState(0);

  React.useEffect(() => {
    console.log('useEffect');
    return () => {
      console.log('cleanup function');
    }
  })
  
  return (
    <div>
      <button type="button" onClick={() => setCount(count + 1)}>
        Count is: { count }
      </button>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

You can see console.log('useEffect'); fires at the start, and then once you press the Count button, console.log('cleanup function'); fires first, followed by console.log('useEffect');.

This cleanup function is great for things like resetting state or clearing specific event bindings. In Vue terms, it's similar to the beforeDestroy / onBeforeUnmount lifecycle.

From all this, we can see useEffect is a bit more involved than useState.

A common useEffect gotcha

A lot of people hit one particular trap when they first start using useEffect: "watching the same value you're assigning," which makes useEffect blow up on the spot. Below is a blow-up example — you can uncomment it and try it yourself, but use it with care (I'm not responsible if it crashes)

jsx
······························function App (){
  const [count, setCount] = React.useState(0);

  // Uncomment with care — be mentally prepared for your computer to freeze
  // React.useEffect(() => {
  //   setCount((pre) => pre + 1);
  // }, [ count ])
  
  return (
    <div>
      { count }
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);
