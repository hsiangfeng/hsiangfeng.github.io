Intro

Let's talk about event handling in React. It's one of those things you'll be doing constantly once you start building anything real, so it's worth slowing down on.

Event handling in Vue

First, a quick detour through Vue — a lot of us land on React after Vue, so this gives us a nice point of comparison.

Vue uses the v-on:eventName="function" pattern. Here's a click event (click) that fires a little greeting function (sayHi):

JavaScript
<a href="https://israynotarray.com/" v-on:click="sayHi">Click me</a>

In practice though, you rarely actually type out v-on:. Most of the time you use the @ shorthand instead:

HTML
<a href="https://israynotarray.com/" @click="sayHi">Click me</a>

And that's the most basic way to handle events in Vue.

Vue event modifiers

If you played with the example above, you'll notice that after the alert pops up, the page jumps straight to my blog (which, okay, was kind of the plan all along).

blog

That's happening because the link has a default behavior — it redirects to the href URL, the same way a form's action path kicks in on submit. To stop it, Vue gives you event modifiers. There are a bunch of them, but I'll only cover one here: .prevent. Just tack .prevent onto your @click and the redirect problem goes away:

HTML
<a href="https://israynotarray.com/" @click.prevent="sayHi">Click me</a>

So with an event modifier we stop the link from jumping away on click.

Okay, that's our quick refresher on Vue event handling and event modifiers. So what about React? Let's keep going.

Event handling in React

We just recapped that Vue handles events with v-on: and @. React's a little different — but if you've ever written an old-school onclick, React's approach is going to feel really familiar:

HTML
<a href="https://israynotarray.com/" onclick="sayHi(event)">Click me</a>

That onclick example is what we call an inline event, usually written as on+[eventName]="function()". In real-world projects we actually try to avoid inline events — I won't get into why here, since that's a whole other topic.

Back to React.

So why did I bother bringing up inline events? Because React's event handling looks about 87% identical to it. Here's how a click event looks in React:

jsx
<a href="https://israynotarray.com/" onClick={ sayHi }>Click me</a>

So yeah, inline events and React event handling are nearly twins, but React's syntax is on+[eventName]={ function } — the event name starts with a capital letter and uses camelCase.

Let me list the two key differences:

  • Inline events are all lowercase; React event handling is camelCase.
  • Inline events take a string; React event handling takes a function.

Does React have event modifiers?

At this point you might be wondering...

Does React have event modifiers like Vue does? And if so, would you write something like onClick.prevent={ sayHi }?

Smart question! And the answer is...

Nope, no such thing!

The reason is simple: React basically feels like writing plain JavaScript, so you have to explicitly call preventDefault(); yourself to stop the default behavior:

JavaScript
······························const App = () => {
  const sayHi = (e) => {
    e.preventDefault()
    window.alert('Hello！');
  }

  return (
    <div>
      <a href="https://israynotarray.com/" onClick={ sayHi }>Click me</a>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

This is exactly why you keep hearing people say that writing React feels like writing plain JavaScript.

Here's a quick list of the common React events:

  • onClick
  • onChange
  • onSubmit
  • onKeyDown
  • onFocus
  • onBlur

And remember: React's syntax is on+[eventName]={ function } — never on+[eventName]="name()"!

Reading descriptions only gets you so far, so I'll write a quick example for each one.

onClick

We've already written onClick examples a bunch of times, but let's look at it once more anyway so the pattern really sticks.

onClick is the most basic click event — when you click, it fires the function you passed in. If you don't pass any arguments and just hand it a function name, React passes in an event (the click event) by default:

jsx
······························const App = () => {
  const handler = (event) => {
    event.preventDefault()
    window.alert('Click！');
  }

  return (
    <div>
      <a href="https://israynotarray.com/" onClick={ handler }>Click me</a>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

(This maps to Vue's @click event.)

onChange

onChange is most often used on input, textarea, and select elements. For example, when we pick something from a select, we can grab the option value:

jsx
··················································const App = () => {
  const handler = (event) => {
    window.alert(`You picked: ${event.target.value}`);
  }

  return (
    <div>
      <select onChange={ handler } name="location">
      　<option value="Taipei">Taipei</option>
      　<option value="Taoyuan">Taoyuan</option>
      　<option value="Hsinchu">Hsinchu</option>
      </select>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

(This maps to Vue's @change event.)

onSubmit

onSubmit is mainly used on form elements. A form basically has three musketeers: the Form, the input fields, and the Submit button.

JavaScript
····························································const App = () => {
  const handlerSubmit = (e) => {
    e.preventDefault();
    window.alert('submit');
  }

  return (
    <div>
      <form onSubmit={ handlerSubmit }>
        <label htmlFor="account">Account</label>
        <input id="account" type="text" />
        <button type="submit">Submit</button>
      </form>
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

Heads up: the for attribute on label has to be written as htmlFor, otherwise you'll get this error: Warning: Invalid DOM property for. Did you mean htmlFor?. That happens because JSX is JavaScript under the hood, and for collides with the reserved word for (as in the for loop), so React renames it.

(This maps to Vue's @submit event.)

onKeyDown

Besides onSubmit and onChange, another common listener is onKeyDown. If you're building something like a ToDoList, you'll reach for onKeyDown all the time — for example, to listen for which key is being pressed:

JavaScript
·······························const App = () => {
  const handler = (e) => {
    e.preventDefault();
    window.alert(`You pressed the ${e.key} key, keyCode is ${e.keyCode}`);
  }

  return (
    <div>
       <input type="text" onKeyDown={ handler } />
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

(This maps to Vue's @keyup event.)

onFocus

onFocus mostly shows up on input fields — it fires the moment we focus on that field:

JavaScript
···························const App = () => {
  const handlerFocus = () => {
    window.alert('Focus');
  }

  return (
    <div>
       <input type="text" onFocus={ handlerFocus } />
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

(This maps to Vue's @focus event.)

onBlur

If focus is when you enter a field, then blur is the opposite — it fires when you leave it:

JavaScript
···························const App = () => {
  const handlerBlur = (e) => {
    window.alert('Blur');
  }

  return (
    <div>
       <input type="text" onBlur={ handlerBlur } />
    </div>
  )
}

const app = document.querySelector('#app');
const root = ReactDOM.createRoot(app);
root.render(<App />);

(This maps to Vue's @blur event.)

And that's pretty much the rundown of the common events you'll run into.
