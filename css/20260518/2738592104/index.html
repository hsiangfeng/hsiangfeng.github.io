為什麼 CSS overflow: scroll 看不到捲軸？macOS 跟 Windows 的 Chrome 差異

#CSS #overflow #捲軸 #scrollbar #macOS #Chrome #前端開發

前言

相信很多寫 CSS 的人都有遇過這個情境，明明在自己的元素上加了 overflow: scroll，但打開 Chrome 卻看不到捲軸，傳給同事看，他那邊卻又跑出一條灰灰的捲軸，明明用的都是 Chrome，怎麼會有這種差異？這一篇就從 overflow 屬性是什麼開始講。

先聊聊 overflow 是什麼

在分析問題之前，先稍微回顧一下 overflow 這個屬性是什麼。

簡單來講，overflow 是用來決定當一個元素裡面的內容超出它本身的寬高時，要怎麼處理那一段超出的部分，這個屬性算是 CSS 裡面非常基本但又非常常用的東西。

舉個例子，假設我有一個寬 200px、高 100px 的 div，但裡面塞了一段超長的文字 or 一張比 200px 還寬的圖片，這時候畫面就會發生兩件事：

  • 內容直接跑出（溢出）容器到外面去
  • 內容被切掉，看不到後面那一段
  • 出現捲軸，讓使用者自己滾動

那這要怎麼解決呢？那就要來講講 overflow 了。

overflow 的四個常見屬性

overflow 屬性有四個比較常用的屬性，這邊用一張表格快速看一下：

屬性 用途
visible 預設屬性，當內容超出容器範圍就會讓它跑出去，不做任何處理。
hidden 內容超出的部分直接被切掉，看不到也沒辦法捲動。
scroll 不管內容有沒有超出，都保留捲軸的位置一直顯示捲軸。
auto 只有當內容真的超出時，捲軸才會出現。

這邊要特別拉 scroll 跟 auto 出來講，因為這兩個屬性最容易搞混，但它們兩者差異只有一個小地方，也就是 scroll 是「永遠都會出現捲軸」，但 auto 則是「內容超出時，需要才會出現捲軸」。

那為什麼 overflow: scroll 在我電腦上看不到捲軸？

這是因為 macOS 預設使用的是 「隱形捲軸（Overlay Scrollbar）」，這種捲軸只有在你正在捲動時才會浮現出來，平常是隱藏的，而且不會占據版面的空間。但你同事如果是用 Windows，那邊的捲軸是 「古典捲軸（Classic Scrollbar）」，會固定保留位置而且永遠看得到。

所以兩個人開同一個網頁、同一個版本的 Chrome，但看到的視覺結果就會不同，根本原因不是 Chrome 不一樣，而是底下的作業系統不一樣。

macOS 的「隱形捲軸」

在 macOS 上，捲軸的顯示行為是由系統設定決定的，查看方式是「設定 → 外觀 → 顯示捲軸」，這邊有三個選項可以選：

通常預設屬性是「自動依揵滑鼠或觸控式軌跡板」，也就是當你用的是觸控板 or Mac 內建的軌跡板時，它就會走「捲動時」的行為，有點饒舌對吧？簡單來講，也就是平常完全看不到捲軸，只有你在捲的當下會浮出一條半透明的細條，停下來幾秒它又會自動淡出消失。

而 Chrome、Safari 跟其他大多數的 Mac 瀏覽器，預設都會跟著這個系統設定走，所以即便你在 CSS 寫了 overflow: scroll，瀏覽器還是會用「隱形捲軸」的方式渲染，不會把捲軸固定地顯示在那邊。

那要怎麼確認到底是不是這個原因呢？很簡單，你只要走兩個步驟驗證：

  1. 把 macOS 的**「顯示捲軸」切到「總是」**
  2. 重新整理一下你的瀏覽器（或者重新開啟）

這時候捲軸就會直接固定地出現了，這代表 CSS 沒寫錯，是作業系統的設定在控制視覺呈現。

為什麼 Windows 沒這個問題？

因為 Windows 從以前到現在都是用「古典捲軸」，捲軸會固定占住版面右側（大約 15 ~ 17px 寬），不管你有沒有在捲都看得到，也不會自動淡出消失。

所以你同事打開同一個頁面時，那條灰灰的細長捲軸就直接掛在右邊，他寫的 overflow: scroll 看起來就是 CSS 預期的樣子，沒有「捲軸到底跑去哪了」的疑問。

這邊順便提一個容易忽略的點，如果你都在 Mac 上開發、可是實際使用者大多用 Windows，那你看到的內容寬度其實會比使用者那邊還多一段（被捲軸吃掉的那段）。做 RWD 切版時，建議借同事的 Windows 開一下看看版面有沒有跑掉，免得交付後才發現右邊有點擠。

那怎麼讓兩邊看起來一樣？

如果你希望不管使用者用什麼系統都看得到捲軸（例如怕 Mac 使用者根本不知道內容還可以往下捲），這邊有幾種做法可以選：

方法一：用 ::-webkit-scrollbar 強制顯示

最直接的做法是用 ::-webkit-scrollbar 這個 CSS 偽元素去自訂捲軸樣式，只要你給它一個明確的 width 或 height，Chrome 就會自動把這個元素的捲軸從隱形捲軸切換成古典捲軸，永遠固定顯示。

寫完之後你會發現 Mac 上的 Chrome 終於把捲軸固定顯示出來了，而且因為它已經變成古典捲軸，會吃掉一段內容寬度，這時候版面就會跟 Windows 上看到的比較接近。

可是這個做法有兩個小坑要記得：

  • ::-webkit-scrollbar 是 Webkit / Blink 系列才支援的非標準屬性，Chrome、Edge、Safari、Opera 都吃，可是 Firefox 不吃
  • iOS Safari 對 ::-webkit-scrollbar 的支援也很有限，大部分情況下還是會走系統的隱形捲軸

方法二：用標準化的 scrollbar-width 跟 scrollbar-color

如果你不想用非標準屬性，CSS 後來也有把捲軸樣式標準化，新增了 scrollbar-width 跟 scrollbar-color 兩個 W3C 標準屬性（出自 CSS Scrollbars Module Level 1），跟 ::-webkit-scrollbar 比起來最大的差異就是它有規格背書。

  • scrollbar-width 可以設 auto、thin 或 none，分別對應「預設粗細」、「細捲軸」、「隱藏捲軸」
  • scrollbar-color 接收兩個顏色值，前者是捲軸 thumb 的顏色，後者是 track 的底色

可是支援度要先注意一下，Firefox 從 64 就支援這兩個屬性了，Chromium 系列（Chrome、Edge）要到 Chrome 121 之後才支援，Safari 則是要 Safari 26.2 之後才有完整支援。如果你的網站還要兼容比較舊的瀏覽器，建議把 ::-webkit-scrollbar 一起寫上去當 fallback。

Note
MDN 文件有提到，當 scrollbar-width 或 scrollbar-color 設成非 auto 的值時，會直接蓋掉 ::-webkit-scrollbar 的樣式。所以兩個一起寫的時候，要注意誰會贏。

為什麼不用 scrollbar-gutter？

看到這邊你可能會想說，CSS 不是還有 scrollbar-gutter 可以拿來保留捲軸位置嗎？那直接用它來解這個問題不是更省事？

可惜不行，這個屬性對 macOS 的隱形捲軸沒效果。

scrollbar-gutter: stable 原本的設計目的是保留捲軸需要的版面空間，避免內容因為捲軸有無而抖動。可是 MDN 文件上有明確寫一行：當瀏覽器使用隱形捲軸時，scrollbar-gutter 不會有任何效果。

簡單來講，它要解決的是古典捲軸那種會占空間的場景下，內容因為捲軸出現/消失而跳動的問題。可是 macOS 預設用的是隱形捲軸，本來就不占空間，自然也沒東西好保留。

所以如果你的目標是讓 Mac 上的 Chrome 永遠看得到捲軸，scrollbar-gutter 解決不了這個問題，還是要回去用 ::-webkit-scrollbar 或 scrollbar-width。

結語

回到開頭那個「同事看得到、你看不到」的情境，這個問題的根源其實不是 CSS 寫錯，也不是 Chrome 哪裡壞掉，而是作業系統層級的捲軸機制本來就不一樣 — macOS 預設走隱形捲軸，Windows 預設走古典捲軸，兩邊對 overflow: scroll 的視覺呈現自然就不同。

知道這件事之後，下次再遇到一樣的狀況就不用懷疑自己 CSS 寫錯，可以直接想到是系統設定差異，再依需求選 ::-webkit-scrollbar or 新版的 scrollbar-width 補上一致的視覺。
