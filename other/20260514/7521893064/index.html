分享網址到 LINE、Discord 沒預覽圖？用 ?v=1 強制重抓社群 OG cache

#Discord #Facebook #Open Graph #og:image #社群分享 #LINE #cache busting #query string

前言

這篇從分享自己部落格文章到社群結果預覽圖一直是舊版的問題切入，講 Open Graph 標籤、社群平台對 og:image / og:title / og:description 的 cache 機制、為什麼 FB / X / LINE / Discord 在你改完 OG 標籤後還是抓不到新內容，以及用 ?v=1 ?v=2 這種 cache-busting query string 強制讓社群 cache 不認得而重新抓取的技巧。也整理 Facebook Sharing Debugger 用法、各平台 cache 大致過期時間、Twitter / X Card Validator 2022 停用後該怎麼測，以及 ?v=1 對 SEO 跟 canonical URL 會不會造成影響。

如果你前一陣子也有在用 Cloudflare Pages 部署部落格 or 試圖鎖預覽網址，可以順便看看 Cloudflare Pages 預覽部署網址會永久保留？用 Cloudflare Access 鎖住外洩風險 那篇，跟這一篇可以一起服用。

事情是這樣的

事情是這樣的，我前幾天寫了一篇 Cloudflare Access 鎖預覽網址的文章，部署完興沖沖把網址分享到 LINE 給朋友看，結果分享出來的卡片沒有封面圖，只有一張預設的章魚貓出現在預覽縮圖上。

當下覺得奇怪，因為文章內第一張就是封面圖，按理講 og:image 應該會抓到那張才對。回頭去看部落格 HTML 確認 og:image 真的有指對封面圖、curl 圖檔 URL 也回 200 OK，可是 LINE 那邊就是不認。

換到 Discord 貼一次、X 貼一次，全部都是同一張預設圖。連自己看了都想笑，明明文章已經改好了，社群這邊就是不更新。

後來查了一下才知道，這不是部落格的問題，而是社群平台自己的 OG cache 在作祟。

為什麼社群會 cache 預覽圖？

先講結論：社群平台會把每個網址的 OG metadata cache 起來，第二次貼同樣的網址不會回你的網站重抓，而是直接拿 cache 顯示。

OG 是 Open Graph 的縮寫，由 Facebook 在 2010 年提出的一個 metadata 規範，目的是讓社群平台分享連結時，可以快速取得這個連結的標題、描述、預覽圖等資訊。網站只要在 <head> 裡放這幾個 meta tag，社群就會抓來顯示

那社群為什麼要 cache？簡單來講就是省成本。

想想看，你發一則貼文夾一個連結，如果 100 個朋友都看到這則貼文，社群平台不可能讓 100 個人的手機分別去打你的網站抓 OG tag，那是 100 個 request。所以社群會在第一次有人貼這個網址時，自己去抓一次 OG metadata 存起來，之後其他人看到這則貼文 or 後來再有人貼相同網址，都從 cache 直接拿，不會再打你的網站。

這個 cache 的設計本身是合理的，可是站在內容創作者角度，當你改了 og:image 想要更新預覽的時候，cache 就變成擋路的東西。

各平台 cache 時間大概抓一下

平台 cache 過期時間 有官方 debugger 嗎？
Facebook 約 7–30 天 有（Sharing Debugger）
LinkedIn 約 7 天 有（Post Inspector）
X（前 Twitter） 約 24–48 小時 已停用預覽功能
Discord 約 30 分鐘 沒有
LINE 官方未公布 沒有
Slack / Telegram 官方未公布 沒有

從這張表可以看出來，Discord 算是 cache 過期最快的，Facebook 跟 LinkedIn 都偏長。LINE 跟 Slack、Telegram 官方都沒公布過數字，遇到的話只能用接下來要講的這招處理。

用 ?v=1 強制重抓 cache

最簡單的強制重抓方式就是在網址後面加一段 query string，例如

對社群的 cache 來說，網址字串不一樣就是不同條目，原本的 https://israynotarray.com/other/20260513/8264193057/ 已經有 cache，可是加上 ?v=1 之後變成「新網址」，cache 找不到對應紀錄，就會回你網站重抓一次最新的 OG metadata。

v=1 的數字隨便挑，下次要再強制刷新就改 v=2、v=3，每次給一個沒貼過的新值就好。也不一定要叫 v，叫 ?refresh=1、?nocache=true 都可以，只要是「以前沒貼過的 query string」就有效。

你的網站收到 ?v=1 不會壞掉

可能你會擔心：那 ?v=1 跑進我網站會不會有事？

放心，多數 framework 跟 web server 預設都會忽略未知的 query string。Hexo、Next.js、Vite、Cloudflare Pages、Vercel 等等，收到 ?v=1 就照原本 path 走，內容跟沒帶 query string 完全一樣，讀者看到的網頁不會差任何一個字。

Note
如果你的網站有自己處理 query string（例如分頁 ?page=2 or 搜尋 ?q=xxx），那只要不挑跟你 query string 同名的字串就好。?v、?refresh、?nocache 這些是很安全的選擇，幾乎不可能跟你網站原本的功能撞名。

各平台對應做法

Facebook：用 Sharing Debugger

Facebook 是唯一還活著的官方 debugger，網址 https://developers.facebook.com/tools/debug/。把你的網址貼進去，點 Debug 看 FB 抓到什麼，如果是舊資料就按 Scrape Again 強制重抓。

Sharing Debugger 也會告訴你如果 og:image 有問題（檔案太大、格式不支援、抓不到等等），對 debug 很有幫助。

Note
Facebook 到 2026 為止官方文件仍然只支援 JPG、PNG、GIF 三種格式當 og:image，WebP 不支援。如果你用 WebP 圖檔 og:image 在 FB 就會壞掉。建議 og:image 用 PNG or JPG，網站其他圖片再用 WebP 提速即可。

另外 og:image 建議尺寸是 1200 x 630 像素，小於 600 x 315 的圖 FB 會顯示成小縮圖 or 直接不顯示。

實際上有時候第一次 Scrape Again 沒用，連續點個兩三次反而會成功，這個沒有什麼合理解釋，純粹是 FB 自己會抽風。

X、LinkedIn、LINE、Discord：直接用 ?v=1

X 的 Card Validator 在 2022 已經拿掉預覽功能，現在剩下的網頁也漸漸難用。LinkedIn 有 Post Inspector 可是限制比較多。LINE、Discord、Slack、Telegram 從以前就沒提供 debugger。

所以對這幾個平台，最快也最通用的解法就是 ?v=1 cache busting。下次貼網址到 LINE 群組之前先在後面加個 ?v=1，社群就會回你網站重抓一次。

Note
已經分享到群組的舊訊息不會更新，社群 cache 是綁在當時那則訊息上，舊訊息抓到什麼就永遠是什麼。要救只能刪掉原訊息，再用 ?v=1 重貼一次。

替代方案：第三方 OG 預覽工具

如果你想在分享前先看看會長什麼樣，市面上也有不少第三方工具可以模擬抓 OG，例如

  • opengraph.dev
  • share-preview.com
  • microlink.io

這些工具是即時抓你網頁的 OG meta，跟社群本身的 cache 無關，所以可以拿來在發文前確認 og:image 真的有抓到。可是要注意，第三方工具能抓不代表 FB / LINE 抓得到，最終還是以官方 debugger（FB Sharing Debugger）為準。

那 SEO 會被影響嗎？

不會，前提是你的網站有設好 canonical URL。

<link rel="canonical"> 是放在 <head> 裡的一個標籤，告訴搜尋引擎「這個頁面的正規網址是哪一個」。當 Google 抓到 https://your-site.com/post/?v=1 這個變體網址，發現

它就會把 ?v=1 當成跟原網址同一頁處理，搜尋權重不會被分散，也不會被視為 duplicate content 而被懲罰。

而且 ?v=1 你只是偶爾為了強制重抓社群 cache 才用一次，不會持續對外推廣這個變體網址，所以實際上 Google 連看到都未必。對 SEO 沒任何實質影響。

Note
如果你不確定自己的網站有沒有 canonical URL，可以打開文章在瀏覽器按 F12 → Elements 找 <link rel="canonical">，看看那個 href 是不是指向你預期的網址。多數現代部落格架構（Hexo、Astro、Next.js、Vite SSG）都會幫你自動加。

補充：分享前先測過再發

最後分享一個我自己被坑過之後養成的習慣：發文之前先用 FB Sharing Debugger 跑一次。

流程很簡單

  1. 文章 publish 完先打開 Facebook Sharing Debugger
  2. 貼上文章網址，點 Debug
  3. 看下方的 og:image、og:title、og:description 有沒有抓對
  4. 如果抓錯就先回去修，修完再 Scrape Again
  5. 確認都沒問題再正式分享到各社群

雖然多一個步驟，可是比起發出去之後才發現沒圖、要刪訊息重貼、還要叫朋友重新點一次，這個事前檢查的 1 分鐘很值得。

結語

社群 cache 這件事其實在大多時候是好的（節省你網站的流量、加快預覽速度），可是當你急著更新 og:image 的時候它就會變成擋路的東西。?v=1 這招說起來很土，可是它真的好用，特別是對 LINE 跟 Discord 這種沒 debugger 的平台來講幾乎是唯一的辦法。

希望這一篇有讓你對社群 OG cache 跟 cache-busting 的概念有更近一步的認識哩～如果你也是用 Hexo or Cloudflare Pages 在跑部落格，可以再看一下 Cloudflare Pages 預覽部署網址會永久保留？用 Cloudflare Access 鎖住外洩風險 那篇，把社群分享 + 部署側的小坑一起補起來哩。
