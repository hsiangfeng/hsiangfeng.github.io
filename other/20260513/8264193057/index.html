Cloudflare Pages 預覽部署網址會永久保留？用 Cloudflare Access 鎖住外洩風險

#Cloudflare #資安 #Cloudflare Pages #Cloudflare Access #Zero Trust #One-time PIN #預覽部署

前言

Cloudflare Pages 每次部署都會產生一組 hash 預覽網址（例如 d682e28c.your-project.pages.dev），這篇從預覽網址永久保留的安全疑慮聊起，整理如何用 Cloudflare Access 免費 50 人額度設定 Self-hosted 應用程式，搭配 Emails ending in 原則把預覽鎖起來，包含建立應用程式步驟、Access 原則的 OR 規則寫法，以及收不到 One-time PIN 信時的三個排查方向。

如果你還沒有自己的 Cloudflare 網域 or Pages 專案，可以先看一下 Google Domains 停止後怎麼辦？用 Cloudflare 建立網域、部署網站完整圖文教學，那一篇有完整的 Cloudflare Pages 部署流程，跟這一篇可以一起服用。

事情是這樣的

最近在玩 Hexo 部落格的自動部署，把專案接到 Cloudflare Pages，每次 push 上去就會自動觸發新的部署。Cloudflare 很貼心，除了正式版的 your-project.pages.dev 之外，每次部署都還會額外給你一組獨立的預覽網址，網址長這樣

那一串 d682e28c 其實就是 commit hash 的前 8 碼，這個機制超實用，你可以拿去分享給同事 or 老師看「我這一版改完長這樣」，不用先合進主分支也能 demo。

可是當下我突然想到一件事：這個預覽網址會保留多久？如果這個連結不小心被別人轉貼出去，會不會變成資安問題？因為有些公司會 Preview 測試一些敏感內容（例如內部專案、還沒公開的功能頁面），如果預覽網址會一直存在，那不就等於說這些內容會一直躺在那邊，等人挖出來看嗎？

於是查了一下…對，永久。

Cloudflare Pages 預覽網址會保留多久？

如同前面所言：永久。

Cloudflare Pages 官方文件 Preview deployments 裡面其實寫滿清楚的

When a preview deployment is published, it is given a unique, hash-based address — for example, <hash>.<project>.pages.dev. These are atomic and may always be visited in the future.

翻成白話就是：每個預覽部署都會綁一組 hash 網址，這個網址是 atomic（原子的、不會變動的），未來都可以一直被造訪。

換句話說，只要你不手動進 Cloudflare 儀表板把那筆部署刪掉，那個網址永遠都打得開，看到的就是當時推上去的那一版內容。

Note
Cloudflare 不僅不會幫你自動清除舊預覽，而且根據官方文件，當一個 Pages 專案累積超過 100 筆部署的時候，連刪除整個專案都會失敗，必須先用 Wrangler 一筆一筆把舊部署清掉才行。

那這樣會有什麼風險？

hash 本身有 8 個字元的隨機性，要靠暴力枚舉去猜中其實不太可能，所以從攻擊面來看，它不是一個容易被掃描器掃到的網址。

可是這不代表沒風險，我列幾個常見情境…

  • 你把預覽連結貼到 Slack、Discord 群組
  • 你把連結傳給設計師、PM、老師看
  • 你 Tweet 出去 or 截圖到部落格上
  • 你 demo 給客戶看，會議結束後沒刪除

只要其中任何一個情境不小心讓不該看到的人看到了，這個網址就回不來了。就算你後來把那個分支砍掉、把專案改名，那筆預覽部署的網址還是會繼續活著，這就是讓我覺得有風險的地方。

因此這邊就來分享一下我自己用的方式，用 Cloudflare Access 在預覽網址前面加一道身分驗證，只有授權過的人才能進去。

用 Cloudflare Access 鎖住預覽

Cloudflare Access 是 Cloudflare Zero Trust 底下的一個 ZTNA（Zero Trust Network Access）服務，它的作用就是在你的網站前面架一道身份驗證關卡，任何人要進去都必須先通過驗證。

那它要錢嗎？根據 Cloudflare 官方 Zero Trust 方案頁面，Zero Trust 的 Free 方案最多 50 個使用者免費，對個人 blog 來講根本就是吃免費的等級。

步驟 1：建立 Self-hosted 應用程式

進到 Cloudflare 儀表板，左側選單找到 Zero Trust（如果是第一次進，會先要你取一個 Team name，那會變成你的登入網址 https://你取的名字.cloudflareaccess.com）。

接著走以下路徑

在目的地這個區塊，填入…

欄位 填什麼
子網域 *
網域 你的 Pages 網域，例如 your-project.pages.dev
路徑 留空

子網域填 * 是關鍵，這樣 <任何 hash>.your-project.pages.dev 全部都會被這個應用程式接管，正式版 your-project.pages.dev 也會被一起保護。

步驟 2：設定 Access 原則

接著要設定誰可以進來。在下方有一塊 Access 原則 區塊

點 建立新原則，填…

欄位 填什麼
原則名稱 例如 僅限自己人
動作 Allow
規則選擇器 結尾為以下的電子郵件（Emails ending in）
值 @gmail.com（你的公司網域）

如果你還想額外讓自己的 gmail 也能登入，可以在同一個原則裡再加一條規則

規則 選擇器 值
規則 1 結尾為以下的電子郵件 @gmail.com
規則 2 電子郵件 your-name@gmail.com

兩條規則之間是 OR 關係，任一條件成立就放行。

步驟 3：儲存應用程式

最後填上應用程式名稱、工作階段持續時間（建議 24 hours），按 建立 就完成了。

完成後開無痕視窗、貼上預覽網址，理論上你應該會看到 Cloudflare 的登入頁要你輸入 email，輸入正確 email 後會收到一封 PIN 信，輸入 PIN 之後就會看到網站內容。

設定好了卻收不到 PIN 信？

如果你跟我一樣，興沖沖設定完之後測試卻一直沒收到 PIN 信，先冷靜，三個地方檢查一下。

第一，垃圾信件夾。Cloudflare 的 PIN 信寄件人是 noreply@notify.cloudflare.com，Gmail 很常把它丟到垃圾郵件 or 促銷內容分頁。標題類似 Your one-time code is... 這種，先去這兩個資料夾翻翻看。

第二，你輸入的 email 沒有符合原則。這個最坑。如果你原則設定的是 @hexschool.com 結尾，可是你登入的時候打的是 gmail.com 的 email，Cloudflare 會直接拒絕、根本不會寄信給你，而且畫面也不會告訴你為什麼不寄，就是靜靜的什麼都沒有。

Note
這個設計從安全性角度來看是合理的（避免讓攻擊者知道哪些 email 在白名單上），可是對第一次設定的人來講真的很容易摸不著頭緒，請務必確認你輸入的 email 有在原則裡。

第三，One-time PIN 沒啟用。到 Zero Trust → 設定 → 驗證 → 登入方法 看看 One-time PIN 有沒有在啟用狀態，如果沒有就點 新增 把它打開。

補充：用 Wrangler 批次刪除舊預覽

如果你想做的不是「擋人看」而是「乾脆把舊預覽通通清掉」，可以用 Cloudflare 官方提供的 Wrangler CLI 一鍵把列表抓出來

配合 wrangler pages deployment delete <id> 一筆一筆刪，效率比點儀表板高很多。如同前面 Note 提到的，當部署累積超過 100 筆，連刪除整個專案都會失敗，所以這個指令在後期維護的時候會滿好用的。

結語

Cloudflare Pages 預覽部署是個雙面刃，該分享的時候很方便、該保密的時候就要記得加把鎖。在這篇文章之前，我其實滿天真地把預覽網址當作「臨時測試環境」，現在回頭看，它根本是個會永久躺在那邊等人挖出來的快照。

希望這一篇有讓你對 Cloudflare Pages 的預覽機制與 Access 保護有更近一步的認識哩～如果你也想把部落格搬上 Cloudflare，可以再看一下 用 Cloudflare Workers 快取 Firebase Storage 圖片，完全免費降低流量費用 這一篇，跟這一篇可以串成一條更完整的 Cloudflare 部署流程。我們下篇文章見囉～
