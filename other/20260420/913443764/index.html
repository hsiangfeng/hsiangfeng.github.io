從 Firebase Storage 搬家到 Cloudflare R2，完整遷移紀錄

#Node.js #Cloudflare #Firebase #Cloudflare R2 #Cloudflare Workers

前言

前陣子寫了一篇用 Cloudflare Workers 快取 Firebase Storage 圖片的文章，當時的想法是在 Firebase Storage 前面擋一層 Cloudflare CDN 來減少流量費用。

但用了一陣子之後發現一個問題，快取終究只是快取，cache miss 的時候還是會打到 Firebase Storage，而且 Firebase 從 2026 年 2 月開始強制所有專案升級到 Blaze（付費）方案，等於免費保護傘沒了，超過免費額度就是直接扣錢，更恐怖的是沒有任何提醒，帳單就這樣默默長上去。

我的部落格有兩個 Firebase Storage bucket，裡面加起來將近 4800 張圖片，分散在兩份帳單裡，每次查看帳單都要開兩個 Firebase Console，想想就覺得麻煩，所以最後就決定一不做二不休，直接把所有圖片搬到 Cloudflare R2。

為什麼選 Cloudflare R2？

在決定搬家之前，我先比較了一下兩邊的費用：

Firebase Storage (Blaze) Cloudflare R2
儲存 $0.026/GB/月 $0.015/GB/月
下載流量 $0.12 ~ $0.15/GB $0（免費）
免費額度 5GB 儲存（限特定 region） 10GB 儲存 + 每月 1000 萬次讀取

最關鍵的差異就是下載流量，Firebase 每 GB 下載要收 $0.12 到 $0.15 美金，而 R2 是零出口流量費，不管你的圖片被讀幾次都不收錢。

部落格圖片就是典型的「寫一次，讀很多次」的場景，每篇文章被瀏覽一次，裡面的圖片就被下載一次，如果哪天某篇文章被大量轉發，Firebase 的流量費用會瞬間爆衝，但 R2 完全不會有這個問題。

以我的 4800 張圖片來說，大概 10GB 左右，剛好完全落在 R2 的免費額度內，也就是說搬過去之後月費是 $0。

遷移步驟

接下來就來聊聊整個遷移的過程。

步驟一：從 Firebase Storage 下載所有圖片

第一步當然是要先把圖片從 Firebase Storage 全部下載下來。4800 張圖片不可能手動下載，所以我寫了一個 Node.js 腳本來處理。

要跑這個腳本需要 Firebase 的服務帳戶金鑰，到 Firebase Console → 專案設定 → 服務帳戶 → 產生新的私密金鑰，就能下載一個 JSON 檔。

這個腳本有幾個重點：

  • 10 條並行下載：不會太快被限流，也不會太慢
  • 斷點續傳：進度會存在 .progress.json 裡，中斷後重跑不會重複下載
  • 自動重試：每個檔案失敗會重試 3 次

跑完之後我看了一下結果，3920 個檔案下載成功，48 個失敗。但仔細看失敗清單，全部都是像 blogImg/Git/ 這樣結尾帶 / 的路徑，根本不是檔案，而是 Firebase Storage 在建立資料夾時產生的佔位物件，所以實際上是全數下載成功，不用擔心。

Note
從 Firebase Storage 下載圖片會產生下載流量費用，但以 4800 張圖片大約 10GB 來算，費用大概是 $0.12/GB × 10GB = $1.2 美金，不到台幣 40 塊，不用擔心。

踩到的坑

整個遷移過程看起來很順利，但其實踩了不少坑，這邊記錄一下，希望能幫到之後也想搬家的人。

坑一：URL 編碼問題

這是最痛的一個坑。

Firebase Storage 的 URL 裡，路徑中的 / 是用 %2F 編碼的，中文字和空格也都是 percent-encoded。替換的時候我一開始用了 decodeURIComponent 把整個路徑都解碼了，結果 URL 裡面就出現了空格和中文字：

Markdown 的圖片語法 ![](url) 遇到空格就會斷掉，所有包含中文或空格的圖片全部壞了。

正確的做法是只把 %2F 轉成 /，其他編碼保持不動：

坑二：上傳路徑前綴問題

上傳到 R2 時，檔案的 key 應該是 blogImg/Git/202107092329.png，但因為本地目錄前綴的字串替換沒對上（./firebase-images/ vs firebase-images/），導致 key 變成了 firebase-images/blogImg/Git/202107092329.png，多了一層目錄。

這個問題在 Worker 端可以修，但很不乾淨。比較好的做法是在上傳腳本裡用正規表示式來處理：

有些圖片 URL 後面莫名其妙多了 &_gl=1*5qrrwu*_ga*... 這一串東西，這是從 Firebase Console 複製 URL 時不小心帶進來的 Google Analytics 跨域追蹤參數，會導致圖片載入失敗。

搬完之後

搬完之後整理一下前後差異：

搬家前 搬家後
儲存位置 2 個 Firebase Storage bucket 1 個 Cloudflare R2 bucket
管理介面 要開 2 個 Firebase Console 只需要 1 個 Cloudflare Dashboard
下載流量費 $0.12/GB，沒有上限 $0
月費 看流量，不可預測 $0（在免費額度內）
圖片域名 2 個不同的域名 統一 images.israynotarray.com
流量暴衝風險 有，帳單直接飆 沒有，流量永遠免費

最大的改變就是再也不用擔心流量費用了，不管哪篇文章被多少人看、圖片被下載多少次，費用就是 $0，我終於可以安心睡覺了（笑）。

結論

整個遷移過程大概花了一個下午，主要時間都在等圖片下載和上傳，腳本跑起來之後基本上就是等它跑完而已。踩的坑大部分都是 URL 編碼相關的問題，但只要知道「只轉 %2F，其他不動」這個原則，就不太會出問題。

如果你也在用 Firebase Storage 存圖片，而且不想再看到那個流量帳單的話，搬到 R2 真的是目前最划算的方案哩～
