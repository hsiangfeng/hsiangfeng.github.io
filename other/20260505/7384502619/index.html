localhost 跟 127.0.0.1 是什麼？工程師笑話與「拒絕連線」常見原因解析

#localhost #127.0.0.1 #loopback #IP 位址 #網路基礎 #本機開發 #後端

前言

這篇把 localhost 跟 127.0.0.1 從 hostname 解析、loopback 的概念、127.0.0.0/8 從 RFC 790 到 RFC 1122 的歷史小故事，到 2005 年那個被 Slashdot 稱為「地表最蠢駭客」的德國 IRC bitchchecker 事件 — 攻擊 127.0.0.1 結果把自己電腦打掛 — 一次講清楚。後半段也整理本機開發遇到「localhost 拒絕連線」最常見的原因，包含 dev server 沒啟動、port 被佔用、瀏覽器 HSTS 強制 HTTPS、hosts 檔案、IPv6 ::1 解析衝突、防火牆等情境的排查方式。

如果你還對 port 這件事有點模糊，建議先讀一下 為什麼大家都用 3000？一文搞懂 Port 的用途與選擇，那一篇可以跟這一篇一起看，會比較完整。

事情是這樣的

最近又看到網路上有人在轉那篇 2005 年 IRC 上的 bitchchecker 事件 — 一位自稱駭客的德國網友嗆說要駭爆管理員，結果攻擊 127.0.0.1 攻擊到把自己電腦打掛 — 順便也翻了 IETF 的 RFC 文件，看到 127 這個數字背後其實有一段滿有趣的小故事。

這時候我想到一件事，這幾年看下來，雖然 localhost 跟 127.0.0.1 是入門開發第一週就會遇到的東西，可是真的能講清楚它們是不是同一個東西、為什麼是 127、拒絕連線到底在拒絕什麼的人其實沒有想像中的多。

所以這一篇就把概念、歷史、笑話、踩坑通通寫進來，希望看完之後你下次再遇到拒絕連線錯誤的時候，可以比較有底氣的去排查問題。

localhost 跟 127.0.0.1 到底是什麼？

簡單來講，127.0.0.1 是 IP 位址，localhost 是 hostname（主機名稱），作業系統會幫你把 localhost 翻譯成 127.0.0.1。

這兩個東西指的是同一個目的地，可是它們的身份不一樣，這個差異在某些踩坑情境下會非常重要。

localhost 是給人看的名字

localhost 不是一組 IP 位址，它是一個 hostname，意思就是這台電腦自己。當你在瀏覽器輸入 http://localhost:3000，你的作業系統會去翻一個叫做 hosts 的檔案，把 localhost 對應到一組 IP 位址。

hosts 檔案的位置如下：

系統 hosts 檔案路徑
macOS / Linux /etc/hosts
Windows C:\Windows\System32\drivers\etc\hosts

打開來看，你會看到類似這樣的內容

意思就是 localhost 這個名字會被翻譯成 127.0.0.1（IPv4）or ::1（IPv6）。所以當你瀏覽器送出 http://localhost:3000 的請求，背後其實是被解析成 http://127.0.0.1:3000 才送出去的。

127.0.0.1 是真正的 IP 位址

127.0.0.1 是一組真實存在的 IPv4 位址，它有一個專有名稱叫做 loopback address（中文叫做迴路位址）。

什麼是 loopback？簡單來講，當你送一個封包到 127.0.0.1，這個封包不會真的跑到外面的網路 or 路由器，而是直接在你電腦內部繞一圈就回來了。

如果用比較生活面的例子來講的話，假設你寄了一封信，可是這封信的收件人是你自己，那這封信根本不需要交給郵差，你直接從信箱拿出來自己看就好了。loopback 就是這個概念。

更冷門的一個事實是，並不是只有 127.0.0.1 是 loopback，整個 127.0.0.0/8 區段（也就是從 127.0.0.0 到 127.255.255.254，超過 1600 萬個位址）都是 loopback 用途，全部都會繞回自己這台電腦。只是因為 127.0.0.1 最方便、最好記，所以大家慣例上都用它。

那 IPv6 的 ::1 又是什麼？

::1 是 IPv6 版本的 loopback 位址。

IPv4 用 32 bit、IPv6 用 128 bit，IPv6 完整寫起來是 0000:0000:0000:0000:0000:0000:0000:0001，可是這樣寫太煩人，所以縮寫成 ::1。

Note
IPv6 的 loopback 只有 ::1 這一個位址，不像 IPv4 是整個 127.0.0.0/8 區段那麼大方。

這個東西看似不重要，可是在本機開發踩雷的時候它會跳出來嚇你，後面會聊到。

為什麼是 127？關於 loopback 的小故事

接下來來聊聊歷史。為什麼是 127 這個數字，而不是 1、2、100 之類的呢？

老實說，沒有特別偉大的理由，純粹是慣例。

把時間拉回 1981 年，當時 Jon Postel（網際網路歷史上最重要的工程師之一）在寫 RFC 790 這份 Assigned Numbers 文件的時候，他習慣會把 Class A IP 位址範圍裡的第一個跟最後一個網路號保留起來，當時 IPv4 Class A 的網路號從 0 到 127，於是 127 就被他先保留下來了，可是當下他並沒有寫明這個保留是要做什麼用的。

到了 1983 年，柏克萊大學在 4.2BSD 的程式碼裡面，由 Joy 跟 Leffler 把 127.0.0.1 實作成 loopback 位址（早期實驗版本他們其實還用過 254.0.0.0），這個實作被廣泛採用之後，127 就變成大家都默認的 loopback 數字。

接著 1986 年，Jon Postel 跟 Joyce Reynolds 在 RFC 990 把 loopback 這件事正式記錄下來，可是仍然只是 assignment 等級的描述而已。

最後到 1989 年的 RFC 1122，才把 127.0.0.0/8 強制定義為 internal host loopback address，並且寫明這個位址不可以出現在主機外的網路上。

整個過程簡單講就是：

年份 文件 / 事件 做了什麼
1981 RFC 790 Jon Postel 把 127 保留，沒寫用途
1983 4.2BSD 程式碼裡實作成 loopback 位址
1986 RFC 990 正式記錄 loopback 這個用法
1989 RFC 1122 強制 127.0.0.0/8 整段都是 loopback

從這張表可以看出來，loopback 是先有慣例、後來才寫進標準的典型案例。下次有人問你為什麼是 127，你就可以告訴他這個答案：Jon Postel 1981 年隨手保留了，柏克萊在 1983 年拿來用，後來大家都這樣用，最後才寫進 RFC。

那個用 127.0.0.1 把自己電腦打掛的德國駭客

接下來這個才是工程師圈真正流傳最廣的 127.0.0.1 笑話 — 重點是它真實發生過。

2005 年 4 月，德國一個嘻哈音樂網站 StopHipHop 的 IRC 頻道 #stophiphop 上，有一位 ID 叫做 bitchchecker 的使用者（中文圈轉述常常變成「某個德國小男孩」，原始 log 自稱 26 歲）在頻道裡鬧事被踢出去，於是嗆聲說要駭爆管理員 Elch 的電腦。

接下來他做的事情，後來在 Slashdot 上被冠上一個稱號 — The Planet’s Most Moronic Hacker（地表最蠢駭客）。

事件經過

bitchchecker 嗆完之後，Elch 想了一下，淡淡丟出一句

好啊，那你來駭我，我的 IP 是 127.0.0.1。

bitchchecker 信了。他打開手上號稱可以遠端刪除別人硬碟的「神兵利器」，把 127.0.0.1 輸進去，按下發射。

幾秒後，IRC 頻道上跳出第一條訊息

Elch 跟旁邊的人也很惡趣味，配合假裝慌張說「啊我的硬碟真的怪怪的！」bitchchecker 越打越得意，然後不出所料 — 又一次 Ping timeout，整場戲就這樣結束。

為什麼這個故事這麼經典？

它把 127.0.0.1 這個概念用最爆笑的方式講清楚：送到 127.0.0.1 的封包永遠會回到自己這台電腦，所以「攻擊 127.0.0.1」就是在攻擊自己，bitchchecker 每按一次發射，斷的都是他自己的網。

更荒謬的是，他不但不知道為什麼斷線，還反過來怪 Elch 的防火牆太強，最後甚至開始幫自己編戰績。整個過程把一個技術概念跟人類的執著一次演完。

所以下次有人嗆說要駭你的電腦，你可以淡淡回他一句

好啊，我的 IP 是 127.0.0.1，盡量打。

附帶一提，工程師圈也有一句改編自《綠野仙蹤》(The Wizard of Oz) 的口號 — There's no place like 127.0.0.1（沒有地方比 127.0.0.1 更像家）— 玩的同樣是「127.0.0.1 永遠回到自己」這個梗，常常被印在 T-shirt 跟貼紙上面。可是論名氣，它在 bitchchecker 事件面前還是矮了一截。

為什麼會出現「localhost 拒絕連線」？

聊完輕鬆的，來聊一下實戰的部分。

「localhost 拒絕連線」（在 Chrome/Edge 是 ERR_CONNECTION_REFUSED，Firefox 會說「無法連線」）這個錯誤背後的意思很簡單，就是你的瀏覽器送了一個封包到 127.0.0.1 的某個 port，可是那個 port 上面沒有東西在聽，或是有東西擋住了它。

底下整理幾種最常見的原因，依照踩到的機率由高到低排序。

dev server 根本沒啟動（最常見）

這個是真的最常發生，可是也最容易被忽略，特別是熬夜寫到一半閃神的時候。

假設你今天要開瀏覽器看 http://localhost:3000，那這個 port 上面要有一個東西在聽才行。檢查方式很簡單

  • 看你的終端機有沒有印出類似 Server is running at http://localhost:3000 的訊息
  • 如果沒有，代表你忘了啟動 dev server，跑一下 npm run dev or pnpm dev or yarn dev 就好

聽起來很白癡，可是這個是排查 connection refused 的第一步，先確認伺服器真的有在跑，再來查其他的。

port 寫錯 or 被佔用

第二常見的是 port 對不起來。譬如你的 dev server 跑在 5173（Vite 預設），可是你瀏覽器卻打成 3000，那當然連不到。

另一種情況是 port 被別的程式佔用了，這時候你 dev server 啟動的時候會報錯，or 它會自己跳到下一個可用的 port，所以還是要回去看終端機的訊息。

要檢查一個 port 到底是不是被佔用，可以用以下指令

如果發現被佔用，可以參考 Mac 系統 Port 號被佔用的解決方式 把佔用的程序殺掉，or 直接把 dev server 改用別的 port。

瀏覽器自動把 http 強制變成 https

這個是踩過的人都知道有多氣的雷。

事情是這樣的，你的瀏覽器有一個叫做 HSTS (HTTP Strict Transport Security) 的快取，當你某個網域曾經有跑 https，瀏覽器就會把它記下來，之後你輸入 http://... 它會自動幫你換成 https://...。

這個機制本來是為了安全（防止 downgrade 攻擊），可是放在 localhost 開發環境就很煩，因為你的本機 dev server 通常只跑 http，不跑 https，於是瀏覽器強制跳到 https 之後，自然就連不到了。

判斷方式：你看一下網址列，如果原本你輸入 http://localhost:3000，可是它變成 https://localhost:3000，那就是這個問題。

解決方法可以參考 解決 Chrome 、 Firefox 或 Edge 等瀏覽器強制跳轉 HTTPS 網址問題 把 HSTS 紀錄清掉。

hosts 檔案被改過 or IPv4 / IPv6 解析衝突

前面有提到 localhost 是透過 hosts 檔案解析成 127.0.0.1 or ::1，那如果這個對應壞了，連線當然也會壞。

最常見的兩種情況：

  • hosts 檔案被某些軟體（譬如某些防毒 or 上網管控工具）改寫，把 localhost 對應到別的 IP
  • localhost 解析到 IPv6 ::1，可是你的 dev server 只綁在 IPv4 127.0.0.1 上面（or 反過來）

第二種情況在 Node.js 早期版本特別常見，那時候 Node.js 17 之後 DNS 解析的預設行為改變，導致 localhost 開始優先解析到 ::1，很多人因此發現原本可以跑的程式突然連不上資料庫。

解法很簡單，把網址改成 http://127.0.0.1:port，跳過 hostname 解析，直接走 IPv4，通常就會通了。

防火牆 or 防毒軟體擋掉 loopback

這個比較少見，可是在公司電腦 or 政府單位電腦上偶爾會發生。

某些比較嚴格的防火牆 or 防毒軟體會把 loopback 流量也列入管控，這時候你的封包送出去會被擋掉，連線就被拒絕了。如果你前面幾項都試過了還是不通，可以試著暫時關掉防火牆 or 把 dev server 加到白名單看看。

特殊情境：自然人憑證之類的本機服務

有一種特殊情境是某些本機服務（譬如台灣的自然人憑證 HiCOS）會在 localhost:61161 這種特定 port 上跑一個小型伺服器，當這個服務的背景程序沒啟動，網頁就會出現「localhost 拒絕連線」。

這個案例我之前有寫過一篇 Windows 自然人憑證網頁總是出現「localhost 拒絕連線」的解決方式，如果你剛好在弄政府網站遇到這個錯誤，可以直接過去看。

結語

這一篇從 localhost 跟 127.0.0.1 的差異、loopback 的歷史、那個經典的工程師笑話，一路講到本機開發踩雷的常見原因，希望可以讓你之後看到「localhost 拒絕連線」這幾個字的時候，不會再像第一次遇到那樣手足無措。

整理一下重點：

  • localhost 是 hostname、127.0.0.1 是 IP，前者透過 hosts 檔案解析成後者
  • 127.0.0.0/8 整段都是 loopback，源自 Jon Postel 1981 年的隨手保留
  • 攻擊 127.0.0.1 等於攻擊自己 — 2005 年 bitchchecker 事件就是最爆笑的證明
  • 連不上的時候優先檢查：dev server 有沒有啟動、port 對不對、HSTS 有沒有把你跳到 https、要不要改用 127.0.0.1 跳過解析

最後送上一句工程師最忠實的好朋友 — 127.0.0.1，永遠不會背叛你（除了它沒在跑的時候），就算你想拿它來嗆人，它也會幫你打你自己（誤）。
