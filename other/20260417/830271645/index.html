用 Cloudflare Workers 快取 Firebase Storage 圖片，完全免費降低流量費用

前言

如果你跟我一樣使用 Firebase Storage 來存放部落格的圖片，那你應該多少會有一個煩惱，那就是流量費用。每當有人造訪你的文章，圖片就會從 Firebase Storage 下載一次，這些流量全部都算在你的 Firebase 帳單上，文章越多、讀者越多，流量費用就越可觀。

那有沒有辦法在中間加一層快取，讓圖片不用每次都從 Firebase Storage 下載呢？答案是可以的，而且完全免費，所以這篇文章就來聊一下如何使用 Cloudflare Workers 搭配自訂子域名，來快取 Firebase Storage 的圖片，讓 Cloudflare 的全球 CDN 幫你擋住大部分的流量。

原理是什麼？

在進入實作之前，我們先來了解一下原理，否則你怎麼會知道為什麼這樣做有用呢？

簡單來講，我們要在 Firebase Storage 前面擋一層 Cloudflare，讓 Cloudflare 當一個「中間人」來幫我們快取圖片：

txt
使用者 → images.yourdomain.com → Cloudflare CDN（快取層）→ Firebase Storage
                                      ↑
                                有快取就直接回傳，不打 Firebase

第一次有人請求圖片時，Cloudflare 會去 Firebase Storage 拿圖片，然後存一份在自己的邊緣節點上。之後再有人請求同一張圖片，Cloudflare 就直接從邊緣節點回傳，完全不會打到 Firebase，也就是說 Firebase 那邊完全不會有流量產生。

這真的免費嗎？

在開始之前，我想先把費用的部分說清楚，畢竟大家最在意的應該就是這個吧？

項目費用
Cloudflare 帳號免費
Cloudflare DNS 託管免費
Cloudflare Workers免費（每天 10 萬次請求，不需信用卡）
Cloudflare CDN 快取流量免費（無流量上限）
Worker 自訂域名免費方案即支援

以上全部免費，你沒看錯。Cloudflare 是少數不按流量收費的 CDN 服務商，甚至有人在免費方案下每月跑 5-9 TB 的流量都沒有問題。

而且快取命中後，請求不會經過 Worker，所以也不會消耗每天 10 萬次的免費額度，對部落格圖片來講基本上完全夠用。

Note 唯一的前提是你需要有一個自己的域名，並且把 DNS 託管到 Cloudflare 上（託管本身也是免費的）。

小心不要踩到收費的坑

雖然 Workers 免費方案不需要信用卡，但 Cloudflare 後台有一些進階功能（如 Durable Objects、Workers AI）可能會引導你升級到 $5/月的付費方案，千萬不要點任何「升級」按鈕，我們只需要最基本的 Worker 功能就夠了，不需要用到那些進階功能。

為什麼用子域名？

你可能會想：「為什麼不直接用 yourdomain.com/images/... 就好？」

原因很簡單，因為如果你的主域名底下已經有部落格或其他專案，路徑很容易衝突。用子域名（例如 images.yourdomain.com）完全獨立，不會影響到主站的任何東西，而且最終圖片網址會像這樣：

txt
https://images.yourdomain.com/blog2/ai/my-post/screenshot.png

比起原本那一長串的 Firebase Storage URL 來說，乾淨又好管理多了。

開始實作

認識完原理跟費用之後，接下來就進入實作的部分吧！

步驟一：把域名 DNS 託管到 Cloudflare

如果你的域名還沒在 Cloudflare 上管理，請先完成以下步驟：

  1. 註冊 Cloudflare 帳號（免費）
  2. 點 Add a site → 輸入你的域名
  3. 選擇 Free 方案
  4. Cloudflare 會給你兩個 nameserver
  5. 到你的域名註冊商把 nameserver 改成 Cloudflare 給的
  6. 回到 Cloudflare 點確認，等待 DNS 生效（通常幾分鐘到幾小時）

Note 如果你的域名已經在 Cloudflare 上了，可以直接跳到步驟二。

步驟二：建立 Cloudflare Worker

  1. 進入 Cloudflare Dashboard → 左側選 Workers & Pages
  2. 點 Create → Create Worker
  3. 選擇 「從 Hello World 開始!」
  4. 取個名字，例如 image-proxy
  5. 點 Deploy 先建立起來，然後點 Edit code
  6. 把預設的程式碼全部刪掉，貼上以下內容：
js
export default {
  async fetch(request) {
    // 限制允許的來源網域
    const allowedOrigins = [
      'https://yourdomain.com',
      'https://www.yourdomain.com',
      'http://localhost', // 本地開發用
    ];

    const referer = request.headers.get('Referer') || '';
    const origin = request.headers.get('Origin') || '';

    const isAllowed =
      (!referer && !origin) || // 允許直接在瀏覽器網址列存取
      allowedOrigins.some((o) => referer.startsWith(o) || origin.startsWith(o));

    if (!isAllowed) {
      return new Response('Forbidden', { status: 403 });
    }

    // 將路徑轉換成 Firebase Storage 的 URL
    const url = new URL(request.url);
    const path = encodeURIComponent(url.pathname.slice(1));
    const firebaseUrl = `https://firebasestorage.googleapis.com/v0/b/YOUR_PROJECT.appspot.com/o/${path}?alt=media`;

    const response = await fetch(firebaseUrl);

    if (!response.ok) {
      return new Response('Not Found', { status: 404 });
    }

    // 設定快取 header
    const headers = new Headers(response.headers);
    headers.set('Cache-Control', 'public, max-age=31536000, immutable');
    headers.set('CDN-Cache-Control', 'max-age=31536000');
    headers.set('Access-Control-Allow-Origin', '*');

    return new Response(response.body, {
      status: response.status,
      headers,
    });
  },
};

Note 請記得把 YOUR_PROJECT.appspot.com 替換成你自己的 Firebase 專案名稱，以及把 allowedOrigins 改成你自己的域名。

  1. 點 Save and Deploy

這段程式碼做了什麼？

看完程式碼之後，你可能會想這段程式碼到底做了什麼事情？其實很簡單，主要做了三件事：

  1. 網域限制：檢查請求的來源，只允許你自己的網域和本地開發環境存取，防止別人盜用你的圖片代理
  2. 反向代理：把請求的路徑轉換成 Firebase Storage 的 URL，然後去 Firebase 拿圖片回來
  3. 設定快取：在回應的 header 中加上快取控制，告訴 Cloudflare CDN 和瀏覽器可以快取這張圖片一年

其中 localhost 的部分是為了本地開發時也能正常顯示圖片，因為我們是用 startsWith 來比對，所以不管你用哪個 port（4000、3000、8080...）都能正常運作，不用擔心本地開發時圖片會顯示不出來。

步驟三：綁定子域名

  1. 在 Worker 頁面 → 點 Settings → Domains & Routes
  2. 點 Add → Custom Domain
  3. 輸入 images.yourdomain.com
  4. Cloudflare 會自動幫你建立 DNS 記錄，點確認即可

步驟四：設定 Firebase Storage Rules

因為我們的 Worker 不會帶 token 去請求 Firebase Storage，所以需要把圖片設為公開可讀。

到 Firebase Console → Storage → Rules，加入以下規則：

txt
rules_version = '2';
service firebase.storage {
  match /b/{bucket}/o {
    match /blog2/{allPaths=**} {
      allow read;
    }
  }
}

這邊只開放了 blog2/ 這個路徑的公開讀取權限，其他路徑完全不受影響。

你可能會擔心安全性的問題，但換個角度想，這些圖片本來就是公開的部落格文章在使用的，任何人都看得到。而且原本的 token 也是直接寫在網頁的 HTML 裡面，任何人打開開發者工具都能拿到，所以把圖片路徑改成公開讀取，實際上並沒有增加額外的安全風險。

步驟五：驗證

打開瀏覽器，輸入以下格式的網址：

txt
https://images.yourdomain.com/blog2/your-folder/your-image.png

如果看到圖片就代表設定成功了！

更新圖片連結

設定完成後，接下來就是要把部落格中所有的 Firebase Storage 圖片連結替換成新的格式了。

原本的連結長這樣：

txt
https://firebasestorage.googleapis.com/v0/b/YOUR_PROJECT.appspot.com/o/blog2%2Ffolder%2Fimage.png?alt=media&token=xxx

替換成：

txt
https://images.yourdomain.com/blog2/folder/image.png

規則很簡單：

  1. 把 Firebase 的域名換成 images.yourdomain.com
  2. 去掉 ?alt=media&token=xxx
  3. 把路徑中的 %2F 替換成 /

如果你的文章很多，手動改不切實際，可以用 sed 批次替換：

bash
# 第一步：替換域名並移除 token
grep -rl "firebasestorage\.googleapis\.com/v0/b/YOUR_PROJECT\.appspot\.com" source/ | \
while IFS= read -r f; do
  sed -i '' 's|https://firebasestorage\.googleapis\.com/v0/b/YOUR_PROJECT\.appspot\.com/o/\([^?]*\)?alt=media&token=[a-f0-9\-]*|https://images.yourdomain.com/\1|g' "$f"
done

# 第二步：把 %2F 替換成 /
for i in $(seq 1 5); do
  grep -rl "images\.yourdomain\.com/[^)]*%2F" source/ 2>/dev/null | \
  while IFS= read -r f; do
    sed -i '' 's|\(images\.yourdomain\.com/[^)]*\)%2F|\1/|g' "$f"
  done
done

Note 第二步需要跑多次，因為 sed 一次只會替換一個 %2F，跑迴圈可以確保所有的 %2F 都被替換乾淨。另外如果你有多個 Firebase 專案的圖片，替換時要注意只替換目標專案的連結，避免誤改到其他專案的圖片 URL。

注意：URL 路徑要用 / 而不是 %2F

這邊有一個很重要的眉角要提醒一下。

Firebase Storage 的 URL 格式是用 %2F 來分隔路徑的（因為整個檔案路徑被當作一個 URL 區段），但在我們自己的子域名下，必須使用正常的 /：

txt
❌ https://images.yourdomain.com/blog2%2Fai%2Fmy-post%2Fimage.png
✅ https://images.yourdomain.com/blog2/ai/my-post/image.png

為什麼呢？因為 Cloudflare 在處理 %2F 的時候會出現問題導致 404，而 Worker 內部會自動幫你把 / 轉換成 Firebase 需要的 %2F 格式（透過 encodeURIComponent），所以你只需要用正常的路徑就好，不用自己去處理編碼的問題。

上傳新圖片後怎麼取得 URL？

那麼以後上傳新圖片的時候該怎麼取得 URL 呢？其實非常簡單，規則就是：

txt
https://images.yourdomain.com/ + Firebase Storage 裡的路徑

舉例來說，你上傳了一張圖到 Firebase Storage 的 blog2/ai/my-post/screenshot.png，那麼 URL 就是：

txt
https://images.yourdomain.com/blog2/ai/my-post/screenshot.png

不需要再去拿什麼 token 了，直接拼路徑就好，是不是方便很多？

快取的運作機制

最後來聊聊快取到底是怎麼運作的，讓你對這個方案更有信心。

我們在 Worker 中設定了兩個關鍵的 header：

js
headers.set('Cache-Control', 'public, max-age=31536000, immutable');
headers.set('CDN-Cache-Control', 'max-age=31536000');

這其實建立了兩層快取：

第一層：Cloudflare CDN 邊緣節點快取

CDN-Cache-Control 會告訴 Cloudflare 的邊緣節點：「這個檔案可以快取一年。」當第一個使用者請求圖片時，Cloudflare 會去 Firebase 拿圖片，然後存一份在離使用者最近的邊緣節點上。之後不管是誰再請求同一張圖片，Cloudflare 都直接從邊緣節點回傳，不經過 Worker，也不打 Firebase。

第二層：瀏覽器本地快取

Cache-Control: public, max-age=31536000, immutable 則是告訴瀏覽器：「這個檔案可以存在本地一年，而且不會變。」所以同一個使用者再次造訪同一篇文章時，瀏覽器會直接用本地的快取，連 Cloudflare 都不用打。

整體流程

第一次請求（冷快取）：

txt
使用者 → Cloudflare 邊緣節點（沒快取）→ Worker → Firebase Storage
                ↓
        存一份在邊緣節點 + 回傳給使用者

第二次請求（CDN 快取命中）：

txt
使用者 → Cloudflare 邊緣節點（有快取）→ 直接回傳

同一使用者再次造訪（瀏覽器快取命中）：

txt
使用者 → 瀏覽器本地快取 → 直接顯示

因為部落格的圖片上傳後幾乎不會修改，所以設定一年的快取時間加上 immutable 是完全合理的。

結論

透過 Cloudflare Workers 搭配 CDN 快取，我們可以在不花一毛錢的情況下，大幅降低 Firebase Storage 的流量消耗，整個設定過程大概也不需要花太多時間，但帶來的效益卻是長期的。

最後整理一下這個方案的優點：

  • 完全免費：不需要信用卡，不需要付費方案
  • 全球加速：Cloudflare 在全球有超過 300 個邊緣節點，圖片從離使用者最近的節點回傳
  • 減少流量：快取命中後完全不打 Firebase，流量歸零
  • 網域限制：可以限制只有你的網站能存取，防止盜連
  • 更短的 URL：從一長串的 Firebase URL 變成乾淨的子域名格式

如果你也有在用 Firebase Storage 來存放圖片，非常推薦試試看這個方案哩～
