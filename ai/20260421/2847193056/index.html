從 3 分到滿分！讓你的網站通過 isitagentready AI Agent Readiness 完整攻略

前言

如果你跟我一樣有在經營技術部落格，那你應該對 SEO 不陌生，之前我也寫過一篇關於 AEO（答案引擎優化）的文章，聊的是怎麼讓 AI 搜尋引擎引用你的內容。不過最近我在社群上看到一個更進一步的東西，就是 Cloudflare 推出的 isitagentready.com，它可以掃描你的網站，告訴你：「你的網站準備好讓 AI Agent 來拜訪了嗎？」

抱著好奇心我就把自己的部落格丟進去掃看看，結果一掃完差點沒從椅子上摔下來，12 個項目只過了 3 個，分數只有慘不忍睹的 3 分（滿分 100），Level 也只有最低等級。

所以這篇就來完整分享我從 3 分修到 100 分滿分的過程，如果你也想試試看的話，可以跟著做。

什麼是 Agent Readiness？為什麼要在意？

在開始修之前，先來聊一下這到底是什麼東西。

你有沒有想過，當 ChatGPT、Claude、Perplexity 這些 AI 工具要存取你的網站的時候，它們是怎麼「看懂」你的網站的？它們不像人類會用眼睛看畫面，它們需要靠一些標準化的方式來知道你的網站有什麼內容、能做什麼事情。

舉個例子好了，就像你開一間店，人類顧客走進來可以自己看招牌、看菜單，可是如果今天來的是一個「AI 顧客」，它需要你在門口掛一個機器可以讀取的 QR Code 或是標準化的標籤，它才知道你這間店賣什麼、要怎麼點餐。

Agent Readiness 講的就是這件事：你的網站有沒有把這些「給 AI 看的門牌」掛好。

isitagentready.com 在測什麼？

這個工具會從四個面向、共 12 個項目來幫你的網站打分數。我先用白話文快速講一下每個項目在幹嘛：

第一類：Discoverability（AI 找不找得到你）3 項

這類在測的是 AI 能不能「發現」你的網站，就像你的店有沒有掛招牌一樣：

  • robots.txt：一個告訴爬蟲「這個網站哪些地方可以看、哪些不能看」的檔案
  • Sitemap：你的網站地圖，告訴 AI 你總共有哪些頁面
  • Link headers：在 HTTP 回應裡面放一些「指路牌」，告訴 AI 去哪裡可以找到更多資訊（像是 API 目錄、RSS 等等）

第二類：Content（AI 看不看得懂你的內容）1 項

  • Markdown Negotiation：當 AI 跟你的網站說「欸，你可以給我 Markdown 格式嗎？」的時候，你的網站能不能回應 Markdown，而不是一堆 HTML 標籤

為什麼 AI 比較想要 Markdown？因為 Markdown 乾淨、沒有一堆 CSS class 跟 div 標籤，AI 處理起來更快、更省 Token。

第三類：Bot Access Control（爬蟲的存取規則）2 項

  • AI bot rules：你的 robots.txt 裡面有沒有針對 AI 爬蟲（像是 GPTBot、ClaudeBot）設定專屬規則
  • Content Signals：你有沒有明確告訴 AI「我的內容可以拿去搜尋，但不能拿去訓練模型」

第四類：Protocol Discovery（協定發現）6 項

這一類比較偏向「進階」的設定，主要是在你的網站上放一些標準化的 JSON 檔案，讓 AI Agent 知道你的網站支援哪些互動方式：

  • API Catalog：公開你的 API 目錄
  • OAuth/OIDC：認證方式的說明
  • OAuth Protected Resource：受保護資源的說明
  • MCP Server Card：MCP 伺服器的基本資訊
  • Agent Skills index：你的網站提供哪些「技能」給 AI 使用
  • WebMCP：透過 JavaScript 直接在網頁上暴露工具給 AI

老實說這 6 項對純部落格來說實際意義不大，比較適合有提供 API 服務的網站。但既然都要做了，就做到滿分吧。

我的起始狀態：只有 3 分

一開始掃描的結果是這樣的：

  • ✅ robots.txt（本來就有）
  • ✅ Sitemap（Hexo 自動產生的）
  • ✅ AI bot rules（之前寫 AEO 那篇的時候就設定過了）
  • ❌ 其他 9 項全部沒過

那接下來就一項一項來修。

修正第一項：Content Signals

這是最簡單的，只要在 robots.txt 裡面加上一行就好。這行的作用是告訴 AI：「我的內容歡迎你拿去搜尋，但不要拿去訓練模型。」

txt
Content-Signal: ai-train=no, search=yes, ai-input=no

三個值的意思：

  • ai-train=no：不要拿我的內容去訓練 AI
  • search=yes：可以拿去做 AI 搜尋索引
  • ai-input=no：不要直接把我的內容當作 AI 的輸入素材

如果你的 robots.txt 是手動維護的，直接加在最後面就好。像我用 Hexo 的話，robots.txt 是透過腳本自動產生的，所以要去改產生 robots.txt 的那個腳本。

修正第二項：Link Response Headers

這一項測的是你的首頁在 HTTP 回應裡面有沒有帶 Link header。

簡單解釋一下什麼是 HTTP header：每次瀏覽器（或 AI Agent）跟你的網站要資料的時候，你的網站除了回傳 HTML 內容之外，還會附帶一些「額外資訊」，這些就是 header。Link header 就是其中一種，用來告訴對方「嘿，這邊還有一些相關資源你可以去看看」。

要注意的是，這邊要的是 HTTP header，不是 HTML 裡面的 <link> 標籤，兩個是不同的東西。

如果你的網站有走 Cloudflare（Domain 的 DNS 有開啟代理，也就是橘色雲朵），那可以用 Cloudflare 的 Transform Rule 來加：

  1. 到 Cloudflare Dashboard，選你的 domain
  2. 進 規則 > 點右上角 建立規則 > 選 回應標頭轉換規則
  3. 規則名稱填 Agent Discovery Link Headers
  4. 條件設定：欄位選「URI 路徑」，運算子選「等於」，值填 /
  5. 往下滑到「則...」的區塊，選取項目選「新增靜態」
  6. 標頭名稱填 Link，值填你要指向的資源（像是 API 目錄、Sitemap、RSS 等等）

這樣設定完之後，每次有人（或 AI）存取你的首頁，Cloudflare 就會自動在回應裡面加上 Link header。

修正第三項：Markdown Negotiation

這一項要讓你的網站支援「內容協商」，也就是當 AI Agent 在請求你的頁面時帶上 Accept: text/markdown 這個 header，你的網站要能回傳 Markdown 格式的內容，而不是一般的 HTML。

這邊的做法是用 Cloudflare Worker 來處理。Worker 你可以想像成是一個「中間人」，所有經過 Cloudflare 的請求都會先過 Worker 一手，Worker 可以看看這個請求是不是 AI Agent 發的（有沒有帶 Accept: text/markdown），如果是的話就把 HTML 轉成 Markdown 再回傳。

這個 Worker 後面會跟 .well-known 的解法一起講，因為它們共用同一個 Worker。

修正第四項：WebMCP

WebMCP 是一個比較新的標準，讓你的網頁可以透過 JavaScript 告訴 AI Agent：「嘿，我這個網站有這些功能你可以用。」

實作方式很簡單，就是在你的網頁裡面加一段 JavaScript，呼叫 navigator.modelContext.provideContext() 來註冊「工具」。

以部落格為例，我註冊了兩個工具，一個是搜尋文章，一個是取得網站基本資訊：

html
<script>
if (navigator.modelContext) {
  navigator.modelContext.provideContext({
    tools: [
      {
        name: "search-articles",
        description: "搜尋 isRayNotArray 部落格的技術文章",
        inputSchema: {
          type: "object",
          properties: {
            query: { type: "string", description: "搜尋關鍵字" }
          },
          required: ["query"]
        },
        execute: async ({ query }) => {
          window.location.href = `/?s=${encodeURIComponent(query)}`;
          return { status: "redirecting", query };
        }
      },
      {
        name: "get-site-info",
        description: "取得 isRayNotArray 部落格的基本資訊",
        inputSchema: { type: "object", properties: {} },
        execute: async () => ({
          name: "isRayNotArray",
          author: "Ray",
          url: "https://israynotarray.com",
          feed: "https://israynotarray.com/atom.xml",
          sitemap: "https://israynotarray.com/sitemap.xml"
        })
      }
    ]
  });
}
</script>

把這段放在你網站的 HTML template 裡面就好，要確保它在頁面載入時就會執行，因為 scanner 是在頁面載入的時候去偵測有沒有 WebMCP 工具的。

修正第五項：.well-known 系列

接下來是最後五個項目，它們都需要在網站的 /.well-known/ 路徑下放置 JSON 檔案。/.well-known/ 是一個網路標準的公開資訊目錄，讓 AI Agent 可以透過固定路徑找到你的網站提供了哪些服務跟能力。

需要放的檔案有：

  • /.well-known/api-catalog - API 目錄
  • /.well-known/openid-configuration - 認證資訊
  • /.well-known/oauth-protected-resource - 受保護資源資訊
  • /.well-known/mcp/server-card.json - MCP 伺服器資訊
  • /.well-known/agent-skills/index.json - Agent Skills 索引

你可能會想說：「我只是一個靜態部落格，又沒有 API 也沒有 OAuth，為什麼要放這些？」，確實這些項目本來是設計給有後端服務的網站用的，對純靜態部落格來說實際上不會有 AI Agent 真的來呼叫你的 OAuth endpoint。但 scanner 檢查的是「這些檔案存不存在、格式對不對」，所以如果想拿滿分的話，放上去就對了。

如果你是部署在 GitHub Pages 上的話要注意，GitHub Pages 沒辦法正常服務 .well-known 目錄下的檔案，即使放了 .nojekyll 也一樣，所以這邊一樣交給 Cloudflare Worker 來處理。

前面 Markdown Negotiation 已經需要 Worker 了，順便把 .well-known 的回應也寫進去就好。做法是在 Worker 裡面建一個對照表，收到 .well-known 路徑的請求時直接回傳對應的 JSON：

javascript
const WELL_KNOWN_FILES = {
  '/.well-known/api-catalog': {
    content: JSON.stringify({
      linkset: [{
        anchor: 'https://你的網域/',
        'service-doc': [{ href: 'https://你的網域/sitemap.xml', type: 'application/xml' }]
      }]
    }),
    type: 'application/linkset+json',
  },
  '/.well-known/mcp/server-card.json': {
    content: JSON.stringify({
      serverInfo: { name: '你的網站名稱', version: '1.0.0' },
      transport: { type: 'streamable-http', endpoint: '/mcp' },
      capabilities: { tools: true, resources: true, prompts: false },
    }),
    type: 'application/json',
  },
  // ... 其他檔案也是同樣的格式
};

export default {
  async fetch(request) {
    const url = new URL(request.url);

    // 如果是 .well-known 的路徑，就直接回傳 JSON
    const wellKnown = WELL_KNOWN_FILES[url.pathname];
    if (wellKnown) {
      return new Response(wellKnown.content, {
        status: 200,
        headers: {
          'Content-Type': wellKnown.type,
          'Access-Control-Allow-Origin': '*',
        },
      });
    }

    // 如果 AI 要求 Markdown 格式，就把 HTML 轉成 Markdown
    const accept = request.headers.get('Accept') || '';
    const response = await fetch(request);
    if (accept.includes('text/markdown') && response.headers.get('Content-Type')?.includes('text/html')) {
      const html = await response.text();
      const markdown = htmlToMarkdown(html);
      return new Response(markdown, {
        status: response.status,
        headers: { 'Content-Type': 'text/markdown; charset=utf-8' },
      });
    }

    // 其他情況就正常回傳
    return response;
  },
};

這個 Worker 同時解決了兩個問題：.well-known 的檔案服務跟 Markdown 內容協商。

Worker 部署步驟

  1. 到 Cloudflare Dashboard，左邊選 Workers 和 Pages
  2. 點 建立 > 建立 Worker
  3. 名稱隨便取，像是 agent-readiness
  4. 點「部署」先用預設程式碼部署上去
  5. 部署完之後點「編輯程式碼」，把你的 Worker 腳本貼上去
  6. 點右上角「部署」

部署完之後還要設定路由，告訴 Cloudflare 哪些請求要經過這個 Worker：

  1. 回到你的 domain 的管理頁面
  2. 左邊選 Workers 路由
  3. 新增路由，路由填 yourdomain.com/*，Worker 選你剛剛建的那個
  4. 儲存

另外你可以把 Worker 的 workers.dev 公開網址關掉，因為它只需要透過路由綁定就好，不需要有自己的公開入口。

Note 你可能會擔心 Cloudflare Worker 免費版一天只有 10 萬次請求會不會不夠用，但以一般技術部落格來說，一天大概幾百到幾千次瀏覽量，連免費額度的 1% 都用不到，完全不用擔心。

最終結果：100 分滿分

全部修完之後重新掃描，結果：

  • Discoverability：100（3/3）
  • Content：100（1/1）
  • Bot Access Control：100（2/2）
  • API, Auth, MCP & Skill Discovery：100（6/6）
  • 總分：100 分，Level 5 Agent-Native 🎉

整理一下最終每項的解法：

項目怎麼解的
robots.txt本來就有
Sitemap本來就有

補充注意事項

  1. GitHub Pages 不支援 .well-known 目錄：即使放了 .nojekyll 也沒用，需要用 Cloudflare Worker 來服務這些檔案
  2. Markdown Negotiation 需要 Worker：靜態網站本身沒辦法做內容協商，需要靠 Cloudflare Worker 來處理

結論

整個修下來其實不複雜，大部分項目都是加個檔案或加個設定就搞定了，最花時間的反而是在踩坑跟除錯。

不過說老實話，這個評測工具有些項目（像是 OAuth）對純部落格來說真的沒什麼實際意義，單純就是為了拿分，但 Discoverability 跟 Content 那幾項倒是蠻值得做的。畢竟 AI Agent 的生態只會越來越成熟，提早把這些基礎建設做好，你的內容被 AI 正確存取跟引用的機會也會更高哩～
