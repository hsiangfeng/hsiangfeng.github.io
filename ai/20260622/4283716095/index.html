讓 /llms.txt 回傳 text/markdown：Cloudflare 回應標頭轉換規則教學

前言

之前我有寫過一篇 SEO 已經不夠了嗎？認識 AEO 答案引擎優化 跟 從 3 分到滿分！讓你的網站通過 isitagentready，裡面有提到 /llms.txt 是用來告訴 AI 搜尋引擎你的網站有哪些東西的索引檔，然後這邊有個關鍵是你必須要把它的 Content-Type 標成 text/markdown。

那這邊我自己也乖乖把 /llms.txt 跟 /llms-full.txt 丟到 gh-pages 上，可是如果你這時候順便 curl -I 一下你會發現回傳的是...

txt
HTTP/2 200
content-type: text/plain

所以這一篇就是要來分享我怎麼用 Cloudflare 的回應標頭轉換規則，把 content-type: text/plain 改成 text/markdown; charset=utf-8，而這整個過程不用寫 Cloudflare Worker，免費方案就夠用哩～

為什麼 /llms.txt 會跑成 text/plain？

這個問題其實很簡單，因為它的副檔名是 .txt（對，就是這樣而已）。

每個 web server 在回應靜態檔的時候，會根據副檔名查一張對應表（叫做 MIME 對應表）來決定 Content-Type 要寫什麼。GitHub Pages 用的是業界很通用的 mime-db 那一套，對應規則大概長這樣：

副檔名對應的 Content-Type
.txttext/plain
.mdtext/markdown
.htmltext/html
.xmlapplication/xml

所以 .txt 結尾的檔案就會被標注成 text/plain，即使你裡面寫的是 markdown 格式，web server 也不會幫你看內容，它只認副檔名。

這時候應該又有一個疑問了：text/plain AI 還是讀得到啊，反正內容是 markdown，給它讀就好不是嗎？

對，AI 確實還是讀得到，可是這邊有兩個小問題：

  1. 少了明確的格式信號： AI 拿到 text/plain 之後會把它當純文字處理，不一定會啟動 markdown 的解析邏輯，這導致標題、列表、連結這些結構就比較容易被當成一般文字看待，那就沒辦法發揮 markdown 格式的優勢了。
  2. 有些工具會看 Content-Type 決定要不要拿： 像 Anthropic、OpenAI 的文件裡都有提到，建議站點在提供 markdown 內容時把 Content-Type 標成 text/markdown，這樣 AI Agent 看到的時候會更明確知道這是要給它看的 markdown 內容，會比較友善地把它當成 markdown 來處理。

所以 Content-Type 的調整就變得格外重要。

為什麼選 Cloudflare 回應標頭轉換規則？

我認為對「就只是想改 Content-Type」這種相對單純的需求來講，用 Cloudflare 的回應標頭轉換規則最簡單，更不用說我自己的網域本來就是託管在 Cloudflare 上（畢竟 gh-pages 本身也沒有地方讓你動 header）。

所以接下來，就來介紹一下該如何設定吧。

設定步驟

整個流程非常簡單，只需要幾分鐘就可以設定完了。

Step 1：進到轉換規則頁面

  1. 登入 Cloudflare Dashboard
  2. 選你要設定的網域（這邊以我的部落格 israynotarray.com 為例）
  3. 左側選單點 規則 → 轉換規則
  4. 找到 回應標頭轉換規則 那一欄，點右上角 建立規則

轉換規則頁面

這邊選擇「回應標頭轉換規則」

建立規則

Step 2：填規則名稱

規則名稱我這邊取 Set markdown content-type，方便我之後一眼看出來這個規則在幹嘛。

你也可以叫 llms.txt markdown header 之類的，反正隨你喜歡。

填規則名稱

Step 3：寫比對條件

接下來這邊稍微有一點坑 XD

所以我個人建議直接使用 編輯運算式 模式比較好，然後把整段貼進去：

txt
(http.request.uri.path eq "/llms.txt") or (http.request.uri.path eq "/llms-full.txt") or (ends_with(http.request.uri.path, ".md"))

貼上比對運算式

這段運算式做了三件事情：

  • 抓 /llms.txt
  • 抓 /llms-full.txt
  • 抓所有 .md 結尾的檔案（如果你有開「每篇文章對應一份 .md」這種設計的話）

所以你的畫面按完成之後應該會變成這樣：

完成後的運算式預覽

Step 4：設定要塞的 header

接下來是最重要的一個步驟，請你往下到 則... 區塊，點 設定新標頭 按鈕，會展開三個欄位讓你填：

  • 動作： 選「設定靜態」（Set static）
  • 標頭名稱： 填 Content-Type
  • 值： 填 text/markdown; charset=utf-8

charset=utf-8 這段建議加上去，不然中文內容在某些 client 上可能會出現編碼錯亂。

Content-Type

放置於 保持預設的「最後」就好，按右下角 部署 就完成啦～

Step 5：用 curl 驗證

部署完之後打開 terminal，跑一下 curl -I 看回應的 header：

bash
curl -I https://你的網域/llms.txt

預期看到的回應大概長這樣：

txt
HTTP/2 200
content-type: text/markdown; charset=utf-8
server: cloudflare

curl

content-type 變成 text/markdown; charset=utf-8 就代表規則生效了。

Note 如果你看到 content-type 還是 text/plain，可能是 Cloudflare 邊緣快取還沒更新。可以加 -H "Cache-Control: no-cache" 重抓一次，或者直接去 Cloudflare Dashboard 的快取頁面點「清除快取」一下。

結語

最後總結一些重點吧。

  1. 如果部署在 gh-pages 上，.txt 結尾的檔案預設會被標成 text/plain，這時候 AI 爬蟲就不太會把它當 markdown 來處理了。
  2. 用 Cloudflare 的回應標頭轉換規則可以很簡單地把 /llms.txt、/llms-full.txt 和所有 .md 檔的 Content-Type 改成 text/markdown; charset=utf-8。
  3. 設定的時候要注意比對條件的寫法，建議直接用編輯運算式的方式，這樣比較不會踩坑（要選「URI 路徑」不是「完整 URI」）。
  4. 設定完之後可以用 curl -I 來驗證 header 是否有正確被改寫。

這條規則做完之後，下次 AI 爬蟲來抓 /llms.txt 就會拿到明確的 markdown 格式信號。

那如果你想做更進階的 Markdown Negotiation（也就是 AI 帶 Accept: text/markdown header 時，把整篇 HTML 文章即時轉成 markdown 回給它），那就要動到 Cloudflare Worker 了，這部分我之前在 從 3 分到滿分！讓你的網站通過 isitagentready 那篇有完整的 Worker 寫法可以參考。

到這邊整個 AEO 的最後一哩路就接起來了。希望這一篇有讓你對 Cloudflare 轉換規則有更近一步的認識哩～
