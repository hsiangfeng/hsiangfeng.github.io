想開始 Vibe Coding？新手推薦先搞懂的這些基礎知識

#Node.js #Git #GitHub #AI #前端框架 #VSCode #Vibe Coding #Terminal #UI Framework #Prompt #新手入門

前言

最近滿多朋友都跑來問我說「我也想用 AI 開發點東西，要從哪裡開始學？」，而這背後其實就是大家最近常聽到的 Vibe Coding。這一篇就來聊一下，新手如果想要踏入 Vibe Coding 的世界，到底有哪些基礎知識是推薦先搞懂的，例如 Git、命令列、UI Framework 等等，免得你做到一半才發現自己卡關卡到不知道該怎麼跟 AI 溝通。

等等，Vibe Coding 不是說好不用懂技術嗎？

在開始講基礎知識之前，先來聊一下這個迷思。

如果你還沒看過我之前寫的 淺談 AI Vibe Coding 當今主流的 BDD 與 SDD，可以先去看一下，裡面有比較完整地介紹 Vibe Coding 是什麼，以及當今主流的兩種開發框架。

簡單來講，Vibe Coding 就是『直接用自然語言來描述功能，由 AI 幫你生成程式碼』的開發方式，重點在於你不用親手寫程式，只要把你想要的東西描述給 AI 聽就好。

聽起來很美好對吧？

可是這邊就要稍微潑一下冷水了，因為 Vibe Coding 不代表「完全不用懂任何技術」。

舉個我自己看過的真實例子，朋友用 AI 做了一個小網站，東西做出來了也很開心，可是某天 AI 改著改著突然把整個首頁弄壞了，而朋友就只能整個刪掉重做，因為他完全不知道怎麼還原到前一個能跑的版本。

這就是為什麼還是要懂一點基礎知識，因為當 AI 出包的時候，你至少要知道怎麼救自己。

那…所以新手要懂哪些東西呢？接下來我就會一個一個講，順序大致是依照「踩坑機率」來排，越前面的越重要。

Note
這篇文章會盡量用最白話的方式來解釋每個東西，假設你完全沒寫過程式碼也看得懂，所以如果你已經有開發經驗，可能會覺得有些地方講得很基本，這是刻意的，請不要見怪。

Git 與 GitHub：你的後悔藥

如果整篇文章你只看得進去一個東西，那我會推薦你先學 Git。

什麼是 Git 呢？簡單來講，Git 就是一個「版本控制工具」，它可以幫你紀錄每一次你改了哪些檔案，這樣當 AI 把程式碼改壞的時候，你可以隨時回到上一個能跑的版本。

如果用比較生活面的例子來講的話，Git 就像是你在打遊戲時的存檔點。每打完一個關卡你就存一次檔，這樣如果下個關卡死了，你還可以回到上一個存檔點重來，而不是從頭開始整個遊戲。

那 GitHub 呢？GitHub 就是一個放 Git 紀錄的雲端平台，你可以把你的專案放上去，不只是備份的功能，還可以在不同電腦之間同步、跟別人協作。

對於新手來講，最少要會以下這幾個基本動作：

動作 說明
git init 在專案資料夾裡面建立 Git 紀錄
git add 把要存檔的檔案加進來
git commit 真正存檔，每次存檔都要寫一句話描述這次改了什麼
git status 看現在有哪些檔案還沒存檔
git diff 看你改了什麼東西
git log 看歷史存檔紀錄

從上面這張表可以看出來，Git 最核心的概念其實就是「存檔」這件事，把這幾個動作搞懂之後，你 80% 的需求都可以解決了。

Note
如果你完全不想碰指令的話，VSCode 內建有 Git 的視覺化介面，GitHub 也有出 GitHub Desktop 這種圖形化工具，可是我還是會推薦先學一下指令，因為很多 AI 工具的教學都是用指令在講，看得懂指令對你會比較有幫助。

至於進階一點的 git branch、git merge 這些等你之後習慣了再學就好，一開始不用一次塞太多。

Terminal（命令列）：跟工具互動的入口

很多新手第一次看到那個全黑色背景閃著游標的視窗會直接嚇到，可是 Vibe Coding 的世界裡，Terminal 是逃不掉的。

那 Terminal 到底是什麼？這邊要先稍微解釋兩個名詞：

  • GUI（圖形使用者介面）： 你日常用的 LINE、Word、Chrome 都屬於這種，有按鈕、有圖示，你用滑鼠點一點就可以操作
  • CLI（命令列介面）： 沒有按鈕、沒有圖示，整個畫面只有一行一行的文字，你要用打字的方式跟電腦下指令

Terminal 就是後者，台灣常翻成『命令列』or『終端機』，簡單來講就是『讓你打字跟電腦溝通』的一個視窗。

為什麼明明有 GUI 這麼方便的東西，工程師還在用 Terminal？因為 Terminal 可以直接執行非常細的指令，速度比一個一個用滑鼠點還快，而且很多開發工具根本沒有圖形介面可以用，只能透過 Terminal 操作。

那為什麼新手會逃不掉呢？因為現在主流的 AI Coding 工具，例如 Claude Code，它本身就是一個跑在 Terminal 裡面的工具，你打開 Terminal 輸入 claude 才能啟動它。其他工具像 Cursor 雖然是編輯器，可是它裡面也內建了一個 Terminal，你還是會用到。

那新手最少要會哪些指令？大概就這幾個：

指令 用途
cd 切換資料夾，例如 cd my-project
ls 看目前資料夾有哪些檔案（Windows 是 dir）
pwd 看你現在在哪個資料夾
mkdir 建立新資料夾
clear 清空畫面

這些指令真的不多，可是對新手來講卻是很重要的入口，因為當你不知道自己現在在哪個資料夾、AI 工具又在那邊跑時，你會搞不清楚它到底是在哪邊執行東西的。

Note
macOS 內建的 Terminal 就堪用了，可是我會推薦裝一下 iTerm2 配上 Oh My Zsh，視覺上比較舒服。Windows 的話可以用 Windows Terminal 配上 WSL，體驗會比預設的 PowerShell 好很多。

編輯器：VSCode 幾乎是現在的預設選擇

雖然 Vibe Coding 主打不用親手寫 code，可是你還是會需要一個地方可以打開檔案、看 AI 寫了什麼、必要的時候手動微調，那就是編輯器。

目前最主流的免費編輯器就是 VSCode（Visual Studio Code），由微軟開發，跨平台、生態系超大、AI 工具的整合度也是各家編輯器裡面最好的。

新手裝完 VSCode 之後，我會推薦至少裝這幾個擴充套件：

  • GitHub Copilot： 即時程式碼補全，雖然 Vibe Coding 不太會用到自己打字，可是有它在 review 的時候比較順
  • GitLens： 看 Git 歷史記錄超直覺，可以看每一行是誰、什麼時候改的
  • Prettier： 自動幫你把程式碼格式化整齊
  • 語言對應的擴充套件： 例如 Volar（Vue）、ESLint（JavaScript）等等，依照你的專案需求安裝

至於 Cursor、Kiro、Antigravity 這些是基於 VSCode 改的 AI 編輯器，如果你已經用過 VSCode 想要換到那邊去也沒什麼門檻，可是新手我還是建議從 VSCode 開始就好，因為大部分教學文章都是用 VSCode 在示範。

Note
如果你想知道現在主流的 AI Coding 工具差在哪裡，可以參考我之前寫的 2026 年 AI Coding 工具怎麼選？GitHub Copilot、Cursor、Claude Code、Kiro 與 Antigravity 完整比較。

一點點 HTML、CSS、JavaScript

看到這個小節你可能會吐槽：「不是說好不用寫 code 嗎？」

別誤會我意思，這邊不是要你會寫，而是要你「看得懂」。

那要先回到一個更基本的問題：網頁到底是什麼東西做的？

你打開一個網站，看到了標題、圖片、按鈕、表格，這些東西其實都是用程式碼寫出來的，就像書是用一個一個字組成的，網頁也是用一行一行的程式碼組成的。

而做網頁主要會用到三種程式語言（or 標記語言），名字看起來很可怕，可是各自負責的事情其實滿好理解：

  • HTML： 負責「網頁裡有什麼東西」，例如『這邊放一個標題、那邊放一張圖片、這裡放一個按鈕』
  • CSS： 負責「這些東西長什麼樣子」，例如『標題要紅色、按鈕要圓角、圖片要置中』
  • JavaScript： 負責「這些東西會做什麼事情」，例如『按鈕被點下去要彈出一個視窗、滑到頁面底部要載入更多文章』

如果用蓋房子的例子來講，HTML 就像是房子的鋼筋水泥，把牆、門、窗的位置定下來；CSS 就像是裝潢，決定要刷什麼顏色、地板用什麼材質、家具怎麼擺；JavaScript 則像是電動裝置，按下開關要開燈、感應器有人經過要自動開門。

你不用會寫，可是你至少要知道「啊這段是 HTML、那段是 CSS」，因為當你想跟 AI 說「把這個按鈕改成藍色」的時候，你才知道要改的是 CSS（外觀）而不是 HTML（結構）。

前端框架：你會聽到 React、Vue、Svelte 這些字

接下來這個是新手最容易誤判的地方。

當你跟 AI 說「幫我做一個網頁」的時候，AI 並不會單純丟給你 HTML/CSS/JavaScript，它很有可能會幫你選一個前端框架來做，例如 React、Vue、Svelte 等等，這些東西其實是 JavaScript 的延伸，不是另一種語言，可是新手第一次接觸時會直接被那些怪語法嚇到。

那前端框架是什麼呢？

簡單來講，前端框架就是『一套幫助你更有效率地建立網頁介面的 JavaScript 工具』，它把網頁拆成一個一個的『元件（Component）』，你可以重複使用這些元件，而不是每次都從零開始寫 HTML/CSS。

那什麼叫做『元件』呢？舉個例子你就懂了：當你打開 YouTube，每一支影片下面那個讚數、按讚按鈕、分享按鈕的組合，每一個影片都長得一樣對吧？這就是一個元件，工程師寫一次，然後讓所有影片都套用，這樣修改的時候只要改一個地方就好。

如果用比較生活面的例子來講的話，沒有前端框架就像是你每次煮飯都要從種米開始種，而前端框架就是直接給你白米、給你料理包，你只要組合起來就可以變成一道菜。

目前最主流的幾個前端框架大概是這些：

框架 說明
React 由 Meta 開發，目前市佔率最高，生態系最完整，學習曲線中等
Vue 在台灣跟亞洲圈滿熱門的，語法相對直覺，新手友善
Svelte 比較新的一個選擇，沒有 Virtual DOM 效能很好，語法很簡潔
Angular 由 Google 開發，比較重型，企業專案用得比較多

那為什麼新手要先知道這些東西呢？因為當 AI 寫出來的程式碼出現 useState、onMounted、$: 這類看起來怪怪的語法時，你才知道它是在用什麼框架，才知道要怎麼問 AI、要去哪邊查文件。

不然你只跟 AI 說「我這段不會」，AI 也不知道你哪邊不會。

Note
如果你還沒有特別偏好，新手第一個建議從 Vue 開始，因為它的學習曲線比較平緩，繁體中文資源也滿多的。如果你已經想要往求職的方向走，那 React 的市場需求會比較大。

UI Framework：Tailwind、Bootstrap、shadcn 這些幫你美化的工具

聊完前端框架，接著順便來認識一個最容易跟它搞混的東西，也就是 UI Framework。

UI Framework 是什麼呢？簡單來講，就是『一套幫你快速做出好看介面的工具』，它會提供現成的元件（按鈕、表格、卡片等等）or 一套已經設計好的樣式系統，讓你不用從零刻 CSS。

那它跟前端框架差在哪呢？

如果還是用煮飯的例子，前端框架是給你白米跟廚房，讓你可以煮出各種菜色；而 UI Framework 比較像是預先做好的醬料包，你淋上去就有那個味道，不用自己慢慢調味。前端框架負責「結構」，UI Framework 負責「外觀」，這兩個是可以同時用的。

目前主流的 UI Framework 大概可以分成這幾種：

名稱 風格 說明
Tailwind CSS 工具類別導向 不給你現成元件，而是給你大量的小工具 class 讓你自由組合
Bootstrap 元件導向 老牌穩健，提供現成的按鈕、導覽列、表單等元件
Material UI（MUI） 元件導向 Google 的 Material Design 風格，搭 React 用
shadcn/ui 拷貝貼上式 直接把元件原始碼複製到你的專案，可以隨意修改
DaisyUI 元件導向 建構在 Tailwind 上面的元件庫

新手不用一次認識所有 UI Framework，一開始接觸到哪個就先學哪個就好。

如果完全沒概念，我會推薦從 Tailwind CSS 開始，因為現在很多 AI 工具預設就會用 Tailwind 寫出來，而且 Tailwind + shadcn/ui 的組合在 2026 年的現在幾乎是新專案的預設選擇。

知道這些差異的好處是，當你看 AI 寫出來的程式碼時，會知道哪些 class 是 Tailwind 的、哪些是自訂的，要改顏色 or 間距的時候才不會改錯地方。

Node.js 與套件管理工具

當你跟 AI 開始做東西的時候，遲早會遇到 npm install 這個指令，然後你會看到一個叫做 node_modules 的資料夾出現，裡面有幾百個你看不懂的東西。

別緊張，這就是 Node.js 的世界。

那 Node.js 又是什麼？

前面有提到 JavaScript 原本是負責讓網頁有互動的程式語言，可是它原本只能跑在瀏覽器裡面，沒辦法直接在你的電腦上執行。Node.js 出現之後，JavaScript 就可以脫離瀏覽器、直接在你的電腦上跑了，就像把原本只能在水裡游的魚變成兩棲動物一樣，去哪都可以。

而現代的前端開發工具大多都依賴 Node.js 來運作（包含 AI 工具的很多輔助程式），所以你還是要裝它。

那套件管理工具是什麼呢？想像一下你做菜要用到醬油、糖、鹽，你不會自己去釀醬油吧？你會去超市買一罐回來。套件管理工具就是程式碼世界的超市，幫你把別人寫好的程式碼下載下來給你用。

目前主流的套件管理工具有這幾個：

工具 說明
npm Node.js 內建，最老牌也最通用
yarn Facebook 推出，曾經比 npm 快很多，現在差距沒那麼明顯
pnpm 用 hard link 共用套件，省硬碟空間且安裝速度快，近期滿熱門的

以我自己來講，目前主要都用 pnpm，因為它在處理大量專案的時候硬碟空間真的省很多。

接下來再來看幾個你一定會遇到的檔案：

  • package.json：你的專案的『身分證』，記錄這個專案叫什麼、用了哪些套件
  • package-lock.json / pnpm-lock.yaml：鎖定每個套件的版本，確保你跟別人裝起來是一樣的
  • node_modules：套件實際下載下來放的地方，這個資料夾通常會很大，所以一般不會 commit 進 Git

新手一開始不用搞懂太多，只要知道「npm install 是把套件裝起來」、「npm run dev 是啟動開發伺服器」這兩個動作就可以解決大部分需求了。

看得懂錯誤訊息

這一點看似簡單，可是其實是 Vibe Coding 過程中超關鍵的能力。

為什麼？因為你跟 AI 合作開發的流程通常是這樣的：

  1. 你描述你要的功能
  2. AI 寫出程式碼
  3. 你跑跑看
  4. 出錯了
  5. 你把錯誤訊息丟回去給 AI
  6. AI 根據錯誤訊息修正

注意第 5 步，如果你看不懂錯誤訊息，甚至不知道錯誤訊息在哪裡看的話，整個流程就斷掉了。

可是新手最常遇到的問題是：「我的網頁打開沒反應，可是我也沒看到任何錯誤啊？」

這是因為錯誤訊息不會自己跳出來給你看，它躲在一個叫做『開發者工具』的地方。你只要在任何網頁上按下鍵盤上的 F12（or 在網頁上按右鍵選『檢查』），就會跑出一個側邊視窗，這個視窗就是開發者工具，是你進入「網頁背後世界」的入口。

開發者工具裡面有很多分頁，新手最常用的是這兩個：

  • Console（主控台）： 程式碼跑出問題的話，紅色的錯誤訊息就會出現在這裡，是新手第一個要學會看的地方
  • Network（網路）： 網頁向後端拿資料的紀錄都在這，當網頁拿不到資料的時候可以來這邊看是不是 API 出問題了

除了瀏覽器之外，錯誤訊息還會出現在這幾個地方：

  • Terminal 視窗： 你執行指令的那個黑框框，套件安裝失敗、伺服器啟動失敗、編譯錯誤都會印在這
  • 編輯器的 Problems 分頁： VSCode 下方會有一個 Problems 標籤，會把語法錯誤列在那邊

新手最容易忽略的就是瀏覽器的 Console，很多人網頁打開沒反應就以為是 AI 寫壞了，結果 F12 一按 Console 早就紅成一片在告訴你問題在哪裡。

把錯誤訊息整段複製給 AI 是最快的修正方式，不要只跟 AI 說「我這個壞了」，因為這樣 AI 又要重新猜一次。直接把那段紅色的英文連同前後文一起貼給 AI，它就會知道問題在哪裡。

環境變數與基本資安觀念

這一點我特別要拉出來講，因為新手最常踩這個坑，而且踩到會很痛。

那先解釋一下什麼是『API Key』，因為新手可能也沒聽過這個詞。

簡單來講，API Key 就像是『某個服務發給你的一張會員卡』，你拿這張卡才能去使用 OpenAI、Google Maps 這類雲端服務。每次你用這張卡，服務就會記下你用了多少，然後跟你收錢 or 從你的免費額度扣款。

所以這張卡是『跟你的錢綁在一起的』，如果被別人拿走，他們就可以拿你的錢去用這些服務。

  • 機密資料放在 .env，並且加進 .gitignore

怎麼跟 AI 溝通：Prompt 與 Markdown

最後這一點看起來不像是技術，可是真的會直接影響你 Vibe Coding 的成敗。

你應該有遇過這種狀況：跟 AI 說一句話，結果它做出來的東西跟你想的差超多。這通常不是 AI 笨，而是你的描述讓 AI 有太多解讀空間。

這時候 Markdown 就是你的好朋友。

那 Markdown 是什麼呢？你可以把它想成『加強版的純文字』，本來純文字就只是一行一行字而已，沒有標題、沒有粗體、沒有條列，可是 Markdown 讓你用簡單的符號就能標示出這些結構，例如用 # 表示標題、- 表示條列、**文字** 表示粗體。

如果你有用過 Notion、Discord、ChatGPT 的話，其實就已經接觸過 Markdown 了，因為這些工具都支援 Markdown 語法。讓你的 Prompt 看起來有結構，AI 也比較好理解你想表達的階層。

結論

回頭整理一下這篇講過的東西：

推薦學習項目 重要性 為什麼
Git 與 GitHub 必學 AI 改壞東西時你的後悔藥
Terminal 命令列 必學 多數 AI 工具的入口
VSCode 編輯器 必學 看程式碼跟手動微調的地方
HTML/CSS/JS 基礎 推薦 至少看得懂 AI 寫了什麼
前端框架認知 推薦 知道 AI 用了 React 還是 Vue
UI Framework 認知 推薦 知道介面的樣式從哪邊來
Node.js 與套件管理 推薦 跑現代前端專案會用到
看錯誤訊息 必學 把錯誤丟回 AI 的關鍵
環境變數與資安 必學 不踩 API Key 外洩這種大坑
Markdown 與 Prompt 必學 跟 AI 溝通的基本功

從上面這張表你會發現，Vibe Coding 推薦先補的基礎其實沒有想像中那麼多，重點都圍繞在「怎麼跟工具互動」、「怎麼跟 AI 溝通」、「東西壞掉怎麼救」這三件事情上。

我自己覺得 Vibe Coding 雖然主打『不用寫程式』，可是實際上你還是要對開發這件事有基本的認知，不然當 AI 出包的時候你就會陷入完全束手無策的窘境。

當然這篇文章列的東西不是要你今天看完明天就全部會，慢慢來就好，挑一兩個你覺得最痛的開始補就可以了，其他的等遇到再學也不遲。

希望這一篇有讓你對 Vibe Coding 該補的基礎知識有更近一步的認識哩～
