看懂 AI 規格表：Token、Context Window、算力，這些數字到底在講什麼？

前言

如果你跟我一樣，最近常常在看各種 AI 產品的介紹頁面，應該對這種畫面不陌生：「支援 1M Token Context Window」、「擁有千億參數」、「Input 每百萬 Token 只要 $3.00」，看起來好像很厲害，但這些數字到底代表什麼意思？

相信很多人一開始看到這些規格表的時候，反應都是「數字越大應該越好吧？」但其實每個數字背後代表的意義都不一樣，有些數字大不一定就是好，理解這些術語之後，你才能真正判斷哪個 AI 適合你的需求，否則就只是被行銷數字牽著走而已。

那麼這篇文章就來一次把這些 AI 廠商最愛講的術語全部拆解清楚，讓你以後再看到這些規格表的時候，不會再霧煞煞。

Token：AI 世界的最小貨幣單位

什麼是 Token？

如果你有在使用 ChatGPT 或 Claude 這類 AI 工具，你一定有聽過「Token」這個詞，畢竟不管是 API（簡單來講就是讓開發者可以用程式去呼叫 AI 服務的介面）計費還是 Context Window 限制，全部都跟 Token 有關。

簡單來講，Token 就是語言模型（也就是 ChatGPT、Claude 這些 AI 背後的核心技術）在處理文字時的最小單位，你可以把它想像成 AI 在閱讀跟產生文字時所使用的「基本貨幣」，每一次 AI 讀取你的輸入或者產生回應，都是以 Token 為單位在計算的。

但 Token 並不等於「一個字」或「一個詞」，它的切法取決於每個模型所使用的 Tokenizer（分詞器），比如英文中 "darkness" 這個字可能會被切成 "dark" 和 "ness" 兩個 Token，而 "I" 就是一個完整的 Token。那麼 Tokenizer 到底是怎麼決定要怎麼切字的呢？

等等，電腦不是只懂 0 跟 1 嗎？為什麼要用 Token？

在繼續聊 Tokenizer 之前，你可能會有一個更根本的疑問：「電腦不是只懂 0 跟 1 嗎？為什麼 AI 不是直接用二進位來處理文字，而是要多搞一個 Token 的概念？」

確實，電腦最底層的運作就是 0 跟 1，所有的文字在電腦裡面最終都會被轉換成一串 0 跟 1 的組合。比如英文字母 A 在電腦裡會被存成 01000001 這 8 個 0 和 1，而中文字 你 因為比較複雜，需要用到 24 個 0 和 1 才能表示。那既然文字已經可以用 0 跟 1 來表示了，為什麼語言模型還需要多一層 Token 的概念呢？

原因是效率跟語意。

如果讓語言模型一個字元一個字元去處理，那模型要處理的長度就會變得非常長。舉例來講，"Hello" 這個字如果一個字元一個字元來處理就是 5 個單位，看起來好像還好對吧？但如果換成一句話 "The quick brown fox jumps over the lazy dog"，一個字元一個字元算就是 43 個單位。而如果我們改用 Token 來處理，透過後面會介紹的切字技術，可能只需要大約 9 到 10 個 Token 就能表示同樣的句子，長度直接縮短了好幾倍。

這個差異在處理長文本的時候會更加明顯，因為語言模型的計算量會隨著處理長度的增加而急遽上升（大約是長度的平方倍），所以如果處理長度是 4 倍，計算量就會變成 16 倍，這在實際運作上是完全不可行的。

除了效率之外，Token 還能捕捉到更有意義的語意單位。一個字元 d 本身沒什麼意義，但 "dark" 這個 Token 就帶有明確的語意，模型可以直接學到「dark」代表「黑暗」的概念，而不需要先從 d、a、r、k 四個字元慢慢拼出這個概念。所以 Token 本質上就是在「電腦的二進位世界」跟「人類的語意世界」之間找到的一個平衡點，既不會太細碎導致效率低落，也不會太粗糙導致詞彙表爆炸。

Tokenizer 是怎麼切字的？

目前主流的語言模型大多使用一種叫做 BPE 的切字演算法（全名是 Byte Pair Encoding，不用特別記），包括 GPT 系列、Claude 系列都是用 BPE 或它的變體。這個演算法的核心邏輯其實不難理解：它會先把文字拆成最小的單位（通常是單一字元），然後不斷地把最常一起出現的「相鄰字元組合」合併成一個新的 Token，重複這個過程直到詞彙表（也就是「這個 Tokenizer 認識的所有 Token 清單」）達到預設的大小為止。

舉個例子，假設我們輸入 "hello" 這個英文單字，BPE 一開始會把它拆成五個字元：

txt
h, e, l, l, o

接著因為 l 跟 l 在訓練資料中經常相鄰出現，所以會先合併成 ll：

txt
h, e, ll, o

然後如果 he 這個組合也很常見，就會再合併：

txt
he, ll, o

最終在經過足夠的合併之後，hello 本身可能就會變成詞彙表中的一個 Token。而像 "darkness" 這個字，則可能被切成 "dark" 和 "ness" 兩個 Token，因為這兩個部分在英文中都是非常常見的組合。有趣的是 "brightness" 也會被切成 "bright" 和 "ness"，你會發現 "ness" 這個後綴會共用同一個 Token 編號，這也幫助模型理解「darkness」和「brightness」之間存在某種語意上的關聯。

而實際上最終訓練完成的 Tokenizer 會有一張固定的詞彙表，大小通常在 32000 到 200000 個 Token 之間，比如 OpenAI 所使用的詞彙表（代號叫 cl100k_base，100k 就是大約 100000 的意思）裡面收錄了大約 100000 個 Token，所有的文字輸入都會被對照這張表來切割。

為什麼中文比英文更「耗」Token？

這邊就要提到一個很現實的問題了。大多數語言模型的 Tokenizer 在訓練的時候，餵給它的文字資料大部分都是英文，所以英文的 Token 效率非常高，大約每 4 個英文字母就對應 1 個 Token，換算下來大約是 1 個英文單字消耗 1 到 1.5 個 Token 左右。

但中文就不一樣了。因為 Tokenizer 主要是從英文資料中學會怎麼切字的，中文字在詞彙表中佔的比例相對少，大部分的中文字需要用 1.5 到 2 個 Token 來表示，有些比較罕見的字甚至需要 3 個 Token。原因是 Tokenizer 的詞彙表大小有限（比如 100000 個位置），但光是中文、日文、韓文的常用漢字加起來就有好幾萬個，不可能全部都各占一個位置，所以很多中文字就得用多個 Token 來組合表示。

舉個具體的比較：

內容文字大約消耗的 Token 數
英文"Hello, how are you?"約 6 個 Token
中文「你好，你好嗎？」約 11 個 Token

明明表達的是差不多的意思，中文卻消耗了將近兩倍的 Token，這也意味著中文使用者在同樣的 Context Window 限制下，能塞進去的資訊量其實比英文使用者少，同時如果你是用 API 來呼叫模型的話，中文的費用也會比英文高一些。

Note 這個問題在近年有逐漸改善，像是 OpenAI 後來推出的 o200k_base Tokenizer，以及各家模型針對多語言的最佳化，都有在縮小中英文的 Token 效率差距，但差異仍然存在。

Input Token vs Output Token

另一個很重要的概念是 Token 分成「Input Token（輸入 Token）」和「Output Token（輸出 Token）」。

Input Token 就是你傳送給 AI 的所有內容，包含你的問題、上傳的文件、System Prompt（系統提示詞，也就是開發者預先設定好的背景指令，告訴 AI 它應該怎麼回應）、對話歷史等等，這些全部加起來就是你的 Input Token 消耗量。

Output Token 則是 AI 回覆給你的內容，每產生一個 Token 就消耗一次。

那為什麼要特別分開呢？因為 AI 產生文字比讀取文字需要更多的運算資源，所以 Output Token 的成本通常比 Input Token 高出好幾倍，以目前主流模型來說，Output 的單價大約是 Input 的 4 到 6 倍左右。這也是為什麼你會看到 API 的定價表上面 Input 跟 Output 是分開列的。

簡單來講，你問的問題越長、AI 回的內容越多，消耗的 Token 就越多，費用也就越高。

Context Window：模型的「工作記憶」上限

Context Window 是什麼？

如果你曾經跟 ChatGPT 聊到後面，發現它開始忘記你前面說過的話，那你已經親身體驗過 Context Window 的限制了。

Context Window 簡單來講就是語言模型在一次對話中能夠處理的「最大 Token 數量」，你可以把它想像成 AI 的工作桌面，桌面越大就能同時攤開越多文件來參考，但桌面大小是有限的，超過上限的內容就會被擠掉或者壓縮掉。

這邊要特別強調一點，Context Window 包含了所有的東西：你的 System Prompt、對話歷史、你上傳的文件以及 AI 的回覆，全部都擠在同一個 Context Window 裡面。所以當 AI 廠商說「支援 1M Token Context Window」的時候，並不是說你可以傳入 1M Token 的問題然後再拿到 1M Token 的回答，而是整個對話的「輸入加上輸出的總量」不能超過這個上限。

那麼 1M Token 到底有多大？這邊先說明一下，在 AI 規格表中常見的 K 跟 M 就是數量單位，K 是 Thousand 也就是「千」，M 是 Million 也就是「百萬」，所以 1M Token 就是 100 萬個 Token，128K 就是 128000 個 Token。那 100 萬個 Token 換算下來大約等於 75 萬個英文單字，或者大概 50 到 65 萬個中文字（因為前面提到中文比英文耗 Token），差不多是好幾本書的內容量。

Context Window 大概有多大？

如果你現在去看各家 AI 的介紹頁面，會發現 Context Window 從幾十萬到上千萬 Token 都有，而且這個數字每隔幾個月就會更新一次，所以這邊就不列出特定的模型規格了，你可以隨時到各家官方網站查到最新的數字。

不過有一個概念值得了解：Context Window 跟「最大輸出 Token」是兩個不同的限制。Context Window 是指整個對話（輸入加輸出）的總量上限，而最大輸出 Token 則是模型單次回覆最多能產生多少 Token。

舉個例子，假設某個模型的 Context Window 是 1M Token，但最大輸出只有 64K Token，那代表它能「讀」的量很大（整個 Context Window 都可以放東西進去），但它單次能「寫」的量最多就是 64K Token，不可能一次回覆整本書給你。有些模型甚至 Context Window 號稱上千萬，但最大輸出可能只有幾千個 Token，所以光看 Context Window 的數字是不夠的，也要注意它的最大輸出限制。

廣告數字跟實際效能是兩回事

這邊有一個非常重要的觀念：廣告上寫的 Context Window 大小跟模型實際能有效處理的量，其實是有落差的。

根據各種獨立測試的結果，大多數模型在接近 Context Window 上限的時候，準確度就會開始明顯下降，模型可能會遺漏你前面提到的重要資訊，或者對前後文的理解出現偏差。一般來說實際能有效使用的範圍大約只有廣告數字的六到七成左右。

這個現象其實跟人類很像，就像你在讀一本 500 頁的書，讀到後面的時候前面章節的細節你其實也記不太清楚了。語言模型也有類似的問題，研究顯示模型對於 Context 開頭跟結尾的內容記憶最深，中間的內容反而最容易被忽略，這也被稱為「Lost in the Middle（中間丟失）」現象，簡單來講就是塞在中間的內容最容易被模型忽略掉。

所以下次看到 AI 廠商宣傳「支援 10M Token」的時候，不要直接就以為你真的可以丟 10M Token 進去然後期待完美的回應，實際使用時還是要考慮有效的處理範圍。

Note 補充一個實務上的小技巧，如果你需要讓 AI 處理大量文件，比起把所有文件一次全部塞進去，分批處理並整理摘要再合併，通常效果會更好。

算力：驅動 AI 背後的引擎

什麼是算力？

你可能常在新聞上看到「算力」、「GPU」、「NVIDIA」這些詞一起出現，那到底什麼是算力呢？

算力簡單來講就是「電腦在一定時間內能完成多少計算的能力」，通常用 FLOPS 來衡量，全名是 Floating Point Operations Per Second，翻成中文就是「每秒可以做多少次浮點運算」。所謂的浮點運算就是處理帶有小數點的數字計算，像是 3.14 × 2.71 這種，而 AI 模型內部幾乎所有的計算都是這類運算，所以 FLOPS 就成了衡量算力最常用的單位。這個數字越大，代表這台電腦（或 GPU，也就是顯示卡上的處理器晶片，後面會詳細說明）每秒能做的計算越多。

常見的算力單位有：

單位全名數量級
GFLOPSGiga FLOPS每秒 10 億次運算
TFLOPSTera FLOPS每秒 1 兆次運算
PFLOPSPeta FLOPS每秒 1000 兆次運算
EFLOPSExa FLOPS每秒 100 萬兆次運算

這些數字聽起來像是天文數字，但訓練一個大型語言模型需要的運算量更是天文中的天文，所以才會需要用上成千上萬片 GPU 同時運算。

訓練 vs 推理

在 AI 的世界裡，算力的使用分成兩個主要階段：訓練（Training） 跟 推理（Inference）。

訓練是讓 AI 模型「學會知識」的過程，就像是讓一個學生讀遍所有的教科書、論文、網頁內容，然後把學到的知識「壓縮」進模型的參數裡面。這個過程需要消耗巨大的算力，像是訓練 GPT-4 等級的模型，據估計需要使用上萬片 GPU 連續運算好幾個月。

推理則是模型在學完之後「運用知識」的過程，也就是你每次向 ChatGPT 提問或者讓 Claude 幫你寫程式碼的時候，模型所執行的計算就是推理。相比訓練，單次推理需要的算力小很多，但因為全世界每天有數以百萬計的使用者在同時使用這些 AI 服務，累積起來的推理算力需求其實也非常驚人。

你可以用一個簡單的比喻來理解：訓練就像是花好幾年時間培養一個醫生，而推理就像是這個醫生每天看診。培養的過程需要投入大量的時間跟資源，但一旦培養完成，每次看診就相對快速。

為什麼 GPU 比 CPU 更適合 AI？

你可能會好奇，為什麼 AI 領域都在用 GPU 而不是一般的 CPU？先簡單介紹一下這兩個東西：

  • CPU（Central Processing Unit，中央處理器）：就是你電腦裡面的「大腦」，負責處理各種運算任務，不管是開瀏覽器、跑程式都靠它
  • GPU（Graphics Processing Unit，圖形處理器）：原本是顯示卡上面專門用來處理遊戲畫面跟 3D 圖形的晶片

那為什麼做 AI 要用原本處理遊戲畫面的晶片？這是因為 AI 的計算本質上就是大量的數字需要同時做加法跟乘法，而 GPU 天生就擅長「同時處理非常多筆簡單的計算」，CPU 則比較擅長「一次處理一件很複雜的事情」。

打個比方，CPU 就像是一個非常聰明的數學教授，一次只能算一道題但算得又快又準。GPU 則像是一千個普通計算員，每個人只會做簡單的加法，但一千個人同時算，總速度就遠超那位教授了。而 AI 的計算恰好就是那種「簡單但大量」的工作，所以 GPU 成了 AI 領域最核心的硬體。

目前 AI 訓練市場幾乎被 NVIDIA 壟斷，佔據了超過 90% 的市場份額，這也是為什麼 NVIDIA 的股價跟市值這幾年成長得如此驚人的主要原因之一。

參數（Parameters）：模型的「腦容量」

什麼是參數？

當你看到新聞說「這個模型有 700 億個參數」或者「據傳 GPT-5 擁有數兆參數」的時候，這個「參數」到底是什麼？

簡單來講，參數就是語言模型在訓練過程中學會的「數值」，每一個參數都是一個數字，代表了模型對某個語言模式的理解程度。這些數字組合在一起，就構成了模型判斷「下一個字應該是什麼」的能力。如果用人腦來比喻的話，參數有點像是大腦中的「突觸連結」，突觸越多，大腦能夠處理的資訊就越複雜。

Note 如果你想更深入了解參數量跟量化的技術細節，包含怎麼根據自己的硬體來選模型，可以參考 想自己跑 LLM？搞懂模型參數與量化讓你不再選錯模型 這篇文章，裡面有更完整的說明。

參數越多越厲害嗎？

很多人直覺會覺得參數越多就越強，早期確實有這樣的趨勢，業界稱之為「Scaling Law（規模法則）」，意思就是只要把模型做大、資料塞多、算力砸夠，模型的能力就會持續提升。但到了 2025 跟 2026 年，大家開始發現單純增加參數量帶來的進步越來越不明顯了。

現在比較流行的做法是一種叫做 MoE（Mixture of Experts，混合專家）的架構設計方式。你可以把它想像成一間大公司裡面有很多不同專長的部門，但每次處理一個案子的時候，只會派最相關的幾個部門去處理，而不是全公司總動員。所以這種架構的模型「總參數量」可能高達數兆，但每次回答問題的時候只會啟用其中一小部分的參數，這些實際被用到的參數就稱為「活躍參數（Active Parameters）」。比如一個宣稱有 2 兆總參數的 MoE 模型，每次回答問題時可能只使用其中 2000 億的活躍參數，這樣既能保持強大的能力，又不會消耗太多運算資源。

所以現在看參數量的時候，更重要的是區分「總參數量」跟「活躍參數量」，只看總參數量已經不太能準確反映模型的實際能力了。

為什麼有些廠商不公開參數量？

你可能有注意到，OpenAI 從 GPT-4 開始就不再公開模型的參數量了，Anthropic 的 Claude 系列也從來沒有正式公開過，Google 的 Gemini 同樣如此。

這背後的原因主要是商業競爭考量，公開參數量等於告訴競爭對手你的模型架構細節，同時也避免市場只用參數量來做簡單粗暴的比較。目前業界對於各家模型參數量的估計，大多都是根據模型行為、推理速度跟定價反推出來的非官方數字，所以看到這些估計的時候，參考一下就好，不用太認真。

模型推理的「調味料」：Temperature、Top-P、Top-K

除了前面講的那些「硬規格」之外，如果你有用過 API 或者一些進階的 AI 工具，可能還會碰到 Temperature、Top-P、Top-K 這幾個參數。這些不是模型本身的規格，而是在模型產生回覆的時候（也就是前面提到的「推理」階段）可以調整的「旋鈕」，它們決定了模型回覆的風格跟隨機程度。

Temperature（溫度）

Temperature 是最常見也最直覺的一個參數，它控制了模型回覆的「創意程度」，數值範圍通常是 0 到 2。前面有提到語言模型在預測下一個字的時候，會計算每個候選字出現的機率（可能性），Temperature 就是用來調整模型要不要「冒險」選那些機率比較低的字。

  • Temperature 越低（接近 0）：模型越傾向選擇機率最高的那個字，回覆會比較確定、一致，適合寫程式碼或回答事實性問題
  • Temperature 越高（接近 1 或以上）：模型越願意選擇機率較低的字，回覆會比較多樣、有創意，適合寫故事或做腦力激盪

簡單來講就是，Temperature 低等於「保守穩定」，Temperature 高等於「大膽冒險」。

Top-P（核心取樣）

Top-P 的邏輯是「只從最有可能的那些候選字裡面挑選，直到它們的機率加起來達到 P%」。比如 Top-P 設定為 0.9，代表模型會先把所有候選字按照機率從高到低排列，然後只考慮機率加起來達到 90% 的那些字，剩下那些極不可能出現的字就直接排除掉。

Top-P 的好處是它是動態調整的，如果模型對下一個字非常有把握，它可能只會考慮 1 到 2 個候選 Token，但如果模型不太確定，它就會考慮更多候選，這讓取樣過程更加靈活。

Top-K

Top-K 則比較直觀，就是「只考慮機率最高的前 K 個 Token」。比如 Top-K 設定為 50，那不管這 50 個 Token 的總機率加起來是多少，模型都固定只從這 50 個裡面選。

相較於 Top-P 的動態調整，Top-K 的候選數量是固定的，所以在某些情境下可能不夠靈活。在 2026 年的實務上，大多數 API 廠商主要提供 Temperature 跟 Top-P 讓使用者調整，Top-K 比較少出現在商業 API 中。

Note 一般建議一次只調整 Temperature 或 Top-P 其中一個就好，同時調整兩個有時候反而會產生互相衝突的效果。如果你不確定要怎麼設定，大多數情境下保持預設值其實就很夠用了。

Token 計價：AI 服務怎麼收費的？

了解了前面那些術語之後，最後來聊一個很實際的問題：AI API 到底是怎麼收費的？

目前主流的 AI API 都是以「每百萬 Token」（通常簡寫成 MTok）為單位來計價的，而且 Input Token 跟 Output Token 的價格會分開計算，Output 的單價通常是 Input 的好幾倍。

具體的價格每家廠商、每個模型都不一樣，而且變動非常快，所以這邊就不列出特定的數字了，你可以到各家的官方定價頁面查看最新的報價。但有幾個觀念是不會變的：

  1. 模型越強越貴：同一個廠商通常會推出「旗艦型」跟「輕量型」兩種等級的模型，旗艦型的能力最強但價格也最高，輕量型便宜很多但能力相對有限，兩者之間的價差可以達到數十倍
  2. Output 比 Input 貴：因為 AI 產生文字比讀取文字需要更多的運算資源，所以 Output Token 的單價通常是 Input 的 4 到 6 倍左右
  3. 價格一直在降：這幾年 AI API 的價格持續快速下降，同等能力的模型比兩年前便宜了好幾倍，對開發者跟一般使用者來說都是好消息

所以一個好的使用策略是：簡單的任務交給便宜的輕量模型處理，真正複雜的任務才出動旗艦模型，這樣可以在成本跟效能之間取得不錯的平衡。

結論

看完這篇文章之後，下次再看到 AI 產品的規格表應該就不會再一頭霧水了，這邊簡單幫你做個重點整理：

  • Token：AI 處理文字的最小單位，不等於字數，中文會比英文消耗更多 Token
  • Context Window：模型一次能處理的最大 Token 總量，實際有效容量通常比廣告數字低，而且要注意最大輸出跟 Context Window 是不同的限制
  • 算力（FLOPS）：衡量運算能力的指標，訓練需要巨大的算力，推理相對較小但累積量驚人
  • 參數（Parameters）：模型學會的數值數量，現在要區分總參數量跟活躍參數量
  • Temperature / Top-P：控制模型回覆風格的調整旋鈕，不是越高越好也不是越低越好

理解這些術語的核心概念比記住具體的數字更重要，畢竟數字每隔幾個月就會更新一次，但概念不會變，希望這篇文章對你有幫助哩～
