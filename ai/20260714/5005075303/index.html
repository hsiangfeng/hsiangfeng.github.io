AI 模型在遠端，但 Coding Agent 的工具可能在本機執行

前言

今天公司在進行內訓時，同事問了幾個讓我印象很深刻的問題：

Claude 是不是跑在我的電腦裡？不然它為什麼可以找到我的專案、修改檔案，甚至自己執行指令？

我覺得這個問題其實滿合理的，畢竟我們看著 Claude Code、Codex CLI、Gemini CLI 這類 AI Agent 工具可以自己讀取檔案、寫/改程式、跑測試時，這眼裡看來真的很像有一個 AI 住在電腦裡。

但其實這個真正的答案本質要拆分成兩個部分，你看到的事情只是其中一部分在本機跑，另一部分則是在遠端伺服器上運算（如：Claude 公司的模型通常在遠端資料中心）。

那這一篇就是要解決你這個疑慮，然後定位會是以小白也能懂的範疇，所以並不會從複雜的 API 或 JSON 開始講。

只要區分清楚「模型」和「Coding Agent」，那這個問題就很好理解啦～

模型是大腦，Coding Agent 是雙手

首先使用 Claude Code 的時候，你可以區分三件事：

  1. Claude 的模型在遠端資料中心： 模型主要負責理解需求、閱讀收到的內容，以及決定下一步該做麼。
  2. Claude Code 是安裝在你電腦上的應用程式： 負責跟遠端模型溝通、管理權限與執行工具的應用程式
  3. 讀檔、改檔、跑測試等動作，是真的會在你的電腦執行： 實際動手的是安裝在你電腦上的 Claude Code，不是遠端模型

如果上面不好理解的話...你可以把它想成底下個比喻：

  • 遠端模型是大腦，本機 Coding Agent 是雙手。
  • 大腦負責想下一步，雙手才真的去碰檔案和執行指令。

Claude 模型與本機 Coding Agent 的分工

所以我們安裝 Claude Code 時，並不是下載一個完整的 Claude 模型到電腦裡，而是一個可以跟模型溝通並使用本機工具的應用程式。

Codex CLI、Gemini CLI 等 Coding Agent 也有類似的核心概念，不過不同的產品能使用哪些工具、在哪裡執行工具，以及什麼時候需要你的允許，還是有所不同。

那當你說「幫我修好這個 Bug」，中間發生了什麼？

這邊我們假設一下情境，也就是你在 Claude Code 對話視窗輸入以下：

Text
請幫我修好 XXX.js 的 Bug，然後跑測試。

其實背後大概是這樣的流程：

  1. 本機 Claude Code 收到你的需求（文字，或稱為 Prompt）
  2. Claude Code 把需求與目前需要的資訊送給遠端（也就是 Claude 公司的模型）
  3. 遠端接收到資訊後，判定認為要先讀取 XXX.js
  4. 模型回傳一個「請讀取 XXX.js」的工具呼叫
  5. 本機 Claude Code 檢查權限後，真的去讀取這個檔案（如果沒有權限也會跳出詢問是否要授權）
  6. Claude Code 把讀到的內容交給遠端模型
  7. 模型看完內容後，再決定如何修改，以及要執行哪個測試

這個過程不會只跑一次。

因為模型會先讀檔、再搜尋其他程式、接著修改內容、執行測試；如果測試失敗，它又會根據錯誤訊息決定下一步。

而這個「模型決定下一步 → 工具執行 → 結果交回模型 → 模型再決定下一步」的循環，就叫做 Agentic Loop（代理迴圈）。

Claude Code 修復 Bug 時，本機工具與遠端模型持續往返的 Agentic Loop

什麼是工具呼叫（Tools Call）？

工具呼叫呢...稍微有點難懂，簡單來講，你安裝 Claude Code 的時候，裡面會有一隻程式，這些程式有很多，像是有...

  • 讀檔工具（File Reader）
  • 改檔工具（File Writer）
  • 刪除工具（File Deleter）

諸如此類，因此遠端模型並不會直接伸手進入你的電腦，它只會告訴 Claude Code：

Text
我下一步想使用「讀檔工具」，目標是 XXX.js。

Claude Code 收到要求後，才會根據目前的權限設定決定：

  • 這個工具能不能使用？
  • 它可以讀取哪個資料夾？
  • 這次操作需不需要先詢問使用者？
  • 執行成功後，要把什麼結果交回模型？

所以「模型想要使用工具」和「工具真的執行成功」是兩件不同的事。

模型負責提出下一步，真正執行是外部程式或執行環境的事

除此之外，工具也不一定都在你的電腦執行，有些工具會在服務商的伺服器或雲端沙盒裡執行，例如 Codex cloud task、Code Interpreter、Web Search 等功能。

但...遠端模型為什麼看得到我的程式碼？

這很有趣，其實這個牽扯到前面的問題，也就是「讀取檔案」的部分。

遠端模型並不會直接看見你的整顆硬碟，也不會自己在你的資料夾裡亂逛。

它能看見 XXX.js，通常是因為本機 Coding Agent 使用讀檔工具取得內容，再把完成任務所需的內容交給模型。

除了檔案內容，模型每次處理任務時還可能取得：

  • 你目前提出的需求
  • 先前的對話內容
  • 專案規則與操作限制
  • 模型可以使用哪些工具
  • 指令輸出、錯誤訊息與測試結果

這些東西合起來我們稱為 上下文（Context）。

模型必須看足夠的上下文，才有辦法知道自己前面做過什麼，以及接下來應該要做什麼跟該怎麼處理。

當對話太長時，Coding Agent 可能只保留相關內容，或把較舊的內容摘要、壓縮。

你可以把它想成：桌面放不下所有資料時，就把不重要的文件收起來，只留下目前工作需要的部分，所以他的專業術語就是「Auto Compact（自動壓縮）」。

Warning

指令雖然在本機執行，但這並不代表資料都不會離開電腦。 Coding Agent 為了讓遠端模型理解專案，是會傳送檔案內容、錯誤訊息與工具結果。 所以實際會傳送哪些資料，仍然要看產品設計、設定與服務條款。

為什麼 Agent 開越多，電腦越容易 Lag？

講了那麼多，終於可以聊聊文章標題了，畢竟要聊這個就需要先解釋一下前面的運作流程。

當 Coding Agent 等待遠端模型回覆時，你的電腦就只是單純等待跟維持應用程式、畫面與網路連線而已，這過程並不會造成什麼負擔、負荷。

而真正容易讓電腦變慢的，是多個 Agent 同時在本機執行工作，例如...

  • 同時搜尋或讀取大量專案檔案
  • 同時執行測試、Lint、型別檢查或完整建置
  • 啟動多個開發伺服器與瀏覽器
  • 安裝套件、解壓縮檔案或建立快取
  • 同時讀寫大量檔案或操作不同的 Git worktree

多個 Agent 同時執行本機工作，造成 CPU、RAM 與磁碟負擔疊加

假設你同時讓三個 Agent 各跑一次完整測試，本質上就和你自己開三個終端機同時跑測試差不多。

這些工作會一起使用本機的 CPU、RAM 與磁碟，那你電腦當然可能變慢啦～

所以這個更正確來講的話：

Agent 開越多，就有可能同時執行很多本機工作，而讓電腦變 Lag 的主要原因是工作量疊加，不是遠端模型隔空吃掉你的本機 GPU。

當然，開三個 Agent 不代表電腦一定會變成三倍慢（如果會這樣，那這件事情多可怕，電腦效能要多好）。

當三個 Agent 都在等待遠端回覆，和三個 Agent 同時建置大型專案，對電腦造成的負擔完全是不同的事情。

那為什麼大家都說 AI 很吃 GPU 跟記憶體？

這個答案本質是 模型本身，畢竟模型很需要大量運算與記憶體，當你只是單純的使用雲端模型時，主要消耗的是遠端資料中心的硬體，不是你的筆電、桌機或伺服器。

這邊我也換個方式比喻：

  • 模型權重就像一整套超大型參考書： 模型要使用學到的能力，就必須讓這些資料可以被系統存取。
  • 記憶體像工作桌： 桌面越大，可以同時攤開的模型資料、對話內容與暫存結果就越多。
  • GPU 像一大群可以同時計算的工作人員： 模型需要進行大量運算，GPU 就很適合一起處理這些重複而龐大的計算。

模型權重像參考書、記憶體像工作桌、GPU 像同時進行計算的大量工作人員

所以當模型執行時，記憶體主要會放：

  • 模型本身的權重
  • 目前對話與輸入需要的 KV Cache
  • 推論過程中的中間結果與執行框架
Note

通常我們講記憶體，是泛指 CPU 記憶體（RAM）與 GPU 記憶體（VRAM），而不是指 HDD 或 SSD 的儲存空間。

舉例來說，使用 7B 的模型時，光是模型權重粗略估算就約需要 14 GB，實際執行還要再加上 KV Cache 與其他開銷，雖然量化可以把模型壓小，但也可能帶來品質、速度或相容性的取捨。

大型雲端模型還可能把權重分散在多張 GPU 或多台機器上，所以你不能把它想成「整個模型一定塞在同一張顯示卡裡」。

簡單一張表看懂模型

接下來，我準備了一張表格讓你看懂「模型在哪裡運算」和「工具在哪裡執行」的差別：

使用情況模型在哪裡運算工具在哪裡執行你的電腦主要負責什麼
一般雲端聊天，沒有使用工具遠端資料中心沒有額外工具顯示畫面、傳送需求與接收回覆
Claude Code、Codex CLI 等本機 Coding Agent模型通常在遠端讀檔、改檔、終端機等工具常在本機執行 Agent 與工具，使用 CPU、RAM、磁碟
Codex cloud task 等雲端 Agent遠端資料中心雲端容器或沙盒送出任務、查看結果與套用變更
Ollama、LM Studio 等本地模型你的電腦通常也在你的電腦連模型推論都自己負責，可能大量使用 GPU／統一記憶體

而這張表最重要的地方是：模型在哪裡運算，和工具在哪裡執行，是兩個不同的問題。

同一個 AI 產品也可能同時提供本機工具與遠端工具。例如資料分析、Code Interpreter、Web Search 或伺服器端 Code Execution。

所以看到「AI 幫我執行程式」時，並不能直接判斷程式一定跑在自己的電腦。

最終還是回歸到你是使用哪個功能，以及那個工具的執行環境。

本機工具真的會動你的電腦，權限真的很重要

模型雖然通常在遠端，在傳送資料過去時，模型商也會做基本敏感資訊過濾，但本機 Coding Agent 執行的動作就可能對你電腦造成影響。

畢竟它真的會...

  • 修改或建立檔案
  • 刪除內容
  • 安裝套件
  • 執行測試或啟動伺服器
  • 在權限允許時連線到外部系統

Claude Code、Codex 等 Coding Agent 會透過權限、核准規則或沙盒限制能夠操作的範圍，但你必須清楚知道並不是每一個動作都一定會先詢問你，而是完全取決於你選擇的模式（Auto Mode）、工作區邊界與操作類型。

如果你是第一次使用 Coding Agent 的話，我會強烈建議你先使用較保守的權限，Code Agent 跳出詢問權限畫面時，也要確認它準備讀寫刪哪裡、是否需要連網，以及動作有沒有破壞性等行為。

如果你無腦按下 Yes，那估計你的隱私、電腦都有可能受到外流或破壞。

結論

最後來總結一下重點：

  1. Claude、GPT 等雲端模型通常在遠端資料中心做演算、推論
  2. Claude Code、Codex CLI 是在本機執行的 Coding Agent 應用程式
  3. 模型只負責提出內容與下一步，真正動手的是本機、雲端或第三方的工具
  4. 多開 Agent 會不會 Lag，要看它們同時在本機執行多少工作。
  5. 只有使用本地模型時，模型推論才會直接大量使用你自己的 GPU／統一記憶體，但本機工具也可能因工作內容而使用 GPU。

所以回到最開始的問題：「Claude 是不是跑在我的電腦裡？」

關於這問題我相信你已經有一個明確答案了，畢竟你只要把「模型在哪裡算」和「工具在哪裡跑」分開看，就不會再把 AI 需要大量 GPU，和 Coding Agent 讓本機變慢，誤認為是同一件事了。
