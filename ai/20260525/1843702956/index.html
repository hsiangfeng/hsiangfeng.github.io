Claude Code Routines 排程怎麼設？/schedule 完整教學與 main 分支推錯解法

#Claude Code #AI #AI Agent #Claude Code Routines #schedule #排程 #Automation

Note
Claude Code 有三種排程：Routines（雲端，關電腦也會跑）、Desktop scheduled tasks（本機，要 App 開著）、/loop（session 內排程，要終端機開著）。/schedule CLI 是建立 Routines 的入口，最短 1 小時間隔，預設只能推 claude/ 前綴分支，要推 main 要去網頁打開 Allow unrestricted branch pushes。

前言

其實我一直都有在使用 Claude Code 的 /schedule 指令，而這個指令就是在建立一個排程任務，每天都會在我指定的時間執行，跑完之後我也會請它 push 到我的 GitHub 上。但，我滿常遇到一個狀況，也就是隔天打開 GitHub 一看，GitHub 分支反而變成了 claude/main-xxxx 這種，而不是我期望的 main 分支 QQ…

所以這篇就來把 Claude Code 的排程功能寫下來，順便把這個讓我很困擾的 claude/main-xxxx 記錄下來並搞清楚。

哦對了，如果你之前看過 Claude Cowork Dispatch 與 Remote Control 那篇，那這篇可以當作另一條延伸路線：Dispatch 跟 Remote Control 是讓你現在用手機遠端控制，而排程是讓 Claude 自動在你不在的時候幫你做事哩。

Claude Code 有三種排程

在進到 /schedule 之前，Claude Code 其實有三種不同的排程方式，這邊也準備一張表格比較一下差異：

了解這三種排程差異之後，接下來我就會著重於 Routines（Cloud） 的部分來介紹。

Routines 是什麼？

Routines 簡單來講，就是你把 prompt 跟它要用的 Repo 等等，放到 Anthropic 雲端，然後它會自己在你設定的時間 or 條件下幫你跑你要的任務。

跟一般的 Claude Code session 最大的差別有兩個：

  1. 它跑在 Anthropic 的雲端環境：不是你的電腦，所以你關電腦它也照跑、出門也照跑、睡覺也照跑。
  2. 它是自主執行的：不會跳出權限提示（也就是 Yes、No），會完全依照你給的 prompt 自己跑完，所以 prompt 也必須非常明確，不然它可能會做出你沒預期到的行為。

Note
如果想要讓自己的每個月訂閱額度不被浪費的話，其實我個人滿推薦使用 Routines 來做一些定期性的任務，例如每天幫我檢查有沒有新的 PR 需要 review、每週幫我整理一下這週的 commit log、每月幫我開一個 PR 更新一下 README 的 stats 圖表之類的。

建立排程：用 /schedule

建立 Routines 有兩種方式：

  1. 網頁的 claude.ai/code/routines
  2. CLI（終端機） /schedule

不論是哪一種都可以，反正兩邊改的是同一筆資料，所以你在哪邊建立、編輯都會即時同步到另一邊。

Note
/schedule CLI 只能建立 Schedule 觸發類型，如果要加 API or GitHub event 觸發，要到 claude.ai/code/routines 網頁端編輯。

除了建立之外，CLI 也支援管理已經存在的 routines：

指令 用途
/schedule list 列出你目前所有的 routines
/schedule update 編輯某個 routine（包含設 cron）
/schedule run 立刻觸發一次，不等下一個排程點

Note
真的懶的話，其實你叫 AI 用 /schedule 幫你建排程也可以…

為什麼我請它推到 main 卻變成 claude/xxx？

了解了基本的建立流程之後，接下來要來講一下我前面所講的 「明明請它推到 main 卻變成 claude/xxx」的問題。

當你在 Routines 裡跑一個會 push 程式碼的 prompt 時，Claude 預設只能推到 claude/ 前綴的分支。所以即使你的 prompt 寫「處理完直接 push 到 main」，Claude 也只能開一個像 claude/main or claude/update-main-xxxx 之類的分支推上去，不會真的動到你的 main。

這不是 bug 是 feature（真的），這樣的設計原因是「避免 routines 不小心改到重要的分支」。畢竟排程任務是自主運作的、沒有任何權限提示也沒人在旁邊盯著，如果它能直接寫 main，一個寫壞的 prompt 就可能讓一個正式的產品雞飛狗跳。

那怎麼解？兩種選擇：

  1. 接受預設行為：讓它推 claude/xxx 分支，然後你開 PR review，最建議的工作流，畢竟排程任務沒有人在旁邊看，先進到一個獨立分支，等你有空再 review 跟合併，相對安全很多。
  2. 打開 Allow unrestricted branch pushes：如果你真的想讓它直接推到 main，可以在 Edit routine 的 Permissions 區塊把對應 Repo 的「Allow unrestricted branch pushes」打開，這樣就沒有分支前綴的限制了，不過要小心你的 prompt 寫好寫滿，確保它不會做出你不想要的改動。

如果你真的就是要讓它直接推到指定分支，可以這樣做：

  1. 到 claude.ai/code/routines 點進你的 routine
  2. 按鉛筆圖示進入 Edit routine
  3. 找到下方的 Permissions 區塊
  4. 把對應 Repo 的「Allow unrestricted branch pushes」勾起來
  5. 存檔

Note
如果你的 main 是受保護的、設了 branch protection rule 的話，就算打開 Allow unrestricted branch pushes，GitHub 那邊還是會擋下來。這時候要嘛在 protection rule 加例外，要嘛還是讓它走 PR 流程比較實際。

排程跑在哪、會用到哪些東西？

除了分支推錯之外，還有幾個 Routines 相關的細節，第一次設排程很容易忽略，這邊一起講一下。

每次都是從 default branch 重新 clone

Routines 每跑一次，都是從你選的 Repo 的 default branch 重新 clone 一份來跑，不會延續上次的 working tree。所以你不能寫一個排程是「接續昨天那次跑到一半的事」，每次都要當作從零開始。

如果你的任務需要參考上次的狀態，可以把狀態寫到 Repo 的某個檔案 or 連接上第三方服務。

一次性排程不吃 daily cap

Routines 有 daily run cap（一天最多能跑幾次的限制），但 one-off 排程不會吃這個 cap，它只會吃你訂閱方案的一般用量。所以如果你想用排程當「兩週後幫我做某件事」這種一次性提醒，可以放心開很多個，不會把 daily cap 用爆。

管理 routines

設好的 routine 要管的話有兩條路。

CLI 端用前面提到的 /schedule list、/schedule update、/schedule run，做基本的列表、編輯、立刻觸發。

網頁端則是到 claude.ai/code/routines，點進每個 routine 的詳細頁，這邊可以做更多事：

  • Run now：立刻觸發一次，不等下一個排程
  • Repeats 開關：暫停 or 恢復排程，暫停不會刪掉 routine 本身
  • 編輯：改名稱、prompt、Repo、環境、Connectors、觸發類型
  • 刪除：移除這個 routine，過去跑過的 session 還會留在 session 列表

另外每次 routine 跑完都會在你的 session 列表開一個新 session，你可以點進去看 Claude 做了什麼、review 變更、開 PR、甚至接著對話繼續做下去。

總結

Claude Code 的排程功能講複雜也不複雜，主要就是搞清楚三種排程的差別、/schedule 怎麼用、然後記得預設只能推 claude/ 前綴分支這件事，剩下的就是依照你的需求調整觸發類型跟環境設定。

我自己後來的習慣是：日常的「每天跑一次某個小事」用 Routines 設 daily、需要存取本機檔案的 batch 任務用 Desktop scheduled task、想在當下 session 裡反覆 poll 某個狀態就用 /loop，三個各有各的場景。

至於那個一直推到 claude/xxx 分支的問題，知道是設計使然之後就釋懷了，反正讓它推 claude/ 分支然後 PR review 也比較安心。如果你跟我一樣有遇到這個困惑，希望這篇有幫你省下翻文件的時間哩～
