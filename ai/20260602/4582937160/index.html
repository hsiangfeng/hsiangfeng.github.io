Claude Code Auto Memory 一直佔用記憶體？清理與檢查完整教學

Note Auto Memory 會將你的 feedback 儲存到 ~/.claude/projects/<專案>/memory/ 底下，雖然不會佔用 RAM，但它卻會佔用你的 Context Window 跟磁碟空間。好消息是記憶檔通常只有 KB 等級，真正讓 ~/.claude/projects/ 膨脹到幾百 MB 甚至 GB 的兇手其實是同一個資料夾底下的 conversation jsonl。Claude Code 預設會用 cleanupPeriodDays（30 天）自動清掉舊的 session 檔，但如果你最近剛好跑了一個超大的 session，等 30 天可能有點久，可以用 find 把大檔案揪出來處理。

前言

前陣子朋友突然私訊我：「ㄟ，Ray，我電腦突然硬碟少了整整 300~500 MB 的空間耶？而且每次啟動 Claude 馬上就少了很多 Context，你有頭緒嗎？」

其實我大概知道原因，因為朋友這陣子正好瘋狂在使用 Claude Code 開發，第一直覺我就幫他看了一下 ~/.claude/projects/，果不其然，這個資料夾已經佔用了 300 多 MB 的空間，而這個東西其實就是 Claude Code 的 Auto Memory 機制導致，所以這一篇就要來聊一下這一塊。

另外，之前我有寫過一篇 Claude Code Auto Memory 是什麼？自動記憶功能完整教學 介紹 Auto Memory 的基本概念跟運作方式，這一篇就接續那一篇，來聊一下 Auto Memory 到底「佔」了什麼、什麼東西才是真的在吃你的磁碟跟 Context Window，以及怎麼處理。

Auto Memory 到底佔用了什麼？

我們先來聊一下 Auto Memory 到底佔了什麼哪些資源，簡單來說就是兩個：

資源佔多少為什麼會佔
Context WindowMEMORY.md 前 200 行或 25KB每次對話啟動時自動載入
磁碟空間通常只有 KB 等級記憶內容寫成 Markdown 檔存在 Home 目錄

Note 這邊要特別提醒一下，我這邊寫的「記憶體」並不是 RAM，而是 Context Window 跟磁碟空間。

那怎麼找到這個 MEMORY.md 呢？它其實就是你專案資料夾底下 memory/ 的索引檔，路徑是 ~/.claude/projects/<專案>/memory/MEMORY.md，裡面會列出你所有的記憶檔（通常是一些 Markdown 檔），每一條是一個索引，Claude 會根據這個索引去讀對應的記憶內容。

Claude Code Auto Memory 的記憶檔結構示意圖

Context Window：MEMORY.md 怎麼吃 token？

首先，Auto Memory 的記憶內容是以 Markdown 檔的形式存在磁碟上的，Claude Code 每次啟動對話的時候會自動把 MEMORY.md 裡面的內容載入到 Context Window 裡面，讓 Claude 在對話過程中可以參考這些記憶。

可是 MEMORY.md 並不是整份載入，而是有一個限制：

Auto Memory 只會載入 MEMORY.md 的前 200 行或前 25KB（取小的那個）到 Context Window 裡面。

也就是說，如果你的 MEMORY.md 超過 200 行或 25KB 的話，超過的部分根本不會被載入到 Context Window 裡面，這也是為什麼有些人使用 Claude 之後，發現自己明明曾經講過，Claude 它也說過它會記憶下來，但它實際上根本不記得了，因為它根本沒看到那些記憶。

這邊也來讓我們看一下實際的 MEMORY.md 內容會是長什麼樣子：

markdown
- [feedback_writing_style.md](./feedback_writing_style.md) - 不要在文案中大量使用「」和 emoji
- [feedback_pnpm.md](./feedback_pnpm.md) - 此專案使用 pnpm，不要用 npm
- [feedback_uiux_skill_for_ui_work.md](feedback_uiux_skill_for_ui_work.md) — 牽涉 UI／UX 的修改一律呼叫 `ui-ux-pro-max:ui-ux-pro-max`

其實每一條都是很單純的一句話索引，加上連結指向實際的記憶內容檔案，這樣的管理方式對於 Context Window 的佔用是非常友善的，而實際的細節只會在對話中需要的時候才會被 Claude 讀取，這樣就不會讓 Context Window 被大量細節佔滿，因此以前面講的 feedback_uiux_skill_for_ui_work.md 它就會像這樣：

md
---
name: UI 相關修改必須透過 ui-ux-pro-max skill
description: 任何牽涉 UI／UX 的調整、修改、新增，都要用 Skill tool 呼叫 ui-ux-pro-max:ui-ux-pro-max
type: feedback
originSessionId: bd73d7ce-893d-423f-2344dd123dw
---
只要這次修改牽涉 **UI／UX**（畫面排版、互動行為、視覺呈現、元件設計、Tailwind class、元件結構等），**必須透過 Skill tool 呼叫 `ui-ux-pro-max:ui-ux-pro-max`** 進行，不要自己直接改。

**Why:** Ray 明確指示「只要跟 UI 扯到你都要呼叫 UIUX SKILL Pro 去調整跟修改或者新增」。交給專門的 skill 處理可確保一致性與品質。

**How to apply:**
- 評估這次修改是否牽涉 UI（任何 .vue 檔的 template / style 區塊、Tailwind class 變更、新元件、互動流程調整都算）
- 牽涉 UI → 呼叫 `Skill({ skill: 'ui-ux-pro-max:ui-ux-pro-max', args: '<簡述要改什麼' })`
- 不牽涉 UI（僅 .ts service／types／composable 邏輯）→ 可自行處理
- 邊界案例（例如 composable 影響 UI 行為但本身不是 UI）：以「最終會不會改到 .vue」為判斷基準
- 呼叫前先準備清楚的需求描述（要改哪個畫面、目前的問題、期望的方向），這個 skill 才能產出有用的建議

這邊你會看 name、description、type、originSessionId 這些 metadata，其實你會發現跟 SKILL 的格式很像，你可以把 Auto Memory 看成一個另一種特殊的 Skill，Claude 會根據這些 metadata 來判斷這條記憶是什麼類型、來自哪個 session，以及它的內容是什麼，因此 description 的描述是非常重要的。

如果你發現你的 Claude Code 總是沒辦法正確讀取你要求的記憶內容，那很可能就是因為 description 寫得不夠清楚，Claude 看不懂它是什麼東西，自然就不會去讀取它了。

怎麼讓 MEMORY.md 瘦下來？

如果你發現 MEMORY.md 已經膨脹到接近 200 行，有幾個方式可以瘦身：

  1. 把細節拆到 Topic 檔案：MEMORY.md 只留索引，每條一行帶連結就好，實際內容寫在獨立的 .md，Claude 需要時會自己去讀
  2. 刪掉過時的記憶：直接在對話中跟 Claude 講「忘記之前關於 X 的規則」，or 自己手動到 ~/.claude/projects/<專案>/memory/ 底下刪掉對應檔案
  3. 手動編輯 MEMORY.md：它就是普通的 Markdown，自己用編輯器打開整理就好

Note 通常大多都是請 AI 整理，而不是自己動手，畢竟 Auto Memory 的重點就是讓 AI 來幫你管理這些記憶內容，自己動手就失去意義了？但理解一下這一塊的結構跟運作方式，對於後續的使用跟排除障礙還是很有幫助。

磁碟空間：真正膨脹的兇手不是 Auto Memory

接下來是磁碟這一塊，這也是大部分人會誤解 Auto Memory 的地方。

我們先看 Auto Memory 本身佔多少。以我自己的專案為例，記憶資料夾長這樣：

bash
du -sh ~/.claude/projects/-Users-rayxu/memory/
# 52K

用 du 看 ~/.claude/projects/<專案>/memory/ 的磁碟用量

整個 memory/ 加起來才 52KB，這個大小基本上你一輩子都用不完。

那...一整個 projects 資料夾呢？

bash
du -sh ~/.claude/projects/
# 339M

用 du 看 ~/.claude/projects/ 的磁碟用量

意外吧？竟然高達 339 MB！所以真正佔用的人到底是誰哩？

答案就是 conversation jsonl — Claude Code 把你的每一次對話都存成一個 .jsonl 檔，每一行是一個 JSON 物件，包含你的 prompt、Claude 的回應、工具呼叫、工具結果，全部寫下來。長對話跟有大量檔案讀寫的 session，單一檔案就可能 MB 等級。

實際看一下：

bash
ls -lh ~/.claude/projects/*/*.jsonl | sort -k5 -h | tail -5
# 5.7M  -....jsonl
# 5.9M  -....jsonl
# 6.2M  -....jsonl
# 6.2M  -....jsonl
# 8.1M  -....jsonl

可以看到最大的單一對話檔已經到 8 MB 了。

用 du 看 ~/.claude/projects/ 底下各專案的磁碟用量

所以下次你看到 ~/.claude/projects/ 一直膨脹，先別怪 Auto Memory，去 du 一下就會發現真正的兇手是這些 .jsonl。

所以該怎麼自動清理？認識 cleanupPeriodDays

ok，其實有一個方式可以解決這件事情，本身 Claude Code 就有內建一個自動清理機制，預設就會把超過 30 天的 session 檔在啟動時自動刪掉，這個設定叫做 cleanupPeriodDays。

NotecleanupPeriodDays 只會清 conversation 紀錄，不會動到 Auto Memory 的記憶檔。你之前寫的記憶都不會被它刪掉，可以放心。

如果你想改保留天數，在 ~/.claude/settings.json 加上這條：

json
{
  "cleanupPeriodDays": 7
}

這樣 Claude Code 每次啟動時就會把超過 7 天的 jsonl 都清掉（最少可以設 1 天，設 0 會出現錯誤哦）。

那如果你完全不想保留任何對話紀錄（隱私考量 or 純粹想省空間），可以直接停用 transcript 寫入：

bash
export CLAUDE_CODE_SKIP_PROMPT_HISTORY=1

把這行加到你的 ~/.zshrc 或 ~/.bashrc，從此 Claude Code 就不會再寫 conversation jsonl 了。代價是你也不能再用 claude --resume 回到之前的對話，魚與熊掌就看你怎麼選擇了。

手動清理大檔案

雖然 cleanupPeriodDays 會自動清理，可是如果你最近兩週剛好跑了一個超大的 session（例如連續開 subagent 解 bug），等 30 天有點久。

這時候可以用 find 把大檔案揪出來：

bash
find ~/.claude/projects -name "*.jsonl" -size +50M -exec ls -lh {} \;

這條指令會找出 ~/.claude/projects/ 底下所有超過 50 MB 的 jsonl 檔。確認過內容你不需要保留之後，再用 -delete 把它們刪掉：

bash
find ~/.claude/projects -name "*.jsonl" -size +50M -delete

Note 刪掉某個 session 的 jsonl 之後，那個 session 就無法用 claude --resume 救回來了。如果你還想保留近期的對話紀錄，刪之前最好先 ls -lh 確認一下檔案的修改時間，挑你真的不會再回頭看的那幾個就好。

順便看哪些專案資料夾最肥

~/.claude/projects/ 底下每個資料夾對應一個你曾經用 Claude Code 開啟的目錄，可是專案如果你早就不維護了，那它對應的資料夾也沒必要留。可以這樣看哪些資料夾最肥：

bash
du -sh ~/.claude/projects/*/ | sort -h | tail -10

確認哪幾個專案是你已經不會再 resume 的，整個資料夾直接刪掉就好。

那資料夾底下還有別的東西嗎？

除了 conversation jsonl 跟 memory/ 之外，每個專案資料夾底下還可能有一些以 UUID 命名的子資料夾（沒有 .jsonl 副檔名的純資料夾），那些是 subagent 跑任務時用的暫存空間。一般來講不需要動它們，cleanupPeriodDays 也會一起清。

如果你有用 Worktree 跑 subagent，舊的 worktree 也會在啟動時被自動清掉，同樣由 cleanupPeriodDays 控制。

希望這篇有讓你對 Auto Memory 的「佔用」有更近一步的認識哩～

下次磁碟爆滿的時候，記得先 du 看一眼是哪邊真的在吃，不要冤枉了 Auto Memory。
