Claude Code Hooks 是什麼？自動跑 Lint、擋 rm -rf、SessionStart 注入 Context 完整教學

Note Claude Code Hooks 是 Claude Code 生命週期裡的「自動觸發點」，會在你期望的時候自動觸發你寫的 Shell 指令。而常用的生命週期有 PreToolUse（工具執行前）、PostToolUse（工具執行後）、UserPromptSubmit（每次送 prompt 前）、SessionStart（session 開啟時）。你需要將相關設定寫在 settings.json 的 hooks 區塊就可以了。

前言

用 Claude Code 開發一段時間後，你應該有遇過類似的狀況，也就是每次叫它改完檔案還要再請它跑一次 Prettier 跟 ESLint（當然也可以手動自己執行 npm run format），但人總是會偷懶，想要在比較放飛的時候開個 --dangerously-skip-permissions，但開了又會怕它跑了 rm -rf 之類的指令；又或者 Context 被 Compact 之後，它就忘了專案的某些重要約定，又要重新講一次 QQ...

所以這篇就要來介紹與分享一下 Claude Code 中很重要的技巧，也就是 Hooks，讓你可以在 Claude Code 的生命週期裡「掛鉤」你想要的自動化動作，確保它在對的時間點幫你做對的事。

另外，之前我有寫過 Claude Code 哪些檔案該 Commit？CLAUDE.md、Settings、Rules、Skills 完整解析，裡面有大致提到 .claude/settings.json 的層級概念，這篇就接續那一篇，把 hooks 這個藏在 settings 裡面的進階功能拉出來獨立詳細介紹一下。

Hook 跟 Skill、CLAUDE.md、Rules 差在哪？

在開始講 hook 怎麼設之前，先讓我們快速回顧一ㄒ它跟其他「教 Claude 做事」的機制差別，這樣會比較清楚彼此之間到底在幹嘛。

機制由誰執行何時觸發適合的場景
CLAUDE.mdClaude每次對話載入專案規範、命名慣例、工作流程
RulesClaude啟動時 or 處理對應路徑檔案時載入模組化的規則，可依路徑生效
SkillClaudeClaude 自己判斷該用時觸發把一整段流程包成可重用的能力
Hook你的 ShellClaude Code 生命週期事件強制執行的自動化動作

如果上面表格看完還是很難懂的話，那你可以這樣理解

  1. CLAUDE.md 跟 Rules 是「規則」
  2. Skill 是「能力」，這三個都是 Claude 自己看完之後再決定要不要照做。
  3. Hook 不一樣，hook 是 Claude Code 本體幫你執行的，Claude 沒有選擇權，到了那個時機就一定會執行的事情。

這邊也簡單舉一個例子，假設你在 CLAUDE.md 裡面寫「每次改完檔案請跑 Prettier」，Claude 大部分時候會聽，可是偶爾會忘記（畢竟 Context 可能爆掉了？！），如果你改用 PostToolUse hook 設定「每次 Edit 或 Write 之後自動跑 Prettier」，那不管 Claude 想不想跑，反正你的 Shell 指令一定會被執行。

所以 hook 適合的的場景是...

我絕對不能容忍 Claude 漏掉的事

像是 lint、format、安全防護這些，而比較模糊、需要 Claude 判斷的事還是寫在 CLAUDE.md 比較合適。

Hook 的設定檔長什麼樣子？

Hook 的設定都是寫在 settings.json 的 hooks 區塊裡面，而它的結構是長這樣：

json
{
  "hooks": {
    "事件名稱": [
      {
        "matcher": "篩選條件",
        "hooks": [
          {
            "type": "command",
            "command": "你要跑的 Shell 指令"
          }
        ]
      }
    ]
  }
}

外層的 key 是「事件名稱」（例如 PreToolUse、PostToolUse），裡面包一個陣列，陣列每一項是一組設定，包含 matcher（用來過濾這組要在什麼狀況下啟動）跟 hooks（實際要跑的指令）。

如果你之前看過 Claude Code Status Line 設定教學，會覺得這個結構滿熟悉的，因為它跟 Status Line 一樣都是「Claude Code 餵資料給你的 Shell，你決定怎麼回應」。

設定檔放在哪？

跟其他 Claude Code 設定一樣，hooks 也有層級分別：

位置範圍該不該 commit
~/.claude/settings.json你所有的專案不會出現在專案 repo
.claude/settings.json整個團隊該 commit
.claude/settings.local.json只有你個人不該 commit（已在 .gitignore）

簡單來講，全團隊都應該裝的 hook（例如 lint）寫在 .claude/settings.json、個人偏好（例如桌面通知）寫在 ~/.claude/settings.json、跟敏感路徑相關的本機規則寫在 .claude/settings.local.json。

Hook 事件有哪些？

聊完設定檔的結構，接下來就是這篇的重頭戲，也就是「到底有哪些事件可以使用？」

老實講，Claude Code 提供的事件超過 20 個，全部寫出來會把人嚇跑，所以這邊我以我自己「實務上會用到的頻率」分類列給你看。

下面這張圖是 Claude Code 一輪典型 session 的生命週期，每個箭頭上面標的就是會觸發的事件，可以對照看 hook 大概在哪個位子會被呼叫：

Claude Code Hooks 事件觸發時機示意圖

最常用的幾個

事件觸發時機經典用途
PreToolUse某個工具要執行之前擋掉危險指令、改寫參數
PostToolUse某個工具成功執行之後自動 format、跑 lint
UserPromptSubmit你按下 Enter 送出 prompt 時注入額外脈絡、過濾敏感資訊
SessionStart開啟 or 恢復 session 時載入專案脈絡、輸出 git 狀態
StopClaude 回完話準備停手時驗證任務有沒有真的完成
NotificationClaude 需要你注意時桌面通知、語音提示

這六個算是屬於最常被用到的事件，許多實戰範例都會繞著它們轉。

其他比較少用但值得知道的

事件觸發時機經典用途
PermissionRequest權限提示要彈出時自動同意特定動作
PermissionDenied自動模式擋掉某個工具呼叫時紀錄被擋的指令
PostToolUseFailure某個工具執行失敗時失敗時通知 or 自動 retry
SubagentStopSubagent 跑完任務時驗證 subagent 結果

SessionEndsession 關閉時清理暫存檔、寫日誌
ConfigChange設定檔在 session 中被改稽核設定變動
CwdChanged / FileChanged工作目錄 / 監看的檔案有變動重新載入環境變數（搭配 direnv）

看到現在為止，應該可以知道 hook 幾乎涵蓋了 Claude Code 跑起來會經過的所有節點，幾乎你想得到的「我希望某件事發生時自動跑某個指令」都做得到，概念就跟寫成是的「生命週期事件」一樣。

matcher 怎麼寫？

接下來我們總要來介紹一下該怎麼撰寫 hook 的條件了。

前面有看到 hook 範例是長這樣：

json
{
  "hooks": {
    "事件名稱": [
      {
        "matcher": "篩選條件",
        "hooks": [
          {
            "type": "command",
            "command": "你要跑的 Shell 指令"
          }
        ]
      }
    ]
  }
}

matcher 決定「這組 hook 在什麼狀況下要啟動」，而它要比對什麼東西會隨事件而不同。最常用的 PreToolUse / PostToolUse 比對的是「工具名稱」，所以最直覺的寫法就是把工具名直接丟進 matcher 欄位。

舉個實際的例子，下面這個 hook 是「Claude 每次要跑 Bash 之前，在終端機上顯示一段提醒」：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -n '{ systemMessage: \"Bash 要執行了\" }'"
          }
        ]
      }
    ]
  }
}

接下來畫面就可以看到 Bash 要執行了 的提醒了。

Bash 要執行了 的提醒示意

matcher 的值就是工具名 Bash，意思是「只在 Claude 用 Bash 工具時這個 hook 才會跑」。

Note 為什麼這邊要用 jq 不用 echo？因為 Claude Code 跟 hook 之間是用「JSON 格式」對話的，如果你直接寫 echo "Bash 要執行了"，Claude Code 收到的就只是一段沒結構的純文字，它看不懂你要它幹嘛，就只會把這段話當成 hook 的執行紀錄收掉（忽略），Claude 永遠看不到。而 jq -n '{ systemMessage: "Bash 要執行了" }' 是在產出一段 JSON，等於是跟 Claude Code 講「這個欄位叫 systemMessage，內容請你塞給 Claude 看」，Claude Code 看到這種它認得的格式，才會真的把訊息餵進去。

如果你想一次盯多個工具，中間用 | 隔開就好。例如下面這個 hook 是「Claude 每次用 Edit 或 Write 改完檔案，自動跑 Prettier」：

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

看到這邊我知道 command 有點複雜，所以我來拆解解釋一下：

  1. Claude 用 Edit / Write 改完檔案的當下，Claude Code 會把一段 JSON 丟給這個 hook，裡面大概長這樣 — { "tool_input": {"file_path": "/path/to/file.ts" }, ... }，告訴你它剛剛改了哪個檔。
  2. 接著 jq -r '.tool_input.file_path' 就是把這段 JSON 拆開，從裡面挖出 tool_input.file_path 這個欄位的值（也就是檔案路徑），-r 的意思是「給我純文字就好，不要有雙引號」。
  3. |（pipe）你就把它當作大隊接力的接力棒，意思是「把上一個指令的結果，當成下一個指令的輸入餵過去」。
  4. xargs 負責把接過來的東西「接在下一個指令的後面當參數」，所以實際跑出來會變成 npx prettier --write /path/to/file.ts。
  5. npx prettier --write 就是呼叫 Prettier 把這個檔案 format 一遍。

Noteprettier 是一款程式碼格式化工具，--write 是它的參數，意思是「直接改檔案」，如果你不想改檔只想看差異，可以換成 --check。如果你的專案不是用 Prettier 而是 ESLint，把 npx prettier --write 換成 npx eslint --fix 就好。

基本上，後面範例你只要看到 jq -r '.xxx' 就是再做這些事情，也就是：

jq -n 將 Claude Code 丟過來的 JSON 解析出來，抓你想要的欄位值，然後丟給下一個指令用。反之前面那個 jq -n 才是反過來「從零產生」一段 JSON 餵回去給 Claude Code。

最後，如果你想要「不管什麼工具都觸發」，那就把 matcher 留空字串：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_name' >> ~/.claude/all-tools.log"
          }
        ]
      }
    ]
  }
}

這個 hook 會在 Claude 用任何工具之前把工具名追加到 log 檔，等於是把整個 session 的工具使用紀錄起來。

到這邊 95% 的場景已經夠用了，要看每個工具叫什麼名字，可以參考之前我寫的 Claude Code 內建 Tools 在做什麼？看懂它怎麼選工具，裡面把每個工具的名字都列出來了。

進階：要篩 MCP 工具就要用 regex

剩下 5% 的情境就是當你要比對 MCP server 提供的工具時。MCP 工具的命名格式長這樣：mcp__<server>__<tool>，舉例來說 GitHub MCP server 提供的搜尋工具叫做 mcp__github__search_repositories。

如果你想一次盯某個 MCP server 提供的所有工具，可以用 regex 的 .* 配對任何字元：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "mcp__github__.*",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'GitHub MCP tool called' >&2"
          }
        ]
      }
    ]
  }
}

.* 是 regex 的「任何字元」，所以 mcp__github__.* 的意思就是「mcp__github__ 開頭，後面接什麼都符合」。Claude Code 判斷要不要當 regex 的規則很單純：matcher 只要不只由「字母、數字、_、|」這四種組成，就會走 JavaScript regex 模式；反之就當完整字串比對。所以前面 Bash、Edit|Write 那種寫法不會被誤判成 regex。

不是所有事件都比工具名

前面講的都是跟工具有關的事件，可是有些事件 matcher 比對的不是工具名，而是「事件本身的某個分類」。

最常見的例子是 SessionStart，它比對的是「session 是怎麼開的」。下面這個 hook 是「Context 被 Compact 後 session 重建時，重新提醒 Claude 專案規則」：

json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo '提醒：套件管理用 pnpm 不要用 npm'"
          }
        ]
      }
    ]
  }
}

看到 echo '提醒：套件管理用 pnpm 不要用 npm'你應該會充滿疑問想著...

「前面你不是說 PreToolUse 的 echo 沒用嗎？怎麼這邊又這樣用？」

這是因為 SessionStart 跟 UserPromptSubmit 是兩個比較特殊的事件，hook 輸出到畫面的內容會被 Claude Code 自動當成「要塞給 Claude 的額外脈絡」收進去，所以直接 echo 就會被 Claude 看到。其他事件（像前面的 PreToolUse）才要走 jq -n '{systemMessage: ...}' 這種 JSON 格式跟 Claude Code 對話。

matcher: "compact" 限定「只在 Compact 後重建 session 時跑」，第一次開 Claude Code 或 resume 舊 session 都不會觸發。SessionStart 的可選值有 startup（第一次開）、resume（恢復舊 session）、clear（清空 context）、compact（壓縮後）。

Notification 也是類似邏輯，它比對的是「通知類型」，常見的有 permission_prompt（Claude 在等你授權）、idle_prompt（Claude 沒事在等下一句），另外還有 auth_success、elicitation_dialog、elicitation_complete、elicitation_response 等比較少用的。每個事件 matcher 對應什麼，可以查 官方 hooks reference，這邊就不一一列了。

進階：用 if 連工具參數也篩

光是 matcher 還不夠精確，因為它只能比對工具名。如果你想說「只在 Claude 跑 git push 的時候才觸發 hook，其他 Bash 指令不要管」，那你需要更細的 if 欄位：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "if": "Bash(git push *)",
            "command": "echo 'Blocked: 請走 PR 不要直接 push 到 main' >&2; exit 2"
          }
        ]
      }
    ]
  }
}

這個範例做的事情是：當 Claude 想跑任何 git push 開頭的 Bash 指令時，hook 就會輸出一段擋下的訊息然後 exit 2 把這個動作擋掉。可是如果 Claude 想跑的是 git status 或 npm test，hook 根本不會被叫起來。

兩個篩選欄位的差別是：

  • matcher: "Bash" 是第一層粗篩，「只看 Bash 工具」。
  • if: "Bash(git push *)" 是第二層細篩，連 Bash 裡面實際要跑的指令也比對，* 是「後面接什麼都行」的萬用字元，跟 permissions 設定用的是同一套語法。

它最大的好處是「只有真的命中條件時才會把 hook 跑起來」，比起每次都把 hook 叫起來再用 jq 解析參數判斷，效能差很多。

Noteif 欄位是後來才加的，比較舊版本的 Claude Code 可能不認得這個欄位，如果你寫了 if 但 hook 行為跟沒寫一樣，先用 claude --version 確認版本然後更新看看。

Hook 怎麼跟 Claude Code 溝通？

理解 matcher 之後，再來看 hook 跟 Claude Code 是「怎麼互傳訊息」的。先看一個最小範例：

bash
#!/bin/bash

# 1. 將 Claude Code 丟過來的 JSON 儲存到 INPUT 變數裡
INPUT=$(cat)

# 2. 輸出一段訊息給 Claude Code 看，因此不會直接在終端機上看到，而是被 Claude Code 當成 hook 的回應收掉
echo "Claude 你好"

# 3. 結束 script，回傳 0 代表「沒事，繼續」，如果你寫 `exit 2` 就代表「有事，擋下這個動作」
exit 0

舉個 PreToolUse 的例子，當 Claude 要跑 npm test 時你的 hook 從 stdin 會收到這樣的 JSON：

json
{
  "session_id": "abc123",
  "cwd": "/Users/ray/myproject",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm test"
  }
}

Note 很多人寫 hook 第一次踩雷就在這邊：以為 exit 1 會擋，但只有 exit 2 才會擋，其他 non-zero 都不會。

用 JSON 回應做更細的控制

只能用 exit 0 跟 exit 2 嗎？不對，還可以做更細的控制（例如自動同意 or 直接改寫 Claude 想跑的指令），方法是 exit 0 然後輸出一段 JSON 到 stdout：

json
{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "請改用 pnpm 不要用 npm"
  }
}

permissionDecision 在 PreToolUse 有四個值：

  • allow：跳過權限提示直接放行
  • deny：擋下來並把理由餵給 Claude
  • ask：照原本的權限流程跳出提示問你
  • defer：hook 沒有意見，回到原本的權限流程（效果等同沒寫這個欄位）

Note Exit 2 跟 JSON 不要混用，Claude Code 看到 exit 2 就會直接讀 stderr 當理由，stdout 的 JSON 會被忽略。要嘛 exit 2 配 stderr，要嘛 exit 0 配 JSON，選一邊就好。

五種 Hook Type

前面範例都是用 type: "command"，這是最常見的形式，可是 Claude Code 還有另外四種：

Type怎麼跑適合的場景
command跑 Shell 指令大多數情境，靈活又簡單
httpPOST event 資料到指定 URL接遠端服務做共用稽核、組織內驗證
mcp_tool呼叫已連線的 MCP server 的某個工具把現成 MCP 工具當成 hook 邏輯
prompt把資料丟給 Claude 模型自己判斷需要語意判斷而不是寫規則的情境
agent派一個 subagent 帶工具去驗證要實際讀檔 or 跑指令才能判斷的情境

90% 的人寫 hook 都會停在 command，可是後面三種（特別是 prompt）滿值得認識的，因為它讓 hook 能做的事情從「死規則」變成「有判斷力的守門員」。

舉個 prompt hook 的例子，假設你想在 Claude 講完話之後，讓另一顆模型檢查它有沒有真的把任務做完：

json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "檢查剛剛 Claude 是否真的完成了所有要求的任務。如果沒有，回覆 {\"ok\": false, \"reason\": \"還缺什麼\"}。"
          }
        ]
      }
    ]
  }
}

這個 hook 觸發時 Claude Code 會自動呼叫一顆模型做單輪 yes/no 判斷，如果回覆判斷沒做完，主 Claude 就會繼續做下去，相當於免費幫你裝了一個品管站。

實戰範例

聊完概念接下來看實際範例。下面這些是我認為日常最常用到、CP 值最高的幾個。

自動 format 改過的檔

每次 Claude 用 Edit 或 Write 改完檔之後，自動跑 Prettier 把格式修好：

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

這邊用 jq 把 stdin 餵進來的 JSON 解析出 file_path 再丟給 Prettier，整個一行解決。如果你的系統沒有 jq，Mac 可以 brew install jq，Ubuntu 是 apt install jq。

如果你的專案不是用 Prettier 而是 ESLint，把 npx prettier --write 換成 npx eslint --fix 就好。

這個 hook 跟專案的 lint / format 規則綁在一起，建議寫在 .claude/settings.json 跟著 commit 給全團隊吃。

擋掉 rm -rf 之類的危險指令

把 hook 寫成一個獨立的 Shell script 會比較好維護，請你先建立 .claude/hooks/block-rm.sh：

bash
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# 只擋真正的 rm -rf / 或 rm -r /，後面接其他路徑（例如 /tmp/foo）放行
if echo "$COMMAND" | grep -qE 'rm[[:space:]]+-rf?[[:space:]]+/([[:space:]]|$)'; then
  echo "Blocked: 不允許從根目錄 rm -rf" >&2
  exit 2
fi

exit 0

這邊用 POSIX 字元類別 [[:space:]] 不用 \s，是因為 \s 是 GNU grep 的擴充，macOS 預設的 BSD grep 不一定支援，會讓比對整個失效。[[:space:]] 是 POSIX 標準寫法，Mac、Linux 各家 grep 都認得。

regex 那段 (/([[:space:]]|$)) 強制 / 後面要接空白或行尾，所以：

  • rm -rf /：會擋掉
  • rm -r /：會擋掉
  • rm -rf /tmp/foo：不會擋掉，因為後面接了路徑不是空白或行尾

這個 pattern 故意寫得很保守，只擋字面上的 rm -rf / 跟 rm -r /。如果你想擋更兇一點（例如 rm -rf /*、rm -rf ~、rm -rf $HOME 這些變形），照同樣模式加幾條 grep -qE 串起來，每多一個 case 就 exit 2 即可。

然後打開在 .claude/settings.json 註冊：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/block-rm.sh"
          }
        ]
      }
    ]
  }
}

$CLAUDE_PROJECT_DIR 是 Claude Code 內建的環境變數，會自己變成專案根目錄（也就是你在哪邊呼叫專案，就會變成哪邊）。

Note 這個 hook 即使在 --dangerously-skip-permissions 模式下也會生效。PreToolUse 是在權限模式判斷之前就跑，所以 hook 是少數可以真正擋住「繞過權限」模式的機制。

這種安全防護建議寫在 .claude/settings.json commit 進 repo，全團隊都有保護。如果你只想自己保險，放 ~/.claude/settings.json 也行。

rm -rf 被擋掉的示意

Compact 之後重新注入專案脈絡

Context 一旦被 Compact，Claude 常常會忘記某些重要約定。可以用 SessionStart 加 compact matcher，在每次壓縮完之後重新塞一段提醒：

json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo '提醒：套件管理用 pnpm 不要用 npm；commit 前一定要跑 pnpm test。'"
          }
        ]
      }
    ]
  }
}

SessionStart 跟 UserPromptSubmit hook 輸出到 stdout 的內容會被當成額外脈絡加進 Claude 的 context，這是它們跟其他事件不太一樣的地方。

如果你想塞動態內容（例如目前的 git 分支、未 commit 的檔案），把 echo 換成你的 script 就好：

bash
#!/bin/bash
BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null)
DIRTY=$(git status --porcelain | wc -l)
echo "目前分支：$BRANCH，未 commit 檔案數：$DIRTY"

這個 hook 寫在 .claude/settings.json 比較適合，因為「用 pnpm」「commit 前要跑 pnpm test」這種是專案內部約定，commit 進去其他人 clone 下來也直接吃得到，不會每個人都要自己設一次。

Mac 桌面通知

Claude 在等你授權 or 等下一個 prompt 的時候通知一下，這樣你可以放心去做別的事：

json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude Code 在等你\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

Linux 把 osascript 換成 notify-send，Windows PowerShell 用 MessageBox.Show，邏輯都一樣。

Note Mac 第一次跑可能沒反應，因為 osascript 是透過內建的 Script Editor app 發通知，要先到「系統設定 → 通知」找到 Script Editor 把通知權限打開才會出現。如果連 Script Editor 都沒看到，先在 Terminal 跑一次 osascript -e 'display notification "test"'，它就會出現在通知設定的清單裡。

桌面通知純粹是個人偏好，寫在 ~/.claude/settings.json 就好，不用 commit 進 repo。

把 Claude 的每個動作通知到 Discord

如果你不想開著終端機盯 Claude，或想把 Claude 在跑什麼丟到 Discord channel 跟團隊共享，可以用 PreToolUse hook 串 Discord webhook。

先到 Discord channel 設定產一個 webhook URL，然後在 shell profile 裡 export 出來（不要寫死在 script 裡，commit 進 repo 等於外洩）：

bash
export DISCORD_WEBHOOK_URL="https://discord.com/api/webhooks/你的-webhook-id/你的-token"

接著建立 ~/.claude/hooks/discord-notify.js：

javascript
#!/usr/bin/env node
const WEBHOOK_URL = process.env.DISCORD_WEBHOOK_URL;

async function main() {
  // 讀進 Claude Code 從 stdin 餵進來的 JSON
  const chunks = [];
  for await (const chunk of process.stdin) chunks.push(chunk);
  const input = JSON.parse(Buffer.concat(chunks).toString());

  const toolName = input.tool_name || "未知工具";
  const toolInput = input.tool_input || {};

  // 組一個 Discord embed，讓訊息漂亮一點
  const embed = {
    color: 0x7289da,
    author: { name: `🔧 ${toolName}` },
    description: toolInput.file_path
      ? `\`${toolInput.file_path}\``
      : toolInput.command
        ? `\`\`\`bash\n${toolInput.command}\n\`\`\``
        : "",
    timestamp: new Date().toISOString(),
  };

  await fetch(WEBHOOK_URL, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ embeds: [embed] }),
  });
}

main().catch(console.error);

最後在 ~/.claude/settings.json 註冊（用個人 settings 讓所有專案都生效）：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "node ~/.claude/hooks/discord-notify.js"
          }
        ]
      }
    ]
  }
}

matcher: "" 是「不管什麼工具都通知」，這樣 Claude 不管要 Read、Edit、Bash 你都會在 Discord 看到一則訊息。如果你想根據工具種類顯示不同 emoji、顏色、欄位，照這個結構往下擴就好。

這整個 hook 是個人通知用、webhook URL 也是你私人的，寫在 ~/.claude/settings.json 個人層就好，不用 commit。

Note Webhook URL 一定要透過環境變數讀，不要寫死在 script 裡。萬一你真的想跟團隊共用一個 webhook，hook 本身可以放 .claude/settings.json，但 webhook URL 還是只放在各自的 shell profile 或 .claude/settings.local.json（後者不會被 commit）。

Log 所有 Bash 指令

如果你想稽核 Claude 到底跑了哪些 Bash 指令，可以在 PostToolUse 加個一行：

json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' >> ~/.claude/bash.log"
          }
        ]
      }
    ]
  }
}

每跑一次 Bash 指令就會被追加到 ~/.claude/bash.log，事後翻很方便。這個是個人稽核用的，寫在 ~/.claude/settings.json 就好。

多個 hook 怎麼合併？

當你同一個事件設了好幾個 hook，它們會全部平行執行，不會誰擋誰，最後 Claude Code 把所有結果合併。PreToolUse 的權限決定採「最嚴格者勝」：deny 蓋過 ask、ask 蓋過 allow。

舉個實際的例子，你可以同時設「Log 所有 Bash 指令」跟「擋掉 rm -rf」兩個 hook：

json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r .tool_input.command >> ~/.claude/bash.log"
          },
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/block-rm.sh"
          }
        ]
      }
    ]
  }
}

當 Claude 試圖跑 rm -rf /tmp/build 時，兩個 hook 會同時跑，log 會被寫進去，rm 會被擋掉。Log 不會因為 rm 被擋就漏記，因為兩個是各自獨立執行的。

怎麼確認 hook 有跑？

設好 hook 卻不確定到底有沒有觸發？Claude Code 提供兩個方式可以查。

第一個是 /hooks 指令，在 Claude Code 裡面直接打 /hooks 會跳出目前已設定的 hook 清單，按事件分組，每個都可以點進去看 matcher 跟 command。/hooks 是唯讀的，要改 hook 還是要編輯 settings.json。

/hooks 指令的介面示意

第二個是 debug log，啟動時加 --debug-file /tmp/claude.log：

bash
claude --debug-file /tmp/claude.log

然後另一個視窗開 tail -f /tmp/claude.log，每次 hook 觸發都會在 log 看到完整的 stdout、stderr、exit code。如果你只想看 hooks 相關的訊息不想被其他 debug 噪音淹沒，可以用 claude --debug hooks 加 filter。
