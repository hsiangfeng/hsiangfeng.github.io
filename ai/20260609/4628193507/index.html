AI 搜尋實際撈走你文章的多少內容？拆 Claude 的 WebSearch 與 WebFetch 規格

前言

之前寫過一篇 SEO 已經不夠了嗎？認識 AEO 答案引擎優化 ，寫完之後我突然有一個想法，也就是：AI 在做網路搜尋的時候，到底會撈走我部落格多少內容？整篇一字不漏？前面 500 字？還是只看到標題就跳走？所以就稍微研究一下，然後這篇就把 Anthropic 官方的 web_search 跟 web_fetch tool 文件翻一遍，順便拿自己的部落格實測一輪，最後整理這對怎麼寫文章、寫文案有什麼具體的影響吧。

先講清楚，「搜尋」跟「撈取」是兩件事

在我們開始往下之前，我認為必須要先把一件事情講清楚，也就是 AI 在做查詢時，「搜尋」 跟 「撈取網頁內文」 不是同一件事，它是兩件事情，而且是分層了兩階段來處理。

  • 第一階段 （WebSearch）：AI 拿著你的問題去呼叫 WebSearch 工具（去查搜尋引擎），這時候回來的會是一份搜尋結果清單，每一筆資料都有網址、標題之類的資訊，可是不包含網頁的整篇內文。
  • 第二階段 （WebFetch）：AI 看完搜尋結果清單之後，開始決定哪幾筆是它值得深入去看，接著再對那幾個網址各使用 WebFetch 工具發一次「把這個頁面的內文給我」的請求，這時候才會真的把內文撈回來。

嗯？你是不是好奇為什麼它不是直接在搜尋階段就把內文一起撈回來？

其實最大問題還是在於 Context （上下文）的限制上，你可以思考一下，如果它每次搜尋就把 10 筆的搜尋結果＋內文都塞進來，那想當然你可以使用的 Context 就很容易爆掉，然後接著你就會開始抱怨 AI 很笨、不記得剛剛的提問（畢竟 Context 爆掉了）。所以它才會把這兩件事分成兩階段，先搜尋拿到一個「清單」，然後再決定要從清單裡面挑哪幾筆去撈內文。

這邊我也順手畫一張流程圖，把兩階段的時序串起來看一眼會更直觀：

AI 搜尋流程圖

了解兩階段的差異之後，接下來就分別來看每個階段實際撈到了什麼。

WebSearch 撈到了什麼？

直接看 Anthropic 官方 web_search tool 文件，每筆搜尋結果裡面只有四個欄位：

  • url：網頁的網址
  • title：網頁的標題
  • page_age：網頁上次更新時間
  • encrypted_content：加密內容，這個不是給 AI 讀文章用的，是多輪對話時拿回來做 citation 用的

是的，就這樣，就這四個。

AI 在搜尋階段（WebSearch）看到的就是「網址、標題、上次更新時間」這三筆人類可讀的資訊，沒有任何內文。

那如果 AI 引用你的內容呢？也有上限

每個 web_search_result_location 的 cited_text 最多是 up to 150 characters of the cited content

簡單來講就是：最多 150 個字元的引用文字。而這些還只是 API 層級的規格而已。

Claude Code 內建的 WebSearch 又再砍一刀，根據 Mikhail Shilkov 拆解 Claude Code 內部行為的這篇文章 裡面有提到...Claude Code 連 page_age 跟 encrypted_content 都丟掉了，只保留 title 跟 url。

所以簡單來講...在搜尋這個階段 AI 看到你網站的東西就是只有 一個標題加一個網址 沒了。

WebFetch 撈到了什麼？

那接下來就要來聊一下，內文何時會被撈出來了，也就是前面有提到的，第二階段 WebFetch。

AI 看完搜尋結果之後，如果決定點開某幾筆，會對每個網址發一次 WebFetch 請求，這時候才會把整篇內文撈回來。那撈取範圍多大？

這邊需要分成兩層來說明，因為 API 跟 Claude Code 的設計不太一樣。

Note

這邊講的 API 是指 Anthropic API 的 web_fetch tool，Claude Code 是指 Anthropic 自己的產品線裡面內建的 WebFetch 功能，兩者的規格跟流程不太一樣。

API 層級的 web_fetch

Anthropic API 的 web_fetch tool 有一個叫 max_content_tokens 的參數，可以由開發者自己設定，雖然官方文件範例用的是 10 萬 tokens。

但官方文件上也有給參考換算：

內容大小估算 Token 數
平均網頁 10 KB約 2,500 tokens
大型文件頁 100 KB約 25,000 tokens
研究論文 PDF 500 KB約 125,000 tokens

由此可知，一篇中長度的中文部落格文章純文字通常 1~2000 tokens 就非常足夠了，遠遠低於 10 萬的上限。所以不太會發生「內文被截斷」的問題，除非你寫了一篇五萬字的超長文章（紅樓夢之類？）。

但這邊要特別注意，web_fetch tool 的 citation 跟 web_search 的設計不太一樣，它是使用 start_char_index / end_char_index 選出文章中的特定位置（雖然文件沒寫死字元上限就是了）。

Claude Code 內建的 WebFetch

Claude Code 內建的 WebFetch 走的是另一個方式。

依照 Mikhail Shilkov 的拆解， WebFetch 它的流程會是這樣的：

  1. 把 HTML 用 Turndown 套件轉成 Markdown
  2. 擷取內文的 100 KB 純文字
  3. 把這 100 KB 丟給 Haiku 3.5 模型
  4. Haiku 根據你給它的 prompt 摘要出答案，再把摘要回傳給主模型

這邊其實真正的關鍵還是在第三步驟，你會發現我們主模型（也就是我們當前使用的 Claude 模型）並不會直接看到網頁的原文，它都是看到 Haiku 摘要過的版本。也就是說，你寫的東西最後變成什麼樣回到主模型，是 Haiku 怎麼讀的決定，不是你寫了多少的決定。

而且引用部分也有限制，Mikhail 從 Claude Code 內部 prompt 抓到的規則是 Enforce a strict 125-character maximum for quotes from any source document，也就是說引用的部分最多只能有 125 個字元。

這時候應該又有一個疑問了，所以 100 KB 的純文字大概是多少呢？ 假設以中文來說，每個字算 3 bytes 的話，100 KB 大概可以裝 3 萬多字；如果是英文，每個字算 1 byte 的話，可以裝十幾萬字。對於一個普通部落格文章來講，再長也撐不到這個量了。所以真正該關注的點應該是「怎麼讓 Haiku 摘到我想被引用的內容」，而不是「我寫了多少內容給 Haiku 看」。

拿我的部落格實測一輪

講了那麼多，這邊我們就來實際測試一下吧！

我這邊是直接請 Claude 呼叫 WebFetch 工具，然後把我部落格首頁（israynotarray.com）的 URL 給它，讓它去抓內文。

那麼該怎麼實際測試呢？我們可以善加利用 Hook 的特性，將每一個階段的輸入輸出都記錄下來，這樣就能清楚看到 AI 在每個階段實際撈到了什麼內容。

底下是我的 Hook 設定，這樣每次 WebSearch 跟 WebFetch 的輸入輸出都會被記錄下來：

JSON
············································································································································································{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "WebSearch|WebFetch",
        "hooks": [
          {
            "type": "command",
            "command": "/claude-tools/hooks/log-web-tool.sh"
          }
        ]
      }
    ],
    "SessionEnd": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/claude-tools/hooks/dump-transcript.sh"
          }
        ]
      }
    ]
  }
}

這邊我提示詞（Prompt）是以下：

Text
請你幫我用 WebFetch 工具抓一下 israynotarray.com 首頁的內容，然後告訴我你從這個頁面摘了什麼內容回來。

實際 Input 內容如下：

JSON
····{
  "url": "https://israynotarray.com",
  "prompt": "請告訴我這個首頁的主要內容：站名、站長介紹、最新文章列表（標題與簡介）、分類、以及任何導覽列或側邊欄資訊。"
}

接著跑完之後，你可以看到 WebFetch 的結果裡面，AI 摘要回來的內容大概是這樣的：

JSON
············{
  "bytes": 1068,
  "code": 200,
  "codeText": "OK",
  "result": "# 網頁內容分析\n\n基於提供的內容片段，我能提供以下資訊：\n\n## 站點結構\n- **主要分類**：AI 相關內容，包括 Gemini 和 Gemini Omni 產品\n\n## 最新文章\n**標題**：「Gemini Omni 影片生成怎麼用？30 個 Reels、產品 Demo、Logo Reveal 提示詞模板」\n\n**簡介摘要**：\n本文介紹了 Google 在 2026 年 5 月 I/O 發布的 Gemini Omni 影片生成功能，說明其如何取代先前的 Veo 3.1。文章涵蓋「一次 10 秒短片、原生同步音訊、中文文字渲染」等新功能，並提供 30 個實用提示詞模板，適用於 Instagram Reels、產品演示、Logo 動畫、B-roll、轉場效果、引言卡片、生活風格與美食旅遊等多種場景。\n\n**發布日期**：2026-06-06  \n**閱讀時間**：約 23 分鐘\n\n## 限制\n提供的內容片段不含站長介紹或完整導覽列資訊。",
  "durationMs": 4603,
  "url": "https://israynotarray.com"
}

發現了嗎？你真正被 AI 撈走的內容其實只有這一小段 1000 字以內的摘要而已，一個網站裡面根本不可能只有這麼一小段內容，剩下的部分根本沒有被 AI 看到。也就是說中介層的 Haiku 模型看完之後，只挑了它認為跟我 prompt 相關的部分摘給我。同樣的頁面如果我用瀏覽器打開，看到的會是一整列文章卡片加 sidebar，可是 Haiku 不會把整列丟回來。

我順便試了一篇舊文，故意把 URL 路徑的某段寫錯，它就只會回傳這樣：

JSON
············{
  "bytes": 0,
  "code": 404,
  "codeText": "Not Found",
  "result": "The server returned HTTP 404 Not Found.\n\nThe response body was not retrieved. If this URL requires authentication, use an authenticated tool (e.g. `gh` for GitHub, or an MCP-provided fetch tool) instead of WebFetch.",
  "durationMs": 588,
  "url": "https://israynotarray.com/dqwdqwdqwd"
}

你會發現連你寫在 404 頁面的內容都撈不到，因為 WebFetch 直接回報了 404 Not Found 的錯誤，AI 根本沒辦法看到你 404 頁面上寫了什麼內容。也就是說，如果你的網站有任何路徑問題、重構過 URL、或者只有前端路由沒有實體頁面，AI 就撈不到內容了。

順帶一提，這部分跟 Claude 官方文件裡的注意事項也對得起來：

The web fetch tool currently does not support websites dynamically rendered with JavaScript.

如果你的部落格是用前端 SPA 撐起來、內容完全靠 JavaScript runtime 渲染的話，那 AI 撈到的可能只是空殼 HTML，看不到任何文章。因此使用靜態產生器（像 Hexo、Astro、Next.js 的 SSG 模式）相對安全，因為 build 出來的就是完整 HTML，AI 一抓就有東西看（歡迎加入 Hexo 行列）。

robots.txt 那層也別忘了

接下來還有一個重點，AI 撈不撈得到你的網站，有一個大前提，也就是 robots.txt。

AI 爬蟲基本上分兩種：搜尋型（會引用 + 連回你網站）跟訓練型（吃內容餵模型，不一定有回引）。常見的對應大概長這樣：

爬蟲類型行為
Claude-SearchBot / Claude-User搜尋Claude 回答時即時抓取，會引用
ClaudeBot訓練抓內容餵 Claude 訓練
OAI-SearchBot / ChatGPT-User搜尋ChatGPT 回答時即時抓取，會引用
GPTBot訓練抓內容餵 GPT 訓練
PerplexityBot搜尋Perplexity 引擎用，會引用
Google-Extended訓練Gemini 訓練用
CCBot訓練Common Crawl 公開資料集

如果你希望被 AI 引用、可是不想被拿去訓練，最常見的策略就是「搜尋型放行、訓練型擋掉」。

這邊我也給你一個可以直接複製貼上使用的 robots.txt 範例：

Text
# 搜尋型 AI 爬蟲：放行（會回引）
User-agent: OAI-SearchBot
Allow: /

User-agent: ChatGPT-User
Allow: /

User-agent: Claude-SearchBot
Allow: /

User-agent: Claude-User
Allow: /

User-agent: PerplexityBot
Allow: /

# 訓練型 AI 爬蟲：擋下（吃資料不回引）
User-agent: GPTBot
Disallow: /

User-agent: ClaudeBot
Disallow: /

User-agent: Google-Extended
Disallow: /

User-agent: CCBot
Disallow: /

User-agent: Bytespider
Disallow: /

# 內容信號：可被搜尋索引、不可訓練、不直接餵 AI 當輸入
Content-Signal: ai-train=no, search=yes, ai-input=no

robots.txt 的詳細寫法可以看我之前寫的 快速掌握 robots.txt 用途，整套 Agent Readiness 的滿分設定可以看 從 3 分到滿分！讓你的網站通過 isitagentready AI Agent Readiness 完整攻略。

這對寫文章有什麼具體影響？

那講完了這些東西，所以這件事情對於寫文章有什麼具體影響呢？答案是有的，你只要理解上面的限制跟細節之後，寫文章就可以針對這四件事情多下功夫。

標題要能獨立成立

搜尋階段 AI 看到你的文章，就只有 title 跟 URL 兩個欄位。

如果你的標題需要副標 or 上下文才看得懂，AI 把它跟其他十幾筆結果排在一起時就會被略過。

舉個對照：

  • 比較弱的標題：「實作筆記」
  • 比較強的標題：「Cloudflare Worker 把 HTML 轉 Markdown 的內容協商完整實作筆記」

後者把「主題、工具、做什麼事、文章類型」都塞進去了，AI 不用點開頁面就知道值不值得撈。

首段就直接給答案

WebFetch 階段，中介的 Haiku 模型是從上往下讀的，前 300 到 500 字決定它摘什麼回去。如果你的開頭是「在我們聊 X 之前，先來回顧一下歷史...」，Haiku 讀到一半發現都是背景沒答案，就直接摘背景了。

正確的寫法是每個 H2 段落開頭第一句話就直接給結論，後面再補脈絡。這個原則我在 SEO 已經不夠了嗎？認識 AEO 答案引擎優化 那篇也有提過，可以一起看。

設計可獨立引用的單句

cited_text 在 web_search 是 150 字元、Claude Code 內建 WebFetch 是 125 字元。這代表 AI 引用你的時候，能塞的就是一段「沒有上下文也能成立」的短句。

寫作的時候有意識地去設計這種句子，例如：

  • 比較弱：「這個跟前面提到的那個方法不太一樣，主要差別在...」（沒有上下文看不懂）
  • 比較強：「llms.txt 是 Answer.AI 共同創辦人 Jeremy Howard 在 2024 年提出的，目的是主動告訴 AI 網站有什麼重要內容。」（不需要前後文就成立）

寫完一段之後可以自己挑一句出來看，如果單獨拿出來給沒看過全文的人都看得懂，那這句就有機會被引用。

結構化內容當 Haiku 的航標

H2、H3、- 列表、表格、```js 程式碼區塊，這些 Markdown 結構對中介摘要層特別有用。Haiku 讀 HTML 轉出來的 Markdown 時，會把標題當「這段在講什麼」的索引，把列表當「主要要點」的訊號，把表格當「資料佐證」的單位。

如果你整篇文章都是純散文段落，Haiku 沒有航標只能靠語意硬讀，摘出來的東西就會偏散；如果你有清楚的結構，Haiku 摘的時候可以直接照航標走，得到的摘要會跟你想被引用的重點對齊。

結語

最後總結一下吧。

所以 AI 搜尋時，實際撈走你部落格多少呢？

我認為答案可以分成三層：

  • 搜尋階段： 只有 title 加 URL
  • 撈內文階段： API 預設裝得下你整篇，可是 Claude Code 走 Haiku 中介摘要 + 100 KB 截斷
  • 引用階段： web_search 是 150 字元，Claude Code WebFetch 是 125 字元

寫文章針對的就是這三道閘門，不是讓 AI 把你整篇背起來。

希望這篇有讓你對 AI 搜尋實際撈走多少內容這件事有更具體的概念哩～

如果你的部落格還沒做 AI bot 分流，可以參考前面那段 robots.txt 範例先把基礎建設起來，剩下的就交給內容慢慢累積了。
